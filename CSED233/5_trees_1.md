### **핵심 개념**

*   **트리 (Tree)**: 노드(Node)와 간선(Edge)으로 구성된 비선형 계층적 자료구조입니다. 데이터 간의 1:N 관계를 표현하는 데 사용됩니다.
*   **노드 (Node)**: 트리의 각 요소를 의미하며, 데이터와 자식 노드에 대한 링크(포인터)를 포함할 수 있습니다.
*   **간선 (Edge)**: 두 노드 사이의 연결을 나타냅니다.
*   **루트 노드 (Root Node)**: 트리의 최상위에 위치한 노드로, 부모 노드가 없습니다. 모든 트리는 하나의 루트 노드를 가집니다.
*   **부모 노드 (Parent Node)**: 특정 노드의 바로 위에 연결된 노드입니다.
*   **자식 노드 (Child Node)**: 특정 노드의 바로 아래에 연결된 노드입니다.
*   **형제 노드 (Sibling Node)**: 같은 부모 노드를 가지는 노드들을 의미합니다.
*   **리프 노드 (Leaf Node) / 단말 노드**: 자식 노드가 없는 노드입니다.
*   **내부 노드 (Internal Node)**: 리프 노드가 아닌 노드로, 자식 노드를 하나 이상 가집니다.
*   **경로 (Path)**: 한 노드에서 다른 노드까지 간선들을 따라 이동하는 일련의 과정입니다.
*   **경로의 길이 (Length of Path)**: 경로를 구성하는 간선의 수입니다.
*   **노드의 깊이 (Depth of Node)**: 루트 노드로부터 특정 노드까지의 경로 길이입니다. 루트 노드의 깊이는 0입니다.
*   **트리의 높이 (Height of Tree)**: 트리에 있는 노드들의 깊이 중 최댓값, 즉 루트로부터 가장 멀리 떨어진 리프 노드까지의 경로 길이입니다.
*   **노드의 차수 (Degree of Node)**: 특정 노드의 자식 노드의 수입니다.
*   **트리의 차수 (Degree of Tree)**: 트리 내 노드들의 차수 중 가장 큰 값입니다.
*   **서브트리 (Subtree)**: 루트 노드를 포함한 원래 트리의 특정 노드를 루트로 하는 부분 트리입니다.

### **코드/수식 해설**

**일반적인 트리 노드 구조 (C/C++ 예시)**
트리는 자식 노드의 수가 가변적일 수 있으므로, 자식 노드를 동적 배열이나 연결 리스트 등으로 관리하는 구조를 생각할 수 있습니다.

```c
// 일반적인 트리 노드 구조 (자식 수가 정해지지 않은 경우)
#include <stdio.h>
#include <stdlib.h>

typedef struct Node {
    int data;
    struct Node** children; // 자식 노드들을 가리키는 포인터 배열
    int num_children;      // 현재 자식 노드의 수
    int capacity_children; // children 배열의 현재 용량
} Node;

// 노드 생성 함수 예시
Node* createNode(int data) {
    Node* newNode = (Node*)malloc(sizeof(Node));
    if (newNode == NULL) {
        perror("Failed to allocate memory for node");
        exit(EXIT_FAILURE);
    }
    newNode->data = data;
    newNode->children = NULL;
    newNode->num_children = 0;
    newNode->capacity_children = 0;
    return newNode;
}

// 자식 노드 추가 함수 예시
void addChild(Node* parent, Node* child) {
    if (parent->num_children >= parent->capacity_children) {
        // 용량이 부족하면 확장
        int new_capacity = parent->capacity_children == 0 ? 2 : parent->capacity_children * 2;
        Node** new_children = (Node**)realloc(parent->children, new_capacity * sizeof(Node*));
        if (new_children == NULL) {
            perror("Failed to reallocate memory for children");
            exit(EXIT_FAILURE);
        }
        parent->children = new_children;
        parent->capacity_children = new_capacity;
    }
    parent->children[parent->num_children++] = child;
}
```

이진 트리는 자식 노드의 수가 최대 두 개로 정해져 있기 때문에 구조가 더 간단합니다.

```c
// 이진 트리 노드 구조 (자식이 최대 2개: left, right)
typedef struct BinaryTreeNode {
    int data;
    struct BinaryTreeNode* left;  // 왼쪽 자식 노드
    struct BinaryTreeNode* right; // 오른쪽 자식 노드
} BinaryTreeNode;

// 이진 노드 생성 함수 예시
BinaryTreeNode* createBinaryTreeNode(int data) {
    BinaryTreeNode* newNode = (BinaryTreeNode*)malloc(sizeof(BinaryTreeNode));
    if (newNode == NULL) {
        perror("Failed to allocate memory for binary tree node");
        exit(EXIT_FAILURE);
    }
    newNode->data = data;
    newNode->left = NULL;
    newNode->right = NULL;
    return newNode;
}
```

**트리의 높이/깊이 수식**

*   **노드 $N$의 깊이 (depth($N$))**:
    *   루트 노드 $R$의 깊이: $depth(R) = 0$
    *   노드 $N$이 루트 노드가 아닐 경우: $depth(N) = depth(Parent(N)) + 1$

*   **노드 $N$의 높이 (height($N$))**:
    *   리프 노드 $L$의 높이: $height(L) = 0$
    *   내부 노드 $N$의 높이: $height(N) = \max(\text{height}(\text{child}_1), \text{height}(\text{child}_2), \dots, \text{height}(\text{child}_k)) + 1$
    *   트리의 높이: 루트 노드의 높이와 같습니다. $Height(\text{Tree}) = height(\text{Root})$

### 핵심 개념

*   **트리의 정의**: 트리는 `노드(nodes)` (원형으로 표현)와 `엣지(edges)` (선으로 표현)의 집합입니다. 노드는 데이터를 저장하는 개체이며, 엣지는 노드 간의 관계를 나타냅니다.
*   **루트(Root)**: 트리는 하나의 특별한 노드를 가집니다. 이 노드를 `루트(root)`라고 부르며, 트리의 가장 상위에 위치하여 모든 다른 노드의 조상이 됩니다. 루트 노드는 부모 노드를 가지지 않습니다.
*   **엣지의 연결**: 트리의 각 엣지는 정확히 두 노드를 연결하며, 이들은 `부모(parent)`와 `자식(child)` 관계를 형성합니다. 엣지는 일반적으로 부모에서 자식으로 향하는 방향성을 가집니다.
*   **노드의 연결성**: 하나의 노드는 여러 엣지에 연결될 수 있습니다. 특히, 부모 노드는 여러 자식 노드를 가질 수 있습니다.
*   **시각화 관례**: 실제 나무는 뿌리가 아래에 있지만, 컴퓨터 과학에서 트리 자료구조를 표현할 때는 관례적으로 `루트 노드`가 가장 위에 오도록 뒤집어서 그립니다. 이는 계층 구조를 직관적으로 파악하기 위함입니다.
*   **노드의 종류**:
    *   **루트 노드 (Root Node)**: 트리의 시작점이 되는 가장 상위 노드 (슬라이드에서는 빨간색으로 표시).
    *   **비-말단 노드 (Non-terminal Node)**: 자식 노드를 하나 이상 가지는 노드 (슬라이드에서는 파란색으로 표시).
    *   **말단 노드 / 리프 노드 (Terminal Node / Leaf Node)**: 자식 노드가 없는 노드. 트리의 가장 아래쪽에 위치합니다 (슬라이드에서는 회색으로 표시).

### 구체적 예시

*   **파일 시스템**: 컴퓨터의 파일 시스템은 대표적인 트리 구조입니다. 최상위 디렉토리(예: `C:\` 또는 `/`)가 루트 노드가 되고, 각 서브 디렉토리는 부모-자식 관계를 가지는 노드가 됩니다. 최종적으로 각 파일은 더 이상 자식(하위 디렉토리나 파일)이 없는 리프 노드가 됩니다.
*   **가계도**: 가족 구성원 간의 관계를 나타내는 가계도 역시 트리 구조로 볼 수 있습니다. 가장 오래된 조상이 루트 노드가 되고, 부모-자식 관계가 엣지로 연결됩니다. 자녀가 없는 구성원은 리프 노드가 됩니다.
*   **웹사이트 내비게이션**: 웹사이트의 메뉴 구조나 페이지 간의 연결도 트리 형태로 구성될 수 있습니다. 홈페이지가 루트가 되고, 각 카테고리나 서브페이지가 노드를 형성합니다.

### 시험 포인트

*   ⭐ 트리의 기본적인 정의 (노드와 엣지의 집합)와 함께 `루트`, `부모`, `자식`, `리프(leaf)` 등 핵심 용어들을 정확히 이해하고 구별할 수 있어야 합니다. 각 용어의 정의와 역할에 대한 질문이 나올 수 있습니다.
*   ⭐ 트리는 `사이클(cycle)`이 없는 `연결된(connected)` 그래프입니다. 이 특성이 일반적인 그래프와 트리를 구분하는 가장 중요한 차이점 중 하나입니다. (슬라이드에 명시되지는 않았지만, 트리의 본질적인 특징이므로 중요합니다.)
*   ⭐ 트리의 시각화 관례(루트가 상단에 위치)를 이해하고, 주어진 트리 구조에서 각 노드의 종류(루트, 비-말단, 리프)를 식별할 수 있어야 합니다.
*   ⭐ 부모-자식 관계의 특징, 즉 하나의 엣지는 두 노드를 연결하며 노드 하나가 여러 엣지에 연결될 수 있다는 점(특히 부모가 여러 자식을 가질 수 있음)을 파악하는 것이 중요합니다.

## Slide 3

**핵심 개념**
이 슬라이드는 바둑과 같은 게임에서 인공지능이 최적의 수를 찾기 위해 사용하는 **몬테카를로 트리 탐색(Monte Carlo Tree Search, MCTS)** 알고리즘을 트리 자료구조의 관점에서 설명합니다.

*   **노드 (Node)**: 각 노드는 바둑 게임의 특정 **말판 상태(Board State)**를 나타냅니다. 즉, 특정 시점까지 돌이 놓인 바둑판의 배치를 의미합니다.
*   **간선 (Edge)**: 각 간선은 한 말판 상태에서 다른 말판 상태로 이어지는 **가능한 다음 수(Possible Move)**를 나타냅니다. 즉, 특정 위치에 돌을 놓는 행위를 의미합니다.

**코드/수식 해설**

*   **a. Selection (선택)**:
    *   MCTS는 트리의 루트 노드(현재 바둑판 상태)에서 시작하여 특정 자식 노드를 선택하며 트리를 아래로 내려갑니다. 이때 선택 기준은 **UCT(Upper Confidence Bound for Trees)** 공식을 기반으로 합니다.
    *   $Q$: 해당 노드(말판 상태)의 평균적인 승리 가치 또는 예상 보상.
    *   $u(P)$: 탐색(exploration)과 활용(exploitation)의 균형을 맞추기 위한 항. 방문 횟수가 적은 노드에 더 많은 탐색 기회를 줍니다. $P$는 정책 신경망(Policy Network)에서 얻은 초깃값 또는 확률을 나타냅니다.
    *   $\max (Q + u(P))$: 현재 노드에서 자식 노드(가능한 다음 수) 중 $Q + u(P)$ 값이 가장 큰 수를 선택하여 트리를 하강합니다.
    *   **UCT 공식 (일반적인 형태)**:
        $$ Q(s, a) + C \sqrt{\frac{\ln N(s)}{N(s, a)}} $$
        여기서 $Q(s, a)$는 상태 $s$에서 행동 $a$를 취했을 때의 평균 가치, $N(s)$는 상태 $s$의 총 방문 횟수, $N(s, a)$는 상태 $s$에서 행동 $a$를 취했을 때의 방문 횟수, $C$는 탐색 상수입니다.

*   **b. Expansion (확장)**:
    *   선택 단계를 통해 도달한 노드가 아직 충분히 탐색되지 않은 경우, 해당 노드에서 **새로운 자식 노드들을 생성**하여 트리를 확장합니다.
    *   $P$ 또는 $p_{\sigma}(\cdot)$: 정책 신경망 $\sigma$로부터 얻은 현재 말판 상태에서의 가능한 각 수의 **확률 분포**입니다. 이 확률을 바탕으로 새로운 자식 노드들이 추가됩니다.

*   **c. Evaluation (평가)**:
    *   새롭게 확장된 노드 또는 선택된 리프 노드에 대해 **가치를 평가**합니다.
    *   $v_{\theta}(\cdot)$: 가치 신경망 $\theta$로부터 예측된 현재 말판 상태의 **승리 확률** 또는 최종 가치입니다.
    *   $\sim p_{\pi}$: 정책 $\pi$에 따라 게임을 끝까지 진행하는 **시뮬레이션(롤아웃)**을 수행하여 최종 결과를 얻습니다.
    *   $r(\cdot)$: 시뮬레이션의 최종 **보상(reward)**을 나타냅니다. 예를 들어, 승리 시 +1, 패배 시 -1. 이 보상은 트리를 거슬러 올라가며 부모 노드들의 가치를 업데이트하는 데 사용됩니다.

**구체적 예시**
알파고(AlphaGo)가 바둑 대국 중 다음 수를 결정하는 과정을 예로 들 수 있습니다.
1.  **Selection**: 현재 바둑판 상태(루트 노드)에서 시작하여, 이전에 탐색했던 수들 중 지금까지 이길 확률이 높다고 평가되면서도 아직 충분히 탐색되지 않은 수(노드)를 선택하여 트리를 깊이 탐색해 나갑니다.
2.  **Expansion**: 만약 선택 경로의 끝에 도달한 노드가 아직 자식 노드를 다 만들지 않았다면, 정책 신경망을 이용하여 다음 가능한 수들을 예측하고, 그 수들을 나타내는 새로운 자식 노드들을 트리에 추가합니다.
3.  **Evaluation**: 새로 생성된 노드 중 하나에서 게임의 최종 결과(승패)를 예측하거나, 실제 게임을 시뮬레이션하여 보상을 얻습니다. 예를 들어, 해당 수가 승리로 이어진다면 +1점, 패배로 이어진다면 -1점을 얻습니다.
4.  **Backpropagation (역전파)**: 얻어진 보상 값은 선택 단계에서 내려왔던 경로를 따라 루트 노드까지 거꾸로 전파됩니다. 이 과정에서 각 노드의 평균 가치($Q$)와 방문 횟수가 업데이트되어 다음 탐색 시 더 나은 수를 선택할 수 있도록 정보를 갱신합니다.

이러한 과정을 반복하여 충분한 탐색 후, 루트 노드의 자식 노드들 중 가장 높은 $Q$ 값을 가진 수를 최종 착수로 결정합니다.

**시험 포인트**
*   ⭐ **트리 노드와 간선이 각각 무엇을 의미하는지** (노드: 말판 상태, 간선: 가능한 수) 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **몬테카를로 트리 탐색(MCTS)의 4가지 주요 단계(Selection, Expansion, Evaluation/Simulation, Backpropagation)의 개념과 각 단계에서 일어나는 일**을 설명할 수 있어야 합니다. (슬라이드에는 Backpropagation이 명시적으로 없지만 MCTS의 핵심 요소입니다.)
*   ⭐ **$Q+u(P)$와 같은 수식이 MCTS에서 어떤 목적(탐색-활용 딜레마 해결)으로 사용되는지** 설명할 수 있어야 합니다.
*   ⭐ **정책 네트워크($P$, $p_{\sigma}$)와 가치 네트워크($v_{\theta}$)의 역할**을 구분할 수 있어야 합니다.
*   ⭐ **알파고와 같은 게임 AI에서 트리 자료구조(MCTS)가 어떻게 활용되는지** 구체적인 예시를 들어 설명할 수 있어야 합니다.

**핵심 개념**
이 슬라이드는 **공간 분할 트리(Spatial Partitioning Tree)**의 한 예시를 제시하며, 특히 3차원 공간을 재귀적으로 8등분하여 표현하는 **옥트리(Octree)**의 개념을 시각화하고 있습니다. 이는 방대한 양의 3D 공간 데이터나 객체들을 효율적으로 관리하고 검색하기 위한 계층적 자료구조입니다.

*   **노드(Node)**: 트리에서 각 원형 노드는 특정 3차원 공간 영역(예: 정육면체)을 나타냅니다.
    *   최상위 루트 노드는 전체 공간을 포괄하는 가장 큰 영역을 의미합니다.
    *   내부 노드는 더 큰 영역이 세분화된 중간 단계의 서브 영역들을 의미합니다.
    *   가장 아래쪽의 리프(Leaf) 노드는 더 이상 분할되지 않는 가장 작은 공간 단위 또는 특정 조건을 만족하는 공간 영역을 나타냅니다. 이 리프 노드들은 해당 공간 영역에 존재하는 객체들(슬라이드 하단의 3D 모델들)에 대한 정보를 포함할 수 있습니다.

*   **엣지(Edge)**: 트리에서 부모 노드와 자식 노드를 연결하는 선은 공간 분할의 계층적 관계를 나타냅니다. 즉, 부모 노드의 공간 영역이 자식 노드들의 공간 영역으로 세분화됨을 의미합니다. 예를 들어, 부모 노드가 한 정육면체를 나타낸다면, 그 자식 노드들은 해당 정육면체를 8등분한 작은 정육면체들을 나타냅니다.

**코드/수식 해설**
슬라이드에는 직접적인 코드나 수식이 제시되어 있지 않습니다. 다만, 옥트리의 노드는 일반적으로 다음과 같은 정보를 포함할 수 있습니다.

```c
// 옥트리 노드의 개념적 구조
typedef struct OctreeNode {
    BoundingBox region; // 이 노드가 담당하는 3차원 공간 영역 (예: 최소/최대 좌표)
    OctreeNode* children[8]; // 8개의 자식 노드 포인터 (해당 공간이 8개로 분할됨)
    ListOfObjects* objects_in_region; // 이 영역에 포함된 객체들의 리스트 (리프 노드에서 주로 사용)
    bool isLeaf; // 리프 노드 여부
    // 기타 메타데이터 (예: 깊이, 부모 노드 포인터 등)
} OctreeNode;
```

**구체적 예시**
슬라이드 하단의 다양한 3D 모델들(건물, 나무, 동물, 탈것 등)은 컴퓨터 그래픽스 렌더링 환경에서 관리되어야 할 '객체'들을 나타냅니다. 옥트리는 이러한 객체들로 구성된 방대한 3D 장면을 효율적으로 관리하고 처리하는 데 사용됩니다.

1.  **장면 관리(Scene Management)**: 거대한 가상 세계를 하나의 큰 큐브로 보고, 이 큐브를 재귀적으로 분할하여 옥트리를 구성합니다. 각 3D 객체는 하나 이상의 옥트리 노드(리프 노드 또는 내부 노드)에 속하게 됩니다.
2.  **컬링(Culling)**: 사용자의 시야(카메라 frustum)에 들어오지 않는 공간 영역(노드)에 있는 객체들은 렌더링에서 제외할 수 있습니다. 이를 통해 불필요한 연산을 줄여 렌더링 성능을 크게 향상시킬 수 있습니다.
3.  **충돌 감지(Collision Detection)**: 두 객체가 충돌하는지 여부를 확인할 때, 모든 객체 쌍을 검사하는 대신, 옥트리를 활용하여 같은 공간 영역에 속한 객체들만 집중적으로 검사함으로써 효율성을 높일 수 있습니다.
4.  **수준별 상세도(Level of Detail, LOD)**: 멀리 있는 객체는 낮은 상세도(간단한 모델)로 렌더링하고, 가까이 있는 객체는 높은 상세도(복잡한 모델)로 렌더링할 때, 옥트리가 객체의 위치와 카메라 거리를 기반으로 적절한 LOD를 선택하는 데 도움을 줄 수 있습니다.

**시험 포인트**
*   ⭐ **옥트리(Octree) 및 쿼드트리(Quadtree)의 정의 및 목적**: 3차원/2차원 공간 분할을 통한 데이터 관리 방식과 주요 응용 분야를 이해해야 합니다.
*   ⭐ **노드와 엣지의 의미**: 주어진 예시(공간 분할)에서 노드가 나타내는 것(공간 영역)과 엣지가 나타내는 것(계층적 관계, 공간 분할)을 정확히 설명할 수 있어야 합니다.
*   ⭐ **공간 분할 트리의 활용 분야 및 장점**: 컴퓨터 그래픽스, GIS, 게임 개발 등에서 왜 이러한 자료구조를 사용하는지(효율적인 검색, 렌더링, 충돌 감지, LOD 관리 등) 설명할 수 있어야 합니다.
*   ⭐ **트리의 재귀적 분할 과정 이해**: 슬라이드의 큐브 분할 그림처럼, 초기 공간이 어떻게 재귀적으로 더 작은 하위 공간으로 나뉘어 트리를 형성하는지 설명할 수 있어야 합니다. (예: 2차원 쿼드트리는 4분할, 3차원 옥트리는 8분할)

## 트리(Tree)의 재귀적 정의 (Recursive Definition)

이 슬라이드는 트리의 기본적인 구조를 재귀적인 방식으로 정의하는 방법을 설명합니다. 이는 트리라는 자료구조의 본질을 이해하는 데 매우 중요합니다.

---

### **핵심 개념**

트리는 다음과 같은 두 가지 규칙을 통해 재귀적으로 정의될 수 있습니다.

1.  **단일 노드는 트리이다 (Base Case)**: 아무런 연결 없이 오직 하나의 노드 $n$ 만 존재해도, 그 자체로 하나의 트리로 간주합니다. 이 노드는 트리의 루트이자 유일한 노드가 됩니다.
2.  **새로운 트리를 구성하는 방법 (Recursive Step)**:
    *   하나의 노드 $n$ 을 준비합니다. 이 노드가 새로운 트리의 루트가 될 것입니다.
    *   별개로 존재하는 $k$ 개의 트리를 준비합니다. 이 트리를 각각 $T_1, T_2, \dots, T_k$ 라 하고, 각 트리의 루트 노드를 $n_1, n_2, \dots, n_k$ 라고 합시다.
    *   노드 $n$ 을 루트로 하고, $T_1, T_2, \dots, T_k$ 의 루트 $n_1, n_2, \dots, n_k$ 를 노드 $n$ 의 자식으로 연결하여 새로운 트리를 구성합니다.
    *   이때, $T_1, T_2, \dots, T_k$ 는 노드 $n$ 의 **서브트리(Subtrees)**가 됩니다.

이 정의를 통해, 트리는 단일 노드에서 시작하여 점차 더 복잡한 구조로 확장될 수 있음을 알 수 있습니다.

### **코드/수식 해설**

슬라이드에 명시된 코드는 없지만, 그림은 트리가 어떻게 재귀적으로 구성되는지 시각적으로 보여줍니다.

*   **상단 그림**: 노드 $n$ 하나가 'Single node'라는 레이블과 함께 그려져 있습니다. 이는 트리의 가장 기본적인 형태, 즉 **재귀의 종료 조건(Base Case)**을 나타냅니다.
*   **하단 그림**:
    *   왼쪽에는 루트 노드 $n$ 과 $k$ 개의 개별 트리 ($T_1, \dots, T_k$)가 각각의 루트 ($n_1, \dots, n_k$)와 함께 그려져 있습니다.
    *   오른쪽에는 이들을 조합하여 'New Tree'가 생성된 모습이 있습니다. 노드 $n$ 이 새로운 트리의 루트가 되고, 각 서브트리 $T_i$ 의 루트 $n_i$ 가 $n$ 의 자식으로 연결됩니다. 이 과정이 바로 **재귀 단계(Recursive Step)**입니다.

$$ \text{Tree} = \begin{cases} \text{A single node } n & \text{(Base Case)} \\ \text{A node } n \text{ with } k \text{ subtrees } T_1, \dots, T_k & \text{(Recursive Step)} \end{cases} $$
여기서 각 서브트리 $T_i$ 역시 위 정의를 따르는 트리입니다.

### **구체적 예시**

1.  **시작**: 노드 'A'는 하나의 트리입니다. 노드 'B'도 트리이고, 노드 'C'도 트리입니다. (Base Case 적용)
2.  **1단계 구성**:
    *   'A'를 루트 노드로 선택합니다.
    *   'B'와 'C'를 각각 서브트리로 간주하고, 'A'의 자식으로 연결합니다.
    *   이제 우리는 'A'를 루트로 하고 'B'와 'C'를 자식으로 하는 트리를 가집니다. (Recursive Step 적용)
3.  **2단계 구성 (재귀 적용)**:
    *   만약 'B' 노드가 다시 'D'와 'E'라는 노드(각각 단일 트리)를 자식으로 가질 수 있습니다.
    *   이때 'B'는 'A'의 서브트리인 동시에, 'D'와 'E'의 루트 노드가 되는 새로운 트리의 루트가 됩니다.

이처럼 트리는 가장 작은 단위인 노드에서 시작하여, 다른 작은 트리들을 자식으로 붙여나가며 계층적인 구조를 형성합니다.

### **시험 포인트**

*   ⭐ **트리의 재귀적 정의**를 설명할 수 있어야 합니다 (Base Case와 Recursive Step을 구분하여 설명).
*   ⭐ **루트 노드**, **서브트리**의 개념을 정확히 이해하고 있어야 합니다. 트리의 재귀적 정의에서 서브트리가 어떻게 형성되는지 파악하는 것이 중요합니다.
*   트리가 **계층적인 구조**를 가지는 이유가 바로 이 재귀적 정의에 있음을 이해해야 합니다.

## Slide 6

이 슬라이드는 트리(Tree) 자료구조에서 노드 간의 계층적 관계를 설명합니다.

---

### **핵심 개념**

*   **Parent/Child (부모/자식)**
    *   **부모 (Parent)**: 어떤 노드에 직접 연결되어 그 노드보다 한 단계 위에 있는 노드를 의미합니다. 즉, 자식 노드를 가리키는 간선(edge)의 시작점이 됩니다.
    *   **자식 (Child)**: 어떤 노드에 직접 연결되어 그 노드보다 한 단계 아래에 있는 노드를 의미합니다. 즉, 부모 노드로부터 오는 간선의 끝점이 됩니다.
    *   예시: 슬라이드에서 "A is the parent of C" (A는 C의 부모), "C is the child of A" (C는 A의 자식)

*   **Ancestor/Descendant (조상/자손)**
    *   **조상 (Ancestor)**: 어떤 노드의 부모 노드뿐만 아니라, 그 부모의 부모 노드(할아버지 노드) 등, 루트(root)까지 이르는 모든 상위 노드를 통틀어 조상이라고 합니다. 자기 자신은 조상에 포함되지 않습니다.
    *   **자손 (Descendant)**: 어떤 노드의 자식 노드뿐만 아니라, 그 자식의 자식 노드(손자 노드) 등, 해당 노드로부터 뻗어나가는 모든 하위 노드를 통틀어 자손이라고 합니다. 자기 자신은 자손에 포함되지 않습니다.
    *   예시: 슬라이드에서 "A is the ancestor of G" (A는 G의 조상), "G is the descendant of A" (G는 A의 자손)

*   **Siblings (형제)**
    *   **형제 (Siblings)**: 동일한 부모 노드를 공유하는 모든 자식 노드들을 형제라고 합니다.
    *   예시: 슬라이드에서 "B, C, D are siblings" (B, C, D는 형제)

### **코드/수식 해설**

해당 슬라이드에는 직접적인 코드나 수식은 포함되어 있지 않습니다. 이 개념들은 트리 구조를 이해하는 데 필요한 기본적인 용어 정의입니다. 트리를 코드로 구현할 때 각 노드(Node) 객체는 보통 `parent` 포인터와 `children` 리스트(또는 배열)를 가질 수 있습니다.

### **구체적 예시**

슬라이드에 제시된 트리 그림을 바탕으로 관계를 살펴보겠습니다.

*   **Parent/Child 관계:**
    *   `A`는 `B`, `C`, `D`의 부모이며, `B`, `C`, `D`는 `A`의 자식입니다.
    *   `B`는 `E`, `F`의 부모이며, `E`, `F`는 `B`의 자식입니다.
    *   `D`는 `G`의 부모이며, `G`는 `D`의 자식입니다.
    *   `G`는 `H`, `I`의 부모이며, `H`, `I`는 `G`의 자식입니다.

*   **Ancestor/Descendant 관계:**
    *   `A`의 자손은 `B, C, D, E, F, G, H, I` 입니다.
    *   `G`의 조상은 `D`, `A` 입니다.
    *   `H`의 조상은 `G`, `D`, `A` 입니다.
    *   `E`의 조상은 `B`, `A` 입니다.
    *   `G`의 자손은 `H`, `I` 입니다.

*   **Siblings 관계:**
    *   `B, C, D`는 모두 `A`를 부모로 가지므로 형제입니다.
    *   `E, F`는 모두 `B`를 부모로 가지므로 형제입니다.
    *   `H, I`는 모두 `G`를 부모로 가지므로 형제입니다.

### **시험 포인트**

*   ⭐ **각 용어의 정확한 정의 숙지**: 부모/자식, 조상/자손, 형제 관계의 정의를 명확히 이해하고 구분할 수 있어야 합니다. 특히 '조상'과 '자손'은 직계 관계를 넘어선 개념임을 기억해야 합니다.
*   ⭐ **주어진 트리에서 관계 식별**: 임의의 트리 구조가 주어졌을 때, 특정 노드에 대해 부모, 자식, 조상, 자손, 형제 노드를 정확하게 찾아낼 수 있어야 합니다.
*   ⭐ **계층적 구조의 이해**: 트리는 노드 간의 1대N 계층적 관계를 나타내는 자료구조임을 인지하고 있어야 합니다.

## Slide 7

**핵심 개념**:
*   **경로 (Path)**: 트리에서 $n_1$부터 $n_k$까지 이어지는 노드들의 순서화된 시퀀스 $<n_1, n_2, \dots, n_k>$를 의미합니다. 여기서 각 $n_i$는 $n_{i+1}$의 부모 노드여야 합니다 ($1 \le i < k$).
*   **경로의 길이 (Length of a Path)**: 경로를 구성하는 노드의 총 개수 $k$에서 1을 뺀 값입니다. 이는 경로를 연결하는 엣지(edge)의 총 개수와 동일합니다.
*   **단말 노드 (Terminal Node / Leaf Node / External Node)**: 자식 노드를 전혀 가지지 않는 노드를 지칭합니다. 트리의 가장 끝단에 위치합니다.
*   **비단말 노드 (Non-terminal Node / Internal Node)**: 하나 이상의 자식 노드를 가지는 노드를 지칭합니다.

**코드/수식 해설**:
경로의 길이는 다음과 같은 수식으로 표현됩니다:
$length = k - 1$
여기서 $k$는 경로를 구성하는 노드의 개수입니다.

**구체적 예시**:
첨부된 트리 다이어그램을 기준으로 설명합니다.
*   **경로 예시**: 노드 'A'에서 'H'로 이어지는 경로는 $<A, D, G, H>$ 입니다.
    *   이 경로에는 'A', 'D', 'G', 'H' 총 4개의 노드가 포함되어 있습니다 ($k=4$).
    *   따라서 경로의 길이는 $4 - 1 = 3$ 입니다. (A-D, D-G, G-H 세 개의 엣지로 연결됩니다.)
*   **단말 노드/비단말 노드 예시**:
    *   **단말 노드**: 'E', 'F', 'C', 'H', 'I'는 자식 노드가 없으므로 단말 노드입니다.
    *   **비단말 노드**: 'A', 'B', 'D', 'G'는 자식 노드를 가지고 있으므로 비단말 노드입니다. 예를 들어 'A'는 'B', 'C', 'D'를 자식으로 가지고, 'D'는 'G'를 자식으로 가집니다.

**시험 포인트**:
*   ⭐ **경로의 정의와 길이 계산**: 주어진 트리의 특정 두 노드 간 경로를 정확히 파악하고, 그 길이를 엣지의 개수로 올바르게 계산할 수 있는지 묻는 문제가 자주 출제됩니다. (노드 개수와 엣지 개수 혼동 주의!)
*   ⭐ **노드 유형 분류**: 트리가 주어졌을 때, 어떤 노드가 단말 노드(leaf node)이고 어떤 노드가 비단말 노드(internal node)인지 정확하게 구분하는 능력이 중요합니다. 이 개념은 트리 순회(traversal) 등 다른 트리 관련 문제의 기본이 됩니다.

## Slide 8

## 데이터구조 (CSED233) 슬라이드 노트: Depth, Height, Level

### 핵심 개념

이 슬라이드는 트리 자료구조에서 노드 및 트리의 깊이(Depth), 높이(Height), 레벨(Level)을 정의합니다. 이 용어들은 트리의 구조를 이해하고 분석하는 데 필수적입니다.

*   **노드 M의 깊이 (Depth of node M)**:
    *   루트(Root)에서 노드 M까지의 경로 길이(간선 수)입니다.
    *   다른 정의로는 노드 M의 조상(ancestors)의 개수와 같습니다 (단, 노드 M 자신은 제외).
    *   루트 노드의 깊이는 0입니다.

*   **노드 M의 높이 (Height of node M)**:
    *   노드 M에서부터 시작하여 가장 멀리 떨어진 리프(leaf) 노드까지의 가장 긴 경로의 길이입니다.
    *   리프 노드의 높이는 0입니다.

*   **트리의 높이 (Height of the tree)**:
    *   트리의 루트 노드의 높이와 같습니다. 이는 트리 내에서 가장 긴 경로의 길이입니다.

*   **레벨 (Level)**:
    *   트리의 각 층을 나타내는 개념입니다.
    *   일반적으로 루트 노드는 레벨 0에 위치하며, 그 자식들은 레벨 1, 그들의 자식들은 레벨 2, 이런 식으로 아래로 갈수록 레벨이 증가합니다.
    *   레벨은 깊이와 동일한 개념이며, 단지 시작 번호(0부터 시작할지 1부터 시작할지)에 대한 명확한 규칙이 없어 구현이나 문맥에 따라 달라질 수 있음을 유의해야 합니다. (이 강의에서는 0부터 시작하는 것을 기본으로 합니다.)

### 코드/수식 해설

#### 깊이 (Depth) 수식 및 재귀적 정의

노드 $M$의 깊이는 루트에서 $M$까지의 간선 수를 세는 것과 같습니다.
$Depth(M)$은 $M$의 부모 노드 $P$의 깊이에 1을 더한 값으로 재귀적으로 정의될 수 있습니다.

*   $Depth(Root) = 0$
*   $Depth(M) = Depth(Parent(M)) + 1 \quad (\text{if M is not the Root})$

#### 높이 (Height) 수식 및 재귀적 정의

노드 $M$의 높이는 $M$을 루트로 하는 서브트리에서 가장 긴 경로의 길이입니다.

*   $Height(M) = 0 \quad (\text{if M is a leaf node})$
*   $Height(M) = 1 + \max(\{ Height(C) \mid C \text{ is a child of M} \}) \quad (\text{if M is not a leaf node})$

### 구체적 예시

첨부된 트리 $T$를 예시로 들어 각 개념을 살펴봅시다.

```
       A (Level 0)
      / | \
     B  C  D (Level 1)
    / \    |
   E   F   G (Level 2)
           / \
          H   I (Level 3)
```

*   **깊이 (Depth)**:
    *   `Depth(A)` = 0
    *   `Depth(B)` = 1 (경로: A-B, 조상: A)
    *   `Depth(E)` = 2 (경로: A-B-E, 조상: A, B)
    *   `Depth(G)` = 2 (경로: A-D-G, 조상: A, D)
    *   `Depth(H)` = 3 (경로: A-D-G-H, 조상: A, D, G)

*   **높이 (Height)**:
    *   `Height(E)` = 0 (리프 노드)
    *   `Height(F)` = 0 (리프 노드)
    *   `Height(C)` = 0 (리프 노드)
    *   `Height(H)` = 0 (리프 노드)
    *   `Height(I)` = 0 (리프 노드)
    *   `Height(B)` = 1 (가장 긴 경로: B-E 또는 B-F)
    *   `Height(G)` = 1 (가장 긴 경로: G-H 또는 G-I)
    *   `Height(D)` = 2 (가장 긴 경로: D-G-H 또는 D-G-I)
    *   `Height(A)` = 3 (가장 긴 경로: A-D-G-H 또는 A-D-G-I)
    *   **트리 $T$의 높이** = `Height(A)` = 3

*   **레벨 (Level)**:
    *   `Level 0`: A
    *   `Level 1`: B, C, D
    *   `Level 2`: E, F, G
    *   `Level 3`: H, I

### 시험 포인트

*   ⭐ **깊이와 높이의 정확한 정의 구별**: 깊이는 루트에서 아래로, 높이는 노드에서 리프로의 가장 긴 경로를 의미한다는 점을 명확히 이해해야 합니다.
*   ⭐ **루트 노드와 리프 노드의 깊이/높이**: 루트 노드의 깊이는 0, 리프 노드의 높이는 0이라는 점을 기억해야 합니다.
*   ⭐ **트리의 높이 정의**: 트리의 높이는 루트 노드의 높이와 동일하다는 것을 알아야 합니다.
*   ⭐ **레벨과 깊이의 관계**: 레벨은 일반적으로 깊이와 같은 개념이며, 루트를 레벨 0으로 시작하는 것이 일반적인 관례입니다. 하지만 시작 번호는 문맥에 따라 달라질 수 있다는 점을 인지해야 합니다.
*   주어진 트리에서 특정 노드의 깊이, 높이, 또는 트리의 높이를 계산하는 문제가 출제될 수 있습니다. 재귀적 정의를 이해하면 계산이 용이합니다.

## Slide 9

이 슬라이드는 트리(Tree) 자료구조에서 노드와 트리의 깊이(Depth) 및 높이(Height) 개념을 예시를 통해 설명합니다.

---

### **핵심 개념**

*   **노드의 깊이 (Depth of a node)**
    *   어떤 노드의 깊이는 루트(root) 노드로부터 해당 노드까지의 경로를 구성하는 엣지(edge)의 개수를 의미합니다.
    *   루트 노드의 깊이는 0입니다.
    *   일반적으로 노드가 위치한 레벨(Level)과 동일합니다.
*   **노드의 높이 (Height of a node)**
    *   어떤 노드의 높이는 해당 노드로부터 가장 멀리 떨어진 리프(leaf) 노드까지의 경로를 구성하는 엣지의 최대 개수를 의미합니다.
    *   리프 노드(자식이 없는 노드)의 높이는 0입니다.
*   **트리의 높이 (Height of a tree)**
    *   트리 전체의 높이는 해당 트리의 루트 노드의 높이와 같습니다.
    *   이는 또한 트리 내의 모든 노드 중에서 가장 깊은 노드의 깊이와도 같습니다. 즉, 트리의 가장 깊은 레벨을 의미합니다.

### **코드/수식 해설**

트리의 깊이와 높이는 재귀적인 정의를 가집니다.

*   **노드의 깊이 (Depth of node $v$)**:
    *   루트 노드 $r$의 경우: $depth(r) = 0$
    *   루트가 아닌 노드 $v$의 경우: $depth(v) = depth(parent(v)) + 1$

*   **노드의 높이 (Height of node $v$)**:
    *   리프 노드 $v$의 경우: $height(v) = 0$
    *   리프가 아닌 노드 $v$의 경우: $height(v) = \max_{c \in children(v)} (height(c)) + 1$

*   **트리의 높이 (Height of tree $T$)**:
    *   $height(T) = height(root(T))$
    *   또는, $height(T) = \max_{v \in T} (depth(v))$

### **구체적 예시**

슬라이드에 제시된 트리 $T$의 예시를 통해 각 개념을 살펴봅니다.

*   **트리 구조**:
    *   루트: `A` (Level 0)
    *   `A`의 자식: `B`, `C`, `D` (Level 1)
    *   `B`의 자식: `E`, `F` (Level 2)
    *   `D`의 자식: `G` (Level 2)
    *   `G`의 자식: `H`, `I` (Level 3)

*   **`Depth (node G) = 2`**:
    *   노드 `G`는 루트 `A`로부터 `A -> D -> G` 경로를 통해 도달할 수 있으며, 이 경로는 2개의 엣지(`A-D`, `D-G`)로 구성됩니다. 따라서 깊이는 2입니다. `G`는 Level 2에 위치합니다.

*   **`Height (node G) = 1`**:
    *   노드 `G`의 자식은 `H`와 `I`입니다. `H`와 `I`는 모두 리프 노드이므로 높이가 0입니다.
    *   `G`에서 `H`로 가는 경로는 1개의 엣지, `G`에서 `I`로 가는 경로는 1개의 엣지입니다.
    *   가장 긴 리프까지의 경로는 1개의 엣지이므로 `G`의 높이는 1입니다. ($height(G) = \max(height(H), height(I)) + 1 = \max(0, 0) + 1 = 1$)

*   **`Height (node F) = 0`**:
    *   노드 `F`는 자식이 없는 리프 노드입니다. 정의에 따라 리프 노드의 높이는 0입니다.

*   **`Height (tree T) = 3`**:
    *   트리 $T$의 루트 노드는 `A`입니다.
    *   루트 `A`로부터 가장 멀리 떨어진 리프 노드는 `H` 또는 `I`입니다.
    *   `A -> D -> G -> H` 경로는 3개의 엣지로 구성됩니다. (`A-D`, `D-G`, `G-H`)
    *   `A -> D -> G -> I` 경로는 3개의 엣지로 구성됩니다. (`A-D`, `D-G`, `G-I`)
    *   다른 리프(`E`, `F`, `C`)로의 경로는 최대 2개의 엣지입니다.
    *   따라서 가장 긴 경로는 3개의 엣지이므로 트리의 높이는 3입니다. ($height(A) = 3$)
    *   또한, 트리의 최대 깊이인 $depth(H)=3$ 또는 $depth(I)=3$과도 일치합니다.

### **시험 포인트**

*   ⭐ **노드의 깊이, 노드의 높이, 트리의 높이**의 정의를 정확히 이해하고 구분할 수 있어야 합니다.
*   ⭐ 주어진 트리에서 특정 노드의 깊이와 높이, 그리고 트리 전체의 높이를 정확하게 계산할 수 있어야 합니다.
*   ⭐ **리프 노드의 높이는 항상 0**이라는 점을 기억해야 합니다.
*   ⭐ 트리의 높이는 **루트 노드의 높이**와 같다는 점, 그리고 **모든 노드의 깊이 중 최댓값**이라는 점을 인지해야 합니다.

## Slide 10

**핵심 개념**
*   **노드의 차수 (Degree of node)**: 특정 노드의 차수는 해당 노드가 가진 자식 노드의 개수를 의미합니다.
*   **리프 노드의 차수 (Degree of leaf)**: 자식 노드를 가지지 않는 리프 노드의 차수는 항상 0입니다.
*   **트리의 차수 (Degree of tree)**: 트리 전체의 차수는 해당 트리에 속한 모든 노드의 차수 중 가장 큰 값(최대값)을 의미합니다.

**코드/수식 해설**
노드의 차수는 자식 노드의 수로 정의됩니다. 리프 노드의 차수는 다음과 같이 표현할 수 있습니다.
$\text{Degree (leaf)} = 0$
트리의 차수는 모든 노드 차수들의 집합 $D = \{\text{degree}(v) \mid v \in \text{nodes of } T\}$ 에 대해 다음과 같습니다.
$\text{Degree (tree } T) = \max(D)$

**구체적 예시**
첨부된 `Tree T`를 예시로 각 노드와 트리의 차수를 계산해 봅시다.

1.  **각 노드의 차수**:
    *   노드 A: 자식 (B, C, D) -> $\text{Degree(A)} = 3$
    *   노드 B: 자식 (E, F) -> $\text{Degree(B)} = 2$
    *   노드 C: 자식 없음 -> $\text{Degree(C)} = 0$ (리프 노드)
    *   노드 D: 자식 (G) -> $\text{Degree(D)} = 1$
    *   노드 E: 자식 없음 -> $\text{Degree(E)} = 0$ (리프 노드)
    *   노드 F: 자식 없음 -> $\text{Degree(F)} = 0$ (리프 노드)
    *   노드 G: 자식 (H, I) -> $\text{Degree(G)} = 2$
    *   노드 H: 자식 없음 -> $\text{Degree(H)} = 0$ (리프 노드)
    *   노드 I: 자식 없음 -> $\text{Degree(I)} = 0$ (리프 노드)

2.  **트리 T의 차수**:
    트리 T의 모든 노드 차수들의 집합은 $\{3, 2, 0, 1, 0, 0, 2, 0, 0\}$ 입니다.
    이 값들 중 최대값은 3입니다.
    따라서, $\text{Degree (Tree T)} = \max(\{3, 2, 0, 1, 0, 0, 2, 0, 0\}) = 3$ 입니다.

**시험 포인트**
*   ⭐ **노드의 차수와 트리의 차수 정의를 정확히 이해하고 구별하는 것이 중요합니다.**
*   ⭐ 특히, **리프 노드의 차수는 항상 0**이라는 점을 기억해야 합니다.
*   ⭐ **트리의 차수는 모든 노드의 차수 중 최댓값**이라는 점을 숙지하고, 주어진 트리에서 정확히 계산할 수 있어야 합니다. (예시처럼 트리를 주고 차수를 묻는 문제가 나올 수 있습니다.)

### Ordered Tree

-   **핵심 개념**:
    *   **정렬된 트리(Ordered Tree)**는 각 노드의 자식 노드들(siblings) 사이에 **선형 순서(linear ordering)**가 존재하는 트리입니다.
    *   즉, 같은 부모를 가진 자식 노드들의 순서가 의미를 가지며, 이 순서가 변경되면 서로 다른 트리로 간주됩니다.
    *   반대로 자식 노드들의 순서가 중요하지 않은 트리는 **비정렬된 트리(Unordered Tree)**라고 합니다.

-   **구체적 예시**:
    슬라이드에 제시된 `Tree T`와 `Tree T'`를 통해 정렬된 트리의 개념을 이해할 수 있습니다.
    *   `Tree T`: 루트 `A`의 자식 노드들은 왼쪽부터 `B, C, D` 순서로 정렬되어 있습니다.
    *   `Tree T'`: 루트 `A`의 자식 노드들은 왼쪽부터 `C, B, D` 순서로 정렬되어 있습니다.
    두 트리는 `A`의 자식으로 `B, C, D` 노드를 가진다는 점에서는 같지만, `B`와 `C`의 **상대적인 순서가 다르기 때문에** `Tree T`와 `Tree T'`는 서로 다른 **정렬된 트리**로 간주됩니다. 만약 자식 노드의 순서가 중요하지 않은 비정렬된 트리였다면 `Tree T`와 `Tree T'`는 같은 트리로 보았을 것입니다.
    실생활 비유로는 컴퓨터 파일 시스템에서 디렉토리 안의 파일 목록이 있습니다. 파일 자체는 동일하더라도 "이름 순 정렬" 또는 "생성일 순 정렬" 등 특정 순서에 따라 목록이 다르게 보일 수 있습니다. 정렬된 트리는 이처럼 특정 기준에 따른 순서를 유지하는 트리 구조를 나타낼 때 사용됩니다.

-   **시험 포인트**:
    *   ⭐ **정렬된 트리와 비정렬된 트리의 차이를 명확히 이해하고 구분할 수 있는가?** (핵심 정의)
    *   ⭐ 주어진 트리의 자식 노드 순서가 변경되었을 때, 이것이 정렬된 트리와 비정렬된 트리 각각에서 어떤 의미(같은 트리인가 다른 트리인가)를 가지는지 설명할 수 있어야 합니다.
    *   트리 순회(Traversal) 등 트리 관련 알고리즘에서 자식 노드의 순서가 중요한지에 따라 알고리즘의 동작 방식이나 결과가 달라질 수 있음을 이해해야 합니다.

## Slide 12

## Forest (포레스트)

### 핵심 개념
데이터 구조에서 **포레스트(Forest)**는 **하나 이상의 서로 연결되지 않은(disjoint) 트리의 집합**을 의미합니다. 각각의 트리는 포레스트 내에서 독립적인 구성 요소입니다. 그래프 이론의 관점에서 보면, 트리는 연결된 비순환 그래프(connected acyclic graph)인 반면, 포레스트는 비순환 그래프(acyclic graph)이지만 반드시 연결될 필요는 없습니다. 즉, 여러 개의 연결되지 않은 컴포넌트를 가질 수 있으며, 각 컴포넌트가 하나의 트리를 이룹니다.

### 코드/수식 해설
포레스트를 직접적으로 나타내는 특정 자료구조는 없지만, 각각의 트리를 구성하는 노드 구조와, 이 트리의 루트들을 관리하는 방식으로 포레스트를 표현할 수 있습니다.

**단일 트리 노드의 일반적인 정의 (예: C 언어)**:
```c
// 트리 노드의 구조 정의
struct TreeNode {
    int data;                   // 노드가 저장하는 데이터
    struct TreeNode *firstChild; // 첫 번째 자식 노드를 가리키는 포인터 (left-child)
    struct TreeNode *nextSibling; // 다음 형제 노드를 가리키는 포인터 (right-sibling)
    // 이 외에 부모 노드 포인터, 자식 노드 리스트 등을 포함할 수 있습니다.
};
```
이러한 `TreeNode` 구조체를 사용하여 각각의 트리를 만들 수 있습니다.

**포레스트의 표현**:
포레스트는 단순히 위와 같은 `TreeNode` 구조체로 만들어진 **여러 트리의 루트 노드들에 대한 포인터들을 모아둔 컬렉션**으로 생각할 수 있습니다.
```c
// 포레스트를 나타내는 방법 중 하나: 트리 루트 포인터들의 배열
#define MAX_TREES 100 // 최대 트리의 개수
struct TreeNode *forest_roots[MAX_TREES]; // 각 요소는 독립적인 트리의 루트를 가리킵니다.

// 또는 동적 크기를 가지는 자료구조 (예: C++의 std::vector)
// std::vector<TreeNode*> forest_roots_vector; // 트리의 루트 포인터들을 저장
```
여기서 핵심은 각 `forest_roots` 배열의 요소들이 가리키는 `TreeNode`가 서로 독립적인 하나의 트리의 루트라는 것입니다.

### 구체적 예시
*   **실생활 비유**: 슬라이드의 이미지처럼 실제 숲은 여러 그루의 나무(트리)들이 모여 있는 곳입니다. 각 나무는 독립적으로 서 있지만, 전체적으로 '숲'이라는 하나의 집합을 이룹니다. 각 나무의 뿌리는 독립적이며, 다른 나무와 직접적으로 연결되어 있지 않습니다.
*   **컴퓨터 과학적 예시**:
    *   **파일 시스템**: 컴퓨터의 파일 시스템은 일반적으로 하나의 루트 디렉토리(/ 또는 C:)를 가진 트리 구조입니다. 하지만 여러 개의 독립적인 드라이브 파티션이 있다면, 각 파티션은 자체적인 루트를 가진 트리가 되고, 이 모든 파일 시스템 트리의 집합이 포레스트가 될 수 있습니다.
    *   **Disjoint Set Union (DSU) 자료구조**: DSU는 서로소 집합들을 표현하고 관리하는 데 사용됩니다. 이 자료구조는 내부적으로 각 집합을 하나의 트리로 표현하며, 모든 트리의 집합이 포레스트를 이룹니다. 각 트리의 루트 노드는 해당 집합의 대표 원소가 됩니다.

### 시험 포인트
*   ⭐ **포레스트의 정의**: 포레스트는 하나 이상의 **독립적인 트리의 집합**이다. 이는 트리가 연결된 비순환 그래프인 반면, 포레스트는 **비연결 비순환 그래프**일 수 있음을 이해하는 것이 중요합니다.
*   ⭐ **트리와의 관계**: 트리는 포레스트의 한 종류이며, 가장 간단한 형태의 포레스트는 단일 트리입니다. 즉, 모든 트리는 포레스트이지만, 모든 포레스트가 단일 트리는 아닙니다.
*   ⭐ **응용**:
    *   어떤 트리의 루트 노드를 제거하면 그 서브트리들이 포레스트를 형성합니다.
    *   그래프 알고리즘, 특히 최소 신장 트리를 찾는 Kruskal 알고리즘의 중간 단계나 서로소 집합 자료구조(DSU) 구현에서 포레스트 개념이 자주 사용됩니다.
*   ⭐ **구현 관점**: 포레스트는 일반적으로 단일 루트를 가지지 않으며, 여러 트리의 루트 노드들에 대한 포인터를 배열, 리스트 등으로 관리함으로써 구현됩니다.

## Slide 13

**핵심 개념**
이 슬라이드는 이진 트리(Binary Tree)의 기본적인 정의를 설명합니다. 이진 트리는 다음 두 가지 핵심 속성을 가집니다:
*   **최대 두 개의 자식**: 모든 노드는 최대 두 개의 자식 노드를 가질 수 있습니다. 즉, 0개, 1개 또는 2개의 자식 노드를 가질 수 있습니다.
*   **좌우 자식 지정**: 각 자식 노드는 명확하게 '왼쪽 자식(left child)' 또는 '오른쪽 자식(right child)'으로 지정됩니다. 이는 자식 노드의 순서와 위치가 중요함을 의미합니다.

**코드/수식 해설**
이 슬라이드에는 직접적인 코드나 수식은 없지만, 이진 트리의 노드를 C/C++에서 구현할 때 일반적으로 다음과 같은 구조체를 사용합니다.

```c
struct Node {
    int data;         // 노드가 저장하는 데이터
    struct Node* left;  // 왼쪽 자식 노드를 가리키는 포인터
    struct Node* right; // 오른쪽 자식 노드를 가리키는 포인터
};
```
여기서 `left`와 `right` 포인터는 해당 위치에 자식 노드가 없는 경우 `NULL` 값을 가집니다.

**구체적 예시**
슬라이드의 하단에 제시된 두 가지 예시를 통해 이진 트리의 중요한 특징을 이해할 수 있습니다.
왼쪽 트리:
```
    O (Root)
   / \
  L   R
 /
L
```
이 트리는 루트 노드의 왼쪽 자식(L)이 또 다른 왼쪽 자식(L)을 가집니다.

오른쪽 트리:
```
    O (Root)
   / \
  L   R
     /
    R
```
이 트리는 루트 노드의 오른쪽 자식(R)이 또 다른 오른쪽 자식(R)을 가집니다.

슬라이드는 이 두 트리가 서로 다르다고 명시($\ne$ different binary trees). 이는 비록 노드의 개수나 전체적인 모양은 비슷해 보일지라도, 자식 노드의 **위치(왼쪽 또는 오른쪽)**가 이진 트리를 구성하는 데 있어 매우 중요하며, 자식의 위치가 다르면 다른 트리로 간주된다는 것을 보여줍니다. 즉, 구조적인 동일성에는 노드 간의 연결뿐만 아니라 그 연결이 '왼쪽'인지 '오른쪽'인지에 대한 정보가 포함됩니다.

**시험 포인트**
*   ⭐ **이진 트리의 정의**: 모든 노드가 '최대 두 개의 자식'을 가지며, 각 자식이 '왼쪽' 또는 '오른쪽'으로 명확히 지정된다는 두 가지 핵심 속성을 정확히 이해해야 합니다.
*   ⭐ **좌우 자식의 중요성**: 이진 트리에서 자식 노드의 상대적인 위치(왼쪽 vs. 오른쪽)는 트리의 구조적 동일성을 판단하는 데 결정적인 요소입니다. 단순히 노드가 연결되어 있다는 것뿐만 아니라, 어느 쪽에 연결되었는지가 중요합니다. 슬라이드의 예시처럼, 동일한 노드 구성이라도 자식의 좌우 위치가 다르면 서로 다른 이진 트리로 간주됩니다.
*   ⭐ 트리가 아닌 일반 그래프와의 차이점을 이해하는 것이 중요합니다. 트리는 사이클이 없는 연결된 그래프이며, 이진 트리는 여기에 더해 자식 노드의 개수와 위치에 대한 제약이 추가된 특수한 형태입니다.

## Slide 14

**핵심 개념**

*   **이진 트리 (Binary Tree)**: 각 노드가 최대 두 개의 자식 노드를 가질 수 있는 트리 자료구조입니다. 이때 두 자식 노드는 각각 '왼쪽 자식(left child)'과 '오른쪽 자식(right child)'으로 명확하게 구분됩니다. 이 구분 자체가 자식 노드 간의 순서 개념을 내포합니다.
*   **정렬 트리 (Ordered Tree)**: 노드의 자식들 간에 순서(order)가 중요한 트리입니다. 즉, 어떤 노드의 자식 노드들의 상대적인 위치나 순서를 바꾸면 서로 다른 트리로 간주됩니다. 반대로 '비정렬 트리(Unordered Tree)'에서는 자식 노드들의 순서가 중요하지 않아, 같은 자식 집합을 가지면 동일한 트리로 봅니다.
*   **퀴즈 답변**: 이진 트리는 자식 노드를 왼쪽과 오른쪽으로 엄격하게 구분하므로, 자식 노드의 위치가 트리의 구조를 정의하는 데 결정적인 역할을 합니다. 따라서 **이진 트리는 정렬 트리(Ordered Tree)입니다.**

**코드/수식 해설**

이 질문은 주로 개념적인 이해를 묻는 것이므로 직접적인 코드나 수식 해설보다는, 이진 트리가 '정렬'된 특성을 가지는 이유를 구조체 정의를 통해 엿볼 수 있습니다.

```c
struct TreeNode {
    int data;
    struct TreeNode* left;  // 왼쪽 자식 포인터
    struct TreeNode* right; // 오른쪽 자식 포인터
};
```
위 C 구조체에서 `left`와 `right`는 명확히 다른 메모리 주소를 가리키는 포인터 필드입니다. 이는 특정 노드의 자식들이 '왼쪽'에 있는지 '오른쪽'에 있는지에 따라 서로 다른 위치와 의미를 가짐을 나타냅니다. 예를 들어, `node->left = childA` 와 `node->right = childB` 는 `node->left = childB` 와 `node->right = childA` 와 완전히 다른 트리를 형성합니다.

**구체적 예시**

*   **이진 트리의 예시 (정렬 트리)**:
    어떤 노드 `P`가 왼쪽 자식 `L`과 오른쪽 자식 `R`을 가질 때,
    `Tree A`: `P`의 왼쪽 자식은 `L`, 오른쪽 자식은 `R`
    `Tree B`: `P`의 왼쪽 자식은 `R`, 오른쪽 자식은 `L`
    `Tree A`와 `Tree B`는 서로 다른 이진 트리로 간주됩니다. 이는 자식 노드의 위치(순서)가 중요하기 때문입니다.

*   **일반적인 (비정렬) 트리의 예시**:
    만약 노드 `X`가 자식 `Y`와 `Z`를 갖는 일반적인 트리(이진 트리가 아닌)라면, 자식 노드의 순서가 중요하지 않은 경우(비정렬 트리라면),
    `Tree C`: `X`의 자식은 `{Y, Z}` (순서 상관 없음)
    `Tree C`에서 `Y`와 `Z`의 순서를 바꾸더라도 동일한 트리로 취급될 수 있습니다. 그러나 이진 트리는 `left`와 `right`가 고정된 순서 개념을 제공하므로 이런 모호함이 없습니다.

**시험 포인트**

*   ⭐ **이진 트리는 정렬 트리(Ordered Tree)이다.** (정답은 'Yes' 또는 'True')
*   ⭐ 이진 트리에서 왼쪽 자식과 오른쪽 자식의 위치는 서로 **구별되며 중요**하다.
*   정렬 트리는 자식 노드의 순서가 중요하게 다뤄지는 트리를 의미하며, 이진 트리는 이 정의에 부합한다.
*   정렬 트리와 비정렬 트리의 개념적 차이를 명확히 이해하고 있어야 합니다.

## Proper, Full Binary Trees

### 핵심 개념

*   **Proper Binary Tree (정 이진 트리)**
    *   모든 노드가 자식을 **0개** 또는 **2개**만 가지는 이진 트리입니다. 즉, 자식을 1개만 가지는 노드는 존재하지 않습니다.
*   **Full Binary Tree (완전 이진 트리 / 포화 이진 트리)**
    *   모든 레벨에서 **최대 개수의 노드**를 가지는 이진 트리입니다.
    *   이는 트리의 모든 내부 노드가 두 개의 자식을 가지며, 모든 리프 노드가 동일한 레벨(트리의 최대 높이)에 존재함을 의미합니다.
    *   이러한 의미에서 종종 "완전 이진 트리 (Complete Binary Tree)" 또는 "포화 이진 트리 (Perfect Binary Tree)"와 혼용되거나 유사한 의미로 사용되기도 합니다.
    *   높이(height)가 $h$인 Full Binary Tree는 총 $(2^{h+1} - 1)$개의 노드를 가집니다.
    *   Full Binary Tree는 항상 Proper Binary Tree입니다. (모든 노드가 0개 또는 2개의 자식을 가짐)

### 코드/수식 해설

높이가 $h$인 Full Binary Tree가 가질 수 있는 총 노드의 수 $N$은 다음과 같습니다:
$$N = 2^{h+1} - 1$$
각 레벨 $L$($0 \le L \le h$)에 있는 노드의 수는 $2^L$개이며, 이를 모두 합한 값입니다.
예를 들어, 높이 $h=2$인 Full Binary Tree의 경우:
*   레벨 0: $2^0 = 1$개
*   레벨 1: $2^1 = 2$개
*   레벨 2: $2^2 = 4$개
총 노드 수 $N = 1 + 2 + 4 = 7$개입니다.
위 수식을 적용하면 $N = 2^{2+1} - 1 = 2^3 - 1 = 8 - 1 = 7$개로 동일합니다.

### 구체적 예시

슬라이드에 제시된 세 가지 이진 트리를 통해 Proper와 Full의 개념을 이해해 봅시다. (편의상 노드는 동그라미로 표시)

1.  **가장 왼쪽 트리**:
    *   루트 노드의 오른쪽 자식이 하나의 자식만 가집니다.
    *   **결과**: Proper Binary Tree가 아닙니다. 따라서 Full Binary Tree도 아닙니다.

2.  **가운데 트리 (proper (not full))**:
    *   루트 노드는 두 자식을 가집니다.
    *   루트의 왼쪽 자식은 두 자식을 가집니다.
    *   루트의 오른쪽 자식은 두 자식을 가집니다.
    *   모든 리프 노드는 0개의 자식을 가집니다.
    *   **결과**: 각 노드가 0개 또는 2개의 자식만을 가지므로 **Proper Binary Tree**입니다.
    *   하지만, 이 트리의 높이는 2입니다. 레벨 2에는 3개의 노드만 존재하며, 이는 최대 개수($2^2 = 4$개)보다 적습니다. 따라서 **Full Binary Tree는 아닙니다.**

3.  **가장 오른쪽 트리 (full)**:
    *   루트 노드는 두 자식을 가집니다.
    *   루트의 왼쪽 자식은 두 자식을 가집니다.
    *   루트의 오른쪽 자식은 두 자식을 가집니다.
    *   모든 리프 노드는 0개의 자식을 가집니다.
    *   **결과**: 각 노드가 0개 또는 2개의 자식만을 가지므로 **Proper Binary Tree**입니다.
    *   이 트리의 높이는 $h=2$입니다.
        *   레벨 0: 1개 노드 ($2^0$)
        *   레벨 1: 2개 노드 ($2^1$)
        *   레벨 2: 4개 노드 ($2^2$)
    *   모든 레벨이 최대 개수의 노드로 채워져 있으므로 **Full Binary Tree**입니다.
    *   총 노드 수는 $1+2+4=7$이며, 이는 공식 $2^{2+1}-1 = 7$과 일치합니다.

### 시험 포인트

*   ⭐ **Proper Binary Tree와 Full Binary Tree의 정의를 정확히 구분하고 설명할 수 있어야 합니다.** 특히, "각 노드가 0개 또는 2개의 자식을 가짐" (Proper)과 "각 레벨이 최대 노드 수로 채워짐" (Full)의 차이를 이해하는 것이 중요합니다.
*   ⭐ **Full Binary Tree의 노드 수 계산 공식** ($2^{h+1} - 1$)을 암기하고 적용할 수 있어야 합니다. 주어진 높이에 따라 트리의 총 노드 수를 계산하거나, 노드 수에 따라 높이를 추론하는 문제가 나올 수 있습니다.
*   ⭐ **Full Binary Tree는 항상 Proper Binary Tree이지만, Proper Binary Tree가 항상 Full Binary Tree인 것은 아니라는 관계**를 이해하고 예시를 들어 설명할 수 있어야 합니다. (가운데 트리 예시 참조)
*   **이진 트리의 용어가 교재에 따라 다르게 정의될 수 있다는 점**에 유의하세요. 본 강의에서 사용하는 정의를 기준으로 학습하는 것이 중요합니다.

## Slide 16

**핵심 개념**:
*   **노드 번호 매김 (Node Numbering)**: 이 슬라이드는 **완전 이진 트리 (full binary tree)**의 각 노드에 고유한 번호를 할당하는 표준 방식을 설명합니다. 이 방식은 트리를 배열(array)로 표현할 때 노드의 위치와 관계를 효율적으로 관리하기 위해 사용됩니다.
*   **완전 이진 트리의 정의**: 슬라이드에서 `full binary tree`로 명시되어 있지만, 제시된 이미지는 모든 레벨이 완전히 채워져 있고 마지막 레벨의 노드들이 가능한 한 왼쪽부터 채워진 **완전 이진 트리 (complete binary tree)**의 특성을 따르는 번호 매김을 보여줍니다. (만약 `full binary tree`가 모든 내부 노드가 두 개의 자식을 가지는 트리를 의미한다면, 이 번호 매김은 `perfect binary tree`나 `complete binary tree`에 더 적합합니다.)
*   **번호 할당 규칙**:
    1.  번호는 1부터 시작하여 최대 $2^{h+1} - 1$까지 할당됩니다. 여기서 $h$는 트리의 높이입니다 (루트를 레벨 0으로 했을 때, 가장 깊은 레벨의 인덱스).
    2.  노드 번호는 **레벨 별로 (from top to bottom)**, 즉 상위 레벨에서 하위 레벨 순서로 할당됩니다.
    3.  동일한 레벨 내에서는 **왼쪽에서 오른쪽으로 (from left to right)** 순서대로 번호가 할당됩니다.

**코드/수식 해설**:
*   높이가 $h$인 **완전 이진 트리**가 가질 수 있는 최대 노드 개수는 다음과 같은 수식으로 표현됩니다. (슬라이드 이미지처럼 모든 레벨이 꽉 찬 트리를 가정할 때):
    $$ \text{Total Nodes} = 2^{h+1} - 1 $$
    예를 들어, 슬라이드 이미지의 트리는 높이가 $h=2$입니다. 이 경우 총 노드 수는 $2^{2+1} - 1 = 2^3 - 1 = 8 - 1 = 7$개가 됩니다.

*   이 번호 매김 방식은 트리를 배열로 구현할 때, 노드의 인덱스를 사용하여 부모-자식 관계를 쉽게 계산할 수 있게 합니다. (여기서 인덱스는 1부터 시작한다고 가정합니다.)
    *   노드 `i`의 왼쪽 자식 (left child) 인덱스: $2i$
    *   노드 `i`의 오른쪽 자식 (right child) 인덱스: $2i + 1$
    *   노드 `i`의 부모 (parent) 인덱스: $\lfloor i/2 \rfloor$

**구체적 예시**:
슬라이드에 제시된 높이 $h=2$인 이진 트리의 노드 번호 매김 과정을 따라가 봅시다. 이 트리는 총 7개의 노드를 가집니다.

1.  **레벨 0 (루트 레벨)**:
    *   가장 상위 레벨의 가장 왼쪽에 있는 노드에 **1**번을 할당합니다.
    *   (노드 `1`)
2.  **레벨 1**:
    *   다음 레벨로 내려와 왼쪽에서 오른쪽 순서로 번호를 할당합니다.
    *   왼쪽 노드에 **2**번, 오른쪽 노드에 **3**번을 할당합니다.
    *   (노드 `2`, 노드 `3`)
3.  **레벨 2 (가장 하위 레벨)**:
    *   그 다음 레벨로 내려와 왼쪽에서 오른쪽 순서로 번호를 할당합니다.
    *   노드 `2`의 왼쪽 자식에 **4**번, 노드 `2`의 오른쪽 자식에 **5**번을 할당합니다.
    *   노드 `3`의 왼쪽 자식에 **6**번, 노드 `3`의 오른쪽 자식에 **7**번을 할당합니다.
    *   (노드 `4`, 노드 `5`, 노드 `6`, 노드 `7`)

이 번호 매김을 통해 배열 기반 트리에서 관계를 확인하면:
*   노드 `2`의 왼쪽 자식은 $2 \times 2 = 4$번 노드이고, 오른쪽 자식은 $2 \times 2 + 1 = 5$번 노드입니다.
*   노드 `7`의 부모는 $\lfloor 7/2 \rfloor = 3$번 노드입니다.

**시험 포인트**:
*   ⭐ **완전 이진 트리의 노드 번호 매김 규칙**: "상위 레벨부터 하위 레벨로, 동일 레벨 내에서는 왼쪽에서 오른쪽으로" 순서를 정확히 이해하고 어떤 완전 이진 트리에 대해서도 올바르게 번호를 매길 수 있어야 합니다.
*   ⭐ **높이 $h$인 완전 이진 트리의 총 노드 수 공식**: $2^{h+1} - 1$을 기억하고 주어진 높이에 대한 노드 수를 계산할 수 있어야 합니다.
*   ⭐ **배열 기반 트리 표현에서 노드 인덱스 관계**: 특정 노드 번호가 주어졌을 때, 그 노드의 부모, 왼쪽 자식, 오른쪽 자식의 번호를 계산하는 방법을 알고 있어야 합니다. 이 원리는 **힙 (Heap)** 자료구조와 같이 배열로 구현되는 트리 기반 자료구조에서 매우 중요하게 사용됩니다.
*   이 번호 매김 순서는 트리를 **레벨 순서 순회 (Level-order Traversal)**하는 순서와 동일합니다.

## Slide 17

### 핵심 개념
노드 번호 매김(Node Numbering)은 이진 트리의 각 노드에 정수 인덱스를 부여하는 방법입니다. 이는 특히 완전 이진 트리(Complete Binary Tree)나 힙(Heap)과 같이 배열 기반으로 트리를 구현할 때 매우 유용합니다. 노드 번호를 이용하면 부모, 자식, 형제 노드의 위치를 포인터 없이 간단한 산술 연산으로 계산할 수 있습니다.

*   **루트 노드**: 항상 노드 `1`이 루트입니다. (1-based indexing)
*   **유효성 조건 (`< n`)**: `n`은 트리의 전체 노드 수를 나타내며, 계산된 자식 또는 형제 노드의 인덱스가 `n`보다 작아야만 해당 노드가 실제로 존재한다고 간주합니다.

### 코드/수식 해설
주어진 노드 $k$에 대해 다른 관계 노드의 번호를 계산하는 수식은 다음과 같습니다:

*   **부모 노드 (Parent of node $k$)**:
    $ \text{node } \lfloor k/2 \rfloor \quad (\text{if } k \neq 1) $
*   **왼쪽 자식 노드 (Left\_child of node $k$)**:
    $ \text{node } 2k \quad (\text{if } 2k < n) $
*   **오른쪽 자식 노드 (Right\_child of node $k$)**:
    $ \text{node } 2k + 1 \quad (\text{if } 2k + 1 < n) $
*   **왼쪽 형제 노드 (Left\_sibling of node $k$)**:
    $ \text{node } k - 1 \quad (\text{if } k \text{ is odd}) $
*   **오른쪽 형제 노드 (Right\_sibling of node $k$)**:
    $ \text{node } k + 1 \quad (\text{if } k \text{ is even and } k + 1 < n) $

### 구체적 예시
슬라이드에 제시된 트리(총 $n=7$개의 노드)를 기준으로 각 규칙을 적용해봅시다.

1.  **노드 1 (루트)**:
    *   왼쪽 자식: $2 \times 1 = 2$. ($2 < 7$ 만족) -> 노드 2
    *   오른쪽 자식: $2 \times 1 + 1 = 3$. ($3 < 7$ 만족) -> 노드 3

2.  **노드 2**:
    *   부모: $\lfloor 2/2 \rfloor = 1$. -> 노드 1
    *   왼쪽 자식: $2 \times 2 = 4$. ($4 < 7$ 만족) -> 노드 4
    *   오른쪽 자식: $2 \times 2 + 1 = 5$. ($5 < 7$ 만족) -> 노드 5
    *   오른쪽 형제: 노드 2는 짝수이므로 $2 + 1 = 3$. ($3 < 7$ 만족) -> 노드 3

3.  **노드 3**:
    *   부모: $\lfloor 3/2 \rfloor = 1$. -> 노드 1
    *   왼쪽 자식: $2 \times 3 = 6$. ($6 < 7$ 만족) -> 노드 6
    *   오른쪽 자식: $2 \times 3 + 1 = 7$. ($7 < 7$ 만족) -> 노드 7
    *   왼쪽 형제: 노드 3은 홀수이므로 $3 - 1 = 2$. -> 노드 2

4.  **노드 4**:
    *   부모: $\lfloor 4/2 \rfloor = 2$. -> 노드 2
    *   자식: 왼쪽 자식 $2 \times 4 = 8$. ($8 \not< 7$이므로 자식이 없습니다.)
    *   오른쪽 형제: 노드 4는 짝수이므로 $4 + 1 = 5$. ($5 < 7$ 만족) -> 노드 5

### 시험 포인트
*   ⭐ **1-기반 인덱싱의 이해**: 모든 노드 번호 매김 규칙은 노드 1을 루트로 하는 1-기반 인덱싱을 전제로 합니다. 0-기반 인덱싱과 혼동하지 않도록 주의하세요.
*   ⭐ **핵심 수식의 암기 및 적용**: 주어진 노드 $k$에 대해 부모, 왼쪽/오른쪽 자식, 왼쪽/오른쪽 형제 노드를 계산하는 수식을 정확히 알고 빠르게 적용할 수 있어야 합니다. 특히 `floor` 함수, $k$의 홀짝 여부 조건 등을 주의 깊게 확인해야 합니다.
*   ⭐ **유효성 조건 (`< n`)**: 자식이나 형제 노드를 계산할 때, 해당 인덱스가 트리의 총 노드 수 `n`보다 작은지 반드시 확인해야 합니다. 이 조건은 노드의 존재 여부를 판단하는 데 필수적입니다.
*   ⭐ **배열 기반 이진 트리 구현**: 이 노드 번호 매김 방식은 배열을 사용하여 힙(Heap) 또는 완전 이진 트리(Complete Binary Tree)를 구현할 때 핵심적인 원리입니다. 특정 노드에 해당하는 배열 인덱스를 통해 다른 관계 노드의 배열 인덱스를 직접 계산할 수 있게 됩니다.

## Complete Binary Tree

### 핵심 개념

완전 이진 트리(Complete Binary Tree)는 Full Binary Tree의 정의를 완화한 형태입니다. 높이가 $h$인 이진 트리가 완전 이진 트리이기 위한 조건은 다음과 같습니다.

1.  **마지막 레벨을 제외한 모든 레벨(Level 0부터 Level $h-1$까지)은 완전히 채워져 있어야 합니다.**
    *   즉, 각 레벨 $i$ ($0 \le i < h$)에는 $2^i$개의 노드가 존재해야 합니다.
2.  **마지막 레벨 $h$는 왼쪽부터 오른쪽으로 채워져 있어야 합니다.**
    *   마지막 레벨 $h$의 노드들은 가능한 한 왼쪽에 밀집해 있어야 하며, 중간에 빈 노드가 있어서는 안 됩니다.

⭐ **주의**: Full Binary Tree는 모든 레벨이 완전히 채워진 트리이므로, Full Binary Tree는 항상 Complete Binary Tree입니다. 하지만 그 역은 성립하지 않습니다.

### 코드/수식 해설

완전 이진 트리의 높이가 $h$이고 노드의 개수가 $N$개일 때, 노드의 개수는 다음과 같은 범위에 있습니다.

$$2^h \le N \le 2^{h+1} - 1$$

여기서 $2^{h+1}-1$은 높이 $h$인 Full Binary Tree가 가질 수 있는 최대 노드 수입니다. 완전 이진 트리는 마지막 레벨이 모두 채워지지 않을 수 있으므로, 최소 $2^h$개의 노드를 가지고 (이 경우 마지막 레벨인 $h$에는 $2^h$개의 노드 중 맨 왼쪽 노드 1개만 있고, 그 이전 레벨이 $h-1$이므로 총 $2^h - 1 + 1 = 2^h$개가 됨), 최대 $2^{h+1}-1$개의 노드를 가집니다.

### 구체적 예시

슬라이드의 두 가지 예시를 통해 완전 이진 트리를 이해해 봅시다. 두 트리 모두 높이(Height)는 2입니다.

1.  **왼쪽 트리**: "Full binary tree with 7 nodes"
    *   **Level 0**: 노드 1 (1개)
    *   **Level 1**: 노드 2, 3 (2개)
    *   **Level 2**: 노드 4, 5, 6, 7 (4개)
    *   이 트리는 모든 레벨이 완전히 채워져 있습니다. 따라서 Full Binary Tree이며, 동시에 Complete Binary Tree의 정의(마지막 레벨을 제외한 모든 레벨이 채워져 있고, 마지막 레벨 $h=2$도 왼쪽부터 오른쪽으로 완전히 채워져 있음)도 만족합니다.

2.  **오른쪽 트리**: "Complete binary tree with 6 nodes"
    *   **Level 0**: 노드 1 (1개) - 완전히 채워짐.
    *   **Level 1**: 노드 2, 3 (2개) - 완전히 채워짐.
    *   **Level 2**: 노드 4, 5, 6 (3개)
        *   이 트리의 마지막 레벨(Level 2)은 3개의 노드를 가집니다. 원래 Full Binary Tree라면 4개의 노드(4, 5, 6, 7)가 있어야 하지만, 노드 7이 비어 있습니다.
        *   하지만 노드들이 **왼쪽부터 오른쪽으로 순서대로(4, 5, 6)** 채워져 있으므로, Complete Binary Tree의 조건을 만족합니다.
    *   이 트리는 모든 레벨이 완전히 채워진 것이 아니므로 Full Binary Tree는 아닙니다. 하지만 Complete Binary Tree의 정의에는 부합합니다.

### 시험 포인트

*   ⭐ **완전 이진 트리의 두 가지 정의 조건**을 정확히 이해하고 설명할 수 있어야 합니다. 특히 **"마지막 레벨 $h$는 왼쪽부터 오른쪽으로 채워져야 한다"**는 조건이 중요합니다.
*   ⭐ **Full Binary Tree와 Complete Binary Tree의 관계**를 명확히 구분할 수 있어야 합니다. 모든 Full Binary Tree는 Complete Binary Tree이지만, 모든 Complete Binary Tree가 Full Binary Tree는 아니라는 점을 인지해야 합니다.
*   ⭐ 주어진 이진 트리가 완전 이진 트리인지 판별할 수 있어야 합니다. (예: 어떤 노드가 비어있을 때, 그 위치가 규칙을 위반하는지 아닌지)
*   ⭐ 완전 이진 트리는 **힙(Heap)** 자료구조 구현에 주로 사용되는 트리 형태입니다. 이 연관성도 기억해두면 좋습니다.

## 이진 트리 구현 방식

### 핵심 개념
이 슬라이드는 이진 트리를 컴퓨터 메모리에 표현하는 두 가지 주요 방법에 대해 설명합니다: **배열 기반 구현 (Array implementation)**과 **연결 리스트 기반 구현 (Linked implementation)**입니다. 각 방식은 고유한 특징과 장단점을 가집니다.

*   **배열 기반 구현**: 주로 **완전 이진 트리(Complete Binary Tree)**에 효율적입니다. 트리의 노드들을 메모리 상의 연속적인 공간인 배열에 저장하며, 노드의 부모-자식 관계는 인덱스 계산을 통해 이루어집니다.
*   **연결 리스트 기반 구현**: 각 노드가 데이터 값과 함께 자식 노드를 가리키는 포인터(또는 참조)를 포함하는 방식으로 구현됩니다. 이는 어떤 형태의 이진 트리에도 유연하게 적용할 수 있으며, 특히 동적인 크기 변화나 희소한(sparse) 트리에 적합합니다.

### 코드/수식 해설

#### 배열 기반 구현 (Array implementation)

슬라이드에서는 `char binary_tree[123];`와 같은 배열 선언을 통해 이진 트리를 구현하는 예를 보여줍니다. 이 방식은 완전 이진 트리에 적합합니다.

```c
char binary_tree[123]; // 123개의 char 값을 저장할 수 있는 배열
```

*   **부모-자식 관계 수식 (0-indexed 배열 기준)**:
    *   인덱스 $i$에 있는 노드의 **왼쪽 자식**은 인덱스 $2i+1$에 위치합니다.
    *   인덱스 $i$에 있는 노드의 **오른쪽 자식**은 인덱스 $2i+2$에 위치합니다.
    *   인덱스 $i$에 있는 노드의 **부모**는 인덱스 $(i-1)/2$에 위치합니다 (단, $i>0$일 때).

*   **부모-자식 관계 수식 (1-indexed 배열 기준)**:
    *   인덱스 $i$에 있는 노드의 **왼쪽 자식**은 인덱스 $2i$에 위치합니다.
    *   인덱스 $i$에 있는 노드의 **오른쪽 자식**은 인덱스 $2i+1$에 위치합니다.
    *   인덱스 $i$에 있는 노드의 **부모**는 인덱스 $i/2$에 위치합니다 (정수 나눗셈).

이러한 인덱스 계산을 통해 별도의 포인터 없이도 노드 간의 관계를 파악하고 탐색할 수 있습니다.

#### 연결 리스트 기반 구현 (Linked implementation)

슬라이드에서는 각 노드가 두 개의 포인터 필드를 가지는 `struct elem` 구조체를 제시합니다.

```c
struct elem {
    char val;         // 노드가 저장할 데이터 값
    struct elem * left;  // 왼쪽 자식 노드를 가리키는 포인터
    struct elem * right; // 오른쪽 자식 노드를 가리키는 포인터
};
```

*   `char val`: 해당 노드에 저장되는 실제 데이터 값입니다. 여기서는 `char` 타입이지만, `int`, `float`, 또는 다른 구조체 포인터 등 어떤 타입도 될 수 있습니다.
*   `struct elem * left`: 이 노드의 왼쪽 자식 노드를 가리키는 포인터입니다. 왼쪽 자식이 없으면 `NULL` 값을 가집니다.
*   `struct elem * right`: 이 노드의 오른쪽 자식 노드를 가리키는 포인터입니다. 오른쪽 자식이 없으면 `NULL` 값을 가집니다.

이 구조체는 노드들이 메모리에 연속적으로 위치하지 않아도, 포인터를 통해 논리적인 연결을 형성하여 이진 트리를 구성할 수 있게 합니다.

### 구체적 예시

*   **배열 기반 구현 예시**:
    완전 이진 트리가 다음과 같이 구성되어 있다고 가정합니다 (0-indexed).
    ```
          A (idx 0)
         / \
        B   C (idx 1, 2)
       / \
      D   E (idx 3, 4)
    ```
    이 트리는 `char binary_tree[] = {'A', 'B', 'C', 'D', 'E'};`와 같이 배열로 저장될 수 있습니다.
    *   `binary_tree[0]`는 'A'를 저장하며, 'A'의 왼쪽 자식(B)은 `binary_tree[2*0+1] = binary_tree[1]`, 오른쪽 자식(C)은 `binary_tree[2*0+2] = binary_tree[2]`에 있습니다.
    *   `binary_tree[1]`는 'B'를 저장하며, 'B'의 왼쪽 자식(D)은 `binary_tree[2*1+1] = binary_tree[3]`, 오른쪽 자식(E)은 `binary_tree[2*1+2] = binary_tree[4]`에 있습니다.

*   **연결 리스트 기반 구현 예시**:
    위와 동일한 트리를 연결 리스트로 구현할 경우:
    1.  `elem *root = (elem*)malloc(sizeof(elem));`
    2.  `root->val = 'A';`
    3.  `elem *nodeB = (elem*)malloc(sizeof(elem));`
    4.  `nodeB->val = 'B';`
    5.  `root->left = nodeB;` (A의 왼쪽 자식으로 B 연결)
    6.  `elem *nodeC = (elem*)malloc(sizeof(elem));`
    7.  `nodeC->val = 'C';`
    8.  `root->right = nodeC;` (A의 오른쪽 자식으로 C 연결)
    ... 이런 식으로 노드들을 동적으로 생성하고 포인터를 연결하여 트리를 구성합니다.

### 시험 포인트

*   ⭐ **배열 구현은 완전 이진 트리에 가장 효율적이며, 그 외의 경우 (희소 트리)에는 메모리 낭비가 심하다는 점을 이해해야 합니다.**
*   ⭐ **연결 리스트 구현은 어떤 형태의 이진 트리에도 적용 가능하며, 삽입/삭제에 유연하지만, 각 노드에 포인터 저장으로 인한 추가적인 공간 오버헤드가 발생한다는 점을 알아야 합니다.**
*   ⭐ **0-indexed 및 1-indexed 배열에서 부모-자식 노드 인덱스 간의 관계 수식을 정확히 암기하고 적용할 수 있어야 합니다.** 특히, 힙(Heap) 구현 시 이 개념이 중요하게 다루어집니다.
*   ⭐ **연결 리스트 노드 구조체를 정의하는 방법 (데이터 필드와 자식 포인터 필드)을 이해하고 직접 작성할 수 있어야 합니다.**
*   각 구현 방식의 **시간/공간 복잡도 측면에서의 장단점**을 비교하여 설명할 수 있어야 합니다. (예: 배열은 노드 탐색 시 $O(1)$ 인덱스 계산, 연결 리스트는 $O(h)$ (트리 높이) 시간 소요 등).

## Slide 20

**핵심 개념**

*   **배열을 이용한 이진 트리 구현**: 이 슬라이드는 이진 트리를 배열(array)을 사용하여 표현하는 방법을 설명합니다. 특히, 완전 이진 트리(Full Binary Tree)의 특성을 활용한 번호 매기기(numbering scheme) 방식을 따릅니다.
*   **노드 번호와 배열 인덱스 매핑**: 트리의 각 노드에 고유한 번호 $k$를 부여하고, 이 노드의 데이터를 배열의 $k$번째 인덱스, 즉 $tree[k]$에 저장합니다. 일반적으로 1부터 번호를 매기는 1-based indexing을 사용합니다.
*   **완전 이진 트리 번호 매기기**: 루트를 1번으로 시작하여 각 레벨을 왼쪽에서 오른쪽으로 채워나가는 방식으로 번호를 부여합니다. 마치 트리를 상층부터 하층으로, 같은 층에서는 왼쪽에서 오른쪽으로 읽어나가는 것과 같습니다.
*   **결손 노드 처리**: 실제 트리에 존재하지 않는 노드(예: 이진 트리가 완전 이진 트리가 아닌 경우 비어있는 위치)가 있더라도 해당 위치의 번호는 유지하고, 배열에서는 해당 인덱스를 비워두거나 특별한 값(null, 0 등)으로 표시합니다.

**코드/수식 해설**

배열로 이진 트리를 구현할 때, 특정 노드의 부모, 왼쪽 자식, 오른쪽 자식의 인덱스는 간단한 수식으로 계산할 수 있습니다 (노드 인덱스가 1부터 시작한다고 가정).

*   **부모 노드 인덱스**: 노드 $k$의 부모 노드 인덱스
    $$Parent(k) = \lfloor k / 2 \rfloor$$
    예를 들어, 노드 4의 부모는 $\lfloor 4/2 \rfloor = 2$ (노드 'b')입니다.
*   **왼쪽 자식 노드 인덱스**: 노드 $k$의 왼쪽 자식 노드 인덱스
    $$LeftChild(k) = 2k$$
    예를 들어, 노드 2의 왼쪽 자식은 $2 \times 2 = 4$ (노드 'd')입니다.
*   **오른쪽 자식 노드 인덱스**: 노드 $k$의 오른쪽 자식 노드 인덱스
    $$RightChild(k) = 2k + 1$$
    예를 들어, 노드 2의 오른쪽 자식은 $2 \times 2 + 1 = 5$ (그림에서는 결손 노드)입니다.

**구체적 예시**

슬라이드에 제시된 이진 트리를 예시로 들어보겠습니다.

1.  **루트 노드 'a'**: 번호 1. 배열 `tree[1]`에 'a' 저장.
2.  **'a'의 왼쪽 자식 'b'**: 번호 2. 배열 `tree[2]`에 'b' 저장.
3.  **'a'의 오른쪽 자식 'c'**: 번호 3. 배열 `tree[3]`에 'c' 저장.
4.  **'b'의 왼쪽 자식 'd'**: 번호 4. 배열 `tree[4]`에 'd' 저장.
5.  **'b'의 오른쪽 자식 (결손)**: 번호 5. 이 노드는 존재하지 않으므로 `tree[5]`는 비어있습니다 (또는 null 값).
6.  **'c'의 왼쪽 자식 'e'**: 번호 6. 배열 `tree[6]`에 'e' 저장.

이러한 방식으로 배열 `tree`는 `[null, 'a', 'b', 'c', 'd', null, 'e', ...]` 와 같이 구성됩니다 (인덱스 0을 사용하지 않는 경우). 여기서 `null`은 해당 위치에 노드가 없음을 나타냅니다.

**시험 포인트**

*   ⭐ **배열을 이용한 이진 트리의 부모/자식 인덱스 계산 공식**: 주어진 노드 $k$에 대해 $Parent(k) = \lfloor k/2 \rfloor$, $LeftChild(k) = 2k$, $RightChild(k) = 2k+1$ 공식은 반드시 암기하고 적용할 수 있어야 합니다. (인덱스 0부터 시작하는 경우 $Parent(k) = \lfloor (k-1)/2 \rfloor$, $LeftChild(k) = 2k+1$, $RightChild(k) = 2k+2$로 변경될 수 있으니 주의)
*   ⭐ **공간 복잡도 (Space Complexity)**: 이 방식은 특히 **완전 이진 트리(Complete Binary Tree)**나 **힙(Heap)**과 같이 노드가 밀집되어 있는 경우 매우 효율적입니다. 그러나 슬라이드의 예시처럼 노드가 중간에 비어있는 경우(예: 노드 5), 해당 위치의 배열 공간은 낭비될 수 있습니다. 트리가 희소(sparse)할수록 공간 낭비가 심해집니다.
*   ⭐ **배열 인덱싱 시작점**: 인덱스를 0부터 시작하는지 1부터 시작하는지에 따라 위의 부모/자식 인덱스 계산 공식이 달라지므로, 문제에서 주어진 인덱싱 방식을 정확히 확인해야 합니다. 슬라이드에서는 1부터 시작하는 1-based indexing을 사용하고 있습니다.
*   ⭐ **장점과 단점**:
    *   **장점**: 노드 간의 관계(부모-자식)를 O(1) 시간에 찾아낼 수 있어 매우 빠릅니다. 포인터 오버헤드가 없습니다.
    *   **단점**: 트리의 구조가 변경될 때(노드 추가/삭제) 배열의 재정렬이 필요할 수 있으며, 희소한 트리의 경우 공간 효율성이 떨어집니다. 최대 노드 수만큼 배열을 미리 할당해야 합니다.

**핵심 개념**
이 슬라이드는 이진 트리(Binary Tree)를 연결 리스트(Linked List)와 유사한 방식으로 구현하는 방법을 설명합니다. 각 이진 트리 노드(node)는 다음과 같은 세 가지 주요 필드를 가집니다:
1.  **값(Value) 필드**: 해당 노드가 저장하는 실제 데이터 또는 값을 나타냅니다.
2.  **왼쪽 자식(Left child) 포인터 필드**: 노드의 왼쪽 자식 노드를 가리키는 포인터입니다.
3.  **오른쪽 자식(Right child) 포인터 필드**: 노드의 오른쪽 자식 노드를 가리키는 포인터입니다.
자식이 없는 경우 (예: 리프 노드), 해당 포인터 필드는 `NULL`을 가리킵니다. 이 방식은 트리의 크기를 동적으로 조절할 수 있고, 메모리를 효율적으로 사용할 수 있게 합니다 (노드가 필요할 때만 할당).

**코드/수식 해설**

이진 트리 노드의 C/C++ 언어 기반의 구조체(struct) 정의는 다음과 같습니다.

```c
// C 언어에서 이진 트리 노드 구조체 정의
typedef struct TreeNode {
    int value;               // 노드가 저장할 데이터
    struct TreeNode *left;   // 왼쪽 자식 노드를 가리키는 포인터
    struct TreeNode *right;  // 오른쪽 자식 노드를 가리키는 포인터
} TreeNode;

// C++ 언어에서 이진 트리 노드 클래스 정의 (더 객체 지향적)
class TreeNode {
public:
    int value;               // 노드가 저장할 데이터
    TreeNode *left;          // 왼쪽 자식 노드 포인터
    TreeNode *right;         // 오른쪽 자식 노드 포인터

    // 생성자
    TreeNode(int val) : value(val), left(nullptr), right(nullptr) {}
};
```

이 구조체(또는 클래스)를 사용하여 메모리에 노드를 생성하고, `left`와 `right` 포인터를 통해 다른 노드들을 연결함으로써 이진 트리를 구성합니다.

**구체적 예시**

슬라이드의 그림은 연결 구현 방식을 통해 구성된 이진 트리의 한 예시를 보여줍니다.
*   각 사각형 블록은 하나의 `TreeNode`를 나타냅니다.
*   각 블록은 왼쪽 포인터(주황색), 값 필드(회색), 오른쪽 포인터(분홍색)의 세 부분으로 구성됩니다.
*   **노드 `a`**: 루트 노드입니다. 값 `a`를 가지고, 왼쪽 자식 노드 `b`와 오른쪽 자식 노드 `c`를 가리킵니다.
*   **노드 `b`**: 값 `b`를 가지고, 왼쪽 자식 노드 `d`를 가리킵니다. 오른쪽 자식은 없으므로 오른쪽 포인터는 `NULL`입니다 (슬래시로 표시).
*   **노드 `c`**: 값 `c`를 가지고, 왼쪽 자식 노드 `e`를 가리킵니다. 오른쪽 자식은 없으므로 오른쪽 포인터는 `NULL`입니다.
*   **노드 `d`**: 리프 노드입니다. 값 `d`를 가지며, 왼쪽과 오른쪽 자식 모두 없으므로 두 포인터 필드 모두 `NULL`입니다.
*   **노드 `e`**: 값 `e`를 가지고, 오른쪽 자식 노드 `f`를 가리킵니다. 왼쪽 자식은 없으므로 왼쪽 포인터는 `NULL`입니다.
*   **노드 `f`**: 리프 노드입니다. 값 `f`를 가지며, 왼쪽과 오른쪽 자식 모두 없으므로 두 포인터 필드 모두 `NULL`입니다.

이러한 포인터들을 통해 노드들이 서로 연결되어 계층적인 트리 구조를 형성합니다.

**시험 포인트**

*   ⭐**이진 트리의 노드 구조**: 연결 구현에서 각 노드가 반드시 값 필드와 두 개의 자식 포인터 필드를 가진다는 점을 이해해야 합니다. (`struct TreeNode` 정의를 스스로 작성할 수 있어야 합니다.)
*   ⭐**NULL 포인터의 의미**: 자식이 없는 경우 `NULL` 포인터가 사용됨을 정확히 알아야 합니다 (예: 리프 노드, 완전하지 않은 노드).
*   ⭐**연결 구현의 장단점**:
    *   **장점**: 동적인 크기 조절, 유연한 노드 추가/삭제, 필요한 만큼만 메모리 할당.
    *   **단점**: 각 노드당 포인터 저장으로 인한 추가적인 메모리 오버헤드, 노드 접근 시 포인터 추적(dereferencing)으로 인한 오버헤드.
*   ⭐**배열 구현과의 비교**: 이진 트리의 배열 구현(주로 완전 이진 트리에 유리)과 연결 구현의 차이점과 각각의 상황별 장단점을 비교하여 설명할 수 있어야 합니다.
*   ⭐**트리 순회(Traversal)**: 연결된 이진 트리를 전위(preorder), 중위(inorder), 후위(postorder) 순회하는 알고리즘을 코드로 구현하거나 설명할 수 있어야 합니다. (다음 슬라이드에서 다룰 내용이지만, 노드 구조 이해는 필수적입니다.)

## Slide 22

**핵심 개념**

이 슬라이드는 배열(Array) 구현과 연결 리스트(Linked List) 구현의 장단점을 묻는 퀴즈입니다. 두 자료구조는 데이터를 저장하고 관리하는 기본적인 방식이며, 각각 고유한 특징과 성능 특성을 가집니다.

*   **배열 (Array) 구현**: 데이터를 연속적인 메모리 공간에 저장하는 방식입니다. 인덱스를 통해 원소에 직접 접근할 수 있습니다.
*   **연결 리스트 (Linked List) 구현**: 데이터를 노드(Node) 형태로 저장하며, 각 노드는 데이터와 다음 노드를 가리키는 포인터(또는 레퍼런스)를 포함합니다. 노드들은 메모리상에 연속적으로 저장되지 않아도 됩니다.

**코드/수식 해설**

두 구현 방식의 핵심적인 성능 차이는 특정 연산의 시간 복잡도에서 나타납니다. 여기서 $N$은 자료구조에 저장된 원소의 개수를 의미합니다.

*   **배열 구현 (동적 배열 기준)**
    *   **임의 접근 (Random Access, 인덱스를 통한 접근)**: $O(1)$
        *   메모리가 연속적이고, 원소의 크기가 동일하므로, 시작 주소와 인덱스를 알면 원하는 원소의 주소를 즉시 계산할 수 있습니다.
        *   주소 계산: `array_start_address + index * element_size`
    *   **삽입/삭제 (중간)**: $O(N)$
        *   특정 위치에 원소를 삽입하거나 삭제하면, 그 이후의 모든 원소를 한 칸씩 밀거나 당겨야 합니다.
    *   **삽입/삭제 (끝)**: $O(1)$ (단, 배열에 공간이 있을 때)
    *   **크기 변경 (Resizing)**: $O(N)$ (새로운 더 큰 배열을 할당하고 모든 원소를 복사해야 함)
        *   일반적으로 동적 배열은 용량이 부족해지면 현재 용량의 2배 등으로 확장하여 $O(N)$ 작업이 발생하지만, 이를 통해 대부분의 삽입 연산은 상각(amortized) $O(1)$에 수행될 수 있습니다.
    *   **공간 복잡도**: $O(N)$ (데이터만 저장)

*   **연결 리스트 구현**
    *   **임의 접근 (Random Access, 인덱스를 통한 접근)**: $O(N)$
        *   원하는 원소에 접근하기 위해선 헤드(Head) 노드부터 순차적으로 탐색해야 합니다.
    *   **삽입/삭제 (중간, 해당 노드의 포인터를 알고 있을 때)**: $O(1)$
        *   앞뒤 노드의 포인터만 변경하면 됩니다.
    *   **삽입/삭제 (중간, 특정 인덱스에 삽입/삭제)**: $O(N)$
        *   해당 인덱스까지 탐색하는 데 $O(N)$이 걸립니다.
    *   **공간 복잡도**: $O(N)$ (데이터 외에 포인터 저장 공간이 필요하므로 배열보다 더 많은 공간을 사용)

**구체적 예시**

*   **배열 구현 예시 (장점)**:
    *   음악 플레이어에서 특정 시간(초)에 해당하는 부분을 바로 재생하는 경우: $O(1)$ 시간에 해당 데이터에 접근 가능.
    *   이미지 처리에서 특정 픽셀의 색상 값을 읽거나 수정하는 경우: 2D 배열 인덱싱을 통해 $O(1)$ 접근.
*   **배열 구현 예시 (단점)**:
    *   학급에서 학생 명단(배열)에 전학생이 중간에 들어오거나, 전학 가는 학생이 생겼을 때: 나머지 학생들의 번호(인덱스)를 전부 조정해야 하므로 번거롭고 시간이 오래 걸립니다.
*   **연결 리스트 구현 예시 (장점)**:
    *   인터넷 브라우저의 '뒤로 가기/앞으로 가기' 기능: 방문 기록을 연결 리스트로 관리하면, 현재 페이지 노드에서 이전/다음 페이지 노드로 포인터만 변경하면 되므로 삽입/삭제가 매우 효율적입니다.
    *   대규모 로그 파일에서 특정 시점의 로그를 추가/삭제하는 경우: 파일의 물리적 위치에 상관없이 효율적으로 처리 가능.
*   **연결 리스트 구현 예시 (단점)**:
    *   도서관에서 100번째 책을 찾을 때, 첫 번째 책부터 순서대로 책장을 넘겨가며 찾아야 하는 경우: 원하는 책에 도달하기까지 시간이 오래 걸립니다.

**시험 포인트**

*   ⭐ **배열의 장단점**:
    *   **장점**:
        *   **빠른 임의 접근 ($O(1)$)**: 인덱스를 통한 직접 접근이 가능합니다.
        *   **우수한 캐시 효율**: 데이터가 연속적으로 저장되어 CPU 캐시 활용에 유리합니다.
        *   **적은 공간 오버헤드**: 데이터 자체 외에 추가적인 포인터 저장 공간이 필요 없습니다.
    *   **단점**:
        *   **비효율적인 삽입/삭제 ($O(N)$)**: 중간 원소 삽입/삭제 시 많은 원소 이동이 필요합니다.
        *   **고정된 크기 (정적 배열)** 또는 **잦은 재할당 (동적 배열)**: 동적 배열은 크기 변경 시 $O(N)$ 비용이 발생합니다.

*   ⭐ **연결 리스트의 장단점**:
    *   **장점**:
        *   **효율적인 삽입/삭제 ($O(1)$)**: 노드의 포인터만 변경하면 되므로 빠릅니다 (단, 해당 노드에 대한 포인터를 알고 있을 경우).
        *   **동적인 크기 조절**: 필요에 따라 자유롭게 노드를 추가하거나 삭제할 수 있습니다.
    *   **단점**:
        *   **느린 임의 접근 ($O(N)$)**: 원하는 원소에 접근하기 위해선 반드시 순차 탐색이 필요합니다.
        *   **높은 공간 오버헤드**: 각 노드가 데이터 외에 포인터를 저장해야 하므로 추가 공간이 필요합니다.
        *   **낮은 캐시 효율**: 노드가 메모리상에 분산되어 있어 캐시 미스 발생 확률이 높습니다.

*   ⭐ **어떤 상황에서 어떤 구현을 선택해야 하는지 이해**:
    *   **임의 접근이 잦고 삽입/삭제가 적은 경우**: 배열 (예: 배열 기반 스택/큐, 해시 테이블의 내부 구현)
    *   **삽입/삭제가 잦고 순차 접근이 주로 필요한 경우**: 연결 리스트 (예: 연결 리스트 기반 스택/큐, 그래프의 인접 리스트)

## Slide 23

---
### 핵심 개념
*   **트리 순회(Tree Traversal)**는 트리의 모든 노드를 빠짐없이, 그리고 각각 **정확히 한 번씩(exactly once)** 방문하는 과정입니다.
*   이 방문 과정은 미리 정해진 **체계적인 방식(systematic way)**에 따라 이루어집니다.
*   노드를 방문하는 동안, 해당 노드에 대한 특정 **액션** (예: 노드 값 출력, 데이터 업데이트, 조건 검사, 연산 수행 등)을 수행합니다.
*   궁극적으로 트리 순회는 계층적인 트리 구조를 **선형적인 순서(linearized sequence)**로 펼쳐 보여주는 방법입니다. 이는 트리의 정보를 1차원 리스트 형태로 변환하여 처리할 때 유용합니다.

### 코드/수식 해설
트리 순회는 일반적으로 재귀(recursion) 또는 반복(iteration, 스택이나 큐 사용) 방식을 통해 구현됩니다. 다음은 이진 트리의 깊이 우선 탐색(DFS) 기반 순회에 대한 개념적인 의사 코드입니다. 'action(node)'의 위치에 따라 다양한 순회 방식이 결정됩니다.

```cpp
// 이진 트리의 노드 구조 (예시)
struct Node {
    char data;
    Node* left;
    Node* right;
};

// 재귀를 이용한 깊이 우선 탐색(DFS) 순회 기본 구조
void traverse(Node* node) {
    if (node == nullptr) { // 기저 조건: 노드가 없으면 종료
        return;
    }

    // 1. 현재 노드에서 액션 수행 (Pre-order: 부모 -> 왼쪽 -> 오른쪽)
    // action(node); // 예: cout << node->data << " ";

    traverse(node->left); // 왼쪽 서브트리 순회

    // 2. 현재 노드에서 액션 수행 (In-order: 왼쪽 -> 부모 -> 오른쪽)
    // action(node); // 예: cout << node->data << " ";

    traverse(node->right); // 오른쪽 서브트리 순회

    // 3. 현재 노드에서 액션 수행 (Post-order: 왼쪽 -> 오른쪽 -> 부모)
    // action(node); // 예: cout << node->data << " ";
}
```

위 의사 코드에서 `action(node)`가 호출되는 시점에 따라 프리오더(Pre-order), 인오더(In-order), 포스트오더(Post-order)와 같은 표준 순회 방식이 정의됩니다.

### 구체적 예시
슬라이드에 제시된 이진 트리를 바탕으로 순회 예시를 살펴보겠습니다.
트리 구조:
*   루트: `F`
*   `F`의 자식: `B` (왼쪽), `G` (오른쪽)
*   `B`의 자식: `A` (왼쪽), `D` (오른쪽)
*   `D`의 자식: `C` (왼쪽), `E` (오른쪽)
*   `G`의 자식: `null` (왼쪽), `I` (오른쪽)
*   `I`의 자식: `H` (왼쪽), `null` (오른쪽)

제시된 순회 예시: `F B A D C E G I H` (각 노드를 한 번씩 방문)

이 순서는 **프리오더(Pre-order) 순회**의 결과와 일치합니다. 프리오더 순회는 "현재 노드 방문 -> 왼쪽 자식 순회 -> 오른쪽 자식 순회"의 원칙을 따릅니다.
1.  **F** 방문
2.  `F`의 왼쪽 자식 `B`로 이동
    3.  **B** 방문
    4.  `B`의 왼쪽 자식 `A`로 이동
        5.  **A** 방문 (더 이상 자식 없음, `A`의 부모 `B`로 복귀)
    6.  `B`의 오른쪽 자식 `D`로 이동
        7.  **D** 방문
        8.  `D`의 왼쪽 자식 `C`로 이동
            9.  **C** 방문 (더 이상 자식 없음, `C`의 부모 `D`로 복귀)
        10. `D`의 오른쪽 자식 `E`로 이동
            11. **E** 방문 (더 이상 자식 없음, `E`의 부모 `D`로 복귀, `D`의 모든 자식 순회 완료, `D`의 부모 `B`로 복귀, `B`의 모든 자식 순회 완료, `B`의 부모 `F`로 복귀)
3.  `F`의 오른쪽 자식 `G`로 이동
    4.  **G** 방문
    5.  `G`의 왼쪽 자식 없음
    6.  `G`의 오른쪽 자식 `I`로 이동
        7.  **I** 방문
        8.  `I`의 왼쪽 자식 `H`로 이동
            9.  **H** 방문 (더 이상 자식 없음, `H`의 부모 `I`로 복귀, `I`의 모든 자식 순회 완료, `I`의 부모 `G`로 복귀, `G`의 모든 자식 순회 완료, `G`의 부모 `F`로 복귀, `F`의 모든 자식 순회 완료)

결과적으로 `F B A D C E G I H` 순서로 모든 노드를 한 번씩 방문하게 됩니다.

### 시험 포인트
*   ⭐ **트리 순회의 정확한 정의**: 모든 노드를 **체계적(systematic)**으로 **정확히 한 번씩(exactly once)** 방문하는 과정임을 기억하세요.
*   ⭐ 순회 시 노드에서 수행되는 **액션(actions)**의 중요성: 순회의 목적은 단순히 노드를 지나가는 것이 아니라, 각 노드에서 특정 작업을 수행하기 위함입니다.
*   ⭐ **트리 선형화(linearizing the tree)**의 개념: 계층적인 트리 구조를 선형적인 순서로 변환하는 것이 순회의 주요 목적 중 하나입니다.
*   ⭐ 이 슬라이드에는 명시되지 않았지만, 실제 시험에서는 **프리오더(Pre-order), 인오더(In-order), 포스트오더(Post-order)**와 같은 **표준 트리 순회 방법**의 원리, 각 순회 결과의 특징, 그리고 재귀 및 반복을 이용한 **구현 방법**을 깊이 있게 다룰 가능성이 매우 높습니다. 이 세 가지 순회는 반드시 숙지해야 합니다.

## Slide 25

**핵심 개념**:
트리 순회(Tree Traversal)는 트리 내의 모든 노드를 체계적으로 방문(처리)하는 과정을 말합니다. 노드를 방문하는 시점, 즉 '액션(Action)'을 수행하는 시점에 따라 세 가지 주요 유형으로 나뉩니다. 각 순회 방식은 재귀 함수를 통해 구현되며, 현재 노드에 대한 액션이 자식 서브트리에 대한 재귀 호출 전, 중간, 후 중 어디에 위치하느냐에 따라 구분됩니다.

1.  **전위 순회 (Preorder Traversal)**: 현재 노드(`n`)를 가장 먼저 방문(처리)하고, 그 다음 왼쪽에서 오른쪽으로 모든 자식 서브트리(`T1`, `T2`, ..., `Tk`)를 순회합니다. (NLR: Node -> Left Subtree -> Right Subtree)
2.  **중위 순회 (Inorder Traversal)**: 슬라이드의 정의에 따르면, 첫 번째 자식 서브트리(`T1`)를 먼저 순회한 후 현재 노드(`n`)를 방문(처리)하고, 그 다음 나머지 자식 서브트리(`T2`, ..., `Tk`)를 순회합니다. ⭐ **이진 트리(Binary Tree)**의 경우, 왼쪽 자식 서브트리 -> 현재 노드 -> 오른쪽 자식 서브트리 순서(LNR)로 방문하며, 특히 이진 탐색 트리의 경우 오름차순으로 정렬된 노드 순서를 얻을 수 있어 중요합니다.
3.  **후위 순회 (Postorder Traversal)**: 왼쪽에서 오른쪽으로 모든 자식 서브트리(`T1`, `T2`, ..., `Tk`)를 순회한 **후** 마지막으로 현재 노드(`n`)를 방문(처리)합니다. (LRN: Left Subtree -> Right Subtree -> Node)

**코드/수식 해설**:

슬라이드의 의사 코드(pseudocode)는 재귀적인 트리 순회 함수의 일반적인 구조를 보여줍니다. `traverse(node)` 함수는 현재 `node`를 인자로 받아, 이 노드에 대한 '액션'을 수행하는 시점에 따라 전위, 중위, 후위 순회가 결정됩니다.

```
Func traverse (node){
    // Preorder: 'Action on the node' happens here
    // Inorder: 'Action on the node' happens here (for binary tree, after left child traversal)
    // Postorder: 'Action on the node' happens here

    // if the node have a left child
    //     call traverse for the left child;
    // if the node have a right child
    //     call traverse for the right child;
    // (Note: For general k-ary trees, this implies iterating over all children)
}
```

각 순회 방식에 대한 재귀적 정의는 다음과 같습니다:
트리 $T$의 루트를 $n$이라고 하고, $n$의 자식 서브트리들을 왼쪽부터 차례대로 $T_1, T_2, \dots, T_k$라고 할 때:

1.  **전위 순회 (Preorder Traversal)**:
    현재 노드를 먼저 방문하고, 이어서 모든 자식 서브트리를 차례대로 전위 순회합니다.
    $$ \text{Preorder}(T) = <n, \text{Preorder}(T_1), \dots, \text{Preorder}(T_k)> $$

2.  **중위 순회 (Inorder Traversal)**:
    첫 번째 자식 서브트리를 중위 순회한 후 현재 노드를 방문하고, 이어서 나머지 자식 서브트리를 차례대로 중위 순회합니다. 이 정의는 이진 트리의 경우 ($k=2$, $T_1$이 왼쪽 서브트리, $T_2$가 오른쪽 서브트리) 일반적인 "Left -> Root -> Right" 순서와 일치합니다.
    $$ \text{Inorder}(T) = <\text{Inorder}(T_1), n, \text{Inorder}(T_2), \dots, \text{Inorder}(T_k)> $$

3.  **후위 순회 (Postorder Traversal)**:
    모든 자식 서브트리를 차례대로 후위 순회한 후, 마지막으로 현재 노드를 방문합니다.
    $$ \text{Postorder}(T) = <\text{Postorder}(T_1), \dots, \text{Postorder}(T_k), n> $$

**구체적 예시**:

다음과 같은 이진 트리를 예시로 들어 각 순회 방식을 살펴보겠습니다.

```
      A
     / \
    B   C
   / \   \
  D   E   F
```

1.  **전위 순회 (Preorder Traversal)**: (NLR)
    *   방문 순서: A -> B -> D -> E -> C -> F
    *   설명: 루트 A를 먼저 방문, 이어서 왼쪽 서브트리(B-D-E)를 전위 순회, 그 다음 오른쪽 서브트리(C-F)를 전위 순회합니다.

2.  **중위 순회 (Inorder Traversal)**: ⭐ (이진 트리에 특화된 의미) (LNR)
    *   방문 순서: D -> B -> E -> A -> C -> F
    *   설명: 왼쪽 서브트리(B-D-E)를 중위 순회한 후 루트 A를 방문, 그 다음 오른쪽 서브트리(C-F)를 중위 순회합니다.

3.  **후위 순회 (Postorder Traversal)**: (LRN)
    *   방문 순서: D -> E -> B -> F -> C -> A
    *   설명: 왼쪽 서브트리(B-D-E)를 후위 순회, 이어서 오른쪽 서브트리(C-F)를 후위 순회한 후 마지막으로 루트 A를 방문합니다.

**시험 포인트**:

*   ⭐ **각 순회 방식의 정의 및 노드 방문 순서**: 전위(NLR), 중위(LNR), 후위(LRN)를 정확히 이해하고 있어야 합니다. (N: Node, L: Left Subtree, R: Right Subtree)
*   ⭐ **재귀적 구현 원리**: 각 순회 방식이 재귀 함수 내에서 노드 처리(Action)가 일어나는 위치와 자식 노드 재귀 호출 순서에 따라 어떻게 달라지는지 파악하는 것이 중요합니다. 슬라이드의 의사 코드와 수식 표기를 정확히 해석할 수 있어야 합니다.
*   ⭐ **주어진 트리에 대한 순회 결과 도출**: 임의의 트리(특히 이진 트리)가 주어졌을 때, 각 순회 방식에 따른 노드 방문 순서를 정확히 맞출 수 있어야 합니다.
*   ⭐ **각 순회 방식의 활용 예시**:
    *   **전위 순회**: 트리를 복사하거나, 표현식을 전위 표기법(prefix notation)으로 변환할 때 유용합니다.
    *   **중위 순회**: 이진 탐색 트리(BST)에서 노드를 오름차순으로 정렬된 순서로 얻을 때 사용됩니다.
    *   **후위 순회**: 트리의 노드를 삭제할 때 (자식 노드부터 삭제해야 부모 노드에 대한 참조 문제가 발생하지 않음), 표현식을 후위 표기법(postfix notation)으로 변환하거나 트리의 메모리를 해제할 때 유용합니다.
*   일반적인 $k$-ary 트리에서 중위 순회의 정의(슬라이드에 제시된 $ <\text{Inorder}(T_1), n, \text{Inorder}(T_2), \dots, \text{Inorder}(T_k)> $ 형태)는 이진 트리의 경우와는 약간 다를 수 있으므로, 슬라이드의 수식 정의를 정확히 이해하는 것이 중요합니다. 보통 중위 순회는 이진 트리에 특화되어 많이 다뤄집니다.

트리 순회(Tree Traversal)는 트리에 있는 모든 노드를 한 번씩 방문하는 과정입니다. 슬라이드에서 질문하는 `Preorder(T)`는 여러 트리 순회 방식 중 하나인 **전위 순회(Preorder Traversal)**를 의미합니다.

*   **전위 순회(Preorder Traversal)**:
    1.  현재 노드를 **방문(Visit)**합니다 (데이터를 처리하거나 출력합니다).
    2.  현재 노드의 모든 자식 노드들을 **왼쪽에서 오른쪽 순서**로 재귀적으로 방문합니다. (즉, 왼쪽 서브트리부터 순회하고, 다음 자식 서브트리를 순회하는 식입니다.)

### 코드/수식 해설

일반적인 트리(자식 노드가 여러 개일 수 있는 트리)에 대한 전위 순회 알고리즘의 재귀적 의사 코드는 다음과 같습니다.

```cpp
void Preorder(Node* node) {
    if (node == nullptr) { // 노드가 없으면 종료
        return;
    }

    // 1. 현재 노드를 방문 (출력)
    cout << node->data << " "; 

    // 2. 왼쪽에서 오른쪽 순서로 모든 자식 노드를 재귀적으로 순회
    for (Node* child : node->children) {
        Preorder(child);
    }
}
```

주어진 슬라이드의 트리에 대해 `Preorder(T)`를 적용해 봅시다. 트리의 노드 구성은 다음과 같습니다:

*   1의 자식: 2 (왼쪽), 3 (오른쪽)
*   2의 자식: 4 (왼쪽), 5 (중앙), 6 (오른쪽)
*   3의 자식: 7 (왼쪽), 8 (오른쪽)
*   7의 자식: 9

전위 순회 과정을 따라가 보면:

1.  `Preorder(1)` 호출
    *   **1 방문** (출력: `1`)
    *   `Preorder(2)` 호출
        *   **2 방문** (출력: `1 2`)
        *   `Preorder(4)` 호출
            *   **4 방문** (출력: `1 2 4`)
            *   4는 자식이 없으므로 종료
        *   `Preorder(5)` 호출
            *   **5 방문** (출력: `1 2 4 5`)
            *   5는 자식이 없으므로 종료
        *   `Preorder(6)` 호출
            *   **6 방문** (출력: `1 2 4 5 6`)
            *   6은 자식이 없으므로 종료
        *   2의 모든 자식 순회 완료

            *   **7 방문** (출력: `1 2 4 5 6 3 7`)

### 시험 포인트

*   ⭐ **세 가지 주요 트리 순회 방식(전위, 중위, 후위)의 정의와 동작 방식**을 정확히 이해하고 있어야 합니다. 특히, 이진 트리가 아닌 일반 트리에서도 적용할 수 있도록 일반화된 규칙을 알아두세요.
*   ⭐ **주어진 트리에 대해 각 순회 방식의 결과를 정확하게 도출**할 수 있는 능력이 중요합니다. 재귀 호출 스택을 머릿속으로 그리거나, 손으로 추적하는 연습을 충분히 해야 합니다.
*   ⭐ 각 순회 방식이 어떤 용도로 활용될 수 있는지 (예: 전위 순회는 디렉토리 구조 출력, 식 트리의 접두사 표기법, 중위 순회는 이진 탐색 트리의 정렬된 노드 출력, 후위 순회는 식 트리의 후위 표기법 및 메모리 해제 등) 알아두면 좋습니다.
*   이 슬라이드의 문제는 전위 순회를 묻고 있지만, 시험에서는 중위 순회 (`Inorder(T)`)나 후위 순회 (`Postorder(T)`)도 나올 수 있으니 모두 연습해 보세요. (참고: 중위 순회는 이진 트리에만 의미가 있습니다. 슬라이드의 트리는 이진 트리가 아니므로, "중위"라는 개념을 적용하기 어렵습니다. 만약 이진 트리라면 "왼쪽-루트-오른쪽" 순서입니다.)

### 핵심 개념

**Postorder Traversal (후위 순회)**는 트리를 순회하는 방법 중 하나로, 특정 노드를 방문하기 전에 해당 노드의 모든 자식 노드(서브트리)를 먼저 방문하는 방식입니다. 구체적으로는 다음 순서를 따릅니다:

1.  **왼쪽 자식 서브트리 순회** (가장 왼쪽 자식부터 순차적으로)
2.  **오른쪽 자식 서브트리 순회** (모든 자식 서브트리를 순회)
3.  **현재 노드 방문 (처리)**

이 순회 방식은 주로 이진 트리에 적용되지만, 슬라이드와 같이 자식이 둘 이상인 일반 트리에도 확장하여 적용할 수 있습니다. 일반 트리의 경우, 모든 자식 서브트리를 왼쪽에서 오른쪽으로 순회한 후 현재 노드를 방문합니다.

### 코드/수식 해설

재귀를 이용한 Postorder traversal의 일반적인 의사 코드(Pseudocode)는 다음과 같습니다:

```
Postorder(node):
  if node is not null:
    for each child in node.children (from left to right):
      Postorder(child)
    visit(node) // Process the current node (e.g., print its value)
```

이 의사 코드는 주어진 슬라이드의 일반 트리에 적용됩니다. 이진 트리의 경우 `for each child` 부분이 `Postorder(node.left)`와 `Postorder(node.right)`로 대체됩니다.

### 구체적 예시

슬라이드에 제시된 트리에 대해 Postorder Traversal을 적용해 봅시다.

트리 구조:
-   루트: 1
-   1의 자식: 2, 3
-   2의 자식: 4, 5, 6
-   3의 자식: 7, 8
-   7의 자식: 9

Postorder 순회 과정:

1.  `Postorder(1)` 호출
    *   자식 2 순회: `Postorder(2)` 호출
        *   자식 4 순회: `Postorder(4)` 호출 -> 4는 리프 노드이므로 **4** 출력
        *   자식 5 순회: `Postorder(5)` 호출 -> 5는 리프 노드이므로 **5** 출력
        *   자식 6 순회: `Postorder(6)` 호출 -> 6은 리프 노드이므로 **6** 출력
        *   모든 자식 순회 완료 후, 노드 2 방문 -> **2** 출력
    *   자식 3 순회: `Postorder(3)` 호출
        *   자식 7 순회: `Postorder(7)` 호출
            *   자식 9 순회: `Postorder(9)` 호출 -> 9는 리프 노드이므로 **9** 출력
            *   모든 자식 순회 완료 후, 노드 7 방문 -> **7** 출력
        *   자식 8 순회: `Postorder(8)` 호출 -> 8은 리프 노드이므로 **8** 출력
        *   모든 자식 순회 완료 후, 노드 3 방문 -> **3** 출력
    *   모든 자식 순회 완료 후, 노드 1 방문 -> **1** 출력

따라서, `Postorder(T)`의 결과는 다음과 같습니다:

**4, 5, 6, 2, 9, 7, 8, 3, 1**

### 시험 포인트

*   ⭐ **트리 순회 방법 (Inorder, Preorder, Postorder)의 정의와 순서**를 정확히 이해하고 있어야 합니다. 각 순회 방식이 노드를 방문하는 순서가 어떻게 다른지 비교할 수 있어야 합니다.
*   ⭐ **주어진 트리에 대한 특정 순회 결과를 도출**할 수 있어야 합니다. 이 문제는 Postorder Traversal을 묻고 있으므로, 이 순서대로 모든 노드를 방문하는 연습을 충분히 해야 합니다.
*   ⭐ **재귀적 사고**를 통해 트리 순회 알고리즘을 구현하거나 이해하는 것이 중요합니다. 특히, Base Case (리프 노드 또는 null 노드)와 Recursive Step (자식 노드를 재귀적으로 호출)를 명확히 파악해야 합니다.
*   ⭐ Postorder Traversal은 특정 응용 분야에서 특히 유용합니다. 예를 들어, 식 트리(expression tree)에서 후위 순회는 후위 표기법(postfix notation)을 생성하는 데 사용되며, 파일을 삭제할 때 자식 파일/폴더를 먼저 삭제하고 부모를 삭제하는 방식 등에도 비유될 수 있습니다. (이번 슬라이드에 직접적인 내용은 아니지만, 응용 분야도 함께 알아두면 좋습니다.)

**핵심 개념**:
Inorder 순회(Inorder Traversal)는 이진 트리(Binary Tree)에서 노드를 방문하는 세 가지 주요 방법 중 하나입니다. 이진 트리의 경우, "왼쪽 서브트리 방문 → 루트 노드 방문 → 오른쪽 서브트리 방문" 순서로 진행됩니다. 하지만 주어진 슬라이드의 트리는 각 노드가 두 개 이상의 자식을 가질 수 있는 일반 트리(General Tree)입니다. 일반 트리에서 Inorder 순회를 정의하는 표준 방식은 다양할 수 있으나, 일반적으로는 첫 번째(가장 왼쪽) 자식 서브트리를 재귀적으로 방문한 후 현재 노드를 방문하고, 이어서 나머지 자식 서브트리들을 왼쪽에서 오른쪽 순서로 재귀적으로 방문하는 방식으로 확장하여 적용합니다. (즉, 첫째 자식 - 루트 - 나머지 자식들 순서)

**코드/수식 해설**:
일반 트리에 대한 Inorder 순회의 의사코드 (확장된 정의):

```pseudocode
Inorder(node):
    if node is not null:
        // 1. 첫 번째 자식 서브트리를 Inorder 순회
        if node has first_child:
            Inorder(node.first_child)
        
        // 2. 현재 노드를 방문 (처리)
        Visit(node)
        
        // 3. 두 번째 자식부터 마지막 자식까지의 서브트리들을 Inorder 순회
        if node has more than one child:
            for each child C_i from node.second_child to node.last_child:
                Inorder(C_i)
```

**구체적 예시**:
주어진 트리에 위 의사코드를 적용하여 Inorder 순회 결과를 도출합니다.

1.  `Inorder(1)` 호출 (루트)
    *   1의 첫 번째 자식은 2이므로 `Inorder(2)` 호출.
        *   2의 첫 번째 자식은 4이므로 `Inorder(4)` 호출.
            *   4는 자식이 없으므로, **4** 방문. (출력: 4)
        *   **2** 방문. (출력: 2)
        *   2의 두 번째 자식은 5이므로 `Inorder(5)` 호출.
            *   5는 자식이 없으므로, **5** 방문. (출력: 5)
        *   2의 세 번째 자식은 6이므로 `Inorder(6)` 호출.
            *   6은 자식이 없으므로, **6** 방문. (출력: 6)
    *   **1** 방문. (출력: 1)
    *   1의 두 번째 자식은 3이므로 `Inorder(3)` 호출.
        *   3의 첫 번째 자식은 7이므로 `Inorder(7)` 호출.
            *   7의 첫 번째 자식은 9이므로 `Inorder(9)` 호출.
                *   9는 자식이 없으므로, **9** 방문. (출력: 9)
            *   **7** 방문. (출력: 7)
        *   **3** 방문. (출력: 3)
        *   3의 두 번째 자식은 8이므로 `Inorder(8)` 호출.
            *   8은 자식이 없으므로, **8** 방문. (출력: 8)

따라서, 주어진 트리의 Inorder 순회 결과는 다음과 같습니다:
**4, 2, 5, 6, 1, 9, 7, 3, 8**

**시험 포인트**:
*   ⭐ **트리 순회(Traversal)의 세 가지 기본 방식(Preorder, Inorder, Postorder)의 정의**를 정확히 이해하고 있어야 합니다. 이진 트리에 국한되지 않고, 일반 트리에서의 순회 방식이 주어졌을 때도 이를 적용할 수 있어야 합니다.
*   ⭐ 특히, Inorder 순회는 **이진 탐색 트리(Binary Search Tree)에서 노드들을 오름차순으로 정렬된 순서로 방문**한다는 핵심적인 특성을 반드시 기억해야 합니다. (다만, 이 문제의 트리는 이진 탐색 트리가 아닙니다.)
*   주어진 트리의 구조를 파악하고, 재귀적으로 정의된 순회 알고리즘을 손으로 직접 추적하여 결과를 도출하는 연습이 중요합니다.

**핵심 개념**:
-   **Degenerate Tree (사향 트리)**: 모든 내부 노드가 정확히 하나의 자식만 가지는 트리를 의미합니다. 이러한 트리는 본질적으로 연결 리스트와 같은 선형 구조를 이룹니다. 트리의 높이가 노드의 수에 비례하여 $O(N)$이 됩니다.
-   **트리 순회 (Tree Traversal)**: 트리의 모든 노드를 방문하는 표준적인 방법으로, 주로 세 가지 방식이 있습니다.
    -   **Preorder Traversal (전위 순회)**: **Root** -> Left Subtree -> Right Subtree (현재 노드를 먼저 방문하고, 왼쪽 자식, 오른쪽 자식 순으로 재귀 방문)
    -   **Inorder Traversal (중위 순회)**: Left Subtree -> **Root** -> Right Subtree (왼쪽 자식들을 먼저 재귀 방문하고, 현재 노드를 방문한 뒤, 오른쪽 자식들을 재귀 방문)
    -   **Postorder Traversal (후위 순회)**: Left Subtree -> Right Subtree -> **Root** (왼쪽 자식들, 오른쪽 자식들을 재귀 방문한 후, 현재 노드를 방문)

**코드/수식 해설**:
일반적인 이진 트리의 순회 함수는 다음과 같습니다. 사향 트리의 경우, `left` 또는 `right` 중 하나만 유효한 자식 노드를 가리키고 다른 하나는 `NULL`이 됩니다.

```c
// Node 구조체 (C/C++ 예시)
struct Node {
    int data;
    struct Node *left;
    struct Node *right;
};

// Preorder Traversal
void preorder(struct Node* node) {
    if (node == NULL) return;
    printf("%d ", node->data);   // 1. Visit Root
    preorder(node->left);        // 2. Traverse Left Subtree
    preorder(node->right);       // 3. Traverse Right Subtree
}

// Inorder Traversal
void inorder(struct Node* node) {
    if (node == NULL) return;
    inorder(node->left);         // 1. Traverse Left Subtree
    printf("%d ", node->data);   // 2. Visit Root
    inorder(node->right);        // 3. Traverse Right Subtree
}

// Postorder Traversal
void postorder(struct Node* node) {
    if (node == NULL) return;
    postorder(node->left);       // 1. Traverse Left Subtree
    postorder(node->right);      // 2. Traverse Right Subtree
    printf("%d ", node->data);   // 3. Visit Root
}
```

**구체적 예시**:
노드 값을 A, B, C, D (루트부터 리프까지)라고 가정합니다.

**1. 모든 자식이 왼쪽 자식인 경우 (All children are left children)**
트리 구조는 다음과 같습니다:
```
  A
 /
B
/
C
/
D
```
-   **Preorder**: A, B, C, D (루트에서 리프까지 순서대로 방문)
-   **Inorder**: D, C, B, A (가장 왼쪽 리프부터 루트까지 역순으로 방문)
-   **Postorder**: D, C, B, A (가장 왼쪽 리프부터 루트까지 역순으로 방문)

관계:
-   **Preorder vs. Postorder**: `A, B, C, D` 와 `D, C, B, A` 로 서로 역순입니다.
-   **Inorder vs. Preorder**: `D, C, B, A` 와 `A, B, C, D` 로 서로 역순입니다.
-   **Inorder vs. Postorder**: `D, C, B, A` 와 `D, C, B, A` 로 **동일**합니다.

**2. 모든 자식이 오른쪽 자식인 경우 (All children are right children)**
트리 구조는 다음과 같습니다:
```
A
 \
  B
   \
    C
     \
      D
```
-   **Preorder**: A, B, C, D (루트에서 리프까지 순서대로 방문)
-   **Inorder**: A, B, C, D (루트에서 리프까지 순서대로 방문)
-   **Postorder**: D, C, B, A (가장 오른쪽 리프부터 루트까지 역순으로 방문)

관계:
-   **Preorder vs. Postorder**: `A, B, C, D` 와 `D, C, B, A` 로 서로 역순입니다.
-   **Inorder vs. Preorder**: `A, B, C, D` 와 `A, B, C, D` 로 **동일**합니다.
-   **Inorder vs. Postorder**: `A, B, C, D` 와 `D, C, B, A` 로 서로 역순입니다.

**결론**:
-   **Preorder vs. Postorder**:
    모든 자식이 왼쪽 또는 오른쪽 자식인 사향 트리의 경우, **Preorder 순회 결과는 Postorder 순회 결과와 항상 역순 관계**를 가집니다.
-   **Inorder vs. Preorder**:
    -   모든 자식이 왼쪽 자식인 경우: 서로 역순입니다.
    -   모든 자식이 오른쪽 자식인 경우: **서로 동일**합니다.
-   **Inorder vs. Postorder**:
    -   모든 자식이 왼쪽 자식인 경우: **서로 동일**합니다.
    -   모든 자식이 오른쪽 자식인 경우: 서로 역순입니다.

**시험 포인트**:
-   ⭐ **사향 트리(Degenerate Tree)의 개념과 특징**을 정확히 이해하고 있어야 합니다.
-   ⭐ 세 가지 **트리 순회(Preorder, Inorder, Postorder)의 정의와 동작 방식**을 완벽하게 숙지해야 합니다. 각 순회가 노드를 방문하는 순서(Root-Left-Right 등)를 잊지 마세요.
-   ⭐ 특정 트리의 구조적 제약(예: 모든 자식이 한쪽으로 치우쳐진 경우)이 주어졌을 때, 각 순회 결과가 어떻게 나오는지 **손으로 직접 그려보거나 추론할 수 있는 능력**이 중요합니다. 이를 통해 순회 결과 간의 관계를 도출할 수 있습니다.
-   ⭐ 이 문제는 트리 순회에 대한 기본적인 이해뿐만 아니라, **트리 구조의 변화가 순회 결과에 미치는 영향**을 묻는 응용 문제입니다.

### **핵심 개념**

*   **이진 탐색 트리 (Binary Search Tree, BST)**
    *   **정의**: 이진 트리의 특수한 형태로, 다음 세 가지 속성을 만족합니다:
        1.  모든 노드 $N$의 왼쪽 서브트리에 있는 모든 노드의 키(key)는 $N$의 키보다 작습니다.
        2.  모든 노드 $N$의 오른쪽 서브트리에 있는 모든 노드의 키는 $N$의 키보다 큽니다.
        3.  왼쪽 및 오른쪽 서브트리도 이진 탐색 트리입니다. (재귀적 정의)
    *   **특징**: 노드의 키 값을 기준으로 정렬된 구조를 유지하여 효율적인 탐색, 삽입, 삭제가 가능합니다. 중복된 키는 일반적으로 허용되지 않거나, 특정 규칙(예: 오른쪽 서브트리에 저장)을 따릅니다.
*   **BST의 주요 연산**
    *   **탐색 (Search)**: 주어진 키 값을 가진 노드를 찾는 연산. 루트에서 시작하여 키 값과 현재 노드의 값을 비교하며 왼쪽 또는 오른쪽 서브트리로 이동합니다.
    *   **삽입 (Insertion)**: 새 키 값을 가진 노드를 트리에 추가하는 연산. 탐색과 유사하게 적절한 삽입 위치(말단 노드의 자식)를 찾아 노드를 추가합니다.
    *   **삭제 (Deletion)**: 주어진 키 값을 가진 노드를 트리에서 제거하는 연산. 삭제할 노드의 자식 유무에 따라 처리 방법이 달라집니다.
        1.  **자식이 없는 노드**: 간단히 해당 노드를 제거합니다.
        2.  **자식이 하나인 노드**: 해당 노드를 제거하고 그 자식을 노드의 부모와 연결합니다.
        3.  **자식이 둘인 노드**: 해당 노드를 제거하고, 그 자리에는 보통 중위 순회 후계자(in-order successor, 오른쪽 서브트리에서 가장 작은 값) 또는 중위 순회 선행자(in-order predecessor, 왼쪽 서브트리에서 가장 큰 값)를 찾아 대체합니다.
*   **트리 순회 (Tree Traversal)**
    *   트리에 있는 모든 노드를 체계적인 방법으로 한 번씩 방문하는 과정. 주로 이진 트리에 적용되며, 재귀적으로 구현됩니다.
    *   **전위 순회 (Pre-order Traversal)**: `Root -> Left Subtree -> Right Subtree`
    *   **중위 순회 (In-order Traversal)**: `Left Subtree -> Root -> Right Subtree` (BST에서는 노드를 오름차순으로 방문)
    *   **후위 순회 (Post-order Traversal)**: `Left Subtree -> Right Subtree -> Root`

### **코드/수식 해설**

*   **BST 노드 구조체 (C/C++ 예시)**
    ```c
    typedef struct Node {
        int data;
        struct Node *left;
        struct Node *right;
    } Node;
    ```

*   **BST 탐색 (Search) 의사코드 (재귀적 구현)**
    ```c
    Node* search(Node* root, int key) {
        // 트리가 비었거나, 현재 노드가 찾는 키를 가진 경우
        if (root == NULL || root->data == key) {
            return root;
        }
        // 키가 현재 노드보다 작으면 왼쪽 서브트리에서 탐색
        if (key < root->data) {
            return search(root->left, key);
        } 
        // 키가 현재 노드보다 크면 오른쪽 서브트리에서 탐색
        else {
            return search(root->right, key);
        }
    }
    ```

*   **BST 삽입 (Insertion) 의사코드 (재귀적 구현)**
    ```c
    Node* insert(Node* root, int key) {
        // 삽입할 위치를 찾으면 새 노드를 생성하여 반환
        if (root == NULL) {
            Node* newNode = (Node*)malloc(sizeof(Node));
            newNode->data = key;
            newNode->left = NULL;
            newNode->right = NULL;
            return newNode;
        }
        // 키가 현재 노드보다 작으면 왼쪽 서브트리에 삽입
        if (key < root->data) {
            root->left = insert(root->left, key);
        } 
        // 키가 현재 노드보다 크면 오른쪽 서브트리에 삽입
        else if (key > root->data) { // 중복 키는 일반적으로 무시하거나 특정 규칙에 따라 처리
            root->right = insert(root->right, key);
        }
        return root; // 변경되지 않은 루트 반환 (재귀 호출의 결과 반영)
    }
    ```

*   **BST 연산의 시간 복잡도**
    *   트리의 높이($h$)에 비례합니다. $n$은 노드의 개수입니다.
    *   최악의 경우 (사향 트리, Skewed tree): 트리가 한쪽으로만 기울어진 경우 (예: 연결 리스트와 유사) $O(n)$
    *   평균적인 경우 (균형 잡힌 트리): $O(\log n)$
    *   탐색, 삽입, 삭제 모두 동일한 시간 복잡도를 가집니다.
    *   이를 수학적으로 표현하면:
        *   최악의 경우: $T(n) = O(n)$
        *   평균의 경우: $T(n) = O(\log n)$

### **구체적 예시**

*   **이진 탐색 트리 (BST) 예시**:
    *   전화번호부에서 특정 이름을 찾는 과정과 유사합니다. '홍길동'을 찾을 때, 현재 페이지의 이름이 '박보검'이라면 뒤 페이지(오른쪽)를 보고, '김철수'라면 앞 페이지(왼쪽)를 보는 식으로 범위를 좁혀나가는 것과 같습니다.
    *   도서관에서 특정 책을 찾을 때, 책꽂이의 정렬 기준(예: 가나다순)에 따라 탐색 범위를 계속 절반으로 줄여나가는 것과 비슷합니다.

*   **중위 순회 (In-order Traversal) 예시**:
    *   BST에 숫자 50, 30, 70, 20, 40, 60, 80이 삽입되었다고 가정해 봅시다. 이 트리를 중위 순회하면 `20, 30, 40, 50, 60, 70, 80`과 같이 오름차순으로 정렬된 결과를 얻을 수 있습니다. 이는 BST의 핵심적인 속성을 보여주는 중요한 예시입니다.
    *   산술 표현식 트리에서 중위 순회를 하면 괄호 없는 중위 표현식을 얻을 수 있습니다 (예: `(A + B) * C` 트리를 중위 순회하면 `A + B * C`).

### **시험 포인트**

*   ⭐ **이진 탐색 트리의 정의와 속성**: 특히 왼쪽 서브트리와 오른쪽 서브트리의 키 값 규칙을 정확히 이해해야 합니다.
*   ⭐ **BST의 탐색, 삽입, 삭제 연산 과정**: 각 연산이 어떻게 이루어지는지 그림을 그리거나 단계별로 설명할 수 있어야 합니다. 특히 삭제 연산 시 자식 노드의 개수에 따른 세 가지 경우를 명확히 구분하고 구현할 수 있어야 합니다.
*   ⭐ **각 연산의 시간 복잡도**: 최악의 경우($O(n)$)와 평균적인 경우($O(\log n)$)의 차이를 이해하고, 왜 그런 복잡도가 나오는지 설명할 수 있어야 합니다. (불균형 트리의 문제점)
*   ⭐ **트리 순회 (Pre-order, In-order, Post-order)**: 각 순회 방식의 정의를 암기하고, 주어진 트리에 대해 각 순회 방식의 결과를 정확히 출력할 수 있어야 합니다. 특히, BST에서 중위 순회의 의미(정렬된 결과)를 강조해서 기억하세요.
*   ⭐ **재귀적 순회 구현**: 재귀 함수를 이용한 순회 코드를 직접 작성하거나 이해하는 능력이 중요합니다.

## Recap: Tree Traversal

### 핵심 개념

트리 순회(Tree Traversal)는 트리 내의 모든 노드를 정확히 한 번씩 방문하는 체계적인 과정을 의미합니다. 슬라이드에서는 트리의 루트 노드($n$)와 그 자식 서브트리들($T_1, T_2, \dots, T_k$)을 방문하는 순서에 따라 세 가지 주요 순회 방식(Preorder, Postorder, Inorder)을 설명합니다. 각 방식은 재귀적으로 정의되며, 루트 노드를 언제 방문하느냐에 따라 구분됩니다.

### 코드/수식 해설

트리 $T$의 루트 노드를 $n$이라고 하고, 그 자식 서브트리들을 왼쪽에서 오른쪽으로 $T_1, T_2, \dots, T_k$라고 할 때, 각 순회 방식의 재귀적 정의는 다음과 같습니다:

1.  **전위 순회 (Preorder Traversal)**
    *   **정의**: 루트 노드를 먼저 방문하고, 그 다음 각 자식 서브트리들을 순서대로 전위 순회합니다.
    *   **수식**:
        $$ \text{Preorder}(T) = < n, \text{Preorder}(T_1), \text{Preorder}(T_2), \dots, \text{Preorder}(T_k) > $$
    *   **구현 (의사코드)**:
        ```cpp
        void preorder(Node* node) {
            if (node == nullptr) {
                return;
            }
            visit(node); // 1. 현재 노드(루트) 방문/처리
            for (Node* child : node->children) {
                preorder(child); // 2. 각 자식 서브트리 재귀적으로 전위 순회
            }
        }
        ```

2.  **후위 순회 (Postorder Traversal)**
    *   **정의**: 각 자식 서브트리들을 순서대로 후위 순회하고, 그 다음 마지막으로 루트 노드를 방문합니다.
    *   **수식**:
        $$ \text{Postorder}(T) = < \text{Postorder}(T_1), \text{Postorder}(T_2), \dots, \text{Postorder}(T_k), n > $$
    *   **구현 (의사코드)**:
        ```cpp
        void postorder(Node* node) {
            if (node == nullptr) {
                return;
            }
            for (Node* child : node->children) {
                postorder(child); // 1. 각 자식 서브트리 재귀적으로 후위 순회
            }
            visit(node); // 2. 현재 노드(루트) 방문/처리
        }
        ```

3.  **중위 순회 (Inorder Traversal)**
    *   **정의**: 첫 번째 자식 서브트리를 중위 순회하고, 그 다음 루트 노드를 방문한 후, 나머지 자식 서브트리들을 순서대로 중위 순회합니다.
    *   **수식**:
        $$ \text{Inorder}(T) = < \text{Inorder}(T_1), n, \text{Inorder}(T_2), \text{Inorder}(T_3), \dots, \text{Inorder}(T_k) > $$
    *   **특징**: 이 중위 순회 정의는 슬라이드에 제시된 일반적인 트리(k개의 자식)에 대한 것이며, 이진 트리($k=2$)의 경우 왼쪽 자식 서브트리, 루트, 오른쪽 자식 서브트리 순서로 방문하는 일반적인 중위 순회와 일치합니다 ($T_1$이 왼쪽, $T_2$가 오른쪽).
    *   **구현 (의사코드)**:
        ```cpp
        void inorder_general(Node* node) {
            if (node == nullptr) {
                return;
            }
            if (!node->children.empty()) {
                inorder_general(node->children[0]); // 1. 첫 번째 자식 서브트리 중위 순회
            }
            visit(node); // 2. 현재 노드(루트) 방문/처리
            for (size_t i = 1; i < node->children.size(); ++i) {
                inorder_general(node->children[i]); // 3. 나머지 자식 서브트리들 중위 순회
            }
        }
        ```

### 구체적 예시

다음과 같은 트리를 예시로 들어 각 순회 방식을 살펴보겠습니다.
```
     A
    /|\
   B C D
```

1.  **전위 순회 (Preorder)**:
    -   `preorder(A)` 호출:
        -   **A 방문**
        -   `preorder(B)` 호출:
            -   **B 방문**
            -   (B는 자식이 없으므로 종료)
        -   `preorder(C)` 호출:
            -   **C 방문**
            -   (C는 자식이 없으므로 종료)
        -   `preorder(D)` 호출:
            -   **D 방문**
            -   (D는 자식이 없으므로 종료)
    -   **결과**: A, B, C, D

2.  **후위 순회 (Postorder)**:
    -   `postorder(A)` 호출:
        -   `postorder(B)` 호출:
            -   (B는 자식이 없으므로 스킵)
            -   **B 방문**
        -   `postorder(C)` 호출:
            -   (C는 자식이 없으므로 스킵)
            -   **C 방문**
        -   `postorder(D)` 호출:
            -   (D는 자식이 없으므로 스킵)
            -   **D 방문**
        -   **A 방문**
    -   **결과**: B, C, D, A

3.  **중위 순회 (Inorder - 슬라이드 정의 기준)**:
    -   `inorder_general(A)` 호출:
        -   (A의 첫 번째 자식 B) `inorder_general(B)` 호출:
            -   (B는 자식이 없으므로 첫 번째 자식 순회 스킵)
            -   **B 방문**
            -   (B는 나머지 자식이 없으므로 스킵)
        -   **A 방문**
        -   (A의 나머지 자식 C) `inorder_general(C)` 호출:
            -   (C는 자식이 없으므로 스킵)
            -   **C 방문**
            -   (C는 나머지 자식이 없으므로 스킵)
        -   (A의 나머지 자식 D) `inorder_general(D)` 호출:
            -   (D는 자식이 없으므로 스킵)
            -   **D 방문**
            -   (D는 나머지 자식이 없으므로 스킵)
    -   **결과**: B, A, C, D

### 시험 포인트

*   ⭐ **각 순회 방식의 재귀적 정의를 정확히 이해하고 구현**할 수 있어야 합니다. 특히 루트 노드를 방문하는 시점에 따라 세 가지 순회가 어떻게 달라지는지 명확히 구분해야 합니다.
*   ⭐ **주어진 트리 그림에 대해 각 순회 결과를 도출**하는 문제는 거의 항상 출제됩니다. 이진 트리에 국한되지 않고 일반 트리에 대해서도 순회할 수 있어야 합니다.
*   ⭐ **특정 순회 결과(특히 이진 트리의 전위/중위 또는 중위/후위)로부터 원래의 트리 구조를 재구성**하는 문제도 자주 나옵니다. 이는 순회 개념에 대한 깊은 이해를 요구합니다.
*   ⭐ 각 순회 방식이 **어떤 상황에서 유용하게 사용되는지**도 알아두면 좋습니다 (예: 전위 순회는 트리를 복사하거나 표현식을 폴란드 표기법으로 변환할 때, 후위 순회는 트리의 모든 노드를 메모리에서 해제할 때, 이진 탐색 트리의 중위 순회는 정렬된 순서로 노드를 얻을 때).

### **핵심 개념**
*   **이진 트리 순회(Binary Tree Traversal)**: 이진 트리에 있는 모든 노드를 체계적으로 방문하는 과정을 말합니다.
*   **깊이 우선 탐색(Depth-first search, DFS)**: 트리의 한 경로를 따라 가능한 한 깊이 내려간 후, 더 이상 탐색할 수 있는 노드가 없으면 되돌아와(백트래킹) 다른 경로를 탐색하는 방식입니다. "Go down first"라는 설명처럼 깊이를 우선합니다.
*   **재귀적 구현**: DFS는 일반적으로 재귀 함수를 사용하여 구현하며, 이때 시스템의 **호출 스택(call STACK)**이 자동으로 깊이 탐색 경로를 관리합니다.
*   **반복적 구현**: 명시적인 스택(stack) 자료구조를 사용하여 재귀 없이도 DFS를 구현할 수 있습니다.
*   **세 가지 주요 순회 방식**: DFS에는 노드를 방문하는 시점에 따라 세 가지 변형이 있습니다.
    *   **전위 순회 (pre-order traversal)**: 현재 노드를 먼저 방문한 후, 왼쪽 서브트리, 그 다음 오른쪽 서브트리를 방문합니다. (Root-Left-Right)
    *   **중위 순회 (in-order traversal)**: 왼쪽 서브트리를 방문한 후, 현재 노드를 방문하고, 그 다음 오른쪽 서브트리를 방문합니다. (Left-Root-Right) (이진 탐색 트리에서 노드를 오름차순으로 정렬된 순서로 얻을 때 유용)
    *   **후위 순회 (post-order traversal)**: 왼쪽 서브트리, 오른쪽 서브트리를 방문한 후, 현재 노드를 방문합니다. (Left-Right-Root) (트리 삭제 또는 후위 표현식 평가에 유용)

### **코드/수식 해설**

**반복적 깊이 우선 탐색(Iterative DFS) 의사 코드**:
슬라이드에 제시된 의사 코드는 스택을 사용하여 깊이 우선 탐색을 수행하는 일반적인 방식을 보여줍니다.

```
// 모든 노드를 방문하는 동안 (스택이 빌 때까지)
While visiting all nodes {
    // 스택에서 현재 처리할 노드를 가져옴
    current_node = stack.pop()

    // 현재 노드 처리 (예: 출력, 값 저장 등. 전위 순회라면 이 시점에 방문)

    // 현재 노드의 모든 자식 노드를 스택에 푸시하여 다음 탐색 준비
    // 깊이 우선 탐색의 순서를 결정하는 핵심 부분 (자식을 푸시하는 순서에 따라 달라짐)
    For each child of the current node
        stack.push( child )
}
```
**설명**:
이 의사 코드는 스택의 LIFO(Last-In, First-Out) 특성을 활용하여 깊이 우선 탐색을 구현합니다. `current_node = stack.pop()`을 통해 가장 최근에 스택에 들어갔던(즉, 가장 깊이 있거나 가장 최근에 발견된) 노드를 꺼내 처리합니다. `stack.push(child)`를 통해 현재 노드의 자식들을 스택에 넣어 다음 탐색 대상을 예약합니다. 특정 순회 방식(전위, 중위, 후위)을 구현하려면 `current_node`를 "방문"하는 시점과 자식 노드를 스택에 `push`하는 순서를 조절해야 합니다. 예를 들어 전위 순회(Root-Left-Right)의 반복적 구현에서는 일반적으로 오른쪽 자식을 먼저 푸시하고 왼쪽 자식을 나중에 푸시하여 왼쪽 자식이 먼저 팝되도록 합니다.

**시간 복잡도**:
$$O(n)$$
*   $n$은 트리에 있는 노드의 총 개수를 의미합니다. 깊이 우선 탐색은 트리의 모든 노드를 정확히 한 번씩 방문하므로, 노드의 개수에 비례하는 시간이 소요됩니다. 각 노드에서 스택 푸시/팝 및 자식 노드 확인 등의 작업은 상수 시간에 이루어집니다.

### **구체적 예시**

슬라이드의 이진 트리를 가지고 **전위 순회(Pre-order Traversal)**를 스택 기반으로 수행하는 과정을 살펴봅시다. (루트 -> 왼쪽 -> 오른쪽)

**트리 구조**:
```
      F
     / \
    B   G
   / \   \
  A   D   I
     / \ /
    C   E H
```

**전위 순회 (방문 순서: F -> B -> A -> D -> C -> E -> G -> I -> H)**

1.  **초기화**: 스택 = `[F]` (루트 노드 F를 푸시)
2.  **F 처리**:
    *   스택에서 F를 `pop`. F를 방문 (출력: `F`).
    *   F의 자식: G (오른쪽), B (왼쪽). 스택에 오른쪽부터 푸시 (G 먼저, B 나중) -> 스택 = `[G, B]`
3.  **B 처리**:
    *   스택에서 B를 `pop`. B를 방문 (출력: `B`).
    *   B의 자식: D (오른쪽), A (왼쪽). 스택에 오른쪽부터 푸시 (D 먼저, A 나중) -> 스택 = `[G, D, A]`
4.  **A 처리**:
    *   스택에서 A를 `pop`. A를 방문 (출력: `A`).
    *   A는 자식이 없음. -> 스택 = `[G, D]`
5.  **D 처리**:
    *   스택에서 D를 `pop`. D를 방문 (출력: `D`).
    *   D의 자식: E (오른쪽), C (왼쪽). 스택에 오른쪽부터 푸시 (E 먼저, C 나중) -> 스택 = `[G, E, C]`
6.  **C 처리**:
    *   스택에서 C를 `pop`. C를 방문 (출력: `C`).
    *   C는 자식이 없음. -> 스택 = `[G, E]`
7.  **E 처리**:
    *   스택에서 E를 `pop`. E를 방문 (출력: `E`).
    *   E는 자식이 없음. -> 스택 = `[G]`
8.  **G 처리**:
    *   스택에서 G를 `pop`. G를 방문 (출력: `G`).
    *   G의 자식: I (오른쪽). 스택에 I 푸시 (G는 왼쪽 자식이 없음) -> 스택 = `[I]`
9.  **I 처리**:
    *   스택에서 I를 `pop`. I를 방문 (출력: `I`).
    *   I의 자식: H (오른쪽). 스택에 H 푸시 (I는 왼쪽 자식이 없음) -> 스택 = `[H]`
10. **H 처리**:
    *   스택에서 H를 `pop`. H를 방문 (출력: `H`).
    *   H는 자식이 없음. -> 스택 = `[]`
11. **종료**: 스택이 비었으므로 순회 종료.

최종 방문 순서는 `F -> B -> A -> D -> C -> E -> G -> I -> H` 입니다.

### **시험 포인트**
*   ⭐ **DFS와 BFS의 근본적인 차이점**: DFS는 스택(재귀 호출 스택 포함)을 사용하고, BFS는 큐를 사용한다는 점을 명확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **재귀적 DFS의 동작 원리**: 재귀 호출이 내부적으로 시스템 콜 스택을 어떻게 활용하여 깊이 탐색을 관리하는지 설명할 수 있어야 합니다.
*   ⭐ **전위, 중위, 후위 순회의 정의와 각각의 방문 순서**: 세 가지 순회 방식의 노드 방문 순서(Root-Left-Right 등)를 정확히 암기하고, 주어진 트리에 대해 각 순회 방식의 결과를 도출할 수 있어야 합니다.
*   ⭐ **반복적 DFS 구현 시 스택의 활용**: 반복적 DFS(특히 전위 순회) 구현 시 자식 노드를 스택에 푸시하는 순서가 방문 순서에 어떤 영향을 미치는지 이해해야 합니다 (일반적으로 오른쪽 자식을 먼저 푸시하여 왼쪽 자식이 먼저 처리되도록 함).
*   ⭐ **모든 트리 순회 방식의 시간 복잡도**: 트리의 모든 노드를 한 번씩 방문하므로 시간 복잡도는 항상 $O(n)$임을 기억해야 합니다.

## Binary Tree Traversal: Breadth-First Search (Level-Order)

### 핵심 개념
**Breadth-First Search (BFS)**는 트리의 모든 노드를 탐색하는 한 가지 방법으로, 현재 레벨의 모든 노드를 먼저 방문한 후 다음 레벨로 넘어가는 방식으로 진행됩니다. 이를 트리에서 적용할 때 **Level-order traversal**이라고 부르며, 트리의 각 레벨(depth)별로 왼쪽에서 오른쪽 순서로 노드를 방문합니다. 이 탐색은 큐(Queue) 자료구조를 활용하여 구현됩니다. 큐의 First-In, First-Out (FIFO) 특성이 레벨별 탐색에 적합합니다.

### 코드/수식 해설
BFS(Level-order traversal)의 기본적인 알고리즘은 다음과 같습니다:

```
While visiting all nodes {
    current node = queue.dequeue()
    For the current node's child
        queue.enqueue( child )
}
```
1.  **초기화**: 탐색을 시작할 루트 노드를 큐에 추가합니다.
2.  **반복**: 큐가 빌 때까지 다음 과정을 반복합니다.
    *   큐에서 하나의 노드를 꺼내(dequeue) 방문합니다. 이 노드가 `current node`가 됩니다.
    *   `current node`의 모든 자식 노드를 큐에 추가(enqueue)합니다.
이 과정은 현재 레벨의 모든 노드가 큐에서 처리되고, 그 다음 레벨의 노드들이 큐에 추가되는 방식으로 진행되어 레벨별 탐색을 보장합니다.

**시간 복잡도 (Time Complexity)**:
$$O(n) \text{, where } n \text{ is \# of nodes in a tree}$$
트리 내의 모든 노드($n$)는 큐에 한 번 삽입(enqueue)되고 한 번 삭제(dequeue)됩니다. 각 노드에서 자식 노드를 탐색하여 큐에 추가하는 과정은 트리의 간선(edge) 수에 비례합니다 (트리에서 간선 수는 $n-1$입니다). 따라서 전체 연산은 노드 수 $n$에 비례하므로 시간 복잡도는 $O(n)$입니다.

### 구체적 예시
첨부된 슬라이드의 트리 다이어그램을 사용하여 Level-order traversal을 진행해 보겠습니다.
트리 구조:
-   루트: F
-   F의 자식: B, G
-   B의 자식: A, D
-   G의 자식: I
-   A의 자식: C
-   D의 자식: E
-   I의 자식: H

탐색 과정:
1.  **시작**: 큐 = `[F]`
2.  **F 처리**: F를 큐에서 꺼냄. F 방문. F의 자식 B, G를 큐에 넣음.
    *   방문 순서: `F`
    *   큐: `[B, G]`
3.  **B 처리**: B를 큐에서 꺼냄. B 방문. B의 자식 A, D를 큐에 넣음.
    *   방문 순서: `F, B`
    *   큐: `[G, A, D]`
4.  **G 처리**: G를 큐에서 꺼냄. G 방문. G의 자식 I를 큐에 넣음.
    *   방문 순서: `F, B, G`
    *   큐: `[A, D, I]`
5.  **A 처리**: A를 큐에서 꺼냄. A 방문. A의 자식 C를 큐에 넣음.
    *   방문 순서: `F, B, G, A`
    *   큐: `[D, I, C]`
6.  **D 처리**: D를 큐에서 꺼냄. D 방문. D의 자식 E를 큐에 넣음.
    *   방문 순서: `F, B, G, A, D`
    *   큐: `[I, C, E]`
7.  **I 처리**: I를 큐에서 꺼냄. I 방문. I의 자식 H를 큐에 넣음.
    *   방문 순서: `F, B, G, A, D, I`
    *   큐: `[C, E, H]`
8.  **C 처리**: C를 큐에서 꺼냄. C 방문. C는 자식이 없음.
    *   방문 순서: `F, B, G, A, D, I, C`
    *   큐: `[E, H]`
9.  **E 처리**: E를 큐에서 꺼냄. E 방문. E는 자식이 없음.
    *   방문 순서: `F, B, G, A, D, I, C, E`
    *   큐: `[H]`
10. **H 처리**: H를 큐에서 꺼냄. H 방문. H는 자식이 없음.
    *   방문 순서: `F, B, G, A, D, I, C, E, H`
    *   큐: `[]`
11. **종료**: 큐가 비었으므로 탐색 종료.

최종 방문 순서는 **F, B, G, A, D, I, C, E, H** 입니다.

### 시험 포인트
*   ⭐ **BFS와 Level-order traversal의 관계**: BFS는 그래프 탐색 알고리즘의 일종이며, 트리에서는 Level-order traversal과 동일한 방식으로 동작합니다.
*   ⭐ **큐(Queue) 자료구조의 활용**: BFS/Level-order traversal은 반드시 큐를 사용하여 구현해야 합니다. 큐의 FIFO 특성 덕분에 레벨별 탐색이 가능합니다. (DFS는 스택 또는 재귀를 사용합니다.)
*   ⭐ **탐색 순서**: 노드를 레벨별로, 같은 레벨 내에서는 일반적으로 왼쪽에서 오른쪽으로 방문합니다.
*   ⭐ **시간 복잡도**: 모든 노드를 한 번씩 방문하고 모든 간선을 한 번씩 확인하므로 $O(n)$의 시간 복잡도를 가집니다. 여기서 $n$은 트리의 노드 수입니다.
*   ⭐ **공간 복잡도**: 최악의 경우 (완전 이진 트리) 큐에 한 레벨의 모든 노드가 저장될 수 있으므로 $O(W)$ 또는 $O(n)$의 공간 복잡도를 가집니다. 여기서 $W$는 트리의 최대 너비(width)입니다.

## Slide 5

*   **핵심 개념**
    *   **이진 트리 (Binary Tree)**: 각 노드가 최대 두 개의 자식 노드(왼쪽 자식, 오른쪽 자식)를 가질 수 있는 계층적 자료구조입니다.
    *   **트리 순회 (Tree Traversal)**: 트리의 모든 노드를 체계적인 방식으로 정확히 한 번씩 방문하는 과정입니다. 트리의 내용을 검색하거나 처리할 때 사용됩니다.
    *   **세 가지 주요 순회 방식**:
        *   **전위 순회 (Preorder Traversal)**: 루트 노드를 먼저 방문한 후 왼쪽 서브트리를 전위 순회하고, 이어서 오른쪽 서브트리를 전위 순회합니다. (Root -> Left -> Right)
        *   **중위 순회 (Inorder Traversal)**: 왼쪽 서브트리를 중위 순회한 후 루트 노드를 방문하고, 이어서 오른쪽 서브트리를 중위 순회합니다. (Left -> Root -> Right) 이진 탐색 트리(Binary Search Tree)의 경우, 노드를 오름차순으로 방문합니다.
        *   **후위 순회 (Postorder Traversal)**: 왼쪽 서브트리를 후위 순회한 후 오른쪽 서브트리를 후위 순회하고, 마지막으로 루트 노드를 방문합니다. (Left -> Right -> Root) 트리를 삭제할 때 자식 노드를 먼저 삭제하고 부모 노드를 나중에 삭제하는 데 유용합니다.
    *   **트리 재구성 (Tree Reconstruction)**: 이진 트리의 정확한 구조를 모르는 상태에서 특정 순회 결과들을 바탕으로 원래의 트리 구조를 찾아내는 문제입니다.

*   **코드/수식 해설**
    트리 순회는 주로 재귀적으로 구현됩니다. 각 순회 방식에 대한 의사 코드는 다음과 같습니다.

    ```c++
    struct TreeNode {
        int val;
        TreeNode* left;
        TreeNode* right;
        TreeNode(int x) : val(x), left(nullptr), right(nullptr) {}
    };

    // 전위 순회 (Preorder Traversal)
    void preorderTraversal(TreeNode* root) {
        if (root == nullptr) {
            return;
        }
        printf("%d ", root->val);       // 1. 현재 노드 방문
        preorderTraversal(root->left);  // 2. 왼쪽 서브트리 순회
        preorderTraversal(root->right); // 3. 오른쪽 서브트리 순회
    }

    // 중위 순회 (Inorder Traversal)
    void inorderTraversal(TreeNode* root) {
        if (root == nullptr) {
            return;
        }
        inorderTraversal(root->left);  // 1. 왼쪽 서브트리 순회
        printf("%d ", root->val);      // 2. 현재 노드 방문
        inorderTraversal(root->right); // 3. 오른쪽 서브트리 순회
    }

    // 후위 순회 (Postorder Traversal)
    void postorderTraversal(TreeNode* root) {
        if (root == nullptr) {
            return;
        }
        postorderTraversal(root->left);  // 1. 왼쪽 서브트리 순회
        postorderTraversal(root->right); // 2. 오른쪽 서브트리 순회
        printf("%d ", root->val);       // 3. 현재 노드 방문
    }
    ```

    **시간 및 공간 복잡도**:
    모든 순회 방식은 트리의 모든 노드를 정확히 한 번씩 방문하므로, 노드 수가 $N$인 트리에 대해 시간 복잡도는 $O(N)$입니다.
    $$ T(N) = O(N) $$
    재귀 호출 시 스택 메모리를 사용하며, 이는 트리의 높이 $H$에 비례합니다. 따라서 공간 복잡도는 $O(H)$입니다. 최악의 경우(편향 트리) $O(N)$, 최적의 경우(균형 트리) $O(\log N)$입니다.
    $$ S(N) = O(H) $$

*   **구체적 예시**
    슬라이드에 제시된 이진 트리에 대해 세 가지 순회를 수행해 봅시다.
    트리 구조:
            50
           /  \
          40   60
         / \   / \
        30 45 55 70
       /  \
      20  35

    *   **전위 순회 (Preorder Traderal)**: `50 40 30 20 35 45 60 55 70`
    *   **중위 순회 (Inorder Traversal)**: `20 30 35 40 45 50 55 60 70`
    *   **후위 순회 (Postorder Traversal)**: `20 35 30 45 40 55 70 60 50`

    **트리 재구성 예시 (Preorder + Inorder 이용)**:
    주어진 Preorder 순회 (`50 40 30 20 35 45 60 55 70`)와 Inorder 순회 (`20 30 35 40 45 50 55 60 70`)를 이용하여 트리를 재구성하는 과정은 다음과 같습니다.
    1.  **Preorder의 첫 번째 요소는 항상 현재 서브트리의 루트입니다.** 따라서 전체 트리의 루트는 `50`입니다.
    2.  **Inorder 순회에서 루트(`50`)를 찾습니다.** `50`의 왼쪽에 있는 요소들(`20 30 35 40 45`)은 루트의 왼쪽 서브트리에 속하고, 오른쪽에 있는 요소들(`55 60 70`)은 루트의 오른쪽 서브트리에 속합니다.
    3.  **이제 각 서브트리에 대해 재귀적으로 이 과정을 반복합니다.**
        *   **왼쪽 서브트리**: Inorder (`20 30 35 40 45`), Preorder에서 해당 노드들의 순서는 (`40 30 20 35 45`). 이 중 첫 번째 노드인 `40`이 왼쪽 서브트리의 루트가 됩니다. Inorder에서 `40`을 기준으로 왼쪽(`20 30 35`)은 `40`의 왼쪽 자식, 오른쪽(`45`)은 `40`의 오른쪽 자식이 됩니다.
        *   **오른쪽 서브트리**: Inorder (`55 60 70`), Preorder에서 해당 노드들의 순서는 (`60 55 70`). 이 중 첫 번째 노드인 `60`이 오른쪽 서브트리의 루트가 됩니다. Inorder에서 `60`을 기준으로 왼쪽(`55`)은 `60`의 왼쪽 자식, 오른쪽(`70`)은 `60`의 오른쪽 자식이 됩니다.
    이 과정을 노드가 없을 때까지 반복하면 원래의 이진 트리를 유일하게 재구성할 수 있습니다.

*   **시험 포인트**
    *   ⭐ **세 가지 순회(Preorder, Inorder, Postorder)의 정확한 정의와 각 순서(Root-Left-Right 등)를 이해하고 암기해야 합니다.**
    *   ⭐ **주어진 이진 트리에 대해 세 가지 순회 결과를 정확하게 도출할 수 있어야 합니다.** (특히 그림으로 주어졌을 때)
    *   ⭐ **두 가지 순회 정보가 주어졌을 때 이진 트리를 재구성하는 방법을 이해해야 합니다.**
        *   **Preorder + Inorder** 조합 또는 **Postorder + Inorder** 조합은 유일하게 이진 트리를 재구성할 수 있습니다.
        *   Preorder와 Postorder 순회만으로는 유일하게 트리를 재구성할 수 없습니다 (예: 루트를 가진 한 개의 노드 트리와, 루트와 왼쪽 자식만 있는 트리의 Preorder와 Postorder는 동일할 수 있음).
    *   ⭐ **각 순회의 재귀적 구현 방식과 그에 따른 시간 복잡도 $O(N)$ 및 공간 복잡도 $O(H)$를 이해해야 합니다.**
    *   ⭐ **재구성 과정에서 Preorder의 첫 번째 요소가 루트임을, Postorder의 마지막 요소가 루트임을 파악하고 Inorder를 통해 좌우 서브트리를 분할하는 원리를 숙지해야 합니다.**

## Slide 6

**핵심 개념**

*   **이진 트리 순회 (Binary Tree Traversal)**
    *   이진 트리에 있는 모든 노드를 체계적인 방법으로 한 번씩 방문하는 과정입니다. 노드를 방문하는 순서에 따라 크게 세 가지 주요 순회 방식이 있습니다.
    *   **중위 순회 (Inorder Traversal)**: 왼쪽 자식 노드 -> 현재 노드 -> 오른쪽 자식 노드 순서로 방문합니다. (Left-Root-Right) 이진 탐색 트리(BST)의 경우, 중위 순회는 노드 값을 오름차순으로 정렬된 형태로 얻을 수 있습니다.
    *   **후위 순회 (Postorder Traversal)**: 왼쪽 자식 노드 -> 오른쪽 자식 노드 -> 현재 노드 순서로 방문합니다. (Left-Right-Root) 트리를 삭제할 때 자식 노드부터 삭제해야 하므로 주로 사용됩니다.
    *   **전위 순회 (Preorder Traversal)**: 현재 노드 -> 왼쪽 자식 노드 -> 오른쪽 자식 노드 순서로 방문합니다. (Root-Left-Right) 트리를 복사하거나 트리의 표현식을 만들 때 유용합니다.

*   **단일 순회와 트리 재구성의 한계**
    *   슬라이드의 핵심은 **어떤 단일 순회 결과(전위, 중위, 후위 중 하나)만으로는 해당 이진 트리의 원래 구조를 유일하게 재구성할 수 없다**는 것입니다. 즉, 동일한 순회 결과를 갖는 여러 이진 트리가 존재할 수 있습니다.
    *   이는 각 노드가 자식 노드(왼쪽, 오른쪽)를 가질 수 있는 경우의 수가 다양하기 때문입니다. 단일 순회로는 부모-자식 관계나 서브트리의 구조에 대한 충분한 정보를 제공하지 못합니다.

**코드/수식 해설**

슬라이드에는 직접적인 코드나 수식이 없지만, 이진 트리 순회의 개념을 이해하기 위해 중위 순회의 일반적인 의사코드를 살펴보겠습니다.

```cpp
// 중위 순회 (Inorder Traversal) 의사코드
void inorderTraversal(Node* node) {
    if (node == nullptr) { // 기저 조건: 노드가 없으면 반환
        return;
    }
    inorderTraversal(node->left);  // 1. 왼쪽 서브트리 재귀적으로 순회
    print(node->data);             // 2. 현재 노드 방문 (예: 데이터 출력)
    inorderTraversal(node->right); // 3. 오른쪽 서브트리 재귀적으로 순회
}
```

*   **설명**: 위 의사코드는 `node`가 `nullptr`이 아닐 때, 먼저 왼쪽 자식 노드를 방문하고, 그 다음 현재 노드의 데이터를 처리한 후, 마지막으로 오른쪽 자식 노드를 방문하는 중위 순회의 전형적인 재귀적 구현을 보여줍니다. 전위 순회는 `print(node->data)`가 재귀 호출(`inorderTraversal(node->left)`)보다 먼저 오고, 후위 순회는 `print(node->data)`가 두 재귀 호출보다 나중에 옵니다.

**구체적 예시**

슬라이드에 제시된 이진 트리와 그 순회 결과를 분석하고, 왜 단일 순회가 유일하지 않은지 예시를 들어 설명합니다.

1.  **슬라이드의 이진 트리 분석:**
    ```
        50
       /  \
      40   60
     / \   / \
    30 45 55 70
   / \
  20 35
    ```
    이 트리에 대해 중위 순회와 후위 순회를 수행하면 다음과 같습니다.
    *   **Inorder Traversal (L-R-R)**:
        *   `20 -> 30 -> 35 -> 40 -> 45 -> 50 -> 55 -> 60 -> 70`
        *   슬라이드: `< 20,30,35,40,45,50,55,60,70 >` (일치)
    *   **Postorder Traversal (L-R-R)**:
        *   `20 -> 35 -> 30 -> 45 -> 40 -> 55 -> 70 -> 60 -> 50`
        *   슬라이드: `< 20,35,30,45,40,55,70,60,50 >` (일치)

2.  **단일 순회가 유일하지 않은 이유 (예시)**:
    `Inorder = < A, B >` 라는 중위 순회 결과가 있다고 가정해 봅시다. 이 결과를 갖는 트리는 다음 두 가지 이상이 가능합니다.

    *   **트리 1:**
        ```
          B
         /
        A
        ```
        이 트리의 중위 순회는 A -> B 입니다.

    *   **트리 2:**
        ```
        A
         \
          B
        ```
        이 트리의 중위 순회는 A -> B 입니다.

    위 두 트리는 중위 순회 결과는 같지만, 그 구조는 완전히 다릅니다. 이처럼 단 하나의 순회 결과만으로는 어떤 트리가 원래의 트리를 대표하는지 특정할 수 없습니다. 이는 전위 순회나 후위 순회 단독으로도 마찬가지입니다.

**시험 포인트**

*   ⭐ **이진 트리의 구조를 유일하게 재구성하려면 최소 두 가지 순회 결과가 필요합니다.** 일반적으로 **중위 순회(Inorder Traversal)**가 포함된 상태에서 다른 순회(전위 순회 또는 후위 순회) 하나가 더 주어지면 트리를 유일하게 재구성할 수 있습니다. (예: Inorder + Preorder, Inorder + Postorder)
*   ⭐ 각 순회 방식(전위, 중위, 후위)의 **정의와 노드 방문 순서**를 정확히 이해하고 있어야 합니다.
*   ⭐ **단일 순회만으로 트리를 유일하게 복원할 수 없는 이유**를 설명할 수 있어야 합니다. (동일한 순회 결과를 갖는 여러 트리가 존재할 수 있기 때문)
*   ⭐ 주어진 이진 트리에 대해 세 가지 순회(전위, 중위, 후위) 결과를 정확히 도출할 수 있어야 합니다.

## Slide 7

**핵심 개념**

이 슬라이드는 두 가지 트리 순회(traversal) 결과만으로 이진 트리(Binary Tree)의 구조를 고유하게(uniquely) 재구성할 수 있는 조건에 대해 설명합니다.

*   **고유한 재구성 조합**: 이진 트리는 다음 세 가지 순회 조합 중 하나를 알면 그 구조를 완벽하게 결정하고 재구성할 수 있습니다.
    1.  `(postorder & inorder)`
    2.  `(preorder & inorder)`
    3.  `(level-order & inorder)`
*   **`inorder`의 역할**: `inorder` 순회는 트리의 루트 노드를 기준으로 왼쪽 서브트리와 오른쪽 서브트리를 구분하는 데 핵심적인 정보를 제공합니다. 어떤 노드가 루트로 식별되면, `inorder` 시퀀스에서 그 노드의 왼쪽에 있는 모든 노드는 왼쪽 서브트리에 속하고, 오른쪽에 있는 모든 노드는 오른쪽 서브트리에 속합니다.
*   **루트 노드 식별자**:
    *   `postorder` 순회에서는 항상 시퀀스의 **마지막** 원소가 현재 서브트리의 루트 노드가 됩니다.
    *   `preorder` 순회에서는 항상 시퀀스의 **첫 번째** 원소가 현재 서브트리의 루트 노드가 됩니다.
    *   `level-order` 순회(너비 우선 탐색)에서도 시퀀스의 첫 번째 원소가 전체 트리의 루트 노드가 됩니다.
*   **모호성 (Ambiguity)**: 위에서 언급된 조합 외의 다른 순회 조합 (예: `preorder`와 `postorder`만으로)은 이진 트리의 구조를 고유하게 결정하지 못하고, 여러 개의 가능한 트리 구조가 존재할 수 있는 모호성을 야기합니다.

**코드/수식 해설**

해당 슬라이드에는 코드 블록이나 수학적 수식이 포함되어 있지 않습니다.

**구체적 예시**

`preorder` 순회 결과가 `[A, B, D, E, C, F]` 이고, `inorder` 순회 결과가 `[D, B, E, A, F, C]` 인 이진 트리를 재구성하는 과정을 예시로 들어보겠습니다.

1.  **루트 찾기**: `preorder`의 첫 번째 원소는 항상 루트입니다. 따라서 `A`가 전체 트리의 루트 노드입니다.
2.  **서브트리 분할 (inorder 활용)**:
    *   `inorder` 시퀀스에서 `A`를 찾으면 `[D, B, E]`는 `A`의 왼쪽 서브트리 노드들이고, `[F, C]`는 `A`의 오른쪽 서브트리 노드들임을 알 수 있습니다.
    *   `inorder` 왼쪽 서브트리: `[D, B, E]`
    *   `inorder` 오른쪽 서브트리: `[F, C]`
3.  **`preorder` 시퀀스 분할**: `preorder`에서 `A`를 제외한 나머지 `[B, D, E, C, F]`를 왼쪽 서브트리와 오른쪽 서브트리에 해당하는 부분으로 나눕니다.
    *   `preorder` 왼쪽 서브트리: `[B, D, E]` (inorder 왼쪽 서브트리 노드들을 포함하는 `preorder` 부분)
    *   `preorder` 오른쪽 서브트리: `[C, F]` (inorder 오른쪽 서브트리 노드들을 포함하는 `preorder` 부분)
4.  **재귀적 반복**: 이제 각 서브트리에 대해 1~3단계를 재귀적으로 반복합니다.
    *   **왼쪽 서브트리**: `preorder = [B, D, E]`, `inorder = [D, B, E]`
        *   루트는 `B`입니다.
        *   `inorder`에서 `B`를 기준으로 `[D]`는 왼쪽, `[E]`는 오른쪽입니다.
        *   `preorder`에서 `B`를 제외한 `[D, E]`에서 `D`는 `B`의 왼쪽 자식, `E`는 `B`의 오른쪽 자식입니다.
    *   **오른쪽 서브트리**: `preorder = [C, F]`, `inorder = [F, C]`
        *   루트는 `C`입니다.
        *   `inorder`에서 `C`를 기준으로 `[F]`는 왼쪽입니다. `C`의 오른쪽은 없습니다.
        *   `preorder`에서 `C`를 제외한 `[F]`는 `C`의 왼쪽 자식입니다.

이 과정을 통해 우리는 다음과 같은 이진 트리를 유일하게 재구성할 수 있습니다:
```
      A
     / \
    B   C
   / \ /
  D  E F
```

**시험 포인트**

*   ⭐ **이진 트리를 고유하게 재구성할 수 있는 두 가지 순회 조합**을 정확하게 암기하고 이해해야 합니다: `(postorder & inorder)`, `(preorder & inorder)`, `(level-order & inorder)`. 이 중 `inorder`가 반드시 포함되어야 한다는 점이 핵심입니다.
*   ⭐ 각 순회 방식(`preorder`, `postorder`, `inorder`, `level-order`)이 **루트 노드나 서브트리를 식별하는 데 어떻게 활용되는지** 명확히 알아야 합니다. 특히 `preorder`는 시퀀스의 첫 번째 요소가 루트, `postorder`는 마지막 요소가 루트, `inorder`는 루트를 기준으로 좌우 서브트리를 분할한다는 점을 기억하세요.
*   ⭐ `preorder`와 `postorder` 조합만으로는 트리를 고유하게 재구성할 수 없는 이유, 즉 **모호성이 발생하는 경우**에 대해 설명할 수 있어야 합니다. (예: 루트가 A이고 왼쪽 자식이 B인 트리와, 루트가 A이고 오른쪽 자식이 B인 트리는 `preorder`와 `postorder`가 같을 수 있습니다.)

### **핵심 개념**

*   **이진 트리 재구성**: 이 슬라이드는 이진 트리의 **중위 순회(Inorder Traversal)**와 **후위 순회(Postorder Traversal)** 결과를 이용하여 원래의 이진 트리를 유일하게 재구성하는 방법을 설명합니다.
*   **중위 순회(Inorder Traversal)**: 왼쪽 자식 $\to$ 루트 $\to$ 오른쪽 자식 순서로 노드를 방문합니다. 이 순회 결과는 특정 노드를 루트로 보았을 때, 그 노드의 왼쪽에 있는 모든 노드들이 왼쪽 서브트리에 속하고, 오른쪽에 있는 모든 노드들이 오른쪽 서브트리에 속함을 알려주는 '분할 기준'이 됩니다.
*   **후위 순회(Postorder Traversal)**: 왼쪽 자식 $\to$ 오른쪽 자식 $\to$ 루트 순서로 노드를 방문합니다. 따라서 후위 순회 결과의 **가장 마지막 원소**는 항상 현재 트리의 루트 노드입니다.

### **코드/수식 해설**

이진 트리를 중위 순회 및 후위 순회로 재구성하는 과정은 재귀적으로 이루어집니다. `buildTree` 함수는 주어진 `inOrder` 배열과 `postOrder` 배열의 특정 범위 내에서 서브트리를 구축합니다.

```python
class TreeNode:
    def __init__(self, val=0, left=None, right=None):
        self.val = val
        self.left = left
        self.right = right

def buildTree(inorder: list[int], postorder: list[int]) -> TreeNode:
    # 1. 기저 조건: 배열이 비어있으면 트리를 만들 수 없음
    if not inorder or not postorder:
        return None

    # 2. 후위 순회의 마지막 원소는 현재 트리의 루트 노드
    root_val = postorder[-1]
    root = TreeNode(root_val)

    # 3. 중위 순회에서 루트 값의 위치를 찾아 좌우 서브트리 분할
    #    (성능 최적화를 위해 inorder_map을 사용하면 O(1)에 찾을 수 있음)
    root_idx_inorder = inorder.index(root_val)

    # 4. 좌우 서브트리의 중위 순회 및 후위 순회 배열 생성
    #    inorder 배열에서 root_idx_inorder를 기준으로 분할
    left_inorder = inorder[:root_idx_inorder]
    right_inorder = inorder[root_idx_inorder + 1:]

    #    postorder 배열에서 마지막 원소(루트)를 제외한 부분을
    #    left_inorder의 길이와 right_inorder의 길이에 맞춰 분할
    #    left_postorder의 길이는 left_inorder의 길이와 같음
    left_postorder = postorder[:len(left_inorder)]
    right_postorder = postorder[len(left_inorder):-1] # -1은 루트를 제외

    # 5. 재귀적으로 좌우 서브트리 구성
    root.left = buildTree(left_inorder, left_postorder)
    root.right = buildTree(right_inorder, right_postorder)

    return root
```

*   **시간 복잡도 분석**:
    *   `inorder.index()` 메서드는 최악의 경우 $O(N)$ 시간이 걸립니다. 이를 해시 맵(hash map)을 사용하여 미리 `value -> index` 매핑을 저장해두면 $O(1)$로 만들 수 있습니다.
    *   배열 슬라이싱(`inorder[:]`)도 최악의 경우 $O(N)$ 시간이 걸립니다.
    *   따라서 해시 맵을 사용하지 않으면 각 재귀 호출마다 $O(N)$의 작업이 발생하여 총 시간 복잡도는 $O(N^2)$이 됩니다.
    *   만약 `inOrder` 인덱스 매핑을 해시 맵으로 미리 구축하고, 배열을 슬라이싱하는 대신 시작 및 끝 인덱스를 전달하는 방식으로 재귀 함수를 구현한다면, 각 재귀 호출이 $O(1)$에 가까워져 전체 시간 복잡도는 $O(N)$으로 최적화될 수 있습니다.

### **시험 포인트**

*   ⭐ **이진 트리 유일성**: 중위 순회와 다른 한 가지 순회(전위 또는 후위 순회) 결과가 주어질 때만 이진 트리를 유일하게 재구성할 수 있습니다. 예를 들어, 전위 순회와 후위 순회만으로는 유일한 트리를 만들 수 없습니다 (예: 루트-왼쪽-오른쪽, 루트-오른쪽-왼쪽의 경우 루트만 알고 좌우 서브트리를 확정할 수 없음).
*   ⭐ **재구성 알고리즘의 단계 이해**:
    1.  주어진 `postOrder` (또는 `preOrder`)에서 현재 서브트리의 루트 노드를 정확히 식별하는 방법.
    2.  식별된 루트 노드를 기준으로 `inOrder` 배열을 좌우 서브트리로 분할하는 방법.
    3.  좌우 서브트리의 크기에 맞춰 `postOrder` (또는 `preOrder`) 배열을 다시 분할하여 다음 재귀 호출에 사용할 배열을 만드는 방법.
    4.  이 과정을 재귀적으로 반복하여 트리를 구축하는 로직.
*   ⭐ **예제 트레이싱**: 주어진 `inOrder`와 `postOrder` 배열을 이용하여 재구성 과정을 단계별로 직접 트레이싱하여 이진 트리를 그려보는 연습이 중요합니다. 슬라이드의 예시와 같이 중간 배열의 분할과 노드 연결 과정을 정확히 이해해야 합니다.
*   ⭐ **시간 복잡도 최적화**: $O(N^2)$ 재귀 구현을 $O(N)$으로 최적화하는 방법을 이해해야 합니다 (예: `inOrder` 인덱스를 해시 맵에 미리 저장하거나, 배열 슬라이싱 대신 인덱스 범위를 전달).

## Slide 9

**핵심 개념**:
이 슬라이드는 이진 트리의 **전위 순회(Preorder Traversal)**와 **중위 순회(Inorder Traversal)** 결과가 주어졌을 때, 해당 이진 트리를 유일하게 재구성하는 알고리즘을 설명합니다. 전위 순회는 '루트 -> 왼쪽 서브트리 -> 오른쪽 서브트리' 순으로 노드를 방문하고, 중위 순회는 '왼쪽 서브트리 -> 루트 -> 오른쪽 서브트리' 순으로 노드를 방문하는 특징을 활용합니다. 이 두 가지 순회는 트리의 구조를 완벽하게 파악하기 위한 필수적인 정보를 제공합니다.

**코드/수식 해설**:
제시된 알고리즘은 재귀적인 방식으로 이진 트리를 구성합니다. 각 단계는 현재 처리할 서브트리의 루트를 식별하고, 이 루트를 기준으로 왼쪽 및 오른쪽 서브트리를 분할한 후, 각 서브트리에 대해 동일한 과정을 재귀적으로 반복합니다.

1.  **루트(Root) 식별**:
    *   전위 순회(Preorder)의 가장 첫 번째 노드는 항상 현재 트리의 루트 노드입니다. 이는 전위 순회의 정의($\text{Root} \rightarrow \text{Left Subtree} \rightarrow \text{Right Subtree}$)에 기반합니다.

2.  **왼쪽 자식(Left Child) 및 왼쪽 서브트리(Left Subtree) 식별**:
    *   중위 순회(Inorder) 결과에서 현재 식별된 `Root` 노드의 위치를 찾습니다.
    *   `Root` 노드의 왼쪽에 있는 모든 노드는 `Root`의 **왼쪽 서브트리(Left_subtree)**를 구성합니다.
    *   이 `Left_subtree`에 포함된 노드들 중에서 전위 순회에 가장 먼저 나타나는 노드가 `Left_child`가 됩니다. (정확히는, 전체 전위 순회에서 `Root` 노드 바로 다음에 오면서 `Left_subtree`에 속하는 노드들이 `Left_subtree`의 전위 순회를 이루며, 그 첫 번째 노드가 `Left_child`입니다.)

3.  **오른쪽 자식(Right Child) 및 오른쪽 서브트리(Right Subtree) 식별**:
    *   유사하게, 중위 순회(Inorder) 결과에서 `Root` 노드의 오른쪽에 있는 모든 노드는 `Root`의 **오른쪽 서브트리(Right_subtree)**를 구성합니다.
    *   이 `Right_subtree`에 포함된 노드들 중에서 전위 순회에 가장 먼저 나타나는 노드가 `Right_child`가 됩니다. (마찬가지로, 전체 전위 순회에서 `Left_subtree`의 노드들을 모두 거친 후 `Right_subtree`에 속하는 노드들로 구성된 전위 순회 부분의 첫 번째 노드가 `Right_child`입니다.)

4.  **재귀적 반복**:
    *   2단계와 3단계에서 식별된 `Left_child`와 `Right_child`를 각각 루트로 하여, 해당 서브트리의 전위 순회 및 중위 순회 부분에 대해 이 과정을 다시 재귀적으로 반복합니다.
    *   이 과정은 모든 노드가 트리에 배치되거나, 특정 서브트리가 비어있어 더 이상 자식을 가질 수 없을 때까지 계속됩니다.

이 알고리즘은 일반적으로 다음과 같은 재귀 함수 형태로 구현될 수 있습니다.

```cpp
// 가상의 함수 시그니처 (실제 구현 시 인덱스나 부분 배열 처리가 필요)
TreeNode* buildTree(vector<int>& preorder, vector<int>& inorder) {
    if (preorder.empty() || inorder.empty()) {
        return nullptr; // 빈 서브트리
    }

    // 1. Preorder의 첫 번째 요소가 현재 서브트리의 루트
    int rootVal = preorder[0];
    TreeNode* root = new TreeNode(rootVal);

    // 2. Inorder에서 루트의 위치를 찾아 왼쪽/오른쪽 서브트리 분할
    auto it = find(inorder.begin(), inorder.end(), rootVal);
    int inorderRootIdx = distance(inorder.begin(), it);

    // 3. 왼쪽 서브트리 구성
    // Inorder: [left_nodes] root [right_nodes]
    // Preorder: root [left_nodes_preorder] [right_nodes_preorder]
    
    // 왼쪽 서브트리의 Inorder 배열 추출
    vector<int> leftInorder(inorder.begin(), it);
    // 왼쪽 서브트리의 Preorder 배열 추출 (leftInorder.size() 만큼)
    vector<int> leftPreorder(preorder.begin() + 1, preorder.begin() + 1 + leftInorder.size());
    root->left = buildTree(leftPreorder, leftInorder);

    // 4. 오른쪽 서브트리 구성
    // 오른쪽 서브트리의 Inorder 배열 추출
    vector<int> rightInorder(it + 1, inorder.end());
    // 오른쪽 서브트리의 Preorder 배열 추출
    vector<int> rightPreorder(preorder.begin() + 1 + leftInorder.size(), preorder.end());
    root->right = buildTree(rightPreorder, rightInorder);

    return root;
}
```

**구체적 예시**:
다음과 같은 이진 트리의 순회 결과가 주어졌다고 가정해 봅시다.
*   **Preorder**: `[A, B, D, E, C, F]`
*   **Inorder**: `[D, B, E, A, F, C]`

단계별 트리의 재구성 과정:

1.  **초기 호출**: `buildTree([A,B,D,E,C,F], [D,B,E,A,F,C])`
    *   **루트**: Preorder의 첫 노드는 `A`입니다. `A`가 트리의 루트가 됩니다.
    *   **Inorder에서 A 찾기**: `[D, B, E]`, `A`, `[F, C]`
        *   왼쪽 서브트리 (Inorder): `[D, B, E]`
        *   오른쪽 서브트리 (Inorder): `[F, C]`

2.  **왼쪽 서브트리 재귀 호출**: `buildTree([B,D,E], [D,B,E])` (A 다음 Preorder와 Inorder 왼쪽 부분)
    *   **루트**: Preorder의 첫 노드는 `B`입니다. `B`가 `A`의 왼쪽 자식이 됩니다.
    *   **Inorder에서 B 찾기**: `[D]`, `B`, `[E]`
        *   왼쪽 서브트리 (Inorder): `[D]`
        *   오른쪽 서브트리 (Inorder): `[E]`

3.  **`B`의 왼쪽 자식 재귀 호출**: `buildTree([D], [D])`
    *   **루트**: `D`입니다. `D`가 `B`의 왼쪽 자식이 됩니다.
    *   `D`는 자식이 없습니다. (Preorder, Inorder 모두 `D`만 남음)

4.  **`B`의 오른쪽 자식 재귀 호출**: `buildTree([E], [E])`
    *   **루트**: `E`입니다. `E`가 `B`의 오른쪽 자식이 됩니다.
    *   `E`는 자식이 없습니다.

5.  **오른쪽 서브트리 재귀 호출**: `buildTree([C,F], [F,C])` (A 다음 BDE를 제외한 Preorder와 Inorder 오른쪽 부분)
    *   **루트**: Preorder의 첫 노드는 `C`입니다. `C`가 `A`의 오른쪽 자식이 됩니다.
    *   **Inorder에서 C 찾기**: `[F]`, `C`, `[]` (Inorder: `[F, C]`)
        *   왼쪽 서브트리 (Inorder): `[F]`
        *   오른쪽 서브트리 (Inorder): `[]` (비어있음)

6.  **`C`의 왼쪽 자식 재귀 호출**: `buildTree([F], [F])`
    *   **루트**: `F`입니다. `F`가 `C`의 왼쪽 자식이 됩니다.
    *   `F`는 자식이 없습니다.

7.  **`C`의 오른쪽 자식 재귀 호출**: `buildTree([], [])`
    *   비어 있으므로, `C`의 오른쪽 자식은 없습니다.

최종적으로 재구성된 트리는 다음과 같습니다.
```
      A
     / \
    B   C
   / \ /
  D   E F
```

**시험 포인트**:
*   ⭐ **전위 순회와 중위 순회 (또는 후위 순회와 중위 순회) 쌍이 주어졌을 때만 유일한 이진 트리를 재구성할 수 있는 이유**를 설명할 수 있어야 합니다. (중위 순회가 루트를 기준으로 좌우 서브트리를 명확히 구분해주기 때문입니다.)
*   ⭐ 주어진 Preorder와 Inorder (혹은 Postorder와 Inorder) 배열을 보고 **이진 트리를 직접 재구성하는 과정**을 단계별로 설명하거나 그려낼 수 있어야 합니다.
*   ⭐ 이 알고리즘의 **재귀적 특성**과 각 재귀 호출 시 Preorder 및 Inorder 배열의 어느 부분이 현재 서브트리에 해당하는지를 정확히 파악하는 방법이 중요합니다.
*   ⭐ 시간 복잡도 분석: 각 노드에 대해 Inorder 배열에서 루트를 찾아야 하므로 $O(N)$의 선형 탐색이 필요하고, 총 $N$개의 노드가 있으므로 전체 시간 복잡도는 $O(N^2)$이 됩니다. 해시 맵 등을 사용하여 Inorder에서 노드의 인덱스를 미리 저장해두면 $O(1)$에 탐색 가능하여 전체 시간 복잡도를 $O(N)$으로 최적화할 수 있습니다.
*   ⭐ 이진 트리의 세 가지 주요 순회(전위, 중위, 후위)의 정의와 특징을 명확히 구분할 수 있어야 합니다.

## Slide 10

**핵심 개념**:
*   **일반 $k$-ary 트리 (General $k$-ary Tree)**: 각 노드가 최대 $k$개의 자식 노드를 가질 수 있는 트리를 의미합니다. 여기서 $k$는 고정된 값이 아닐 수도 있고, 각 노드마다 실제 자식 수는 0부터 $k$까지 다양할 수 있습니다. 슬라이드 우측의 그림은 노드 'b'가 3개의 자식을 가지므로, 일반 트리의 예시를 보여줍니다.
*   **트리 구현 방식**: 일반 트리를 컴퓨터 메모리에 표현하고 조작하기 위한 여러 가지 방법들이 있습니다. 슬라이드에서 언급된 세 가지 주요 구현 방식들은 각각의 장단점을 가지며, 이들을 이해하고 분석하는 것이 중요합니다.

**코드/수식 해설**:
이 슬라이드에는 직접적인 코드나 수식이 제시되어 있지 않지만, 언급된 세 가지 구현 방식의 개념적인 구조를 설명하고 C/C++ pseudo-code 형태로 간략히 표현해 볼 수 있습니다.

1.  **Simple array implementation (단순 배열 구현)**:
    *   **개념**: 트리의 노드들을 선형 배열에 저장하는 방식입니다. 이진 트리의 경우 부모-자식 관계를 배열 인덱스로 쉽게 계산할 수 있지만, 일반 $k$-ary 트리의 경우 각 노드의 자식 수가 가변적이고 $k$ 값도 고정되어 있지 않을 수 있어 효율적인 인덱스 계산이 어렵습니다. 일반적으로 각 노드 구조체 안에 자식 노드들을 가리키는 고정 크기 배열을 두거나, 전체 노드 배열에서 자식들의 인덱스를 저장하는 방식으로 구현됩니다.
    *   **예시 구조**: (최대 $K$개의 자식을 가정)
        ```c
        #define K_MAX 10 // 최대 자식 노드 수 가정

        typedef struct Node {
            char data;
            int child_indices[K_MAX]; // 자식 노드들의 배열 내 인덱스
            int num_children;         // 실제 자식 노드 수
        } Node;

        Node tree_nodes[MAX_NODES]; // 모든 노드를 저장할 배열
        ```
    *   **장단점**: 자식 노드에 대한 접근이 빠르지만, $K_{MAX}$가 크고 실제 자식 수가 적은 노드가 많다면 메모리 낭비가 심할 수 있습니다.

2.  **List-of-Children implementation (자식 리스트 구현)**:
    *   **개념**: 각 노드가 자신의 자식 노드들을 가리키는 포인터들의 리스트(예: 연결 리스트 또는 동적 배열)를 유지하는 방식입니다.
    *   **예시 구조**:
        ```c
        // 자식 노드를 연결 리스트로 관리하는 경우
        typedef struct ChildNodeList {
            struct Node* child_ptr;        // 자식 노드에 대한 포인터
            struct ChildNodeList* next;    // 다음 형제 노드에 대한 포인터
        } ChildNodeList;

        typedef struct Node {
            char data;
            ChildNodeList* children_head; // 자식 리스트의 시작 (가장 왼쪽 자식)
        } Node;
        ```
    *   **장단점**: 필요한 만큼만 메모리를 할당하므로 `Simple array implementation`보다 공간 효율적일 수 있습니다. 자식 노드의 추가/삭제가 용이하지만, 특정 자식을 찾기 위해서는 리스트를 순회해야 할 수 있습니다.

3.  **Left-Child/Right-Sibling implementation (왼쪽 자식/오른쪽 형제 구현)**:
    *   **개념**: 일반 트리를 이진 트리 형태로 변환하여 표현하는 효율적인 방법입니다. 각 노드는 다음 두 개의 포인터만을 가집니다.
        *   `left_child`: 자신의 가장 왼쪽 자식을 가리킵니다.
        *   `right_sibling`: 자신과 같은 부모를 가진 노드 중 자신 바로 오른쪽에 있는 형제 노드를 가리킵니다.
    *   **예시 구조**:
        ```c
        typedef struct Node {
            char data;
            struct Node* left_child;    // 가장 왼쪽 자식 노드에 대한 포인터
            struct Node* right_sibling; // 오른쪽 형제 노드에 대한 포인터
        } Node;
        ```
    *   **장단점**: 임의의 개수의 자식을 갖는 일반 트리를 고정된 두 개의 포인터만으로 표현할 수 있어 메모리 효율적입니다. 또한, 이진 트리에 대한 많은 알고리즘을 재활용할 수 있다는 장점이 있습니다. 그러나 모든 자식을 찾으려면 `left_child`를 통해 첫 자식에 접근한 후 `right_sibling` 포인터를 계속 따라가야 하므로 $O(k)$ 시간이 소요될 수 있습니다.

**구체적 예시**:
슬라이드에 제시된 트리 구조를 `Left-Child/Right-Sibling` 방식으로 표현해 봅시다.
```
      a
     / \
    b   c
   /|\
  d e f
```

`Left-Child/Right-Sibling` 표현은 다음과 같이 이진 트리 형태로 변환됩니다.
*   **노드 'a'**: `left_child`는 'b', `right_sibling`은 NULL (루트이므로)
*   **노드 'b'**: `left_child`는 'd', `right_sibling`은 'c'
*   **노드 'c'**: `left_child`는 NULL, `right_sibling`은 NULL
*   **노드 'd'**: `left_child`는 NULL, `right_sibling`은 'e'
*   **노드 'e'**: `left_child`는 NULL, `right_sibling`은 'f'
*   **노드 'f'**: `left_child`는 NULL, `right_sibling`은 NULL

이를 시각적으로 표현하면, `left_child`는 일반적인 이진 트리의 왼쪽 자식처럼 아래로 내려가고, `right_sibling`은 형제 노드를 가리키며 옆으로 이어집니다.

```
       a
       |
       v
       b --- (right_sibling) --> c
       |
       v
       d --- (right_sibling) --> e --- (right_sibling) --> f
       |
       v
      NULL
```
(위 그림에서 수직선은 `left_child`를, 수평선은 `right_sibling`을 나타냅니다.)

**시험 포인트**:
*   ⭐ **일반 $k$-ary 트리의 정의**를 정확히 설명하고, 일반 트리와 이진 트리의 차이점을 이해해야 합니다.
*   ⭐ **세 가지 주요 트리 구현 방식(Simple array, List-of-Children, Left-Child/Right-Sibling) 각각의 개념, 특징, 장단점**을 비교하여 설명할 수 있어야 합니다.
*   ⭐ 특히, **Left-Child/Right-Sibling 구현**이 일반 트리를 이진 트리 형태로 변환하여 표현하는 방식임을 이해하고, 주어진 일반 트리를 이 방식으로 변환한 그림이나 설명을 제시할 수 있어야 합니다. 이는 일반 트리를 다루는 중요한 기법 중 하나입니다.
*   각 구현 방식에 대한 **시간 및 공간 복잡도 분석** (예: 특정 노드의 모든 자식을 찾는 시간, 노드 삽입/삭제의 복잡도)에 대한 기본적인 이해가 요구될 수 있습니다. (이 슬라이드에서는 다음 분석을 예고하지만, 개념적 중요성은 미리 인지해야 합니다.)

## Slide 11

**핵심 개념**:
이 슬라이드는 트리를 표현하는 간단한 방법 중 하나인 **부모 포인터(Parent point) 표현** 방식을 설명합니다. 이 방식은 두 개의 선형 배열을 사용하여 트리의 구조와 각 노드의 정보를 저장합니다.

*   **`Tree[k] = m`**: 노드 `k`의 부모가 노드 `m`임을 나타냅니다. 즉, 배열의 인덱스 `k`는 노드 자신을, 해당 인덱스에 저장된 값 `m`은 그 노드의 부모를 가리킵니다. 루트 노드의 경우, 부모가 없으므로 특별한 값(예: 0 또는 -1)을 저장하여 이를 표시합니다.
*   **`Label[k]`**: 노드 `k`의 레이블(값, 정보)을 저장합니다. 이 배열은 `Tree[]` 배열과 병렬로(parallel) 사용됩니다.
*   **자식 순서 미지정**: 이 표현 방식은 특정 노드의 자식들 간의 순서를 명시적으로 저장하지 않습니다. 자식들을 찾으려면 `Tree[]` 배열 전체를 스캔해야 합니다.
*   **`struct` 사용 가능**: 슬라이드에 언급된 바와 같이, 배열 대신 `struct`를 사용하여 노드 정보를 구조적으로 표현하는 방법도 가능합니다 (예: 각 노드 구조체에 `parent` 필드와 `label` 필드를 포함).

**코드/수식 해설**:
트리 노드 `k`의 부모가 노드 `m`인 관계는 다음과 같이 표현됩니다.

$$
Tree[k] = m \iff \text{node } m \text{ is the parent of node } k
$$

각 노드 `k`의 실제 값(레이블)은 `Label[k]` 배열에 저장됩니다.

```c
// C++를 가정한 배열 기반 트리 표현 예시
// 노드의 개수 N에 따라 배열의 크기가 결정됩니다.
int Tree[N+1];     // Tree[k]는 노드 k의 부모 인덱스를 저장
char Label[N+1];   // Label[k]는 노드 k의 값을 저장 (예시로 char 타입)

// 예를 들어, 노드 1이 루트이고, 노드 2의 부모가 1인 경우
Tree[1] = 0; // 루트 노드의 부모는 0 (또는 다른 특별한 값)
Label[1] = 'a';

Tree[2] = 1; // 노드 2의 부모는 노드 1
Label[2] = 'b';
```

**구체적 예시**:
슬라이드에 제시된 예시 트리를 통해 `Tree[]`와 `Label[]` 배열의 구성을 살펴봅시다.

```
트리 구조:
      1 (a)
     /   \
   2 (b)  5 (c)
  / | \
3(d) 6(e) 4(f)
```

이 트리를 `Tree[]`와 `Label[]` 배열로 표현하면 다음과 같습니다:

| 노드 인덱스 `k` | 1 | 2 | 3 | 4 | 5 | 6 | ... | n |
| :-------------- | :-: | :-: | :-: | :-: | :-: | :-: | --- | - |
| `Tree[k]` (부모) | 0 | 1 | 2 | 2 | 1 | 2 | ... |   |
| `Label[k]` (값) | a | b | d | f | c | e | ... |   |

*   **노드 1 (a)**: 루트 노드이므로 `Tree[1] = 0`으로 부모가 없음을 표시하고, `Label[1] = 'a'`입니다.
*   **노드 2 (b)**: 부모가 노드 1이므로 `Tree[2] = 1`이고, `Label[2] = 'b'`입니다.
*   **노드 3 (d)**: 부모가 노드 2이므로 `Tree[3] = 2`이고, `Label[3] = 'd'`입니다.
*   **노드 4 (f)**: 부모가 노드 2이므로 `Tree[4] = 2`이고, `Label[4] = 'f'`입니다.
*   **노드 5 (c)**: 부모가 노드 1이므로 `Tree[5] = 1`이고, `Label[5] = 'c'`입니다.
*   **노드 6 (e)**: 부모가 노드 2이므로 `Tree[6] = 2`이고, `Label[6] = 'e'`입니다.

이 예시를 통해 각 노드의 인덱스가 고유하며, 해당 인덱스의 `Tree[]` 값으로 부모 노드를, `Label[]` 값으로 노드 자체의 정보를 O(1) 시간에 찾을 수 있음을 알 수 있습니다.

**시험 포인트**:
*   ⭐**부모 포인터 표현의 원리**: `Tree[k] = m`이 노드 `k`의 부모가 `m`임을 의미한다는 것을 정확히 이해해야 합니다. 루트 노드의 부모가 어떻게 표현되는지 (예: 0 또는 특별한 값)도 중요합니다.
*   ⭐**장점**: 특정 노드의 부모를 O(1) 시간에 빠르게 찾을 수 있습니다. 구현이 간단하고, 배열을 사용하여 메모리 지역성을 높일 수 있습니다.
*   ⭐**단점**: 특정 노드의 모든 자식을 찾으려면 `Tree[]` 배열 전체를 스캔해야 하므로 O(N)의 시간이 소요됩니다. 자식들 간의 순서를 표현할 수 없습니다. 이는 특정 트리 순회(예: 전위/중위/후위 순회)를 구현할 때 추가적인 노력이 필요함을 의미합니다.
*   ⭐**다른 트리 표현 방식과의 비교**: 이 방식이 인접 리스트(adjacency list), 자식 포인터(child pointer), 좌측 자식-우측 형제(left-child right-sibling) 표현 등 다른 트리 표현 방식과 어떻게 다른지, 각 방식의 장단점을 비교할 수 있어야 합니다. (예: 이 방식은 자식을 찾는 데 비효율적이지만 부모 찾기는 효율적임)
*   ⭐특정 노드의 깊이를 계산하거나 트리의 높이를 구하는 알고리즘을 부모 포인터 표현을 사용하여 구현하는 방법에 대한 질문이 나올 수 있습니다. (부모를 따라 루트까지 올라가는 방식으로 깊이 계산)

## Slide 12

**핵심 개념**:
*   **트리의 배열 표현**: 이 슬라이드는 일반적인 트리를 효율적으로 표현하고 연산하기 위한 간단한 배열 기반 접근 방식을 제시합니다. 특히, 각 노드의 부모 인덱스를 저장하는 `Tree[]` 배열과 각 노드의 실제 데이터(레이블)를 저장하는 `Label[]` 배열을 사용합니다. `Tree[k]`는 노드 `k`의 부모 노드 인덱스를 나타내며, `Label[k]`는 노드 `k`에 해당하는 데이터를 저장합니다.
*   **시간 복잡도 분석**: 특정 트리 연산(`Parent`, `Leftmost_Child`)에 대해 이 배열 표현 방식에서의 시간 복잡도를 분석합니다.

**코드/수식 해설**:
*   **`Parent(k, T)` 연산**:
    노드 인덱스 $k$가 주어졌을 때, 해당 노드의 부모를 찾는 연산입니다. `Tree[]` 배열이 각 노드의 부모 인덱스를 직접 저장하고 있으므로, 노드 $k$의 부모는 `Tree[k]` 값을 한 번 조회하는 것으로 즉시 알 수 있습니다.
    $$ \text{Parent}(k, T) = \text{Tree}[k] $$
    따라서 시간 복잡도는 $O(1)$입니다.

*   **`Leftmost_Child(k, T)` 연산**:
    노드 인덱스 $k$가 주어졌을 때, 해당 노드의 가장 왼쪽 자식을 찾는 연산입니다. 슬라이드는 "Not well defined (no order of children)"이라며 자식 간의 순서가 명확히 정의되어 있지 않음을 지적합니다.
    `Tree[]` 배열은 각 노드의 부모를 알려주지만, 어떤 노드의 자식들을 직접적으로 연결해주지는 않습니다. 따라서 노드 $k$의 자식들을 찾으려면 `Tree[]` 배열 전체를 순회하면서 부모가 $k$인 모든 노드 $i$를 찾아야 합니다. 즉, $i$를 $1$부터 $n$까지 순회하며 $\text{Tree}[i] == k$ 인 경우를 탐색합니다. '가장 왼쪽'이라는 개념을 인덱스가 가장 작은 자식으로 정의하더라도, 전체 배열을 스캔해야 하므로, 최악의 경우 $n$개의 노드를 모두 확인해야 합니다.
    ```cpp
    int Leftmost_Child(int k, int[] Tree, int n) {
        for (int i = 1; i <= n; i++) { // n은 노드의 총 개수
            if (Tree[i] == k) {
                return i; // 가장 먼저 발견된 자식 (가장 왼쪽)
            }
        }
        return -1; // 자식이 없는 경우
    }
    ```
    따라서 시간 복잡도는 $O(n)$입니다.

**구체적 예시**:
슬라이드에 제시된 `Tree T`와 `Tree[]` 배열을 통해 연산을 살펴봅니다.
`Tree T`: Root `a`(1), Children of `a` are `b`(2), `c`(5). Children of `b` are `d`(3), `f`(4), `e`(6).

*   **`Parent(4, T)` (노드 `f`의 부모 찾기)**:
    `Tree[]` 배열에서 인덱스 4에 해당하는 값을 확인합니다.
    `Tree[4] = 2`
    이는 노드 `f` (인덱스 4)의 부모가 노드 `b` (인덱스 2)임을 나타냅니다. 배열의 한 요소를 직접 참조하는 $O(1)$ 연산입니다.

*   **`Leftmost_Child(2, T)` (노드 `b`의 가장 왼쪽 자식 찾기)**:
    노드 `b` (인덱스 2)의 자식을 찾기 위해 `Tree[]` 배열을 순회합니다.
    - `Tree[1] = 0`
    - `Tree[2] = 1`
    - `Tree[3] = 2` (노드 `d`, 인덱스 3) $\rightarrow$ `b`의 자식 발견. 이것이 현재까지 가장 작은 인덱스의 자식이므로 `Leftmost_Child`는 `3`입니다.
    - `Tree[4] = 2` (노드 `f`, 인덱스 4) $\rightarrow$ `b`의 자식이나 `3`보다 인덱스가 크므로 `Leftmost_Child`는 여전히 `3`입니다.
    - `Tree[5] = 1`
    - `Tree[6] = 2` (노드 `e`, 인덱스 6) $\rightarrow$ `b`의 자식이나 `3`보다 인덱스가 크므로 `Leftmost_Child`는 여전히 `3`입니다.
    이 과정을 위해 모든 노드의 부모 정보를 확인해야 하므로 $O(n)$ 시간이 소요됩니다. 최종적으로 노드 `b`의 가장 왼쪽 자식은 노드 `d` (인덱스 3)입니다.

**시험 포인트**:
*   ⭐ **트리 배열 표현의 이해**: 각 노드의 부모 인덱스를 저장하는 `parent array`의 개념과 역할(`Tree[k] = parent_of_k`)을 정확히 이해해야 합니다.
*   ⭐ **`Parent` 연산의 효율성**: `parent array`를 사용하는 경우 `Parent(k, T)` 연산은 배열의 직접 접근을 통해 $O(1)$의 시간 복잡도를 가짐을 설명할 수 있어야 합니다.
*   ⭐ **`Leftmost_Child` 연산의 비효율성**: 일반적인 트리에서 `Leftmost_Child(k, T)` 연산은 자식의 순서가 명확하지 않으며, `parent array` 방식으로는 자식 노드를 찾기 위해 전체 배열을 스캔해야 하므로 $O(n)$의 시간 복잡도를 가짐을 설명할 수 있어야 합니다.
*   ⭐ **트리 표현 방식별 장단점**: 이 `parent array` 방식은 부모를 찾는 데는 매우 효율적이지만, 자식을 찾는 데는 비효율적이라는 점을 다른 트리 표현 방식(예: 자식 리스트, 이진 트리 등)과 비교하여 논할 수 있는 능력이 중요합니다.

## Slide 13

### **핵심 개념**

이 슬라이드는 일반 트리(general tree)의 재귀적 정의와 이를 컴퓨터 메모리에 표현하는 한 가지 방법인 **부모 배열(parent array) 표현**을 소개합니다. 슬라이드 제목인 "Simple Approach: Time Complexity"는 이러한 표현 방식이 향후 트리에 대한 연산의 시간 복잡도를 분석하는 데 기초가 됨을 암시합니다.

1.  **트리의 재귀적 정의 (`Create_m`)**:
    *   `Create_m(v, T_1, T_2, ..., T_m)`는 루트 노드 `v`와 `m`개의 서브트리 `T_1, T_2, ..., T_m`를 결합하여 하나의 새로운 트리를 생성하는 함수 또는 연산을 의미합니다. 이는 트리가 재귀적으로 "루트 노드와 그에 연결된 0개 이상의 서브트리 집합"으로 정의될 수 있음을 보여주는 일반적인 방법입니다.

2.  **부모 배열 표현 (Parent Array Representation)**:
    *   트리의 각 노드에 고유한 정수 인덱스(대부분 1부터 시작)를 부여하고, 각 노드의 부모 노드 인덱스를 배열에 저장하여 트리를 표현하는 방식입니다.
    *   `T[]` 배열의 `T[i]`는 인덱스 `i`를 가진 노드의 부모 노드의 인덱스를 저장합니다.
    *   루트 노드의 경우, 부모가 없으므로 특별한 값(예: `0` 또는 `-1`)을 저장하여 루트임을 표시합니다.
    *   `Label[]` 배열은 각 인덱스에 해당하는 노드의 실제 데이터(레이블)를 저장합니다.

### **코드/수식 해설**

트리의 `Create_m` 함수는 개념적으로 다음과 같은 재귀적 구조를 가집니다:

```
// 일반 트리 노드 구조체 (의사 코드)
struct Node {
    ValueType data;
    Node* parent;
    List<Node*> children; // 자식 노드들을 저장하는 리스트
};

// Create_m 함수의 의사 코드
// v: 새로운 루트가 될 노드의 값
// subtrees: v의 자식이 될 서브트리들의 리스트
Node* Create_m(ValueType v_data, List<Node*> subtrees) {
    Node* root_node = new Node(v_data); // 새로운 루트 노드 생성
    root_node->parent = nullptr; // 루트의 부모는 없음

    for (Node* subtree_root : subtrees) {
        root_node->children.add(subtree_root); // 서브트리의 루트를 현재 루트의 자식으로 추가
        subtree_root->parent = root_node;      // 서브트리 루트의 부모를 현재 루트로 설정
    }
    return root_node;
}
```

부모 배열 `P`와 노드 레이블 배열 `L`의 관계는 다음과 같이 수식으로 표현될 수 있습니다:

$P[i] = \text{parent\_index if node } i \text{ is not root}$
$P[i] = 0 \text{ (or a sentinel value like -1) if node } i \text{ is root}$
$L[i] = \text{data\_value of node } i$

여기서 $i$는 노드의 고유 인덱스입니다.

### **구체적 예시**

슬라이드에 제시된 두 개의 서브트리 `T_1`과 `T_2`의 부모 배열 표현을 살펴봅니다. 각 노드 옆의 숫자는 해당 노드의 배열 인덱스입니다.

**예시 1: 서브트리 `T_1` (회색 노드)**

`T_1` 트리의 구조:
*   `a` (인덱스 1) - 루트
    *   `b` (인덱스 2)
        *   `d` (인덱스 3)
        *   `e` (인덱스 6)
        *   `f` (인덱스 4)
    *   `c` (인덱스 5)

이 트리의 부모 배열 `T_1[]`과 레이블 배열 `Label[]`은 다음과 같습니다:

| 인덱스 ($i$) | 1 (a) | 2 (b) | 3 (d) | 4 (f) | 5 (c) | 6 (e) |
| :----------: | :---: | :---: | :---: | :---: | :---: | :---: |
|   `T_1[i]`   |   0   |   1   |   2   |   2   |   1   |   2   |
|  `Label[i]`  |   a   |   b   |   d   |   f   |   c   |   e   |

*   `T_1[1] = 0`: 노드 `a`는 `T_1`의 루트이므로 부모가 0으로 표시됩니다.
*   `T_1[2] = 1`: 노드 `b`의 부모는 노드 `a` (인덱스 1)입니다.
*   `T_1[3] = 2`: 노드 `d`의 부모는 노드 `b` (인덱스 2)입니다.
*   `T_1[4] = 2`: 노드 `f`의 부모는 노드 `b` (인덱스 2)입니다.
*   `T_1[5] = 1`: 노드 `c`의 부모는 노드 `a` (인덱스 1)입니다.
*   `T_1[6] = 2`: 노드 `e`의 부모는 노드 `b` (인덱스 2)입니다.

**예시 2: 서브트리 `T_2` (노란색 노드)**

`T_2` 트리의 구조:
*   `x` (인덱스 1) - 루트
    *   `y` (인덱스 2)
    *   `z` (인덱스 3)

이 트리의 부모 배열 `T_2[]`는 다음과 같습니다:

| 인덱스 ($i$) | 1 (x) | 2 (y) | 3 (z) |
| :----------: | :---: | :---: | :---: |
|   `T_2[i]`   |   0   |   1   |   1   |

*   `T_2[1] = 0`: 노드 `x`는 `T_2`의 루트입니다.
*   `T_2[2] = 1`: 노드 `y`의 부모는 노드 `x` (인덱스 1)입니다.
*   `T_2[3] = 1`: 노드 `z`의 부모는 노드 `x` (인덱스 1)입니다.

### **시험 포인트**

*   ⭐ **일반 트리의 재귀적 정의를 이해하고 설명할 수 있어야 합니다.** (`Create_m` 함수의 의미)
*   ⭐ **부모 배열을 사용한 트리 표현 방식의 원리(각 인덱스가 나타내는 것, 0의 의미 등)를 완벽하게 이해해야 합니다.**
*   ⭐ **주어진 트리 그림으로부터 부모 배열을 정확히 구성하거나, 반대로 부모 배열로부터 트리의 계층 구조를 그려낼 수 있어야 합니다.** (특히, 루트 노드와 자식 노드들의 부모 인덱스 관계를 명확히 파악)
*   이러한 표현 방식의 **장단점** (예: 부모를 찾는 것은 $O(1)$이지만, 자식을 찾는 것은 모든 노드를 순회해야 하므로 $O(N)$의 시간 복잡도를 가질 수 있음)을 논할 수 있어야 합니다.
*   특정 트리 연산(예: 노드의 깊이 계산, 두 노드의 최소 공통 조상 찾기 등)에 대한 **시간/공간 복잡도**를 부모 배열 표현 방식의 관점에서 분석할 수 있어야 합니다. (예: 노드의 깊이 계산은 해당 노드부터 루트까지 부모 배열을 따라 올라가므로 트리의 높이 $h$에 비례하는 $O(h)$의 시간이 소요됩니다.)

### Simple Approach: Time Complexity

**핵심 개념**:
이 슬라이드는 여러 개의 기존 트리를 새로운 루트 노드 `v` 아래에 결합하는 `Create_m(v, T_1, T_2, ..., T_m)` 연산의 시간 복잡도를 다룹니다. 특히, 트리를 배열 기반으로 표현할 때 발생하는 "인덱스 재조정(re-indexing)" 문제를 중심으로 설명합니다.

*   **배열 기반 트리 표현**: 트리의 노드들을 메모리 상의 연속적인 배열에 저장하는 방식입니다. 노드들은 특정 규칙에 따라 배열의 인덱스에 매핑되며, 부모-자식 관계는 이 인덱스를 통해 표현됩니다 (예: 부모 인덱스를 저장하는 배열).
*   **`Create_m` 연산**: 새로운 노드 `v`를 루트로 하고, 기존의 `m`개 트리 $T_1, T_2, ..., T_m$를 `v`의 자식 서브트리로 연결하여 하나의 큰 트리를 만드는 연산입니다.
*   **인덱스 재조정의 필요성**: 배열 기반 표현에서 `T_1`을 제외한 $T_2, ..., T_m$의 노드들은 새로운 통합 트리에서 자신의 고유한 인덱스를 가져야 합니다. 이는 $T_1$의 노드 수와 `v` 노드의 인덱스를 고려하여 $T_2, ..., T_m$의 모든 노드 인덱스를 일괄적으로 변경하고, 해당 노드들의 부모-자식 관계를 나타내는 모든 인덱스 포인터도 업데이트해야 함을 의미합니다. 이 과정이 "Quite hard to handle"하며 시간 복잡도의 주된 원인이 됩니다.

**코드/수식 해설**:
슬라이드에 직접적인 코드는 없지만, 배열 기반 트리의 병합 과정을 추론할 수 있습니다. 각 트리가 부모 인덱스를 저장하는 배열 `parent[]`로 표현된다고 가정합니다.

1.  **초기 트리 배열**:
    *   `T1[]`: `[0, 1, 2, 2, 1, 2]` (인덱스 `1`~`6` 사용. `0`은 루트 `a`의 부모, `1`은 `b`와 `c`의 부모 `a`, `2`는 `d, e, f`의 부모 `b`)
    *   `T2[]`: `[0, 1, 1]` (인덱스 `1`~`3` 사용. `0`은 루트 `x`의 부모, `1`은 `y`와 `z`의 부모 `x`)
    (슬라이드 그림의 배열 값은 노드 자체의 데이터를 나타낼 수도 있으나, 여기서는 부모 인덱스라고 가정하고 설명합니다.)

2.  **`Create_m(v, T_1, T_2)` 연산 및 통합 배열 `T[]` 생성**:
    *   새로운 노드 `v`가 추가되고, `T_1`의 노드들과 `T_2`의 노드들이 하나의 배열 `T[]`에 순서대로 배치됩니다.
    *   그림 하단의 `T[]` 배열은 다음과 같은 인덱스 매핑을 보여줍니다:
        *   `T1`의 노드들은 `T[]`의 인덱스 `1`부터 `6`까지 차지합니다.
        *   새로운 루트 `v`는 `T[]`의 인덱스 `7`을 차지합니다.
        *   `T2`의 노드들은 `T[]`의 인덱스 `8`부터 `10`까지 차지합니다.

    *   **인덱스 및 부모 포인터 업데이트 과정**:
        *   **`v` 노드**: `T[7] = 0` (새로운 전체 트리의 루트이므로 부모 없음).
        *   **`T1` 노드들 (`a`~`f`)**:
            *   `T1`의 루트 `a` (원래 `T1[1]`)는 이제 `v`를 부모로 갖습니다. `v`의 인덱스가 `7`이므로, `T[1] = 7`로 업데이트됩니다.
            *   `T1` 내부의 다른 노드들(예: `b`의 부모 `a`, `d`의 부모 `b` 등)의 인덱스 참조는 상대적인 위치가 변하지 않았으므로, `T[2..6]`의 값은 `[1, 2, 2, 1, 2]`로 유지됩니다.
        *   **`T2` 노드들 (`x`~`z`)**:
            *   `T2`의 노드들은 통합 `T[]` 배열에서 `7`만큼 인덱스가 밀립니다.
                *   `x` (원래 `T2[1]`)는 `8`로, `y` (원래 `T2[2]`)는 `9`로, `z` (원래 `T2[3]`)는 `10`으로 인덱스가 변경됩니다.
            *   `T2`의 루트 `x` (새로운 인덱스 `8`)는 이제 `v`를 부모로 갖습니다. `v`의 인덱스가 `7`이므로, `T[8] = 7`로 업데이트됩니다.
            *   `T2` 내부의 다른 노드들의 부모 포인터도 새로운 인덱스에 맞게 업데이트됩니다:
                *   `y` (새로운 인덱스 `9`)의 부모는 `x` (새로운 인덱스 `8`)이므로, `T[9] = 8`로 업데이트됩니다.
                *   `z` (새로운 인덱스 `10`)의 부모는 `x` (새로운 인덱스 `8`)이므로, `T[10] = 8`로 업데이트됩니다.
    *   이 모든 인덱스 및 포인터 업데이트 작업은 각 `T_i`의 모든 노드를 방문해야 하므로 상당한 비용이 발생합니다.

**구체적 예시**:
당신이 여러 개의 독립된 도서관(각각 $T_1, T_2, ..., T_m$)을 가지고 있다고 상상해 봅시다. 각 도서관은 책들을 고유한 번호(인덱스)로 관리하고 있습니다. 이제 이 모든 도서관을 하나의 거대한 중앙 도서관(새로운 루트 `v`를 가진 트리)으로 통합하려고 합니다.

*   **배열 기반 표현**: 각 도서관의 책 위치 정보가 번호 순서대로 목록(배열)에 저장되어 있습니다. 예를 들어, "1번 책은 2번 책장에 있다" 식으로요.
*   **통합 문제**:
    1.  $T_1$ 도서관의 책들은 기존 번호를 유지하거나 일부만 변경됩니다 (예: $T_1$의 모든 책이 이제 중앙 도서관의 첫 번째 섹션에 속하게 되므로, $T_1$의 "루트 책"은 중앙 도서관의 "루트 책"에 연결됩니다).
    2.  하지만 $T_2$ 도서관의 책들을 가져오면, 기존의 "1번 책"이 이미 $T_1$ 도서관에 있을 수 있으므로 충돌이 발생합니다. 따라서 $T_2$ 도서관의 모든 책 번호(`1, 2, 3...`)를 중앙 도서관 전체 체계에 맞춰 새로운 번호(`8, 9, 10...`)로 바꿔주어야 합니다.
    3.  `T_2` 도서관 내부에서 "1번 책이 2번 책장에 있다"고 되어 있던 정보도 "8번 책이 9번 책장에 있다" 식으로 모두 수정해야 합니다.
*   **시간 복잡도**: 이처럼 모든 책의 번호와 그 번호가 참조되는 모든 정보를 수동으로 수정해야 하므로, 통합하는 도서관의 책이 많을수록 작업 시간은 비례하여 늘어납니다.

**시험 포인트**:
*   ⭐ **`Create_m` 연산의 시간 복잡도**: 배열 기반 트리 표현에서 `Create_m(v, T_1, ..., T_m)` 연산의 시간 복잡도는 $O(N)$입니다. 여기서 $N$은 새로 생성될 전체 트리의 노드 수($1 + \sum N_i$)입니다. 이는 $T_2, ..., T_m$의 모든 노드에 대한 인덱스 재조정 및 관련 포인터 업데이트가 필요하기 때문입니다.
*   ⭐ **링크드 리스트 기반 트리의 시간 복도와 비교**: 만약 트리가 링크드 리스트(포인터) 기반으로 구현되었다면, `Create_m` 연산은 새로운 노드 `v`를 생성하고 `m`개의 포인터를 연결하는 `O(m)` 또는 `O(1)`의 상수 시간 복잡도를 가집니다. 이는 배열 기반 표현과의 중요한 차이점으로, 배열 기반의 단점을 보여줍니다.
*   ⭐ **인덱스 재조정의 구체적인 작업**: $T_2, ..., T_m$의 모든 노드에 대해 수행해야 하는 작업은 다음과 같습니다:
    1.  각 노드의 **고유 인덱스**를 새로운 전체 트리 범위에 맞춰 변경합니다.
    2.  해당 노드의 **부모를 가리키는 포인터(인덱스)**를 변경된 인덱스에 맞춰 업데이트합니다.
    3.  (트리 구현 방식에 따라) 해당 노드의 **자식을 가리키는 포인터(인덱스)**도 업데이트해야 할 수 있습니다.
*   **배열 기반 트리 구현의 장단점**:
    *   **장점**: 메모리 연속성, 캐시 효율성, 인접 노드 접근 시 빠른 속도 (특히 완전 이진 트리의 경우).
    *   **단점**: 삽입/삭제 시 인덱스 변경 및 데이터 이동 비용, 유연하지 못함, 특히 `Create_m`과 같은 병합 연산에서 높은 시간 복잡도.

## Slide 15

**핵심 개념**

*   **List-of-Children Approach (자식 리스트 방식)**: 트리를 표현하는 한 가지 방법으로, 각 노드에 해당 노드의 자식들을 연결 리스트(linked list) 형태로 저장하는 방식입니다. 기존의 부모 포인터만 사용하는 방식(Parent Array 등)과 달리, 부모와 자식 정보를 모두 효율적으로 관리할 수 있도록 고안되었습니다.
*   **Node Array (노드 배열)**: 트리의 모든 노드를 배열의 형태로 저장합니다. 각 배열 엔트리는 노드에 대한 정보를 담으며, 여기에는 노드의 레이블(값), 부모 노드의 인덱스, 그리고 자식 노드들의 연결 리스트 시작 포인터가 포함됩니다.
*   **Parent and Children Information**: 이 방식의 핵심 아이디어는 부모 노드 정보($Pa$)와 자식 노드 정보($Ch$)를 모두 명시적으로 유지하여, 한 노드에서 부모나 자식으로의 접근을 용이하게 하는 것입니다.

**코드/수식 해설**

트리의 노드 정보는 `Node array` 형태로 저장됩니다. 각 노드 `k`에 대해 다음 필드들이 정의됩니다:

*   `k`: 노드의 고유 인덱스 (1부터 시작).
*   `La`: 노드의 레이블 (값). 예시에서는 문자를 사용했습니다.
*   `Pa`: 부모 노드의 인덱스. 루트 노드의 경우 `0` (혹은 `NULL`)으로 표기합니다.
*   `Ch`: 해당 노드의 자식 노드들을 가리키는 연결 리스트의 시작 포인터(혹은 첫 번째 자식 노드의 인덱스). 자식 노드가 없는 리프 노드의 경우 비어 있거나 특별한 값(예: `NULL`)을 가집니다.

$$ \text{Node Array}[k] = \{ \text{Label}(k), \text{Parent}(k), \text{ChildrenListPointer}(k) \} $$

`ChildrenListPointer(k)`는 연결 리스트의 헤드를 가리키며, 이 연결 리스트는 노드 `k`의 모든 자식 노드들의 인덱스를 순서대로 저장합니다. 각 연결 리스트의 노드는 자식 노드의 인덱스와 다음 자식 노드를 가리키는 포인터로 구성됩니다.

**구체적 예시**

슬라이드에 제시된 트리를 살펴보겠습니다.

1.  **트리 구조**:
    *   노드 `1 (a)`: 루트. 자식 `2 (b)`, `5 (c)`.
    *   노드 `2 (b)`: 부모 `1 (a)`. 자식 `3 (d)`, `6 (e)`, `4 (f)`.
    *   노드 `3 (d)`: 부모 `2 (b)`. 자식 없음. (리프)
    *   노드 `4 (f)`: 부모 `2 (b)`. 자식 없음. (리프)
    *   노드 `5 (c)`: 부모 `1 (a)`. 자식 없음. (리프)
    *   노드 `6 (e)`: 부모 `2 (b)`. 자식 없음. (리프)

2.  **Node Array 매핑**:
    *   `k=1 (a)`: `La='a'`, `Pa=0`. `Ch`는 자식 `2 (b)`와 `5 (c)`를 담는 연결 리스트 `[2] -> [5]`를 가리킵니다.
    *   `k=2 (b)`: `La='b'`, `Pa=1`. `Ch`는 자식 `3 (d)`, `6 (e)`, `4 (f)`를 담는 연결 리스트 `[3] -> [6] -> [4]`를 가리킵니다.
    *   `k=3 (d)`: `La='d'`, `Pa=2`. `Ch`는 비어있습니다 (리프 노드).
    *   `k=4 (f)`: `La='f'`, `Pa=2`. `Ch`는 비어있습니다.
    *   `k=5 (c)`: `La='c'`, `Pa=1`. `Ch`는 비어있습니다.
    *   `k=6 (e)`: `La='e'`, `Pa=2`. `Ch`는 비어있습니다.

**시험 포인트**

*   ⭐ **List-of-Children Approach의 구조 이해**: `Node array`와 `Linked Lists of children`의 상호작용 방식과 각 필드(`La`, `Pa`, `Ch`)의 의미를 정확히 이해해야 합니다.
*   ⭐ **장단점**:
    *   **장점**:
        *   특정 노드의 모든 자식을 쉽게 찾을 수 있습니다 (연결 리스트 순회).
        *   부모 노드 정보도 즉시 얻을 수 있습니다 (`Pa` 필드).
        *   노드의 추가/삭제 시 자식 노드 리스트의 변경이 비교적 용이합니다 (연결 리스트의 특성).
    *   **단점**:
        *   자식 노드에 접근하기 위해 연결 리스트를 순회해야 하므로, 특정 순서의 $k$번째 자식을 찾는 데는 $O(k)$ 시간이 걸릴 수 있습니다.
        *   `Node array` 외에 `Linked Lists of children`을 위한 추가적인 메모리 오버헤드가 발생합니다.
*   ⭐ **다른 트리 표현 방식과의 비교**: 이 방식이 Parent array, Left-Child Right-Sibling (LCRS) 등 다른 트리 표현 방식과 어떻게 다른지, 각 방식의 장단점과 시간/공간 복잡도를 비교할 수 있어야 합니다. (예: 특정 노드의 모든 자식 찾기, 부모 찾기, 형제 찾기 등 작업의 복잡도)
*   ⭐ **트리 순회 (Traversal)**: 이 구조에서 DFS, BFS와 같은 트리의 기본 순회 알고리즘이 어떻게 구현될 수 있는지 생각해 볼 필요가 있습니다. `Pa`와 `Ch` 정보를 활용하여 효율적인 순회가 가능합니다.

## Slide 16

**핵심 개념**:
이 슬라이드는 'List-of-Children' (자식 리스트) 방식을 사용하여 여러 트리를 하나의 통합된 자료구조에 효율적으로 저장하고 관리하는 방법을 설명합니다. 특히 `Create2(v, T1, T2)`와 같이 두 개의 기존 트리 `T1`과 `T2`를 새로운 노드 `v`의 자식으로 결합하는 개념을 구현하기 위한 내부적인 데이터 표현 방식을 보여줍니다.

핵심 아이디어는 다음과 같습니다:
*   **단일 `Node array`**: 여러 트리의 모든 노드들을 하나의 배열(Array)에 저장하여 메모리 연속성을 확보하고 노드 관리를 중앙집중화합니다.
*   **노드 정보**: 각 배열의 요소는 하나의 노드에 해당하며, 노드의 값(Label), 부모 노드의 인덱스(Parent), 그리고 자식 노드들의 리스트 시작점(Children list head)을 포함합니다.
*   **자식 리스트**: 각 노드의 자식 노드들은 연결 리스트 형태로 관리됩니다. `Ch` 필드는 이 연결 리스트의 시작을 가리키며, 리스트 내부의 노드들은 자식 노드의 인덱스를 저장하고 다음 자식을 가리킵니다.
*   **트리 결합 준비**: 이러한 방식은 두 개 이상의 트리가 메모리에 분산되지 않고 한 곳에 통합되어 저장되므로, 이들을 하나의 새로운 트리로 결합하는 연산(`Create2`)을 효율적으로 수행할 수 있는 기반을 제공합니다.

**코드/수식 해설**:
슬라이드는 트리를 표현하는 `Node array`의 구조를 보여줍니다. 각 배열 항목은 노드를 나타내며 다음과 같은 필드를 가집니다.

```
Node_Array[k] = { Label: La, Parent: Pa, Children_List_Head: Ch }
```
여기서,
*   `k`: 노드의 고유 인덱스 (1부터 시작). 이 인덱스는 배열의 인덱스와 일치합니다.
*   `La`: 노드에 저장된 실제 값 또는 레이블 (예: 'x', 'a', 'y').
*   `Pa`: 해당 노드의 부모 노드 인덱스. 만약 노드가 트리의 루트라면, `Pa`는 `0` (혹은 `null`)으로 표시됩니다.
*   `Ch`: 해당 노드의 자식 노드 인덱스들을 저장하는 연결 리스트의 시작점(헤드 포인터 또는 인덱스)입니다. 자식이 없는 리프 노드의 경우 이 필드는 비어있거나 `null`을 가리킵니다.

예를 들어, 슬라이드에 제시된 `Node_Array[1]`은 다음과 같이 해석됩니다:
*   노드 인덱스 `k = 1`
*   노드 값 `La = 'x'`
*   부모 인덱스 `Pa = 0` (노드 'x'가 `T1`의 루트임을 의미)
*   자식 리스트 `Ch`는 인덱스 `3`과 `4`를 포함하는 연결 리스트를 가리킵니다. 이는 노드 'x'의 자식이 노드 'y' (인덱스 3)와 노드 'z' (인덱스 4)임을 나타냅니다.

**구체적 예시**:
슬라이드 좌측에는 `T1`과 `T2`라는 두 개의 트리가 있습니다.
*   **`T1`**: 루트 'x' (인덱스 1) 아래에 'y' (인덱스 3)와 'z' (인덱스 4)를 자식으로 가집니다.
*   **`T2`**: 루트 'a' (인덱스 5) 아래에 'b' (인덱스 2)와 'c' (인덱스 6)를 자식으로 가집니다. 'b'는 다시 'd' (인덱스 7)를 자식으로 가집니다.

이 두 트리의 모든 노드 정보는 중앙의 `Node array`에 통합되어 저장됩니다.
*   노드 1 ('x'): `Pa=0`, `Ch`는 `[3] -> [4]` 리스트를 가리킴.
*   노드 2 ('b'): `Pa=5` (노드 'a'의 자식), `Ch`는 `[7]` 리스트를 가리킴.
*   노드 5 ('a'): `Pa=0`, `Ch`는 `[2] -> [6]` 리스트를 가리킴.

이러한 통합된 표현 덕분에, 만약 새로운 연산 `Create2(new_root, T1_root_idx, T2_root_idx)`를 통해 `T1`과 `T2`를 `new_root`의 자식으로 만들고자 한다면, `new_root`의 `Ch` 필드에 `T1_root_idx`와 `T2_root_idx`를 추가하고, `T1_root_idx`와 `T2_root_idx`에 해당하는 노드들의 `Pa` 필드를 `new_root`의 인덱스로 업데이트하기만 하면 됩니다.

**시험 포인트**:
*   ⭐ **List-of-Children 표현 방식의 구조와 특징**: 노드 배열, 부모 포인터, 자식 리스트가 어떻게 상호작용하여 트리를 표현하는지 정확히 이해하고 설명할 수 있어야 합니다. (예: 임의의 노드의 자식을 효율적으로 찾을 수 있지만, 부모를 찾는 것은 `O(1)`이라는 장점).
*   ⭐ **각 필드의 역할**: `k`, `La`, `Pa`, `Ch` 각 필드가 트리의 구조적 정보를 어떻게 나타내는지 명확하게 설명할 수 있어야 합니다. 특히 루트 노드의 `Pa=0`과 `Ch` 필드가 연결 리스트의 헤드를 가리키는 방식에 주목하세요.
*   ⭐ **트리 연산의 구현 원리**: `Create2`와 같은 트리 결합/생성 연산이 이 `Node array` 구조에서 어떻게 `Pa`와 `Ch` 필드를 업데이트하여 구현될 수 있는지 논리적으로 설명할 수 있어야 합니다.
*   ⭐ **이러한 방식의 시간 복잡도 분석**: 특정 노드의 부모 찾기, 자식 찾기, 특정 자식 노드 찾기 등의 연산에 대한 시간 복잡도를 분석할 수 있어야 합니다. (예: 부모 찾기 `O(1)`, 모든 자식 순회 `O(degree)`, 특정 값을 가진 자식 찾기 `O(degree)`).

## Slide 17

이 슬라이드는 `List-of-Children` 접근 방식을 사용하여 트리를 결합하는 `Create2` 연산을 설명합니다.

### **핵심 개념**
*   **List-of-Children 접근 방식 (List-of-Children Approach)**: 이 방식은 각 노드가 자신의 자식 노드들을 연결 리스트 형태로 저장하는 트리 표현 방법입니다. 이를 통해 노드의 차수(degree)에 제한을 두지 않고 유연하게 일반적인(n-ary) 트리를 표현할 수 있습니다. 슬라이드의 `Node array`에서 `Ch` (Children) 컬럼은 각 노드의 자식 노드들의 인덱스를 담은 연결 리스트의 시작 포인터 역할을 합니다.
*   **트리 결합 연산 (`Create2(v, T1, T2)`)**: 주어진 두 개의 독립적인 트리 `T1`과 `T2`를 새로운 노드 `v`의 자식으로 만들어 하나의 큰 트리로 합치는 연산입니다. `List-of-Children` 표현 방식을 사용하면 이 연산을 매우 효율적으로 수행할 수 있습니다.

### **코드/수식 해설**
슬라이드에 명시적인 코드는 없지만, `Node array` 구조를 통해 노드 표현 및 연산 방식을 유추할 수 있습니다. `Node array`는 각 노드를 `k` (Key/Index), `La` (Label/Value), `Pa` (Parent Index), `Ch` (Children List Pointer) 네 가지 필드로 표현합니다.

`Create2(v, T1, T2)` 연산의 동작은 다음과 같습니다:

1.  새로운 노드 `v`를 생성하고, `Node array`에 새로운 엔트리로 추가합니다 (예: 인덱스 9).
2.  `v`는 새로운 트리의 루트가 되므로, `v`의 `Pa` (Parent) 필드를 `0` (또는 `null`)으로 설정합니다.
3.  `v`의 `Ch` (Children) 필드를 `T1`의 루트 노드 인덱스 (예: 1)와 `T2`의 루트 노드 인덱스 (예: 5)를 포함하는 연결 리스트의 시작 포인터로 설정합니다.
4.  `T1`의 루트 노드 (예: `x`, 인덱스 1)의 `Pa` 필드를 `v`의 인덱스 (9)로 변경합니다.
5.  `T2`의 루트 노드 (예: `a`, 인덱스 5)의 `Pa` 필드를 `v`의 인덱스 (9)로 변경합니다.

이러한 단계들은 트리의 크기(`T1`, `T2` 내부의 노드 수)와 관계없이 항상 일정한 수의 노드 필드(새로운 `v` 노드, `T1`의 루트, `T2`의 루트)만 변경하므로, 이 연산의 시간 복잡도는 다음과 같습니다:

$$ O(1) $$

이는 특히 이진 트리를 결합하는 경우에 해당하며, 다른 복잡한 트리 구조라도 핵심적으로 변경되는 노드의 수는 상수이기 때문에 일반적으로 $O(1)$로 간주됩니다.

### **구체적 예시**
슬라이드의 그림은 `Create2(v, T1, T2)` 연산이 완료된 후의 트리 구조와 `Node array` 상태를 보여줍니다.

*   원래 `T1`은 루트 `x` (k=1)를 가지며 자식 `y` (k=3), `z` (k=4)를 가지고, `T2`는 루트 `a` (k=5)를 가지며 자식 `b` (k=2), `c` (k=6)를 가집니다 (여기서 `b`는 자식 `d` (k=7)를 가집니다).
*   `Create2(v, T1, T2)` 연산으로 새로운 노드 `v` (k=9)가 생성됩니다.
*   `Node array`를 보면, `v` (k=9)의 `Pa`는 `0`으로 설정되어 전체 트리의 루트임을 나타냅니다.
*   `v`의 `Ch`는 `1` (노드 `x`)과 `5` (노드 `a`)를 가리키는 연결 리스트로 설정되어 `x`와 `a`가 `v`의 자식임을 보여줍니다.
*   `x` (k=1)의 `Pa`는 `9` (노드 `v`)로 변경되었고, `a` (k=5)의 `Pa`도 `9` (노드 `v`)로 변경되었습니다. 이는 `x`와 `a`가 이제 `v`의 직접적인 자식임을 의미합니다.
*   `x`와 `a` 아래의 서브트리 구조 (예: `x`의 자식으로 `y`, `z`가 있고, `a`의 자식으로 `b`, `c`가 있는 등)는 `Pa` 필드만 변경되었을 뿐, 내부적으로는 그대로 유지됩니다.

### **시험 포인트**
*   ⭐ **`Create2` 연산의 시간 복잡도**: `List-of-Children` 접근 방식에서 두 트리를 결합하는 `Create2` 연산의 시간 복잡도가 $O(1)$인 이유를 정확히 설명할 수 있어야 합니다. 이는 트리의 크기와 무관하게 항상 일정한 수의 포인터만 변경하기 때문입니다.
*   ⭐ **`List-of-Children` 표현의 장점**: 이 방식이 이진 트리가 아닌 일반적인 N-ary 트리를 유연하게 표현할 수 있다는 점과 그 이유(각 노드의 자식 수에 제한이 없음)를 이해해야 합니다.
*   ⭐ **Node array의 각 필드 역할**: `k` (Key), `La` (Label), `Pa` (Parent), `Ch` (Children) 각 필드가 노드 정보에서 어떤 역할을 하는지 명확히 알아야 합니다. 특히 `Pa`와 `Ch` 필드가 부모-자식 관계를 어떻게 나타내는지 중요합니다.
*   ⭐ **트리 결합 시 변경되는 사항**: `Create2` 연산 수행 시, 어떤 노드(새로운 루트 노드, 결합되는 서브트리 루트 노드)의 어떤 필드(`Pa`, `Ch`)가 변경되는지 구체적으로 설명할 수 있어야 합니다.

**핵심 개념**

이 슬라이드는 **List-of-Children Approach** (자식 목록 방식)를 사용하여 트리를 배열로 표현할 때, 여러 트리를 결합하는 과정에서 발생할 수 있는 비효율성을 설명합니다. 특히, 각 트리가 별개의 배열에 저장되어 있을 경우, 이들을 하나의 통합된 배열로 만들려고 할 때 인덱스를 재조정해야 하는 문제점을 지적합니다.

*   **List-of-Children (배열 기반)**: 각 노드를 배열의 한 요소로 표현하며, 각 노드의 자식들은 별도의 연결 리스트(또는 다른 보조 자료구조)를 통해 해당 노드의 인덱스(또는 포인터)로 참조됩니다. 슬라이드에서는 `La` (Label/Value), `Pa` (Parent Index), `Ch` (Children List Header/Pointer) 세 가지 필드를 사용하여 노드를 표현합니다.
*   **트리 결합의 비효율성**: 만약 여러 트리가 각각 독립적인 배열 인덱스를 사용하여 표현되어 있다면, 이들을 하나의 큰 배열로 합칠 때 두 번째 트리의 노드들부터는 새로운 인덱스를 할당받아야 합니다. 이 과정에서 기존의 부모-자식 관계를 나타내는 인덱스들을 모두 재조정해야 하므로 $O(n)$의 시간 복잡도가 발생합니다.

**코드/수식 해설**

슬라이드에 제시된 트리 표현 방식은 다음과 같은 테이블 구조를 사용합니다 (개념적인 표현):

| k (Index) | La (Label) | Pa (Parent Index) | Ch (Children Link) |
| :-------- | :--------- | :---------------- | :----------------- |
| 1         | x          | /                 | (Link to 2) -> (Link to 3) |
| 2         | y          | 1                 | /                  |
| 3         | z          | 1                 | /                  |

여기서 `k`는 배열 인덱스를 나타내고, `La`는 노드의 값을, `Pa`는 부모 노드의 인덱스를, `Ch`는 자식 노드들의 인덱스를 가리키는 연결 리스트의 시작 포인터(또는 첫 번째 자식의 인덱스)를 나타냅니다.

트리 결합 시 인덱스 재조정으로 인한 시간 복잡도는 다음과 같습니다.
$$O(n)$$
여기서 $n$은 결합되는 전체 노드의 수를 의미합니다. 두 번째 (그리고 이후) 트리의 모든 노드에 대해 새로운 인덱스를 계산하고, 그 노드의 부모-자식 관계 인덱스들을 업데이트해야 하므로 $n$에 비례하는 시간이 소요됩니다.

**구체적 예시**

슬라이드에는 두 개의 트리가 각각 독립적인 List-of-Children 방식으로 표현되어 있습니다.

**예시 1: $T_1$**
*   트리 구조: `x`가 루트, `y`와 `z`가 `x`의 자식.
*   배열 표현:
    *   `k=1`: `La=x`, `Pa=/` (루트), `Ch`는 `2`와 `3`을 가리키는 리스트.
    *   `k=2`: `La=y`, `Pa=1`, `Ch=/`.
    *   `k=3`: `La=z`, `Pa=1`, `Ch=/`.

**예시 2: $T_2$**
*   트리 구조: `a`가 루트, `b`와 `c`가 `a`의 자식, `d`가 `b`의 자식.
*   배열 표현:
    *   `k=1`: `La=a`, `Pa=/` (루트), `Ch`는 `2`와 `3`을 가리키는 리스트.
    *   `k=2`: `La=b`, `Pa=1`, `Ch`는 `4`를 가리키는 리스트.
    *   `k=3`: `La=c`, `Pa=1`, `Ch=/`.
    *   `k=4`: `La=d`, `Pa=2`, `Ch=/`.

만약 $T_1$과 $T_2$를 하나의 배열로 결합한다고 가정해 봅시다. $T_1$이 인덱스 1-3을 사용하고, $T_2$가 뒤이어 배치된다면, $T_2$의 노드들은 이제 인덱스 4부터 시작해야 합니다.
*   `T_2`의 'a' (원래 인덱스 1)는 새 인덱스 4가 됩니다.
*   `T_2`의 'b' (원래 인덱스 2)는 새 인덱스 5가 됩니다.
*   `T_2`의 'c' (원래 인덱스 3)는 새 인덱스 6이 됩니다.
*   `T_2`의 'd' (원래 인덱스 4)는 새 인덱스 7이 됩니다.

이때, 'a'의 자식 목록은 (4, 5)가 아닌 (5, 6)으로, 'b'의 자식 목록은 (7)로, 'b', 'c'의 부모는 (4)로, 'd'의 부모는 (5)로 모두 변경되어야 합니다. 이처럼 **모든 노드의 부모 및 자식 인덱스 정보를 재계산하고 업데이트하는 작업**이 필요하며, 이것이 $O(n)$의 비용을 발생시킵니다.

**시험 포인트**

*   ⭐ **List-of-Children 배열 기반 표현 방식의 이해**: 각 노드 필드(`La`, `Pa`, `Ch`)의 역할과 노드 인덱스 간의 관계를 설명할 수 있어야 합니다.
*   ⭐ **복수 트리 결합 시 발생하는 문제점**: 각 트리가 독립적인 배열에 저장될 때, 이들을 하나의 배열로 결합하는 과정에서 왜 비효율적인지, 특히 인덱스 재조정이 왜 필요한지 이해해야 합니다.
*   ⭐ **시간 복잡도**: 트리 결합 시 발생하는 인덱스 재조정 작업의 시간 복잡도가 $O(n)$인 이유를 설명할 수 있어야 합니다. 여기서 $n$은 결합될 총 노드의 수입니다.

- **핵심 개념**:
`List-of-Children Approach`는 트리를 표현하는 한 가지 방법으로, 각 노드가 자신의 자식 노드들을 가리키는 포인터 리스트(일반적으로 연결 리스트)를 유지합니다. 슬라이드는 이 접근 방식이 가지는 두 가지 주요 문제를 지적합니다.

1.  **우측 형제(right sibling) 접근의 어려움**: 특정 노드(`y`)의 바로 다음 형제 노드(`z`)에 접근하기 위해서는 해당 노드의 부모 노드(`x`)를 먼저 찾고, 그 부모 노드의 자식 리스트를 탐색하여 `y`를 찾은 후 `y`의 다음 원소를 확인해야 합니다. 이는 부모의 자식 수에 비례하는 시간 복잡도를 가질 수 있습니다.
2.  **노드 정보의 중복 저장**: 각 자식 노드에 대한 정보(혹은 참조)가 두 군데에 저장됩니다. 하나는 `node array`와 같은 주 노드 정보 저장 공간이고, 다른 하나는 해당 노드의 부모가 관리하는 자식 연결 리스트입니다. 이는 메모리 사용의 비효율성을 초래할 수 있습니다.

- **코드/수식 해설**:
  슬라이드에는 직접적인 코드는 없지만, `List-of-Children` 구조를 표현하는 의사 코드는 다음과 같을 수 있습니다.

  ```c
  struct Node {
      char data;           // 노드의 데이터 (e.g., 'x', 'y', 'z')
      int parent_idx;      // 부모 노드의 인덱스 (Pa)
      struct ChildNode* children_head; // 자식 노드 리스트의 시작을 가리키는 포인터 (Ch)
  };

  struct ChildNode {
      int child_idx;       // 자식 노드의 인덱스
      struct ChildNode* next; // 다음 자식 노드 포인터
  };

  // 노드 정보를 저장하는 배열
  struct Node node_array[MAX_NODES];
  ```
  위 구조에서 `node_array[1]`은 노드 `x`의 정보를 담고, `node_array[1].children_head`는 `y`와 `z`를 가리키는 `ChildNode` 연결 리스트의 시작을 가리킵니다. `ChildNode` 내부의 `child_idx`는 실제 노드 정보를 담고 있는 `node_array`의 인덱스를 참조합니다. 예를 들어, `node_array[1].children_head`가 가리키는 `ChildNode`는 `child_idx=2`를 가지고 `node_array[2]` (`y`)를 참조하고, 그 `next`는 `child_idx=3`을 가지는 `ChildNode` (`node_array[3]` - `z`)를 참조합니다.

- **구체적 예시**:
  트리 `T1`에서 노드 `y`의 우측 형제 `z`를 찾는 과정을 예로 들어봅시다.
  1.  노드 `y`의 정보를 통해 `parent_idx` (부모 `x`의 인덱스 `1`)를 얻습니다.
  2.  `node_array[1]`(`x`)에 접근하여 `x`의 `children_head` 포인터를 통해 자식 연결 리스트를 시작합니다.
  3.  연결 리스트를 순차적으로 탐색하며 각 `ChildNode`의 `child_idx`가 `y`의 인덱스(`2`)와 일치하는 노드를 찾습니다.
  4.  `y`를 나타내는 `ChildNode`를 찾으면, 그 `next` 포인터를 통해 `z`를 나타내는 `ChildNode` (인덱스 `3`)에 접근할 수 있습니다.
  이 과정에서 `y`를 찾기 위해 연결 리스트를 순차적으로 탐색해야 하므로, `x`의 자식 수($$degree(x)$$)에 비례하는 시간이 소요될 수 있습니다.

- **시험 포인트**:
  *   ⭐ `List-of-Children` 방식의 트리가 주어졌을 때, 특정 노드의 우측 형제를 찾는 `시간 복잡도`를 분석하고 설명할 수 있어야 합니다. (최악의 경우 부모의 자식 수, 즉 $O(degree(parent))$).
  *   ⭐ 노드 정보 중복으로 인한 `공간 복잡도`의 비효율성을 설명하고, 어떤 정보가 어떻게 중복되는지 예시를 들어 설명할 수 있어야 합니다. (자식 노드의 참조가 `node array`와 부모의 `children_head` 연결 리스트에 중복).
  *   ⭐ 이 슬라이드에서 제시된 `List-of-Children` 방식의 단점들을 다른 트리 표현 방식(예: `Parent Array`, `First Child/Next Sibling`)의 장점과 비교하여 설명하는 문제가 나올 수 있습니다.

### 핵심 개념
Left-Child/Right-Sibling (LCRS) Approach는 자식의 수가 정해지지 않은 **일반 트리(General Tree)**를 **이진 트리(Binary Tree)** 형태로 효율적으로 표현하는 방법입니다. 각 노드는 오직 두 개의 포인터만을 저장합니다:
1.  **Left Child (왼쪽 자식)**: 해당 노드의 가장 왼쪽 자식을 가리킵니다. (부모-자식 관계)
2.  **Right Sibling (오른쪽 형제)**: 해당 노드와 같은 부모를 가지는 바로 다음 오른쪽 형제 노드를 가리킵니다. (형제 관계)

이 방식을 통해, 노드가 가질 수 있는 자식의 수에 관계없이 항상 고정된 수의 포인터(두 개)만 유지하여 메모리 사용을 효율적으로 할 수 있습니다.

### 코드/수식 해설
이 슬라이드에는 별도의 수식이 없으므로, LCRS 표현을 위한 노드 구조를 C/C++ 언어의 `struct` 형태로 제시합니다.

```c
// LCRS Approach를 위한 트리 노드 구조체 정의
typedef struct Node {
    DataType data;           // 노드가 저장할 데이터 (예: int, char 등)
    struct Node* leftChild;  // 현재 노드의 가장 왼쪽 자식 노드를 가리키는 포인터
    struct Node* rightSibling; // 현재 노드의 바로 다음 오른쪽 형제 노드를 가리키는 포인터
} Node;
```

위 구조에서 `leftChild` 포인터는 수직적으로 아래 방향(부모 -> 첫 번째 자식)의 연결을 나타내며, `rightSibling` 포인터는 수평적으로 옆 방향(형제 -> 형제)의 연결을 나타냅니다.

### 구체적 예시
슬라이드의 그림을 통해 LCRS 변환 과정을 이해할 수 있습니다.

*   **변환 전 (왼쪽 그림)**: 일반 트리입니다.
    *   루트 노드는 세 개의 자식을 가집니다.
    *   루트의 왼쪽 자식은 두 개의 자식을 가집니다.
*   **변환 후 (오른쪽 그림)**: LCRS 방식으로 표현된 트리입니다.
    *   **빨간색 선**: `leftChild` 포인터를 나타냅니다. 각 부모는 자신의 가장 왼쪽 자식에게만 빨간색 화살표를 보냅니다.
        *   루트 노드 -> 첫 번째 자식
        *   첫 번째 자식 -> 그 자식의 첫 번째 자식
    *   **초록색 선**: `rightSibling` 포인터를 나타냅니다. 각 노드는 자신의 바로 다음 오른쪽 형제에게만 초록색 화살표를 보냅니다.
        *   루트의 첫 번째 자식 -> 루트의 두 번째 자식
        *   루트의 두 번째 자식 -> 루트의 세 번째 자식
        *   첫 번째 자식의 첫 번째 자식 -> 그 자식의 두 번째 자식

이러한 방식으로 모든 노드는 오직 하나의 `leftChild` 포인터와 하나의 `rightSibling` 포인터만 가짐으로써, 일반 트리가 이진 트리와 유사한 구조로 변환됩니다. 모든 자식은 `leftChild`를 통해 가장 왼쪽 자식에게 도달한 후, `rightSibling` 체인을 따라 탐색할 수 있습니다.

**실생활 비유**:

## Slide 21

**핵심 개념**
*   **Left-Child/Right-Sibling (LCRS) 접근 방식**: 트리를 표현하는 한 가지 방법으로, 각 노드가 자신의 **부모 (parent)**, **가장 왼쪽 자식 (leftmost child)**, 그리고 **오른쪽 형제 (right sibling)**에 대한 포인터(또는 배열 인덱스)를 유지합니다.
*   **정적(Static) 구현**: 이 슬라이드에서는 LCRS 접근 방식을 배열(array)을 사용하여 정적으로 구현하는 방법을 설명합니다. 각 노드는 배열의 특정 인덱스에 저장되며, 포인터들은 해당 노드의 배열 인덱스로 표현됩니다.
*   **구조체 필드**: 각 노드는 일반적으로 다음 정보를 저장합니다.
    *   `Lc` (Leftmost Child): 가장 왼쪽 자식 노드의 인덱스. 자식이 없으면 null 또는 특정 값(예: 0, -1)으로 표시됩니다.
    *   `La` (Label/Value): 노드 자체의 데이터(값).
    *   `Pa` (Parent): 부모 노드의 인덱스. 루트 노드는 null로 표시됩니다.
    *   `Rs` (Right Sibling): 바로 오른쪽 형제 노드의 인덱스. 오른쪽 형제가 없으면 null로 표시됩니다.
*   **효율성**: 이 방법은 자식의 수가 가변적이거나 많은 일반 트리(N-ary tree)에 대해 효율적입니다. 각 노드가 고정된 수의 포인터(`Lc`, `Pa`, `Rs`)만 저장하므로, 모든 자식 포인터를 미리 배열로 할당할 필요가 없어 메모리 낭비를 줄일 수 있습니다. 자식들을 순회할 때는 `Lc`를 따라간 후 `Rs`들을 순차적으로 따라가게 됩니다.

**코드/수식 해설**

슬라이드에 제시된 표는 LCRS 접근 방식의 정적 배열 구현을 보여줍니다. 각 행은 하나의 노드를 나타내며, `k`는 해당 노드의 배열 인덱스입니다.

다음은 표의 각 열에 대한 설명입니다:

*   `k`: 배열 인덱스. 각 노드는 이 인덱스에 저장됩니다. (예: `a`는 인덱스 3, `b`는 인덱스 2)
*   `Lc` (Left Child): 해당 노드의 가장 왼쪽 자식 노드의 배열 인덱스. 자식이 없는 리프 노드는 빈 값(`/`)으로 표시됩니다.
*   `La` (Label/Value): 해당 노드에 저장된 실제 값 또는 레이블. (예: `k=3`은 `a` 값을 가짐)
*   `Pa` (Parent): 해당 노드의 부모 노드의 배열 인덱스. 루트 노드(`k=3(a)`)는 부모가 없으므로 빈 값(`/`)으로 표시됩니다.
*   `Rs` (Right Sibling): 해당 노드의 바로 오른쪽 형제 노드의 배열 인덱스. 오른쪽 형제가 없는 노드(`k=1(f)`, `k=3(a)`, `k=6(c)`)는 빈 값(`/`)으로 표시됩니다.

예시 테이블 (`T root`는 `k=3`을 가리킵니다):

| k | Lc | La | Pa | Rs |
|---|----|----|----|----|
| 1 | /  | f  | 2  | /  |
| 2 | 7  | b  | 3  | 6  |
| 3 | 2  | a  | /  | /  |
| 4 | /  | /  | /  | /  |
| 5 | /  | e  | 2  | 1  |
| 6 | /  | c  | 3  | /  |
| 7 | /  | d  | 2  | 5  |

**구체적 예시**

슬라이드의 `Tree T`와 배열 표를 통해 LCRS가 어떻게 트리를 표현하는지 살펴봅시다.

1.  **루트 노드 `a` (인덱스 3)의 정보**:
    *   `La` 필드는 `a`입니다.
    *   `Pa` 필드는 `/` (부모가 없으므로)입니다.
    *   `Lc` 필드는 `2`입니다. 이는 `a`의 가장 왼쪽 자식이 배열 인덱스 `2`에 있는 노드(`b`)임을 의미합니다.
    *   `Rs` 필드는 `/` (오른쪽 형제가 없으므로)입니다.

2.  **노드 `b` (인덱스 2)의 자식들을 탐색하는 경우**:
    *   `b`의 `Lc` 필드는 `7`입니다. 따라서 `b`의 가장 왼쪽 자식은 배열 인덱스 `7`에 있는 노드(`d`)입니다.
    *   이제 `d`의 형제를 찾으려면 `d`의 `Rs` 필드를 확인합니다. `d`의 `Rs`는 `5`입니다. 이는 `d`의 오른쪽 형제가 인덱스 `5`에 있는 노드(`e`)임을 의미합니다.
    *   `e`의 `Rs`는 `1`입니다. 이는 `e`의 오른쪽 형제가 인덱스 `1`에 있는 노드(`f`)임을 의미합니다.
    *   `f`의 `Rs`는 `/`입니다. 이는 `f`가 `b`의 자식들 중 가장 오른쪽에 있는 노드임을 의미합니다.
    *   이처럼 `Lc`와 `Rs` 포인터를 순차적으로 따라가면 한 부모의 모든 자식들을 효율적으로 순회할 수 있습니다.

**시험 포인트**

*   ⭐ **LCRS 접근 방식의 원리 이해**: 각 노드가 부모, 가장 왼쪽 자식, 오른쪽 형제 포인터(또는 인덱스)를 어떻게 사용하여 트리 구조를 나타내는지 정확히 이해해야 합니다.
*   ⭐ **장점**: 이 구현이 다수의 자식을 가질 수 있는 일반 트리에서 왜 효율적인지 (고정된 메모리 오버헤드, 모든 자식 포인터를 저장할 필요 없음) 설명할 수 있어야 합니다.
*   ⭐ **배열-기반 LCRS 표 해석 및 구성**: 주어진 트리 구조를 보고 LCRS 배열 표를 직접 채우거나, 반대로 주어진 배열 표를 보고 트리 구조를 그릴 수 있어야 합니다. 특히 `Lc`, `Pa`, `Rs`의 의미와 null 값 처리 방법을 명확히 이해해야 합니다.
*   ⭐ **트리 순회**: LCRS 표현에서 특정 노드의 모든 자식 노드들을 순회하는 방법 (Lc를 따라간 후 Rs들을 순차적으로 따라가는 방식)을 설명하고, 필요한 경우 가상 코드(pseudocode)로 작성할 수 있어야 합니다.
*   ⭐ LCRS가 '정적' 구현이라는 의미를 파악해야 합니다. 이는 배열 크기가 고정되어 있음을 의미하며, 동적 할당 방식과 대비될 수 있습니다.

## Left-Child/Right-Sibling 접근 방식: 동적 구현

### 핵심 개념
*   **정적 Left-Child/Right-Sibling 방식의 한계**: 기존에 배열(array)을 사용하여 트리의 노드들을 저장하는 정적(static) Left-Child/Right-Sibling 방식은 트리의 크기가 가변적일 때 고정된(fixed) 공간 할당으로 인해 비효율적이거나 확장이 어렵다는 문제가 있었습니다. 즉, 동적인 공간 할당(dynamic space allocation)이 불가능하다는 단점이 있습니다.
*   **동적 Left-Child/Right-Sibling 방식 도입**: 이러한 정적 방식의 고정 공간 할당 문제를 해결하기 위해, 연결 리스트(linked list)의 아이디어를 차용하여 노드들을 동적으로 연결하는 동적(dynamic) Left-Child/Right-Sibling 방식이 제안됩니다. 이 방식에서는 각 노드가 자신의 데이터와 함께 '가장 왼쪽 자식 노드(leftChild)'를 가리키는 포인터와 '오른쪽에 위치한 형제 노드(rightSibling)'를 가리키는 포인터를 가집니다. 이를 통해 필요한 만큼만 메모리를 할당하여 트리를 유연하게 구성하고 변경할 수 있습니다.

### 코드/수식 해설
각 노드는 다음과 같은 구조체로 표현될 수 있습니다.

```c
// C 언어 기준 Node 구조체 정의
struct Node {
    char data;                 // 노드의 데이터 값 (예: 'a', 'b')
    struct Node* leftChild;    // 자신의 가장 왼쪽 자식 노드를 가리키는 포인터
    struct Node* rightSibling; // 자신의 바로 오른쪽 형제 노드를 가리키는 포인터
};

// 트리의 루트 노드를 가리키는 포인터
struct Node* T;
```

이 방식에서 트리의 노드 개수가 $N$개일 때, 각 노드는 고정된 수의 포인터(일반적으로 2개)와 데이터를 저장하므로, 전체 공간 복잡도는 $O(N)$입니다.

### 구체적 예시
슬라이드에 제시된 트리를 동적 Left-Child/Right-Sibling 방식으로 표현한 예시를 살펴보겠습니다.

*   **원래 트리 구조**:
    ```
        a
       / \
      b   c
     /|\
    d e f
    ```
*   **동적 연결 표현**:
    *   `T` (루트 포인터)는 노드 `'a'`를 가리킵니다.
    *   **노드 `'a'`**: `data = 'a'`
        *   `leftChild`는 노드 `'b'`를 가리킵니다. (`'a'`의 가장 왼쪽 자식)
        *   `rightSibling`은 `NULL`입니다. (`'a'`는 루트이므로 형제가 없습니다.)
    *   **노드 `'b'`**: `data = 'b'`
        *   `leftChild`는 노드 `'d'`를 가리킵니다. (`'b'`의 가장 왼쪽 자식)
        *   `rightSibling`은 노드 `'c'`를 가리킵니다. (`'b'`의 바로 오른쪽 형제)
    *   **노드 `'c'`**: `data = 'c'`
        *   `leftChild`는 `NULL`입니다. (`'c'`는 자식이 없습니다.)
        *   `rightSibling`은 `NULL`입니다. (`'c'`는 `'b'`의 가장 오른쪽 형제이므로 더 이상 형제가 없습니다.)
    *   **노드 `'d'`**: `data = 'd'`
        *   `leftChild`는 `NULL`입니다. (`'d'`는 자식이 없습니다.)
        *   `rightSibling`은 노드 `'e'`를 가리킵니다. (`'d'`의 바로 오른쪽 형제)
    *   **노드 `'e'`**: `data = 'e'`
        *   `leftChild`는 `NULL`입니다. (`'e'`는 자식이 없습니다.)
        *   `rightSibling`은 노드 `'f'`를 가리킵니다. (`'e'`의 바로 오른쪽 형제)
    *   **노드 `'f'`**: `data = 'f'`
        *   `leftChild`는 `NULL`입니다. (`'f'`는 자식이 없습니다.)
        *   `rightSibling`은 `NULL`입니다. (`'f'`는 `'e'`의 가장 오른쪽 형제이므로 더 이상 형제가 없습니다.)

이처럼 각 노드는 오직 두 개의 포인터(`leftChild`, `rightSibling`)만으로 전체 트리의 모든 관계를 표현할 수 있으며, 이는 임의의 다진 트리(general tree)를 이진 트리(binary tree) 형태로 논리적으로 변환하여 표현하는 효과를 가집니다.

### 시험 포인트
*   ⭐ **정적 vs. 동적 구현의 차이점**: 정적 Left-Child/Right-Sibling 방식이 배열을 사용하여 고정된 공간을 할당하는 반면, 동적 방식은 포인터를 사용하여 노드를 연결함으로써 동적인 공간 할당과 유연한 트리 구조 변경이 가능하다는 핵심 차이점을 이해해야 합니다.
*   ⭐ **각 포인터의 역할**: `leftChild` 포인터가 현재 노드의 '가장 왼쪽 자식'을, `rightSibling` 포인터가 현재 노드의 '바로 오른쪽 형제'를 가리킨다는 것을 정확히 알고 있어야 합니다. 이 두 포인터만으로 트리의 모든 관계를 표현할 수 있다는 점이 중요합니다.
*   ⭐ **일반 트리의 이진 트리 표현**: 이 Left-Child/Right-Sibling 방식은 임의의 다진 트리(general tree)를 두 개의 포인터만을 사용하여 이진 트리(binary tree) 형태로 표현하는 일반적인 방법입니다. `leftChild`는 이진 트리의 왼쪽 자식, `rightSibling`은 이진 트리의 오른쪽 자식에 해당한다고 생각할 수 있습니다.
*   ⭐ **장단점**: 동적 구현의 장점은 트리의 크기가 동적으로 변할 때 효율적이고, 노드 추가/삭제가 용이하다는 것입니다. 단점으로는 부모 노드를 직접 찾아가는 것이 어렵고 (parent pointer가 없는 경우), 특정 자식 노드를 찾기 위해 `rightSibling` 체인을 따라가야 하므로 탐색 시간이 길어질 수 있으며, 각 노드마다 포인터를 저장하기 위한 추가적인 메모리 오버헤드가 발생할 수 있습니다.

## Slide 23

**핵심 개념**:

*   **k-ary 트리**: 각 노드가 최대 $k$개의 자식을 가질 수 있는 일반적인 트리를 말합니다. 예를 들어, 이진 트리는 2-ary 트리의 한 종류입니다.
*   **Left-Child/Right-Sibling (LCRS) 표현**: 일반적인 $k$-ary 트리를 이진 트리(Binary Tree) 형태로 표현하는 효율적인 방법입니다. 이 방식을 사용하면, 각 노드는 항상 최대 두 개의 포인터만 가지게 되어 이진 트리의 구조를 활용하여 일반 트리를 구현할 수 있습니다.
    *   **Left-Child**: 원래 트리의 특정 노드의 **첫 번째 자식 노드**를 가리킵니다.
    *   **Right-Sibling**: 원래 트리의 특정 노드와 같은 부모를 가진 **다음 형제 노드**를 가리킵니다.
*   **LCRS의 장점**:
    1.  노드가 가질 수 있는 자식 수에 대한 사전 지식이 필요 없어집니다 (항상 최대 2개의 포인터만 관리).
    2.  이진 트리와 동일한 노드 구조를 사용하므로 코딩이 단순해집니다.

**코드/수식 해설**:

일반적인 트리 노드는 자식 노드를 가리키는 포인터들의 배열이나 리스트를 가질 수 있습니다. 예를 들어, 다음과 같습니다:

```c
// 일반적인 k-ary 트리 노드 구조
struct GeneralTreeNode {
    int data;
    struct GeneralTreeNode* children[MAX_K]; // MAX_K는 k의 최댓값
    int num_children; // 현재 자식 수
};
```

하지만 Left-Child/Right-Sibling 접근법을 사용하면, 노드 구조가 이진 트리와 동일하게 두 개의 포인터만을 가지도록 변경됩니다.

```c
// Left-Child/Right-Sibling (LCRS) 표현 노드 구조
struct LCRSTreeNode {
    int data;
    struct LCRSTreeNode* left_child;   // 첫 번째 자식 노드를 가리킴 (빨간 선)
    struct LCRSTreeNode* right_sibling; // 다음 형제 노드를 가리킴 (녹색 선)
};
```

이 변환을 통해, 원래 트리의 부모-첫 번째 자식 관계는 `left_child` 포인터로, 같은 부모를 가진 형제 관계는 `right_sibling` 포인터로 대체됩니다.

공간 복잡도 측면에서, $N$개의 노드를 가진 트리를 가정할 때:
*   `GeneralTreeNode` 방식은 각 노드가 최대 $MAX\_K$개의 포인터를 가질 수 있으므로, 최악의 경우 $N \times MAX\_K \times \text{pointer size}$의 공간이 필요할 수 있습니다 (실제로는 `num_children`에 따라 달라짐).
*   `LCRSTreeNode` 방식은 각 노드가 항상 2개의 포인터만을 가지므로, $N \times 2 \times \text{pointer size}$의 공간이 필요합니다.
이는 $MAX\_K$가 2보다 큰 경우 (대부분의 일반 트리) LCRS 방식이 더 효율적인 고정된 공간을 사용함을 의미합니다.

**구체적 예시**:

슬라이드의 원본 트리와 Left-Child/Right-Sibling 변환 규칙을 적용한 이진 트리를 분석해봅시다.

**원본 트리 (General Tree):**
```
      a
     / \
    b   c
   /|\
  d e f
```

**Left-Child/Right-Sibling (LCRS) 변환 규칙:**
*   **Left-Child (빨간 선)**: 노드의 첫 번째 자식을 가리킵니다.
*   **Right-Sibling (녹색 선)**: 노드의 다음 형제를 가리킵니다.

이 규칙에 따라 원본 트리를 변환하면 다음과 같습니다:

1.  **노드 `a`**:
    *   `a`의 첫 번째 자식은 `b`입니다. $\rightarrow$ `a.left_child` = `b` (슬라이드 그림의 빨간 선)
    *   `a`의 다음 형제는 없습니다. $\rightarrow$ `a.right_sibling` = `NULL`
2.  **노드 `b`**:
    *   `b`의 첫 번째 자식은 `d`입니다. $\rightarrow$ `b.left_child` = `d` (슬라이드 그림의 빨간 선)
    *   `b`의 다음 형제는 `c`입니다. $\rightarrow$ `b.right_sibling` = `c` (슬라이드 그림의 녹색 선)
3.  **노드 `c`**:
    *   `c`는 자식이 없습니다. $\rightarrow$ `c.left_child` = `NULL`
    *   `c`는 다음 형제가 없습니다. $\rightarrow$ `c.right_sibling` = `NULL`
4.  **노드 `d`**:
    *   `d`는 자식이 없습니다. $\rightarrow$ `d.left_child` = `NULL`
    *   `d`의 다음 형제는 `e`입니다. $\rightarrow$ `d.right_sibling` = `e`
5.  **노드 `e`**:
    *   `e`는 자식이 없습니다. $\rightarrow$ `e.left_child` = `NULL`
    *   `e`의 다음 형제는 `f`입니다. $\rightarrow$ `e.right_sibling` = `f`
6.  **노드 `f`**:
    *   `f`는 자식이 없습니다. $\rightarrow$ `f.left_child` = `NULL`
    *   `f`는 다음 형제가 없습니다. $\rightarrow$ `f.right_sibling` = `NULL`

**⭐주의사항**: 슬라이드에 제시된 그림의 오른쪽 이진 트리는 `d`의 `left_child`가 `e`를, `e`의 `left_child`가 `f`를 가리키는 것으로 표현되어 있습니다. 하지만 표준 Left-Child/Right-Sibling 변환 규칙에 따르면 `d, e, f`는 `b`의 자식들이므로, `d`는 `b`의 `left_child`, `e`는 `d`의 `right_sibling`, `f`는 `e`의 `right_sibling`이 되어야 합니다. 또한 `d, e, f`는 자식이 없으므로 이들의 `left_child`는 모두 `NULL`이어야 합니다. 따라서 슬라이드 그림의 `d`, `e`, `f` 부분은 표준 LCRS 규칙을 잘못 표현했거나, 다른 변환 방식을 나타낼 수 있으므로 혼동하지 않도록 주의해야 합니다. 시험에서는 일반적인 LCRS 규칙을 정확히 적용하는 것이 중요합니다.

**시험 포인트**:

*   ⭐ **k-ary 트리**의 정의와 일반 트리를 이진 트리로 표현하는 LCRS 변환의 필요성을 설명할 수 있어야 합니다.
*   ⭐ **Left-Child/Right-Sibling (LCRS) 변환의 핵심 원리**: `left_child`와 `right_sibling` 포인터가 각각 무엇을 가리키는지 정확하게 이해하고 설명할 수 있어야 합니다. (⭐**첫 번째 자식**, ⭐**다음 형제**)
*   ⭐ LCRS 변환을 통해 얻는 **장점**들을 명확히 기술할 수 있어야 합니다 (자식 수에 대한 유연성, 이진 트리 구현의 단순성).
*   ⭐ **주어진 일반 트리를 LCRS 규칙에 따라 이진 트리로 변환하는 과정을 ⭐정확히 그릴 수 있어야 합니다.** 특히, 형제 노드들이 `right_sibling` 링크로 연결되고, 자식 노드들은 `left_child` 링크로만 연결되는 방식을 명확히 구분해야 합니다. 슬라이드의 예시 그림과 표준 LCRS 규칙의 차이점을 인지하고 있다면 더욱 좋은 점수를 받을 수 있습니다.

## Slide 24

**핵심 개념**:
이 슬라이드는 일반 트리(General Tree)나 트리들의 숲(Forest)을 이진 트리(Binary Tree)로 변환하는 "Left-Child/Right-Sibling Approach"를 설명합니다. 이 변환 방식은 "Knuth Transform"이라고도 불립니다. 일반 트리는 각 노드가 여러 자식(arbitrary number of children)을 가질 수 있지만, 이진 트리는 각 노드가 최대 두 개의 자식(left child, right child)만 가질 수 있습니다. 이 기법은 일반 트리를 이진 트리 구조로 효율적으로 표현하고 구현하기 위한 표준 방법입니다.

변환은 크게 두 단계로 이루어집니다:
1.  **일반 트리를 이진 트리로 변환**: 다음 규칙을 적용하여 노드 간의 관계를 재정의합니다.
    *   **가장 왼쪽 자식 (Leftmost child)** $\rightarrow$ **왼쪽 자식 (Left child)**: 부모 노드의 첫 번째 자식은 이진 트리의 왼쪽 자식이 됩니다.
    *   **오른쪽 형제 (Right sibling)** $\rightarrow$ **오른쪽 자식 (Right child)**: 동일한 부모를 가진 형제 노드는 이진 트리의 오른쪽 자식이 됩니다. 이때, 트리 루트들은 가상의 형제로 간주될 수 있습니다.
    *   이 변환된 구조를 시각적으로 이해하기 위해, `left-child/right-sibling` 관계가 형성된 트리를 시계 방향으로 45도 회전한다고 생각할 수 있습니다.
2.  **변환된 이진 트리의 연결 리스트 구현**: 변환된 이진 트리는 노드 구조체 내에 `left_child`와 `right_child` 포인터를 사용하여 연결 리스트 형태로 구현될 수 있습니다.

**코드/수식 해설**:
이 슬라이드에는 직접적인 코드나 수식은 없지만, 변환 규칙은 다음과 같이 개념적으로 표현될 수 있습니다.
일반 트리의 각 노드는 다음 두 가지 관계를 가집니다:
*   자식 노드들과의 관계
*   형제 노드들과의 관계

이것이 이진 트리로 변환될 때는 다음과 같이 매핑됩니다:
*   일반 트리의 `가장 왼쪽 자식`은 이진 트리의 `왼쪽 자식`으로 연결됩니다.
*   일반 트리의 `오른쪽 형제`는 이진 트리의 `오른쪽 자식`으로 연결됩니다.

예를 들어, 일반 트리의 노드 구조는 다음과 같을 수 있습니다:
```c
struct G_TreeNode {
    char data;
    struct G_TreeNode* first_child; // 첫 번째 자식 노드를 가리킴
    struct G_TreeNode* next_sibling; // 다음 형제 노드를 가리킴
    // 필요에 따라 부모 노드 포인터 등 추가 가능
};
```
이러한 구조는 `first_child`를 이진 트리의 `left_child`로, `next_sibling`을 이진 트리의 `right_child`로 직접 매핑하여 이진 트리를 표현하는 것과 동일합니다.

**구체적 예시**:
하나의 일반 트리를 예로 들어보겠습니다.

**원래 일반 트리 (Conceptual)**:
```
      A
     /|\
    B C D
   / \
  E   F
```
여기서 A는 B, C, D를 자식으로 가집니다. B는 E, F를 자식으로 가집니다.

**Left-Child/Right-Sibling 변환**:

1.  **A 노드**:
    *   A의 가장 왼쪽 자식은 B이므로, B는 A의 이진 트리 `left_child`가 됩니다.
    *   A의 오른쪽 형제는 없지만, 트리 루트가 여러 개라면 해당 루트들이 오른쪽 형제로 처리됩니다. (이 예시에서는 단일 트리이므로 해당 없음)
2.  **B 노드**:
    *   B의 가장 왼쪽 자식은 E이므로, E는 B의 이진 트리 `left_child`가 됩니다.
    *   B의 오른쪽 형제는 C이므로, C는 B의 이진 트리 `right_child`가 됩니다.
3.  **C 노드**:
    *   C는 자식이 없으므로, C의 이진 트리 `left_child`는 NULL이 됩니다.
    *   C의 오른쪽 형제는 D이므로, D는 C의 이진 트리 `right_child`가 됩니다.
4.  **D 노드**:
    *   D는 자식이 없으므로, D의 이진 트리 `left_child`는 NULL이 됩니다.
    *   D의 오른쪽 형제는 없으므로, D의 이진 트리 `right_child`는 NULL이 됩니다.
5.  **E 노드**:
    *   E는 자식도 형제도 없으므로, E의 이진 트리 `left_child`와 `right_child`는 모두 NULL이 됩니다.
6.  **F 노드**:
    *   F는 자식도 형제도 없으므로, F의 이진 트리 `left_child`와 `right_child`는 모두 NULL이 됩니다.

**변환된 이진 트리 (시계 방향 45도 회전 시각화)**:

```
      A
     /
    B  --- (right_child link)
   / \   /
  E   C --- (right_child link)
       \ /
        D
       / \
     (NULL)(NULL)
```
(이진 트리의 왼쪽 자식은 원래 트리의 수직 자식 관계를, 오른쪽 자식은 원래 트리의 수평 형제 관계를 나타냅니다.)

**시험 포인트**:
*   ⭐ **Left-Child/Right-Sibling Approach의 목적**: 일반 트리를 왜 이진 트리로 변환하는가? (이진 트리의 표준화된 구현 방식을 통해 효율적인 저장 및 처리를 가능하게 하기 위함)
*   ⭐ **변환 규칙**: 일반 트리의 `가장 왼쪽 자식`이 이진 트리의 `왼쪽 자식`으로, `오른쪽 형제`가 이진 트리의 `오른쪽 자식`으로 매핑된다는 것을 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **다른 이름**: 이 변환 방식이 `Knuth Transform`으로도 불린다는 것을 기억해야 합니다.
*   ⭐ **주어진 일반 트리를 Left-Child/Right-Sibling 방식으로 이진 트리로 변환하는 과정을 직접 수행하고 결과를 그릴 수 있어야 합니다.** (가장 중요한 시험 포인트 중 하나입니다.)
*   변환된 이진 트리의 노드 구조가 어떻게 정의될 수 있는지 (`left_child`와 `right_child` 포인터를 포함하는) 이해해야 합니다.

## 데이터구조 (CSED233) 강의 노트: Donald Knuth와 트리 변환

### **핵심 개념**
이 슬라이드는 컴퓨터 과학의 거장 **도널드 크누스(Donald Knuth)**의 업적과 데이터 구조에서의 중요한 개념인 **일반 트리(General Tree)를 이진 트리(Binary Tree)로 변환하는 방법**을 소개합니다.

1.  **도널드 크누스**: "The Art of Computer Programming (TAOCP)"의 저자이자 TeX 조판 시스템의 개발자로, 알고리즘 분석 및 컴퓨터 과학 전반에 걸쳐 지대한 영향을 끼친 인물입니다. 데이터 구조와 알고리즘 학습에 있어 그의 저서는 정수와 같은 역할을 합니다.
2.  **일반 트리**: 각 노드가 가질 수 있는 자식 노드의 수에 제한이 없는 트리 구조입니다.
3.  **이진 트리**: 각 노드가 최대 두 개의 자식 노드(왼쪽 자식, 오른쪽 자식)만 가질 수 있는 트리 구조입니다.
4.  **일반 트리의 이진 트리 표현 (Left-Child, Right-Sibling Representation)**: 일반 트리를 이진 트리 구조로 효율적으로 표현하는 방법입니다. 이는 일반 트리를 다루는 복잡성을 줄이고, 이진 트리에 적용되는 다양한 알고리즘을 활용할 수 있게 합니다.

### **코드/수식 해설**

일반 트리의 각 노드는 자식 노드의 수가 가변적이지만, 이진 트리 표현에서는 각 노드가 `firstChild`와 `nextSibling`이라는 두 개의 포인터만을 가집니다.

*   **`firstChild` (빨간색 선)**: 해당 노드의 첫 번째 자식 노드를 가리킵니다. (이진 트리의 '왼쪽 자식' 역할)
*   **`nextSibling` (초록색 선)**: 해당 노드의 바로 다음 형제 노드를 가리킵니다. (이진 트리의 '오른쪽 자식' 역할)

이 표현법에서, 어떤 노드의 자식 노드는 그 노드의 `firstChild` 포인터를 따라가고, 이어서 `nextSibling` 포인터들을 따라가면서 찾을 수 있습니다.

**노드 구조 (C++ 예시):**
```cpp
struct GeneralTreeNode {
    char data;
    GeneralTreeNode* firstChild; // 첫 번째 자식 (이진 트리의 왼쪽 자식)
    GeneralTreeNode* nextSibling; // 다음 형제 (이진 트리의 오른쪽 자식)

    GeneralTreeNode(char val) : data(val), firstChild(nullptr), nextSibling(nullptr) {}
};
```

### **구체적 예시**

슬라이드에 제시된 트리 변환 예시를 따릅니다.

**[원본 일반 트리]**
```
      a
     / \
    b   c
   /|\
  d e f
```

**[변환된 이진 트리 (Left-Child, Right-Sibling Representation)]**
1.  **노드 `a`**:
    *   첫 번째 자식은 `b`입니다. (`a` -> `b`는 빨간색 선)
    *   형제 노드는 없습니다. (`a`의 `nextSibling`은 `nullptr`)
2.  **노드 `b`**:
    *   첫 번째 자식은 `d`입니다. (`b` -> `d`는 빨간색 선)
    *   다음 형제는 `c`입니다. (`b` -> `c`는 초록색 선)
3.  **노드 `c`**:
    *   자식 노드가 없습니다. (`c`의 `firstChild`는 `nullptr`)
    *   형제 노드가 없습니다. (`c`의 `nextSibling`은 `nullptr`)
4.  **노드 `d`**:
    *   첫 번째 자식은 `e`의 형제인 `d`의 자식이 아니므로 `d`의 `firstChild`는 `nullptr`입니다. (`d` 자체는 원본 트리에서 자식이 없었음.)
    *   다음 형제는 `e`입니다. (`d` -> `e`는 초록색 선)
5.  **노드 `e`**:
    *   자식 노드가 없습니다. (`e`의 `firstChild`는 `nullptr`)
    *   다음 형제는 `f`입니다. (`e` -> `f`는 초록색 선)
6.  **노드 `f`**:
    *   자식 노드가 없습니다. (`f`의 `firstChild`는 `nullptr`)
    *   형제 노드가 없습니다. (`f`의 `nextSibling`은 `nullptr`)

이 변환을 통해 원본 트리의 계층 구조는 유지되면서도, 각 노드가 두 개의 포인터만을 가지는 이진 트리 형태로 효율적으로 저장 및 처리될 수 있습니다. 마치 가계도를 저장할 때 "나의 첫째 자식"과 "나의 다음 형제"만 기록하는 방식과 유사합니다.

### **시험 포인트**

*   **⭐ 도널드 크누스의 중요성**: 컴퓨터 과학, 특히 알고리즘 및 데이터 구조 분야에서의 그의 기여를 이해하고 설명할 수 있어야 합니다. (TAOCP, TeX)
*   **⭐ 일반 트리의 이진 트리 표현 방식**: `Left-Child, Right-Sibling` 변환의 개념을 정확히 이해하고, 주어진 일반 트리를 이진 트리 형태로 변환하는 과정을 설명하거나 직접 그릴 수 있어야 합니다.
*   **⭐ `firstChild`와 `nextSibling` 포인터의 역할**: 각 포인터가 변환된 이진 트리에서 어떤 관계를 나타내는지 (왼쪽 자식 vs. 오른쪽 자식) 명확히 구분할 수 있어야 합니다.
*   **변환의 장점**: 왜 일반 트리를 이진 트리로 변환하여 사용하는지 (구현의 단순화, 이진 트리 알고리즘 활용 등) 설명할 수 있어야 합니다.

**핵심 개념**
이 슬라이드는 **일반 트리(General Tree)를 이진 트리(Binary Tree)로 변환하는 방법**을 설명합니다. 일반 트리는 하나의 노드가 여러 개의 자식 노드를 가질 수 있는 반면, 이진 트리는 각 노드가 최대 두 개의 자식(왼쪽 자식, 오른쪽 자식)만 가질 수 있습니다. 일반 트리를 이진 트리로 변환하는 가장 일반적인 방법 중 하나는 **"왼쪽 자식-오른쪽 형제(Left-Child, Right-Sibling)" 표현**입니다.

변환의 핵심 규칙은 다음과 같습니다:
1.  **각 노드의 '가장 왼쪽 자식(Leftmost child)'은 이진 트리의 '왼쪽 자식(Left child)'이 됩니다.**
2.  **각 노드의 '다른 자식들(형제 노드들)'은 이진 트리의 '오른쪽 자식(Right child)'으로 연결됩니다.** 즉, 한 노드의 두 번째 자식은 첫 번째 자식의 오른쪽 자식이 되고, 세 번째 자식은 두 번째 자식의 오른쪽 자식이 되는 식으로 형제 노드들이 오른쪽으로 체인처럼 연결됩니다.

**코드/수식 해설**
이 슬라이드에는 직접적인 코드나 수식은 없지만, 개념적으로 일반 트리의 노드 구조를 이진 트리의 노드 구조로 매핑하는 방식을 설명할 수 있습니다.

**일반 트리 노드 정의 (개념적)**
```c
struct GeneralTreeNode {
    Data data;
    GeneralTreeNode* parent;
    std::vector<GeneralTreeNode*> children; // 여러 개의 자식을 가질 수 있음
};
```

**이진 트리 노드 정의**
```c
struct BinaryTreeNode {
    Data data;
    BinaryTreeNode* left_child;  // 원래 트리의 가장 왼쪽 자식
    BinaryTreeNode* right_child; // 원래 트리의 다음 형제
};
```

**변환 매핑 (개념적)**
일반 트리의 `GeneralTreeNode G`를 이진 트리의 `BinaryTreeNode B`로 변환할 때:
*   `B.data = G.data;`
*   `B.left_child = Convert(G.children[0]);` (G의 가장 왼쪽 자식을 B의 왼쪽 자식으로)
*   `B.right_child = Convert(G.children[1]);` (G의 첫 번째 형제(두 번째 자식)를 B의 오른쪽 자식으로. 단, 실제로는 `G.children[0]`의 `right_child`가 `G.children[1]`이 됨)

실제로는 각 노드를 순회하며 다음과 같이 연결합니다:
$$ \text{For each node } P \text{ in the general tree:} $$
$$ \text{If } P \text{ has at least one child } C_1: $$
$$ \quad P_{\text{binary}}.\text{left\_child} = C_{1, \text{binary}} $$
$$ \text{For children } C_i, C_{i+1} \text{ where } C_{i+1} \text{ is the next sibling of } C_i: $$
$$ \quad C_{i, \text{binary}}.\text{right\_child} = C_{i+1, \text{binary}} $$
여기서 $P_{\text{binary}}$는 일반 트리 노드 $P$에 대응하는 이진 트리 노드를 의미합니다.

**구체적 예시**
슬라이드의 이미지를 통해 변환 과정을 살펴보겠습니다.

1.  **왼쪽 예시 (더 복잡한 트리)**
    *   **원본 트리**: 루트는 3개의 자식을 가집니다. 그중 첫 번째 자식(왼쪽에서 첫 번째)은 2개의 자식을 가집니다.
    *   **변환 후 이진 트리**:
        *   원본 루트의 **가장 왼쪽 자식**이 새로운 이진 트리의 루트의 **왼쪽 자식**이 됩니다 (슬라이드에서 빨간색 선).
        *   원본 루트의 나머지 두 자식은 형제 관계이므로, 첫 번째 자식의 **오른쪽 자식**이 두 번째 자식이 되고, 두 번째 자식의 **오른쪽 자식**이 세 번째 자식이 되는 식으로 연결됩니다.
        *   마찬가지로, 원래 루트의 가장 왼쪽 자식이었던 노드도 2개의 자식을 가지므로, 그 노드의 **가장 왼쪽 자식**은 이진 트리의 **왼쪽 자식**이 되고 (슬라이드에서 빨간색 선), 그 노드의 다음 자식은 **오른쪽 자식**이 됩니다.

2.  **오른쪽 예시 (선형에 가까운 트리)**
    *   **원본 트리**: 루트는 2개의 자식을 가집니다. 그중 첫 번째 자식은 다시 2개의 자식을 가집니다.
    *   **변환 후 이진 트리**:
        *   원본 루트의 **가장 왼쪽 자식**이 이진 트리의 루트의 **왼쪽 자식**이 됩니다 (슬라이드에서 빨간색 선).
        *   원본 루트의 두 번째 자식은 첫 번째 자식의 **오른쪽 자식**이 됩니다.
        *   원본 루트의 가장 왼쪽 자식이었던 노드도 2개의 자식을 가지므로, 그 노드의 **가장 왼쪽 자식**은 이진 트리의 **왼쪽 자식**이 되고 (슬라이드에서 빨간색 선), 그 노드의 다음 자식은 **오른쪽 자식**이 됩니다.

**시험 포인트**
*   ⭐ **일반 트리를 이진 트리로 변환하는 '왼쪽 자식-오른쪽 형제' 방법의 개념과 규칙을 정확히 이해하고 설명할 수 있어야 합니다.**
*   ⭐ **주어진 일반 트리를 변환 규칙에 따라 이진 트리로 그릴 수 있어야 합니다.** (특히, 어떤 간선이 왼쪽 자식 관계를 나타내고 어떤 간선이 오른쪽 형제 관계를 나타내는지 구분할 수 있어야 합니다.)
*   이 변환의 장점: 모든 트리를 표준화된 이진 트리 구조로 처리할 수 있어 알고리즘 구현이 단순해집니다.
*   이 변환의 단점: 원래 트리의 깊이(depth)와 폭(width)이 이진 트리의 깊이에 복합적으로 영향을 미쳐, 원래 넓은(wide) 트리가 이진 트리에서는 매우 깊어질 수 있습니다.

## Slide 27

**핵심 개념**:
이 슬라이드는 **일반 트리(General Tree)**를 **이진 트리(Binary Tree)**로 변환하는 방법 중 하나인 **Left-Child Right-Sibling (LCRS)** 변환의 핵심 원리를 설명합니다. 이 변환 방식은 일반 트리의 노드들이 가질 수 있는 임의의 자식 수를 이진 트리의 최대 2개의 자식(왼쪽, 오른쪽)으로 제한하면서도, 원본 트리의 구조 정보를 보존하는 데 사용됩니다.

핵심 규칙은 다음과 같습니다:
1.  **`Right sibling` $\rightarrow$ `Right child`**: 일반 트리에서 한 노드의 바로 오른쪽에 있는 형제(sibling) 노드는 변환된 이진 트리에서 해당 노드의 **오른쪽 자식(right child)**이 됩니다.
2.  **`Tree roots are assumed to be siblings`**: 여러 개의 트리가 모여 있는 포레스트(forest)를 하나의 이진 트리로 변환할 경우, 각 트리의 루트 노드들이 서로 형제 관계인 것처럼 취급되어 오른쪽 자식 링크로 연결될 수 있습니다. (단일 트리의 경우, 루트는 오른쪽 자식을 가지지 않을 수 있습니다.)
3.  **`Remove the other links`**: 위의 두 규칙에 따라 재해석되는 링크 외의 모든 다른 링크(예: 일반 트리에서 부모가 첫 번째 자식 외의 다른 자식에게 직접 연결하는 링크)는 제거됩니다. 일반 트리의 '첫 번째 자식' 링크는 이진 트리의 '왼쪽 자식' 링크로 해석됩니다.

**코드/수식 해설**:
슬라이드에 직접적인 코드나 수식은 없지만, 다이어그램의 색상별 선들이 변환 규칙을 시각적으로 나타냅니다.
*   **녹색 선**: 일반 트리에서 **부모가 첫 번째 자식**에게 연결하는 링크를 나타냅니다. 이 링크는 변환된 이진 트리에서 **왼쪽 자식** 링크로 재해석됩니다.
*   **빨간색 선**: 일반 트리에서 **노드와 그 다음 형제**를 연결하는 링크를 나타냅니다. 이 링크는 변환된 이진 트리에서 **오른쪽 자식** 링크로 재해석됩니다.
*   **검은색 선**: 일반 트리에서 **부모가 첫 번째 자식이 아닌 다른 자식**들에게 직접 연결하는 링크를 나타냅니다. 이 링크들은 LCRS 변환 과정에서 **제거**됩니다. (이 자식들은 첫 번째 자식의 '오른쪽 자식' 체인을 통해 간접적으로 연결됩니다.)

**구체적 예시**:
슬라이드의 왼쪽 다이어그램(원본 일반 트리)과 오른쪽 다이어그램(변환된 이진 트리)을 비교하여 LCRS 변환 과정을 이해할 수 있습니다.

**1. 원본 일반 트리 (왼쪽)**:
*   맨 위 루트 노드는 3개의 자식을 가집니다 (녹색, 검은색, 검은색 선으로 연결). 이 중 가장 왼쪽의 자식은 녹색 선으로 연결되어 있습니다.
*   녹색 선으로 연결된 노드 (루트의 첫 번째 자식)는 2개의 자식을 가집니다 (녹색, 검은색 선으로 연결). 이 중 가장 왼쪽의 자식은 녹색 선으로 연결되어 있습니다.
*   각 노드와 그 바로 다음 형제는 빨간색 선으로 연결되어 있습니다. 예를 들어, 루트의 첫 번째 자식과 두 번째 자식은 빨간색 선으로 연결됩니다.

**2. 변환된 이진 트리 (오른쪽)**:
변환 규칙이 적용되어 다음과 같이 변경됩니다.
*   **녹색 선**: 원본 트리의 '첫 번째 자식' 링크는 이진 트리의 '왼쪽 자식' 링크로 유지됩니다.
    *   루트의 왼쪽 자식은 원래 루트의 첫 번째 자식이었던 노드입니다.
    *   그 왼쪽 자식의 왼쪽 자식은 원래 해당 노드의 첫 번째 자식이었던 노드입니다.
*   **빨간색 선**: 원본 트리의 '다음 형제' 링크는 이진 트리의 '오른쪽 자식' 링크로 재해석됩니다.
    *   루트의 왼쪽 자식 노드에서 오른쪽으로 연결된 빨간색 선은 이제 그 노드의 '오른쪽 자식'을 가리킵니다. 이 노드는 원래 루트의 두 번째 자식이었던 노드입니다.
    *   이 '오른쪽 자식' 노드에서 다시 오른쪽으로 연결된 빨간색 선은 원래 루트의 세 번째 자식이었던 노드를 가리킵니다.
    *   이러한 방식으로, 일반 트리의 모든 자식들은 부모의 첫 번째 자식으로부터 시작하여 오른쪽 자식 체인으로 연결됩니다.
*   **검은색 선**: 원본 트리의 부모에서 첫 번째 자식이 아닌 다른 자식에게 직접 연결되던 검은색 선은 모두 제거됩니다. 이는 해당 자식들이 이제 첫 번째 자식의 오른쪽 형제 체인을 통해 접근되기 때문입니다.

**시험 포인트**:
*   ⭐ **LCRS 변환의 정의 및 목적**: 일반 트리를 이진 트리로 변환하는 LCRS 방식의 개념과 왜 이러한 변환이 필요한지 (일반 트리를 이진 트리 구조로 표현하여 처리 효율을 높이거나 단순화하기 위함) 이해해야 합니다.
*   ⭐ **각 링크의 변환 규칙**: 일반 트리의 '첫 번째 자식'이 이진 트리의 '왼쪽 자식'이 되고, '다음 형제'가 '오른쪽 자식'이 된다는 핵심 규칙을 정확히 기억해야 합니다.
*   ⭐ **불필요한 링크 제거의 의미**: 첫 번째 자식 및 다음 형제 링크 외의 모든 링크(검은색 선)가 제거되는 이유를 설명할 수 있어야 합니다. 이는 이진 트리의 정의(각 노드가 최대 두 개의 자식만 가짐)를 만족시키고 정보의 중복을 피하기 위함입니다.
*   **변환된 이진 트리의 특징**: LCRS 변환을 통해 생성된 이진 트리의 높이 및 노드 간 관계가 원본 일반 트리와 어떻게 달라지는지 파악할 수 있어야 합니다. 예를 들어, 일반 트리의 깊이(depth)는 이진 트리의 왼쪽으로만 연결된 체인의 길이가 되며, 일반 트리의 너비(width)는 이진 트리의 오른쪽으로만 연결된 체인의 길이가 됩니다.

## Slide 28

이 슬라이드는 일반 트리(General Tree)를 이진 트리(Binary Tree)로 변환하는 방법을 시각적으로 설명하고 있습니다. 이는 자료구조 구현의 중요한 기법 중 하나입니다.

---

### **핵심 개념**

*   **일반 트리를 이진 트리로 변환 (Converting a General Tree to a Binary Tree)**:
    일반 트리는 노드가 가질 수 있는 자식의 수에 제한이 없는 트리입니다. 이를 이진 트리 (각 노드가 최대 두 개의 자식만 가질 수 있는 트리)로 변환하는 기법은 일반 트리를 더 효율적으로 저장하고 처리하기 위해 사용됩니다. 이 변환의 핵심은 **'첫 번째 자식-다음 형제 (First-Child, Next-Sibling)'** 또는 **'좌측 자식-우측 형제 (Left-Child, Right-Sibling)'** 표현입니다.

    *   **첫 번째 자식 (First-Child) $\rightarrow$ 좌측 자식 (Left-Child)**: 원본 일반 트리에서 노드의 '첫 번째 자식'은 변환된 이진 트리에서 해당 노드의 '좌측 자식'이 됩니다.
    *   **다음 형제 (Next-Sibling) $\rightarrow$ 우측 자식 (Right-Child)**: 원본 일반 트리에서 노드의 '다음 형제'는 변환된 이진 트리에서 해당 노드의 '우측 자식'이 됩니다.
*   **시각적 변환 (Visual Transformation)**: 슬라이드의 "Rotate clockwise by 45°"는 이러한 변환 과정을 시각적으로 이해하는 데 도움을 줍니다. 일반 트리의 수직적인 자식 링크와 수평적인 형제 링크를 45도 시계 방향으로 회전시켜 이진 트리의 좌측 자식과 우측 자식 링크처럼 보이게 하는 비유입니다.
*   **연결 리스트 구현 (Linked Implementation)**: 이 변환을 통해 일반 트리를 이진 트리 노드 구조(각 노드가 `left`와 `right` 두 개의 포인터만 가짐)를 사용하여 연결 리스트 형태로 쉽게 구현할 수 있게 됩니다.

### **코드/수식 해설**

일반 트리의 노드 구조는 다음과 같이 표현될 수 있습니다 (의사 코드):

```c
// 일반 트리 노드 구조 (예시: 자식 배열 또는 리스트)
struct GeneralTreeNode {
    Data_Type data;
    // 배열로 자식들을 관리하는 경우
    struct GeneralTreeNode* children[MAX_CHILDREN]; 
    int num_children;
    
    // 또는 연결 리스트로 자식들을 관리하는 경우
    // struct GeneralTreeNode* first_child; 
};
```

이를 "첫 번째 자식-다음 형제" 방식으로 이진 트리로 변환하면, 각 노드는 다음과 같은 구조를 가지게 됩니다:

```c
// 이진 트리 노드 구조 (변환 후)
struct BinaryTreeNode {
    Data_Type data;
    struct BinaryTreeNode* left_child;  // 원본 노드의 첫 번째 자식
    struct BinaryTreeNode* right_child; // 원본 노드의 다음 형제
};
```

슬라이드에서 제시된 그림은 이러한 변환된 이진 트리의 시각적 표현입니다.
- 빨간색 선은 `left_child` 포인터(원본 트리의 첫 번째 자식)를 나타냅니다.
- 초록색 선은 `right_child` 포인터(원본 트리의 다음 형제)를 나타냅니다.

### **구체적 예시**

다음과 같은 일반 트리를 가정해 봅시다.

```
     A
    /|\
   B C D
  / \
 E   F
```

이 트리를 "첫 번째 자식-다음 형제" 원칙에 따라 이진 트리로 변환하면 다음과 같습니다:

1.  **노드 A**:
    *   `A->left_child`는 `B` (A의 첫 번째 자식).
    *   `A->right_child`는 `NULL` (A는 부모의 형제가 아니므로).
2.  **노드 B**:
    *   `B->left_child`는 `E` (B의 첫 번째 자식).
    *   `B->right_child`는 `C` (B의 다음 형제).
3.  **노드 C**:
    *   `C->left_child`는 `NULL` (C는 자식이 없음).
    *   `C->right_child`는 `D` (C의 다음 형제).
4.  **노드 D**:
    *   `D->left_child`는 `NULL`.
    *   `D->right_child`는 `NULL`.
5.  **노드 E**:
    *   `E->left_child`는 `NULL`.
    *   `E->right_child`는 `F` (E의 다음 형제).
6.  **노드 F**:
    *   `F->left_child`는 `NULL`.
    *   `F->right_child`는 `NULL`.

결과적인 이진 트리의 구조는 원래 트리와는 다르게, 주로 오른쪽으로 치우친(right-skewed) 형태를 띠게 됩니다. 슬라이드의 그림은 이러한 변환된 이진 트리의 예시를 보여주고 있습니다. 가장 위의 노드에서 빨간색 선으로 연결된 노드가 첫 번째 자식, 초록색 선으로 연결된 노드는 그 첫 번째 자식의 다음 형제이거나, 상위 노드의 자식 중 두 번째 이후의 자식이 변환된 형태일 수 있습니다.

### **시험 포인트**

*   ⭐ **일반 트리를 이진 트리로 변환하는 '첫 번째 자식-다음 형제(또는 좌측 자식-우측 형제)' 표현 방식**을 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ 왜 이러한 변환이 필요한지 (예: 구현 단순화, 이진 트리 알고리즘 활용) 그 **목적과 장점**을 알아야 합니다.
*   ⭐ 주어진 일반 트리를 이진 트리로 직접 **변환하는 과정**을 예시를 통해 보여줄 수 있어야 합니다.
*   ⭐ 변환 후 이진 트리의 `left_child` 포인터와 `right_child` 포인터가 원본 일반 트리의 어떤 관계를 나타내는지 설명할 수 있어야 합니다.

## Slide 29

이 슬라이드는 일반 트리를 **Left-Child/Right-Sibling (LCRS) 표현 방식**으로 변환하고, 이를 **연결 리스트(Linked Representation)** 형태로 메모리에 구현하는 방법을 설명합니다.

---

### **핵심 개념**

*   **일반 트리 (General Tree)**: 자식 노드의 수에 제한이 없는 트리를 의미합니다. 슬라이드 좌측 상단의 그림이 일반 트리의 예시입니다. 특정 노드(예: `b`)는 여러 자식(d, e, f)을 가질 수 있습니다.
*   **Left-Child/Right-Sibling (LCRS) 표현**: 일반 트리를 이진 트리의 형태로 표현하는 표준적인 방법 중 하나입니다. 각 노드는 최대 두 개의 포인터를 가집니다.
    *   **Left Child (Lc)**: 해당 노드의 *첫 번째 자식*을 가리킵니다. (슬라이드 좌하단 그림의 빨간색 선)
    *   **Right Sibling (Rs)**: 해당 노드와 *같은 부모를 가지는 다음 형제 노드*를 가리킵니다. (슬라이드 좌하단 그림의 초록색 선)
    *   이 변환을 통해 일반 트리는 개념적으로 이진 트리가 됩니다.
*   **연결 리스트 구현 (Linked Representation)**: LCRS 구조를 실제 메모리에서 포인터를 이용해 구현하는 방식입니다. 각 노드는 자신의 데이터와 Lc, Pa, Rs 포인터를 저장하는 구조체로 표현됩니다.

---

### **코드/수식 해설**

LCRS 트리의 노드 구조는 C 언어 기준으로 다음과 같이 정의될 수 있습니다.

```c
typedef struct TreeNode {
    char data;                 // 노드의 데이터 (예: 'a', 'b', 'c')
    struct TreeNode *leftChild;    // 첫 번째 자식 노드를 가리키는 포인터 (Lc)
    struct TreeNode *rightSibling; // 다음 형제 노드를 가리키는 포인터 (Rs)
    struct TreeNode *parent;       // 부모 노드를 가리키는 포인터 (Pa) - 선택적이지만 유용함
} TreeNode;
```

*   `data`: 노드에 저장될 실제 값(Label). 슬라이드에서는 `La`로 표시되어 있습니다.
*   `leftChild`: 노드의 첫 번째 자식(`Lc`)을 가리킵니다. 자식이 없으면 `NULL`입니다.
*   `rightSibling`: 노드의 오른쪽 형제(`Rs`)를 가리킵니다. 오른쪽 형제가 없으면 `NULL`입니다.
*   `parent`: 노드의 부모(`Pa`)를 가리킵니다. 루트 노드의 부모는 `NULL`입니다. 이 필드는 LCRS 표현 자체의 필수 요소는 아니지만, 상위 노드로의 탐색이 필요한 경우 유용하게 사용됩니다.

### **시험 포인트**

*   ⭐ **LCRS 변환 원리 이해**: 일반 트리가 주어졌을 때, LCRS 표현으로 어떻게 변환되는지 (어떤 엣지가 Left Child가 되고 어떤 엣지가 Right Sibling이 되는지) 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **노드 구조와 각 필드의 의미**: `leftChild`, `rightSibling`, `parent` 포인터가 각각 무엇을 가리키는지 명확히 알아야 합니다. 특히 `parent` 필드의 필요성과 유용성을 이해하는 것이 중요합니다.
*   ⭐ **LCRS 트리의 장단점**: 일반 트리를 LCRS로 표현했을 때의 메모리 사용 효율성 및 탐색(첫 자식, 다음 형제, 부모)의 시간 복잡도를 분석할 수 있어야 합니다. (예: `N`개의 노드를 가진 일반 트리를 LCRS로 표현하면 $N$개의 노드가 필요하며, 각 노드는 고정된 수의 포인터를 가집니다.)
*   ⭐ **탐색 및 순회**: LCRS 표현된 트리에서 특정 노드의 첫 번째 자식, 다음 형제, 부모를 찾는 방법뿐만 아니라, 전위/중위/후위 순회 등의 트리 순회 알고리즘이 어떻게 적용될 수 있는지 생각해 볼 필요가 있습니다.
*   **응용**: 일반 트리를 이진 트리 형태로 다룰 수 있게 해주므로, 이진 트리에 적용되는 많은 알고리즘을 일반 트리에 적용할 수 있는 기반이 됩니다.

## Traversal: Original vs. Converted Binary

### 핵심 개념

이 슬라이드는 **일반 트리(General Tree)**를 **이진 트리(Binary Tree)**로 변환하는 방법 중 하나인 **Left-Child Right-Sibling (LCRS)** 변환과, 이 변환이 트리의 순회(Traversal) 결과에 미치는 영향을 설명합니다.

*   **일반 트리 $T$**: 각 노드가 임의의 수의 자식(0개 이상)을 가질 수 있는 트리입니다. 슬라이드의 좌측 그림이 일반 트리의 예시입니다.
*   **이진 트리 $T'$**: 각 노드가 최대 두 개의 자식(왼쪽 자식, 오른쪽 자식)을 가질 수 있는 트리입니다.
*   **LCRS 변환 (Left-Child Right-Sibling Conversion)**: 일반 트리를 이진 트리로 표현하는 표준적인 방법입니다.
    *   일반 트리의 각 노드 $N$에 대해:
        1.  $N$의 **첫 번째 자식(First Child)**은 변환된 이진 트리 $T'$에서 $N$의 **왼쪽 자식(Left Child)**이 됩니다.
        2.  $N$의 **다음 형제(Next Sibling)**는 변환된 이진 트리 $T'$에서 $N$의 **오른쪽 자식(Right Child)**이 됩니다.
*   **트리 순회 (Tree Traversal)**: 트리의 모든 노드를 방문하는 체계적인 방법으로, 주로 전위(Preorder), 중위(Inorder), 후위(Postorder) 순회가 있습니다.
    *   **전위 순회 (Preorder)**: 현재 노드 방문 -> 왼쪽/첫 번째 자식 방문 -> 오른쪽/나머지 자식 방문.
    *   **중위 순회 (Inorder)**: 왼쪽/첫 번째 자식 방문 -> 현재 노드 방문 -> 오른쪽/나머지 자식 방문. (일반 트리에서는 정의가 다양할 수 있습니다.)
    *   **후위 순회 (Postorder)**: 왼쪽/첫 번째 자식 방문 -> 오른쪽/나머지 자식 방문 -> 현재 노드 방문.

### 코드/수식 해설

LCRS 변환은 일반 트리의 다중 자식 관계를 이진 트리의 왼쪽/오른쪽 자식 관계로 매핑합니다. 이 변환을 통해 일반 트리의 순회 결과와 변환된 이진 트리의 특정 순회 결과 사이에 다음과 같은 중요한 등가 관계가 성립합니다.

*   **일반 트리 $T$의 전위 순회(Preorder)**는 변환된 이진 트리 $T'$의 전위 순회와 **동일합니다.**
    $$ \text{Preorder}(T) = \text{Preorder}(T') $$
    *   `Preorder(node)` 함수 (일반 트리):
        ```
        function preorder_general(node):
            if node is null: return
            visit(node)
            for each child in node.children:
                preorder_general(child)
        ```
    *   `Preorder(node)` 함수 (이진 트리):
        ```
        function preorder_binary(node):
            if node is null: return
            visit(node)
            preorder_binary(node.left)
            preorder_binary(node.right)
        ```
    *   LCRS 변환에서 일반 트리의 `node.children` 리스트의 첫 번째 자식이 이진 트리의 `node.left`가 되고, 그 다음 형제들이 `node.right` 체인으로 연결되기 때문에, `root`를 먼저 방문하고 `left` (첫 번째 자식 서브트리)를 재귀적으로 방문한 다음 `right` (다음 형제 서브트리)를 방문하는 방식이 일반 트리의 자식들을 순서대로 방문하는 방식과 일치하게 됩니다.

*   **일반 트리 $T$의 후위 순회(Postorder)**는 변환된 이진 트리 $T'$의 중위 순회와 **동일합니다.**
    $$ \text{Postorder}(T) = \text{Inorder}(T') $$
    *   `Postorder(node)` 함수 (일반 트리):
        ```
        function postorder_general(node):
            if node is null: return
            for each child in node.children:
                postorder_general(child)
            visit(node)
        ```
    *   `Inorder(node)` 함수 (이진 트리):
        ```
        function inorder_binary(node):
            if node is null: return
            inorder_binary(node.left)
            visit(node)
            inorder_binary(node.right)
        ```
    *   이 등가 관계는 LCRS 변환의 독특한 특성으로, 일반 트리의 `자식들 먼저 방문 후 루트 방문` (Postorder)이 이진 트리의 `왼쪽 자식 서브트리 방문 후 루트 방문, 그 다음 오른쪽 자식 서브트리 방문` (Inorder)과 맞아떨어지기 때문입니다. 이진 트리의 `left`는 일반 트리의 첫 번째 자식 서브트리를, `right`는 그 첫 번째 자식의 다음 형제 서브트리를 나타내므로 이러한 매핑이 가능합니다.

### 구체적 예시

슬라이드에 제시된 일반 트리 $T$와 이를 LCRS 방식으로 변환한 이진 트리 $T'$를 바탕으로 순회 결과를 확인해봅시다. (슬라이드의 $T'$ 다이어그램은 시각적으로 약간 혼동을 줄 수 있으나, LCRS 규칙에 따라 올바르게 변환하면 아래와 같은 구조와 순회 결과가 나옵니다.)

**1. Original General Tree $T$**:
```
     a
    /|\
   b c
  /|\
 d e f
```

*   **Preorder $T$**: `a` (루트) -> `b`의 서브트리 (`d,e,f`) -> `c`의 서브트리
    *   결과: `a b d e f c`
*   **Inorder $T$**: (첫 번째 자식 중위, 루트, 나머지 자식 중위 - 한 가지 정의)
    *   `b`의 중위 (`d` -> `b` -> `e` -> `f`) -> `a` -> `c`의 중위 (`c`)
    *   결과: `d b e f a c`
*   **Postorder $T$**: `b`의 서브트리 -> `c`의 서브트리 -> `a` (루트)
    *   `d,e,f` -> `b` -> `c` -> `a`
    *   결과: `d e f b c a`

**2. Converted Binary Tree $T'$ (LCRS 변환)**:
일반 트리 $T$를 LCRS 규칙에 따라 변환하면 다음과 같은 이진 트리가 됩니다.
*   `a`: 첫 자식 `b` (왼쪽), 형제 없음 (오른쪽 = null)
*   `b`: 첫 자식 `d` (왼쪽), 다음 형제 `c` (오른쪽)
*   `c`: 자식 없음 (왼쪽 = null), 형제 없음 (오른쪽 = null)
*   `d`: 자식 없음 (왼쪽 = null), 다음 형제 `e` (오른쪽)
*   `e`: 자식 없음 (왼쪽 = null), 다음 형제 `f` (오른쪽)
*   `f`: 자식 없음 (왼쪽 = null), 형제 없음 (오른쪽 = null)

이를 구조로 그리면 다음과 같습니다:
```
           a
          /
         b
        / \
       d   c  (c is right child of b)
        \
         e    (e is right child of d)
          \
           f  (f is right child of e)
```

*   **Preorder $T'$**: (Root, Left, Right)
    *   `a` -> `b` 서브트리 (`b` -> `d` 서브트리 (`d` -> `e` 서브트리 (`e` -> `f` 서브트리 (`f`))) -> `c` 서브트리 (`c`))
    *   결과: `a b d e f c`
*   **Inorder $T'$**: (Left, Root, Right)
    *   `d` 서브트리의 중위 (`d` -> `e` 서브트리의 중위 (`e` -> `f` 서브트리의 중위 (`f`))) -> `b` -> `c` 서브트리의 중위 (`c`) -> `a`
    *   결과: `d e f b c a`
*   **Postorder $T'$**: (Left, Right, Root)
    *   `d` 서브트리의 후위 (`f` -> `e` -> `d`) -> `c` 서브트리 (`c`) -> `b` -> `a`
    *   결과: `f e d c b a`

슬라이드에 제시된 순회 결과와 비교해보면:
*   Preorder $T$ (`a b d e f c`) == Preorder $T'$ (`a b d e f c`) (일치)
*   Postorder $T$ (`d e f b c a`) == Inorder $T'$ (`d e f b c a`) (일치)
*   슬라이드의 Inorder $T$ (`d b e f a c`)는 일반 트리에 대한 정의 중 하나이며, Postorder $T'$ (`f e d c b a`)와는 직접적인 관계가 없습니다.

### 시험 포인트

*   ⭐ **LCRS (Left-Child Right-Sibling) 변환 규칙**을 정확히 이해하고, 주어진 일반 트리를 이진 트리로 변환하는 과정을 설명하거나 직접 변환할 수 있어야 합니다. (첫 자식은 왼쪽 자식, 다음 형제는 오른쪽 자식)
*   ⭐ 일반 트리와 이진 트리의 **전위, 중위, 후위 순회** 정의를 명확히 알고, 주어진 트리 구조에 대해 올바른 순회 결과를 도출할 수 있어야 합니다.
*   ⭐ **일반 트리와 LCRS 변환된 이진 트리 간의 순회 등가 관계**를 정확히 기억해야 합니다. 특히, 다음 두 가지는 핵심 암기 사항입니다:
    *   `Preorder(General Tree) = Preorder(Converted Binary Tree)`
    *   `Postorder(General Tree) = Inorder(Converted Binary Tree)`
*   ⭐ 이 등가 관계를 활용하여, 예를 들어 "어떤 일반 트리의 후위 순회 결과가 주어졌을 때, LCRS 변환된 이진 트리의 중위 순회 결과는 무엇인가?" 와 같은 질문에 답할 수 있어야 합니다.

### **핵심 개념**
이 슬라이드는 **일반 트리(General Tree)를 이진 트리로 변환한 후, 다시 원래의 일반 트리로 복원하는 방법**에 대한 퀴즈를 제시하고 있습니다. 핵심 개념은 다음과 같습니다:

1.  **일반 트리(General Tree)**: 각 노드가 가질 수 있는 자식의 수에 제한이 없는 트리 구조입니다.
2.  **변환된 이진 트리 (First Child, Next Sibling Representation)**: 일반 트리를 이진 트리 형태로 표현하는 한 가지 방법입니다.
    *   **왼쪽 자식(Left Child)** 포인터는 해당 노드의 **첫 번째 자식(First Child)**을 가리킵니다. (슬라이드에서는 **빨간색 선**으로 표현)
    *   **오른쪽 자식(Right Child)** 포인터는 해당 노드의 **다음 형제(Next Sibling)**를 가리킵니다. (슬라이드에서는 **초록색 선**으로 표현)
    *   이러한 방식으로 일반 트리의 계층 구조와 형제 관계를 이진 트리의 두 포인터(Left, Right)로 표현할 수 있습니다.
3.  **변환된 이진 트리에서 일반 트리로의 복원**: 변환된 이진 트리(오른쪽)를 분석하여 원래의 일반 트리(왼쪽) 구조를 재구성하는 과정입니다. 이는 `leftChild` 포인터를 따라 첫 번째 자식을 찾고, 그 첫 번째 자식으로부터 `rightSibling` 포인터를 따라 다른 형제들을 찾아 해당 노드의 자식 목록을 완성하는 재귀적인 과정입니다.

### **코드/수식 해설**

일반 트리를 FCNS(First Child, Next Sibling) 이진 트리로 변환하는 것의 역방향으로, FCNS 이진 트리를 다시 일반 트리로 변환하는 함수는 다음과 같습니다.

```c++
// 일반 트리 노드 구조체
struct GeneralTreeNode {
    char data;
    std::vector<GeneralTreeNode*> children; // 자식 노드들을 저장하는 벡터 (여러 개 가능)

    GeneralTreeNode(char d) : data(d) {}
    void addChild(GeneralTreeNode* child) {
        children.push_back(child);
    }
};

// FCNS 이진 트리 노드 구조체
// (First Child, Next Sibling)
struct FCNS_BinaryTreeNode {
    char data;
    FCNS_BinaryTreeNode* leftChild;    // 일반 트리의 첫 번째 자식 (슬라이드의 빨간 선)
    FCNS_BinaryTreeNode* rightSibling; // 일반 트리의 다음 형제 (슬라이드의 초록 선)

    FCNS_BinaryTreeNode(char d) : data(d), leftChild(nullptr), rightSibling(nullptr) {}
};

// FCNS 이진 트리를 일반 트리로 변환하는 함수
GeneralTreeNode* convertFCNS_to_General(FCNS_BinaryTreeNode* fcns_root) {
    // 1. 기본 케이스: FCNS 노드가 null이면, 변환할 것이 없음
    if (fcns_root == nullptr) {
        return nullptr;
    }

    // 2. 현재 FCNS 노드에 해당하는 새로운 일반 트리 노드 생성
    GeneralTreeNode* general_node = new GeneralTreeNode(fcns_root->data);

    // 3. 자식 처리: FCNS 노드의 'leftChild' 포인터는 일반 트리의 첫 번째 자식을 가리킴.
    //    이 첫 번째 자식으로부터 'rightSibling' 포인터들을 따라가며 모든 형제들을 찾아 자식 목록에 추가
    
    FCNS_BinaryTreeNode* current_fcns_child = fcns_root->leftChild; // 첫 번째 자식부터 시작

    while (current_fcns_child != nullptr) {
        // 현재 FCNS 자식 노드(및 그 하위 트리)를 재귀적으로 변환
        GeneralTreeNode* child_node = convertFCNS_to_General(current_fcns_child);
        
        // 변환된 자식 노드를 현재 general_node의 자식 목록에 추가
        general_node->addChild(child_node);

        // FCNS 표현에서 다음 형제로 이동
        current_fcns_child = current_fcns_child->rightSibling;
    }

    // 4. 새로 생성된 일반 트리 노드 반환
    return general_node;
}
```

이 변환 과정의 시간 복잡도는 $O(N)$ 입니다. 각 FCNS 이진 트리 노드가 정확히 한 번 방문되어 일반 트리 노드를 생성하고, 각 `leftChild`와 `rightSibling` 링크도 한 번씩만 따라가기 때문입니다. 공간 복잡도 또한 $O(N)$이며, 이는 새로운 일반 트리 노드를 저장하고 재귀 호출 스택(최악의 경우 트리의 높이, 즉 $O(N)$)을 위한 공간입니다.

### **구체적 예시**

슬라이드의 오른쪽 **변환된 이진 트리**에서 왼쪽 **일반 트리**로 복원하는 과정을 `convertFCNS_to_General` 함수를 통해 추적해봅시다.

1.  `convertFCNS_to_General('a')` 호출:
    *   `general_node_a` 생성.
    *   `a`의 `leftChild`는 `b`입니다 (빨간 선).
    *   `current_fcns_child`가 `b`가 됩니다.
2.  `while (b != nullptr)` 루프 시작:
    *   `convertFCNS_to_General('b')` 호출:
        *   `general_node_b` 생성.
        *   `b`의 `leftChild`는 `d`입니다 (빨간 선).
        *   `current_fcns_child_of_b`가 `d`가 됩니다.
        *   `while (d != nullptr)` 루프 시작:
            *   `convertFCNS_to_General('d')` 호출: `general_node_d` 생성. `d`는 `leftChild`가 없으므로 자식 없음. `general_node_d` 반환.
            *   `general_node_b`에 `general_node_d`를 자식으로 추가.
            *   `d`의 `rightSibling`은 `e`입니다 (초록 선). `current_fcns_child_of_b`가 `e`가 됩니다.
        *   `while (e != nullptr)` 루프 계속:
            *   `convertFCNS_to_General('e')` 호출: `general_node_e` 생성. `e`는 `leftChild`가 없으므로 자식 없음. `general_node_e` 반환.
            *   `general_node_b`에 `general_node_e`를 자식으로 추가.
            *   `e`의 `rightSibling`은 `f`입니다 (초록 선). `current_fcns_child_of_b`가 `f`가 됩니다.
        *   `while (f != nullptr)` 루프 계속:
            *   `convertFCNS_to_General('f')` 호출: `general_node_f` 생성. `f`는 `leftChild`가 없으므로 자식 없음. `general_node_f` 반환.
            *   `general_node_b`에 `general_node_f`를 자식으로 추가.
            *   `f`의 `rightSibling`은 `nullptr`입니다. `current_fcns_child_of_b`가 `nullptr`가 됩니다.
        *   `while` 루프 종료. `general_node_b`는 자식 `d`, `e`, `f`를 가집니다. `general_node_b` 반환.
    *   `general_node_a`에 `general_node_b`를 자식으로 추가.
    *   `b`의 `rightSibling`은 `c`입니다 (초록 선). `current_fcns_child`가 `c`가 됩니다.
3.  `while (c != nullptr)` 루프 계속:
    *   `convertFCNS_to_General('c')` 호출: `general_node_c` 생성. `c`는 `leftChild`가 없으므로 자식 없음. `general_node_c` 반환.
    *   `general_node_a`에 `general_node_c`를 자식으로 추가.
    *   `c`의 `rightSibling`은 `nullptr`입니다. `current_fcns_child`가 `nullptr`가 됩니다.
4.  `while` 루프 종료. `general_node_a`는 자식 `b`, `c`를 가집니다. 최종적으로 재구성된 일반 트리의 루트 `general_node_a`가 반환됩니다.

이 과정은 슬라이드 왼쪽에 제시된 일반 트리 구조와 정확히 일치합니다.

### **시험 포인트**

*   ⭐ **First Child, Next Sibling (FCNS) 이진 트리 변환 규칙을 정확히 이해하고 있어야 합니다.**
    *   `leftChild` (빨간 선)는 첫 번째 자식을 가리키고, `rightSibling` (초록 선)은 다음 형제를 가리킨다는 것을 숙지하세요.
*   ⭐ **주어진 일반 트리를 FCNS 이진 트리로 그리거나, 반대로 FCNS 이진 트리를 일반 트리로 복원할 수 있어야 합니다.** 슬라이드의 그림과 같이 구체적인 노드를 이용한 변환/복원 문제가 자주 출제됩니다.
*   ⭐ FCNS 이진 트리에서 일반 트리로 복원하는 **알고리즘의 논리적 흐름 (재귀적 처리 방식)**을 이해하고 설명할 수 있어야 합니다. 특히 `leftChild`를 통해 첫 자식을 찾고, `rightSibling`을 통해 나머지 형제들을 찾아 자식 목록을 구성하는 부분이 중요합니다.
*   FCNS 변환 및 복원 과정의 **시간 및 공간 복잡도**를 분석할 수 있어야 합니다. (대부분 $O(N)$)

## Slide 32

## 데이터구조 (CSED233) 슬라이드 노트: 트리 요약

### 핵심 개념

본 슬라이드는 트리에 대한 핵심 개념들을 요약하고 있습니다. 트리는 계층적 데이터를 표현하는 데 사용되는 비선형 자료구조로, 각 노드는 0개 이상의 자식 노드를 가질 수 있습니다. 특히 이 슬라이드에서는 트리의 탐색 방법, 재구성, 일반적인 트리의 표현 방식, 그리고 트리 연산의 시간 복잡도와 구현 방식에 대해 다룹니다.

### 1. Tree traversals (트리 순회)

**핵심 개념**: 트리에 있는 모든 노드를 체계적으로 방문하는 방법입니다. 주로 다음 세 가지 순회 방식이 있습니다:
*   **Preorder Traversal (전위 순회)**: (1) 현재 노드 방문 → (2) 왼쪽 서브트리 순회 → (3) 오른쪽 서브트리 순회
*   **Inorder Traversal (중위 순회)**: (1) 왼쪽 서브트리 순회 → (2) 현재 노드 방문 → (3) 오른쪽 서브트리 순회
*   **Postorder Traversal (후위 순회)**: (1) 왼쪽 서브트리 순회 → (2) 오른쪽 서브트리 순회 → (3) 현재 노드 방문
이 외에도 레벨 단위로 순회하는 **Level-order Traversal (레벨 순회)**이 있으며, BFS(너비 우선 탐색) 기반으로 구현됩니다.

**코드/수식 해설**:
```cpp
// 이진 트리의 전위 순회 (Preorder Traversal)
void preorder(Node* node) {
    if (node == nullptr) {
        return;
    }
    visit(node->data);    // 현재 노드 처리
    preorder(node->left); // 왼쪽 자식으로 이동
    preorder(node->right);// 오른쪽 자식으로 이동
}

// 이진 트리의 중위 순회 (Inorder Traversal)
void inorder(Node* node) {
    if (node == nullptr) {
        return;
    }
    inorder(node->left);  // 왼쪽 자식으로 이동
    visit(node->data);    // 현재 노드 처리
    inorder(node->right); // 오른쪽 자식으로 이동
}

// 이진 트리의 후위 순회 (Postorder Traversal)
void postorder(Node* node) {
    if (node == nullptr) {
        return;
    }
    postorder(node->left);   // 왼쪽 자식으로 이동
    postorder(node->right);  // 오른쪽 자식으로 이동
    visit(node->data);       // 현재 노드 처리
}
```

**구체적 예시**:
다음 이진 트리가 있다고 가정합니다:
```
      A
     / \
    B   C
   / \
  D   E
```
*   **Preorder**: A, B, D, E, C (부모 먼저)
*   **Inorder**: D, B, E, A, C (왼쪽 → 부모 → 오른쪽, 이진 탐색 트리에서는 정렬된 순서로 나옴)
*   **Postorder**: D, E, B, C, A (자식들 먼저, 부모 마지막. 트리를 삭제할 때 유용)

**시험 포인트**: ⭐ 각 순회 방식의 개념을 정확히 이해하고 주어진 트리에 대해 올바른 순회 결과를 도출할 수 있어야 합니다. ⭐ 각 순회 방식이 특정 문제(예: 이진 탐색 트리에서 정렬된 출력, 식 표현 트리 계산)에서 어떻게 활용되는지 알아두세요.

### 2. Binary tree reconstruction (이진 트리 재구성)

**핵심 개념**: 두 가지 순회 결과 (예: 중위 순회와 전위 순회 또는 중위 순회와 후위 순회)를 이용하여 원래의 유일한 이진 트리를 재구성하는 과정입니다. 일반적으로 중위 순회 결과는 노드의 상대적인 순서를, 전위/후위 순회 결과는 부모 노드의 위치 정보를 제공합니다.

**구체적 예시**:
주어진 순회 결과:
*   Preorder: `A B D E C`
*   Inorder: `D B E A C`

1.  Preorder의 첫 번째 원소 `A`가 루트 노드임을 알 수 있습니다.
2.  Inorder에서 `A`의 위치를 찾습니다. `D B E`는 `A`의 왼쪽 서브트리에 속하고, `C`는 `A`의 오른쪽 서브트리에 속합니다.
3.  Preorder에서 `D B E`와 `C`를 찾습니다. Preorder의 `B D E`는 왼쪽 서브트리의 Preorder, `C`는 오른쪽 서브트리의 Preorder입니다.
4.  왼쪽 서브트리 `(Pre: B D E, In: D B E)`를 재귀적으로 구성합니다. `B`가 루트, `D`가 왼쪽, `E`가 오른쪽이 됩니다.
5.  오른쪽 서브트리 `(Pre: C, In: C)`를 재귀적으로 구성합니다. `C`가 루트가 됩니다.
결과적으로 위에 제시된 예시 트리가 재구성됩니다.

**시험 포인트**: ⭐ 전위/중위 순회 또는 후위/중위 순회 결과가 주어졌을 때 이진 트리를 재구성하는 과정을 단계별로 설명하거나 직접 재구성할 수 있어야 합니다.

### 3. General k-ary tree (일반 k-아리 트리)

**핵심 개념**: 각 노드가 가질 수 있는 자식 노드의 개수에 제한이 없는 일반적인 트리 구조를 의미합니다. 'k-ary'는 명칭일 뿐, k가 특정 상수가 아닐 수 있으며, 각 노드는 0개부터 임의의 개수(최대 N-1개)의 자식을 가질 수 있습니다. 이진 트리는 k=2인 k-ary 트리의 특정 경우입니다.

**시험 포인트**: ⭐ 이진 트리와 일반 트리의 차이점을 설명할 수 있어야 합니다. 일반 트리의 표현 방식이 왜 이진 트리와 다른지 이해해야 합니다.

### 4. List-of-Children representation (자식 리스트 표현)

**핵심 개념**: 일반 k-ary 트리를 구현하는 한 가지 방법으로, 각 노드가 자신의 모든 자식 노드들을 가리키는 포인터 리스트(또는 배열)를 가지는 방식입니다.

**코드/수식 해설**:
```cpp
struct Node {
    int data;
    std::vector<Node*> children; // 자식 노드들을 저장하는 리스트
};
```

**구체적 예시**:
노드 A가 B, C, D 세 개의 자식을 가지고 있다면, A 노드의 `children` 벡터는 B, C, D 노드를 가리키는 포인터들을 포함하게 됩니다.

**시험 포인트**: ⭐ List-of-Children 표현 방식의 개념과 장단점을 이해해야 합니다 (장점: 자식 추가/삭제 용이, 단점: 노드당 가변적인 메모리 오버헤드, 자식 탐색에 $O(degree)$ 시간 소요).

### 5. Left-child / right-sibling representation (왼쪽 자식 / 오른쪽 형제 표현)

**핵심 개념**: 일반 k-ary 트리를 이진 트리와 동일한 구조로 변환하여 표현하는 방식입니다. 각 노드는 두 개의 포인터를 가집니다:
*   **Left-child (왼쪽 자식)**: 해당 노드의 첫 번째 자식을 가리킵니다.
*   **Right-sibling (오른쪽 형제)**: 해당 노드의 바로 오른쪽 형제 노드를 가리킵니다.
이 방식은 일반 트리를 이진 트리의 구조로 "변환"하므로, 이진 트리에 적용되는 많은 알고리즘을 일반 트리에 적용할 수 있게 해줍니다.

**코드/수식 해설**:
```cpp
struct Node {
    int data;
    Node* firstChild; // 첫 번째 자식 노드를 가리킴
    Node* nextSibling; // 다음 형제 노드를 가리킴
};
```

**구체적 예시**:
```
      A
     /|\
    B C D
   / \
  E   F
```
이 일반 트리는 Left-child / right-sibling 표현으로 다음과 같이 변환됩니다:
*   A: `firstChild` -> B, `nextSibling` -> `nullptr`
*   B: `firstChild` -> E, `nextSibling` -> C
*   C: `firstChild` -> `nullptr`, `nextSibling` -> D
*   D: `firstChild` -> `nullptr`, `nextSibling` -> `nullptr`
*   E: `firstChild` -> `nullptr`, `nextSibling` -> F
*   F: `firstChild` -> `nullptr`, `nextSibling` -> `nullptr`

이러한 변환 후, A의 `firstChild`를 따라가면 B에, B의 `nextSibling`을 따라가면 C에, C의 `nextSibling`을 따라가면 D에 도달하는 식입니다. 이는 마치 A가 B만 자식으로 가지고, B가 E를 자식으로 가지고 C를 형제로 가지며, C는 D를 형제로 가지는 이진 트리처럼 보입니다.

**시험 포인트**: ⭐ Left-child / right-sibling 표현 방식이 일반 트리를 어떻게 이진 트리 구조로 변환하는지 그림과 함께 설명할 수 있어야 합니다. ⭐ 이 방식의 장점(이진 트리 알고리즘 활용)과 단점(직접적인 자식 접근의 비효율성)을 이해해야 합니다.

### 6. Time complexity of tree operations (트리 연산의 시간 복잡도)

**핵심 개념**: 트리에 대한 기본적인 연산(탐색, 삽입, 삭제, 순회)을 수행하는 데 걸리는 시간입니다. 트리의 종류(이진 탐색 트리, 힙, AVL 트리 등)와 구조(균형 vs. 불균형)에 따라 복잡도가 달라집니다.
*   **순회 (Traversal)**: 모든 노드를 한 번씩 방문하므로 일반적으로 트리의 노드 수 $N$에 비례하여 $O(N)$의 시간이 소요됩니다.
*   **탐색, 삽입, 삭제**:
    *   **불균형 이진 탐색 트리 (Worst Case)**: 트리가 한쪽으로 기울어진 경우 (연결 리스트와 유사), 트리의 높이 $h$가 $N$이 될 수 있어 $O(N)$이 됩니다.
    *   **균형 이진 탐색 트리 (Average Case)**: AVL 트리나 Red-Black 트리와 같이 균형을 유지하는 트리에서는 트리의 높이가 $O(\log N)$이므로, 탐색, 삽입, 삭제 모두 $O(\log N)$의 시간이 소요됩니다.
    *   **힙 (Heap)**: 삽입/삭제는 $O(\log N)$, 최대/최소 원소 접근은 $O(1)$.
    *   **일반 트리**: 특정 노드의 자식 탐색은 List-of-Children 방식에서는 자식 수에 비례, Left-child / Right-sibling 방식에서는 형제 노드를 따라가야 하므로 최악의 경우 $O(N)$이 될 수 있습니다.

**수식 해설**:
트리의 높이를 $h$라고 할 때,
*   완전 이진 트리의 경우 $h = \log_2 N$ ($N$은 노드 수)
*   편향 트리의 경우 $h = N$

**시험 포인트**: ⭐ 각 트리 연산(탐색, 삽입, 삭제, 순회)에 대한 일반적인 시간 복잡도를 숙지하고, ⭐ 트리의 균형 여부가 시간 복잡도에 미치는 영향을 설명할 수 있어야 합니다. ⭐ 특정 자료구조(예: 이진 탐색 트리, 힙)에서의 연산별 시간 복잡도를 명확히 구분하여 알아야 합니다.

### 7. Knuth transform (크누스 변환)

**핵심 개념**: Knuth transform은 일반적으로 "Left-child / Right-sibling representation"과 동일한 개념을 지칭하며, Donal Knuth가 "The Art of Computer Programming"에서 소개한 방식입니다. 일반 트리를 이진 트리 형태로 변환하여 표현함으로써, 일반 트리를 다루는 복잡성을 줄이고 이진 트리를 위한 기존 알고리즘을 활용할 수 있게 하는 강력한 기법입니다.

**시험 포인트**: ⭐ Knuth transform이 Left-child / Right-sibling representation과 같은 개념임을 이해하고, 이 변환의 목적과 효과 (일반 트리를 이진 트리로 변환하여 알고리즘 적용 용이)를 설명할 수 있어야 합니다.

### 8. Static & dynamic tree implementations (정적 및 동적 트리 구현)

**핵심 개념**: 트리를 메모리에 저장하고 관리하는 방식에 따라 정적 구현과 동적 구현으로 나눌 수 있습니다.

*   **Static Tree Implementation (정적 트리 구현)**:
    *   **특징**: 배열(array)을 기반으로 트리를 구현합니다. 주로 완전 이진 트리나 힙과 같이 노드의 인덱스만으로 부모/자식 관계를 계산할 수 있는 특정 형태의 트리에서 효율적입니다.
    *   **장점**:
        *   메모리 할당 및 해제 오버헤드가 없습니다.
        *   메모리 접근이 연속적이어서 캐시 효율이 좋습니다.
        *   포인터 오버헤드가 없습니다.
    *   **단점**:
        *   트리의 최대 크기를 미리 알아야 합니다 (고정 크기).
        *   삽입/삭제 시 노드의 이동이 발생할 수 있어 비효율적일 수 있습니다.
        *   공간 낭비가 발생할 수 있습니다 (sparse tree의 경우).
    *   **코드/수식 해설**: 노드 `i`의 왼쪽 자식은 `2i+1`, 오른쪽 자식은 `2i+2`, 부모는 `(i-1)/2` (0-indexed)와 같은 수식으로 관계를 계산합니다.

*   **Dynamic Tree Implementation (동적 트리 구현)**:
    *   **특징**: 포인터(pointer)를 사용하여 노드들을 연결하는 방식으로 구현합니다. 각 노드는 자식 노드를 가리키는 포인터를 가집니다. 일반적인 이진 탐색 트리, AVL 트리, Red-Black 트리 등에서 사용됩니다.
    *   **장점**:
        *   트리의 크기가 동적으로 변할 수 있습니다 (유연성).
        *   삽입/삭제 시 노드 이동 없이 포인터만 변경하면 되므로 효율적입니다.
        *   필요한 만큼만 메모리를 사용합니다.
    *   **단점**:
        *   메모리 할당 및 해제 오버헤드가 있습니다.
        *   포인터 저장을 위한 추가 메모리 오버헤드가 있습니다.
        *   메모리 접근이 비연속적이어서 캐시 효율이 떨어질 수 있습니다.
    *   **코드/수식 해설**:
        ```cpp
        struct Node {
            int data;
            Node* left;
            Node* right;
        };
        ```

**구체적 예시**:
*   **정적 구현**: 힙(Heap)은 완전 이진 트리의 특성을 가지므로 배열로 구현하기 매우 적합합니다.
*   **동적 구현**: 대부분의 이진 탐색 트리(BST)는 노드의 삽입/삭제가 빈번하게 일어나므로 포인터를 이용한 동적 구현이 일반적입니다.

**시험 포인트**: ⭐ 정적 및 동적 트리 구현 방식의 장점과 단점을 비교하여 설명하고, ⭐ 특정 트리 자료구조(예: 힙 vs. BST)에 어떤 구현 방식이 더 적합한지 이유와 함께 설명할 수 있어야 합니다. ⭐ 각 구현 방식에서의 노드 접근 및 관계 표현 방법을 이해해야 합니다.

## [CSED233-01] Data Structure Algorithm Analysis

---

### **핵심 개념**

이 슬라이드는 데이터 구조 강의의 첫 번째 주제인 **알고리즘 분석(Algorithm Analysis)**의 도입 부분입니다. 여기서 다루는 핵심 개념은 알고리즘의 효율성을 평가하는 방법이며, 특히 **시간 복잡도(Time Complexity)**를 주요 지표로 사용합니다. 알고리즘의 시간 복잡도는 입력 데이터의 크기($n$)가 증가함에 따라 알고리즘이 작업을 완료하는 데 필요한 연산 횟수가 얼마나 증가하는지를 나타냅니다. 그래프는 다양한 시간 복잡도 함수($O(1), O(\log n), O(n), O(n^2), O(n^3), O(n!)$)를 시각적으로 비교하여 입력 크기에 따른 성능 변화를 보여줍니다.

### **코드/수식 해설**

이 슬라이드에서는 알고리즘의 점근적 상한을 나타내는 **Big-O 표기법**을 중심으로 설명합니다. Big-O 표기법 $O(g(n))$은 $n$이 충분히 커질 때 알고리즘의 실행 시간이 함수 $g(n)$에 비례하거나 그보다 빠르게 증가하지 않음을 의미합니다. 즉, 알고리즘의 **최악의 경우(Worst-case) 성능**을 나타내는 지표로 주로 사용됩니다.

그래프에 표시된 주요 Big-O 복잡도들은 다음과 같습니다:

1.  **$O(1)$ (상수 시간)**: 입력 크기 $n$에 관계없이 연산 횟수가 일정합니다.
    ```latex
    f(n) = c \text{ (where c is a constant)}
    ```
2.  **$O(\log n)$ (로그 시간)**: 입력 크기가 두 배가 될 때 연산 횟수가 미미하게 증가합니다 (예: 1회).
    ```latex
    f(n) = c \log n
    ```
3.  **$O(n)$ (선형 시간)**: 입력 크기 $n$에 비례하여 연산 횟수가 증가합니다.
    ```latex
    f(n) = c n
    ```
4.  **$O(n^2)$ (제곱 시간)**: 입력 크기 $n$의 제곱에 비례하여 연산 횟수가 증가합니다.
    ```latex
    f(n) = c n^2
    ```
5.  **$O(n^3)$ (세제곱 시간)**: 입력 크기 $n$의 세제곱에 비례하여 연산 횟수가 증가합니다.
    ```latex
    f(n) = c n^3
    ```
6.  **$O(n!)$ (팩토리얼 시간)**: 입력 크기 $n$의 팩토리얼에 비례하여 연산 횟수가 폭발적으로 증가합니다. 매우 비효율적인 알고리즘을 나타냅니다.
    ```latex
    f(n) = c n!
    ```

그래프에서 볼 수 있듯이, $n$ 값이 커질수록 $O(1) < O(\log n) < O(n) < O(n^2) < O(n^3) < O(n!)$ 순으로 시간 복잡도가 급격히 증가합니다. 즉, 곡선이 아래쪽에 있을수록 더 효율적인 알고리즘입니다.

### **구체적 예시**

*   **$O(1)$ (상수 시간)**: 배열에서 특정 인덱스(예: `array[5]`)의 요소를 접근하는 경우. 입력 크기(배열의 크기)가 아무리 커져도 한 번의 연산으로 값을 찾을 수 있습니다.
*   **$O(\log n)$ (로그 시간)**: 정렬된 배열에서 이진 탐색(Binary Search)을 수행하는 경우. 탐색할 때마다 탐색 범위가 절반으로 줄어들어 필요한 연산 횟수가 입력 크기의 로그에 비례합니다.
*   **$O(n)$ (선형 시간)**: 배열의 모든 요소를 한 번씩 순회하며 특정 값을 찾는 선형 탐색(Linear Search)을 수행하는 경우. 입력 크기가 $n$이면 최대 $n$번의 비교 연산이 필요합니다.
    ```c
    int linearSearch(int arr[], int n, int target) {
        for (int i = 0; i < n; i++) {
            if (arr[i] == target) {
                return i; // Found
            }
        }
        return -1; // Not found
    }
    ```
*   **$O(n^2)$ (제곱 시간)**: 버블 정렬(Bubble Sort)과 같이 중첩된 두 개의 반복문을 사용하여 배열을 정렬하는 경우. 외부 루프가 $n$번, 내부 루프가 최대 $n$번 반복되어 $n \times n = n^2$에 비례하는 연산이 필요합니다.
    ```c
    void bubbleSort(int arr[], int n) {
        for (int i = 0; i < n - 1; i++) {
            for (int j = 0; j < n - i - 1; j++) {
                if (arr[j] > arr[j+1]) {
                    // swap arr[j] and arr[j+1]
                    int temp = arr[j];
                    arr[j] = arr[j+1];
                    arr[j+1] = temp;
                }
            }
        }
    }
    ```
*   **$O(n!)$ (팩토리얼 시간)**: 외판원 문제(Traveling Salesman Problem)를 무차별 대입(brute force) 방식으로 해결할 때 모든 가능한 경로를 탐색하는 경우. 경로의 수가 입력 도시의 수에 대한 팩토리얼로 증가하여 매우 빠르게 비효율적이 됩니다.

### **시험 포인트**

*   ⭐ **Big-O 표기법의 정확한 정의와 의미**를 이해하고 설명할 수 있어야 합니다. (상한을 나타낸다는 점이 중요)
*   ⭐ **각 주요 시간 복잡도($O(1), O(\log n), O(n), O(n^2), O(n^3), O(n!)$)의 상대적인 성능 차이**를 그래프처럼 시각적으로 인지하고, $n$ 값이 커질수록 어떤 복잡도가 더 나쁜 성능을 가지는지 순서대로 나열할 수 있어야 합니다.
*   ⭐ **주어진 간단한 코드 스니펫이나 알고리즘의 시간 복잡도를 분석**하여 Big-O 표기법으로 표현할 수 있어야 합니다. (반복문의 중첩 여부, 재귀 함수의 깊이 등이 핵심)
*   ⭐ 알고리즘 분석 시 **최악(Worst-case), 최선(Best-case), 평균(Average-case) 시간 복잡도**의 개념을 구분하고, Big-O 표기법이 주로 최악의 경우를 나타내는 이유를 설명할 수 있어야 합니다.

**핵심 개념**

*   **컴파일러 (Compiler) 및 링커 (Linker)**:
    *   **컴파일러**: C++ 소스 코드를 컴퓨터가 이해할 수 있는 기계어 코드를 포함하는 오브젝트 파일로 변환합니다. 문법 오류를 검사하고, 코드를 최적화하는 역할을 합니다.
    *   **링커**: 여러 오브젝트 파일들과 필요한 라이브러리(예: `iostream` 같은 표준 라이브러리)를 묶어 하나의 실행 가능한 프로그램(실행 파일)을 생성합니다.
*   **변수 (Variable)**: 데이터를 저장하기 위한 메모리 공간에 붙여진 이름입니다. 프로그램 실행 중 값을 변경할 수 있습니다. (예: `int sum;`)
*   **배열 (Array)**: 동일한 타입의 여러 데이터를 메모리에 연속적으로 저장하는 자료구조입니다. 인덱스(0부터 시작)를 통해 각 요소에 직접 접근할 수 있습니다.
*   **포인터 (Pointer)**: 메모리 주소를 저장하는 변수입니다. 다른 변수의 메모리 위치를 가리키는 데 사용되며, 동적 메모리 관리와 연결 리스트 같은 복잡한 자료구조 구현에 필수적입니다.
*   **동적 메모리 (Dynamic Memory)**: 프로그램 실행 중에 필요한 만큼 메모리를 할당하고 해제하는 기능입니다. `new` 연산자로 할당하고, `delete` 연산자로 해제합니다. 이를 통해 유연한 크기의 자료구조를 만들 수 있습니다.
*   **반복문 (Loop)**: 특정 코드 블록을 여러 번 반복 실행하는 제어 구조입니다. `for` 루프, `while` 루프, `do-while` 루프 등이 있습니다.
*   **`break` 및 `continue` 문**:
    *   **`break`**: 현재 실행 중인 가장 가까운 반복문(for, while, do-while) 또는 `switch` 문을 즉시 종료하고 다음 문장으로 넘어갑니다.
    *   **`continue`**: 현재 반복문의 나머지 부분을 건너뛰고, 다음 반복을 시작합니다. (반복 조건 검사 단계로 이동)

**코드/수식 해설**

아래 C++ 코드는 배열의 일부 요소를 순회하며 합계를 계산하는 예시입니다. 배열 내에 음수가 발견되면 반복을 중단하는 `break` 문이 사용되었습니다.

```cpp
int a[10] = {1,2,-3,-4,-5,5,4,3,-2,-1};
int sum = 0;

for (int i = 0; i < 10; i++) {
    if (a[i] < 0) break;
    sum += a[i];
}

cout << "sum : " << sum << endl;
```

1.  `int a[10] = {1,2,-3,-4,-5,5,4,3,-2,-1};`: `int` 타입의 10개 요소를 가질 수 있는 `a`라는 이름의 배열을 선언하고 초기화합니다. 배열의 인덱스는 0부터 9까지입니다.
2.  `int sum = 0;`: 정수형 변수 `sum`을 선언하고 0으로 초기화합니다. 이 변수는 배열 요소들의 합을 저장할 것입니다.
3.  `for (int i = 0; i < 10; i++) { ... }`: `i`를 0부터 시작하여 9까지 1씩 증가시키면서 반복합니다. 즉, 배열의 모든 요소를 순서대로 접근합니다.
4.  `if (a[i] < 0) break;`: 현재 배열 요소 `a[i]`의 값이 0보다 작은지(음수인지) 검사합니다. 만약 음수라면, `break` 문이 실행되어 `for` 루프를 즉시 종료합니다.
5.  `sum += a[i];`: `a[i]`가 음수가 아니라면 (즉, `if` 조건이 거짓이라면), 현재 `a[i]`의 값을 `sum`에 더합니다.
6.  `cout << "sum : " << sum << endl;`: 루프가 종료된 후, 최종 `sum` 값을 콘솔에 출력합니다.

**구체적 예시**

위 코드의 실행 흐름을 단계별로 추적해 봅시다.

*   `a` 배열: `{1, 2, -3, -4, -5, 5, 4, 3, -2, -1}`
*   `sum` 초기값: `0`

1.  `i = 0`:
    *   `a[0]`는 `1`입니다.
    *   `1 < 0`은 거짓입니다.
    *   `sum = 0 + 1 = 1`.
2.  `i = 1`:
    *   `a[1]`는 `2`입니다.
    *   `2 < 0`은 거짓입니다.
    *   `sum = 1 + 2 = 3`.
3.  `i = 2`:
    *   `a[2]`는 `-3`입니다.
    *   `-3 < 0`은 참입니다.
    *   `break` 문이 실행되어 `for` 루프가 즉시 종료됩니다.

루프가 종료된 후, `cout` 문이 실행되어 최종 `sum` 값인 `3`이 출력됩니다.
출력 결과: `sum : 3`

**시험 포인트**

*   ⭐ **C++ 기본 문법의 숙지**: 변수 선언, 배열 초기화 및 접근, `for` 루프, `if` 조건문 등은 자료구조를 구현하는 데 필수적이므로 정확히 이해하고 있어야 합니다.
*   ⭐ **`break`와 `continue` 문의 정확한 동작 방식**: 특히 반복문 제어 흐름에 미치는 영향을 정확히 파악해야 합니다. `break`는 루프를 완전히 탈출하고, `continue`는 현재 반복만 건너뛰고 다음 반복으로 넘어갑니다.
*   ⭐ **배열의 인덱스 접근**: 배열은 0부터 시작하는 인덱스를 사용하며, 인덱스 범위를 벗어나면 런타임 오류(undefined behavior)가 발생할 수 있음을 유의해야 합니다.
*   ⭐ **포인터 및 동적 메모리**: 이 슬라이드에는 직접적인 코드가 없지만, 연결 리스트, 트리 등 여러 자료구조에서 포인터와 동적 메모리(`new`, `delete`)가 핵심적으로 사용되므로 이 개념들을 철저히 복습하고 이해해야 합니다. 이들이 메모리를 어떻게 관리하는지, 메모리 누수(memory leak)를 방지하는 방법 등을 알아두면 좋습니다.

## Slide 3

**핵심 개념**

*   **알고리즘의 정의**: 알고리즘은 특정 문제를 해결하기 위한 "단계별(step-by-step) 절차"이며, 이 절차는 "유한한 시간(finite amount of time)" 내에 완료되어야 합니다. 이는 단순히 문제를 해결하는 것을 넘어, 해결 가능성(solvability)과 효율성(efficiency)을 동시에 고려해야 함을 의미합니다.
*   **효율적인 알고리즘의 필요성**: 슬라이드의 마지막 질문("Can you think about a better algorithm?")은 주어진 문제를 해결하는 것을 넘어, 더 효율적이거나 개선된 방법을 탐색하는 것이 중요함을 암시합니다. 이는 데이터 구조 강의에서 다루는 시간/공간 복잡도 분석의 기초적인 동기가 됩니다.

**코드/수식 해설**

첨부된 C++ 코드는 배열의 요소들을 순회하며 합계를 계산하되, 첫 번째 음수 값을 만나면 더 이상 합계를 계산하지 않고 반복을 중단하는 알고리즘을 구현합니다.

```cpp
int a[10] = {1,2,-3,-4,-5,5,4,3,-2,-1}; // 정수형 배열 'a'를 선언하고 초기화
int sum = 0; // 합계를 저장할 변수 'sum'을 0으로 초기화

for (int i = 0; i < 10; i++) { // 인덱스 'i'를 0부터 9까지 1씩 증가시키며 반복
    if (a[i] < 0) break; // 현재 배열 요소 'a[i]'가 음수이면 반복을 즉시 중단
    sum += a[i]; // 'a[i]'가 음수가 아니면 'sum'에 'a[i]' 값을 더함
}

cout << "sum : " << sum << endl; // 최종 'sum' 값을 출력
```

*   **배열 초기화**: `int a[10] = {1,2,-3,-4,-5,5,4,3,-2,-1};`는 10개의 정수 요소를 가진 배열 `a`를 선언하고 지정된 값들로 초기화합니다.
*   **합계 변수**: `int sum = 0;`은 배열의 양수 값들을 누적할 변수 `sum`을 선언하고 0으로 초기화합니다.
*   **`for` 루프**: `for (int i = 0; i < 10; i++)`는 배열의 첫 번째 요소(인덱스 0)부터 마지막 요소(인덱스 9)까지 순차적으로 접근합니다.
*   **조건문과 `break`**: `if (a[i] < 0) break;`는 현재 검사하는 `a[i]` 값이 음수인지 확인합니다. 만약 음수이면 `break` 문이 실행되어 `for` 루프가 즉시 종료됩니다. 이는 배열의 나머지 요소들을 더 이상 검사하지 않음을 의미합니다.
*   **합계 누적**: `sum += a[i];`는 `a[i]`가 음수가 아닐 경우에만 `sum` 변수에 `a[i]`의 값을 더합니다.
*   **결과 출력**: `cout << "sum : " << sum << endl;`는 최종 계산된 `sum` 값을 콘솔에 출력합니다.

**구체적 예시**

주어진 배열 `a = {1, 2, -3, -4, -5, 5, 4, 3, -2, -1}`에 대해 위 코드가 어떻게 동작하는지 살펴보겠습니다.

1.  `sum = 0`으로 초기화됩니다.
2.  `i = 0`: `a[0]`은 `1`입니다. `1 < 0`은 거짓이므로, `sum = 0 + 1 = 1`이 됩니다.
3.  `i = 1`: `a[1]`은 `2`입니다. `2 < 0`은 거짓이므로, `sum = 1 + 2 = 3`이 됩니다.
4.  `i = 2`: `a[2]`은 `-3`입니다. `-3 < 0`은 참이므로, `break` 문이 실행됩니다.
5.  `for` 루프가 즉시 종료되고, 현재 `sum` 값인 `3`이 출력됩니다.

따라서 이 코드의 최종 출력은 "sum : 3"이 됩니다.

**시험 포인트**

*   **알고리즘의 정의와 특징**: 알고리즘이 '단계별 절차'이며 '유한한 시간' 내에 문제를 해결해야 한다는 정의를 정확히 이해하고 설명할 수 있어야 합니다. ⭐
*   **배열 순회 및 조건부 합계 계산**: 주어진 조건(음수 발견 시 중단)에 따라 배열 요소를 순회하며 합계를 계산하는 코드를 작성하거나 분석할 수 있어야 합니다. ⭐
*   **`break` 문 사용**: 반복문(loop) 내에서 특정 조건이 만족될 때 반복을 즉시 중단시키는 `break` 문의 정확한 사용법과 동작 방식을 이해해야 합니다. ⭐
*   **코드 추적 및 결과 예측**: 주어진 코드와 입력 값에 대해 실행 과정을 머릿속으로 추적하여 최종 결과를 정확히 예측할 수 있는 능력이 중요합니다. ⭐
*   **알고리즘의 효율성 고려**: "Can you think about a better algorithm?" 질문은 단순한 문제 해결을 넘어 시간/공간 복잡도 측면에서 더 효율적인 접근 방식이 무엇인지 고민하게 합니다. 이 슬라이드 자체는 복잡도 분석을 직접 다루진 않지만, 이 질문을 통해 앞으로 배우게 될 시간/공간 복잡도 분석의 필요성을 이해하는 것이 중요합니다. ⭐

## Slide 4

### **핵심 개념**

본 슬라이드는 **알고리즘 분석 (Algorithm Analysis)**의 필요성을 제기합니다. 동일한 문제를 해결하는 여러 알고리즘이 있을 때, 어떤 알고리즘이 더 효율적인지, 즉 "더 좋은지"를 객관적으로 판단하기 위한 기준과 방법을 탐구하는 것이 핵심입니다. 단순히 코드를 실행해 보는 것만으로는 정확한 비교가 어려우며, 실행 환경(하드웨어, 운영체제, 컴파일러)이나 입력 데이터의 크기에 따라 결과가 달라질 수 있기 때문에, 이에 독립적인 평가 기준이 필요합니다.

### **코드/수식 해설**

알고리즘의 성능을 평가하는 주된 기준은 **시간 복잡도 (Time Complexity)**와 **공간 복잡도 (Space Complexity)**입니다. 이들은 입력 데이터의 크기($n$)에 따라 알고리즘이 소모하는 시간과 메모리 양이 어떻게 증가하는지를 나타냅니다. 주로 **점근적 표기법 (Asymptotic Notation)**을 사용하여 표현합니다.

가장 흔히 사용되는 점근적 표기법은 **빅-오(Big-O) 표기법**입니다.

*   **빅-오 표기법 ($O$)**: 알고리즘의 최악의 경우 (Upper Bound) 성능을 나타냅니다. 즉, 아무리 나빠도 이보다는 더 나쁘지 않다는 것을 보장합니다.

    어떤 함수 $T(n)$이 $O(g(n))$에 속한다는 것은, $n$이 충분히 커질 때 $T(n)$이 $g(n)$보다 빠르거나 같은 속도로 증가한다는 의미입니다. 수학적으로는 다음과 같이 정의됩니다.
    $$ T(n) \in O(g(n)) \text{ if there exist positive constants } c \text{ and } n_0 \text{ such that } 0 \le T(n) \le c \cdot g(n) \text{ for all } n \ge n_0 $$
    여기서 $T(n)$은 알고리즘의 실제 연산 횟수 또는 자원 사용량을 나타내고, $g(n)$은 비교 대상이 되는 간단한 함수(예: $n$, $\log n$, $n^2$)입니다.

### **구체적 예시**

두 가지 다른 검색 알고리즘을 통해 "어떤 알고리즘이 더 좋은지"를 판단하는 과정을 살펴봅시다.

**문제**: $N$개의 정수가 저장된 배열에서 특정 값 $X$를 찾는 알고리즘을 비교해 봅시다.

1.  **선형 탐색 (Linear Search)**
    *   **동작 방식**: 배열의 첫 번째 원소부터 마지막 원소까지 $X$와 비교하며 순차적으로 탐색합니다.
    *   **시간 복잡도**: 최악의 경우(배열 끝에 있거나 없는 경우) 모든 $N$개의 원소를 확인해야 합니다.
        $$ O(N) $$
    ```c
    int linear_search(int arr[], int n, int x) {
        for (int i = 0; i < n; i++) {
            if (arr[i] == x) {
                return i; // Found at index i
            }
        }
        return -1; // Not found
    }
    ```

2.  **이진 탐색 (Binary Search)** (단, 배열이 **정렬**되어 있어야 함)
    *   **동작 방식**: 배열의 중간 원소와 $X$를 비교하여, $X$가 작으면 왼쪽 절반, 크면 오른쪽 절반에서 다시 탐색합니다. 탐색 범위를 절반씩 줄여나갑니다.
    *   **시간 복잡도**: 탐색 범위가 절반씩 줄어들므로, $N$이 1이 될 때까지 $\log_2 N$번의 단계를 거칩니다.
        $$ O(\log N) $$
    ```c
    int binary_search(int arr[], int n, int x) {
        int low = 0;
        int high = n - 1;
        while (low <= high) {
            int mid = low + (high - low) / 2;
            if (arr[mid] == x) {
                return mid; // Found at index mid
            } else if (arr[mid] < x) {
                low = mid + 1;
            } else {
                high = mid - 1;
            }
        }
        return -1; // Not found
    }
    ```

**비교**:
*   $N=100$일 때, 선형 탐색은 최대 100번의 비교가 필요하지만, 이진 탐색은 약 $\log_2 100 \approx 7$번의 비교면 충분합니다.
*   $N=1,000,000$일 때, 선형 탐색은 최대 1,000,000번, 이진 탐색은 약 $\log_2 1,000,000 \approx 20$번의 비교가 필요합니다.

이처럼 입력 크기 $N$이 커질수록 $O(\log N)$ 알고리즘이 $O(N)$ 알고리즘보다 훨씬 효율적입니다. 이를 통해 이진 탐색이 선형 탐색보다 "더 좋은" 알고리즘이라고 판단할 수 있습니다 (물론, 이진 탐색은 배열이 정렬되어야 한다는 전제 조건이 있습니다).

### **시험 포인트**

*   ⭐ **알고리즘 분석의 필요성**: 왜 알고리즘의 효율성을 객관적으로 평가해야 하는지 (실행 환경, 입력 크기 등에 독립적인 평가) 설명할 수 있어야 합니다.
*   ⭐ **시간 복잡도와 공간 복잡도**: 두 가지 주요 성능 측정 기준을 정의하고 설명할 수 있어야 합니다.
*   ⭐ **점근적 표기법 (특히 빅-오 표기법)**: 알고리즘의 성능을 표현하는 데 사용되는 표기법의 의미와 용도를 정확히 이해하고 있어야 합니다. (최악의 경우, 상한)
*   ⭐ **"더 나은" 알고리즘의 의미**: 시간 복잡도가 낮은 알고리즘이 대규모 입력에서 더 효율적임을 이해하고, 간단한 알고리즘(예: 선형 탐색 vs 이진 탐색)의 시간 복잡도를 비교할 수 있어야 합니다.

## Comparing Algorithms

이 슬라이드는 두 개 이상의 알고리즘을 비교하고 평가하는 방법에 대한 서론을 다룹니다. 특히 알고리즘의 '효율성'이라는 핵심 개념과 이를 측정하는 두 가지 주요 지표인 '시간 복잡도'와 '공간 복잡도'를 소개합니다.

---

### **핵심 개념**

*   **알고리즘 비교 기준**: 여러 알고리즘 중에서 어떤 알고리즘이 특정 문제 해결에 더 적합하고 '좋은' 알고리즘인지 판단하기 위한 기준이 필요합니다.
*   **효율성 (Efficiency)**: 알고리즘의 성능을 평가하는 가장 중요한 척도입니다. "얼마나 빨리 작업을 완료하는가?" 또는 "얼마나 적은 자원을 사용하는가?" 와 같은 질문에 대한 답을 제공합니다.
*   **시간 복잡도 (Time Complexity)**: 알고리즘이 주어진 작업을 완료하는 데 필요한 '실행 시간'을 측정하는 지표입니다. 일반적으로 입력 데이터의 크기($N$)가 증가함에 따라 알고리즘의 실행 시간이 어떻게 변하는지에 초점을 맞춥니다.
*   **공간 복잡도 (Space Complexity)**: 알고리즘이 작업을 수행하는 데 필요한 '메모리 공간'의 양을 측정하는 지표입니다. 이 또한 입력 데이터의 크기($N$)에 따라 알고리즘이 사용하는 메모리 양이 어떻게 변하는지 분석합니다.

### **코드/수식 해설**

이 슬라이드에서는 직접적인 코드나 수식이 제시되지 않습니다. 하지만 앞으로 데이터 구조 및 알고리즘 수업에서 시간/공간 복잡도를 분석할 때 주로 사용하게 될 점근적 표기법(Asymptotic Notation)은 다음과 같습니다.

예를 들어, 어떤 알고리즘의 실행 시간이 입력 크기 $N$에 비례하여 $c_1 N + c_2$와 같이 증가한다면, 우리는 최고차항만 고려하여 시간 복잡도를 $O(N)$이라고 표현합니다.

$$ T(N) = c_1 N + c_2 \Rightarrow O(N) $$

어떤 알고리즘이 $N^2$에 비례하는 시간을 갖는다면, 다음과 같이 표현합니다.

$$ T(N) = c_1 N^2 + c_2 N + c_3 \Rightarrow O(N^2) $$

### **구체적 예시**

*   **시간 복잡도 예시**: 특정 숫자 `X`를 배열에서 찾는 두 가지 방법을 생각해 봅시다.
    1.  **순차 탐색 (Sequential Search)**: 배열의 처음부터 끝까지 하나씩 확인하며 `X`를 찾습니다.
    2.  **이진 탐색 (Binary Search)**: (정렬된 배열에서만 가능) 배열의 중간 값을 확인하여 `X`가 더 작은 쪽에 있는지, 더 큰 쪽에 있는지 판단하고 탐색 범위를 절반씩 줄여나갑니다.
    입력 배열의 크기가 매우 클 때, 이진 탐색이 순차 탐색보다 훨씬 빠르게 `X`를 찾을 수 있습니다. 이는 이진 탐색의 시간 복잡도가 일반적으로 순차 탐색보다 더 효율적이기 때문입니다.

*   **공간 복잡도 예시**: 크기가 $N$인 배열의 모든 요소를 복사하여 새로운 배열을 만드는 알고리즘을 생각해 봅시다.
    *   이 경우, 원본 배열 외에 $N$개의 요소를 저장할 새로운 배열이 필요하므로, 알고리즘이 사용하는 공간 복잡도는 $O(N)$이 됩니다. 반면, 단순히 배열의 모든 요소를 출력하기만 하는 알고리즘은 추가적인 $N$ 크기의 배열이 필요 없으므로, $O(1)$과 같이 매우 적은 추가 공간만 사용할 수 있습니다.

### **시험 포인트**

*   ⭐ 알고리즘을 비교하는 가장 중요한 척도는 **효율성**입니다.
*   ⭐ 효율성을 평가하는 두 가지 핵심 지표는 **시간 복잡도**와 **공간 복잡도**입니다.
*   ⭐ **시간 복잡도**는 알고리즘의 **실행 시간**, **공간 복잡도**는 알고리즘의 **메모리 사용량**을 나타낸다는 것을 정확히 이해해야 합니다.
*   ⭐ 이 두 가지 복잡도는 주로 **입력 데이터의 크기 ($N$)**에 따라 어떻게 변하는지로 분석합니다.

## Slide 6

## 알고리즘 비교 방법

### 핵심 개념
동일한 문제를 해결하는 여러 알고리즘 중 어떤 것이 더 효율적인지 판단하기 위한 두 가지 주요 비교 방법이 있습니다.

1.  **Empirical studies (실험적 연구)**:
    *   실제 알고리즘 코드를 구현한 후, 다양한 입력 데이터에 대해 직접 실행 시간을 측정하여 성능을 비교하는 방법입니다.
    *   특정 하드웨어 및 소프트웨어 환경에서 알고리즘이 실제로 어떻게 동작하는지 관찰할 수 있습니다.
2.  **Theoretical analysis (이론적 분석)**:
    *   알고리즘의 동작을 수학적으로 모델링하고, 입력 크기($n$)에 대한 연산 횟수(시간 복잡도)나 사용되는 메모리 양(공간 복잡도)을 분석하여 성능을 예측하는 방법입니다.
    *   특정 환경에 구애받지 않고 알고리즘의 본질적인 효율성을 평가할 수 있습니다.

### 코드/수식 해설
본 슬라이드에는 직접적인 코드나 수식이 제시되어 있지 않지만, 이론적 분석에서 사용되는 핵심 개념인 시간/공간 복잡도 분석의 기초가 되는 점근적 표기법(Asymptotic Notations)을 이해하는 것이 중요합니다.

*   **시간 복잡도**: 알고리즘이 문제를 해결하는 데 걸리는 시간을 입력 크기($n$)의 함수로 표현합니다. 주로 연산 횟수를 기준으로 합니다.
*   **공간 복잡도**: 알고리즘이 문제를 해결하는 데 필요한 메모리 공간을 입력 크기($n$)의 함수로 표현합니다.

가장 흔히 사용되는 점근적 표기법은 **빅-오(Big-O) 표기법**입니다.
*   **빅-오 표기법($O(g(n))$)**: 어떤 함수 $f(n)$이 $O(g(n))$이라는 것은 $n$이 충분히 커질 때 $f(n)$이 $g(n)$의 상수배보다 빠르게 증가하지 않음을 의미합니다. 즉, 알고리즘의 상한(Upper Bound) 또는 최악의 경우(Worst Case) 실행 시간을 나타내는 데 사용됩니다.

수식으로 표현하면 다음과 같습니다:
어떤 함수 $f(n)$이 $O(g(n))$이라면, 다음을 만족하는 양의 상수 $c$와 $n_0$가 존재한다.
$$
\exists c > 0, n_0 \ge 1 \text{ such that } 0 \le f(n) \le c \cdot g(n) \quad \forall n \ge n_0
$$

예를 들어, 알고리즘의 정확한 연산 횟수가 $T(n) = 2n^2 + 3n + 5$라면, 가장 지배적인 항은 $n^2$이므로 $O(n^2)$으로 표기할 수 있습니다.

### 구체적 예시
1.  **실험적 연구 (Empirical studies)** 예시:
    *   두 가지 정렬 알고리즘(예: 버블 정렬, 퀵 정렬)을 C언어로 구현합니다.
    *   크기가 $1000, 10000, 100000$인 무작위 정수 배열을 생성하여 각 알고리즘에 적용합니다.
    *   `time()` 함수 등을 사용하여 각 실행마다 걸린 실제 시간을 측정하고 기록합니다.
    *   측정된 시간 데이터를 바탕으로 입력 크기에 따른 성능 그래프를 그려 두 알고리즘의 효율성을 비교합니다.
    *   **비유**: 두 대의 경주용 자동차가 같은 트랙을 실제로 달려 누가 더 빠른지 시간을 재어 비교하는 것과 같습니다.

2.  **이론적 분석 (Theoretical analysis)** 예시:
    *   버블 정렬의 시간 복잡도는 $O(n^2)$로 분석됩니다 (최악 및 평균).
    *   퀵 정렬의 평균 시간 복잡도는 $O(n \log n)$으로 분석됩니다.
    *   이론적으로 $n$이 충분히 커질수록 $n \log n$이 $n^2$보다 훨씬 느리게 증가하므로, 퀵 정렬이 버블 정렬보다 더 효율적이라고 결론을 내립니다. 실제 구현 없이도 수학적 분석만으로 성능 우위를 예측할 수 있습니다.
    *   **비유**: 두 대의 경주용 자동차의 설계도(엔진 출력, 무게, 공기 역학 등)를 보고 수학적 계산을 통해 누가 더 빠를지 예측하는 것과 같습니다.

### 시험 포인트
*   ⭐ **알고리즘 비교의 두 가지 주요 방법(실험적 연구, 이론적 분석)을 명확히 설명하고 그 차이점을 이해하는 것이 중요합니다.**
*   ⭐ 각 방법의 **장점과 단점**을 정확히 파악하고 있어야 합니다.
    *   **실험적 연구의 장점**: 실제 환경에서의 성능 측정, 시스템의 특정 요소(캐시, OS 등) 영향 반영.
    *   **실험적 연구의 단점**: 실행 환경에 의존적, 특정 입력 데이터에 국한, 구현 및 테스트 비용 발생, 대규모 입력에 대한 테스트의 어려움.
    *   **이론적 분석의 장점**: 하드웨어/소프트웨어 환경에 독립적, 모든 입력 크기에 대한 일반적인 성능 예측, 알고리즘 본질적 효율성 파악.
    *   **이론적 분석의 단점**: 상수 인자나 저차항 무시로 실제 성능과 차이 발생 가능, 최악의 경우만 고려하는 경우 평균 성능과 다를 수 있음.
*   ⭐ **이론적 분석의 핵심인 시간/공간 복잡도 개념과 빅-오(Big-O) 표기법**이 무엇을 의미하는지 설명할 수 있어야 합니다.

알고리즘의 **경험적 연구 (Empirical Studies)**는 실제 코드를 구현하고 실행하여 그 성능(실행 시간 및 공간 사용량)을 측정하는 방식입니다. 이는 알고리즘의 효율성을 실제 환경에서 검증하는 데 사용됩니다.

*   **진행 과정**:
    1.  알고리즘을 프로그래밍 언어로 구현합니다.
    2.  다양한 입력값(varying inputs)으로 프로그램을 실행합니다.
    3.  프로그램의 실행 시간(runtimes)과 공간 요구사항(space requirements)을 측정합니다.

*   **한계점 (Limitations)**:
    1.  **구현 노력**: 알고리즘을 직접 구현하는 데 많은 노력과 시간이 필요합니다.

    3.  **환경 의존성**: 공정한 비교를 위해서는 반드시 동일한 하드웨어(H/W) 및 소프트웨어(S/W) 환경(environments)에서 측정해야 합니다. 운영체제, 컴파일러, 프로세서 등의 차이가 결과에 영향을 미칠 수 있습니다.

**코드/수식 해설**:
본 슬라이드에는 코드나 수식이 포함되어 있지 않습니다.

**구체적 예시**:
두 가지 정렬 알고리즘(예: 버블 정렬과 퀵 정렬)의 성능을 비교한다고 가정해 봅시다.
1.  **구현**: C/C++ 또는 Python 등으로 두 정렬 알고리즘을 각각 구현합니다.
2.  **실행 및 측정**:
    *   작은 배열 (예: 100개 원소)로 각각 정렬하고 시간을 측정합니다.
    *   중간 배열 (예: 10,000개 원소)로 각각 정렬하고 시간을 측정합니다.
    *   큰 배열 (예: 1,000,000개 원소)로 각각 정렬하고 시간을 측정합니다.
    *   이때, `clock()` 함수나 `std::chrono` 라이브러리를 사용하여 시간을 측정하고, 운영체제 도구를 통해 메모리 사용량을 확인할 수 있습니다.
3.  **결과 분석**: 측정된 시간을 비교하여 어떤 알고리즘이 특정 입력 크기에서 더 빠른지, 또는 어떤 메모리 효율성을 보이는지 분석합니다. 하지만 이 결과는 구현의 완성도, 사용된 컴파일러, CPU 성능 등에 따라 달라질 수 있습니다.

**시험 포인트**:
*   ⭐ **경험적 연구의 목적과 과정**: 알고리즘의 실제 성능(실행 시간, 공간 요구량)을 측정하는 방법과 그 단계를 이해해야 합니다.
*   ⭐ **경험적 연구의 주요 한계점**: 특히 `Lots of effort`, `may not apply to other inputs`, `same H/W and S/W environments` 세 가지 한계점을 명확히 기억해야 합니다. 이 한계점들 때문에 이론적 분석(시간/공간 복잡도 분석, Big-O 표기법 등)이 왜 중요한지에 대한 배경 지식이 됩니다.

## Slide 8

**핵심 개념**

*   **경험적 연구 (Empirical Studies)**: 알고리즘 및 자료구조 분석에서 경험적 연구란, 이론적인 분석(예: 빅-O 표기법을 사용한 점근적 복잡도 분석)과는 다르게, 실제 코드를 다양한 입력 데이터에 대해 실행시켜 **측정 가능한 성능 지표(실행 시간, 메모리 사용량 등)를 수집하고 분석**하는 방법을 의미합니다. 이는 특정 하드웨어, 운영체제, 컴파일러, 그리고 실제 데이터셋의 특성이 알고리즘 성능에 미치는 영향을 파악하는 데 필수적입니다.
*   **이론적 분석의 한계 보완**: 이론적 분석은 데이터 크기 $N$이 무한대로 커질 때의 일반적인 성능 경향을 예측하지만, 상수 계수(constant factors), 낮은 차수의 항, 그리고 시스템 수준의 최적화(예: 캐싱 효과) 등 실제 성능에 큰 영향을 미치는 요소를 무시하는 경향이 있습니다. 경험적 연구는 이러한 이론적 분석의 한계를 보완하여, 특정 환경에서 알고리즘이 어떻게 동작할지 더 정확하게 이해할 수 있도록 돕습니다.

**코드/수식 해설**

경험적 연구는 주로 다음과 같은 방식으로 코드의 실행 시간을 측정합니다.

```cpp
#include <chrono> // C++에서 시간 측정을 위한 헤더

// ... 알고리즘 구현 ...

auto start = std::chrono::high_resolution_clock::now(); // 시작 시간 기록

// 측정하고자 하는 알고리즘 또는 코드 블록 실행
my_algorithm(input_data);

auto end = std::chrono::high_resolution_clock::now();   // 종료 시간 기록

// 시간 차이 계산 (마이크로초 단위)
auto duration = std::chrono::duration_cast<std::chrono::microseconds>(end - start);
std::cout << "Execution time: " << duration.count() << " microseconds" << std::endl;
```

수식적으로 볼 때, 이론적 시간 복잡도가 $O(N \log N)$인 알고리즘의 실제 실행 시간 $T(N)$은 일반적으로 다음과 같이 표현될 수 있습니다.
$$ T(N) = c_1 N \log N + c_2 N + c_3 \log N + c_4 $$
여기서 $c_1, c_2, c_3, c_4$는 상수 계수이며, 이 값들은 알고리즘의 구체적인 구현, 컴파일러의 최적화 수준, 하드웨어 특성(CPU 클럭, 캐시 크기 등)에 따라 크게 달라집니다. 경험적 연구는 이러한 상수 계수들과 낮은 차수의 항들이 실제 유한한 $N$ 값에서 성능에 미치는 영향을 정량적으로 파악하는 데 사용됩니다.

**구체적 예시**

*   **동일한 점근 복잡도를 가진 알고리즘 비교**: 퀵 정렬과 병합 정렬은 모두 평균적으로 $O(N \log N)$의 시간 복잡도를 가집니다. 하지만 특정 크기의 배열에서, 퀵 정렬은 캐시 효율성이나 상수 계수 때문에 병합 정렬보다 빠를 수 있고, 최악의 경우에는 병합 정렬이 더 안정적인 성능을 보일 수 있습니다. 경험적 연구는 다양한 데이터셋 크기와 특성에 대해 두 알고리즘을 실제로 실행시켜 어떤 알고리즘이 특정 환경에서 더 나은 성능을 보이는지 측정하고 비교할 수 있습니다.
*   **이론과 실제 성능의 괴리 분석**: 이론적으로는 $O(N)$으로 매우 효율적이라고 예상되는 특정 자료구조 연산이, 실제 구현 시 메모리 접근 패턴으로 인해 캐시 미스(cache miss)가 빈번하게 발생하여 예상보다 느리게 동작할 수 있습니다. 예를 들어, 불필요한 동적 메모리 할당/해제 작업이 빈번하게 일어나는 경우, 이론적 복잡도는 낮더라도 실제 실행 시간은 크게 증가할 수 있습니다. 경험적 연구를 통해 이러한 성능 병목 현상(bottleneck)의 원인을 식별하고 최적화 방향을 찾을 수 있습니다.

**시험 포인트**

*   ⭐ **경험적 연구의 주요 이점 (Benefits of Empirical Studies)**:
    1.  **실제 환경 성능 측정**: 특정 하드웨어, 컴파일러, 운영체제, 그리고 실제 데이터셋의 특성을 고려한 **정확한 실행 시간 및 메모리 사용량**을 측정할 수 있습니다.
    2.  **상수 계수 및 낮은 차수 항의 중요성 파악**: 이론적 분석에서 무시되는 상수 계수나 낮은 차수의 항이 **작거나 중간 크기의 입력에서 성능에 미치는 실제 영향**을 알 수 있습니다.
    3.  **이론적 예측 검증 및 보완**: 이론적 분석의 가정을 검증하고, 캐싱 효과, 메모리 접근 패턴 등 **이론적 분석이 설명하기 어려운 실제 시스템 요인의 영향**을 파악하여 보완적인 통찰을 얻을 수 있습니다.
    4.  **최적의 알고리즘/구현 선택**: 여러 알고리즘이나 동일한 알고리즘의 여러 구현 중에서 **실제 운영 환경에서 가장 효율적인 것을 선택**하는 데 결정적인 근거를 제공합니다.
    5.  **성능 병목 현상(bottleneck) 식별**: 코드의 어떤 부분이 실제 성능 저하의 주범인지 파악하여 **최적화가 필요한 부분을 명확히 지시**해 줍니다.
*   ⭐ **경험적 연구의 한계점**:
    1.  **환경 의존성**: 측정 결과가 특정 하드웨어, 소프트웨어 환경에 종속되어 일반화하기 어렵습니다.
    2.  **다양한 테스트 케이스 필요**: 의미 있는 결과를 얻기 위해서는 다양한 입력 데이터 크기와 특성을 가진 광범위한 테스트 케이스가 필요하며, 이는 많은 시간과 노력을 요구합니다.
    3.  **오버헤드**: 시간 측정 자체에도 약간의 오버헤드가 발생할 수 있습니다.
    4.  **정확한 측정의 어려움**: 멀티태스킹 환경이나 OS 스케줄링 등으로 인해 일관된 측정이 어려울 수 있습니다.

## Slide 9

**핵심 개념**:
*   **이론적 분석의 초점**: 알고리즘의 **구현(implementation)**이 아닌, 알고리즘의 **고수준(high-level) 추상적 설명**에 집중합니다. 이는 특정 프로그래밍 언어나 하드웨어에 얽매이지 않고 알고리즘 본연의 효율성을 평가하기 위함입니다.
*   **성능 측정의 기준**: 알고리즘의 **실행 시간(running time)**과 **공간 요구사항(space requirements)**을 **입력 크기 $n$의 함수**로 간주합니다. 이를 통해 입력 데이터의 크기가 변함에 따라 알고리즘의 성능이 어떻게 변화하는지 예측하고 비교할 수 있습니다.
*   **분석 범위**: 일반적으로 **모든 가능한 입력(all possible inputs)**에 대해 알고리즘의 동작을 고려합니다. 이는 최악의 경우(worst-case)나 평균적인 경우(average-case) 성능을 분석하여 알고리즘의 신뢰성을 보장하려는 목적을 가집니다.
*   **환경 독립성**: 이론적 분석 결과는 특정 **하드웨어/소프트웨어 환경에 독립적**입니다. 즉, 어떤 컴퓨터나 운영체제에서 실행되든 알고리즘의 이론적인 성능 특성(예: 시간 복잡도)은 변하지 않습니다.

**코드/수식 해설**:
해당 슬라이드에는 직접적인 코드나 수식이 포함되어 있지 않지만, "입력 크기 $n$의 함수"라는 개념을 수학적 표기법으로 나타내면 다음과 같습니다.
*   알고리즘의 실행 시간 $T$를 입력 크기 $n$의 함수로 표기: $T(n)$
*   알고리즘의 공간 요구사항 $S$를 입력 크기 $n$의 함수로 표기: $S(n)$
이러한 함수들은 나중에 배울 점근적 표기법(Asymptotic Notation)을 통해 간략화됩니다.

**구체적 예시**:
*   **알고리즘 추상화**: 배열에서 특정 숫자를 찾는 **선형 탐색(Linear Search)** 알고리즘을 생각해봅시다.
    *   **구현 예시 (C++)**:
        ```cpp
        int linearSearch(int arr[], int n, int target) {
            for (int i = 0; i < n; i++) {
                if (arr[i] == target) {
                    return i; // 찾음
                }
            }
            return -1; // 못 찾음
        }
        ```
    *   **추상적 설명**: "배열의 첫 번째 요소부터 마지막 요소까지 순서대로 방문하여 각 요소가 목표 값과 일치하는지 비교한다."
    이론적 분석은 코드 한 줄 한 줄의 실행 시간(예: `arr[i] == target` 비교에 걸리는 CPU 사이클 수)보다는, **추상적 설명에 기반한 주요 연산(여기서는 비교 연산)의 횟수**를 $n$의 함수로 표현하는 데 집중합니다. 최악의 경우, 목표 값이 배열의 마지막에 있거나 없으면 $n$번의 비교 연산이 필요하므로, 이 알고리즘의 시간 복잡도는 $O(n)$이 됩니다.
*   **환경 독립성**: 위 선형 탐색 알고리즘이 구형 컴퓨터에서 100밀리초가 걸리고 최신 컴퓨터에서 1밀리초가 걸릴 수 있습니다. 실행 시간 자체는 다르지만, **입력 크기 $n$이 두 배가 되면 실행 시간도 대략 두 배가 될 것**이라는 "선형적인 증가"라는 이론적 분석 결과($O(n)$)는 어떤 환경에서든 동일하게 유지됩니다.

**시험 포인트**:
*   ⭐ **이론적 분석의 목적**: 알고리즘의 **성능을 입력 크기 $n$의 함수로 나타내고**, 이를 통해 **하드웨어/소프트웨어 환경에 구애받지 않는 보편적인 효율성**을 평가하는 것임을 명확히 이해해야 합니다.
*   ⭐ **'구현'과 '고수준 설명'의 차이**를 구분하고, 이론적 분석이 어디에 초점을 맞추는지 설명할 수 있어야 합니다.
*   ⭐ **시간 복잡도($T(n)$)와 공간 복잡도($S(n)$)**라는 개념이 무엇을 의미하며, 왜 입력 크기 $n$에 대한 함수로 표현하는지 그 중요성을 인지해야 합니다. 이는 앞으로 배우게 될 Big-O 표기법의 기반이 됩니다.

## Slide 10

**핵심 개념**:
이 슬라이드는 알고리즘의 **이론적 분석(Theoretical Analysis)**, 즉 주로 점근적 시간/공간 복잡도(예: 빅-오 표기법) 분석의 한계점에 대해 묻고 있습니다. 이론적 분석은 알고리즘의 효율성을 입력 크기($n$)가 무한히 커질 때의 경향성으로 평가하기 때문에, 실제 시스템에서의 동작이나 특정 상황에서 발생하는 미묘한 성능 차이를 반영하지 못하는 단점이 있습니다.

**코드/수식 해설**:
이론적 분석은 알고리즘의 실행 시간 $T(n)$을 빅-오(Big-O) 표기법 $O(f(n))$으로 표현합니다. 여기서 $f(n)$은 $n$이 충분히 커질 때 $T(n)$의 상한을 나타내는 함수입니다.

예를 들어, 두 알고리즘 A와 B가 모두 $O(n^2)$의 시간 복잡도를 가진다고 가정해봅시다.
알고리즘 A의 실제 실행 시간 함수가 $T_A(n) = 100n^2 + 50n + 1000$이고,
알고리즘 B의 실제 실행 시간 함수가 $T_B(n) = 2n^2 + 1000n + 50$이라고 할 때,
이론적 분석만으로는 두 알고리즘이 "동일한" $O(n^2)$ 효율을 가진다고 판단합니다.
하지만 실제로는 $n$이 작은 값일 때, 또는 특정 상수 인자가 매우 클 때 $T_A(n)$과 $T_B(n)$의 성능 차이가 매우 클 수 있습니다.
$n$이 매우 커지면 최고차항($n^2$)이 지배적이 되므로 상수 인자(100과 2)는 무시될 수 있지만, 현실적인 $n$ 값에서는 이 상수 인자가 중요한 영향을 미칩니다.

**구체적 예시**:
1.  **상수 인자와 낮은 차수 항의 무시**:
    *   두 개의 정렬 알고리즘, A는 최악의 경우 $100n \log n$ 시간이 걸리고, B는 최악의 경우 $2n^2$ 시간이 걸린다고 가정합시다. 이론적으로 A는 $O(n \log n)$, B는 $O(n^2)$이므로 A가 B보다 훨씬 빠릅니다.
    *   하지만 만약 알고리즘 A의 실제 구현이 매우 복잡하여 상수 인자가 $C_A = 100$이고, 알고리즘 B가 매우 간단하여 상수 인자가 $C_B = 2$라면,
        *   $n=10$일 때: $T_A(10) \approx 100 \times 10 \times 3.32 \approx 3320$, $T_B(10) \approx 2 \times 100 = 200$. 이 경우 B가 더 빠릅니다.
        *   $n=1000$일 때: $T_A(1000) \approx 100 \times 1000 \times 9.97 \approx 997,000$, $T_B(1000) \approx 2 \times 1000^2 = 2,000,000$. 이 경우 A가 더 빠릅니다.
    *   즉, 이론적 분석은 $n$이 매우 커질 때의 경향만 보여주며, 실제 작은 입력 크기나 특정 환경에서는 상수 인자가 성능에 더 큰 영향을 미칠 수 있습니다.

2.  **하드웨어 및 환경적 요인**:
    *   아무리 이론적으로 효율적인 알고리즘이라도, 특정 하드웨어(CPU 캐시, 메모리 접근 속도, 디스크 I/O)나 소프트웨어 환경(운영체제 스케줄링, 프로그래밍 언어, 컴파일러 최적화)에 따라 실제 실행 시간이 크게 달라질 수 있습니다. 예를 들어, 캐시 효율이 높은 알고리즘은 이론적으로는 $O(n)$이라도 캐시 미스가 잦은 $O(n \log n)$ 알고리즘보다 실제로는 느릴 수 있습니다.
    *   이론적 분석은 이러한 시스템 레벨의 세부 사항을 고려하지 않습니다.

**시험 포인트**:
*   ⭐ **이론적 분석의 한계점 3가지 이상을 서술하시오.** (주요 답변 예시: 상수 인자 및 낮은 차수 항 무시, 하드웨어/소프트웨어 환경의 영향 미반영, 특정 입력 데이터 패턴에 따른 성능 변화 예측 어려움)
*   ⭐ **빅-오(Big-O) 표기법이 실제 실행 시간을 정확히 예측하지 못하는 이유를 설명하고, 이러한 한계를 보완하기 위한 방법(예: 실험적 분석)을 제시하시오.**
*   ⭐ **이론적 분석과 실험적 분석(Empirical Analysis)의 차이점과 각각의 장단점을 비교하시오.** (이론적 분석은 일반화되고 추상적이지만 구현 없이 가능, 실험적 분석은 현실적이지만 특정 환경에 종속되고 구현이 필요함)

## Slide 11

**핵심 개념**:
-   **점근적 분석 (Asymptotic Analysis)**: 알고리즘의 효율성을 평가하는 방법론으로, 입력의 크기 $n$이 "매우 커질 때" 알고리즘의 시간 복잡도나 공간 복잡도가 어떻게 변화하는지 그 **극한 동작(limiting behavior)**을 설명하는 데 중점을 둡니다. 이는 실제 실행 시간의 정확한 값이 아니라, $n$이 커질수록 성능이 어떻게 "증가하는 경향"을 보이는지를 파악하는 데 사용됩니다.
-   **하위 차수 항 무시 (Lower-order terms are not important)**: $n$이 충분히 커질 때, 알고리즘의 성능을 나타내는 함수에서 가장 큰 영향을 미치는 것은 "최고차항(dominant term)"입니다. 최고차항보다 차수가 낮은 항들은 $n$이 커질수록 전체 값에 미치는 영향이 점차 미미해지므로, 점근적 분석에서는 이들을 무시합니다. 마찬가지로, 최고차항의 상수 계수(constant coefficient)도 $n$이 커질 때의 성장률 자체에는 영향을 주지 않으므로 무시합니다.

**코드/수식 해설**:
주어진 함수 $f(n)$이 다음과 같을 때:
$$f(n) = n^2 + 3n$$
$n$이 무한히 커질 때($n \to \infty$), 각 항의 증가 속도를 비교해 봅시다.
-   $n^2$ 항은 $n$이 커질수록 제곱에 비례하여 증가합니다.
-   $3n$ 항은 $n$이 커질수록 선형적으로 증가합니다.

예를 들어,
-   $n = 10$: $f(10) = 10^2 + 3 \times 10 = 100 + 30 = 130$
-   $n = 100$: $f(100) = 100^2 + 3 \times 100 = 10000 + 300 = 10300$
-   $n = 1000$: $f(1000) = 1000^2 + 3 \times 1000 = 1000000 + 3000 = 1003000$

$n$이 커질수록 $3n$이 $n^2$에 비해 차지하는 비율은 급격히 감소합니다 (위 예시에서 약 23% $\to$ 2.9% $\to$ 0.3%). 따라서 $n$이 매우 커지면 $3n$ 항은 $n^2$ 항에 비해 **무시할 수 없을 정도로 작아집니다(insignificant)**. 이 경우, 함수 $f(n)$은 "점근적으로 $n^2$에 동등하다(asymptotically equivalent to $n^2$)"고 표현하며, 이는 흔히 빅-O 표기법($O(n^2)$)의 기초가 됩니다.

**구체적 예시**:
여러분들이 자동차를 타고 여행을 간다고 가정해 봅시다. 총 여행 시간은 다음과 같은 요소들로 구성될 수 있습니다.
-   고속도로 주행 시간: (거리 $D$) / (평균 속도 $V_h$)
-   휴게소 정차 시간: (휴게소 방문 횟수 $k$) * (1회당 정차 시간 $T_s$)
-   톨게이트 통과 시간: (톨게이트 개수 $m$) * (1회당 통과 시간 $T_t$)

이것을 함수로 표현하면 $f(D) = \frac{1}{V_h} D + k T_s + m T_t$ 와 같이 될 수 있습니다.
만약 여행 거리 $D$가 매우 길어진다면 (예를 들어, 서울-부산 왕복), 고속도로 주행 시간($\frac{1}{V_h} D$)이 전체 여행 시간에서 압도적인 비중을 차지하게 됩니다. 휴게소 정차 시간이나 톨게이트 통과 시간은 고정된 값($k T_s + m T_t$) 또는 거리에 비례하더라도 그 계수가 작아 전체 시간에 미치는 영향이 매우 작아집니다.
이처럼 "매우 긴 거리($D \to \infty$)"를 가정할 때, 여행 시간의 주된 요인은 "고속도로 주행 시간"이 되며, 나머지 시간은 "하위 차수 항"으로 간주되어 무시할 수 있습니다. 점근적 분석은 이와 같이 "스케일이 커졌을 때" 어떤 요소가 지배적인지를 파악하는 데 유용합니다.

**시험 포인트**:
-   ⭐ **점근적 분석의 핵심 목표**: 알고리즘의 **확장성(scalability)**을 예측하는 데 왜 "극한 동작(limiting behavior)"을 분석하는지 설명할 수 있어야 합니다. (즉, 작은 입력에서는 큰 의미가 없지만, 실제 대규모 문제 해결 시 중요한 지표가 됨)
-   ⭐ **하위 차수 항 무시 원칙**: $f(n) = an^k + bn^{k-1} + \dots + c$ 형태의 함수에서 $n \to \infty$ 일 때, 왜 $an^k$ 항이 지배적인지, 그리고 $b, c$ 등의 상수 계수도 무시하는 이유를 명확하게 설명할 수 있어야 합니다.
-   ⭐ **다항식 함수에서 최고차항 식별**: 주어진 시간 복잡도 함수에서 최고차항을 정확하게 식별하고, 점근적으로 동등한 함수($n^2$, $n \log n$, $n^3$ 등)를 도출할 수 있어야 합니다. 이는 나중에 빅-O 표기법을 학습할 때 핵심적인 기초가 됩니다.

## Slide 12

**핵심 개념**
점근적 분석(Asymptotic Analysis)은 알고리즘의 시간 또는 공간 복잡도를 평가할 때, 입력 크기 $N$이 무한대로 커질 때(즉, '극한적인 상황'에서) 함수의 '극한적인 행동(limiting behavior)'을 설명하는 방법입니다. 이 분석은 실제 실행 시간이나 메모리 사용량의 정확한 값을 계산하기보다는, $N$이 커짐에 따라 성능이 어떻게 변화하는지 그 '성장률'에 초점을 맞춥니다.

특히, 점근적 분석에서 중요한 원칙 중 하나는 **"상수는 중요하지 않다 (Constants do not matter)"**는 것입니다. 이는 $N$이 매우 커질 때, 복잡도 함수에 곱해지는 상수 인자가 전체 함수의 증가율에 미치는 영향이 미미해지기 때문입니다.

**코드/수식 해설**
슬라이드에서 제시된 $6N^2$와 $20N^2$ 두 함수를 통해 이 개념을 이해할 수 있습니다.

*   **$N \gg 20$**: 이 표기는 $N$이 20보다 훨씬 큰 값, 즉 매우 큰 값을 의미합니다.
*   $N$이 매우 클 때, 두 함수 $6N^2$와 $20N^2$는 모두 $N^2$이라는 항에 의해 그 증가 속도가 결정됩니다. 여기서 6과 20은 상수 인자입니다.
*   $N^2$ 항은 $N$이 커질수록 다른 저차항(예: $N$ 또는 상수)보다 훨씬 더 빠르게 증가하므로, 전체 함수의 증가를 지배하는 '주요 원인(driving the function's increase)'이 됩니다.
*   $N \rightarrow \infty$ (N이 무한대로 갈 때), $6N^2$는 $20N^2$와 "점근적으로 동등(asymptotically equivalent)"하다고 말합니다. 이는 두 함수의 비율이 상수에 수렴함을 의미합니다:
    $$ \lim_{N \to \infty} \frac{20N^2}{6N^2} = \lim_{N \to \infty} \frac{20}{6} = \frac{10}{3} $$
    이처럼 비율이 유한한 양의 상수에 수렴하면, 두 함수는 점근적으로 동일한 성장률을 가진다고 봅니다. 따라서 빅-O 표기법에서는 두 함수 모두 $O(N^2)$로 표현됩니다.

**구체적 예시**
두 개의 웹 서버가 있습니다.
*   서버 A는 요청 처리 시간이 $1000N$ 밀리초라고 가정합니다.
*   서버 B는 요청 처리 시간이 $N^2$ 밀리초라고 가정합니다.

여기서 $N$은 초당 들어오는 요청의 수입니다.
*   $N=10$ (초당 10개 요청):
    *   서버 A: $1000 \times 10 = 10000$ 밀리초 (10초)
    *   서버 B: $10^2 = 100$ 밀리초 (0.1초)
    *   이 경우 서버 B가 훨씬 빠릅니다. $N^2$ 항의 상수가 1이지만 $1000N$의 상수가 1000이므로 상수의 차이가 중요하게 작용합니다.
*   $N=1000$ (초당 1000개 요청):
    *   서버 A: $1000 \times 1000 = 1,000,000$ 밀리초 (1000초 = 약 16.7분)
    *   서버 B: $1000^2 = 1,000,000$ 밀리초 (1000초 = 약 16.7분)
    *   이 경우 두 서버의 처리 시간이 비슷해집니다.
*   $N=10000$ (초당 10000개 요청):
    *   서버 A: $1000 \times 10000 = 10,000,000$ 밀리초 (10000초 = 약 2.78시간)
    *   서버 B: $10000^2 = 100,000,000$ 밀리초 (100000초 = 약 27.8시간)
    *   이제 서버 A가 서버 B보다 훨씬 빨라집니다.

이 예시에서 알 수 있듯이, 작은 $N$에서는 상수($1000$ vs $1$)가 중요하게 보일 수 있지만, $N$이 특정 지점을 넘어서면 최고차항($N$ vs $N^2$)이 전체 성능을 지배하게 됩니다. 점근적 분석은 이처럼 $N$이 '매우 커질 때'의 경향성을 파악하는 데 중점을 둡니다.

**시험 포인트**
*   ⭐ 점근적 분석은 입력 크기 $N$이 무한대로 커질 때의 '극한적인 행동'을 분석하는 것임을 정확히 이해해야 합니다.
*   ⭐ **상수는 점근적 분석에서 무시됩니다.** 알고리즘의 효율성을 빅-O 표기법으로 나타낼 때, 상수 인자는 제외하고 가장 높은 차수의 항(dominant term)만 사용하여 표현합니다. 예를 들어, $5N^3 + 2N^2 + 100$의 시간 복잡도는 $O(N^3)$ 입니다.
*   ⭐ 왜 상수가 무시되는지 그 이유를 설명할 수 있어야 합니다: $N$이 충분히 커질 때, 최고차항이 저차항 및 상수항보다 훨씬 더 빠르게 증가하여 전체 함수의 증가율을 지배하게 되므로, 상수 인자는 상대적으로 중요성이 떨어지기 때문입니다.

## Best, Worst, and Average Cases

### 핵심 개념

알고리즘의 실행 시간은 주어진 입력의 크기($N$)뿐만 아니라, 입력 데이터의 '내용'에 따라서도 달라질 수 있습니다. 이러한 다양한 입력 시나리오에 따라 알고리즘의 성능을 세 가지 주요 케이스로 분석합니다.

1.  **최선(Best Case) 입력**:
    *   알고리즘이 가장 빨리 실행될 때의 입력 시나리오를 의미합니다.
    *   "적어도 이만큼의 시간이 걸린다" (At least, takes this much)는 하한선을 나타냅니다.

2.  **평균(Average Case) 입력**:
    *   일반적인 또는 전형적인 입력이 주어졌을 때 알고리즘의 실행 시간을 의미합니다.
    *   "보통 이만큼의 시간이 걸린다" (Usually, it takes this much)고 할 수 있습니다.
    *   모든 가능한 입력의 분포에 대한 가정이 필요하며, 정확한 분석이 **어렵습니다** (Difficult to determine).

3.  **최악(Worst Case) 입력**:
    *   알고리즘이 가장 오래 실행될 때의 입력 시나리오를 의미합니다.
    *   "최대 이만큼의 시간이 걸릴 수 있다" (It could take up to this)는 상한선을 나타냅니다.
    *   평균 경우보다 **분석하기 더 쉽고** (Easier to analyze), 사용자 경험이 중요한 **상호작용적 애플리케이션(interactive applications)에서 매우 중요합니다.**
    *   대부분의 이론적 분석은 안정적인 성능 보장을 위해 **최악의 경우 입력에 초점을 맞춥니다.**

### 코드/수식 해설

알고리즘의 시간 복잡도는 주로 $N$에 대한 함수로 표현되며, 최악의 경우 분석 시 빅-O 표기법($O$)을 사용합니다. 최선의 경우 분석에는 빅-오메가($\Omega$) 표기법을, 평균 또는 최악/최선이 같은 경우에 빅-세타($\Theta$) 표기법을 사용하기도 합니다.

예를 들어, 배열에서 특정 값을 찾는 선형 탐색(Linear Search)의 경우를 살펴보겠습니다.

```c
int linearSearch(int arr[], int n, int target) {
    for (int i = 0; i < n; i++) {
        if (arr[i] == target) {
            return i; // 찾음
        }
    }
    return -1; // 못 찾음
}
```

*   **최선 경우**: `target`이 배열의 첫 번째 요소(`arr[0]`)인 경우. 비교 연산이 1번만 일어납니다.
    *   시간 복잡도: $\Omega(1)$
*   **최악 경우**: `target`이 배열의 마지막 요소(`arr[n-1]`)이거나 배열에 존재하지 않는 경우. 배열의 모든 $N$개 요소를 확인해야 합니다.
    *   시간 복잡도: $O(N)$
*   **평균 경우**: `target`이 평균적으로 배열의 중간에 있을 경우. 대략 $N/2$번의 비교 연산이 일어납니다.
    *   시간 복잡도: $\Theta(N)$ (상수 계수를 무시하므로 $O(N)$과 동일하게 표현 가능)

### 구체적 예시

**도서관에서 특정 책을 찾는 경우**로 비유할 수 있습니다.

*   **최선 경우**: 찾는 책이 도서관 입구의 신간 코너에 바로 놓여 있는 경우. 아주 빠르게 찾을 수 있습니다.
*   **최악 경우**: 찾는 책이 도서관 가장 안쪽 구석의 가장 높은 선반에 있거나, 심지어 도서관에 없는 경우. 도서관 전체를 샅샅이 뒤져야 하므로 시간이 가장 오래 걸립니다.
*   **평균 경우**: 대략 중간쯤 되는 서가에서 찾는 책을 발견하는 경우. 최선보다는 오래 걸리지만 최악보다는 빠릅니다.

이 비유에서 도서관의 책 개수($N$)는 입력 크기이고, 책의 위치가 입력 내용에 해당합니다. 우리가 도서관 시스템을 설계한다면, 사용자가 "최악의 경우에도 이 정도 시간 안에는 책을 찾을 수 있을 것"이라고 보장하기 위해 최악의 경우를 분석하는 것이 중요합니다.

### 시험 포인트

*   ⭐ **최선, 평균, 최악 각 케이스의 정의와 의미를 명확히 이해해야 합니다.**
*   ⭐ **대부분의 알고리즘 이론적 분석이 최악의 경우(Worst Case)에 초점을 맞추는 이유를 설명할 수 있어야 합니다.** (안정적인 성능 보장, 상호작용성, 분석의 용이성)
*   ⭐ **선형 탐색 등 간단한 알고리즘을 예시로 각 케이스의 시간 복잡도를 분석할 수 있어야 합니다.** (예: 선형 탐색의 최악은 $O(N)$, 최선은 $O(1)$)
*   ⭐ **빅-O 표기법이 주로 최악의 경우 성능을 나타내는 데 사용된다는 것을 기억하세요.**

## Slide 14

**핵심 개념**:
*   **점근적 분석 (Asymptotic Analysis)**: 알고리즘의 성능을 입력 크기 $n$이 무한대로 커질 때($n \to \infty$) 어떻게 변하는지 이론적으로 분석하는 방법입니다. 이는 실제 실행 시간을 측정하는 대신, 알고리즘이 수행하는 기본 연산 횟수의 증가 추세를 파악하여 알고리즘의 효율성을 비교하는 데 사용됩니다.
*   **실행 시간 함수 $T(n)$**: 알고리즘에 주어진 입력의 크기 $n$에 대한 총 실행 시간(또는 연산 횟수)을 나타내는 함수입니다.
*   **성장률 (Growth Rate)**: $T(n)$이 입력 크기 $n$이 증가함에 따라 얼마나 빠르게 증가하는지를 나타내는 척도입니다. 점근적 분석은 이 성장률을 포착하여 알고리즘의 장기적인 성능을 평가합니다.
*   **점근적 표기법 (Asymptotic Notations)**: 알고리즘의 성장률을 수학적으로 표현하는 데 사용되는 표기법으로, 다음 세 가지가 주요합니다.
    1.  **Big-O ($O$)**: 알고리즘의 **최악의 경우 (Worst Case)** 실행 시간의 **상한 (Upper Bound)**을 나타냅니다. 즉, 아무리 나빠도 이 시간 복잡도 이상으로는 걸리지 않는다는 의미입니다.
    2.  **Big-Theta ($\Theta$)**: 알고리즘의 **평균적인 경우 (Average Case)** 실행 시간의 **정확한 성장률**을 나타냅니다. 상한과 하한이 같아서 알고리즘의 실제적인 성장률을 가장 정확하게 표현합니다.
    3.  **Big-Omega ($\Omega$)**: 알고리즘의 **최선의 경우 (Best Case)** 실행 시간의 **하한 (Lower Bound)**을 나타냅니다. 즉, 아무리 좋아도 이 시간 복잡도보다는 최소한 더 걸린다는 의미입니다.

**코드/수식 해설**:
점근적 표기법은 수식적으로 다음과 같이 정의됩니다.

*   **Big-O ($O$) - 상한 (Worst Case)**
    함수 $T(n)$이 $O(g(n))$이라는 것은, 적당한 양의 상수 $c$와 충분히 큰 $n_0$에 대해 모든 $n \ge n_0$에 대하여 $T(n)$이 $c \cdot g(n)$보다 작거나 같다는 것을 의미합니다.
    $$T(n) \in O(g(n)) \iff \exists c > 0, n_0 > 0 \text{ such that } T(n) \le c \cdot g(n) \text{ for all } n \ge n_0$$

*   **Big-Omega ($\Omega$) - 하한 (Best Case)**
    함수 $T(n)$이 $\Omega(g(n))$이라는 것은, 적당한 양의 상수 $c$와 충분히 큰 $n_0$에 대해 모든 $n \ge n_0$에 대하여 $T(n)$이 $c \cdot g(n)$보다 크거나 같다는 것을 의미합니다.
    $$T(n) \in \Omega(g(n)) \iff \exists c > 0, n_0 > 0 \text{ such that } T(n) \ge c \cdot g(n) \text{ for all } n \ge n_0$$

*   **Big-Theta ($\Theta$) - 정확한 성장률 (Average Case)**
    함수 $T(n)$이 $\Theta(g(n))$이라는 것은, 적당한 양의 상수 $c_1, c_2$와 충분히 큰 $n_0$에 대해 모든 $n \ge n_0$에 대하여 $T(n)$이 $c_1 \cdot g(n)$보다 크거나 같고 $c_2 \cdot g(n)$보다 작거나 같다는 것을 의미합니다. 이는 $T(n)$이 $O(g(n))$이면서 동시에 $\Omega(g(n))$임을 의미합니다.
    $$T(n) \in \Theta(g(n)) \iff \exists c_1 > 0, c_2 > 0, n_0 > 0 \text{ such that } c_1 \cdot g(n) \le T(n) \le c_2 \cdot g(n) \text{ for all } n \ge n_0$$

**구체적 예시**:
배열에서 특정 값을 찾는 **선형 탐색 (Linear Search)** 알고리즘을 예로 들어봅시다. 입력 크기 $n$은 배열의 요소 개수입니다.

*   **Worst case (Big-O)**: 찾으려는 값이 배열의 마지막에 있거나 배열에 아예 없는 경우입니다. 이 경우, 알고리즘은 배열의 모든 $n$개 요소를 하나씩 확인해야 합니다. 따라서 시간 복잡도는 $O(n)$입니다.
    *   **예시**: 1000개의 숫자가 있는 배열에서 마지막 1000번째 숫자를 찾거나, 없는 숫자를 찾는 경우. 1000번의 비교가 필요합니다.
*   **Best case (Big-Omega)**: 찾으려는 값이 배열의 첫 번째에 있는 경우입니다. 이 경우, 알고리즘은 첫 번째 비교에서 값을 찾고 종료됩니다. 따라서 시간 복잡도는 $\Omega(1)$입니다.
    *   **예시**: 1000개의 숫자가 있는 배열에서 첫 번째 숫자를 찾는 경우. 1번의 비교만 필요합니다.
*   **Average case (Big-Theta)**: 찾으려는 값이 평균적으로 배열의 중간쯤에 존재한다고 가정하는 경우입니다. 대략 $n/2$번의 비교가 필요하며, 상수 $1/2$를 무시하면 시간 복잡도는 $\Theta(n)$입니다. (선형 탐색의 경우 최악과 평균의 성장률이 같으므로 $\Theta(n)$으로 표현됩니다.)
    *   **예시**: 1000개의 숫자가 있는 배열에서 평균적으로 500번째 숫자를 찾는 경우. 약 500번의 비교가 필요합니다.

**시험 포인트**:
*   ⭐ **점근적 분석의 중요성**: 왜 실제 시간 측정 대신 점근적 분석을 사용하여 알고리즘의 성능을 평가하는지, 그 이유(하드웨어, OS 등의 환경 독립성, 큰 입력에 대한 확장성 예측)를 설명할 수 있어야 합니다.
*   ⭐ **Big-O, Big-Theta, Big-Omega의 개념**: 각 표기법이 어떤 케이스(최악, 평균, 최선)를 나타내며, 각각 상한, 정확한 성장률, 하한 중 무엇을 의미하는지 명확히 구분하고 설명할 수 있어야 합니다.
*   ⭐ **각 표기법의 수식적 정의**: 위에 제시된 Big-O, Big-Omega, Big-Theta의 수학적 정의(특히 상수 $c$와 $n_0$의 의미)를 이해하고 정확히 표현할 수 있어야 합니다.
*   ⭐ **알고리즘별 시간 복잡도 분석**: 주어진 간단한 알고리즘에 대해 최악, 최선, 평균 경우의 시간 복잡도를 Big-O, Big-Omega, Big-Theta 표기법으로 분석하고 도출하는 연습이 중요합니다.

### Big-O Notation

-   **핵심 개념**: Big-O 표기법($O(f(n))$)은 알고리즘의 시간 복잡도 또는 공간 복잡도를 분석할 때 사용되며, 입력 크기 $n$이 충분히 커질 때(점근적 관점) 알고리즘의 실행 시간 $T(n)$의 **성장률에 대한 점근적 상한선(Asymptotic Upper Bound)**을 나타냅니다. 이는 알고리즘이 아무리 오래 걸려도 특정 함수 $f(n)$의 상수배보다 더 느려지지 않음을 보장합니다.

-   **코드/수식 해설**:
    Big-O 표기법의 정의는 다음과 같습니다:

    An algorithm is $O(f(n))$ if there exist a constant $c > 0$ & an integer constant $n_0 \geq 1$ such that $T(n) \leq c \cdot f(n)$ for all $n \geq n_0$.

    *   $T(n)$: 입력 크기 $n$에 대한 알고리즘의 실제 실행 시간 또는 연산 횟수를 나타내는 함수.
    *   $f(n)$: $T(n)$의 성장률을 비교하기 위한 기준 함수(일반적으로 $n$, $n^2$, $\log n$, $n \log n$ 등).
    *   $c$: 양의 상수 (상한선을 조정하기 위한 계수).
    *   $n_0$: 양의 정수 상수 (특정 입력 크기 이상에서 상한선이 유효함을 보장).

    위 정의는 $T(n)$이 $n_0$ 이상의 모든 $n$에 대해 $c \cdot f(n)$보다 작거나 같음을 의미합니다. 즉, $T(n)$은 $f(n)$의 성장률을 넘어설 수 없다는 뜻입니다.

    이를 표현할 때는 $T(n) \in O(f(n))$ 또는 $T(n) = O(f(n))$으로 씁니다.

    슬라이드의 그래프는 이 개념을 시각적으로 보여줍니다.
    *   왼쪽 그래프: $T(n)$ (빨간색 선)이 어떤 $n_0$ 이후로 $c \times n$ (검은색 선) 아래에 머무르는 것을 보여줍니다. 이는 $T(n)$이 $O(n)$임을 의미합니다.
    *   오른쪽 그래프: $T(n)$ (빨간색 선)이 어떤 $n_0$ 이후로 $c \times n^2$ (검은색 선) 아래에 머무르는 것을 보여줍니다. 이는 $T(n)$이 $O(n^2)$임을 의미합니다.

-   **구체적 예시**:
    다음과 같은 간단한 코드를 생각해 봅시다.

    ```c
    void printArray(int arr[], int n) {
        for (int i = 0; i < n; i++) { // n번 반복
            printf("%d ", arr[i]);   // 상수 시간 연산
        }
    }
    ```
    이 `printArray` 함수는 배열 `arr`의 모든 원소를 출력합니다. `for` 루프는 입력 크기 $n$에 따라 $n$번 반복되며, 각 반복 내의 `printf` 연산은 상수 시간($k$)이 걸립니다.
    따라서 총 실행 시간 $T(n)$은 대략 $n \times k$가 됩니다.
    Big-O 정의에 따르면, $T(n) = k \cdot n$은 $c=k$ (예: $k=1$, $c=1$)이고 $f(n) = n$일 때 $T(n) \leq c \cdot f(n)$을 만족합니다 (모든 $n \geq 1$에 대해).
    따라서 이 함수의 시간 복잡도는 $O(n)$입니다.

    **실생활 비유**:
    "강남에서 부산까지 자동차로 이동하는 시간"을 계산한다고 해봅시다. 다양한 요인(교통 체증, 운전자의 숙련도, 도로 상황 등)에 따라 실제 시간은 달라지겠지만, 우리는 "대략 4시간에서 6시간 정도 걸릴 것이다"라고 예상할 수 있습니다. 여기서 "6시간"은 상한선과 같습니다. 즉, 아무리 오래 걸려도 6시간 안에 도착할 것이라고 가정하는 것이 Big-O 표기법의 상한선 개념과 유사합니다. 실제 걸리는 시간 $T(n)$이 특정 함수 $f(n)$의 상수배($c \cdot f(n)$)를 넘지 않는다는 것을 의미합니다.

-   **시험 포인트**:
    *   ⭐ **Big-O 표기법의 정확한 정의를 서술하고 각 변수($T(n)$, $f(n)$, $c$, $n_0$)의 의미를 설명**할 수 있어야 합니다.
    *   ⭐ Big-O가 알고리즘의 **점근적 상한선(Asymptotic Upper Bound)**을 의미한다는 것을 이해해야 합니다. 최악의 경우에도 이 시간을 넘지 않는다는 보장으로 활용됩니다.
    *   주어진 간단한 코드 스니펫에 대해 Big-O 시간 복잡도를 분석하고 그 이유를 설명할 수 있어야 합니다.
    *   상수 $c$와 $n_0$가 왜 필요한지, 그리고 Big-O 표기법이 큰 $n$ 값에 대해서만 유의미한 이유(점근적 분석)를 이해해야 합니다.

## Slide 16

이 슬라이드는 알고리즘의 효율성을 나타내는 시간 복잡도(Time Complexity) 또는 공간 복잡도(Space Complexity)를 표기하는 데 사용되는 대표적인 점근적 성장률(Asymptotic Growth Rate)들을 설명합니다. 입력 데이터의 크기 $n$에 따라 알고리즘이 얼마나 많은 시간이나 공간 자원을 사용하는지 빅-오(Big-O) 표기법으로 나타냅니다.

---

### **핵심 개념**

*   **빅-오(Big-O) 표기법**: 알고리즘의 성능을 입력 크기 $n$에 대한 함수로 표현할 때, 가장 크게 영향을 미치는 항만을 남기고 상수를 무시하여 간략하게 나타내는 방법입니다. 이는 알고리즘의 상한(upper bound)을 나타내며, $n$이 충분히 커졌을 때의 성능 추이를 분석하는 데 사용됩니다.
*   **주요 성장률**:
    *   `O(1)` (상수 시간): 입력 크기 $n$에 관계없이 항상 일정한 시간이 소요됩니다.
    *   `O(log n)` (로그 시간): 입력 크기가 커질수록 필요한 시간이 매우 조금씩 증가합니다.
    *   `O(n)` (선형 시간): 입력 크기 $n$에 비례하여 시간이 증가합니다.
    *   `O(n log n)` (선형 로그 시간): 선형 시간보다 약간 더 빠르게 증가하지만, `O(n^2)`보다는 훨씬 효율적입니다.
    *   `O(n^2)` (제곱 시간): 입력 크기 $n$의 제곱에 비례하여 시간이 증가합니다. 중첩된 반복문에서 흔히 나타납니다.
    *   `O(n^3)` (세제곱 시간): 입력 크기 $n$의 세제곱에 비례하여 시간이 증가합니다.
    *   `O(2^n)` (지수 시간): 입력 크기 $n$에 대해 시간이 기하급수적으로 증가합니다. $n$이 조금만 커져도 실용적으로 사용하기 어렵습니다.
    *   `O(n!)` (팩토리얼 시간): 입력 크기 $n$의 팩토리얼에 비례하여 시간이 증가합니다. 지수 시간보다 훨씬 빠르게 증가하며, $n$이 매우 작을 때만 사용할 수 있습니다.

그래프는 $n$이 증가함에 따라 각 성장률이 얼마나 빠르게 증가하는지를 시각적으로 보여주며, `O(1)`부터 `O(2^n)`까지 효율성이 크게 차이남을 알 수 있습니다.

### **코드/수식 해설**

알고리즘의 시간/공간 복잡도는 주로 다음 수학적 함수들로 표현됩니다. 여기서 $n$은 알고리즘의 입력 크기를 나타냅니다.

*   **상수 시간 (Constant Time)**:
    $O(1)$
    $n$의 크기와 무관하게 연산 횟수가 일정합니다.
*   **로그 시간 (Logarithmic Time)**:
    $O(\log n)$
    입력 크기가 증가할 때 연산 횟수는 매우 느리게 증가합니다. 주로 데이터를 반씩 줄여나가는 알고리즘에서 나타납니다.
*   **선형 시간 (Linear Time)**:
    $O(n)$
    입력 크기 $n$에 비례하여 연산 횟수가 증가합니다.
*   **선형 로그 시간 (Log-linear Time)**:
    $O(n \log n)$
    $O(n)$과 $O(n^2)$의 중간 정도의 효율성을 가집니다. 효율적인 정렬 알고리즘에서 자주 볼 수 있습니다.
*   **제곱 시간 (Quadratic Time)**:
    $O(n^2)$
    입력 크기 $n$의 제곱에 비례하여 연산 횟수가 증가합니다. 주로 이중 반복문 구조에서 나타납니다.
*   **세제곱 시간 (Cubic Time)**:
    $O(n^3)$
    입력 크기 $n$의 세제곱에 비례하여 연산 횟수가 증가합니다. 주로 삼중 반복문 구조에서 나타납니다.
*   **지수 시간 (Exponential Time)**:
    $O(2^n)$
    입력 크기 $n$이 1씩 증가할 때마다 연산 횟수가 2배로 증가하는 등 기하급수적으로 증가합니다.
*   **팩토리얼 시간 (Factorial Time)**:
    $O(n!)$
    입력 크기 $n$의 팩토리얼에 비례하여 연산 횟수가 증가합니다. $n$이 조금만 커져도 엄청난 연산량을 요구합니다.

### **구체적 예시**

*   **`O(1)`**: 배열에서 특정 인덱스의 원소를 직접 접근하는 경우.
    ```c++
    int arr[10];
    int value = arr[5]; // 입력 크기(배열 크기)와 무관하게 1번의 연산으로 접근
    ```
*   **`O(log n)`**: 이진 탐색(Binary Search) 알고리즘. 정렬된 배열에서 중간값을 기준으로 탐색 범위를 절반씩 줄여나갑니다.
*   **`O(n)`**: 배열의 모든 원소를 한 번씩 순회하며 합계를 계산하거나, 최댓값을 찾는 경우.
    ```c++
    int sum = 0;
    for (int i = 0; i < n; i++) {
        sum += arr[i]; // n번의 연산
    }
    ```
*   **`O(n log n)`**: 병합 정렬(Merge Sort)이나 힙 정렬(Heap Sort)과 같은 효율적인 정렬 알고리즘.
*   **`O(n^2)`**: 버블 정렬(Bubble Sort), 선택 정렬(Selection Sort) 또는 이중 반복문을 사용하여 모든 쌍을 비교하는 경우 (예: 인접 행렬로 표현된 그래프에서 모든 두 노드 간의 경로를 찾는 초기 단계).
    ```c++
    for (int i = 0; i < n; i++) {
        for (int j = 0; j < n; j++) {
            // n * n 번의 연산
        }
    }
    ```
*   **`O(n^3)`**: 플로이드-워셜(Floyd-Warshall) 알고리즘을 사용해 모든 쌍 최단 경로를 찾는 경우 (그래프의 정점 수가 $n$일 때), 또는 3차원 행렬 곱셈.
*   **`O(2^n)`**: 재귀적으로 모든 부분집합을 생성하거나, 백트래킹을 이용한 브루트 포스(Brute-Force) 방법으로 NP-hard 문제(예: 작은 크기의 외판원 문제)를 해결하는 경우.
*   **`O(n!)`**: 모든 가능한 순열을 생성하는 경우. (예: 브루트 포스 방식으로 외판원 문제를 해결할 때 모든 순열을 탐색하는 경우)

### **시험 포인트**

*   ⭐**각 Big-O 표기법의 의미와 성장률 특징**: `O(1)`부터 `O(n!)`까지 각 표기법이 나타내는 시간/공간 복잡도의 의미와 $n$ 값에 따른 성장 속도의 상대적 크기(예: `O(n^2)`가 `O(n log n)`보다 훨씬 빠르게 증가)를 정확히 이해해야 합니다. 그래프를 통해 시각적으로 이해하는 것도 중요합니다.
*   ⭐**알고리즘의 시간 복잡도 분석**: 특정 코드 스니펫이나 주어진 알고리즘에 대해 Big-O 표기법으로 시간 복잡도를 분석할 수 있어야 합니다. 반복문의 중첩 정도, 재귀 호출의 깊이, 데이터 처리 방식 등을 고려합니다.
*   ⭐**주요 자료구조 및 알고리즘의 시간 복잡도 암기**: 이진 탐색(`O(log n)`), 배열 순회(`O(n)`), 병합 정렬/힙 정렬(`O(n log n)`), 버블 정렬/선택 정렬(`O(n^2)`) 등 자주 등장하는 자료구조 연산 및 알고리즘의 평균 및 최악 시간 복잡도를 알고 있어야 합니다.
*   ⭐**효율성 비교**: 여러 알고리즘 중 어떤 것이 더 효율적인지(낮은 성장률을 가질수록 효율적) Big-O 표기법을 통해 비교하고 선택할 수 있어야 합니다.

## Big-O Notation: Examples

---

### **핵심 개념**
Big-O 표기법($O$)은 함수의 **점근적 상한(asymptotic upper bound)**을 나타냅니다. 이는 입력 크기 $n$이 충분히 커질 때, 주어진 함수 $T(n)$의 성장률이 특정 함수 $g(n)$의 성장률을 넘지 않음을 의미합니다.

정확한 정의는 다음과 같습니다:
함수 $T(n)$이 $O(g(n))$에 속한다는 것은, 양의 상수 $c$와 $n_0$가 존재하여 모든 $n \ge n_0$에 대해 다음 부등식을 만족한다는 것입니다.
$$0 \le T(n) \le c \cdot g(n)$$
시간 복잡도 분석에서는 최고차항(dominant term)만을 고려하며, 계수(constant factor)와 낮은 차수의 항들은 무시합니다. 이는 $n$이 매우 커질 때 이들의 영향력이 미미해지기 때문입니다.

---

### **코드/수식 해설**

**1. $T(n) = (n+1)^2$은 $O(n^2)$이다.**

이 예시는 $T(n)$이 $O(n^2)$에 속함을 정의에 따라 증명하는 과정입니다.
먼저 $T(n)$을 전개합니다:
$$T(n) = (n+1)^2 = n^2 + 2n + 1$$
이제 $0 \le n^2 + 2n + 1 \le c \cdot n^2$를 만족하는 $c$와 $n_0$를 찾아야 합니다.
$n \ge 1$일 때, 다음 부등식들이 성립합니다:
*   $2n \le 2n^2$
*   $1 \le n^2$

이것들을 활용하여 $T(n)$의 상한을 추정합니다:
$$n^2 + 2n + 1 \le n^2 + 2n^2 + n^2 = 4n^2$$
따라서, $n \ge 1$일 때 $T(n) \le 4n^2$가 성립합니다.
여기서 우리는 **$c=4$** 와 **$n_0=1$** 을 선택할 수 있습니다.
이로써 $T(n) = (n+1)^2$이 $O(n^2)$에 속함을 증명했습니다.

**2. 추가 예시들**

*   $3n^3 \in O(n^3)$: 이는 **tight bound (타이트 상한)** 입니다. 함수 $3n^3$의 점근적 상한이 실제 성장률 $n^3$과 동일하다는 의미입니다. $c=3, n_0=1$로 증명 가능합니다.
*   $3n^3 \in O(n^4)$: 이는 **loose bound (루즈 상한)** 입니다. $n^3$은 $n^4$보다 느리게 성장하므로 $O(n^4)$에도 속하지만, 가능한 가장 "tight"한 상한은 아닙니다. $c=1, n_0=1$로 증명 가능합니다 (왜냐하면 $3n^3 \le 1 \cdot n^4$ for $n \ge 3$ 이므로 $c=1, n_0=3$이 됩니다).
*   $3n^3 + 2n^2 + 8 \in O(n^3)$: 최고차항은 $n^3$이므로, Big-O 표기에서는 낮은 차수의 항($2n^2$, $8$)은 무시됩니다.
*   $2^{100} \in O(1)$: $2^{100}$은 상수(매우 큰 상수일지라도)이므로, 입력 $n$의 크기에 관계없이 일정한 시간이 걸린다는 의미의 $O(1)$에 속합니다.
*   $3\log(n) + 5 \in O(\log(n))$: 최고차항인 $\log(n)$만 남으며, 상수 계수 $3$과 상수항 $5$는 무시됩니다.

---

### **구체적 예시**
여러분에게 어떤 과제를 줬을 때 "이 과제는 아무리 오래 걸려도 `n`명의 친구와 한 번씩 전화하는 시간($O(n)$)을 넘지 않을 거야"라고 말하는 것과 같습니다.
만약 과제를 수행하는 데 걸리는 실제 시간이 $T(n) = n+5$분이라고 가정합시다. $n$이 1명일 때는 $6$분, $n$이 10명일 때는 $15$분, $n$이 100명일 때는 $105$분입니다.
반면 $c \cdot g(n)$은 $2 \cdot n$이라고 할 수 있습니다. $n$이 1명일 때는 $2$분, $n$이 10명일 때는 $20$분, $n$이 100명일 때는 $200$분입니다.
$n$이 5보다 크거나 같을 때($n_0=5$), $T(n) = n+5 \le 2n$이 항상 성립합니다. ($5 \le n$).
이처럼 $n$이 충분히 커지면 $T(n)$은 $c \cdot g(n)$보다 항상 작거나 같게 되어, $T(n) \in O(n)$이 됩니다.

---

### **시험 포인트**
*   ⭐ **Big-O 표기법의 정의($0 \le T(n) \le c \cdot g(n)$ for all $n \ge n_0$)를 정확하게 암기하고 있어야 합니다.** 특히 $c$와 $n_0$가 무엇을 의미하는지, 그리고 이들을 어떻게 찾을 수 있는지 이해하는 것이 중요합니다.
*   ⭐ 주어진 함수의 **가장 "tight"한 Big-O 상한을 찾는 연습**을 충분히 하세요. 예를 들어 $3n^3+2n^2+8$은 $O(n^3)$이지 $O(n^4)$라고 답하는 것은 틀린 것은 아니지만, 문제에서 요구하는 바는 대개 가장 근접한(tightest) 상한입니다.
*   ⭐ 상수항($2^{100}$)과 낮은 차수의 항($2n^2, 8$)이 Big-O 표기에서 왜 무시되는지 그 원리를 설명할 수 있어야 합니다. 즉, $n$이 무한대로 갈 때 최고차항의 영향이 지배적이기 때문입니다.
*   ⭐ **Tight bound**와 **Loose bound**의 개념 차이를 정확히 구분하고 설명할 수 있어야 합니다. $3n^3 \in O(n^3)$은 tight bound의 예시이고, $3n^3 \in O(n^4)$은 loose bound의 예시입니다.

### Big-O 표기법의 주요 성질

이 슬라이드는 알고리즘의 시간/공간 복잡도를 분석할 때 사용되는 Big-O 표기법의 기본적인 성질들을 설명합니다. 이 성질들은 여러 부분으로 구성된 알고리즘의 전체 복잡도를 계산하는 데 필수적입니다.

---

-   **핵심 개념**: Big-O 표기법은 함수의 점근적 상한(asymptotic upper bound)을 나타내며, 특히 입력 크기 $n$이 매우 커질 때 함수의 증가율을 표현합니다. 여기에 제시된 성질들은 복수의 시간 복잡도를 가진 연산이나 서브루틴이 결합될 때 전체 복잡도를 어떻게 도출하는지 알려줍니다.

-   **코드/수식 해설**:
    *   **덧셈 규칙 (Addition rule)**:
        두 함수 $T_1(n)$과 $T_2(n)$의 Big-O 복잡도가 각각 $O(f(n))$과 $O(g(n))$일 때, 두 함수를 더한 $T_1(n) + T_2(n)$의 복잡도는 $f(n)$과 $g(n)$ 중 증가율이 더 큰 함수의 Big-O로 결정됩니다.
        $$T_1(n) \in O(f(n)) \text{ and } T_2(n) \in O(g(n)) \Rightarrow T_1(n) + T_2(n) \in O(\max\{f(n), g(n)\})$$

    *   **곱셈 규칙 (Product rule)**:
        두 함수 $T_1(n)$과 $T_2(n)$의 Big-O 복잡도가 각각 $O(f(n))$과 $O(g(n))$일 때, 두 함수를 곱한 $T_1(n) \cdot T_2(n)$의 복잡도는 $f(n)$과 $g(n)$의 곱의 Big-O로 결정됩니다.
        $$T_1(n) \in O(f(n)) \text{ and } T_2(n) \in O(g(n)) \Rightarrow T_1(n) \cdot T_2(n) \in O(f(n) \cdot g(n))$$

    *   **기타 성질 (Others)**:
        1.  **상수 곱셈 규칙**: $a > 0$인 임의의 상수 $a$에 대해, $T(n) \in O(f(n))$이면 $a \cdot T(n) \in O(f(n))$입니다. 이는 Big-O 표기법이 상수 계수를 무시한다는 것을 의미합니다.
            $$\text{For any constant } a > 0, T(n) \in O(f(n)) \Rightarrow a \cdot T(n) \in O(f(n))$$
        2.  **전이성 (Transitivity)**: $T(n)$이 $f(n)$에 의해 상한이 결정되고, $f(n)$이 $g(n)$에 의해 상한이 결정된다면, $T(n)$도 $g(n)$에 의해 상한이 결정됩니다.
            $$T(n) \in O(f(n)) \text{ and } f(n) \in O(g(n)) \Rightarrow T(n) \in O(g(n))$$
        3.  **다항식 규칙 (Polynomial rule)**: 차수(degree)가 $d$인 다항식 $T(n)$의 Big-O 복잡도는 최고 차항에 따라 $O(n^d)$가 됩니다.
            $$T(n) : \text{polynomial of degree } d \Rightarrow T(n) \in O(n^d)$$

-   **구체적 예시**:
    *   **덧셈 규칙 예시**:
        알고리즘 A가 $T_1(n) = 3n^2 + 2n$의 시간 복잡도를 가지고 ($O(n^2)$), 알고리즘 B가 $T_2(n) = 5n + 7$의 시간 복잡도를 가진다고 ($O(n)$) 합시다. 이 두 알고리즘을 순차적으로 실행한다면 (즉, A 실행 후 B 실행), 전체 시간 복잡도는 $T_1(n) + T_2(n) = (3n^2 + 2n) + (5n + 7) = 3n^2 + 7n + 7$이 됩니다. Big-O 표기법으로는 $O(\max\{n^2, n\}) = O(n^2)$ 입니다. 항상 더 높은 증가율을 가진 항이 전체를 지배합니다.

    *   **곱셈 규칙 예시**:
        배열을 순회하며(복잡도 $O(n)$) 각 원소에 대해 이진 탐색(복잡도 $O(\log n)$)을 수행하는 알고리즘이 있다고 가정해 봅시다. 이 경우 전체 시간 복잡도는 각 단계의 복잡도를 곱한 $O(n \cdot \log n) = O(n \log n)$이 됩니다.

    *   **기타 성질 예시**:
        *   **상수 곱셈**: $T(n) = 50n^2$의 알고리즘은 $O(n^2)$ 입니다. $10 \cdot T(n) = 500n^2$ 역시 $O(n^2)$ 입니다. 상수 계수는 Big-O에서 무시됩니다.
        *   **다항식**: $T(n) = 7n^3 - 2n^2 + 100$은 3차 다항식이므로, $O(n^3)$입니다. 최고 차항 $n^3$이 $n$이 커질수록 다른 항들을 압도하기 때문입니다.

-   **시험 포인트**:
    *   ⭐ 두 개 이상의 알고리즘 또는 코드 블록이 순차적으로 실행될 때 (덧셈 관계), 전체 복잡도는 **가장 높은 증가율을 가진 항**으로 결정됩니다. ($O(\max\{f(n), g(n)\})$)
    *   ⭐ 루프 안에 다른 루프나 함수 호출이 있을 때 (곱셈 관계), 전체 복잡도는 각 부분의 복잡도를 **곱한 값**으로 결정됩니다. ($O(f(n) \cdot g(n))$)
    *   ⭐ Big-O 표기법에서 **상수 계수는 무시**됩니다. (예: $5n^2$는 $O(n^2)$)
    *   ⭐ 다항식 형태의 시간 복잡도($3n^2 + 5n + 10$)는 **최고 차항**의 증가율로 표현됩니다. (예: $O(n^2)$)
    *   이러한 Big-O 성질들을 이용하여 주어진 코드 스니펫의 시간 복잡도를 정확하게 분석하는 문제가 출제될 수 있습니다.

## Slide 19

**핵심 개념**:
*   **Big-Omega ($\Omega$)**: 알고리즘의 점근적 하한(Asymptotic Lower Bound)을 나타냅니다. 이는 알고리즘의 실행 시간이 아무리 빨라도 특정 함수 $f(n)$의 상수배보다 적게 걸릴 수 없음을 의미합니다. 즉, "최소한 이만큼은 걸린다"는 보장을 제시할 때 사용됩니다.
*   **Big-Theta ($\Theta$)**: 알고리즘의 점근적 상한(Upper Bound)과 하한(Lower Bound)이 동일할 때 사용됩니다. 이는 알고리즘의 실행 시간이 특정 함수 $f(n)$의 상수배 사이에서 점근적으로 성장한다는 것을 의미하며, 알고리즘의 '정확한' 점근적 성장률을 나타냅니다.

**코드/수식 해설**:

1.  **Big-Omega ($\Omega$) 정의**:
    어떤 함수 $T(n)$이 $\Omega(f(n))$에 속한다는 것은, 충분히 큰 $n$에 대해 $T(n)$이 $f(n)$의 어떤 양의 상수배보다 항상 크거나 같음을 의미합니다.

    $$T(n) \in \Omega(f(n)) \text{ if there exist constants } c > 0 \text{ and } n_0 \ge 1 \text{ such that }$$
    $$T(n) \ge c \cdot f(n) \text{ for all } n \ge n_0$$

    여기서 $T(n)$은 알고리즘의 실행 시간 또는 자원 사용량을 나타내는 함수이며, $f(n)$은 비교의 기준이 되는 함수입니다. 상수 $c$와 $n_0$는 $T(n)$이 $c \cdot f(n)$ 아래로 떨어지지 않는 시점을 정의하는 임계값입니다.

2.  **Big-Theta ($\Theta$) 정의**:
    어떤 함수 $T(n)$이 $\Theta(f(n))$에 속한다는 것은, $n$이 충분히 커질 때 $T(n)$이 $f(n)$의 두 개의 다른 양의 상수배 사이에 존재함을 의미합니다. 이는 $T(n)$이 $O(f(n))$이면서 동시에 $\Omega(f(n))$일 때 성립합니다.

    $$T(n) \in \Theta(f(n)) \text{ if there exist constants } c_1 > 0, c_2 > 0 \text{ and } n_0 \ge 1 \text{ such that }$$
    $$c_1 \cdot f(n) \le T(n) \le c_2 \cdot f(n) \text{ for all } n \ge n_0$$

    이 정의는 $T(n)$의 성장률이 $f(n)$의 성장률과 점근적으로 동일하다는 것을 나타냅니다. $c_1 \cdot f(n)$은 하한을, $c_2 \cdot f(n)$은 상한을 형성합니다.

**구체적 예시**:
*   **Big-Omega ($\Omega$)**:
    퀵 정렬(Quick Sort) 알고리즘을 생각해봅시다. 퀵 정렬의 최악의 경우 시간 복잡도는 $O(n^2)$이지만, 최선의 경우와 평균적인 경우의 시간 복잡도는 $O(n \log n)$입니다.
    이때, 퀵 정렬은 $\Omega(n \log n)$이라고 말할 수 있습니다. 즉, 아무리 데이터 배열이 이상적으로 주어져서 정렬이 빨리 된다 하더라도, $n \log n$보다 더 빠르게 정렬될 수는 없다는 '하한'을 제시하는 것입니다. $n \log n$은 대부분의 비교 기반 정렬 알고리즘의 이론적 하한이기도 합니다.

*   **Big-Theta ($\Theta$)**:
    배열에서 특정 원소를 찾는 선형 탐색(Linear Search) 알고리즘을 예로 들어봅시다.
    *   최선의 경우: 찾는 원소가 배열의 맨 앞에 있어서 1번의 비교만으로 찾을 때. ($O(1)$)
    *   최악의 경우: 찾는 원소가 배열의 맨 뒤에 있거나 배열에 없어서 $n$번의 비교를 할 때. ($O(n)$)
    이 알고리즘의 시간 복잡도는 Big-O 표기로는 $O(n)$입니다. 그런데 이 알고리즘은 또한 $\Omega(n)$이기도 합니다. 왜냐하면 아무리 빨리 끝난다 해도, 원소가 없거나 맨 뒤에 있을 경우 $n$번의 연산이 필요하므로 최소한 $n$에 비례하는 연산이 필요할 수 있기 때문입니다 (엄밀히는 $c \cdot n$이 아니라 $1 \cdot n$에 가까움).
    따라서 선형 탐색의 '정확한' 점근적 성장률은 $\Theta(n)$입니다. 이는 $n$이 충분히 커질 때, 선형 탐색의 실행 시간이 $n$에 비례하여 증가한다는 것을 의미합니다.

**시험 포인트**:
*   ⭐ **각 점근적 표기법 ($\Omega$, $\Theta$, $O$)의 정의를 수식과 함께 정확히 설명하고, 이들의 의미(상한, 하한, 정확한 성장률)를 구분할 수 있어야 합니다.** 특히, $c > 0$ 및 $n_0 \ge 1$ 같은 상수 조건들을 놓치지 마세요.
*   ⭐ **주어진 함수 $T(n)$에 대해 Big-Omega 또는 Big-Theta 표기법을 사용하여 증명하는 과정(적절한 $c$, $c_1$, $c_2$, $n_0$ 찾기)을 이해하고 적용할 수 있어야 합니다.**
*   ⭐ **Big-O는 '최대' 성능을, Big-Omega는 '최소' 성능을, Big-Theta는 '평균적' 또는 '정확한' 성능을 나타내는 용도로 쓰일 수 있음을 이해하는 것이 중요합니다.** 예를 들어, 정렬 알고리즘의 최악 시간 복잡도는 $O(f(n))$으로 표현하고, 이론적 하한은 $\Omega(g(n))$으로 표현하며, 정렬 알고리즘 자체의 일반적인 성능은 $\Theta(h(n))$으로 표현할 수 있습니다.
*   ⭐ **그래프 상에서 $T(n)$과 $cf(n)$ ($c_1f(n), c_2f(n)$) 곡선이 어떤 관계를 가지는지 시각적으로 이해하고 설명할 수 있어야 합니다.** $n_0$ 이후의 관계가 핵심입니다.

## Asymptotic Algorithm Analysis

### 핵심 개념

*   **점근적 알고리즘 분석 (Asymptotic Algorithm Analysis)**: 알고리즘의 실행 시간(running time)을 입력 크기(input size)의 함수로 결정하고, 이를 **빅-O 표기법(big-Oh notation)**으로 표현하는 방법입니다. 실제 물리적인 시간 측정 대신, 알고리즘이 수행하는 기본 연산(primitive operations)의 횟수를 분석하여 알고리즘의 효율성을 평가합니다.
*   **실행 시간 (Running Time)**: 알고리즘이 특정 작업을 완료하는 데 필요한 시간입니다. 점근적 분석에서는 컴퓨터의 속도나 특정 프로그래밍 언어의 특징 등 하드웨어/소프트웨어 환경에 의존하지 않는 추상적인 연산 횟수를 사용합니다.
*   **입력 크기 (Input Size)**: 알고리즘에 주어지는 데이터의 양을 나타내는 매개변수입니다. 보통 $n$으로 표현하며, 예를 들어 배열의 원소 수, 연결 리스트의 노드 수, 그래프의 정점/간선 수 등이 될 수 있습니다.
*   **기본 연산 (Primitive Operations)**: 컴퓨터가 일정하고 짧은 시간 안에 수행할 수 있는 가장 기본적인 연산들을 의미합니다.
    *   변수 할당 (e.g., `x = 10`)
    *   산술 연산 (e.g., `a + b`, `c * d`)
    *   비교 연산 (e.g., `x < y`, `a == b`)
    *   배열 인덱싱 (e.g., `arr[i]`)
    *   함수 호출, 리턴 등
*   **최악의 경우 (Worst-Case)**: 알고리즘의 실행 시간이 가장 길어지는 입력 상황을 의미합니다. 점근적 분석은 일반적으로 최악의 경우를 기준으로 수행하여, 알고리즘 성능의 상한(upper bound)을 보장합니다.
*   **빅-O 표기법 (Big-O Notation)**: 알고리즘의 실행 시간 함수 $T(n)$의 점근적 상한(asymptotic upper bound)을 나타내는 수학적 표기법입니다. 입력 크기가 무한대로 커질 때, $T(n)$이 특정 함수 $g(n)$보다 빠르게 증가하지 않음을 의미합니다.

### 코드/수식 해설

**빅-O 표기법의 정의**:
함수 $T(n)$이 $O(g(n))$이라는 것은, 양의 상수 $c$와 $n_0$가 존재하여 모든 $n \ge n_0$에 대해 다음 부등식이 성립함을 의미합니다.

$$ T(n) \le c \cdot g(n) \quad \text{for all } n \ge n_0 $$

여기서
*   $T(n)$: 알고리즘의 실제 실행 시간(또는 기본 연산 횟수)을 나타내는 함수입니다.
*   $g(n)$: 알고리즘의 실행 시간을 "대략적으로" 표현하는 간단한 함수 (예: $n$, $n^2$, $\log n$)입니다.
*   $c$: 양의 상수입니다. $T(n)$의 성장률이 $g(n)$의 성장률을 넘지 않도록 크기를 조절하는 역할을 합니다.
*   $n_0$: 양의 상수입니다. 충분히 큰 입력 크기($n_0$ 이상)에 대해서만 위 부등식이 성립하면 됩니다. 작은 입력에 대해서는 빅-O 표기법이 의미하는 바가 크지 않습니다.

### 구체적 예시

**예시: 배열에서 특정 값 찾기 (선형 탐색)**

정수 배열 `arr` (크기 $n$)에서 특정 값 `target`을 찾는 선형 탐색(Linear Search) 알고리즘을 생각해 봅시다.

```c
int linearSearch(int arr[], int n, int target) {
    for (int i = 0; i < n; i++) { // (1)
        if (arr[i] == target) {   // (2)
            return i;             // (3)
        }
    }
    return -1;                    // (4)
}
```

1.  **기본 연산 분석**:
    *   `i = 0`: 할당 연산 (1회)
    *   `i < n`: 비교 연산 (반복 횟수 + 1회)
    *   `i++`: 증가 연산 (반복 횟수)
    *   `arr[i] == target`: 배열 인덱싱, 비교 연산 (반복 횟수)
    *   `return i`: 리턴 연산 (1회)
    *   `return -1`: 리턴 연산 (1회)

2.  **최악의 경우 (Worst-Case) 찾기**:
    *   `target`이 배열에 없거나, 배열의 가장 마지막 원소인 경우
    *   이 경우, `for` 루프는 $n$번 모두 실행됩니다.
    *   루프 내의 `arr[i] == target` 비교 연산도 $n$번 발생합니다.
    *   총 기본 연산 횟수는 대략 $1 + (n+1) + n + n \approx 3n + 2$ 회입니다. (자세히 따지면 더 복잡하지만, $n$에 비례하는 형태가 중요합니다.)

3.  **빅-O 표기법으로 표현**:
    *   알고리즘의 최악의 경우 실행 시간 함수 $T(n) = 3n + 2$라고 가정합니다.
    *   $g(n) = n$이라고 할 때, 충분히 큰 $n$에 대해 $3n+2 \le c \cdot n$이 성립하는 $c$와 $n_0$를 찾을 수 있습니다. 예를 들어, $c=4$로 두면 $3n+2 \le 4n$은 $n \ge 2$일 때 성립합니다.
    *   따라서 선형 탐색 알고리즘의 최악의 경우 시간 복잡도는 ⭐**$O(n)$**⭐ 입니다.

### 시험 포인트

*   ⭐**점근적 분석의 목적**: 알고리즘의 효율성을 하드웨어/소프트웨어 독립적으로, 그리고 대규모 입력에 대해 평가하기 위함입니다.
*   ⭐**빅-O 표기법의 의미**: 알고리즘 실행 시간의 **점근적 상한(asymptotic upper bound)**을 나타냅니다. 즉, 아무리 나빠도 이 시간 복잡도 이상으로 느려지지 않음을 보장합니다.
*   ⭐**최악의 경우(Worst-Case) 분석의 중요성**: 대부분의 알고리즘 분석에서 최악의 경우를 사용하는 이유는, 이는 성능의 "보장"을 제공하기 때문입니다. 실제 시스템에서 발생할 수 있는 가장 느린 상황에 대비할 수 있습니다.
*   **기본 연산(Primitive Operations)의 개념**: 코드의 각 줄이 대략 몇 개의 기본 연산으로 구성되는지 이해하는 것이 중요합니다.
*   **시간 복잡도 계산 절차**:
    1.  주어진 알고리즘의 각 연산이 몇 번 수행되는지 입력 크기 $n$의 함수로 나타냅니다.
    2.  가장 많은 연산이 수행되는 **최악의 시나리오**를 식별합니다.
    3.  최악의 경우 총 기본 연산 횟수 $T(n)$을 찾습니다.
    4.  $T(n)$에서 가장 영향력이 큰 항만 남기고 계수와 낮은 차수의 항을 제거하여 빅-O 표기법으로 표현합니다. (예: $3n^2 + 2n + 5 \rightarrow O(n^2)$)

## Slide 21

**핵심 개념**:
이 슬라이드는 알고리즘의 성능을 평가하는 데 사용되는 비점근적 분석(Asymptotic Analysis)의 기본 가정을 설명합니다. 비점근적 분석은 알고리즘의 실행 시간이 입력 크기 $N$에 따라 어떻게 변화하는지, 즉 알고리즘의 성장률을 대략적으로 파악하는 데 중점을 둡니다. 이를 위해 다음과 같은 두 가지 주요 가정을 합니다.

1.  **기본 연산(Basic operations)은 상수 시간(constant time)이 소요됩니다.**
    이는 산술 연산 (고정 너비 정수), 변수 할당, 배열의 특정 인덱스에 접근, 두 단순 값 비교와 같은 기본적인 명령들이 입력의 크기에 관계없이 항상 동일한 시간이 걸린다고 가정하는 것입니다. 실제 컴퓨터 내부의 미세한 시간 차이는 무시하고, 이러한 연산들의 비용을 $O(1)$로 간주합니다.

2.  **다른 복잡한 연산들은 기본 연산들의 합(summations) 또는 곱(products)으로 구성됩니다.**
    *   **연속적인 문장들(Consecutive statements)**: 순차적으로 실행되는 문장들의 총 시간은 각 문장의 시간을 합산합니다.
    *   **반복문(Loops)**: 반복문의 전체 비용은 '반복문 본체(loop body)의 비용'과 '반복 횟수(number of loops)'를 곱하여 계산합니다. 이는 중첩 반복문 분석에 특히 중요하게 적용됩니다.

이러한 가정들은 알고리즘의 효율성을 수학적으로 모델링하고, 서로 다른 알고리즘의 성능을 객관적으로 비교할 수 있는 추상화된 기반을 제공합니다.

**코드/수식 해설**:

1.  **상수 시간 ($O(1)$) 연산**:
    다음과 같은 C 코드는 모두 상수 시간 연산으로 간주됩니다.
    ```c
    int x = 10;           // 할당
    int y = x + 5;        // 산술 연산 및 할당
    int arr[100];
    arr[0] = y;           // 배열 인덱스 접근 및 할당
    if (x < y) {          // 단순 값 비교
        // ...
    }
    ```
    각 라인의 실행 시간은 입력 크기(예: 배열의 전체 크기나 변수 x, y의 값)에 영향을 받지 않으므로, 모두 $O(1)$로 표현됩니다.

2.  **연속적인 문장의 합산**:
    두 개의 상수 시간 연산이 연속으로 있으면 총 시간은 합산됩니다.
    $$ T(\text{statement 1}) + T(\text{statement 2}) = O(1) + O(1) = O(1) $$
    즉, 여러 개의 상수 시간 연산이 순차적으로 실행되어도 전체 시간 복잡도는 여전히 $O(1)$입니다.

3.  **반복문 (Loop)의 비용 계산**:
    반복문의 총 비용은 `(반복문 본체 비용) * (반복 횟수)`로 계산됩니다.
    ```c
    for (int i = 0; i < N; i++) {
        // 반복문 본체 (O(1) 연산)
        int result = i * 2; // O(1)
    }
    ```
    이 반복문 본체는 상수 시간 $O(1)$ 연산을 포함하고, $N$번 실행됩니다. 따라서 총 시간 복잡도는 다음과 같습니다:
    $$ T(\text{loop}) = (\text{cost of loop body}) \times (\text{number of loops}) $$
    $$ T(\text{loop}) = O(1) \times N = O(N) $$

    중첩 반복문의 경우, 이러한 곱셈 규칙이 확장됩니다.
    ```c
    for (int i = 0; i < N; i++) {
        for (int j = 0; j < N; j++) {
            // 반복문 본체 (O(1) 연산)
            int sum = i + j; // O(1)
        }
    }
    ```
    이 예시의 시간 복잡도는 다음과 같습니다:
    $$ T(\text{nested loop}) = (\text{cost of inner loop body}) \times (\text{number of inner loops}) \times (\text{number of outer loops}) $$
    $$ T(\text{nested loop}) = O(1) \times N \times N = O(N^2) $$

**구체적 예시**:

*   **상수 시간 연산의 비유**:
    도서관에서 책을 찾을 때, 책의 정확한 위치(예: "컴퓨터공학 섹션, 3번째 줄, 5번째 책")를 알고 있다면 도서관의 규모(전체 책의 수)와 관계없이 책을 집어 드는 시간은 거의 동일합니다. 이는 배열의 특정 인덱스에 $O(1)$로 접근하는 것과 유사합니다. 반면, 책의 제목만 알고 처음부터 끝까지 찾아야 한다면, 책의 수가 많을수록 시간이 더 걸릴 것입니다.

*   **반복문 시간 복잡도 계산 예시**:
    어떤 학교의 학생 $N$명의 시험 점수가 `scores` 배열에 저장되어 있습니다. 이 점수들을 모두 출력하는 함수를 생각해 봅시다.

    ```c
    void print_scores(int scores[], int N) {
        for (int i = 0; i < N; i++) {
            printf("%d\n", scores[i]); // (1)
        }
    }
    ```
    1.  `scores[i]`에 접근하는 것은 $O(1)$ 연산입니다.
    2.  `printf` 함수로 값을 출력하는 것도 (콘솔 출력의 오버헤드는 무시하고) $O(1)$ 기본 연산으로 간주합니다.
    따라서 반복문 본체 `printf("%d\n", scores[i]);`는 총 $O(1)$의 비용을 가집니다.
    이 반복문은 `i`가 `0`부터 `N-1`까지, 총 $N$번 실행됩니다.
    그러므로 `print_scores` 함수의 전체 시간 복잡도는 $O(1) \times N = O(N)$이 됩니다. 학생 수가 많아질수록 모든 점수를 출력하는 데 걸리는 시간도 학생 수에 비례하여 증가함을 알 수 있습니다.

**시험 포인트**:

*   ⭐**어떤 종류의 연산들이 상수 시간 $O(1)$으로 간주되는지 정확히 파악해야 합니다.** (산술 연산, 할당, 배열 인덱스 접근, 단순 비교) 이는 시간 복잡도 분석의 가장 기본적인 단위입니다.
*   ⭐**코드의 여러 부분이 순차적으로 실행될 때는 각 부분의 시간 복잡도를 합산하고, 반복문 안의 코드는 반복 횟수만큼 곱하여 계산하는 원칙을 확실히 이해해야 합니다.** 특히 중첩 반복문의 $O(N^2)$, $O(N \log N)$ 등 복잡도 계산에 중요합니다.
*   ⭐**비점근적 분석은 정확한 실행 시간보다는 입력 크기에 따른 알고리즘의 '성장률'을 예측하고 비교하는 데 목적이 있음을 이해해야 합니다.** 이러한 가정들은 분석을 단순화하고 핵심적인 성능 특성을 파악하기 위한 도구입니다.

## Slide 22

**핵심 개념**
이 슬라이드는 **점근적 알고리즘 분석(Asymptotic Algorithm Analysis)**의 구체적인 예시로, 행렬 곱셈 알고리즘의 시간 복잡도를 계산하는 과정을 보여줍니다. 점근적 분석은 입력 크기 $N$이 매우 커질 때 알고리즘의 실행 시간(또는 공간)이 어떻게 증가하는지를 수학적으로 예측하는 방법입니다. 여기서는 **Big O 표기법($O$)**을 사용하여 알고리즘의 실행 시간의 상한(upper bound)을 나타냅니다.

**코드/수식 해설**

**코드 설명:**
제시된 프로그램 세그먼트는 두 개의 $N \times N$ 행렬 $A$와 $B$를 곱하여 결과 행렬 $C$를 계산하는 표준 행렬 곱셈 알고리즘입니다.

```
for i:=1 to n do
  for j:=1 to n do begin
    C[i,j]:=0;
    for k:=1 to n do
      C[i,j]:=C[i,j]+A[i,k]*B[k,j]
  end
```

*   **가장 바깥 루프 (`for i:=1 to n do`)**: `i`는 행렬 $C$의 행 인덱스를 나타냅니다. 이 루프는 `n`번 반복됩니다.
*   **중간 루프 (`for j:=1 to n do begin`)**: `j`는 행렬 $C$의 열 인덱스를 나타냅니다. 이 루프는 `n`번 반복되며, 바깥 루프의 각 반복마다 실행됩니다.
*   **`C[i,j]:=0;`**: 행렬 $C$의 각 원소를 0으로 초기화하는 연산입니다. 이는 상수 시간 $c_1$이 소요됩니다.
*   **가장 안쪽 루프 (`for k:=1 to n do`)**: `k`는 행렬 곱셈의 정의에 따라 중간 합계를 계산하는 데 사용되는 인덱스입니다. 이 루프는 `n`번 반복되며, 중간 루프의 각 반복마다 실행됩니다.
*   **`C[i,j]:=C[i,j]+A[i,k]*B[k,j]`**: 실제 곱셈과 덧셈이 일어나는 연산입니다. 이는 상수 시간 $c_2$가 소요됩니다 (하드웨어에 따라 곱셈과 덧셈의 비용이 다르겠지만, 점근적 분석에서는 상수로 간주).

**수식 해설:**
알고리즘의 총 실행 시간 $T(N)$을 계산하는 과정은 다음과 같습니다.

$$
T(n) = \sum_{i=1}^{n} \sum_{j=1}^{n} \left(c_1 + \sum_{k=1}^{n} c_2\right)
$$
*   가장 안쪽 `k` 루프는 $c_2$ 비용의 연산을 $n$번 반복하므로 총 $c_2 \cdot n$ 시간이 걸립니다.
*   그 다음, `C[i,j]:=0`의 상수 비용 $c_1$과 `k` 루프의 비용 $c_2 \cdot n$을 합산합니다.
*   중간 `j` 루프는 이 $(c_1 + c_2 \cdot n)$ 연산을 $n$번 반복합니다.

$$
= \sum_{i=1}^{n} \sum_{j=1}^{n} (c_1 + c_2 \cdot n)
$$
*   따라서 `j` 루프의 내부 블록은 총 $n \cdot (c_1 + c_2 \cdot n) = c_1 \cdot n + c_2 \cdot n^2$ 시간이 걸립니다.
*   가장 바깥 `i` 루프는 이 $(c_1 \cdot n + c_2 \cdot n^2)$ 연산을 $n$번 반복합니다.

$$
= \sum_{i=1}^{n} (c_1 \cdot n + c_2 \cdot n^2)
$$
*   최종적으로 총 실행 시간 $T(N)$은 $n \cdot (c_1 \cdot n + c_2 \cdot n^2) = c_1 \cdot n^2 + c_2 \cdot n^3$ 입니다.

$$
= c_1 \cdot n^2 + c_2 \cdot n^3
$$
Big O 표기법은 $N$이 매우 커질 때 가장 지배적인(최고차) 항만 고려하므로, $n^3$이 가장 지배적인 항입니다. 따라서 이 알고리즘의 시간 복잡도는 $O(N^3)$ 입니다.

$$
\Rightarrow T(n) \in O(n^3)
$$

**구체적 예시**
$N=3$인 $3 \times 3$ 행렬 $A$와 $B$를 곱하여 $C$를 얻는다고 가정해 봅시다.
*   행렬 $C$에는 $3 \times 3 = 9$개의 원소 $C_{ij}$가 있습니다.
*   하나의 원소 $C_{ij}$를 계산하기 위해, 예를 들어 $C_{11}$을 계산하려면 $C_{11} = A_{11}B_{11} + A_{12}B_{21} + A_{13}B_{31}$ 와 같이 3번의 곱셈과 2번의 덧셈(총 3번의 `C[i,j]:=C[i,j]+A[i,k]*B[k,j]` 연산)이 필요합니다.
*   각 $C_{ij}$ 원소에 대해 $N$번의 `k` 루프 연산이 필요하므로, 총 $N^2$개의 원소를 계산하려면 약 $N^2 \times N = N^3$번의 핵심 연산이 수행됩니다.
*   실생활 비유: 만약 여러분이 $N$개의 층을 가진 건물에 살고 있고, 각 층마다 $N$개의 방이 있으며, 각 방에는 $N$개의 서랍이 있다고 상상해 보세요. 모든 서랍을 열어보고 그 안에 있는 물건을 하나씩 확인하는 작업은 $N \times N \times N = N^3$번의 행동을 요구합니다. 이것이 바로 $O(N^3)$ 시간 복잡도의 직관적인 의미입니다.

**시험 포인트**
*   ⭐**Big O 표기법의 정의와 의미**: $N$이 충분히 클 때 알고리즘의 실행 시간(또는 공간)이 어떻게 증가하는지 나타내는 상한(Upper Bound) 표기법임을 이해해야 합니다. 상수항이나 저차항은 무시됩니다.
*   ⭐**중첩 루프의 시간 복잡도 계산**: 중첩된 `for` 루프가 있을 때, 각 루프의 반복 횟수를 곱하여 전체 시간 복잡도를 계산하는 방법을 숙지해야 합니다. 특히, 루프 변수의 범위가 입력 크기 $N$에 따라 어떻게 달라지는지 파악하는 것이 중요합니다.
*   ⭐**주요 알고리즘의 시간 복잡도 암기/이해**: 선형 탐색 $O(N)$, 이진 탐색 $O(\log N)$, 정렬 알고리즘 (버블, 삽입, 선택: $O(N^2)$, 퀵, 병합: $O(N \log N)$) 등 자주 사용되는 알고리즘의 시간 복잡도는 기본적으로 알아두는 것이 좋습니다.
*   ⭐**상수 계수 및 저차항 무시의 원리**: $N$이 매우 커질 때 최고차항이 압도적으로 중요해지므로, 복잡도 분석 시 상수 계수와 저차항이 왜 무시되는지 설명할 수 있어야 합니다.

## Slide 23

**핵심 개념**:
이 슬라이드는 점근적 분석(Asymptotic Analysis), 즉 Big-O 표기법과 같은 시간/공간 복잡도 분석의 한계점을 설명합니다. 점근적 분석은 알고리즘의 확장성(scalability)을 이해하는 데 유용하지만, 실제 시스템의 동작을 완벽하게 반영하지 못하는 세 가지 주요 한계가 있습니다.

1.  **상수 인자(Constant Factors) 무시**: Big-O 표기법은 최고차항만을 고려하고 상수 인자를 무시합니다. 하지만 실제로는 이 상수 인자가 작은 입력 크기($n$) 범위에서 알고리즘의 성능을 결정하는 중요한 요소가 될 수 있습니다.
2.  **메모리 계층(Memory Hierarchy) 고려 부족**: 점근적 분석은 일반적으로 모든 메모리 접근 비용이 동일하다고 가정하지만, 실제 컴퓨터 시스템에서는 캐시(Cache), 주 메모리(Main Memory, MM), 하드 디스크 드라이브(Hard Disk Drive, HDD)와 같은 메모리 계층에 따라 데이터 접근 시간이 크게 다릅니다. 캐시 메모리가 가장 빠르고, HDD가 가장 느립니다.
3.  **연산 비용과 메모리 접근 비용의 불균형**: 알고리즘이 수행하는 계산(CPU 연산) 비용보다 주 메모리에서 데이터를 가져오는(메모리 접근) 비용이 훨씬 클 수 있습니다. 이로 인해 때로는 더 많은 계산을 수행하는 프로그램이 불필요한 메모리 접근을 줄여서 전체 실행 시간이 더 짧아지는 역설적인 상황이 발생할 수 있습니다.

**코드/수식 해설**:
1.  **상수 인자 예시**:
    알고리즘 A의 시간 복잡도가 $1000n$이고, 알고리즘 B의 시간 복잡도가 $n^2$이라고 가정합니다.
    *   점근적 분석 관점에서: $O(n)$은 $O(n^2)$보다 우수합니다.
    *   하지만 특정 $n$ 값 범위에서는 다릅니다. 예를 들어, $n < 1000$일 때, $1000n$은 $n^2$보다 크므로 알고리즘 A가 알고리즘 B보다 더 많은 시간이 걸립니다.
        *   $n = 10 \Rightarrow 1000 \times 10 = 10000$ vs $10^2 = 100$
        *   $n = 500 \Rightarrow 1000 \times 500 = 500000$ vs $500^2 = 250000$
        *   $n = 1000 \Rightarrow 1000 \times 1000 = 1000000$ vs $1000^2 = 1000000$
        *   $n = 2000 \Rightarrow 1000 \times 2000 = 2000000$ vs $2000^2 = 4000000$
    $n \geq 1000$이 되어서야 $1000n \leq n^2$이 성립하며, 이때부터 알고리즘 A가 B보다 빨라집니다.
2.  **메모리 계층 속도 비교**:
    $$ \text{Cache Memory } \ll \text{ MM } \ll \text{ HDD} $$
    이는 메모리 접근 속도(또는 비용)를 나타내며, 캐시 메모리가 가장 빠르고 하드 디스크 드라이브가 가장 느리다는 것을 의미합니다. 여기서 $\ll$는 '훨씬 작다' 또는 '훨씬 빠르다'는 의미로 사용됩니다.
3.  **연산 비용 vs. 메모리 접근 비용**:
    $$ \text{Cost (fetch from main memory) } \gg \text{ Cost (operation in CPU)} $$
    주 메모리에서 데이터를 가져오는 비용이 CPU에서 수행하는 연산(덧셈, 곱셈 등) 비용보다 훨씬 크다는 것을 나타냅니다.

**구체적 예시**:
*   **상수 인자**: 두 가지 정렬 알고리즘이 있다고 가정해 봅시다. `Insertion Sort`는 $O(n^2)$이지만, 작은 $n$에 대해서는 매우 효율적인 상수 인자를 가질 수 있습니다. 반면, `Merge Sort`는 $O(n \log n)$으로 점근적으로 빠르지만, 재귀 호출 및 메모리 할당 등으로 인해 더 큰 상수 인자를 가질 수 있습니다. 만약 데이터 크기($n$)가 항상 작다면 (예: $n < 50$), `Insertion Sort`가 `Merge Sort`보다 더 빠르게 동작할 수 있습니다.
*   **메모리 계층**: 배열을 순차적으로 탐색하는 알고리즘($O(n)$)과 이진 트리를 탐색하는 알고리즘($O(\log n)$)이 있습니다. 이진 트리가 점근적으로 더 빠르지만, 트리의 노드가 메모리에 흩어져 저장되어 캐시 미스가 자주 발생한다면, 순차적 배열 접근 알고리즘이 캐시 효율 덕분에 실제로는 더 빠르게 동작할 수 있습니다.
*   **연산 vs. 메모리 접근**: 특정 값을 계산해야 할 때, 계산된 값을 매번 주 메모리에 쓰고 다시 읽어오는 대신, 레지스터나 캐시에 저장하여 여러 번 재사용하는 것이 좋습니다. 만약 계산 비용이 저렴하다면, 한 번 계산한 값을 저장해두지 않고 필요할 때마다 다시 계산하는 것이 (메모리 접근 없이 CPU 내부에서만 처리할 수 있다면) 주 메모리 접근을 줄여 더 빠를 수도 있습니다. 예를 들어, 삼각함수 테이블을 미리 만들어두고 메인 메모리에서 찾아 쓰는 것보다, CPU에서 직접 계산하는 것이 더 빠를 수 있는 경우가 있습니다.

**시험 포인트**:
*   ⭐ **점근적 분석의 세 가지 주요 한계점**을 설명하고 각 한계점이 발생하는 이유와 실제 성능에 미치는 영향을 예시와 함께 서술할 수 있어야 합니다.
*   ⭐ 특히, **"상수 인자가 점근적 복잡도가 더 높은 알고리즘을 특정 구간에서 더 빠르게 만들 수 있는 경우"**에 대한 이해와 예시를 설명할 수 있어야 합니다. ($1000n$ vs $n^2$ 예시)
*   ⭐ **메모리 계층 구조(캐시, 주 메모리, HDD)**의 속도 차이가 알고리즘의 실제 실행 시간에 어떻게 영향을 미치는지, 그리고 Big-O 표기법이 이를 왜 반영하지 못하는지 설명할 수 있어야 합니다.
*   ⭐ **"계산량이 많은 프로그램이 계산량이 적은 프로그램보다 빠를 수 있는 역설적인 상황"**이 왜 발생하는지 (즉, CPU 연산 비용보다 메모리 접근 비용이 훨씬 크다는 점)를 설명하고 예시를 들 수 있어야 합니다.

## Intuition for Asymptotic Notation

### Big-O Notation ($O$)

*   **핵심 개념**: Big-O 표기법은 함수의 점근적 상한(upper bound)을 나타냅니다. 즉, 입력 크기 $n$이 충분히 커질 때, 함수의 실행 시간이나 공간 복잡도가 아무리 나빠도 특정 함수 $g(n)$을 넘어가지 않음을 의미합니다. 주로 알고리즘의 **최악의 경우(worst-case)** 성능을 분석할 때 사용됩니다. "점근적으로 $g(n)$보다 작거나 같다"는 의미를 가집니다.

*   **코드/수식 해설**:
    함수 $f(n)$이 $O(g(n))$이라는 것은 다음과 같이 정의됩니다:
    $$
    f(n) \in O(g(n)) \text{ if there exist positive constants } c \text{ and } n_0 \text{ such that } 0 \le f(n) \le c \cdot g(n) \text{ for all } n \ge n_0.
    $$
    여기서 $f(n)$은 분석 대상 알고리즘의 실제 복잡도 함수이고, $g(n)$은 우리가 비교하고자 하는 단순화된 함수(예: $n^2$, $n \log n$ 등)입니다.

*   **구체적 예시**:
    어떤 알고리즘의 실행 시간이 $f(n) = 2n^2 + 3n + 5$라고 가정해 봅시다. 이 알고리즘은 $O(n^2)$입니다.
    왜냐하면, $n$이 충분히 커질 때 ($n \ge 1$이라고 가정), $2n^2 + 3n + 5 \le 3n^2$로 만들 수 있는 상수 $c=3$과 $n_0=5$ (예를 들어, $n=5$일 때 $2(25)+3(5)+5 = 50+15+5=70$, $3(25)=75$)가 존재하기 때문입니다. 실제로는 $c=2$만으로도 가능하지만, 중요한 것은 **상한**이 존재한다는 것입니다.
    예를 들어, 배열에서 특정 값을 찾는 순차 탐색(Linear Search)의 최악의 경우 복잡도는 $O(n)$입니다. (배열의 모든 원소를 탐색해야 할 경우)

*   **시험 포인트**:
    *   ⭐ Big-O는 알고리즘의 **상한(upper bound)** 또는 **최악의 경우(worst-case)** 성능을 나타낸다는 점을 명확히 이해해야 합니다.
    *   주어진 함수에 대해 Big-O 표기법을 올바르게 적용하는 방법 (최고차항만 남기고 상수 및 낮은 차수 항 무시).
    *   $f(n) = an^k + \dots$ 형태의 다항식에서 Big-O는 $O(n^k)$가 됨을 알고 있어야 합니다.

### Big-Omega Notation ($\Omega$)

*   **핵심 개념**: Big-Omega 표기법은 함수의 점근적 하한(lower bound)을 나타냅니다. 즉, 입력 크기 $n$이 충분히 커질 때, 함수의 실행 시간이나 공간 복잡도가 아무리 좋아도 특정 함수 $g(n)$보다는 항상 크거나 같음을 의미합니다. 주로 알고리즘의 **최고의 경우(best-case)** 성능을 분석할 때 사용되지만, 어떤 문제 자체가 가질 수 있는 최소한의 복잡도를 표현하는 데 더 중요하게 활용될 수 있습니다. "점근적으로 $g(n)$보다 크거나 같다"는 의미를 가집니다.

*   **코드/수식 해설**:
    함수 $f(n)$이 $\Omega(g(n))$이라는 것은 다음과 같이 정의됩니다:
    $$
    f(n) \in \Omega(g(n)) \text{ if there exist positive constants } c \text{ and } n_0 \text{ such that } 0 \le c \cdot g(n) \le f(n) \text{ for all } n \ge n_0.
    $$

*   **구체적 예시**:
    어떤 알고리즘의 실행 시간이 $f(n) = 2n^2 + 3n + 5$라고 가정해 봅시다. 이 알고리즘은 $\Omega(n^2)$입니다.
    왜냐하면, $n$이 충분히 커질 때 ($n \ge 1$), $c \cdot n^2 \le 2n^2 + 3n + 5$로 만들 수 있는 상수 $c=1$과 $n_0=1$ (또는 더 작은 $c=2$와 $n_0=1$)이 존재하기 때문입니다.
    예를 들어, 배열에서 특정 값을 찾는 순차 탐색(Linear Search)의 최고 경우 복잡도는 $\Omega(1)$입니다. (찾는 원소가 배열의 첫 번째에 있을 경우)

*   **시험 포인트**:
    *   ⭐ Big-Omega는 알고리즘의 **하한(lower bound)** 또는 **최고의 경우(best-case)** 성능을 나타낸다는 점을 이해해야 합니다.
    *   Big-O와 Big-Omega는 상반되는 개념으로, 각각 최악과 최고의 시나리오를 나타냄을 기억하세요.

### Big-Theta Notation ($\Theta$)

*   **핵심 개념**: Big-Theta 표기법은 함수의 점근적 상한과 하한이 같다는 것을 나타내어, 함수가 특정 함수 $g(n)$과 **점근적으로 동일한 성장률**을 가짐을 의미합니다. 이는 알고리즘의 실행 시간이나 공간 복잡도가 주어진 함수 $g(n)$에 비례하여 성장하며, $g(n)$을 넘어가지도 않고 $g(n)$보다 현저히 작지도 않음을 나타냅니다. 즉, **엄격한(tight) 경계**를 제공하며, 알고리즘의 평균적인 경우(average-case) 또는 가장 정확한 복잡도를 표현할 때 사용됩니다. "점근적으로 $g(n)$과 같다"는 의미를 가집니다.

*   **코드/수식 해설**:
    함수 $f(n)$이 $\Theta(g(n))$이라는 것은 다음과 같이 정의됩니다:
    $$
    f(n) \in \Theta(g(n)) \text{ if there exist positive constants } c_1, c_2 \text{ and } n_0 \text{ such that } 0 \le c_1 \cdot g(n) \le f(n) \le c_2 \cdot g(n) \text{ for all } n \ge n_0.
    $$
    이는 $f(n) \in O(g(n))$ 이고 동시에 $f(n) \in \Omega(g(n))$ 일 때 성립합니다.

*   **구체적 예시**:
    어떤 알고리즘의 실행 시간이 $f(n) = 2n^2 + 3n + 5$라고 가정해 봅시다. 이 알고리즘은 $\Theta(n^2)$입니다.
    이는 $f(n) \in O(n^2)$ 이고 동시에 $f(n) \in \Omega(n^2)$이기 때문입니다. 즉, $n$이 충분히 커질 때, $c_1 \cdot n^2 \le 2n^2 + 3n + 5 \le c_2 \cdot n^2$를 만족하는 $c_1=1, c_2=3$ (또는 $c_1=2, c_2=3$)과 $n_0$ (예: $n_0=5$)가 존재합니다.
    예를 들어, 배열에서 특정 값을 찾는 순차 탐색(Linear Search)의 평균 복잡도는 $\Theta(n)$입니다. (평균적으로 배열의 절반을 탐색해야 할 경우)

*   **시험 포인트**:
    *   ⭐ Big-Theta는 알고리즘의 **정확한 점근적 성장률(tight bound)**을 나타낸다는 점을 가장 중요하게 이해해야 합니다.
    *   Big-O와 Big-Omega의 교집합으로 볼 수 있다는 관계를 알고 있어야 합니다.
    *   대부분의 경우 알고리즘의 복잡도를 표현할 때 가장 이상적으로 사용되는 표기법입니다.

## Slide 25

- **핵심 개념**:
`if` 문이 루프 내부에 존재하고 특정 반복에서만 실행될 경우, 해당 `if` 블록의 실행 여부가 전체 알고리즘의 수행 시간에 영향을 미칩니다. 점근적 분석에서는 일반적으로 **최악의 경우(Worst-Case)**를 기준으로 시간 복잡도를 평가합니다. 이는 `if` 문 조건이 가장 많은 연산을 유발하는 방향으로 참이 될 때의 상황을 가정하는 것입니다. 이와 더불어, **최선의 경우(Best-Case)**나 **평균적인 경우(Average-Case)**에 대한 분석도 가능합니다.

- **코드/수식 해설**:

다음과 같은 의사 코드를 예시로 들어 설명합니다.

```cpp
void exampleFunction(int arr[], int n) {
    for (int i = 0; i < n; i++) { // N번 반복되는 외부 루프
        // 항상 실행되는 기본 연산 (O(1))
        arr[i] = arr[i] * 2; 

        // 특정 조건에서만 실행되는 if 문
        if (arr[i] % 2 == 0) { // 이 조건이 항상 참일 수도, 한 번도 참이 아닐 수도, 때때로 참일 수도 있다.
            // if 블록 내부의 추가 연산 (O(1) 또는 그 이상)
            arr[i] = arr[i] + 1; 
        }
    }
}
```

1.  **점근적 분석 적용 방법**:
    *   **최악의 경우 (Worst-Case Analysis)**: `if` 문 안의 조건이 항상 참이 되어 `if` 블록 내의 코드가 루프의 모든 반복에서 실행된다고 가정합니다. 이 경우, 각 루프 반복당 `O(1)`의 기본 연산과 `O(1)`의 추가 연산이 모두 수행됩니다. 따라서 전체 루프는 $N$번 반복되므로 총 연산 횟수는 $N \times (O(1) + O(1)) = O(N)$이 됩니다.
    *   **최선의 경우 (Best-Case Analysis)**: `if` 문 안의 조건이 단 한 번도 참이 되지 않아 `if` 블록 내의 코드가 전혀 실행되지 않는다고 가정합니다. 이 경우, 각 루프 반복당 `O(1)`의 기본 연산만 수행됩니다. 전체 루프는 $N$번 반복되므로 총 연산 횟수는 $N \times O(1) = O(N)$이 됩니다.
    *   **평균적인 경우 (Average-Case Analysis)**: `if` 문 안의 조건이 참이 될 확률이나 빈도를 통계적으로 분석하여 평균적인 연산 횟수를 계산합니다. 이 분석은 일반적으로 더 복잡하며, 입력 데이터의 분포에 대한 가정이 필요합니다. 예를 들어, `if` 문이 $P$의 확률로 참이 된다면, 각 반복의 평균 비용은 $C_1 + P \times C_2$와 같이 계산될 수 있습니다.

2.  **수식 표기**:
    알고리즘의 실행 시간을 나타내는 함수를 $T(N)$이라고 할 때,
    *   **최악의 경우 시간 복잡도**: $T_{worst}(N) \in O(f(N))$
    *   **최선의 경우 시간 복잡도**: $T_{best}(N) \in \Omega(g(N))$
    *   **평균적인 경우 시간 복잡도**: $T_{avg}(N) \in \Theta(h(N))$ (만약 $T_{avg}(N)$의 상한과 하한이 동일하다면)

- **구체적 예시**:

정렬되지 않은 배열에서 특정 값(key)을 찾는 순차 검색(Sequential Search) 알고리즘을 예로 들어봅시다.

```cpp
int sequentialSearch(int arr[], int n, int key) {
    for (int i = 0; i < n; i++) { // 배열의 길이 n만큼 반복
        if (arr[i] == key) { // key를 찾았는지 확인하는 if 문
            return i; // 찾으면 바로 반환
        }
    }
    return -1; // 배열 끝까지 찾아도 없으면 -1 반환
}
```

*   **최악의 경우**: `key`가 배열의 맨 마지막에 있거나, 배열에 아예 존재하지 않는 경우입니다. 이 경우 `for` 루프는 $N$번 모두 반복되고, `if` 문 조건 검사도 $N$번 수행됩니다. 따라서 시간 복잡도는 $O(N)$입니다.
*   **최선의 경우**: `key`가 배열의 첫 번째 위치에 있는 경우입니다. 이 경우 `for` 루프는 단 한 번만 실행되고, `if` 조건이 바로 참이 되어 함수가 즉시 `return`됩니다. 따라서 시간 복잡도는 $O(1)$입니다.
*   **평균적인 경우**: `key`가 배열 내 어느 위치에 있을 확률이 동일하다고 가정할 때, 평균적으로 약 $N/2$번의 비교가 일어납니다. 따라서 시간 복잡도는 $O(N)$입니다.

대부분의 경우, 순차 검색 알고리즘의 시간 복잡도를 언급할 때는 **최악의 경우를 기준으로 $O(N)$**이라고 말합니다.

- **시험 포인트**:

*   ⭐ `if` 문이 포함된 알고리즘의 점근적 분석 시, 특별한 명시가 없다면 **최악의 경우(Worst-Case)를 가정하여 분석**하는 것이 원칙입니다. 최악의 경우는 알고리즘이 가능한 한 가장 많은 연산을 수행할 때를 의미합니다.
*   ⭐ 최악의 경우, 최선의 경우, 평균적인 경우의 정의와 각각의 분석 방법을 정확히 이해해야 합니다. 특히, `if` 문 조건이 가장 비싼 경로(즉, 가장 많은 연산을 유발하는 방향)로 참이 되는 상황을 최악의 경우로 가정합니다.
*   ⭐ 실제 코드나 의사 코드를 보고 각 연산의 실행 횟수를 계산하여 전체 시간 복잡도를 도출하는 능력이 중요합니다. `if` 문 내부의 연산이 `O(1)`이 아닐 수 있다는 점(예: `if` 문 안에 또 다른 루프나 재귀 호출이 있는 경우)을 항상 염두에 두어야 합니다.

## Slide 26

이 슬라이드는 고정 크기 배열의 한계와 동적 확장을 위한 전략 두 가지를 제시합니다. 배열의 크기가 부족할 때, 새로운 더 큰 배열을 할당하고 기존 원소들을 복사하는 과정의 효율성을 비교하는 것이 핵심입니다.

---

### **핵심 개념**

*   **고정 크기 배열의 한계**: C/C++와 같은 언어에서 배열은 메모리에 연속적으로 할당되며, 생성 시 정해진 크기를 변경할 수 없습니다. 따라서 저장해야 할 원소의 수가 배열의 현재 용량을 초과할 경우 문제가 발생합니다.
*   **배열 확장 (Resizing)**: 고정 크기 배열의 한계를 극복하기 위해, 배열이 가득 찼을 때 새로운 더 큰 배열을 할당하고, 기존 배열의 모든 원소를 새 배열로 복사한 후, 기존 배열을 해제하는 과정을 거칩니다. 이 과정은 비용이 많이 들 수 있습니다.
*   **확장 전략**: 새 배열의 크기를 결정하는 두 가지 주요 전략을 제시합니다.
    *   **Option 1: Additive Increase (상수량 증가)**: 현재 배열 크기에 고정된 상수 값(예: 10)을 더하여 새 배열의 크기를 정합니다.
    *   **Option 2: Multiplicative Increase (배율 증가)**: 현재 배열 크기에 고정된 배율(예: 2배)을 곱하여 새 배열의 크기를 정합니다.

### **코드/수식 해설**

배열 확장 시 가장 비용이 많이 드는 작업은 `Copy old array into new one`입니다. 현재 배열에 `N`개의 원소가 있다면, 이들을 모두 새로운 배열로 복사하는 데는 $O(N)$ 시간이 소요됩니다.

#### **Option 1: Additive Increase (상수량 증가)**

*   **설명**: 배열이 가득 찰 때마다 크기를 `c` (예: 10)만큼 늘립니다.
*   **시간 복잡도 분석**: `N`개의 원소를 저장하기 위해 배열이 `N/c`번 확장된다고 가정합니다. 각 확장 시 복사하는 원소의 수는 `c, 2c, 3c, ..., (N-c)`와 같이 증가합니다.
*   **총 비용**:
    $$O(c + 2c + 3c + \dots + (N-c)) = O(c \sum_{i=1}^{N/c} i) = O(c \cdot \frac{(N/c)(N/c+1)}{2}) = O(N^2)$$
    총 `N`개의 원소를 추가하는 데 드는 비용은 $O(N^2)$입니다. 각 원소당 평균(분할 상환) 비용은 $O(N)$입니다. 이는 `N`이 커질수록 매우 비효율적입니다.

#### **Option 2: Multiplicative Increase (배율 증가)**

*   **설명**: 배열이 가득 찰 때마다 크기를 `k`배 (예: 2배) 늘립니다. 초기 용량이 `C`라고 가정합니다.
*   **시간 복잡도 분석**: `N`개의 원소를 저장하기 위해 배열은 `log_k(N/C)`번 확장됩니다. 각 확장 시 복사하는 원소의 수는 `C, kC, k^2C, ..., k^{m-1}C`와 같이 증가합니다. 이때 $k^m C \ge N$을 만족하는 최소 `m`입니다.
*   **총 비용**:
    $$O(C + kC + k^2C + \dots + k^{m-1}C) = O(C \sum_{i=0}^{m-1} k^i) = O(C \frac{k^m - 1}{k-1})$$
    $k^m C \approx N$이므로, $C \frac{k^m - 1}{k-1} \approx O(N)$.
    총 `N`개의 원소를 추가하는 데 드는 비용은 $O(N)$입니다. 이는 각 원소당 **분할 상환 (Amortized) 비용**이 $O(1)$임을 의미합니다. `N`개의 원소를 추가하는 데 총 $O(N)$ 시간이 걸리므로, 평균적으로 원소 하나를 추가하는 데 상수 시간이 걸린다는 것입니다.

```cpp
// 예시: 배열 확장 로직 (C++ pseudo-code)

// Option 1: Additive Increase
void resize_additive(int*& arr, int& capacity, int current_size, int increment) {
    int* new_arr = new int[capacity + increment];
    for (int i = 0; i < current_size; ++i) {
        new_arr[i] = arr[i];
    }
    delete[] arr; // 이전 배열 메모리 해제
    arr = new_arr;
    capacity += increment;
}

// Option 2: Multiplicative Increase
void resize_multiplicative(int*& arr, int& capacity, int current_size, int factor) {
    int new_capacity = (capacity == 0) ? 1 : capacity * factor; // 초기 용량 0일 경우 예외 처리
    int* new_arr = new int[new_capacity];
    for (int i = 0; i < current_size; ++i) {
        new_arr[i] = arr[i];
    }
    delete[] arr; // 이전 배열 메모리 해제
    arr = new_arr;
    capacity = new_capacity;
}
```

### **구체적 예시**

도서관의 책꽂이를 비유해 봅시다. 책꽂이 하나에 100권의 책을 꽂을 수 있다고 가정합니다.

*   **시나리오**: 새로운 책이 계속 들어와 현재 책꽂이가 가득 찼습니다.
*   **Option 1 (상수량 증가)**: "책꽂이가 가득 찼으니, 20권 더 꽂을 수 있는 책꽂이를 하나 더 사자!"
    *   현재 100권이 꽂혀있는데, 120권짜리 새 책꽂이를 삽니다. 기존 100권을 새 책꽂이로 옮깁니다. (20권의 여유)
    *   다시 가득 차면 (120권), 140권짜리 새 책꽂이를 사서 120권을 옮깁니다.
    *   이 방식은 매번 옮기는 비용은 크지만, 추가되는 여유 공간은 항상 일정하여 금방 다시 가득 차게 됩니다. 장기적으로 많은 책이 들어올수록 계속해서 책을 옮기는 작업이 반복되어 비효율적입니다.

*   **Option 2 (배율 증가)**: "책꽂이가 가득 찼으니, 기존 크기의 2배짜리 책꽂이를 하나 더 사자!"
    *   현재 100권이 꽂혀있는데, 200권짜리 새 책꽂이를 삽니다. 기존 100권을 새 책꽂이로 옮깁니다. (100권의 여유)
    *   다시 가득 차면 (200권), 400권짜리 새 책꽂이를 사서 200권을 옮깁니다.
    *   이 방식은 책을 옮기는 횟수 자체는 줄어들고, 한 번 옮길 때마다 훨씬 많은 여유 공간이 생겨 다음 확장까지의 간격이 길어집니다. 따라서 장기적으로 봤을 때 훨씬 효율적입니다.

### **시험 포인트**

*   ⭐ **배열 확장의 필요성**: 동적 데이터 관리를 위해 고정 크기 배열을 확장해야 하는 이유를 설명할 수 있어야 합니다. (예: 동적 배열, `std::vector`의 내부 동작)
*   ⭐ **두 가지 확장 전략의 비교**:
    *   **상수량 증가** (Option 1) 방식의 **총 시간 복잡도는 $O(N^2)$**, **분할 상환 시간 복잡도는 $O(N)$**임을 이해해야 합니다.
    *   **배율 증가** (Option 2) 방식의 **총 시간 복잡도는 $O(N)$**, **분할 상환 시간 복잡도는 $O(1)$**임을 이해해야 합니다.
*   ⭐ **분할 상환 분석 (Amortized Analysis)**의 개념: 특정 연산의 최악의 경우 비용은 높지만, 일련의 연산들에 대한 평균 비용은 낮을 때 사용되는 분석 기법입니다. 배열의 배율 증가 방식에서 원소 추가의 분할 상환 비용이 $O(1)$이라는 것은 매우 중요합니다.
*   ⭐ 실제 대부분의 동적 배열 구현 (`std::vector` 등)에서 **배율 증가 (보통 1.5배 또는 2배)** 방식을 채택하는 이유를 설명할 수 있어야 합니다.

## Stretchy Array (version 1)

---

### **핵심 개념**
`Stretchy Array`는 고정된 크기를 가지는 일반 배열의 한계를 극복하기 위해 동적으로 크기를 조절할 수 있도록 설계된 자료구조입니다. 내부적으로는 일반 배열을 사용하지만, 배열이 가득 찼을 때 자동으로 더 큰 새 배열을 할당하고 기존 요소를 복사하여 확장하는 기능을 제공합니다. 이는 실제 저장된 요소의 개수(`count`)와 배열의 최대 용량(`maxSize`)을 관리하여 이루어집니다.

### **코드/수식 해설**

**`StretchyArray` 구조:**
*   `maxSize`: 현재 배열이 수용할 수 있는 최대 요소의 개수 (양의 정수, 1부터 시작).
*   `array`: 실제 데이터를 저장하는 내부 배열. 크기는 `maxSize`와 동일.
*   `count`: 현재 `array`에 저장된 실제 요소의 개수.

**`put(x)` 메서드:**
`put(x)`는 배열의 끝에 요소 `x`를 추가하는 연산입니다.

```
put(x): add x to the end of the array
    if maxSize == count
        make new array of size (maxSize + 5)
        copy old array contents to new array
        maxSize = maxSize + 5
    array[count] = x
    count = count + 1
```

1.  **확장 조건 확인**: `if maxSize == count`
    *   현재 배열에 저장된 요소의 개수(`count`)가 배열의 최대 용량(`maxSize`)과 같으면 배열이 가득 찼다는 의미입니다.
2.  **배열 확장**:
    *   `make new array of size (maxSize + 5)`: 기존 `maxSize`보다 5만큼 더 큰 새로운 배열을 생성합니다.
    *   `copy old array contents to new array`: 기존 배열에 있던 모든 요소를 새로 생성된 배열로 복사합니다. 이 과정은 현재 `count`개의 요소를 복사하므로 $O(\text{count})$ 또는 $O(\text{maxSize})$의 시간 복잡도를 가집니다.
    *   `maxSize = maxSize + 5`: `StretchyArray`의 `maxSize` 값을 새로 확장된 크기로 업데이트합니다.
3.  **요소 추가**: `array[count] = x`
    *   배열이 가득 차지 않았거나, 확장된 후에 새로운 요소를 `count` 인덱스(현재 마지막 요소 다음 위치)에 추가합니다.
4.  **카운트 증가**: `count = count + 1`
    *   새로운 요소가 추가되었으므로 `count`를 1 증가시킵니다.

### **구체적 예시**

초기 `StretchyArray`가 `maxSize = 2`, `count = 0`, `array = []` (크기 2) 상태라고 가정해봅시다.

1.  **`put(10)` 호출**:
    *   `maxSize` (2)와 `count` (0)가 같지 않습니다.
    *   `array[0] = 10`
    *   `count = 1`
    *   상태: `maxSize = 2`, `count = 1`, `array = [10, _]`

2.  **`put(20)` 호출**:
    *   `maxSize` (2)와 `count` (1)가 같지 않습니다.
    *   `array[1] = 20`
    *   `count = 2`
    *   상태: `maxSize = 2`, `count = 2`, `array = [10, 20]`

3.  **`put(30)` 호출**:
    *   `maxSize` (2)와 `count` (2)가 같습니다! **배열 확장 필요.**
    *   새 배열 생성: `newArray` (크기 `maxSize + 5 = 2 + 5 = 7`)
    *   기존 요소 복사: `[10, 20]`을 `newArray`로 복사. `newArray = [10, 20, _, _, _, _, _]`
    *   `array`가 `newArray`를 가리키도록 변경.
    *   `maxSize = 7`
    *   `array[2] = 30`
    *   `count = 3`
    *   상태: `maxSize = 7`, `count = 3`, `array = [10, 20, 30, _, _, _, _]`

이후에는 `maxSize`가 7이므로, `count`가 7이 될 때까지 추가적인 확장 없이 요소를 계속 추가할 수 있습니다.

### **시험 포인트**

*   ⭐ **동적 배열의 필요성**: 고정 크기 배열의 한계를 설명하고, `Stretchy Array`와 같은 동적 배열이 필요한 이유를 설명할 수 있어야 합니다.
*   ⭐ **`put(x)` 연산의 동작 방식**: 특히 배열이 가득 찼을 때 확장 로직(새 배열 생성, 요소 복사, `maxSize` 업데이트)을 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **시간 복잡도**:
    *   일반적인 `put(x)` 연산은 `array[count] = x`와 `count++`이므로 $O(1)$의 시간 복잡도를 가집니다.
    *   그러나 배열 확장(resizing)이 발생하는 경우, `count`개의 요소를 새 배열로 복사해야 하므로 $O(\text{count})$ 또는 $O(\text{maxSize})$의 시간 복잡도가 발생합니다.
    *   `+5` 방식의 확장은 매번 일정 크기만 늘리므로, 많은 요소를 추가할 때 확장 비용이 자주 발생하게 됩니다. (이후 수업에서 다룰 **상수배 확장**(`*2`) 방식과 비교하여 Amortized Analysis 관점에서 어떤 차이가 있는지 이해하는 것이 중요합니다.)
*   **멤버 변수의 역할**: `maxSize`, `array`, `count` 각각의 역할과 이들이 어떻게 동적 배열의 상태를 관리하는지 이해해야 합니다.

## Slide 28

### Stretchy Array (Version 2)

---

-   **핵심 개념**:
    'Stretchy Array'는 일반적인 고정 크기 배열의 한계를 극복하기 위해 동적으로 크기를 조절할 수 있도록 구현된 자료구조입니다. 내부적으로 고정 크기 배열을 사용하지만, 요소 추가 시 공간이 부족해지면 자동으로 더 큰 배열을 새로 할당하고 기존 요소를 복사하여 마치 '늘어나는' 배열처럼 동작하게 합니다. 이는 `java.util.ArrayList`나 `std::vector`와 같은 동적 배열의 기본적인 동작 원리입니다.

    *   `maxSize`: 현재 배열에 할당된 총 공간(물리적 크기)을 나타냅니다. 0부터 시작하는 양의 정수입니다.
    *   `array`: 실제 데이터를 저장하는 내부 배열입니다. 크기는 `maxSize`입니다.
    *   `count`: 현재 배열에 실제로 저장된 요소의 개수(논리적 크기)를 나타냅니다.

-   **코드/수식 해설**:
    `put(x)` 메서드는 배열의 끝에 새로운 요소 `x`를 추가하는 역할을 합니다. 이때, 배열의 공간이 부족한 경우(꽉 찬 경우) 자동으로 크기를 확장하는 로직이 포함되어 있습니다.

    ```
    put(x): add x to the end of the array
        if maxSize == count
            make new array of size (maxSize * 2) // 현재 크기의 두 배로 새 배열 할당
            copy old array contents to new array // 기존 요소들을 새 배열로 복사
            maxSize = maxSize * 2               // maxSize 업데이트
        array[count] = x                       // 새 요소 추가
        count = count + 1                      // 요소 개수 증가
    ```

    해당 코드의 핵심은 `if maxSize == count` 조건입니다.
    *   `maxSize == count`: 현재 할당된 배열의 공간(`maxSize`)이 실제로 채워진 요소의 개수(`count`)와 같다는 의미로, 배열이 가득 찼음을 나타냅니다.
    *   이 경우, `maxSize * 2` 크기의 새로운 배열을 생성하고 기존 배열의 모든 요소를 새로운 배열로 복사합니다. 이 복사 작업은 $O(\text{count})$ 또는 $O(\text{maxSize})$의 시간 복잡도를 가집니다.
    *   복사가 완료되면 `maxSize`를 새 배열의 크기로 업데이트하고, 그 후 `array[count] = x`를 통해 새 요소를 추가하고 `count`를 1 증가시킵니다.
    *   새로운 배열의 할당 및 요소 복사 작업은 비용이 많이 들지만, 크기를 두 배로 늘림으로써 전체적으로는 $N$개의 요소를 추가하는 데 드는 평균 시간(상환 분석)은 $O(1)$이 됩니다.

-   **구체적 예시**:
    초기 `maxSize = 0`, `count = 0`, `array = []` 라고 가정합니다.

    1.  `put(10)` 호출:
        *   `maxSize == count` (0 == 0) 이므로 조건 만족.
        *   새 배열 크기: `0 * 2 = 0`. (최초 `maxSize`가 0일 경우 초기 크기를 정하는 규칙이 필요하며, 일반적으로는 최소 크기(예: 1)로 시작합니다. 슬라이드의 `(starts at 0)` 문구는 초기 `count`를 의미할 수도 있으므로, `maxSize`는 최초 put 시 특정 값(예: 1)으로 초기화된다고 가정하는 것이 더 합리적입니다.)
        *   **재해석**: 일반적으로 Stretchy Array는 최초 `maxSize`가 0이 아니라 최소 크기(예: 1 또는 4)로 시작하고 `count`만 0으로 시작합니다. 슬라이드에서는 `maxSize: positive integer (starts at 0)`라고 되어 있으나, 이는 `count`의 초기값을 설명하는 부분과 겹쳐서 다소 혼동이 있을 수 있습니다. 여기서는 `maxSize`가 최초 `put` 호출 시 1 또는 그 이상으로 초기화된다고 가정하고 진행하겠습니다.
        *   **다시 예시**: `maxSize = 2`, `count = 0`, `array = [_, _]` (현재 비어있음)
        *   `put(10)`:
            *   `maxSize == count` (2 == 0) -> 거짓.
            *   `array[0] = 10`. `array = [10, _]`
            *   `count = 1`.
        *   `put(20)`:
            *   `maxSize == count` (2 == 1) -> 거짓.
            *   `array[1] = 20`. `array = [10, 20]`
            *   `count = 2`.
        *   `put(30)`:
            *   `maxSize == count` (2 == 2) -> 참! (배열이 꽉 참)
            *   새 배열 생성: `newArray` of size `maxSize * 2` (즉, 4).
            *   기존 요소 복사: `newArray = [10, 20, _, _]`.
            *   `array`는 이제 `newArray`를 가리킴. `maxSize = 4`.
            *   `array[2] = 30`. `array = [10, 20, 30, _]`
            *   `count = 3`.
        *   이후 `put` 호출 시 `maxSize`가 4이므로, `count`가 4가 될 때까지는 리사이징 없이 요소를 추가할 수 있습니다. `count`가 4가 되면 `maxSize * 2 = 8`로 다시 확장됩니다.

-   **시험 포인트**:
    *   ⭐ **동적 배열의 필요성**: 고정 크기 배열의 한계를 설명하고 Stretchy Array와 같은 동적 배열이 필요한 이유를 설명할 수 있어야 합니다. (메모리 낭비 vs. 유연성)
    *   ⭐ **리사이징 전략**: 배열이 꽉 찼을 때 `maxSize * 2`로 크기를 늘리는 전략의 의미와 다른 전략(예: `maxSize + C`)과 비교하여 장단점을 설명할 수 있어야 합니다. (특히 상환 분석 관점에서 $O(1)$의 `put` 연산 비용 달성)
    *   ⭐ **시간 복잡도**:
        *   일반적인 `put(x)` 연산의 시간 복잡도는 $O(1)$입니다.
        *   배열 확장(resizing)이 발생하는 경우, `make new array`와 `copy old array contents` 때문에 $O(\text{maxSize})$ 또는 $O(\text{count})$의 시간이 소요됩니다.
        *   `N`개의 요소를 추가하는 총 시간 복잡도를 상환 분석(Amortized Analysis) 관점에서 설명하고, 개별 연산의 비용이 아닌 전체 연산의 평균 비용이 $O(1)$임을 증명할 수 있어야 합니다. (총 $N$번의 `put` 연산의 누적 시간 복잡도는 $O(N)$임을 이해해야 합니다.)
    *   ⭐ **공간 복잡도**: Stretchy Array의 공간 복잡도는 현재 요소의 개수 `count`에 비례하여 $O(count)$입니다. 다만, 실제 할당된 `maxSize`는 `count`보다 크거나 같으므로, $O(maxSize)$가 됩니다. 확장 전략에 따라 $maxSize$는 $2^k$ 형태가 될 수 있습니다.
    *   구현 시 `maxSize`가 0으로 초기화되는 경우 첫 `put` 연산에서 어떤 예외 처리가 필요할지 고려해야 합니다 (예: 최소 크기 1 또는 특정 값으로 초기화).

## Slide 29

### **핵심 개념**:
`put(x)`는 동적 배열(stretchy array 또는 dynamic array)에 새로운 원소 `x`를 추가하는 연산입니다. 동적 배열은 저장할 데이터가 많아지면 자동으로 내부 배열의 크기를 확장하는 특징을 가집니다. 이 슬라이드는 `put(x)` 연산의 성능 비용, 특히 내부 배열이 가득 찼을 때 발생하는 재할당(resizing) 과정과 그에 따른 시간 복잡도를 분석합니다.

### **코드/수식 해설**:
슬라이드에 제시된 `put(x)` 연산의 의사 코드(pseudocode)는 다음과 같습니다:

```
if maxSize == count
    make new array of bigger size
    copy old array contents to new array
    update maxSize
array[count] = x
count = count + 1
```

1.  **배열 용량 확인**: `if maxSize == count`
    *   현재 배열에 저장된 원소의 개수(`count`)가 배열의 최대 용량(`maxSize`)과 같은지 확인합니다. 이 과정은 상수 시간 $O(1)$이 소요됩니다.
2.  **배열 확장 (재할당)**: `if` 조건이 참일 때(배열이 가득 찼을 때) 실행됩니다.
    *   `make new array of bigger size`: 현재 크기보다 큰(예: 2배) 새로운 배열을 생성합니다.
    *   `copy old array contents to new array`: 기존 배열에 있던 모든 원소를 새 배열로 복사합니다. 현재 배열에 $N$개의 원소가 있다면, 이 복사 과정은 $O(N)$ 시간이 소요됩니다. 이것이 `put(x)` 연산에서 가장 비용이 많이 드는 부분입니다.
    *   `update maxSize`: `maxSize` 변수를 새 배열의 크기로 업데이트합니다.
3.  **원소 추가 및 카운트 증가**:
    *   `array[count] = x`: 새 원소 `x`를 배열의 다음 빈 위치(`count` 인덱스)에 저장합니다. $O(1)$ 시간이 소요됩니다.
    *   `count = count + 1`: 저장된 원소의 개수를 1 증가시킵니다. $O(1)$ 시간이 소요됩니다.

`put(x)` 연산의 시간 복잡도는 다음과 같습니다:
*   **최적/평균 케이스 (Best/Average Case)**: 배열에 여유 공간이 있어 재할당이 필요 없을 때, 원소 추가 및 `count` 증가만 발생합니다. 이 경우 시간 복잡도는 $O(1)$입니다.
*   **최악 케이스 (Worst Case)**: 배열이 가득 차서 재할당이 필요한 경우, 기존 $N$개의 원소를 새 배열로 복사하는 과정 때문에 $O(N)$ 시간이 소요됩니다. 여기서 $N$은 재할당 시점의 `count` 값입니다.

따라서, `put(x)` 연산의 최악 시간 복잡도 $O(f(n))$에서 $f(n)$은 현재 배열의 원소 수 $n$에 해당합니다.
$$ \text{Worst-case time complexity of put(x)} = O(n) $$
하지만 동적 배열의 `put(x)`는 대개 상환 분석(amortized analysis)을 통해 평균적으로 $O(1)$의 시간 복잡도를 가진다고 평가됩니다. 즉, 드물게 발생하는 $O(N)$ 재할당 비용이 여러 $O(1)$ 연산에 분산되어 전체 연산 시퀀스에 대한 평균 비용은 낮아집니다.

### **구체적 예시**:
초기 용량이 2인 동적 배열에 `put` 연산을 수행하며 용량이 어떻게 변하는지 살펴봅시다.

1.  **초기 상태**: `array = [_, _]`, `maxSize = 2`, `count = 0`
2.  `put(10)`:
    *   `maxSize (2) == count (0)`? `False`.
    *   `array[0] = 10` $\rightarrow$ `array = [10, _]`
    *   `count = 1`
    *   **비용**: $O(1)$
3.  `put(20)`:
    *   `maxSize (2) == count (1)`? `False`.
    *   `array[1] = 20` $\rightarrow$ `array = [10, 20]`
    *   `count = 2`
    *   **비용**: $O(1)$
4.  `put(30)`:
    *   `maxSize (2) == count (2)`? `True`. (배열이 가득 참)
    *   **재할당 발생**:
        *   새 배열(`newArray`)을 용량 4로 생성 (예: 기존 크기의 2배).
        *   `[10, 20]`를 `newArray`로 복사.
        *   `maxSize = 4`로 업데이트. `array`는 `newArray`를 가리키게 됨.
    *   `array[2] = 30` $\rightarrow$ `array = [10, 20, 30, _]`
    *   `count = 3`
    *   **비용**: $O(2)$ (원소 2개 복사)
5.  `put(40)`:
    *   `maxSize (4) == count (3)`? `False`.
    *   `array[3] = 40` $\rightarrow$ `array = [10, 20, 30, 40]`
    *   `count = 4`
    *   **비용**: $O(1)$
6.  `put(50)`:
    *   `maxSize (4) == count (4)`? `True`. (배열이 가득 참)
    *   **재할당 발생**: 새 배열 용량 8로 생성, 기존 4개 원소 복사.
    *   **비용**: $O(4)$ (원소 4개 복사)

### **시험 포인트**:
*   ⭐ **`put(x)` 연산의 최악 시간 복잡도**: 동적 배열에서 `put(x)` 연산 시, 배열이 가득 차서 재할당이 필요한 경우의 시간 복잡도는 $O(N)$입니다. 여기서 $N$은 재할당 시점의 배열 원소 개수입니다.
*   ⭐ **재할당(Resizing) 과정의 이해**: `put(x)` 연산 중 배열 크기가 부족할 때, `더 큰 새 배열 생성`, `기존 원소 복사`, `새 배열 참조`의 3단계가 핵심입니다. 이 중 `기존 원소 복사`가 가장 비용이 큰 작업입니다.
*   ⭐ **상환 분석 (Amortized Analysis) 개념**: 비록 단일 `put(x)` 연산이 최악의 경우 $O(N)$이 될 수 있지만, 동적 배열의 `put(x)` 연산은 대개 상환 시간 복잡도(amortized time complexity) 관점에서 $O(1)$의 성능을 가집니다. 이는 여러 연산에 걸쳐 평균적으로 계산된 비용이며, 시험에서 이 개념을 물을 수 있습니다.

데이터구조 (CSED233) 강의 자료 분석 노트

---

### **핵심 개념**
이 슬라이드는 **Stretchy Array (동적 배열)**에 새로운 원소 `x`를 추가하는 `put(x)` 연산의 **시간 복잡도**를 분석합니다. 동적 배열은 고정 크기 배열의 한계를 극복하기 위해 필요에 따라 자동으로 크기를 조절하는 자료구조입니다. `put(x)` 연산은 대부분 상수 시간 $O(1)$에 수행되지만, 배열이 가득 차서 **재할당(resizing)**이 필요한 경우 시간이 더 오래 걸릴 수 있습니다. 슬라이드는 특히 이 **최악의 경우(worst-case)** 시나리오에 초점을 맞춥니다.

### **코드/수식 해설**
`put(x)` 연산의 각 단계와 해당 시간 복잡도는 다음과 같습니다.

```
if maxSize == count
    make new array of bigger size
    copy old array contents to new array
    update maxSize
array[count] = x
count = count + 1
```

1.  `if maxSize == count`
    *   현재 배열의 최대 크기(`maxSize`)와 저장된 원소의 개수(`count`)를 비교하여 배열이 가득 찼는지 확인합니다.
    *   **시간 복잡도**: $O(1)$ (상수 시간)

2.  **배열이 가득 찼을 때 (재할당 과정)**
    *   `make new array of bigger size`
        *   현재 배열보다 더 큰 크기의 새로운 배열을 할당합니다.
        *   **시간 복잡도**: $O(1)$ (상수 시간. 메모리 할당 자체는 일반적으로 상수 시간으로 간주됩니다.)
    *   `copy old array contents to new array`
        *   기존 배열에 있던 모든 원소들을 새로 할당된 더 큰 배열로 복사합니다.
        *   **시간 복잡도**: $O(n)$ (선형 시간). 현재 배열에 $n$개의 원소가 있다면, 모든 $n$개의 원소를 복사해야 하므로 $n$에 비례하는 시간이 소요됩니다.
    *   `update maxSize`
        *   `maxSize` 변수를 새 배열의 크기로 업데이트합니다.
        *   **시간 복잡도**: $O(1)$ (상수 시간)

3.  **새 원소 추가 및 카운트 업데이트**
    *   `array[count] = x`
        *   새로운 원소 `x`를 배열의 `count` 인덱스(다음 빈 위치)에 저장합니다.
        *   **시간 복잡도**: $O(1)$ (상수 시간. 배열 인덱스 접근은 상수 시간입니다.)
    *   `count = count + 1`
        *   저장된 원소의 개수를 1 증가시킵니다.
        *   **시간 복잡도**: $O(1)$ (상수 시간)

**전체 시간 복잡도 분석:**
*   **최선의 경우 (Best-case)**: 배열이 가득 차지 않아 재할당이 필요 없는 경우.
    $O(1)$ (조건 확인) + $O(1)$ (원소 추가) + $O(1)$ (카운트 증가) = $O(1)$
*   **최악의 경우 (Worst-case)**: 배열이 가득 차 재할당이 필요한 경우.
    $O(1)$ (조건 확인) + $O(1)$ (새 배열 할당) + $O(n)$ (원소 복사) + $O(1)$ (maxSize 업데이트) + $O(1)$ (원소 추가) + $O(1)$ (카운트 증가) = $O(n)$

슬라이드에서 강조하듯이, `put(x)` 연산의 **최악의 경우 시간 복잡도는 $O(n)$**입니다. 여기서 $n$은 현재 배열에 저장된 원소의 개수입니다. 이는 `copy old array contents to new array` 단계에서 모든 원소를 복사해야 하기 때문입니다.

### **구체적 예시**
동적 배열에 원소를 추가하는 과정을 실생활 비유로 들어보겠습니다.
당신이 책꽂이(배열)에 책(데이터)을 꽂는다고 상상해 보세요.

*   **배열이 가득 차지 않은 경우**: 빈 칸이 있으면 책을 바로 꽂을 수 있습니다. 이것은 즉시(상수 시간, $O(1)$) 완료됩니다.
*   **배열이 가득 찬 경우 (재할당)**:
    1.  책꽂이에 더 이상 빈 칸이 없습니다 (`maxSize == count`).
    2.  새 책을 꽂기 위해 더 큰 새 책꽂이(새로운 배열)를 구매합니다 (`make new array of bigger size`, $O(1)$).
    3.  기존 책꽂이에 있던 모든 책을 새 책꽂이로 옮겨야 합니다 (`copy old array contents to new array`). 책이 많을수록 옮기는 데 시간이 오래 걸립니다. 예를 들어 책이 100권이라면 100권 모두 옮겨야 하므로 $O(n)$ 시간이 소요됩니다.
    4.  이제 새 책꽂이의 크기 정보를 업데이트합니다 (`update maxSize`, $O(1)$).
    5.  새 책을 새 책꽂이의 빈 칸에 꽂고 (`array[count] = x`, $O(1)$).
    6.  꽂은 책의 총 개수를 업데이트합니다 (`count = count + 1`, $O(1)$).

이 경우 새 책을 꽂는 `put(x)` 연산은 사실상 이사하는 과정(`O(n)`)이 포함되므로, 전체적으로 $O(n)$의 시간이 소요됩니다.

### **시험 포인트**
*   ⭐ **`put(x)` 연산의 최악의 경우 시간 복잡도와 그 이유**: `put(x)` 연산이 최악의 경우 $O(n)$의 시간 복잡도를 가지는 이유를 "재할당 시 기존 원소 복사" 과정과 연결하여 명확하게 설명할 수 있어야 합니다. (특히 `copy old array contents to new array` 단계의 $O(n)$ 복잡도)
*   ⭐ **재할당(Resizing)의 조건**: 동적 배열에서 언제 재할당이 발생하는지 (`maxSize == count`) 이해하고 있어야 합니다.
*   ⭐ **각 연산 단계별 시간 복잡도**: `put(x)`를 구성하는 각 의사 코드 라인의 시간 복잡도를 Big-O 표기법으로 분석할 수 있어야 합니다.
*   ⭐ **Big-O 표기법의 의미**: $O(1)$과 $O(n)$이 각각 무엇을 의미하는지 정확히 알고 있어야 합니다.

## 데이터구조 (CSED233) 슬라이드 노트 - 상환 분석 (Amortized Analysis)

---

### 핵심 개념

*   **동적 배열의 확장 문제**: 동적 배열(dynamic array)에 새로운 요소를 `put(x)` 연산을 통해 추가할 때, 배열이 가득 차 있으면 더 큰 새로운 배열을 할당하고 기존 요소를 새 배열로 복사하는 과정이 필요합니다. 이 배열 확장(resizing) 연산은 배열의 크기 `k`에 비례하는 $O(k)$의 비용이 발생하며, 이는 단일 `put(x)` 연산의 최악의 경우 시간 복잡도를 $O(N)$ (N은 현재 배열의 크기)로 만듭니다.
*   **평균 성능 (Average Performance)**: 여러 번의 연산에 걸친 전체 비용을 연산 횟수로 나누어 얻는 평균적인 성능을 의미합니다. 단일 연산의 최악의 경우가 드물게 발생하더라도, 전체적인 관점에서 성능이 어떠한지 파악하는 데 유용합니다.
*   **상환 분석 (Amortized Analysis)**: 일련의 연산들에 대한 총 비용을 분석하여, 각 연산의 평균 비용을 계산하는 기법입니다. 특정 연산이 간헐적으로 높은 비용을 발생시키더라도, 다른 많은 연산들이 낮은 비용으로 수행되어 전체 연산 시퀀스의 평균 비용은 낮아질 수 있음을 보여줍니다. 즉, 드물게 발생하는 고비용 연산이 빈번하게 발생하는 저비용 연산에 "상환"된다는 개념입니다.

### 코드/수식 해설

슬라이드에 제시된 수식은 `n`번의 `put` 연산에 대한 평균 비용을 계산하는 방법을 나타냅니다.

$$
\frac{\sum_{i=1}^{n} \text{cost of calling put for the ith time}}{n} = O(?)
$$

*   **$\sum_{i=1}^{n} \text{cost of calling put for the ith time}$**: `i`번째 `put` 연산에 소요된 비용을 `n`번의 연산에 대해 모두 합산한 총 비용을 의미합니다.
*   **$n$**: 총 `put` 연산의 횟수입니다.
*   **$O(?)$**: 이 총 비용을 `n`으로 나누어 얻은 평균 비용의 점근적 상한(Big-O notation)을 묻고 있습니다. 상환 분석을 통해 동적 배열의 `put` 연산의 경우 이 평균 비용이 $O(1)$임을 보일 수 있습니다.

### 구체적 예시

**동적 배열(Dynamic Array)의 `put` (삽입) 연산**

C++의 `std::vector`나 Java의 `ArrayList`와 같은 동적 배열은 내부적으로 고정 크기 배열을 사용하며, 공간이 부족할 때마다 배열의 크기를 확장합니다. 일반적으로 현재 크기의 두 배로 확장하는 전략을 사용합니다.

1.  **초기 상태**: 빈 배열, 크기 = 1 (편의상)
2.  **첫 번째 `put` (x1)**:
    *   배열: `[x1]` (크기 1 사용)
    *   비용: $O(1)$ (단순 삽입)
3.  **두 번째 `put` (x2)**:
    *   배열이 가득 찼으므로, 크기를 2배로 확장 (크기 1 -> 크기 2). 기존 요소 `x1`을 새 배열로 복사.
    *   배열: `[x1, x2]`
    *   비용: $O(1)$ (확장 및 복사 `x1`) + $O(1)$ (삽입 `x2`) = $O(2)$
4.  **세 번째 `put` (x3)**:
    *   배열: `[x1, x2, x3]` (크기 2 사용, 크기 4로 확장할 준비)
    *   비용: $O(1)$ (단순 삽입)
5.  **네 번째 `put` (x4)**:
    *   배열이 가득 찼으므로, 크기를 2배로 확장 (크기 2 -> 크기 4). 기존 요소 `x1, x2, x3`을 새 배열로 복사.
    *   배열: `[x1, x2, x3, x4]`
    *   비용: $O(3)$ (확장 및 복사 `x1, x2, x3`) + $O(1)$ (삽입 `x4`) = $O(4)$
    ...
6.  **$N$개의 요소를 삽입할 때의 총 비용**:
    *   $N$번의 단순 삽입 비용: $N \times O(1) = O(N)$
    *   배열 확장(복사) 비용:
        *   크기 1에서 2로 확장 시 1개 복사
        *   크기 2에서 4로 확장 시 2개 복사
        *   크기 4에서 8로 확장 시 4개 복사
        *   ...
        *   크기 $N/2$에서 $N$으로 확장 시 $N/2$개 복사 (대략)
        *   총 복사 비용은 $1+2+4+...+N/2 = N-1$ (등비수열 합) 이므로 $O(N)$
    *   **총 비용** = $O(N)$ (단순 삽입) + $O(N)$ (확장/복사) = $O(N)$
7.  **평균 상환 비용**: 총 비용 $O(N)$을 $N$번의 연산으로 나누면 $O(N)/N = O(1)$이 됩니다.

따라서 동적 배열의 `put` 연산은 최악의 경우 $O(N)$이지만, 일련의 $N$개 연산에 대한 **상환 시간 복잡도는 $O(1)$** 입니다.

### 시험 포인트

*   ⭐ **상환 분석의 정의와 중요성**: 단일 연산의 최악의 경우 성능이 나쁘더라도, 여러 연산에 걸친 평균적인 성능을 효율적으로 평가하는 방법임을 이해해야 합니다.
*   ⭐ **동적 배열 `put` 연산의 상환 시간 복잡도**: 동적 배열의 `put` 연산은 최악의 경우 $O(N)$이지만, 상환 분석을 통해 $O(1)$임을 증명하는 과정을 설명할 수 있어야 합니다. 특히, 배열 크기 확장 전략(예: 2배 확장)이 왜 $O(1)$ 상환 비용을 가능하게 하는지 이해하는 것이 중요합니다.
*   ⭐ **최악의 경우 분석(Worst-Case Analysis)과 상환 분석(Amortized Analysis)의 차이점**:
    *   최악의 경우 분석은 단일 연산이 가질 수 있는 가장 높은 비용을 고려합니다.
    *   상환 분석은 일련의 연산들에 대한 총 비용을 평균화하여, 전체적인 효율성을 평가합니다. 이 두 분석 기법의 목적과 적용 상황을 명확히 구분할 수 있어야 합니다.

## Slide 32

## Amortized Analysis of StretchyArray Version 1

### 핵심 개념
- **StretchyArray (동적 배열)**: C++의 `std::vector`나 Java의 `ArrayList`와 같이, 필요에 따라 자동으로 크기를 조절할 수 있는 배열입니다. 고정된 크기의 배열과 달리, 요소를 추가할 공간이 부족할 경우 내부적으로 더 큰 배열을 할당하고 기존 요소들을 복사하는 방식으로 크기를 늘립니다.
- **분할 상환 분석 (Amortized Analysis)**: 특정 자료구조 연산의 비용을 분석하는 기법 중 하나입니다. 단일 연산의 최악의 경우 비용이 매우 높을 수 있지만, 일련의 연산 시퀀스 전체를 놓고 보았을 때 평균적인 비용이 낮게 유지되는 경우에 그 효율성을 평가하는 데 사용됩니다. 이 슬라이드는 `StretchyArray`의 한 버전(`Version 1`)이 요소를 추가할 때 발생하는 비용을 기록하여 분할 상환 분석의 기초 자료를 제공합니다.

### 코드/수식 해설
제시된 슬라이드는 `StretchyArray`에 요소를 `add`하는 연산에 대한 단계별 비용을 보여주는 표입니다.

**표의 각 열 의미**:
- `i`: `add` 연산의 순서입니다.
- `maxSize`: 현재 `StretchyArray`가 최대로 저장할 수 있는 요소의 개수(배열의 실제 할당 크기)입니다.
- `count`: 현재 `StretchyArray`에 저장된 실제 요소의 개수입니다.
- `cost`: `i`번째 `add` 연산에 소요된 비용입니다.
- `comments`: 해당 연산의 비용에 대한 추가 설명입니다.

**비용 계산 로직 분석**:
1.  **일반적인 삽입**: `count < maxSize`인 경우, 즉 배열에 여유 공간이 있는 경우 새 요소를 추가하는 비용은 상수 시간 $O(1)$입니다. (표에서 `cost=1`인 행들).
2.  **배열 확장 및 삽입**: `count == maxSize`인 경우, 배열이 가득 찼으므로 확장이 필요합니다.
    *   **새로운 배열 할당**: 현재 `maxSize`보다 큰 새로운 배열을 할당합니다.
    *   **기존 요소 복사**: 기존 `maxSize`개의 요소들을 새 배열로 복사합니다. 이 비용은 `maxSize`에 비례합니다 ($O(\text{maxSize})$).
    *   **새 요소 삽입**: 새 배열에 현재 요소를 삽입합니다 (상수 시간 $O(1)$).
    *   따라서 확장 시 총 비용은 `기존 maxSize + 1`이 됩니다. (표에서 `cost=0+1`, `5+1`, `10+1`인 행들).

**`StretchyArray Version 1`의 확장 전략**:
이 표에서 `maxSize`의 변화를 보면, `5 -> 10 -> 15`로 **일정한 크기(`5`)만큼 증가**하는 전략을 사용하고 있음을 알 수 있습니다.

$N$개의 요소를 삽입하는 총 비용 $T(N)$을 분할 상환 분석을 통해 계산하면:
총 삽입 횟수 $N$과 배열 확장 시 복사 비용의 합으로 구성됩니다.
배열은 `count = 0, 5, 10, 15, ...`일 때 확장됩니다.
각 확장 시 복사 비용은 `0, 5, 10, ...`입니다.
대략 $N/5$번의 확장이 발생합니다.

$T(N) = \sum_{j=1}^{N} \text{cost}_j \approx N \times (\text{기본 삽입 비용}) + \sum_{k=0}^{\lfloor (N-1)/5 \rfloor} (\text{복사 비용 for k-th resize})$
여기서 복사 비용은 $0, 5, 10, \ldots, 5 \times \lfloor (N-1)/5 \rfloor$ 와 같습니다.
$T(N) \approx N + \sum_{k=0}^{\lfloor (N-1)/5 \rfloor} 5k = N + 5 \times \frac{\lfloor (N-1)/5 \rfloor \times (\lfloor (N-1)/5 \rfloor + 1)}{2}$

$N$이 충분히 클 때, $\lfloor (N-1)/5 \rfloor \approx N/5$ 이므로,
$T(N) \approx N + 5 \times \frac{(N/5)^2}{2} = N + 5 \times \frac{N^2/25}{2} = N + \frac{N^2}{10}$

따라서 $N$개 연산에 대한 평균 비용(분할 상환 비용)은 다음과 같습니다:
$$ \text{Amortized Cost per operation} = \frac{T(N)}{N} \approx \frac{N + \frac{N^2}{10}}{N} = 1 + \frac{N}{10} $$
이 비용은 $N$에 선형적으로 비례하며, $O(N)$의 분할 상환 비용을 가집니다. 이는 매우 비효율적인 확장 전략입니다.

### 구체적 예시
이 `StretchyArray`에 요소를 하나씩 추가하는 과정을 가상으로 따라가 봅시다. 마치 당신이 물건을 보관하기 위해 가방에 물건을 넣는 것과 같습니다.
- **초기 상태 (i=0)**: 빈 가방입니다. (`maxSize=0`, `count=0`)
- **i=1**: 첫 번째 물건을 넣으려고 합니다. 가방이 없으니, 작은 가방(최대 5개 수용)을 새로 만들고 물건을 넣습니다. (비용 `0` (복사) + `1` (삽입) = `1`)
- **i=2 ~ i=5**: 네 개의 물건을 추가로 넣습니다. 가방에 공간이 충분하니 각각 쉽게 넣을 수 있습니다. (각각 비용 `1`)
- **i=6**: 여섯 번째 물건을 넣으려 합니다. 그런데 가방(최대 5개 수용)이 가득 찼습니다! 어쩔 수 없이 더 큰 가방(최대 10개 수용)을 새로 가져와서, 기존 가방에 있던 5개의 물건을 새 가방으로 옮기고, 새 물건을 넣습니다. (비용 `5` (복사) + `1` (삽입) = `6`)
- **i=7 ~ i=10**: 네 개의 물건을 더 추가합니다. 새 가방에 공간이 충분합니다. (각각 비용 `1`)
- **i=11**: 열한 번째 물건을 넣으려 합니다. 또다시 가방(최대 10개 수용)이 가득 찼습니다! 다시 더 큰 가방(최대 15개 수용)을 가져와서 기존 10개의 물건을 옮기고, 새 물건을 넣습니다. (비용 `10` (복사) + `1` (삽입) = `11`)
이러한 과정이 계속 반복됩니다. 가끔 가방을 교체하는(배열 확장) 매우 비싼 작업이 발생하지만, 대부분의 작업은 저렴합니다. 이 테이블은 이러한 각 단계별 실제 비용을 보여줍니다.

### 시험 포인트
- ⭐ **분할 상환 분석의 중요성**: 동적 배열과 같이 특정 연산(확장)에서 높은 비용이 발생하는 자료구조의 실제 효율성을 $N$개의 연산 시퀀스에 대한 평균 비용으로 평가하는 방법임을 이해해야 합니다.
- ⭐ **`StretchyArray Version 1`의 성능 문제**: 이 슬라이드의 `Version 1`은 배열 크기를 **고정된 양(+5)** 만큼 늘립니다. 이 전략은 $N$개의 요소를 삽입할 때 총 비용이 $O(N^2)$이 되고, 각 연산의 분할 상환 비용은 $O(N)$이 됩니다. 이는 매우 비효율적입니다.
- ⭐ **효율적인 동적 배열의 확장 전략**: 실제 효율적인 동적 배열(예: `std::vector`, `ArrayList`)은 `maxSize`를 **두 배**로 늘리는 전략을 사용합니다. 이 경우 $N$개의 요소를 삽입하는 총 비용은 $O(N)$이 되고, 각 연산의 분할 상환 비용은 $O(1)$이 됩니다. 이 차이점을 명확히 이해해야 합니다.
- ⭐ **비용 계산 원리**: 배열 확장 시 발생하는 비용(`기존 maxSize` 복사 비용 + `1` 삽입 비용)과 일반 삽입 비용(`1`)의 차이를 이해하고, 테이블에서 각 `cost` 값이 어떻게 도출되었는지 설명할 수 있어야 합니다.
- ⭐ **Big-O 표기법과의 연관**: 단일 연산의 최악의 경우 비용과 분할 상환 비용이 어떻게 다를 수 있는지 설명할 수 있어야 합니다. (예: 최악의 경우 확장 시 $O(N)$ 비용이지만, 효율적인 전략에서는 분할 상환으로 $O(1)$).

## Slide 33

**핵심 개념**:
*   **분할 상환 분석 (Amortized Analysis)**: 단일 연산의 최악의 경우 비용이 매우 높더라도, 일련의 연산 시퀀스 전체에 대한 평균 비용은 효율적일 수 있음을 분석하는 기법입니다. 이는 동적 배열과 같이 때때로 큰 비용을 수반하는 연산이 발생하는 자료구조의 성능을 평가할 때 중요합니다.
*   **StretchyArray (동적 배열)의 가산적(Additive) 확장**: 이 슬라이드에서 `StretchyArray Version 1`은 배열이 가득 찼을 때 `maxSize` (최대 저장 가능 크기)를 일정한 상수 (여기서는 5)만큼 증가시키는 전략을 사용합니다. 이는 배열의 크기를 *선형적으로* 증가시키는 방식입니다.

**코드/수식 해설**:
슬라이드는 `StretchyArray`에 요소를 추가(`add`)하는 연산의 비용을 테이블로 보여줍니다. `cost` 열은 각 `add` 연산에 드는 비용을 나타냅니다.
*   `add` 연산 시, 배열에 충분한 공간이 있으면 새 요소를 추가하는 데 `1`의 비용이 듭니다.
*   배열이 가득 차서 리사이징이 필요한 경우, 새로운 (더 큰) 배열을 할당하고 기존 `maxSize`개의 요소를 새 배열로 복사한 다음, 새 요소를 추가합니다. 이때의 비용은 `기존 maxSize` (복사 비용) + `1` (새 요소 추가 비용)이 됩니다.

테이블 예시:
*   `i=1`: 초기 `maxSize`가 0이었고, 첫 요소를 추가하기 위해 `maxSize`가 5로 증가합니다. `0`개 요소를 복사하고 `1`개 요소를 추가하여 `cost = 0 + 1`.
*   `i=2`부터 `i=5`까지: `maxSize`가 5인 배열에 여유 공간이 있으므로 각 `add` 연산의 `cost = 1`.
*   `i=6`: `maxSize`가 5였는데 배열이 가득 찼습니다. `maxSize`를 10으로 증가시키며, 기존 5개의 요소를 복사한 후 1개 요소를 추가합니다. `cost = 5 + 1`.
*   `i=11`: `maxSize`가 10이었는데 배열이 가득 찼습니다. `maxSize`를 15로 증가시키며, 기존 10개의 요소를 복사한 후 1개 요소를 추가합니다. `cost = 10 + 1`.

$N$개의 요소를 `StretchyArray`에 추가하는 총 비용 $T(N)$을 계산해 봅시다.
리사이징은 `maxSize`가 $0, 5, 10, 15, \dots, 5k$가 될 때 발생합니다.
$N$개의 요소를 추가하는 과정에서 대략 $k = \lfloor (N-1)/5 \rfloor$번의 리사이징이 발생합니다.
$j$번째 리사이징 ($j=0, 1, \dots, k-1$)에서는 $5j$개의 요소를 복사합니다.
총 비용 $T(N)$은 각 요소 추가 비용의 합과 리사이징 시 복사 비용의 합으로 구성됩니다.
$$ T(N) = \sum_{p=1}^{N} (\text{p번째 요소 추가 비용}) = N \quad (\text{모든 요소에 대한 기본 비용}) + \sum_{j=0}^{k-1} (5j) \quad (\text{리사이징 시 복사 비용}) $$
여기서 $k \approx N/5$이므로, 리사이징 시 복사 비용의 합은 다음과 같습니다:
$$ \sum_{j=0}^{N/5-1} (5j) = 5 \sum_{j=0}^{N/5-1} j $$
등차수열의 합 $\sum_{j=0}^{M-1} j = \frac{(M-1)M}{2}$ 공식을 사용하고 $M \approx N/5$를 대입하면:
$$ 5 \cdot \frac{(N/5-1)(N/5)}{2} \approx 5 \cdot \frac{N^2/25}{2} = \frac{N^2}{10} $$
따라서, $N$개의 요소를 추가하는 총 비용 $T(N)$은 근사적으로 다음과 같습니다:
$$ T(N) \approx N + \frac{N^2}{10} $$
이는 $O(N^2)$의 시간 복잡도를 가집니다.
$N$번의 `add` 연산에 대한 분할 상환(평균) 비용은 총 비용을 $N$으로 나눈 값입니다:
$$ \text{Amortized Cost} = \frac{T(N)}{N} = \frac{O(N^2)}{N} = O(N) $$

**구체적 예시**:
마트에서 쇼핑 카트의 크기를 늘리는 상황에 비유해 봅시다.
*   **가산적 확장 방식**: 쇼핑 카트가 꽉 찰 때마다, 현재 카트 용량에 '봉투 5개'만큼의 추가 용량을 더해서 새로운 카트를 가져오는 방식입니다.
    *   처음에는 작은 카트(0개 용량)로 시작합니다.
    *   물건을 하나 담으려는데 카트가 없으면, 5개 용량의 카트를 가져와 1개를 담습니다. (비용: 카트 마련 + 1개 담기)
    *   5개 용량 카트가 꽉 차면, 10개 용량 카트를 가져와 기존 5개의 물건을 옮기고 새 물건을 담습니다. (비용: 5개 옮기기 + 카트 마련 + 1개 담기)
    *   10개 용량 카트가 꽉 차면, 15개 용량 카트를 가져와 기존 10개의 물건을 옮기고 새 물건을 담습니다. (비용: 10개 옮기기 + 카트 마련 + 1개 담기)
이 방식은 담아야 할 물건의 수가 많아질수록, 새로운 물건 하나를 담기 위해 옮겨야 할 기존 물건의 수가 계속 늘어나므로, 결국 물건 하나를 담는 평균 비용이 점점 더 비싸지게 됩니다 ($O(N)$).

**시험 포인트**:
*   ⭐ **가산적 확장 전략의 분할 상환 시간 복잡도**: `StretchyArray Version 1`과 같이 배열 크기를 상수로 증가시키는 가산적 확장 전략이 $N$개의 요소를 추가할 때 총 $O(N^2)$의 비용이 들고, 따라서 한 연산당 분할 상환 비용이 $O(N)$임을 이해하고 설명할 수 있어야 합니다.
*   ⭐ **테이블 해석 능력**: 주어진 테이블에서 `maxSize`, `count`, `cost`가 어떻게 변하는지, 특히 리사이징이 언제 발생하고 그때 `cost`가 어떻게 계산되는지 정확히 파악하는 것이 중요합니다.
*   ⭐ **분할 상환 분석의 중요성**: 특정 연산의 최악의 경우 비용이 아닌, 일련의 연산에 대한 평균 비용을 분석하는 이유를 설명할 수 있어야 합니다.
*   ⭐ **가산적 확장과 승법적(Multiplicative) 확장의 비교**: 일반적으로 동적 배열에서 `maxSize`를 2배, 1.5배와 같이 상수를 곱하여 확장하는 승법적 전략이 가산적 확장보다 효율적이며, 승법적 확장의 분할 상환 비용은 $O(1)$임을 알아두세요. (이 슬라이드에서는 가산적 확장만 다루지만, 시험에서 비교 문제로 나올 수 있습니다.)

## Slide 34

## Amortized Analysis of StretchyArray Version 1

### 핵심 개념

*   **상환 분석 (Amortized Analysis)**: 일부 연산은 매우 높은 비용을 가질 수 있지만, 전체 연산 시퀀스를 고려했을 때 평균적으로는 낮은 비용을 가지는 경우를 분석하는 기법입니다. 최악의 경우 비용($O(N)$)이 자주 발생하지 않도록 하여 전체적인 효율성을 평가합니다.
*   **동적 배열 (StretchyArray)**: 크기가 고정된 일반 배열과 달리, 필요에 따라 자동으로 크기를 조절할 수 있는 배열입니다. 요소가 추가되어 배열이 꽉 차면, 더 큰 공간을 할당하고 기존 요소들을 새 공간으로 복사하는 과정을 거칩니다.
*   **선형 증가 (Additive Growth) 방식**: 동적 배열의 크기를 조절할 때, 현재 용량에 고정된 크기만큼(예: +5, +10 등) 더해서 늘리는 방식입니다. 이 슬라이드에서는 고정된 크기 5만큼 용량이 증가하는 StretchyArray를 가정합니다.

### 코드/수식 해설

`put` 연산을 `n`번 수행한다고 가정할 때, `StretchyArray Version 1`의 비용 분석은 다음과 같습니다.

*   `put` 연산의 총 횟수: `n`
*   배열 스트레칭(크기 조절) 횟수: `k`
*   `n`과 `k`의 관계: `n = 5k` (배열 용량이 5씩 증가하며, `k`번 스트레칭 후 `n`개의 요소를 담을 수 있게 됨)

1.  **스트레칭 비용 합계**:
    배열이 `k`번 스트레칭될 때의 총 비용은 다음과 같습니다.
    처음 `n`개의 `put` 연산 자체는 각각 $O(1)$의 비용을 가지지만, 배열이 꽉 차서 용량을 늘리는 (stretch) 작업은 추가 비용을 발생시킵니다. 여기서는 $k$번의 스트레칭 동안 발생하는 총 복사 비용을 나타냅니다.
    $$
    \text{Stretch Cost} = 0 + 5 + 10 + \ldots + 5(k-1)
    $$
    이는 등차수열의 합으로 나타낼 수 있습니다.
    $$
    = 5(0 + 1 + 2 + \ldots + (k-1)) \\
    = 5 \frac{(k-1)k}{2}
    $$

2.  **총 비용 (Total Cost)**:
    총 비용은 `n`번의 `put` 연산(스트레칭 없이 각 연산이 1의 비용을 갖는다고 가정)과 `k`번의 스트레칭 비용의 합입니다.
    $$
    \text{Total Cost} = n + 5 \frac{k(k-1)}{2}
    $$
    여기서 $k = n/5$ 이므로, 이를 대입합니다.
    $$
    = n + 5 \frac{(n/5)(n/5-1)}{2} \\
    = n + \frac{n(n/5-1)}{2} \\
    = n + \frac{n^2/5 - n}{2} \\
    = n + \frac{n^2}{10} - \frac{n}{2} \\
    = \frac{n}{2} + \frac{n^2}{10}
    $$
    슬라이드에서는 $\approx n + n^2/10$으로 근사하고 있습니다. 큰 $n$에 대해 이 두 항 모두 $O(n^2)$에 해당합니다.

3.  **`put(x)` 연산의 상환 비용 (Amortized Cost)**:
    상환 비용은 총 비용을 총 연산 횟수 `n`으로 나눈 값입니다.
    $$
    \text{Amortized Cost} = \frac{\text{Total Cost}}{n} \\
    = \frac{\frac{n}{2} + \frac{n^2}{10}}{n} \\
    = \frac{1}{2} + \frac{n}{10}
    $$
    결과적으로 `put(x)` 연산의 상환 비용은 $O(n)$이 됩니다.

### 구체적 예시

가상의 `StretchyArray`가 처음 5개의 공간을 가지고 시작하고, 꽉 찰 때마다 5개씩 공간을 늘린다고 가정해 봅시다.

1.  **초기 상태**: 배열 크기 5. `put` 연산 5번까지는 $O(1)$ 비용으로 처리됩니다.
2.  **첫 번째 스트레칭**: 6번째 `put` 연산을 위해 배열이 꽉 찼습니다. 새롭게 5개 공간을 늘려 총 10개 공간을 만들고, 기존 5개 요소를 복사합니다. 비용: 5. (슬라이드의 $5(k-1)$에서 $k=2$일 때 $5(1)=5$에 해당)
3.  **두 번째 스트레칭**: 11번째 `put` 연산을 위해 배열이 꽉 찼습니다. 새롭게 5개 공간을 늘려 총 15개 공간을 만들고, 기존 10개 요소를 복사합니다. 비용: 10. (슬라이드의 $5(k-1)$에서 $k=3$일 때 $5(2)=10$에 해당)
4.  이러한 방식으로 배열이 `n`번 `put`될 때까지 스트레칭 비용이 누적됩니다. `k`번 스트레칭이 일어났다면, 총 `n=5k`개의 요소를 넣은 것이며, 스트레칭에 든 총 복사 비용은 $0+5+10+\dots+5(k-1)$이 됩니다.

**실생활 비유**:
매번 손님이 올 때마다 정확히 1칸만 늘릴 수 있는 주차장을 생각해보세요. 꽉 찰 때마다 1칸씩 늘리는 작업을 반복한다면, 주차 공간이 늘어날수록 새 공간을 만들고 차들을 다시 배치하는(복사하는) 비용이 점점 더 커지게 됩니다. 이는 한 명의 손님(put 연산)을 받을 때마다 드는 평균 비용이 주차장 크기에 비례하여 증가하는 상황과 유사합니다.

### 시험 포인트

*   ⭐ **상환 분석의 목적**: 특정 연산의 최악의 경우 비용이 높더라도, 일련의 연산 시퀀스에 대한 평균 비용을 평가하여 자료구조의 실제 효율성을 판단하는 데 사용됩니다.
*   ⭐ **동적 배열의 Additive Growth 방식의 비효율성**: Additive Growth 방식으로 배열 크기를 조절하는 경우, `n`개의 요소를 추가하는 총 비용은 $O(n^2)$이며, `put` 연산의 상환 비용은 $O(n)$이 됩니다.
*   **이를 개선하기 위한 방식**: 이러한 비효율성 때문에 대부분의 실용적인 동적 배열(예: C++의 `std::vector`, Java의 `ArrayList`)은 배열 크기를 *곱셈적*(multiplicative)으로 증가시키는 전략(예: 2배)을 사용하여 `put` 연산의 상환 비용을 $O(1)$로 만듭니다. 이 슬라이드의 분석은 왜 Additive Growth가 좋지 않은지를 보여주는 중요한 예시입니다.

## Slide 35

이 슬라이드는 동적 배열(StretchyArray)의 분할 상환 분석(Amortized Analysis)을 "Version 2"라는 제목 아래 표 형태로 보여줍니다. 배열이 가득 찼을 때 크기를 두 배로 늘리는 전략을 사용하는 경우, 각 삽입 작업의 실제 비용과 전체적인 비용 경향을 분석합니다.

---

### **핵심 개념**

*   **동적 배열 (Dynamic Array / StretchyArray)**: 고정된 크기를 가지는 일반 배열과 달리, 필요에 따라 자동으로 크기를 조절할 수 있는 배열입니다. `std::vector` (C++), `ArrayList` (Java) 등이 이에 해당합니다.
*   **배열 크기 확장 전략**: 이 슬라이드에서는 동적 배열이 가득 찼을 때 현재 `maxSize`를 두 배로 늘리는 전략을 사용합니다.
*   **분할 상환 분석 (Amortized Analysis)**: 최악의 경우(resizing이 발생하는 경우)의 비용이 매우 높지만, 이러한 고비용 작업이 드물게 발생하며, 저비용 작업들이 많이 수행되어 전체 작업 시퀀스에 대한 평균 비용이 낮아지는 경우를 분석하는 방법입니다. 즉, 특정 작업 한 번의 비용이 아니라, 일련의 `N`개 작업에 대한 총 비용을 `N`으로 나누어 평균적인 비용을 계산합니다.

### **코드/수식 해설**

슬라이드에 직접적인 코드는 없지만, `StretchyArray`의 `add` (삽입) 작업 로직을 다음 의사 코드로 유추할 수 있습니다.

```
function add(element):
    if count == maxSize:
        newMaxSize = maxSize * 2
        newArray = allocate(newMaxSize)
        copyElements(currentArray, newArray, count) // 비용 발생: count
        currentArray = newArray
        maxSize = newMaxSize
    currentArray[count] = element // 비용 발생: 1
    count = count + 1
```

**비용 분석**:
`cost` 열은 각 `add` 작업에 드는 비용을 나타냅니다.
*   **일반적인 삽입**: `count < maxSize`인 경우, 단순히 요소를 배열에 추가하고 `count`를 증가시키는 비용 `1`이 듭니다.
*   **확장(Resizing)을 동반한 삽입**: `count == maxSize`인 경우,
    1.  새로운, 더 큰 배열을 할당합니다.
    2.  기존 `count`개의 요소를 새 배열로 복사합니다. 이때 `count`만큼의 비용이 발생합니다.
    3.  새 요소를 새 배열에 추가합니다. 이때 비용 `1`이 발생합니다.
    총비용은 `count + 1`이 됩니다.

**수식**:
`N`개의 요소를 삽입하는 경우의 총 비용 `T(N)`은 다음과 같이 분석할 수 있습니다.
$T(N) = \sum_{i=1}^{N} \text{cost}_i$

확장 비용은 배열의 크기가 $1 \to 2 \to 4 \to 8 \to \dots \to 2^k$ (여기서 $2^k \ge N$)로 늘어날 때 발생합니다. 각 확장에서 이전 `maxSize`만큼의 요소를 복사합니다.
따라서, 총 복사 비용은 다음과 같습니다.
$1 + 2 + 4 + \dots + 2^{k-1} = 2^k - 1$

총 삽입 비용은 각 요소당 `1`이므로 `N`입니다.
그러므로 총 비용은 대략 다음과 같습니다.
$T(N) \approx N + (2^k - 1)$
여기서 $2^k$는 $N$에 가장 가까운 2의 거듭제곱이므로, $2^k < 2N$ 입니다.
$T(N) < N + (2N - 1) = 3N - 1$

`N`개의 작업에 대한 평균(분할 상환) 비용은 다음과 같습니다.
$\frac{T(N)}{N} < \frac{3N - 1}{N} \approx 3$

따라서 `add` 연산의 분할 상환 시간 복잡도는 $O(1)$ 입니다.

### **구체적 예시**

슬라이드의 표가 좋은 예시입니다. `i`는 삽입 횟수, `maxSize`는 현재 배열의 최대 용량, `count`는 현재 요소 수, `cost`는 해당 `i`번째 삽입의 비용입니다.

*   **`i=1`**: `maxSize=1`, `count=1`. 비용은 `1`. 첫 요소를 삽입합니다. (현재 배열이 가득 참)
*   **`i=2`**: `maxSize`를 2배로 늘려 `2`로 만듭니다. 기존 요소 `1`개를 복사하고, 새 요소 `1`개를 삽입합니다. 비용은 `1` (복사) + `1` (삽입) = `2`. (현재 배열이 가득 참)
*   **`i=3`**: `maxSize`를 2배로 늘려 `4`로 만듭니다. 기존 요소 `2`개를 복사하고, 새 요소 `1`개를 삽입합니다. 비용은 `2` (복사) + `1` (삽입) = `3`.
*   **`i=4`**: `maxSize=4`, `count=4`. 비용은 `1`. 새 요소를 삽입합니다. (현재 배열이 가득 참)
*   **`i=5`**: `maxSize`를 2배로 늘려 `8`로 만듭니다. 기존 요소 `4`개를 복사하고, 새 요소 `1`개를 삽입합니다. 비용은 `4` (복사) + `1` (삽입) = `5`.

이러한 패턴이 반복되어, 비용 `1`짜리 작업이 여러 번 수행되다가, `maxSize`만큼의 복사 비용이 드는 작업이 한 번씩 발생합니다. 이 비용이 전체적으로 분산되어 평균 비용이 낮아지게 됩니다.

### **시험 포인트**

*   ⭐ **동적 배열의 `add` 연산이 분할 상환 $O(1)$ 시간 복잡도를 갖는 이유를 설명하라.** (이 슬라이드의 내용이 핵심 설명이 됩니다.)
*   ⭐ **분할 상환 분석의 개념과 언제 유용한지 설명하라.** (개별 최악의 경우의 비용은 높지만, 드물게 발생하여 전체 시퀀스에 대한 평균 비용이 낮아질 때 사용합니다.)
*   동적 배열의 크기 확장 전략 중 '두 배로 늘리는' 방식과 '고정량만큼 늘리는' 방식의 시간 복잡도 차이 (고정량은 $O(N)$이 될 수 있음)를 비교하여 설명하라.
*   주어진 `N`개의 삽입 작업에 대한 동적 배열의 총 비용을 계산하는 방법을 제시하라.

### 핵심 개념
- **분할 상환 분석 (Amortized Analysis)**: 개별 연산의 최악의 경우 시간 복잡도는 높을 수 있지만, 일련의 연산 시퀀스 전체를 고려했을 때 평균적으로 각 연산의 비용이 낮은 경우를 분석하는 기법입니다. 특정 고비용 연산이 드물게 발생하고, 대부분의 연산은 저비용으로 이루어질 때 유용합니다.
- **`StretchyArray` (동적 배열)**: C++의 `std::vector`나 Java의 `ArrayList`처럼 필요에 따라 크기를 자동으로 조절하는 배열을 의미합니다. 원소를 추가할 때 현재 배열의 용량이 부족하면, 더 큰 새 배열을 할당하고 기존 원소들을 새 배열로 복사한 뒤 새 원소를 추가합니다.
- **배열 확장 전략 (Doubling Strategy)**: `StretchyArray`에서 배열의 용량이 부족할 때, 현재 용량의 2배로 확장하는 전략입니다. 이 슬라이드는 이 전략이 분할 상환적으로 효율적임을 보여줍니다.

### 코드/수식 해설
제시된 슬라이드는 `StretchyArray`에 원소를 하나씩 추가하는 `add` (또는 `push_back`) 연산의 비용을 단계별로 보여줍니다.

| `i` | `maxSize` | `count` | `cost` | `comments` |
|---|---|---|---|---|
| `_` | `1` | `0` | `_` | `Initial state` |
| `1` | `1` | `1` | `1` | `_` |
| `2` | `2` | `2` | `1 + 1` | `Copy array of size 1` |
| `3` | `4` | `3` | `2 + 1` | `Copy array of size 2` |
| `4` | `4` | `4` | `1` | `_` |
| `5` | `8` | `5` | `4 + 1` | `_` |
| `6` | `8` | `6` | `1` | `_` |
| `7` | `8` | `7` | `1` | `_` |
| `8` | `8` | `8` | `1` | `_` |
| `9` | `16` | `9` | `8 + 1` | `Copy array of size 8` |
| `10` | `16` | `10` | `1` | `_` |
| `11` | `16` | `11` | `1` | `_` |
| `...` | `...` | `...` | `...` | `...` |

- `i`: 현재까지 추가된 원소의 총 개수를 의미하며, `count`와 동일합니다.
- `maxSize`: 현재 `StretchyArray`가 할당받은 물리적인 배열의 최대 용량입니다.
- `count`: 현재 배열에 저장된 유효한 원소의 개수입니다.
- `cost`: `i`-번째 원소를 추가하는 데 드는 비용입니다.
    - 새로운 원소 1개를 추가하는 기본 비용 `1`은 항상 발생합니다.
    - 만약 원소를 추가하기 전에 `count`가 `maxSize`와 같아져서 배열을 확장해야 한다면, 새로운 더 큰 배열을 할당하고 기존 `count`개의 원소를 새 배열로 복사하는 비용이 추가됩니다. 이 복사 비용은 직전 `maxSize`와 같습니다. 따라서 총 `cost = (이전 maxSize) + 1`이 됩니다.

슬라이드의 빨간색 박스에 언급된 대로 "Enlarge steps happen basically when i is a power of 2"는 배열의 `maxSize`가 2의 거듭제곱일 때 확장이 이루어짐을 의미합니다. 정확히는 `count`가 현재 `maxSize`와 같아져서 다음 원소(`i`)를 추가해야 할 때 `maxSize`를 2배로 늘립니다.

**확장 발생 시점과 비용**:
- `i=2`일 때: `count=1`, `maxSize=1`. 2번째 원소 추가 시 `maxSize` 부족. `maxSize`를 $1 \times 2 = 2$로 확장. 기존 1개 원소 복사 + 1개 원소 추가 $\rightarrow cost = 1 + 1$.
- `i=3`일 때: `count=2`, `maxSize=2`. 3번째 원소 추가 시 `maxSize` 부족. `maxSize`를 $2 \times 2 = 4$로 확장. 기존 2개 원소 복사 + 1개 원소 추가 $\rightarrow cost = 2 + 1$.
- `i=5`일 때: `count=4`, `maxSize=4`. 5번째 원소 추가 시 `maxSize` 부족. `maxSize`를 $4 \times 2 = 8$로 확장. 기존 4개 원소 복사 + 1개 원소 추가 $\rightarrow cost = 4 + 1$.
- `i=9`일 때: `count=8`, `maxSize=8`. 9번째 원소 추가 시 `maxSize` 부족. `maxSize`를 $8 \times 2 = 16$으로 확장. 기존 8개 원소 복사 + 1개 원소 추가 $\rightarrow cost = 8 + 1$.

**`N`개의 원소 추가에 대한 총 비용 및 분할 상환 분석**:
`N`개의 원소를 `StretchyArray`에 추가하는 총 비용 $TotalCost(N)$를 계산해봅시다.
각 원소를 추가하는 기본 비용 `1`은 $N$번 발생합니다.
배열 확장으로 인한 복사 비용은 `maxSize`가 $1, 2, 4, \dots, 2^{k-1}$일 때 각각 $1, 2, 4, \dots, 2^{k-1}$만큼 발생합니다. 이때 $2^{k-1} < N \le 2^k$ (즉, `N`번째 원소를 추가한 후의 최종 `maxSize`는 $2^k$입니다).
총 복사 비용은 다음과 같습니다:
$$ \text{CopyCost} = 1 + 2 + 4 + \dots + 2^{k-1} $$
이는 등비수열의 합 공식에 따라 $2^k - 1$이 됩니다.
$2^k$는 대략 $N$의 2배 미만($N < 2^k \le 2N$)이므로, CopyCost는 대략 $N$에 비례합니다.
`TotalCost(N)`는 새로운 원소를 추가하는 비용과 배열 복사 비용의 합입니다.
$$ TotalCost(N) = N \quad (\text{for adding new elements}) + (2^k - 1) \quad (\text{for copies}) $$
대략적으로 $TotalCost(N) \approx N + N = 2N$이 됩니다.
따라서 `N`번의 `add` 연산에 대한 분할 상환 비용(Amortized Cost per operation)은 다음과 같습니다:
$$ \text{Amortized Cost per operation} = \frac{\text{TotalCost}(N)}{N} \approx \frac{2N}{N} = 2 $$
즉, `StretchyArray`의 `add` 연산은 분할 상환적으로 $O(1)$의 시간 복잡도를 가집니다.

### 구체적 예시
여러분들이 도서관에서 책을 정리하는 작업을 한다고 상상해봅시다.
처음에는 아주 작은 책장(최대 1권)을 가지고 있습니다.
1.  **1번째 책**을 받습니다. 책장에 넣습니다. (비용: 1)
2.  **2번째 책**을 받으려고 하는데, 책장이 꽉 찼습니다. 새 책장(2권)을 가져와서 기존 책 1권을 옮기고, 2번째 책을 넣습니다. (비용: 1(옮기는 비용) + 1(넣는 비용) = 2)
3.  **3번째 책**을 받으려고 하는데, 새 책장이 꽉 찼습니다. 더 큰 책장(4권)을 가져와서 기존 책 2권을 옮기고, 3번째 책을 넣습니다. (비용: 2(옮기는 비용) + 1(넣는 비용) = 3)
4.  **4번째 책**을 받습니다. 책장이 아직 여유 있으므로 바로 넣습니다. (비용: 1)
5.  **5번째 책**을 받으려고 하는데, 책장이 꽉 찼습니다. 더 큰 책장(8권)을 가져와서 기존 책 4권을 옮기고, 5번째 책을 넣습니다. (비용: 4(옮기는 비용) + 1(넣는 비용) = 5)

이러한 방식으로, 대부분의 책은 바로 책장에 들어가지만, 가끔 책장을 바꿔야 할 때는 이전 책장의 모든 책을 새 책장으로 옮기는 큰 작업이 발생합니다. 하지만 이러한 큰 작업은 점점 덜 자주 발생하며 (책장 용량이 2배씩 늘어나므로), 총 책의 개수에 비례하여 평균적으로는 효율적인 작업이 됩니다.

### 시험 포인트
- ⭐ **분할 상환 분석의 개념 및 필요성**: 개별 연산의 최악의 경우 비용이 높아도 전체 시퀀스의 평균 비용이 낮은 경우를 분석하는 기법임을 이해하고, 왜 동적 배열 분석에 적합한지 설명할 수 있어야 합니다.
- ⭐ **`StretchyArray`의 동적 확장 전략**: 용량이 부족할 때 배열을 2배로 확장하는 전략 (`doubling strategy`)이 분할 상환적으로 $O(1)$ 비용을 보장하는 핵심임을 이해해야 합니다.
- ⭐ **총 비용 계산 및 분할 상환 비용 유도**: `N`개의 원소를 추가하는 총 비용이 $O(N)$임을 유도하고, 이를 통해 각 연산의 분할 상환 비용이 $O(1)$임을 증명할 수 있어야 합니다. 특히 복사 비용이 등비수열의 합($1 + 2 + 4 + \dots + 2^{k-1} \approx N$)으로 계산됨을 알아야 합니다.
- ⭐ **다른 확장 전략과의 비교**: 만약 용량을 상수만큼(예: 1칸씩) 늘린다면, 각 확장이 일어날 때마다 $O(N)$의 복사 비용이 발생하고, 총 $N$번 연산에 대해 $O(N^2)$의 총 비용이 들어 분할 상환 비용이 $O(N)$이 된다는 점을 이해하고, 왜 2배 확장이 우수한지 설명할 수 있어야 합니다.

## Slide 37

### **핵심 개념**
이 슬라이드는 동적 배열(StretchyArray 또는 ArrayList, Vector 등)의 `put` (삽입) 연산에 대한 **상환 분석(Amortized Analysis)**을 설명합니다. 상환 분석은 일련의 연산에 대한 평균적인 비용을 평가하는 방법으로, 일부 연산이 매우 비싸더라도 전체 시퀀스에 걸쳐 분산된 평균 비용이 낮을 수 있음을 보여줍니다.

동적 배열은 미리 정해진 크기를 가지는 일반 배열과 달리, 요소가 추가될 때 배열의 용량이 부족해지면 자동으로 크기를 확장합니다. 이때 발생하는 배열 확장(resizing)은 기존 요소를 새롭고 더 큰 배열로 복사해야 하므로 O(N)의 비용이 드는 비싼 연산입니다. 하지만 이 슬라이드는 이러한 비싼 연산이 드물게 발생하며, 전체 삽입 연산 시퀀스를 고려했을 때 `put` 연산의 평균(상환) 비용이 O(1)임을 증명합니다.

### **코드/수식 해설**

슬라이드에서는 `n`개의 `put` 연산이 수행된다고 가정하며, 특히 $n = 2^k$ ($k$는 정수) 형태의 연산 수를 가정합니다. 이는 분석을 단순화하기 위함입니다.

1.  **개별 `put` 연산 비용**:
    `n`번의 `array[count] = x` 호출은 각각 O(1)의 비용을 가집니다. (배열 확장이 필요 없는 경우)

2.  **배열 확장(resizing) 비용**:
    배열은 용량이 가득 찰 때마다 크기를 확장합니다. 일반적으로 기존 용량의 두 배로 확장합니다.
    *   초기 용량 1에서 첫 번째 확장: 1개 요소를 복사하여 용량 2로 만듦. 비용 1.
    *   용량 2에서 두 번째 확장: 2개 요소를 복사하여 용량 4로 만듦. 비용 2.
    *   용량 4에서 세 번째 확장: 4개 요소를 복사하여 용량 8로 만듦. 비용 4.
    *   ...
    *   용량 $2^{k-1}$에서 마지막 $k$번째 확장: $2^{k-1}$개 요소를 복사하여 용량 $2^k$로 만듦. 비용 $2^{k-1}$.

    총 $k$번의 배열 확장 비용은 다음 등비수열의 합으로 표현됩니다:
    $$ \approx (1 + 2 + 4 + \dots + 2^{k-1}) $$
    이 등비수열의 합은 다음과 같습니다:
    $$ \sum_{i=0}^{k-1} 2^i = 2^k - 1 $$

3.  **총 비용 계산**:
    `n`개의 `put` 연산을 수행하는 데 드는 총 비용은 개별 `put` 연산 비용과 모든 배열 확장 비용의 합입니다.
    $$ \text{Total cost} \approx n + (1 + 2 + 4 + \dots + 2^{k-1}) $$
    위에서 계산한 등비수열의 합을 대입합니다:
    $$ \approx n + (2^k - 1) $$
    여기서 가정 $n = 2^k$를 대입하면:
    $$ \approx n + n - 1 = 2n - 1 $$

4.  **상환 비용 (Amortized Cost)**:
    `put(x)` 연산 하나당 평균 비용, 즉 상환 비용은 총 비용을 총 연산 횟수 `n`으로 나눈 값입니다.
    $$ \text{Amortized cost for put}(x) = \frac{2n - 1}{n} $$
    $$ = 2 - \frac{1}{n} $$
    $$ = O(1) $$
    `n`이 매우 커지면 $\frac{1}{n}$은 0에 가까워지므로, `put` 연산의 상환 비용은 상수 시간인 O(1)이 됩니다.

### **구체적 예시**

처음에는 용량이 1인 `StretchyArray`가 있다고 가정하고, 8개의 요소를 삽입하는 과정을 예로 들어보겠습니다 ($n=8$). $n=8$이므로 $2^k=8$, 즉 $k=3$ 입니다.

| 연산 번호 | 현재 요소 수 | 현재 용량 | `put` 비용 (O(1) 삽입) | Resizing 여부 | Resizing 비용 (요소 복사 수) | 누적 총 비용 |
| :-------: | :----------: | :-------: | :--------------------: | :-----------: | :---------------------------: | :----------: |
|     1     |      0       |     1     |           1            |       No      |               0               |      1       |
|     2     |      1       |     1     |           1            |      Yes      |            1 (용량 2)           |    1+1+1=3   |
|     3     |      2       |     2     |           1            |       No      |               0               |    3+1=4     |
|     4     |      3       |     2     |           1            |      Yes      |            2 (용량 4)           |    4+1+2=7   |
|     5     |      4       |     4     |           1            |       No      |               0               |    7+1=8     |
|     6     |      5       |     4     |           1            |       No      |               0               |    8+1=9     |
|     7     |      6       |     4     |           1            |      Yes      |            4 (용량 8)           |   9+1+4=14   |
|     8     |      7       |     8     |           1            |       No      |               0               |   14+1=15    |

총 8개의 `put` 연산에 대한 총 비용은 15입니다.
슬라이드의 공식에 따르면 $2n - 1 = 2 \times 8 - 1 = 16 - 1 = 15$. 일치합니다.
상환 비용은 $\frac{15}{8} = 1.875 \approx O(1)$ 입니다.
이처럼, 간헐적으로 발생하는 비싼 O(N) 리사이징 연산이 많은 O(1) 연산에 의해 '분산'되어 평균적으로는 O(1)이 됩니다.

### **시험 포인트**

*   ⭐ **상환 분석의 정의 및 목적**: 특정 연산의 최악의 경우(Worst-case) 비용이 높더라도, 일련의 연산에 대한 평균 비용(Average-case for sequence)이 낮을 수 있음을 증명하는 분석 기법임을 이해해야 합니다.
*   ⭐ **동적 배열의 `put` 연산이 O(1) 상환 비용을 가지는 이유**: 배열 용량이 가득 찼을 때 두 배로 확장하는 전략(doubling strategy) 덕분입니다. 이는 리사이징 비용이 등비수열의 합으로 나타나며, 전체 연산 수에 비례하는 총 비용을 만듭니다.
*   ⭐ **총 비용 및 상환 비용 계산 과정**: 슬라이드에 제시된 $2n-1$ 이라는 총 비용과 $O(1)$ 상환 비용을 유도하는 과정을 정확히 이해하고 설명할 수 있어야 합니다. (특히 등비수열의 합 $1+2+\dots+2^{k-1} = 2^k-1$ 활용).
*   ⭐ **만약 배열 확장 전략이 `+C` (고정량 증가)였다면?**: 용량이 가득 찰 때마다 `C`만큼 증가시켰다면, 리사이징이 훨씬 더 자주 발생하고 각 리사이징 비용도 커져 상환 비용이 O(1)이 되지 않고 O(N)이 될 수 있음을 이해해야 합니다. (이는 doubling strategy의 중요성을 강조).

## Slide 38

이 슬라이드는 동적 배열(Dynamic Array)에서 원소 추가(`put(x)`) 시 발생하는 재할당(reallocation) 비용을 **상환 분석(Amortized Analysis)** 관점에서 설명하고 있습니다.

---

### **핵심 개념**

*   **상환 분석 (Amortized Analysis)**: 특정 자료구조에 대해 일련의 연산들이 수행될 때, 개별 연산의 최악의 경우(worst-case) 비용이 아닌, 전체 시퀀스에 걸친 '평균적인' 비용을 측정하는 분석 기법입니다. 이는 특정 연산이 간헐적으로 매우 높은 비용을 발생시키지만, 그 발생 빈도가 낮아 전체적으로는 효율적인 경우를 평가할 때 유용합니다.
*   **동적 배열의 크기 조절 전략**:
    *   **상수량 증가 (Constant Amount Stretch)**: 배열의 공간이 부족할 때마다 현재 크기에서 일정한 상수만큼(예: +1, +10) 늘리는 전략입니다. 이 경우, `put(x)` 연산의 상환 시간 복잡도는 $O(n)$이 됩니다.
    *   **배수 증가 (Doubling Size)**: 배열의 공간이 부족할 때마다 현재 크기의 특정 배수만큼(예: 2배) 늘리는 전략입니다. 이 경우, `put(x)` 연산의 상환 시간 복잡도는 $O(1)$이 됩니다.

---

### **코드/수식 해설**

동적 배열에 `put(x)` 연산을 통해 원소를 추가하는 과정에서 배열의 현재 용량이 가득 찼을 경우, 다음과 같은 재할당 절차가 필요합니다.

```c
void put(int x) {
    if (current_size == capacity) { // 배열이 가득 찼으면 재할당 필요
        // 1. 새로운 용량 결정 (전략에 따라 달라짐)
        int new_capacity;
        // 상수량 증가: new_capacity = capacity + CONSTANT;
        // 배수 증가: new_capacity = capacity * FACTOR; (일반적으로 2)

        // 2. 새로운, 더 큰 배열 할당
        ElementType* new_array = (ElementType*)malloc(new_capacity * sizeof(ElementType));

        // 3. 기존 배열의 모든 원소를 새 배열로 복사 (이 부분이 비용 발생)
        for (int i = 0; i < current_size; i++) {
            new_array[i] = array[i];
        }

        // 4. 이전 배열 메모리 해제 및 포인터 업데이트
        free(array);
        array = new_array;
        capacity = new_capacity;
    }
    // 5. 새 원소 추가
    array[current_size++] = x;
}
```

*   **배열 재할당 시 비용**: 위 코드에서 3단계의 `for` 루프는 현재 `current_size`에 비례하는 $O(current\_size)$의 시간이 소요됩니다. 이를 $O(n)$으로 표기할 수 있습니다.

*   **상수량 증가 전략 ($O(n)$ 상환 시간 복잡도)**:
    초기 용량이 $C_0$이고 매번 $K$만큼 용량을 늘린다고 가정해봅시다. $N$개의 원소를 추가하기 위해 $N/K$번의 재할당이 발생할 수 있습니다. 각 재할당 시 복사되는 원소의 수는 $C_0, C_0+K, C_0+2K, \dots$ 와 같이 증가합니다. $N$번의 `put` 연산을 수행했을 때, 총 복사 비용은 대략 $\sum_{i=1}^{N/K} O(i \cdot K) = O(N^2)$이 됩니다.
    $N$번의 `put` 연산에 대한 총 비용이 $O(N^2)$이므로, 개별 `put` 연산의 상환 비용은 $O(N^2)/N = O(N)$이 됩니다.

*   **배수 증가 전략 ($O(1)$ 상환 시간 복잡도)**:
    초기 용량이 $C_0$이고 매번 2배로 늘린다고 가정해봅시다. $N$개의 원소를 추가하기 위해 용량은 $C_0, 2C_0, 4C_0, \dots, 2^k C_0 \ge N$ 과 같이 증가합니다. 재할당이 발생하는 시점의 복사 비용은 각각 $C_0, 2C_0, 4C_0, \dots, 2^{k-1}C_0$입니다.
    $N$개의 원소를 추가하기 위한 총 복사 비용은 이들 비용의 합이 됩니다:
    $$ \sum_{i=0}^{k-1} 2^i C_0 = C_0 (2^k - 1) $$
    여기서 $2^{k-1}C_0 < N \le 2^k C_0$ 이므로 $2^k C_0 \approx 2N$입니다. 따라서 총 복사 비용은 대략 $O(2N) = O(N)$이 됩니다.
    $N$번의 `put` 연산에 대한 총 비용은 $O(N)$ (복사 비용) + $O(N)$ (실제 원소 추가 비용) = $O(N)$이 됩니다.
    결과적으로, 개별 `put` 연산의 상환 비용은 $O(N)/N = O(1)$이 됩니다.

### **구체적 예시**

*   **상수량 증가 전략 (비효율적인 이사):**
    당신이 방 1개짜리 아파트에 살다가, 방이 하나 더 필요할 때마다 옆집의 벽을 허물어 방 1개만 늘리는 경우를 상상해 보세요. 처음에는 괜찮겠지만, 방이 10개 필요하면 10번의 공사가 필요하고, 공사할 때마다 모든 짐을 새 방으로 옮겨야 합니다. 방이 $N$개가 될 때까지 총 $N$번 공사를 해야 하고, 각 공사마다 기존 $N$개의 방의 짐을 옮기는 비용이 발생하므로, 전체 공사 비용은 $N$이 커질수록 기하급수적으로 늘어납니다. 평균적으로 한 번 방을 늘리는 데 $O(N)$ 시간이 걸리는 셈입니다.

*   **배수 증가 전략 (효율적인 이사):**
    이번에는 방 1개짜리 아파트에 살다가, 방이 하나 더 필요할 때마다 현재 집 크기의 "두 배" 되는 새로운 집으로 이사하는 경우를 생각해 보세요.
    1.  방 1개 → 방 2개 (1개 옮김)
    2.  방 2개 → 방 4개 (2개 옮김)
    3.  방 4개 → 방 8개 (4개 옮김)
    4.  방 8개 → 방 16개 (8개 옮김)
    ...
    방이 16개인 집이 필요할 때까지 옮긴 총 짐의 수는 $1+2+4+8 = 15$개입니다. 이는 마지막에 얻은 집 크기(16개)와 거의 같습니다. 즉, 재할당(이사) 비용은 발생하지만, 이 비용의 총합이 최종적으로 확보한 전체 공간 크기에 비례합니다. 이사 횟수가 적고, 한 번 이사할 때 드는 비용이 다음에 훨씬 긴 시간 동안 상쇄되므로, 평균적으로 한 번 방을 늘리는 데 드는 비용은 상수 시간 $O(1)$에 불과하게 됩니다. Java의 `ArrayList`나 C++의 `std::vector` 같은 동적 배열 구현체들이 이 배수 증가 전략을 사용하여 효율성을 확보합니다.

### **시험 포인트**

*   ⭐ **상환 분석의 개념**: 상환 분석이 무엇이며, 왜 동적 배열과 같은 자료구조의 성능을 평가하는 데 중요한지 이해해야 합니다. 최악의 경우와 상환 분석의 차이를 설명할 수 있어야 합니다.
*   ⭐ **동적 배열 크기 조절 전략별 상환 시간 복잡도**:
    *   상수량 증가 전략의 `put(x)` 상환 시간 복잡도는 $O(n)$입니다.
    *   배수 증가 전략의 `put(x)` 상환 시간 복잡도는 $O(1)$입니다.
    이 두 가지 결과를 반드시 기억하고 있어야 합니다.
*   ⭐ **배수 증가 전략이 상수량 증가 전략보다 효율적인 이유**: 재할당 시 발생하는 복사 비용이 기하급수적으로 늘어나는 배열 크기 덕분에, 전체 `put` 연산 수에 걸쳐 분산되었을 때 평균 비용이 상수로 수렴하는 원리를 설명할 수 있어야 합니다. (위 "코드/수식 해설"의 수식적 증명 또는 "구체적 예시"의 비유를 활용)

### **핵심 개념**

*   **알고리즘(Algorithm)**:
    어떤 문제를 해결하기 위한 유한하고 명확하며 순서가 있는 단계들의 집합입니다. 컴퓨터 과학에서 알고리즘은 특정 작업을 수행하는 일련의 지침을 의미하며, 효율적인 알고리즘의 설계는 문제 해결의 핵심입니다.

*   **알고리즘 성능 비교 (Empirical vs. Asymptotic Theoretical Comparison)**:
    *   **실험적 비교(Empirical comparison)**: 실제 구현된 알고리즘을 다양한 입력 데이터에 대해 실행하여 소요되는 시간을 측정하고 비교하는 방법입니다. 실제 환경에서의 성능을 파악할 수 있지만, 특정 하드웨어, 소프트웨어, 입력 데이터에 의존적입니다.
    *   **점근적 이론 비교(Asymptotic theoretical comparison)**: 입력 크기($n$)가 충분히 클 때 (즉, 점근적 관점), 알고리즘의 시간 또는 공간 복잡도가 어떻게 성장하는지를 수학적으로 분석하여 비교하는 방법입니다. 하드웨어/소프트웨어 환경에 독립적이며, 대규모 입력에 대한 알고리즘의 확장성을 예측하는 데 유용합니다. ⭐**시험 포인트**: 두 비교 방식의 장단점과 왜 점근적 분석이 더 보편적으로 사용되는지 이해하는 것이 중요합니다.

*   **최선, 최악, 평균 케이스(Best, Worst, Average Cases)**:
    알고리즘의 성능은 입력 데이터의 특성에 따라 달라질 수 있으므로, 세 가지 시나리오를 고려하여 분석합니다.
    *   **Best Case (최선 케이스)**: 알고리즘이 작업을 수행하는 데 가장 적은 시간이 걸리는 최적의 입력 시나리오입니다.
    *   **Worst Case (최악 케이스)**: 알고리즘이 작업을 수행하는 데 가장 많은 시간이 걸리는 최악의 입력 시나리오입니다. 대부분의 경우, 우리는 최악 케이스 성능을 최적화하는 데 집중합니다. ⭐**시험 포인트**: 최악 케이스가 특히 중요한 이유는 시스템의 안정성을 보장하기 위함입니다.
    *   **Average Case (평균 케이스)**: 모든 가능한 입력에 대해 평균적으로 소요되는 시간입니다. 모든 입력 분포를 고려해야 하므로 분석하기 가장 어렵습니다.

*   **Big O notation (빅-오 표기법)**:
    알고리즘의 점근적 상한(asymptotic upper bound)을 나타내는 표기법입니다. 입력 크기 $n$이 증가함에 따라 알고리즘의 실행 시간 또는 공간 사용량이 어떻게 증가하는지 그 **최대 성장률**을 표현합니다. 즉, 알고리즘이 아무리 나빠도 이 시간 복잡도를 넘어서지는 않음을 의미합니다.

*   **분할 상환 분석(Amortized analysis)**:
    일련의 연산들에 대해 총 비용을 분석한 후, 이 총비용을 각 개별 연산에 "분할 상환"하여 개별 연산의 평균 비용을 계산하는 분석 기법입니다. 특정 연산은 매우 비쌀 수 있지만, 이러한 비싼 연산은 드물게 발생하고, 대부분의 연산은 저렴하게 수행될 때 유용합니다. 일반적인 평균 케이스 분석과 달리 입력 데이터의 확률 분포에 의존하지 않고, 최악의 경우를 포함한 전체 연산 시퀀스에 대한 평균 비용을 계산합니다. ⭐**시험 포인트**: 분할 상환 분석의 개념을 정확히 이해하고, 동적 배열의 예시를 통해 설명할 수 있어야 합니다.

### **코드/수식 해설**

*   **Big O notation 수식**:
    함수 $f(n)$이 $O(g(n))$이라는 것은, 양의 상수 $c$와 충분히 큰 $n_0$가 존재하여 모든 $n \ge n_0$에 대해 다음 관계가 성립함을 의미합니다.
    $$0 \le f(n) \le c \cdot g(n)$$
    여기서 $f(n)$은 알고리즘의 실제 실행 시간 또는 연산 횟수를 나타내며, $g(n)$은 일반적으로 $n, \log n, n^2$ 등과 같은 단순 함수로, 알고리즘의 성장률을 나타내는 지표입니다. 최고차항만 고려하고 상수 계수는 무시합니다.

### **구체적 예시**

*   **Best, Worst, Average Cases 예시 (선형 탐색)**:
    정렬되지 않은 배열에서 특정 값 `x`를 찾는 선형 탐색(Linear Search)을 생각해 봅시다.
    *   **Best Case**: 배열의 첫 번째 위치에서 `x`를 찾는 경우. 비교 횟수는 1번이므로 시간 복잡도는 $O(1)$입니다.
    *   **Worst Case**: 배열의 마지막 위치에서 `x`를 찾거나, 배열에 `x`가 없어 끝까지 탐색하는 경우. 비교 횟수는 배열 크기 $n$번이므로 시간 복잡도는 $O(n)$입니다.
    *   **Average Case**: `x`가 배열에 존재하고 각 위치에 있을 확률이 동일하다면, 평균적으로 $n/2$번의 비교가 필요합니다. 따라서 시간 복잡도는 $O(n)$입니다.

*   **Big O notation 예시**:
    *   배열에서 특정 인덱스의 요소를 접근하는 연산: $O(1)$ (상수 시간)
    *   정렬되지 않은 배열에서 특정 값을 찾는 선형 탐색: $O(n)$ (선형 시간)
    *   정렬된 배열에서 특정 값을 찾는 이진 탐색: $O(\log n)$ (로그 시간)
    *   버블 정렬과 같은 간단한 정렬 알고리즘: $O(n^2)$ (제곱 시간)

*   **분할 상환 분석 예시 (동적 배열의 `push_back` 연산)**:
    C++의 `std::vector`나 Python의 `list`와 같은 동적 배열은 `push_back` (또는 `append`) 연산을 통해 요소를 추가합니다.
    *   대부분의 `push_back` 연산은 배열 끝에 단순히 요소를 추가하는 것이므로 $O(1)$ 시간이 소요됩니다.
    *   하지만 배열이 꽉 찼을 때, 더 큰 메모리 공간을 할당하고(예: 현재 크기의 두 배), 기존의 모든 요소를 새로운 공간으로 복사해야 합니다. 이 경우, $n$개의 요소를 복사해야 하므로 $O(n)$ 시간이 소요됩니다.
    *   $N$개의 `push_back` 연산을 연속적으로 수행한다고 가정해봅시다. 크기 확장(reallocation)은 배열 크기가 1, 2, 4, 8, ... $2^k$가 될 때 발생합니다. 총 $N$개의 요소를 추가하는 동안 발생하는 복사 연산의 총합은 대략 $1+2+4+...+N/2 \approx N$입니다. 따라서 전체 $N$개의 연산에 대한 총 비용은 대략 $N$ (단순 추가) + $N$ (복사) $= 2N$이 됩니다.
    *   이 총 비용 $2N$을 $N$개의 연산에 분할 상환하면, 각 `push_back` 연산의 상환 비용은 $2N/N = 2$가 되어 $O(1)$이 됩니다. 즉, 개별적으로는 비싼 $O(n)$ 연산이 있지만, 전체적으로 보면 각 `push_back` 연산은 평균적으로 $O(1)$의 비용을 갖는다고 볼 수 있습니다.

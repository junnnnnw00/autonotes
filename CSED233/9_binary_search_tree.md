## Binary Search Tree

### 핵심 개념

이 슬라이드는 **이진 탐색 트리(Binary Search Tree, BST)**에 대한 강의의 시작을 알립니다. 이진 탐색 트리는 특정 규칙을 만족하는 이진 트리(Binary Tree)의 한 종류로, 효율적인 데이터 탐색, 삽입, 삭제를 가능하게 하는 자료구조입니다.

BST의 핵심 속성은 다음과 같습니다:
1.  **이진 트리 속성**: 모든 노드는 최대 두 개의 자식 노드(왼쪽 자식, 오른쪽 자식)를 가집니다.
2.  **탐색 속성**:
    *   모든 노드의 왼쪽 서브트리(left subtree)에 있는 값들은 해당 노드의 값보다 작습니다.
    *   모든 노드의 오른쪽 서브트리(right subtree)에 있는 값들은 해당 노드의 값보다 큽니다.
    *   중복된 값은 허용되지 않거나, 특정 규칙(예: 항상 오른쪽으로 삽입)에 따라 처리됩니다.

이러한 속성 덕분에 BST는 정렬된 데이터를 유지하면서도 동적인 삽입 및 삭제가 가능하여, 데이터베이스 인덱스나 사전 구현 등에 활용됩니다.

### 코드/수식 해설

BST의 기본 구성 요소인 노드(Node)의 구조와 주요 연산(탐색, 삽입, 삭제)의 의사 코드는 다음과 같습니다.

**1. 노드 구조**

```c
typedef struct Node {
    int key;
    struct Node *left;
    struct Node *right;
    struct Node *parent; // 선택적
} Node;
```

**2. 탐색 (Search) 연산**
루트에서 시작하여, 찾으려는 `key`와 현재 노드의 `key`를 비교하여 왼쪽 또는 오른쪽으로 이동합니다.

```c
Node* search(Node* root, int key) {
    if (root == NULL || root->key == key) {
        return root;
    }
    if (key < root->key) {
        return search(root->left, key);
    } else {
        return search(root->right, key);
    }
}
```

**3. 삽입 (Insert) 연산**
탐색 연산과 유사하게 적절한 위치를 찾아 새 노드를 삽입합니다.

```c
Node* insert(Node* root, int key) {
    if (root == NULL) {
        Node* newNode = (Node*)malloc(sizeof(Node));
        newNode->key = key;
        newNode->left = NULL;
        newNode->right = NULL;
        // newNode->parent = NULL; // 필요시 설정
        return newNode;
    }
    if (key < root->key) {
        root->left = insert(root->left, key);
        // if (root->left->parent == NULL) root->left->parent = root; // 필요시 설정
    } else if (key > root->key) { // 중복 키는 일반적으로 허용하지 않거나 오른쪽으로 삽입
        root->right = insert(root->right, key);
        // if (root->right->parent == NULL) root->right->parent = root; // 필요시 설정
    }
    return root;
}
```

**4. 삭제 (Delete) 연산**
삭제 연산은 세 가지 경우로 나뉩니다.
*   **리프 노드(Leaf Node) 삭제**: 단순히 노드를 제거합니다.
*   **하나의 자식을 가진 노드 삭제**: 삭제할 노드의 자식을 그 부모와 직접 연결합니다.
*   **두 개의 자식을 가진 노드 삭제**: 삭제할 노드의 오른쪽 서브트리에서 가장 작은 값(successor)을 찾거나, 왼쪽 서브트리에서 가장 큰 값(predecessor)을 찾아 삭제할 노드의 위치로 옮긴 후, 해당 successor/predecessor 노드를 원래 위치에서 삭제합니다. 이 중 successor를 찾는 방법이 일반적입니다.

**시간 복잡도 (Time Complexity)**

BST의 주요 연산(탐색, 삽입, 삭제)의 시간 복잡도는 트리의 높이($h$)에 비례합니다.
*   **최악의 경우 (Worst Case)**: 트리가 한쪽으로 치우쳐(skewed tree) 연결 리스트와 같은 형태가 될 때 발생합니다. 이 경우 높이 $h$는 노드 수 $N$과 같아집니다.
    $$ O(h) = O(N) $$
*   **평균의 경우 (Average Case)**: 트리가 비교적 균형 잡혀(balanced tree) 있을 때 발생합니다. 이 경우 높이 $h$는 $\log N$에 비례합니다.
    $$ O(h) = O(\log N) $$

### 구체적 예시

다음 숫자들을 순서대로 BST에 삽입하는 과정을 예시로 들어보겠습니다: `8, 3, 10, 1, 6, 14, 4, 7, 13`

1.  **8 삽입**: `8`이 루트가 됩니다.
    ```
      8
    ```
2.  **3 삽입**: `3 < 8`이므로 `8`의 왼쪽 자식으로 삽입됩니다.
    ```
      8
     /
    3
    ```
3.  **10 삽입**: `10 > 8`이므로 `8`의 오른쪽 자식으로 삽입됩니다.
    ```
      8
     / \
    3  10
    ```
4.  **1 삽입**: `1 < 8`, `1 < 3`이므로 `3`의 왼쪽 자식으로 삽입됩니다.
    ```
      8
     / \
    3  10
   /
  1
    ```
5.  **6 삽입**: `6 < 8`, `6 > 3`이므로 `3`의 오른쪽 자식으로 삽입됩니다.
    ```
      8
     / \
    3  10
   / \
  1   6
    ```
...이런 식으로 계속 진행됩니다.

**실생활 비유**:
BST는 도서관의 책 분류 시스템과 유사합니다. 예를 들어, ISBN 번호(또는 저자 이름의 알파벳 순서)를 기준으로 책을 정렬하여 보관한다고 상상해 보세요.
*   새로운 책을 삽입할 때: 서가에서 적절한 위치(작은 책은 왼쪽, 큰 책은 오른쪽)를 찾아 꽂아 넣습니다.
*   특정 책을 찾을 때: 루트(가운데 서가)에서 시작하여 찾으려는 책보다 번호가 작으면 왼쪽 서가로, 크면 오른쪽 서가로 이동하며 찾습니다. 이렇게 하면 모든 책을 다 뒤져보지 않고도 효율적으로 책을 찾을 수 있습니다.

### 시험 포인트

*   ⭐ **이진 탐색 트리의 정의 및 핵심 속성**: 왼쪽 서브트리 노드 값 < 현재 노드 값 < 오른쪽 서브트리 노드 값.
*   ⭐ **삽입, 삭제, 탐색 연산의 동작 방식**: 특히 두 자식을 가진 노드의 삭제 방법 (successor 또는 predecessor 사용)을 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **시간 복잡도 분석**:
    *   평균 경우 $O(\log N)$
    *   최악 경우 $O(N)$ (skewed tree일 때)
    *   최악의 경우를 피하기 위한 균형 탐색 트리(Balanced BST)의 필요성 (AVL 트리, 레드-블랙 트리 등으로 이어지는 개념)을 인지하는 것도 중요합니다.
*   ⭐ **BST의 장단점**: 빠른 탐색(평균), 동적 데이터 처리 가능 vs. 최악의 경우 성능 저하 가능성.
*   ⭐ **트리 순회(Traversal) 방법**: 전위(Pre-order), 중위(In-order), 후위(Post-order) 순회 방법을 BST에 적용했을 때의 결과. 특히 중위 순회 시 정렬된 순서로 노드들이 방문된다는 점은 매우 중요합니다.

---

### **핵심 개념**

*   **트리 (Tree)**: 계층적인 데이터를 표현하는 데 사용되는 비선형 자료구조입니다. 최상위 노드인 '루트(root)'를 시작으로, 각 노드는 0개 이상의 자식(child) 노드를 가질 수 있으며, 한 노드는 하나의 부모(parent) 노드만을 가집니다 (루트 노드 제외). 슬라이드에 보이는 나무 그림처럼 가지를 뻗어나가는 형태를 가집니다.
*   **이진 트리 (Binary Tree)**: 모든 노드가 최대 두 개의 자식 노드를 가질 수 있는 특별한 형태의 트리입니다. 이 두 자식 노드는 보통 '왼쪽 자식(left child)'과 '오른쪽 자식(right child)'으로 구분됩니다. 슬라이드의 나무가 0과 1의 이진수로 구성되어 있듯이, 컴퓨터 시스템 내부에서는 이러한 이진적인 구조가 매우 자주 활용됩니다. 자식이 없거나, 하나만 있거나, 두 개 모두 있을 수 있습니다.

### **코드/수식 해설**

이진 트리의 노드는 일반적으로 데이터와 두 개의 자식 포인터를 포함하는 구조체로 표현됩니다.

```c
// C 언어에서 이진 트리 노드를 정의하는 예시
typedef struct TreeNode {
    int data;                 // 노드가 저장할 데이터
    struct TreeNode *left;    // 왼쪽 자식 노드를 가리키는 포인터
    struct TreeNode *right;   // 오른쪽 자식 노드를 가리키는 포인터
} TreeNode;
```

이진 트리의 특성 중 하나는 높이(height)에 따른 노드 개수입니다.
*   높이가 $h$인 이진 트리가 가질 수 있는 **최대 노드 개수**: $N_{max} = 2^{h+1} - 1$
*   $N$개의 노드를 가지는 완전 이진 트리의 **최소 높이**: $h_{min} = \lfloor \log_2 N \rfloor$ (또는 $h_{min} = \lceil \log_2(N+1) \rceil - 1$)

### **구체적 예시**

*   **가족 관계도**: 실제 가족의 족보는 트리의 일종입니다. 한 부모 밑에 여러 자식이 있을 수 있으므로 일반적인 트리이지만, 특정 기준(예: 장남과 차남만 명시)으로 볼 때는 이진 트리의 개념을 적용할 수도 있습니다.
*   **이진 탐색 트리 (Binary Search Tree, BST)**: 데이터를 효율적으로 탐색하기 위한 이진 트리의 한 종류입니다. 특정 노드를 기준으로 왼쪽 서브트리에는 그 노드보다 작은 값들만, 오른쪽 서브트리에는 큰 값들만 저장됩니다. 예를 들어, 사전에서 단어를 찾을 때 중간 지점을 기준으로 왼쪽/오른쪽을 탐색해 나가는 방식과 유사합니다.
*   **컴퓨터 파일 시스템**: 디렉토리(폴더) 안에 다른 디렉토리나 파일이 계층적으로 저장되는 구조도 트리 자료구조의 좋은 예시입니다. 루트 디렉토리(/ 또는 C:\)가 트리의 루트가 됩니다.

### **시험 포인트**

*   ⭐ **트리 및 이진 트리의 정의와 기본 용어**: 루트, 부모, 자식, 형제, 조상, 후손, 리프(leaf), 서브트리, 깊이(depth), 높이(height), 차수(degree) 등 핵심 용어를 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **이진 트리의 종류**: 완전 이진 트리(Complete Binary Tree), 포화 이진 트리(Full Binary Tree), 편향 트리(Skewed Tree) 등 각 종류의 특성과 차이점을 숙지해야 합니다.
*   ⭐ **이진 트리의 순회(Traversal) 방법**: 전위 순회(Preorder Traversal), 중위 순회(Inorder Traversal), 후위 순회(Postorder Traversal)의 원리를 이해하고 주어진 트리에 대해 각 순회 결과를 도출할 수 있어야 합니다.
*   ⭐ **이진 트리의 기본 연산**: 노드의 삽입, 삭제, 탐색 연산의 과정을 이해하고 구현할 수 있어야 합니다. 특히 이진 탐색 트리의 삽입/탐색 과정은 중요합니다.
*   ⭐ **이진 트리의 활용**: 힙(Heap), 이진 탐색 트리(BST), AVL 트리, 레드-블랙 트리(Red-Black Tree) 등 이진 트리를 기반으로 하는 고급 자료구조의 기본 개념을 알아두는 것이 좋습니다.

### Binary Search Tree (BST)

-   **핵심 개념**:
    이 슬라이드는 이진 탐색 트리(Binary Search Tree, BST)의 기본 정의와 핵심 속성을 설명합니다.

    *   **이진 트리**: BST는 기본적으로 이진 트리입니다. 각 노드는 하나의 `(key, value)` 쌍을 가집니다. 여기서 `key`는 트리에 저장된 모든 노드에서 **유일(unique)** 해야 합니다.
    *   **BST 속성 (BST property)**: BST의 가장 중요한 특징은 `key`에 의해 노드들이 정렬되어 있다는 것입니다.
        *   모든 노드의 `key`는 **전체 순서(total order)** 관계를 가집니다. 즉, 서로 다른 두 키 `X`와 `Y`에 대해 항상 `X < Y`이거나 `X > Y` 둘 중 하나만 성립합니다.
        *   **핵심 정렬 규칙**: 특정 노드의 `key`는 해당 노드의 **왼쪽 서브트리에 있는 모든 키보다 커야 하며**, **오른쪽 서브트리에 있는 모든 키보다 작아야** 합니다. 이 규칙은 트리의 모든 노드에 재귀적으로 적용됩니다.
    *   **다른 이름**: 이진 탐색 트리는 때때로 `ordered/sorted binary tree`라고도 불립니다.

-   **코드/수식 해설**:
    해당 슬라이드에는 직접적인 코드나 수식 해설 내용이 없습니다. BST의 개념적 정의에 초점을 맞추고 있습니다.

-   **구체적 예시**:
    숫자 키를 사용하는 BST를 상상해봅시다.
    1.  루트 노드에 키 `10`이 있다고 가정해봅시다.
    2.  `10`의 왼쪽 서브트리에는 키 `10`보다 작은 모든 값들만 올 수 있습니다 (예: `5`, `3`, `7`).
    3.  `10`의 오른쪽 서브트리에는 키 `10`보다 큰 모든 값들만 올 수 있습니다 (예: `15`, `12`, `18`).
    4.  이 규칙은 재귀적으로 적용됩니다. 예를 들어, `10`의 왼쪽 자식인 `5`를 보면, `5`의 왼쪽 서브트리에는 `5`보다 작은 값(예: `3`)이, `5`의 오른쪽 서브트리에는 `5`보다 큰 값(예: `7`)이 위치할 것입니다.

    이러한 구조 덕분에 특정 값을 찾을 때마다 탐색 범위를 절반씩 줄여나갈 수 있어 효율적인 탐색이 가능합니다.

-   **시험 포인트**:
    *   BST의 **핵심 속성**을 정확하게 이해하고 설명할 수 있어야 합니다. (특정 노드의 키와 그 왼쪽/오른쪽 서브트리 키들의 관계) ⭐
    *   `key`가 **unique**하고 **total order** 관계를 가진다는 조건의 중요성. ⭐
    *   BST의 개념을 실제 트리 구조로 그리고, 어떤 키가 유효한 BST인지 아닌지 판단하는 문제. ⭐
    *   `ordered/sorted binary tree`와 같은 다른 이름도 숙지.

### 이진 탐색 트리 (Binary Search Tree, BST): 탐색 예시

---

**핵심 개념**:

*   **이진 탐색 트리 (BST)**:
    *   이진 트리(Binary Tree)의 한 종류로, 모든 노드 $N$에 대해 다음과 같은 속성을 만족하는 트리입니다.
        1.  $N$의 왼쪽 서브트리에 있는 모든 키(key)는 $N$의 키보다 작습니다.
        2.  $N$의 오른쪽 서브트리에 있는 모든 키는 $N$의 키보다 큽니다.
        3.  왼쪽 및 오른쪽 서브트리 또한 이진 탐색 트리입니다.
    *   이러한 속성 덕분에 효율적인 탐색, 삽입, 삭제 연산을 수행할 수 있습니다.
*   **탐색 (Search) 연산**:
    *   특정 키를 BST 내에서 찾는 연산입니다. 루트 노드부터 시작하여 탐색할 키와 현재 노드의 키를 비교하면서 트리를 내려갑니다.
    *   찾는 키가 현재 노드의 키보다 작으면 왼쪽 서브트리로, 크면 오른쪽 서브트리로 이동합니다.
    *   찾는 키와 현재 노드의 키가 같으면 탐색 성공, 해당 노드를 반환합니다.
    *   더 이상 이동할 자식 노드가 없는데 키를 찾지 못했다면 탐색 실패입니다.

---

**코드/수식 해설**:

BST 탐색 연산의 의사 코드 (재귀 방식):

```cpp
Node* Search(Node* root, int key) {
    // 1. 트리가 비었거나, 키를 찾은 경우
    if (root == nullptr || root->key == key) {
        return root; // root가 nullptr이면 찾지 못함, key와 같으면 해당 노드 반환
    }

    // 2. 찾는 키가 현재 노드의 키보다 작은 경우 (왼쪽 서브트리 탐색)
    if (key < root->key) {
        return Search(root->left, key);
    }
    // 3. 찾는 키가 현재 노드의 키보다 큰 경우 (오른쪽 서브트리 탐색)
    else { // key > root->key
        return Search(root->right, key);
    }
}
```

*   **시간 복잡도**:
    *   BST의 탐색 시간 복잡도는 트리의 높이($h$)에 비례합니다.
    *   최선 (Best Case): 트리가 완벽하게 균형 잡혀 있는 경우 (Balanced Tree), 높이 $h \approx \log N$ ($N$은 노드의 수).
        $O(\log N)$
    *   최악 (Worst Case): 트리가 한쪽으로 치우쳐 링크드 리스트와 유사한 형태인 경우 (Skewed Tree), 높이 $h \approx N$.
        $O(N)$
    *   평균 (Average Case): $O(\log N)$ (무작위 삽입을 가정할 경우)

---

**구체적 예시**:

주어진 BST 그림에서 `Search(8, D)` (여기서 `D`는 데이터베이스나 전체 트리를 의미) 연산 과정을 단계별로 살펴보겠습니다. 목표 키는 `8`입니다.

1.  **시작**: 루트 노드 `12`에서 탐색을 시작합니다.
    *   `8 < 12`이므로, 왼쪽 서브트리로 이동합니다.
2.  **두 번째 단계**: 현재 노드 `5`에 도착합니다.
    *   `8 > 5`이므로, 오른쪽 서브트리로 이동합니다.
3.  **세 번째 단계**: 현재 노드 `8`에 도착합니다.
    *   `8 == 8`이므로, 키를 찾았습니다.
4.  **결과**: 노드 `8`을 반환하고 탐색을 종료합니다.

만약 `Search(4, D)`를 수행한다면:
1.  루트 `12` (`4 < 12`) $\rightarrow$ 왼쪽으로 이동 (노드 `5`)
2.  노드 `5` (`4 < 5`) $\rightarrow$ 왼쪽으로 이동 (노드 `3`)
3.  노드 `3` (`4 > 3`) $\rightarrow$ 오른쪽으로 이동 (자식 노드가 없음)
4.  `nullptr`에 도달했으므로, 키 `4`는 트리에 존재하지 않습니다. 탐색 실패.

---

**시험 포인트**:

*   ⭐ **BST의 정의 및 속성**을 정확히 이해하고 설명할 수 있어야 합니다. (왼쪽 자식 < 부모 < 오른쪽 자식)
*   ⭐ **BST 탐색 (Search) 알고리즘의 동작 방식**을 주어진 트리 예시를 통해 설명할 수 있어야 합니다. 즉, 특정 키를 찾아가는 과정을 단계별로 추적하는 문제가 자주 출제됩니다.
*   ⭐ **BST 탐색의 시간 복잡도**를 최선, 최악, 평균 경우에 대해 정확히 알고 있어야 합니다. 특히 최악의 경우($O(N)$)와 최선의 경우($O(\log N)$)가 발생하는 트리의 형태(skewed vs. balanced)를 이해하는 것이 중요합니다.
*   BST는 이진 트리 중에서도 **탐색에 특화된 자료구조**라는 점을 기억하세요.

## BST: Search(x, D)

### 핵심 개념
이 슬라이드는 이진 탐색 트리(Binary Search Tree, BST)에서 특정 키 `x`를 탐색하는 `Search` 연산의 과정과 시간 복잡도를 설명합니다. BST는 노드의 키를 기준으로 정렬된 구조를 가지므로, 탐색 시 불필요한 노드를 건너뛸 수 있어 효율적입니다.

`Search` 연산은 다음 규칙에 따라 수행됩니다:
1.  현재 노드의 키와 찾고자 하는 키 `x`를 비교합니다.
2.  `x`가 현재 노드의 키와 같으면 탐색 성공입니다.
3.  `x`가 현재 노드의 키보다 작으면 왼쪽 서브트리로 이동하여 탐색을 계속합니다.
4.  `x`가 현재 노드의 키보다 크면 오른쪽 서브트리로 이동하여 탐색을 계속합니다.
5.  탐색 중 리프 노드에 도달했는데도 `x`를 찾지 못하면, `x`는 트리에 존재하지 않는 것입니다.

### 코드/수식 해설
BST 탐색 연산의 의사 코드는 다음과 같습니다 (재귀 방식):

```c
Node* search(Node* root, int key) {
    // 트리가 비어있거나 키를 찾았을 경우
    if (root == NULL || root->data == key) {
        return root;
    }

    // 키가 현재 노드보다 작으면 왼쪽 서브트리 탐색
    if (key < root->data) {
        return search(root->left, key);
    }
    // 키가 현재 노드보다 크면 오른쪽 서브트리 탐색
    else {
        return search(root->right, key);
    }
}
```

**시간 복잡도 분석**:
BST의 `Search` 연산의 시간 복잡도는 트리의 높이(height)에 비례합니다. 탐색은 항상 루트부터 시작하여 한 번에 하나의 경로를 따라 내려가므로, 최악의 경우 트리의 가장 깊은 노드까지 도달해야 할 수 있기 때문입니다.
$$
\text{Time Complexity} = O(\text{height})
$$

*   **최적의 경우 (Balanced BST)**: 만약 BST가 균형 트리(Balanced Tree, 예: AVL 트리, Red-Black 트리)라면, 높이는 노드 수 $n$에 대해 로그 스케일로 유지됩니다. 이 경우 시간 복잡도는 다음과 같습니다.
    $$
    O(\log n) \quad \text{if BST is balanced}
    $$
*   **최악의 경우 (Skewed BST)**: BST가 특정 방향으로만 노드가 추가되어 선형 리스트(Linear List) 형태로 기울어진다면 (예: 모든 노드가 부모보다 크거나 작은 값만 연속적으로 삽입), 트리의 높이는 노드 수 $n$과 같아질 수 있습니다. 이 경우 시간 복잡도는 다음과 같습니다.
    $$
    O(n) \quad \text{if BST is a linear list (worst case)}
    $$

### 구체적 예시
슬라이드에 제시된 BST에서 `Search(8, D)`를 수행하는 과정을 살펴보겠습니다.

1.  **시작**: 루트 노드 `12`에서 탐색을 시작합니다.
2.  **`12`와 `8` 비교**: `8 < 12`이므로, 왼쪽 자식으로 이동합니다. (경로: `12` ->)
3.  **`5`와 `8` 비교**: 현재 노드는 `5`입니다. `8 > 5`이므로, 오른쪽 자식으로 이동합니다. (경로: `12` -> `5` ->)
4.  **`8`와 `8` 비교**: 현재 노드는 `8`입니다. `8 == 8`이므로, 키를 찾았습니다. 탐색을 종료하고 노드 `8`을 반환합니다.

이 예시에서 탐색 경로는 `12` -> `5` -> `8`이며, 총 3개의 노드를 방문했습니다. 이는 트리의 높이에 비례하는 연산임을 보여줍니다.

### 시험 포인트
*   ⭐ **BST 탐색 원리**: `Search` 연산이 현재 노드의 키와 찾고자 하는 키를 비교하여 왼쪽 또는 오른쪽 서브트리로 이동하는 과정을 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **시간 복잡도 $O(height)$**: BST의 `Search`, `Insert`, `Delete` 등 대부분의 연산은 트리의 높이에 비례하는 시간 복잡도를 가집니다. 이를 반드시 기억해야 합니다.
*   ⭐ **균형/불균형 트리의 성능 차이**:
    *   BST가 균형 잡혀 있을 때 ($O(\log n)$)의 효율성과
    *   최악의 경우 (선형 리스트 형태, $O(n)$)로 퇴화했을 때의 비효율성을 명확히 구분하고, 왜 이런 차이가 발생하는지 설명할 수 있어야 합니다.
*   ⭐ **균형 트리의 필요성**: $O(n)$의 최악의 경우를 피하기 위해 AVL 트리나 Red-Black 트리와 같은 자가 균형 BST가 왜 필요한지 이해하는 것이 중요합니다.

## Slide 7

**핵심 개념**:
이 슬라이드는 이진 탐색 트리(Binary Search Tree, BST)에 저장된 모든 키(key)들을 오름차순으로 정렬하는 방법을 묻고 있습니다. BST는 모든 노드에 대해 왼쪽 서브트리의 모든 키는 해당 노드의 키보다 작고, 오른쪽 서브트리의 모든 키는 해당 노드의 키보다 큰 고유한 특성을 가집니다. 이 특성 덕분에 특별한 순회 방법으로 키들을 자동으로 정렬할 수 있습니다.

**코드/수식 해설**:
BST에서 키를 오름차순으로 정렬하는 가장 효율적인 방법은 **중위 순회(In-order Traversal)**를 수행하는 것입니다. 중위 순회는 재귀적으로 다음 단계를 따릅니다:
1.  왼쪽 자식 노드를 방문합니다.
2.  현재 노드의 키를 처리(예: 출력 또는 저장)합니다.
3.  오른쪽 자식 노드를 방문합니다.

이 순서대로 트리를 탐색하면 BST의 정의에 의해 키들이 항상 오름차순으로 처리됩니다.

C++ 또는 유사한 의사 코드:
```cpp
// BST 노드 구조체 정의 (예시)
struct Node {
    int key;
    Node* left;
    Node* right;

    Node(int val) : key(val), left(nullptr), right(nullptr) {}
};

// 중위 순회 함수 (오름차순으로 키를 출력)
void inorderTraversal(Node* root) {
    if (root == nullptr) {
        return;
    }
    inorderTraversal(root->left);  // 1. 왼쪽 서브트리 방문
    std::cout << root->key << " "; // 2. 현재 노드 키 처리 (출력)
    inorderTraversal(root->right); // 3. 오른쪽 서브트리 방문
}

// 사용 예시:
// Node* root = ... // 슬라이드에 제시된 BST를 생성합니다.
// inorderTraversal(root); // 모든 키가 오름차순으로 출력됩니다.
```

**시간 복잡도**:
트리에 $N$개의 노드가 있을 때, 중위 순회는 모든 노드를 정확히 한 번씩 방문하므로 시간 복잡도는 $O(N)$입니다.

$$
\text{Time Complexity: } O(N)
$$

**공간 복잡도**:
재귀 호출은 스택 메모리를 사용합니다. 스택의 최대 깊이는 트리의 높이 $H$에 비례합니다.
*   균형 잡힌 BST의 경우 $H = \log N$이므로, 공간 복잡도는 $O(\log N)$입니다.
*   최악의 경우 (예: 한쪽으로 편향된 트리) $H = N$이므로, 공간 복잡도는 $O(N)$이 됩니다.

$$
\text{Space Complexity: } O(H) \quad (\text{where } H \text{ is the height of the tree})
$$

**구체적 예시**:
슬라이드에 제시된 BST에 중위 순회를 적용하면 다음과 같은 순서로 키들이 처리됩니다:

```
inorderTraversal(12)
  inorderTraversal(5)
    inorderTraversal(3)
      inorderTraversal(nullptr) -> return
      print 3 -> Output: 3
      inorderTraversal(nullptr) -> return
    print 5 -> Output: 3 5
    inorderTraversal(8)
      inorderTraversal(6)
        inorderTraversal(nullptr) -> return
        print 6 -> Output: 3 5 6
        inorderTraversal(nullptr) -> return
      print 8 -> Output: 3 5 6 8
      inorderTraversal(nullptr) -> return
  print 12 -> Output: 3 5 6 8 12
  inorderTraversal(17)
    inorderTraversal(nullptr) -> return
    print 17 -> Output: 3 5 6 8 12 17
    inorderTraversal(20)
      inorderTraversal(19)
        inorderTraversal(nullptr) -> return
        print 19 -> Output: 3 5 6 8 12 17 19
        inorderTraversal(nullptr) -> return
      print 20 -> Output: 3 5 6 8 12 17 19 20
      inorderTraversal(nullptr) -> return
```
최종 결과는 `3 5 6 8 12 17 19 20`으로, 모든 키가 오름차순으로 정렬됩니다.

**시험 포인트**:
*   ⭐ **BST의 정의와 속성**: BST의 핵심 규칙(왼쪽 < 현재 < 오른쪽)을 정확히 이해하고 있어야 합니다. 이 규칙이 중위 순회를 통해 정렬된 결과를 얻을 수 있는 근거입니다.
*   ⭐ **트리 순회 종류**: 전위(Pre-order), 중위(In-order), 후위(Post-order) 순회의 차이점과 각 순회가 어떤 목적으로 사용되는지(특히 BST에서 중위 순회의 역할)를 숙지해야 합니다.
*   ⭐ **시간 및 공간 복잡도 분석**: BST에서 중위 순회를 통한 정렬의 시간 복잡도가 $O(N)$이며, 공간 복잡도는 트리의 균형 여부에 따라 $O(\log N)$ 또는 $O(N)$이 될 수 있음을 정확히 설명할 수 있어야 합니다.
*   ⭐ **일반 이진 트리와의 차이**: 만약 트리가 BST가 아니라 일반적인 이진 트리였다면, 단순히 중위 순회만으로는 정렬된 결과를 얻을 수 없으며, 모든 노드의 키를 추출하여 별도의 정렬 알고리즘(예: $O(N \log N)$)을 사용해야 한다는 점을 비교할 수 있어야 합니다.

## Slide 8

- **핵심 개념**:
  - **트리 순회 (Tree Traversal)**: 트리에 있는 모든 노드를 특정 순서로 방문하는 과정입니다. 이 슬라이드에서는 이진 탐색 트리(Binary Search Tree, BST)에서의 중위 순회(Inorder Traversal)를 다룹니다.
  - **이진 탐색 트리 (Binary Search Tree, BST)**: 각 노드의 왼쪽 서브트리에 있는 모든 노드의 값은 현재 노드의 값보다 작고, 오른쪽 서브트리에 있는 모든 노드의 값은 현재 노드의 값보다 큰 속성을 가진 이진 트리입니다. 중복된 값을 허용하지 않는 것이 일반적입니다.
  - **중위 순회 (Inorder Traversal)**: 트리를 순회하는 세 가지 주요 방법(전위 순회, 중위 순회, 후위 순회) 중 하나입니다. 중위 순회는 다음 순서로 노드를 방문합니다:
    1. 왼쪽 서브트리 (Left Subtree)를 재귀적으로 방문합니다.
    2. 현재 노드 (Root)를 방문합니다.
    3. 오른쪽 서브트리 (Right Subtree)를 재귀적으로 방문합니다.
  - ⭐ **BST의 중위 순회의 특징**: 이진 탐색 트리를 중위 순회하면 노드들이 오름차순으로 정렬된 순서로 방문됩니다. 이는 BST의 핵심적인 속성과 중위 순회의 특성이 결합된 결과이며, BST의 가장 중요한 응용 중 하나입니다.

- **코드/수식 해설**:
  - **중위 순회 재귀 함수 (C++ 예시)**:
    ```cpp
    struct Node {
        int data;
        Node* left;
        Node* right;
        // Constructor, etc.
    };

    void inorderTraversal(Node* root) {
        if (root == nullptr) { // 기저 조건: 노드가 없으면 반환
            return;
        }
        inorderTraversal(root->left);  // 1. 왼쪽 서브트리 재귀적으로 순회
        std::cout << root->data << " "; // 2. 현재 노드 데이터 방문 (출력)
        inorderTraversal(root->right); // 3. 오른쪽 서브트리 재귀적으로 순회
    }
    ```
  - **시간 복잡도**: 트리의 모든 노드를 한 번씩 방문합니다. 따라서 노드의 개수가 $N$개일 때, 각 노드 방문에 상수 시간이 소요되므로 전체 시간 복잡도는 $O(N)$입니다. 재귀 호출의 관점에서는 다음과 같이 표현할 수 있습니다:
    $$T(N) = T(N_L) + T(N_R) + C$$
    여기서 $N_L$은 왼쪽 서브트리의 노드 수, $N_R$은 오른쪽 서브트리의 노드 수, $C$는 현재 노드를 처리하는 상수 시간입니다. 일반적으로 $N = N_L + N_R + 1$ 입니다.

- **구체적 예시**:
  제시된 이진 탐색 트리 `T` (루트 `4`)에 대한 중위 순회 과정은 다음과 같습니다:
  1. `inorderTraversal(4)` 호출.
  2. `inorderTraversal(4->left)` 즉, `inorderTraversal(2)` 호출.
     a. `inorderTraversal(2->left)` 즉, `inorderTraversal(1)` 호출.
        i. `inorderTraversal(1->left)` (null) -> return.
        ii. `1` 출력. (현재까지 출력: `1`)
        iii. `inorderTraversal(1->right)` (null) -> return.
     b. `1` 방문 후 `2`로 돌아와 `2` 출력. (현재까지 출력: `1 2`)
     c. `inorderTraversal(2->right)` 즉, `inorderTraversal(3)` 호출.
        i. `inorderTraversal(3->left)` (null) -> return.
        ii. `3` 출력. (현재까지 출력: `1 2 3`)
        iii. `inorderTraversal(3->right)` (null) -> return.
  3. `1, 2, 3` 방문 후 `4`로 돌아와 `4` 출력. (현재까지 출력: `1 2 3 4`)
  4. `inorderTraversal(4->right)` 즉, `inorderTraversal(7)` 호출.
     a. `inorderTraversal(7->left)` 즉, `inorderTraversal(6)` 호출.
        i. `inorderTraversal(6->left)` 즉, `inorderTraversal(5)` 호출.
           1. `inorderTraversal(5->left)` (null) -> return.
           2. `5` 출력. (현재까지 출력: `1 2 3 4 5`)
           3. `inorderTraversal(5->right)` (null) -> return.
        ii. `5` 방문 후 `6`으로 돌아와 `6` 출력. (현재까지 출력: `1 2 3 4 5 6`)
        iii. `inorderTraversal(6->right)` (null) -> return.
     b. `5, 6` 방문 후 `7`로 돌아와 `7` 출력. (현재까지 출력: `1 2 3 4 5 6 7`)
     c. `inorderTraversal(7->right)` 즉, `inorderTraversal(8)` 호출.
        i. `inorderTraversal(8->left)` (null) -> return.
        ii. `8` 출력. (현재까지 출력: `1 2 3 4 5 6 7 8`)
        iii. `inorderTraversal(8->right)` (null) -> return.
  
  따라서 최종 중위 순회 결과는 슬라이드에 명시된 바와 같이 `Inorder(T) = 1, 2, 3, 4, 5, 6, 7, 8` 입니다.

- **시험 포인트**:
  - ⭐ **중위 순회의 정의와 재귀적 동작 방식**을 정확히 이해하고 설명할 수 있어야 합니다. (왼쪽 -> 루트 -> 오른쪽 순서)
  - ⭐ **주어진 이진 탐색 트리에 대한 중위 순회 결과**를 정확하게 도출할 수 있어야 합니다. 복잡한 트리에 대해서도 재귀 호출의 흐름을 추적하는 능력이 중요합니다.
  - ⭐ **이진 탐색 트리의 중위 순회가 항상 오름차순으로 정렬된 원소를 반환한다**는 핵심 속성을 반드시 기억해야 합니다. 이 속성은 BST의 근본적인 활용 목적과 직결됩니다.
  - ⭐ **트리 순회의 시간 복잡도**는 트리의 모든 노드를 한 번씩 방문하므로 항상 $O(N)$임을 이해해야 합니다.

### Quiz: Inorder Traversal을 통한 BST 재구성 가능성

-   **핵심 개념**:
    -   **이진 탐색 트리 (BST)**: 모든 노드에 대해 왼쪽 서브트리의 키는 현재 노드의 키보다 작고, 오른쪽 서브트리의 키는 현재 노드의 키보다 크거나 같습니다.
    -   **중위 순회 (Inorder Traversal)**: 왼쪽 서브트리 방문 -> 현재 노드 방문 -> 오른쪽 서브트리 방문 순서로 노드를 탐색합니다. BST의 중위 순회 결과는 항상 오름차순으로 정렬된 노드 키의 시퀀스를 얻습니다.
    -   슬라이드의 퀴즈는 *오직* BST의 중위 순회 결과만 주어졌을 때, 원래의 BST 구조를 *유일하게* 재구성할 수 있는지를 묻습니다.

-   **코드/수식 해설**:
    주어진 중위 순회 결과만으로는 트리를 유일하게 재구성할 수 없습니다. 이는 동일한 중위 순회 결과를 생성하는 여러 가지 다른 BST 구조가 존재하기 때문입니다. 중위 순회는 요소들의 상대적인 순서만 알려줄 뿐, 어떤 요소가 루트가 되고 어떤 요소가 자식 노드가 되는지에 대한 구조적인 정보는 충분히 제공하지 못합니다.

    예를 들어, 노드의 개수가 $N$개일 때, $N$개의 노드를 가질 수 있는 유일한 중위 순회 리스트로부터 생성될 수 있는 이진 탐색 트리의 개수는 Catalan Number $C_N$으로 주어집니다.
    $$
    C_N = \frac{1}{N+1}\binom{2N}{N}
    $$
    $N=3$일 때 $C_3 = \frac{1}{4}\binom{6}{3} = \frac{1}{4} \times \frac{6 \times 5 \times 4}{3 \times 2 \times 1} = \frac{20}{4} = 5$ 이므로, 3개의 노드를 가지는 중위 순회 리스트에서는 5가지의 서로 다른 BST 구조가 가능합니다. 따라서 유일하게 재구성할 수 없습니다.

-   **구체적 예시**:
    중위 순회 결과가 `[1, 2, 3]`이라고 가정해 봅시다.
    이 중위 순회 결과를 가질 수 있는 몇 가지 다른 BST 구조는 다음과 같습니다:

    1.  **균형 잡힌 트리**:
        ```
              2
             / \
            1   3
        ```
        (중위 순회: 1 -> 2 -> 3)

    2.  **오른쪽으로 치우친 트리 (right-skewed)**:
        ```
            1
             \
              2
               \
                3
        ```
        (중위 순회: 1 -> 2 -> 3)

    3.  **왼쪽으로 치우친 트리 (left-skewed)**:
        ```
            3
           /
          2
         /
        1
        ```
        (중위 순회: 1 -> 2 -> 3)

    위 세 가지 예시 모두 동일한 중위 순회 결과 `[1, 2, 3]`을 가집니다. 하지만 이들은 서로 다른 트리 구조를 가집니다. 따라서 오직 중위 순회 결과만으로는 원래의 트리 구조를 유일하게 재구성할 수 없습니다.

-   **시험 포인트**:
    -   ⭐ **결론**: BST의 중위 순회(inorder traversal)만으로는 원래 트리 구조를 유일하게 재구성할 수 없습니다. (정답: No)
    -   ⭐ **이유**: 동일한 중위 순회 결과를 갖는 여러 BST 구조가 존재하기 때문입니다. 중위 순회는 노드의 상대적인 순서만 제공하며, 어떤 노드가 루트인지, 각 서브트리의 구조가 어떻게 형성되는지에 대한 충분한 정보를 주지 못합니다.
    -   **추가 정보**:
        -   이진 트리를 유일하게 재구성하기 위해서는 일반적으로 **두 가지 순회 결과**가 필요합니다.
        -   가장 일반적인 조합은 **중위 순회 (inorder traversal)**와 **전위 순회 (preorder traversal)**입니다. (전위 순회는 항상 루트를 먼저 방문하므로, 이를 통해 트리의 루트를 식별하고 중위 순회로 좌우 서브트리를 구분할 수 있습니다.)
        -   **중위 순회 (inorder traversal)**와 **후위 순회 (postorder traversal)** 조합으로도 유일하게 재구성할 수 있습니다. (후위 순회는 항상 루트를 마지막에 방문합니다.)
        -   전위 순회와 후위 순회만으로는 유일하게 재구성할 수 없는 경우가 존재합니다 (예: 한쪽 자식만 있는 노드가 있는 경우).

## Slide 10

**핵심 개념**
*   **이진 탐색 트리 (BST)의 Inorder 순회**: 이진 탐색 트리의 Inorder 순회 (Left -> Root -> Right)는 트리의 모든 노드를 오름차순으로 정렬된 형태로 방문합니다. 이는 BST의 핵심 속성인 '왼쪽 자식 노드 < 부모 노드 < 오른쪽 자식 노드'에 의해 보장됩니다.
*   **트리 구조 복원 문제**: 주어진 순회 결과(여기서는 Inorder 순회)만으로 원래의 이진 탐색 트리 구조를 유일하게 다시 만들 수 있는지에 대한 질문입니다.

**코드/수식 해설**
Inorder 순회 자체는 간단한 재귀 함수로 구현됩니다.
```c
void inorderTraversal(Node* root) {
    if (root == NULL) {
        return;
    }
    inorderTraversal(root->left);  // 왼쪽 서브트리 순회
    printf("%d ", root->data);     // 현재 노드(root) 방문
    inorderTraversal(root->right); // 오른쪽 서브트리 순회
}
```
BST의 경우, 이 함수를 실행하면 `root->data`가 항상 정렬된 순서로 출력됩니다. 예를 들어, 노드 값이 1, 2, 3인 BST라면 Inorder 순회 결과는 항상 `1 2 3`이 됩니다.

**구체적 예시**
Inorder 순회 결과가 `[1, 2, 3]`으로 주어졌다고 가정해 봅시다. 이 순회 결과만으로는 다음 두 가지 이상의 서로 다른 BST를 구분하여 복원할 수 없습니다.

1.  **우측으로 치우친 트리 (Right-skewed tree)**:
    ```
      1
       \
        2
         \
          3
    ```
    이 트리의 Inorder 순회 결과는 `1 -> 2 -> 3` 입니다.

2.  **균형 잡힌 트리**:
    ```
        2
       / \
      1   3
    ```
    이 트리의 Inorder 순회 결과 역시 `1 -> 2 -> 3` 입니다.

위 예시들처럼, 동일한 Inorder 순회 결과에 대해 여러 가지 다른 트리 구조가 존재할 수 있습니다. 이는 Inorder 순회 결과가 노드들의 상대적인 순서만 알려줄 뿐, 어떤 노드가 부모-자식 관계를 가지는지, 즉 트리의 '계층적 구조'에 대한 정보를 충분히 제공하지 못하기 때문입니다.

**시험 포인트**
*   ⭐ **Inorder 순회만으로는 BST를 유일하게 재구성할 수 없습니다.** (정답: No)
*   ⭐ **BST를 유일하게 재구성하기 위해서는 Inorder 순회 외에 `Preorder 순회` 또는 `Postorder 순회` 중 하나가 추가로 필요합니다.** (정답: Preorder traversal or Postorder traversal)
    *   **Preorder (Root -> Left -> Right)**: Preorder 순회 결과의 첫 번째 요소는 항상 전체 트리의 Root 노드입니다. 이 Root 노드의 값을 Inorder 순회 결과에서 찾으면, Inorder 배열이 Root를 기준으로 왼쪽과 오른쪽 서브트리에 속하는 노드들로 나뉘어지게 됩니다. 이 정보를 사용하여 왼쪽 서브트리와 오른쪽 서브트리를 재귀적으로 구성할 수 있습니다.
    *   **Postorder (Left -> Right -> Root)**: Postorder 순회 결과의 마지막 요소는 항상 전체 트리의 Root 노드입니다. Preorder와 유사하게, Inorder 배열에서 이 Root를 기준으로 서브트리를 분할하고 재귀적으로 구성할 수 있습니다.
*   일반적인 이진 트리(BST가 아닌)의 경우에도, 트리를 유일하게 복원하기 위해서는 두 가지 다른 순회 결과가 필요하며, 그 중 하나는 반드시 Inorder 순회여야 합니다. (예외적으로 Full Binary Tree와 같이 특별한 속성을 가진 트리는 단일 순회로 복원이 가능할 수도 있지만, 일반적인 경우에 해당하지 않습니다.)

## Slide 11

### **핵심 개념**
이 슬라이드는 이진 트리 (Binary Tree)의 순회(Traversal) 결과를 이용해 원래의 트리를 재구성(Reconstruction)하는 알고리즘에 대한 질문을 담고 있습니다. 특히, Inorder Traversal(중위 순회)과 Preorder Traversal(전위 순회)이 주어졌을 때 트리를 재구성하는 방법과, 이 방법이 이진 탐색 트리(BST)의 특정 속성에 의존하는지 여부를 묻습니다.

*   **Inorder Traversal (중위 순회)**: 왼쪽 자식 -> 루트 -> 오른쪽 자식 순서로 노드를 방문합니다. 이 순회는 주어진 서브트리에서 루트 노드를 기준으로 왼쪽 서브트리 노드들과 오른쪽 서브트리 노드들을 명확하게 구분해주는 중요한 정보를 제공합니다.
*   **Preorder Traversal (전위 순회)**: 루트 -> 왼쪽 자식 -> 오른쪽 자식 순서로 노드를 방문합니다. 이 순회는 항상 현재 처리하는 서브트리의 루트 노드를 가장 먼저 알려줍니다.
*   **Tree Reconstruction**: 두 가지 종류의 순회 (예: Inorder + Preorder, Inorder + Postorder)가 주어졌을 때, 해당 이진 트리의 고유한 구조를 복원하는 과정입니다. 이때 모든 노드의 키는 중복되지 않아야 합니다.

### **코드/수식 해설**
트리 재구성 알고리즘은 일반적으로 재귀적으로 구현됩니다. Preorder 순회에서 가장 첫 번째 원소는 현재 처리할 서브트리의 루트 노드입니다. 이 루트 노드를 Inorder 순회에서 찾으면, Inorder 순회에서 해당 루트 노드의 왼쪽에 있는 원소들은 모두 왼쪽 서브트리를 구성하고, 오른쪽에 있는 원소들은 모두 오른쪽 서브트리를 구성합니다.

```cpp
// 가상의 트리 노드 구조체
struct TreeNode {
    int val;
    TreeNode* left;
    TreeNode* right;
    TreeNode(int x) : val(x), left(nullptr), right(nullptr) {}
};

// 재귀적인 트리 재구성 함수 (C++ 예시)
// preorder: 전위 순회 배열 (전체)
// inorder: 중위 순회 배열 (전체)
// preStart, preEnd: preorder 배열의 현재 구간 (재구성할 서브트리)
// inStart, inEnd: inorder 배열의 현재 구간 (재구성할 서브트리)
// inorderMap: inorder 배열에서 값의 인덱스를 빠르게 찾기 위한 맵 (값 -> 인덱스)
TreeNode* buildTree(const std::vector<int>& preorder,
                    int preStart, int preEnd,
                    const std::vector<int>& inorder,
                    int inStart, int inEnd,
                    const std::unordered_map<int, int>& inorderMap) {
    // 기저 조건: 유효하지 않은 구간 (빈 서브트리)
    if (preStart > preEnd || inStart > inEnd) {
        return nullptr;
    }

    // preorder의 첫 번째 원소가 현재 서브트리의 루트
    int rootVal = preorder[preStart];
    TreeNode* root = new TreeNode(rootVal);

    // inorder에서 루트의 위치를 찾음
    // 이 위치를 기준으로 왼쪽/오른쪽 서브트리를 분할
    int rootInInorder = inorderMap.at(rootVal);

    // inorder에서 루트의 왼쪽에 있는 노드들의 개수
    // 이 개수는 왼쪽 서브트리를 구성하는 노드들의 개수와 같다
    int numsLeft = rootInInorder - inStart;

    // 왼쪽 서브트리 재귀 호출
    // preorder: preStart+1 (현재 루트 다음) ~ preStart+numsLeft (왼쪽 서브트리 끝)
    // inorder: inStart ~ rootInInorder-1 (왼쪽 서브트리 끝)
    root->left = buildTree(preorder, preStart + 1, preStart + numsLeft,
                           inorder, inStart, rootInInorder - 1, inorderMap);

    // 오른쪽 서브트리 재귀 호출
    // preorder: preStart+numsLeft+1 (왼쪽 서브트리 다음) ~ preEnd
    // inorder: rootInInorder+1 (오른쪽 서브트리 시작) ~ inEnd
    root->right = buildTree(preorder, preStart + numsLeft + 1, preEnd,
                            inorder, rootInInorder + 1, inEnd, inorderMap);

    return root;
}
```
**시간 복잡도**: `inorderMap`을 사용하여 Inorder 배열에서 루트의 위치를 $O(1)$에 찾을 수 있으므로, 각 노드에 대해 일정한 작업이 수행됩니다. 따라서 전체 $N$개의 노드를 처리하는 데 걸리는 시간 복잡도는 $O(N)$이 됩니다.

### **구체적 예시**
Inorder: `[D, B, E, A, F, C, G]`
Preorder: `[A, B, D, E, C, F, G]`

1.  **Preorder의 첫 번째 원소 `A`가 루트입니다.**
2.  Inorder에서 `A`를 찾습니다. `[D, B, E]` (왼쪽 서브트리), `A` (루트), `[F, C, G]` (오른쪽 서브트리).
    *   왼쪽 서브트리의 노드들: `D, B, E`
    *   오른쪽 서브트리의 노드들: `F, C, G`
3.  Preorder에서 `A`를 제외한 `[B, D, E, C, F, G]`를 왼쪽과 오른쪽 서브트리의 Preorder로 분할합니다.
    *   왼쪽 서브트리의 크기는 3이므로, `[B, D, E]`가 왼쪽 서브트리의 Preorder가 됩니다.
    *   나머지 `[C, F, G]`가 오른쪽 서브트리의 Preorder가 됩니다.

이제 각 서브트리에 대해 재귀적으로 동일한 과정을 반복합니다:

*   **왼쪽 서브트리 (`[D, B, E]` in Inorder, `[B, D, E]` in Preorder)**:
    *   `B`가 루트. Inorder에서 `B`를 찾으면 `[D]` (왼쪽), `B` (루트), `[E]` (오른쪽).
    *   왼쪽 자식: `D` (Preorder: `[D]`, Inorder: `[D]`)
    *   오른쪽 자식: `E` (Preorder: `[E]`, Inorder: `[E]`)
*   **오른쪽 서브트리 (`[F, C, G]` in Inorder, `[C, F, G]` in Preorder)**:
    *   `C`가 루트. Inorder에서 `C`를 찾으면 `[F]` (왼쪽), `C` (루트), `[G]` (오른쪽).
    *   왼쪽 자식: `F` (Preorder: `[F]`, Inorder: `[F]`)
    *   오른쪽 자식: `G` (Preorder: `[G]`, Inorder: `[G]`)

최종 재구성된 트리는 다음과 같습니다:
```
      A
     / \
    B   C
   / \ / \
  D  E F  G
```

**질문에 대한 답변**:
재구성 알고리즘은 **이진 탐색 트리(BST)의 속성(왼쪽 자식 < 루트 < 오른쪽 자식)**에 근본적으로 의존하지 않습니다. 위에서 설명한 알고리즘은 단지 `preorder`의 첫 번째 원소가 루트이고, `inorder`에서 해당 루트를 기준으로 왼쪽과 오른쪽 서브트리의 원소들이 나뉜다는 **구조적 정보**만을 사용합니다. 노드의 값 크기 관계를 비교하는 BST 속성은 트리 순회 결과에 특정한 순서(예: inorder 순회 결과가 정렬된 상태)를 가져다주지만, 주어진 순회 결과로부터 트리를 재구성하는 로직 자체는 그 속성을 직접 활용하지 않습니다. 따라서 이 알고리즘은 **임의의 이진 트리(arbitrary binary tree)에도 동일하게 적용**됩니다.

### **시험 포인트**
*   ⭐ **Inorder + Preorder (또는 Postorder) 순회로 이진 트리 재구성 원리**: Preorder/Postorder는 루트를 식별하고, Inorder는 서브트리를 분할하는 역할을 한다는 것을 이해하고 설명할 수 있어야 합니다.
*   ⭐ **BST 속성과의 독립성**: 트리 재구성 알고리즘이 BST의 속성(값의 정렬 규칙)에 의존하지 않고, 임의의 이진 트리에 대해서도 작동한다는 점을 명확히 설명할 수 있어야 합니다. 이는 재구성 알고리즘이 순회에 의해 제공되는 `구조적 관계`만을 활용하기 때문입니다.
*   ⭐ **재구성 과정의 재귀적 특성**: 서브트리에 대해 동일한 과정을 반복하는 재귀적인 사고방식과 이를 코드로 구현하는 방법을 알아야 합니다.
*   ⭐ **효율적인 구현**: `inorderMap`과 같은 보조 자료구조를 활용하여 Inorder 배열 내에서 루트 노드의 위치를 $O(1)$ 시간에 찾아 $O(N)$의 시간 복잡도로 트리를 재구성할 수 있음을 이해해야 합니다. (`inorderMap`을 사용하지 않으면, Inorder 배열에서 루트를 찾는데 매번 $O(N)$이 걸려 최악의 경우 $O(N^2)$이 될 수 있습니다.)

## Slide 12

**핵심 개념**

*   **이진 탐색 트리 (Binary Search Tree, BST)**: 이진 트리 중 하나로, 다음 세 가지 속성을 만족합니다:
    1.  각 노드의 왼쪽 서브트리에 있는 모든 원소의 키는 해당 노드의 키보다 작습니다.
    2.  각 노드의 오른쪽 서브트리에 있는 모든 원소의 키는 해당 노드의 키보다 큽니다.
    3.  왼쪽 및 오른쪽 서브트리도 이진 탐색 트리입니다.
    4.  (일반적으로) 중복된 키는 허용하지 않습니다.
*   **BST 삽입 (Insert)**: 새로운 데이터를 BST에 추가하는 연산입니다. BST의 속성을 유지하면서 적절한 위치에 새 노드를 삽입해야 합니다.

**코드/수식 해설**

BST에 새 값 `x`를 삽입하는 과정은 재귀적 또는 반복적으로 구현될 수 있습니다. 기본적인 아이디어는 다음과 같습니다:

1.  트리의 루트에서 시작합니다.
2.  현재 노드의 키와 삽입할 값 `x`를 비교합니다.
    *   `x`가 현재 노드의 키보다 작으면 왼쪽 서브트리로 이동합니다.
    *   `x`가 현재 노드의 키보다 크면 오른쪽 서브트리로 이동합니다.
    *   (만약 `x`가 현재 노드의 키와 같다면, 중복을 허용하지 않는 BST에서는 삽입하지 않거나 오류를 반환합니다.)
3.  이 과정을 `NULL` 포인터를 만날 때까지 반복합니다. `NULL` 포인터를 만나는 지점이 `x`를 삽입할 새로운 노드의 위치가 됩니다.

다음은 삽입 연산의 재귀적 의사 코드입니다:

```cpp
// BST 노드 구조체 (예시)
struct Node {
    int key;
    Node* left;
    Node* right;
    Node(int val) : key(val), left(nullptr), right(nullptr) {}
};

// BST에 새 값(data)을 삽입하는 함수
Node* insert(Node* root, int data) {
    // 트리가 비어있거나, 삽입할 위치를 찾은 경우 (새 노드 생성)
    if (root == nullptr) {
        return new Node(data);
    }

    // 삽입할 값이 현재 노드의 키보다 작은 경우 왼쪽 서브트리로 이동
    if (data < root->key) {
        root->left = insert(root->left, data);
    }
    // 삽입할 값이 현재 노드의 키보다 큰 경우 오른쪽 서브트리로 이동
    else if (data > root->key) { // 중복 키는 허용하지 않는다고 가정
        root->right = insert(root->right, data);
    }
    // else if (data == root->key) { // 중복을 허용하지 않는 경우 처리 }

    return root; // 변경된 서브트리의 루트 반환
}
```

삽입 연산의 시간 복잡도는 트리의 높이($h$)에 비례합니다.
$$ T(n) = O(h) $$
*   완전히 균형 잡힌 트리의 경우 $h = \log_2 n$이므로 $O(\log n)$입니다.
*   편향된 트리의 경우 $h = n$이므로 최악의 경우 $O(n)$입니다.

**구체적 예시**

주어진 BST에 `13`을 삽입하는 과정을 단계별로 살펴보겠습니다.

현재 BST 구조:
```
      12
     /  \
    5    17
   / \   / \
  3   8 NULL 20
     /     /
    6     19
```

1.  **시작**: 루트 노드 `12`에서 시작합니다.
    *   삽입할 값 `13`은 `12`보다 큽니다. 따라서 `12`의 오른쪽 자식으로 이동합니다.
2.  **현재 노드 `17`**:
    *   삽입할 값 `13`은 `17`보다 작습니다. 따라서 `17`의 왼쪽 자식으로 이동합니다.
3.  **현재 노드 `17`의 왼쪽 자식**: `NULL`입니다.
    *   `NULL`을 만났으므로, 여기에 `13`을 새로운 노드로 삽입합니다.

**삽입 후 BST 구조**:
```
      12
     /  \
    5    17
   / \   / \
  3   8  13  20
     /     /
    6     19
```
새로운 노드 `13`이 `17`의 왼쪽 자식으로 성공적으로 삽입되었습니다.

**시험 포인트**

*   ⭐ **BST의 핵심 속성**: 왼쪽 자식 < 부모 노드 < 오른쪽 자식 관계를 항상 기억해야 합니다.
*   ⭐ **삽입 과정**: 주어진 BST에 특정 값을 삽입하는 과정을 정확히 추적하고, 최종 결과 트리를 그릴 수 있어야 합니다. 이는 코딩 테스트나 서술형 문제에서 자주 출제됩니다.
*   ⭐ **시간 복잡도**: BST 삽입의 평균 및 최악 시간 복잡도($O(\log n)$와 $O(n)$)를 이해하고 설명할 수 있어야 합니다. 이는 트리의 높이에 따라 달라집니다.
*   ⭐ **재귀적 사고**: 삽입 연산은 재귀적으로 생각하기 좋은 문제입니다. 재귀 호출 스택의 동작을 이해하는 것이 중요합니다.

### BST: Insert(x, D)

---

-   **핵심 개념**:
    이 슬라이드는 이진 탐색 트리(BST: Binary Search Tree)에 새로운 노드 `(x, D)`를 삽입하는 과정을 설명합니다. 여기서 `x`는 키(key), `D`는 데이터(data)를 의미합니다. BST는 다음 세 가지 중요한 속성을 만족해야 합니다:
    1.  모든 노드의 키는 유일(unique)합니다. (강의에 따라 중복을 허용하는 경우도 있으나, 일반적으로 유일하다고 가정합니다.)
    2.  노드의 왼쪽 서브트리에 있는 모든 노드의 키는 해당 노드의 키보다 작습니다.
    3.  노드의 오른쪽 서브트리에 있는 모든 노드의 키는 해당 노드의 키보다 큽니다.
    삽입 연산은 이 BST 속성을 유지하면서 새로운 노드를 트리의 적절한 위치에 추가하는 과정입니다.

-   **코드/수식 해설**:
    BST 삽입 알고리즘은 새 키 `x`가 들어갈 적절한 위치(말단 노드의 자식 위치)를 찾을 때까지 트리를 순회하는 방식으로 동작합니다.

    ```c
    // 이진 탐색 트리의 노드 구조체 (간소화)
    typedef struct Node {
        int key;
        void* data; // 실제 데이터
        struct Node* left;
        struct Node* right;
    } Node;

    // 새로운 노드를 생성하는 함수 (생략)
    Node* createNode(int key, void* data);

    // BST에 새로운 키와 데이터를 삽입하는 함수
    Node* insert(Node* root, int key, void* data) {
        // 1. 트리가 비어있으면 새로운 노드를 루트로 만듭니다.
        if (root == NULL) {
            return createNode(key, data);
        }

        // 2. 키를 현재 노드의 키와 비교합니다.
        if (key < root->key) {
            // 새 키가 현재 노드의 키보다 작으면 왼쪽 서브트리로 이동합니다.
            root->left = insert(root->left, key, data);
        } else if (key > root->key) {
            // 새 키가 현재 노드의 키보다 크면 오른쪽 서브트리로 이동합니다.
            root->right = insert(root->right, key, data);
        }
        // else if (key == root->key) {
        //     // 키가 이미 존재하면 삽입하지 않거나 데이터만 업데이트할 수 있습니다.
        //     // 여기서는 무시한다고 가정합니다.
        // }

        // 3. (재귀 호출 후) 현재 서브트리의 루트를 반환합니다.
        return root;
    }
    ```

    삽입 연산의 시간 복잡도는 새 노드가 삽입될 위치를 찾기 위해 트리의 높이만큼 노드를 탐색해야 하므로, 트리의 높이 $h$에 비례합니다.
    $$ O(h) $$
    *   최악의 경우 (트리가 한쪽으로 치우쳐진 경우): $h = n$ (여기서 $n$은 트리의 노드 수)이므로, 시간 복잡도는 $O(n)$이 됩니다.
    *   최적의 경우 (트리가 균형 잡힌 경우): $h = \log_2 n$이므로, 시간 복잡도는 $O(\log n)$이 됩니다.

-   **구체적 예시**:

    **현재 BST 상태**:
    ```
            12
           /  \
          5    17
         / \   /  \
        3   8 13   20
            /      /
           6      19
    ```

    1.  **`Insert(13, D)`**: (슬라이드에 표시된 경로)
        *   루트 노드 `12`에서 시작합니다. `13 > 12`이므로 오른쪽 자식 `17`로 이동합니다. (슬라이드에서 `12`에서 `17`로 가는 빨간 선)
        *   노드 `17`에 도착합니다. `13 < 17`이므로 왼쪽 자식으로 이동합니다. (슬라이드에서 `17`에서 `13`으로 가는 빨간 선)
        *   노드 `17`의 왼쪽 자식이 현재 `NULL`입니다. 여기에 `13`을 새로운 노드로 삽입합니다. (슬라이드에서 분홍색으로 강조된 `13` 노드)
        *   결과적으로 `17`의 왼쪽 자식으로 `13` 노드가 연결됩니다.

    2.  **`Insert(18, D)`**: (슬라이드에서 다음으로 삽입할 값)
        *   루트 노드 `12`에서 시작합니다. `18 > 12`이므로 오른쪽 자식 `17`로 이동합니다.
        *   노드 `17`에 도착합니다. `18 > 17`이므로 오른쪽 자식 `20`으로 이동합니다.
        *   노드 `20`에 도착합니다. `18 < 20`이므로 왼쪽 자식 `19`로 이동합니다.
        *   노드 `19`에 도착합니다. `18 < 19`이므로 왼쪽 자식으로 이동합니다.
        *   노드 `19`의 왼쪽 자식이 현재 `NULL`입니다. 여기에 `18`을 새로운 노드로 삽입합니다.

        삽입 후 예상되는 BST 상태 (노드 `18`은 `19`의 왼쪽 자식이 됨):
        ```
                12
               /  \
              5    17
             / \   /  \
            3   8 13   20
                /      /
               6      19
                     /
                    18
        ```

-   **시험 포인트**:
    *   ⭐**BST 속성**: 왼쪽 자식 < 부모 < 오른쪽 자식 규칙을 정확히 이해하고 설명할 수 있어야 합니다.
    *   ⭐**삽입 과정**: 주어진 BST에 특정 값을 삽입할 때, 올바른 탐색 경로를 따라가서 정확한 위치에 삽입하는 과정을 단계별로 시뮬레이션할 수 있어야 합니다. (예: `Insert(18, D)`와 같은 문제)
    *   ⭐**시간 복잡도**: BST 삽입의 시간 복잡도 $O(h)$와 최악/최적/평균 경우의 시나리오 ($O(n)$ vs $O(\log n)$)를 이해하고 설명할 수 있어야 합니다.
    *   ⭐재귀 또는 반복문을 이용한 BST 삽입 함수의 구현 방식을 파악하는 것도 중요합니다.

## Slide 14

**핵심 개념**:
이 슬라이드는 **이진 탐색 트리 (Binary Search Tree, BST)**에 새로운 요소를 삽입하는 `Insert(x, D)` 연산에 대한 퀴즈를 제시합니다.
*   **이진 탐색 트리(BST)**는 다음 세 가지 주요 속성을 만족하는 이진 트리입니다:
    1.  모든 노드의 왼쪽 서브트리에 있는 값은 해당 노드의 값보다 작습니다.
    2.  모든 노드의 오른쪽 서브트리에 있는 값은 해당 노드의 값보다 큽니다.
    3.  왼쪽 및 오른쪽 서브트리 또한 각각 이진 탐색 트리입니다.
*   `Insert(x, D)` 연산은 새로운 값 `x`를 BST `D`에 삽입하는 과정으로, 이진 탐색 트리의 속성을 유지하면서 `x`가 들어갈 적절한 위치(대부분 리프 노드의 자식)를 찾아 새 노드를 추가합니다.

**코드/수식 해설**:
BST에서 `Insert` 연산은 트리를 순회하며 새 노드가 삽입될 위치를 탐색하는 방식으로 이루어집니다. 재귀적으로 구현될 때의 핵심 로직은 다음과 같습니다.

```cpp
// BST Node 구조체 (예시)
struct Node {
    int key;
    Node* left;
    Node* right;
    Node(int val) : key(val), left(nullptr), right(nullptr) {}
};

// Insert 연산 (재귀 함수)
Node* insert(Node* root, int key) {
    // 1. 현재 노드가 nullptr이면, 새 노드를 생성하여 반환 (삽입 위치 발견)
    if (root == nullptr) {
        return new Node(key);
    }

    // 2. 삽입할 키와 현재 노드의 키를 비교하여 적절한 서브트리로 이동
    if (key < root->key) {
        root->left = insert(root->left, key); // 왼쪽 서브트리로 재귀 호출
    } else if (key > root->key) {
        root->right = insert(root->right, key); // 오른쪽 서브트리로 재귀 호출
    }
    // else if (key == root->key) { /* 중복 처리: 무시, 카운트 증가 등 (구현에 따라 다름) */ }

    // 3. (재귀 호출 후) 현재 노드의 주소를 반환하여 부모 노드의 자식 포인터가 업데이트되도록 함
    return root;
}
```

**시간 복잡도**:
BST의 `Insert` 연산은 루트에서부터 리프 노드까지 경로를 탐색하는 데 걸리는 시간에 비례합니다. 이 경로는 트리의 높이($h$)와 같습니다. 따라서 시간 복잡도는 $O(h)$입니다.
*   **최선(Best Case)**: 트리가 거의 완벽하게 균형 잡혀 있을 때 ($h = \log N$, 여기서 $N$은 노드의 수)
    $$ O(\log N) $$
*   **최악(Worst Case)**: 트리가 한쪽으로 완전히 치우쳐 링크드 리스트와 같은 형태일 때 ($h = N$)
    $$ O(N) $$
*   **평균(Average Case)**: 무작위 순서로 삽입이 이루어질 경우, 트리는 대체로 균형을 유지하며 높이는 약 $O(\log N)$이 됩니다.
    $$ O(\log N) $$

슬라이드에서 묻는 "Time complexity = ?"는 일반적으로 최악의 경우를 포함하는 일반적인 시간 복잡도를 묻는 것이므로, $O(h)$ 또는 최악의 경우를 명시하여 $O(N)$으로 답할 수 있습니다.

**구체적 예시**:
슬라이드에 제시된 초기 BST (13과 18이 삽입되기 전을 가정)는 다음과 같다고 추정합니다:
```
      12
     /  \
    5    17
   / \     \
  3   8     20
     /      /
    6      19
```

1.  **`Insert(13, D)` 과정**:
    *   루트 `12`에서 시작합니다. `13 > 12`이므로 오른쪽 자식 `17`로 이동합니다.
    *   `17`에 도달합니다. `13 < 17`이므로 왼쪽 자식으로 이동합니다.
    *   `17`의 왼쪽 자식이 `nullptr`이므로, 여기에 `13`을 새 노드로 삽입합니다.
    *   (슬라이드에서 13이 핑크색으로 표시되어 새로 삽입되었음을 암시)

2.  **`Insert(18, D)` 과정**:
    *   루트 `12`에서 시작합니다. `18 > 12`이므로 오른쪽 자식 `17`로 이동합니다.
    *   `17`에 도달합니다. `18 > 17`이므로 오른쪽 자식 `20`으로 이동합니다.
    *   `20`에 도달합니다. `18 < 20`이므로 왼쪽 자식 `19`로 이동합니다.
    *   `19`에 도달합니다. `18 < 19`이므로 왼쪽 자식으로 이동합니다.
    *   `19`의 왼쪽 자식이 `nullptr`이므로, 여기에 `18`을 새 노드로 삽입합니다.
    *   (슬라이드에서 18까지 파란색 선으로 연결되어 삽입 경로를 암시)

위 두 연산이 완료되면 슬라이드에 제시된 최종 트리가 형성됩니다.

**시험 포인트**:
*   ⭐ **BST의 정의 및 속성**: 이진 탐색 트리의 세 가지 핵심 속성(왼쪽 < 부모 < 오른쪽)을 정확하게 이해하고 설명할 수 있어야 합니다. 이 속성을 위반하는 트리가 주어졌을 때 BST가 아님을 판단할 수 있어야 합니다.
*   ⭐ **`Insert` 연산 과정**: 특정 값을 BST에 삽입할 때, 어떤 경로를 탐색하며 최종적으로 어디에 노드가 추가되는지 단계별로 설명하거나 그림으로 그릴 수 있어야 합니다.
*   ⭐ **시간 복잡도 분석**: `Insert` 연산의 시간 복잡도가 트리의 높이 $h$에 비례하는 이유를 이해하고, 최선($O(\log N)$) 및 최악($O(N)$)의 경우를 구분하여 설명할 수 있어야 합니다. 트리의 균형이 시간 복잡도에 미치는 영향은 매우 중요합니다.

## Slide 15

### 데이터구조 (CSED233) 퀴즈 노트

**핵심 개념**:
이 퀴즈는 이진 탐색 트리(BST)의 근본적인 특성과 트리의 '높이(height)'가 트리의 구조에 따라 어떻게 달라질 수 있는지에 대한 이해를 묻습니다.

*   **이진 탐색 트리 (BST)**: 모든 노드에 대해 다음 두 가지 속성을 만족하는 이진 트리입니다.
    1.  노드의 왼쪽 서브트리에 있는 모든 키는 현재 노드의 키보다 작습니다.
    2.  노드의 오른쪽 서브트리에 있는 모든 키는 현재 노드의 키보다 큽니다.
*   **트리의 높이 (Height)**: 루트 노드에서 가장 깊은 리프 노드까지의 가장 긴 경로에 있는 엣지(간선)의 개수를 의미합니다. (일부 정의에서는 노드의 개수를 포함하기도 하지만, 일반적으로 엣지 개수를 사용합니다.)
*   **$O(\log n)$ 높이**: $n$개의 노드를 가진 BST가 완벽하게 또는 거의 완벽하게 균형 잡혀 있을 때의 이상적인 높이입니다. 탐색, 삽입, 삭제 연산의 시간 복잡도가 이 높이에 따라 $O(\log n)$이 됩니다. 이는 이진 탐색의 효율성과 유사합니다.
*   **$O(n)$ 높이**: $n$개의 노드를 가진 BST가 한쪽으로 완전히 치우쳐져(skewed) 연결 리스트와 유사한 형태를 가질 때의 최악의 높이입니다. 이 경우 높이는 $n-1$ 또는 $n$에 비례하며, 탐색, 삽입, 삭제 연산의 시간 복잡도가 $O(n)$으로 저하됩니다.

**코드/수식 해설**:
이 문제에서는 구체적인 코드가 아닌, 트리의 높이를 나타내는 점근적 표기법($O$ 표기)에 대한 이해가 중요합니다.

*   $n$개의 노드를 가진 이진 탐색 트리의 **최소 높이**는 다음과 같습니다.
    $$h_{min} = \lfloor \log_2 n \rfloor$$
    이 경우 높이의 점근적 상한은 $O(\log n)$입니다.
*   $n$개의 노드를 가진 이진 탐색 트리의 **최대 높이**는 다음과 같습니다.
    $$h_{max} = n - 1 \quad \text{(엣지 개수 기준)}$$
    이 경우 높이의 점근적 상한은 $O(n)$입니다.

이러한 높이 범위는 트리가 어떻게 구성되느냐에 따라 달라집니다. $O(\log n)$은 가장 효율적인 균형 잡힌 트리의 경우를 나타내며, $O(n)$은 가장 비효율적인 편향된 트리의 경우를 나타냅니다.

**구체적 예시**:
네, 두 BST가 정확히 동일한 키 집합을 가지고 있더라도 하나는 $O(\log n)$ 높이를, 다른 하나는 $O(n)$ 높이를 가질 수 있습니다. 이는 키들이 BST에 삽입되는 순서에 따라 트리의 구조가 달라지기 때문입니다.

예를 들어, 키 집합이 $\{10, 20, 30\}$이라고 가정해 봅시다. 여기서 $n=3$입니다.

1.  **$O(\log n)$ 높이를 가지는 BST (균형 트리)**:
    *   키를 `20, 10, 30` 순서로 삽입하면, 트리는 다음과 같이 균형 잡힌 형태를 이룹니다.
    ```
          20
         /  \
        10  30
    ```
    이 트리의 높이는 1입니다 (루트에서 리프 10 또는 30까지 엣지 1개). 이는 $O(\log 3)$에 해당합니다.

2.  **$O(n)$ 높이를 가지는 BST (편향 트리)**:
    *   키를 `10, 20, 30` 순서로 **오름차순** 삽입하면, 트리는 오른쪽으로 완전히 편향됩니다.
    ```
        10
          \
           20
             \
              30
    ```
    이 트리의 높이는 2입니다 (루트 10에서 리프 30까지 엣지 2개). 이는 $O(3)$에 해당합니다.
    (마찬가지로 `30, 20, 10` 순서로 **내림차순** 삽입하면 왼쪽으로 편향된 트리가 되어 동일하게 $O(n)$ 높이를 가집니다.)

이 예시를 통해 알 수 있듯이, 같은 키 집합을 사용하더라도 삽입 순서에 따라 트리의 구조와 높이가 극명하게 달라질 수 있습니다.

**시험 포인트**:
*   ⭐ **BST 높이의 점근적 범위**: $n$개의 노드를 가진 BST의 높이는 최소 $O(\log n)$ (완전히 균형 잡힌 경우)에서 최대 $O(n)$ (완전히 편향된 경우)까지 가능하다는 사실을 정확히 이해하고 있어야 합니다.
*   ⭐ **삽입 순서의 중요성**: 동일한 키 집합이라도, 키들이 BST에 삽입되는 순서에 따라 트리의 형태(구조)가 크게 달라질 수 있으며, 이는 트리의 높이와 모든 연산(탐색, 삽입, 삭제)의 성능에 직접적인 영향을 미칩니다.
*   ⭐ **자가 균형 이진 탐색 트리의 필요성**: 일반 BST는 최악의 경우 $O(n)$의 성능을 보일 수 있기 때문에, 실제 응용에서는 항상 $O(\log n)$의 성능을 보장하기 위해 AVL 트리나 레드-블랙 트리와 같은 자가 균형 이진 탐색 트리를 사용한다는 점을 숙지해야 합니다.

## Slide 16

**핵심 개념**:
이 슬라이드는 이진 탐색 트리(Binary Search Tree, BST)에서 특정 노드를 삭제하는 여러 경우 중 가장 간단한 경우, 즉 삭제할 노드가 **리프(Leaf) 노드**일 때의 동작 방식을 설명합니다. BST는 다음 조건을 만족하는 트리입니다:
*   모든 노드의 왼쪽 서브트리에는 노드 값보다 작은 값만 포함됩니다.
*   모든 노드의 오른쪽 서브트리에는 노드 값보다 큰 값만 포함됩니다.
*   왼쪽 및 오른쪽 서브트리도 BST입니다.

리프 노드는 자식 노드가 없는 노드를 의미합니다.

**코드/수식 해설**:
리프 노드를 삭제하는 알고리즘은 다음과 같습니다.
1.  **노드 탐색**: 삭제할 값 `x`를 가진 노드를 찾기 위해 트리를 순회합니다. 이 과정에서 삭제할 노드의 **부모(Parent) 노드**를 함께 기록합니다.
2.  **리프 노드 확인**: 찾은 노드 `x`가 왼쪽 자식도 없고 오른쪽 자식도 없는지 확인합니다. 이 경우 `x`는 리프 노드입니다.
3.  **노드 삭제**:
    *   부모 노드의 해당 자식 포인터(만약 `x`가 부모의 왼쪽 자식이었다면 `parent->left`, 오른쪽 자식이었다면 `parent->right`)를 `NULL`로 설정합니다.
    *   `x` 노드가 차지하고 있던 메모리를 해제합니다.

시간 복잡도는 다음과 같습니다:
*   노드를 탐색하는 데 걸리는 시간: 트리의 높이에 비례합니다. 최악의 경우(트리가 한쪽으로 기울어진 경우) $O(N)$ (N은 노드의 개수), 최선의 경우(균형 트리) $O(\log N)$입니다.
*   부모 노드의 포인터를 `NULL`로 설정하고 노드를 해제하는 데 걸리는 시간: $O(1)$입니다.
따라서, 리프 노드 삭제의 전체 시간 복잡도는 $O(h)$이며, 여기서 $h$는 트리의 높이입니다.

**구체적 예시**:
슬라이드에 제시된 이진 탐색 트리와 `Delete(6, D)` 연산을 예시로 들어보겠습니다.

1.  **초기 BST 상태**:
    *   루트: 12
    *   왼쪽 서브트리: 5 (root), 3 (left of 5), 8 (right of 5), 6 (left of 8)
    *   오른쪽 서브트리: 17 (root), 13 (left of 17), 20 (right of 17), 19 (left of 20), 18 (left of 19)

2.  **`Delete(6, D)` 연산**:
    *   **단계 1: 노드 6 탐색**:
        *   루트 12에서 시작. 6 < 12 이므로 왼쪽으로 이동 (현재 노드: 5, 부모: 12)
        *   노드 5에서. 6 > 5 이므로 오른쪽으로 이동 (현재 노드: 8, 부모: 5)
        *   노드 8에서. 6 < 8 이므로 왼쪽으로 이동 (현재 노드: 6, 부모: 8)
        *   노드 6을 찾았습니다. 부모 노드는 8입니다.
    *   **단계 2: 리프 노드 확인**: 노드 6은 왼쪽 자식도 없고 오른쪽 자식도 없습니다. 따라서 리프 노드입니다.
    *   **단계 3: 노드 6 삭제**:
        *   부모 노드 8의 왼쪽 자식 포인터를 `NULL`로 설정합니다 (`8->left = NULL`).
        *   노드 6에 할당된 메모리를 해제합니다.

3.  **삭제 후 BST 상태**: 노드 6과 8-6 연결이 사라집니다. 나머지 트리는 변경되지 않습니다.

**시험 포인트**:
*   ⭐ BST에서 리프 노드를 삭제하는 것이 가장 간단한 경우이며, 노드를 찾은 후 부모 노드의 포인터만 `NULL`로 설정하면 됩니다.
*   ⭐ BST 삭제의 시간 복잡도는 노드를 찾는 과정에 의해 결정되며, $O(h)$입니다. 여기서 $h$는 트리의 높이입니다. (균형 트리에서는 $O(\log N)$, 불균형 트리에서는 $O(N)$)
*   ⭐ BST 삭제는 리프 노드 삭제 외에도, 한 개의 자식을 가진 노드 삭제, 두 개의 자식을 가진 노드 삭제의 세 가지 주요 경우가 있음을 알고 있어야 합니다.

## Slide 17

**핵심 개념**:
이 슬라이드는 이진 탐색 트리(Binary Search Tree, BST)에서 노드를 삭제하는 연산 중 가장 기본적인 경우인 '리프 노드(leaf node)'를 삭제하는 방법을 설명합니다. 리프 노드는 자식 노드가 없는 노드를 의미합니다. BST의 노드 삭제는 크게 세 가지 경우로 나눌 수 있으며, 이 슬라이드는 그중 자식이 없는 노드를 삭제하는 "쉬운 경우(Easy case: no children)"에 해당합니다.

**코드/수식 해설**:
BST에서 특정 키(`key`)를 가진 노드를 삭제하는 과정은 먼저 해당 노드를 트리에서 찾아야 합니다. 노드를 찾은 후, 해당 노드가 리프 노드인 경우 다음과 같은 논리 흐름으로 삭제가 이루어집니다.

1.  **노드 탐색**: 삭제할 키를 가진 노드를 찾기 위해 BST의 탐색 규칙을 따릅니다.
    *   현재 노드의 키보다 삭제할 키가 작으면 왼쪽 자식으로 이동합니다.
    *   현재 노드의 키보다 삭제할 키가 크면 오른쪽 자식으로 이동합니다.
    *   삭제할 키와 현재 노드의 키가 같으면 노드를 찾은 것입니다.
2.  **리프 노드 확인**: 찾은 노드가 왼쪽 자식도 없고 오른쪽 자식도 없는지 확인합니다.
    *   `node->left == NULL` 이고 `node->right == NULL` 인 경우 리프 노드입니다.
3.  **삭제 처리**:
    *   해당 노드의 부모 노드가 이 노드를 가리키던 포인터를 `NULL`로 설정합니다.
    *   삭제할 노드가 사용하던 메모리를 해제합니다.

다음은 리프 노드를 삭제하는 부분에 대한 간소화된 의사 코드입니다:

```c
// Simplified pseudocode for deleting a leaf node in a BST
Node* deleteNode(Node* root, int key_to_delete) {
    if (root == NULL) return NULL; // Tree is empty

    Node* current = root;
    Node* parent = NULL;

    // 1. Find the node with key_to_delete and its parent
    while (current != NULL && current->key != key_to_delete) {
        parent = current;
        if (key_to_delete < current->key) {
            current = current->left;
        } else {
            current = current->right;
        }
    }

    // If node was not found
    if (current == NULL) {
        return root;
    }

    // 2. Case 1: Node to be deleted is a leaf node (no children)
    if (current->left == NULL && current->right == NULL) {
        if (parent == NULL) { // Deleting the root node, and it's a leaf
            free(current);
            return NULL; // Tree becomes empty
        }
        // Disconnect the leaf node from its parent
        if (parent->left == current) {
            parent->left = NULL;
        } else {
            parent->right = NULL;
        }
        free(current); // Free the memory of the deleted node
        return root; // Return the original root (or updated root if root was deleted)
    }

    // Other deletion cases (one child or two children) would be handled here
    // ...

    return root;
}
```

**구체적 예시**:
슬라이드에 제시된 BST에서 키 `6`을 가진 노드를 삭제하는 예시를 따릅니다.

1.  **노드 탐색 (Find the location of the node whose key = 6)**:
    *   루트 노드 `12`에서 시작합니다. `6 < 12`이므로 왼쪽 자식 `5`로 이동합니다.
    *   `5`에서 `6 > 5`이므로 오른쪽 자식 `8`로 이동합니다.
    *   `8`에서 `6 < 8`이므로 왼쪽 자식 `6`으로 이동합니다. 이제 삭제할 노드 `6`을 찾았습니다. 이때 `8`이 노드 `6`의 부모 노드입니다.
2.  **리프 노드 확인**: 노드 `6`은 어떠한 자식 노드도 가지고 있지 않으므로 리프 노드입니다 (easy case: no children).
3.  **삭제 처리**:
    *   노드 `6`의 부모 노드인 `8`은 `6`을 왼쪽 자식으로 가리키고 있었습니다. 이 `8`의 왼쪽 자식 포인터를 `NULL`로 변경합니다. (즉, `8->left = NULL;`)
    *   노드 `6`이 사용하던 메모리를 운영체제로 반환합니다. (`free(6)`)
    *   결과적으로 노드 `6`은 트리에서 완전히 제거되고, 노드 `8`의 왼쪽 가지는 비어있게 됩니다.

**시험 포인트**:
*   ⭐ BST에서 노드를 탐색하는 기본 규칙(`key` 비교를 통한 좌/우 이동)을 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ BST 노드 삭제의 세 가지 주요 경우(리프 노드 삭제, 자식 노드가 하나인 경우 삭제, 자식 노드가 두 개인 경우 삭제)를 구분하고, 이 중 리프 노드 삭제가 가장 간단하다는 것을 알아야 합니다.
*   ⭐ **리프 노드 삭제 시에는 부모 노드의 해당 자식 포인터를 `NULL`로 설정하고, 삭제할 노드의 메모리를 해제하는 것이 핵심**입니다. 이를 통해 트리의 구조적 무결성을 유지하면서 노드를 제거합니다.

## Slide 18

**핵심 개념**
이 슬라이드는 이진 탐색 트리(BST: Binary Search Tree)에서 **리프 노드(Leaf Node)**를 삭제하는 가장 간단한 경우를 설명합니다. 리프 노드는 자식 노드가 없는 노드를 의미합니다. BST의 삭제 연산은 크게 세 가지 경우로 나뉘는데, 리프 노드 삭제는 그 중 가장 기본적인 경우입니다. BST의 특성을 유지하면서 노드를 삭제해야 합니다.

**코드/수식 해설**
리프 노드를 삭제하는 과정은 다음과 같습니다:

1.  **삭제할 노드 찾기**: 주어진 값을 가진 노드를 트리를 순회하여 찾아냅니다. 이 과정에서 해당 노드의 부모 노드도 함께 추적합니다.
2.  **노드가 리프인지 확인**: 찾아낸 노드가 자식 노드를 가지고 있지 않은지 확인합니다.
3.  **부모 포인터 업데이트**: 노드가 리프 노드임이 확인되면, 그 노드를 가리키던 부모 노드의 포인터(왼쪽 자식 또는 오른쪽 자식)를 `NULL`로 설정하여 연결을 끊습니다.
4.  **노드 메모리 해제**: 삭제된 노드가 차지하던 메모리를 운영체제에 반환합니다.

수식적으로 특별히 복잡한 부분은 없지만, 노드를 찾는 과정은 트리의 높이($h$)에 비례합니다.
검색 및 삭제 연산의 시간 복잡도는 다음과 같습니다.
$$ T(n) = O(h) $$
여기서 $n$은 트리의 노드 수이며, $h$는 트리의 높이입니다. 균형 잡힌 BST의 경우 $h = \log n$이므로 $O(\log n)$이 되지만, 최악의 경우(한쪽으로 치우친 트리) $h=n$이므로 $O(n)$이 됩니다.

**구체적 예시**
첨부된 슬라이드의 트리를 예시로 들어 설명하겠습니다.

*   **초기 BST**:
    ```
          12
         /  \
        5    17
       / \   / \
      3   8 13 20
               /
              19
             /
            18
    ```
*   **삭제할 노드**: 이 슬라이드의 맥락상 '리프 노드 8'을 삭제하는 과정을 가정합니다 (슬라이드 하단의 "After Delete(6, D)"는 오기일 가능성이 높으며, 노드 6은 현재 트리에 존재하지 않습니다).
*   **삭제 과정**:
    1.  루트(12)부터 시작하여 8을 찾습니다. 12 < 8이 아니므로 왼쪽 자식(5)으로 이동합니다.
    2.  5 < 8이므로 오른쪽 자식(8)으로 이동합니다. 노드 8을 찾았습니다. 이 때, 노드 5가 노드 8의 부모 노드임을 기록합니다.
    3.  노드 8은 왼쪽 자식(NULL)도 오른쪽 자식(NULL)도 없으므로 리프 노드입니다.
    4.  노드 5의 오른쪽 자식 포인터를 `NULL`로 설정합니다 (`5.right_child = NULL`).
    5.  노드 8의 메모리를 해제합니다.
*   **삭제 후 BST**:
    ```
          12
         /  \
        5    17
       /     / \
      3     13 20
               /
              19
             /
            18
    ```
    슬라이드에서는 삭제된 노드 8이 있던 자리에 파란색 점선 원으로 표시되어, 해당 노드가 제거되었음을 시각적으로 보여줍니다.

**시험 포인트**
*   ⭐ **BST 삭제의 세 가지 주요 경우**를 구분하고 각 경우에 대한 처리 방법을 이해해야 합니다. (리프 노드 삭제, 자식이 하나인 노드 삭제, 자식이 둘인 노드 삭제)
*   ⭐ **리프 노드 삭제**는 가장 간단한 경우이며, 해당 노드를 가리키던 **부모 노드의 포인터를 `NULL`로 업데이트**하고 노드를 해제하면 됩니다.
*   삭제 연산의 시간 복잡도는 트리의 높이에 따라 달라진다는 점($O(h)$)을 기억해야 합니다. ⭐ **최악의 경우($O(n)$)와 최적의 경우($O(\log n)$)를 구분**할 수 있어야 합니다.

## BST: Degree 1 노드 삭제

### 핵심 개념

이 슬라이드는 이진 탐색 트리(Binary Search Tree, BST)에서 **자식 노드가 하나인 노드(Degree 1 Node)**를 삭제하는 방법에 대해 설명합니다.

*   **이진 탐색 트리(BST)**: 모든 노드에서 왼쪽 서브트리의 키는 해당 노드의 키보다 작고, 오른쪽 서브트리의 키는 해당 노드의 키보다 큰 특성을 만족하는 트리입니다.
*   **노드의 차수(Degree of a Node)**: 특정 노드의 자식 노드 개수를 의미합니다.
    *   **Degree 0**: 자식 노드가 없는 노드 (리프 노드).
    *   **Degree 1**: 자식 노드가 하나 있는 노드.
    *   **Degree 2**: 자식 노드가 두 개 있는 노드.
*   **Degree 1 노드 삭제 원리**: 삭제할 노드($N$)가 유일한 자식($C$)을 가질 때, $N$의 부모($P$)가 $N$을 건너뛰고 $C$를 직접 가리키도록 연결을 변경하여 $N$을 제거합니다. 이는 BST의 속성을 위반하지 않으면서 노드를 효율적으로 제거하는 방법입니다.

### 코드/수식 해설

Degree 1 노드 삭제 알고리즘의 일반적인 절차는 다음과 같습니다. (별도의 수식은 사용되지 않음)

1.  **삭제할 노드 탐색**: 트리에서 주어진 값을 가진 노드 $N$을 찾습니다. 이 과정은 루트부터 시작하여 BST의 속성을 따라 내려가며 탐색합니다.
2.  **Degree 1 노드 확인**: 노드 $N$이 하나의 자식 노드 $C$를 가지고 있는지 확인합니다 (왼쪽 자식 또는 오른쪽 자식 중 하나만 존재).
3.  **부모-자식 연결 재설정**: 노드 $N$의 부모 노드 $P$를 식별합니다.
    *   만약 $N$이 $P$의 왼쪽 자식이었다면, $P$의 왼쪽 포인터가 $N$ 대신 $C$를 가리키도록 변경합니다.
    *   만약 $N$이 $P$의 오른쪽 자식이었다면, $P$의 오른쪽 포인터가 $N$ 대신 $C$를 가리키도록 변경합니다.
    *   **특수 케이스**: 만약 $N$이 루트 노드이고 Degree 1이라면, $C$가 새로운 루트 노드가 됩니다.
4.  **노드 해제**: 노드 $N$의 메모리를 해제합니다.

```cpp
// Node 구조체 (예시)
struct Node {
    int data;
    Node* left;
    Node* right;
};

// BST에서 Degree 1 노드를 삭제하는 부분 함수 (가상 코드)
Node* deleteNode(Node* root, int key) {
    // 1. 삭제할 노드 탐색 (재귀적으로 찾아 내려감)
    if (root == nullptr) {
        return root;
    }
    if (key < root->data) {
        root->left = deleteNode(root->left, key);
    } else if (key > root->data) {
        root->right = deleteNode(root->right, key);
    } else { // key == root->data, 즉 삭제할 노드를 찾았음
        // 2. Degree 1 노드 확인
        if (root->left == nullptr) { // 왼쪽 자식이 없으므로, 오른쪽 자식이 있거나 리프 노드 (Degree 0)
            Node* temp = root->right;
            delete root; // 현재 노드(root) 삭제
            return temp; // 오른쪽 자식을 부모에게 연결
        } else if (root->right == nullptr) { // 오른쪽 자식이 없으므로, 왼쪽 자식이 있음 (Degree 1)
            Node* temp = root->left;
            delete root; // 현재 노드(root) 삭제
            return temp; // 왼쪽 자식을 부모에게 연결
        }
        // Degree 2 노드 처리 (이 슬라이드의 범위를 넘어감)
        // ...
    }
    return root;
}
```

### 구체적 예시

슬라이드에 제시된 `Delete(20, D)` 예시를 통해 Degree 1 노드 삭제 과정을 살펴보겠습니다.

**초기 BST 상태:**

```
      12
     /  \
    5    17
   / \   / \
  3   8 13  20
             \
              19
             /
            18
```

1.  **삭제할 노드 식별**: `Delete(20, D)` 명령에 따라 값 20을 가진 노드를 찾습니다. 노드 20은 노드 17의 오른쪽 자식입니다.
2.  **노드 20의 차수 확인**: 노드 20은 왼쪽 자식은 없고, 오른쪽 자식으로 노드 19를 가지고 있습니다. 따라서 노드 20은 **Degree 1 노드**입니다.
3.  **연결 재조정**:
    *   노드 20의 부모는 노드 17입니다.
    *   노드 20의 유일한 자식은 노드 19입니다.
    *   노드 17은 원래 노드 20을 오른쪽 자식으로 가리키고 있었습니다. 이제 이 연결을 변경하여 노드 19를 직접 가리키도록 합니다. 즉, `17->right`가 `19`를 가리키게 됩니다.
4.  **노드 20 제거**: 노드 20은 트리에서 제거되고 메모리에서 해제됩니다.

**삭제 후 BST 상태:**

```
      12
     /  \
    5    17
   / \   / \
  3   8 13  19  (<- 노드 17의 오른쪽 자식이 노드 19로 변경)
             /
            18
```
이 과정을 통해 BST의 속성(왼쪽 < 부모 < 오른쪽)은 여전히 유지됩니다.

### 시험 포인트

*   ⭐ **BST 노드 삭제의 3가지 케이스**: 리프 노드(Degree 0), Degree 1 노드, Degree 2 노드에 대한 삭제 로직을 명확히 이해하고 구분할 수 있어야 합니다. 각 케이스별 처리 방식이 다릅니다.
*   ⭐ **Degree 1 노드 삭제 메커니즘**: 삭제할 노드를 그 유일한 자식 노드로 대체하는 방법(부모와 자식 간의 연결 변경)을 정확히 설명하고 구현할 수 있어야 합니다.
*   ⭐ **삭제 후 BST 속성 유지**: 노드 삭제 후에도 BST의 핵심 속성(`left < parent < right`)이 깨지지 않고 유지되는지 확인하는 것이 중요합니다. Degree 1 노드 삭제는 이 속성을 항상 유지합니다.
*   ⭐ **시간 복잡도**: 노드 탐색과 연결 재조정 모두 트리의 높이($h$)에 비례하므로, 최악의 경우 $O(N)$ (skewed tree) 또는 평균적으로 $O(\log N)$ (balanced tree)의 시간 복잡도를 가집니다.

## BST (이진 탐색 트리): Degree 1 노드 삭제

### 핵심 개념
이 슬라이드는 이진 탐색 트리(BST)에서 **단 하나의 자식 노드(Degree 1)**를 가진 노드를 삭제하는 방법에 대해 설명합니다. BST에서 노드를 삭제하는 경우는 크게 세 가지로 나뉩니다:
1.  **자식이 없는 리프 노드 (Degree 0)** 삭제
2.  **자식이 하나인 노드 (Degree 1)** 삭제
3.  **자식이 두 개인 노드 (Degree 2)** 삭제

Degree 1 노드를 삭제하는 경우는 리프 노드를 삭제하는 경우와 유사하게 비교적 간단합니다. 삭제할 노드($N$)의 부모 노드($P$)가 $N$의 유일한 자식 노드($C$)를 직접 가리키도록 연결을 변경하여, $N$을 트리에서 제외시킵니다.

### 코드/수식 해설
주어진 슬라이드에 명시적인 코드는 없지만, Degree 1 노드를 삭제하는 로직은 다음과 같은 포인터 재연결로 구현됩니다.
삭제할 노드 $N$을 찾았을 때, $N$의 부모 노드를 $P$, $N$의 유일한 자식 노드를 $C$라고 합시다.

1.  $N$이 $P$의 왼쪽 자식일 경우: $P \rightarrow \text{left_child} = C$
2.  $N$이 $P$의 오른쪽 자식일 경우: $P \rightarrow \text{right_child} = C$

이 과정을 통해 $N$은 트리 구조에서 제거되며, $N$이 차지하던 메모리는 해제될 수 있습니다.

수식적으로 표현하면,
-   삭제할 노드의 키를 $k_D$
-   삭제할 노드를 $N_D$
-   $N_D$의 부모 노드를 $P$
-   $N_D$의 유일한 자식 노드를 $C$ 라고 할 때,
삭제 후 $P$는 $C$를 가리키게 됩니다.

$$
\text{if } P.\text{left_child} = N_D \text{ then } P.\text{left_child} = C \\
\text{else if } P.\text{right_child} = N_D \text{ then } P.\text{right_child} = C
$$

### 구체적 예시
슬라이드에 제시된 `Delete(20, D)` 예시를 통해 과정을 살펴봅시다.

1.  **삭제할 노드 찾기**: 트리를 탐색하여 키 값이 20인 노드를 찾습니다. 그림에서 보라색으로 강조된 노드 20이 그 대상입니다.
2.  **노드 유형 파악**: 노드 20은 자식 노드로 19만을 가지고 있으므로, Degree 1 노드입니다.
3.  **부모와 자식 파악**:
    *   노드 20의 부모 노드는 17입니다.
    *   노드 20의 유일한 자식 노드는 19입니다.
4.  **연결 재설정**: 노드 20이 부모 노드 17의 오른쪽 자식이므로, 노드 17의 오른쪽 자식 포인터가 노드 20 대신 노드 19를 가리키도록 변경합니다.
    *   **삭제 전**: `17 -> (오른쪽 자식) -> 20 -> (왼쪽 자식) -> 19`
    *   **삭제 후**: `17 -> (오른쪽 자식) -> 19`
5.  **메모리 해제**: 노드 20은 이제 트리에서 분리되었으므로, 해당 노드가 사용하던 메모리를 해제합니다.

결과적으로, 20이라는 키 값을 가진 노드는 트리에서 제거되고, 17과 19는 직접 연결됩니다.

### 시험 포인트
*   ⭐ **BST 노드 삭제의 3가지 경우**: 리프 노드, 자식이 하나인 노드, 자식이 둘인 노드 각각의 삭제 방법과 차이점을 정확히 이해해야 합니다. 특히, 이 슬라이드에서 다루는 자식이 하나인 노드의 삭제는 부모-자식 간의 연결 재설정으로 이루어진다는 점을 기억하세요.
*   ⭐ **자식이 하나인 노드 삭제의 효율성**: 자식이 둘인 노드 삭제(inorder successor/predecessor를 찾아 교체)에 비해 구현이 간단하고, 재귀적으로 트리의 다른 부분을 건드리지 않아도 되므로 효율적입니다. 시간 복잡도는 노드를 찾는 데 필요한 시간과 동일하게 트리의 높이($h$)에 비례하여 $O(h)$입니다. 균형 잡힌 트리(예: AVL 트리, Red-Black 트리)에서는 $O(\log N)$입니다.
*   **실제 구현 시 주의사항**: 삭제할 노드가 루트 노드인 경우 (부모 노드가 없는 경우)와 같은 엣지 케이스 처리 방법을 숙지해야 합니다. 이 경우 새로운 루트가 해당 노드의 유일한 자식이 됩니다.

### 핵심 개념

이 슬라이드는 이진 탐색 트리(Binary Search Tree, BST)에서 **한 개의 자식 노드를 가진 노드 (Degree 1 Node)를 삭제하는 과정**을 설명합니다.

*   **Degree 1 노드 삭제 원칙**: 삭제될 노드 `D`가 유일한 자식 노드 `C`를 가질 경우, `D`의 부모 노드 `P`는 `D`를 가리키던 포인터를 `D`의 자식 노드 `C`를 직접 가리키도록 변경하여 `D`를 트리에서 연결 해제합니다.
*   **BST 속성 유지**: 이 방법을 통해 노드를 삭제해도 이진 탐색 트리의 핵심 속성 (왼쪽 자식 < 부모 < 오른쪽 자식)은 계속해서 유지됩니다. 삭제될 노드의 자식 노드는 삭제될 노드의 기존 위치를 대체하기 때문입니다.

### 코드/수식 해설

BST에서 `D` 노드를 삭제할 때, `D`의 부모 노드 `P`와 `D`의 유일한 자식 노드 `C` 간의 포인터 연결을 변경합니다.

```pseudocode
// D는 삭제할 노드, P는 D의 부모 노드
DeleteNodeWithOneChild(Node D, Node P):
    Node C = null // D의 유일한 자식 노드

    // D가 왼쪽 자식만 가질 경우
    if D.left is not null and D.right is null:
        C = D.left
    // D가 오른쪽 자식만 가질 경우 (슬라이드 그림에 해당)
    else if D.right is not null and D.left is null:
        C = D.right

    // D가 P의 왼쪽 자식인 경우
    if P.left is D:
        P.left = C
    // D가 P의 오른쪽 자식인 경우 (슬라이드 그림에 해당)
    else if P.right is D:
        P.right = C

    // D 노드의 메모리 해제
    free(D)
```

이 연산의 시간 복잡도는 노드를 찾아가는 과정에 트리의 높이에 비례합니다.
$$
O(h)
$$
여기서 $h$는 트리의 높이입니다. 균형 잡힌 트리의 경우 $h \approx \log n$이므로 $O(\log n)$이 됩니다.

### 구체적 예시

슬라이드 이미지는 노드 `19`를 삭제하는 상황을 보여줍니다 (텍스트 "After Delete(20, D)"는 삭제 후 상태를 나타내는 일반적인 주석일 수 있으며, 그림 자체는 Degree 1 노드인 `19`의 삭제를 시각적으로 설명합니다).

1.  **초기 상태**:
    *   노드 `17`의 오른쪽 자식은 `19`입니다.
    *   노드 `19`는 왼쪽 자식으로 `18`을 가지고 있으며, 오른쪽 자식은 없습니다. 따라서 `19`는 한 개의 자식 (`18`)을 가진 Degree 1 노드입니다.
2.  **삭제 과정**:
    *   `19`를 삭제하기 위해, `19`의 부모 노드인 `17`은 자신의 오른쪽 자식 포인터를 `19`가 아닌 `19`의 유일한 자식 노드인 `18`을 직접 가리키도록 변경합니다.
    *   그림에서 빨간색 선은 삭제 경로를, 파란색 점선 원과 화살표는 `17`의 포인터가 `19`를 건너뛰고 `18`로 재연결되는 것을 시각적으로 보여줍니다.
3.  **최종 상태**:
    *   노드 `19`는 트리에서 제거되고, `17`의 오른쪽 자식은 이제 `18`이 됩니다. 즉, `12` -> `17` -> `18` 경로가 형성됩니다.

### 시험 포인트

*   ⭐ **BST 노드 삭제의 세 가지 경우**: 리프 노드 삭제, 하나의 자식을 가진 노드 삭제 (본 슬라이드), 두 개의 자식을 가진 노드 삭제. 각 경우의 처리 방법을 정확히 이해하고 비교할 수 있어야 합니다.
*   ⭐ **BST 속성 유지**: 노드 삭제 후에도 이진 탐색 트리 속성(왼쪽 자식 < 부모 < 오른쪽 자식)이 유지되는 방식과 그 중요성을 파악해야 합니다. 하나의 자식을 가진 노드 삭제 시에는 해당 자식 노드가 삭제된 노드의 자리를 대체함으로써 속성이 자연스럽게 유지됩니다.
*   ⭐ **시간 복잡도 분석**: BST 노드 삭제 연산의 최악 및 평균 시간 복잡도를 알고, 트리의 높이와 관련하여 설명할 수 있어야 합니다 (최악 $O(h)$, 평균 $O(\log n)$).

- **핵심 개념**:
이 슬라이드는 이진 탐색 트리(BST)에서 자식 노드를 두 개 가진 노드(Degree 2 Node)를 삭제하는 첫 번째 단계를 다룹니다. BST에서 노드 삭제는 자식의 수에 따라 세 가지 경우로 나뉩니다:
1.  **자식 노드가 없는 경우 (Leaf Node)**: 해당 노드를 단순히 제거합니다.
2.  **자식 노드가 하나인 경우**: 해당 노드를 제거하고, 그 자식 노드를 삭제된 노드의 부모와 직접 연결합니다.
3.  **자식 노드가 두 개인 경우 (Deg. 2 Node)**: 이 경우가 가장 복잡하며, 슬라이드에서 `Delete(x, D)`가 이 경우에 해당함을 보여줍니다. 이 경우, 삭제할 노드를 다른 노드의 값으로 대체한 후, 대체된 노드를 삭제하는 방식을 사용합니다. 일반적으로 대체 노드는 Inorder Successor (오른쪽 서브트리에서 가장 작은 값) 또는 Inorder Predecessor (왼쪽 서브트리에서 가장 큰 값)가 됩니다.

- **코드/수식 해설**:
이 슬라이드는 특정 삭제 연산이 필요한 상황을 시각적으로 보여주는 단계이므로, 직접적인 코드나 수식은 포함되어 있지 않습니다. 그러나 Deg. 2 노드 삭제의 핵심 아이디어는 Inorder Successor(혹은 Predecessor)를 찾아 값을 복사하고, 그 Successor 노드를 제거하는 것입니다. 다음은 Inorder Successor를 찾는 함수의 의사코드입니다:

```cpp
// BST에서 특정 서브트리의 최솟값(Inorder Successor)을 찾는 함수
Node* findMin(Node* node) {
    Node* current = node;
    // 왼쪽 자식이 없을 때까지 계속 왼쪽으로 이동
    while (current != nullptr && current->left != nullptr) {
        current = current->left;
    }
    return current;
}
```
삭제할 노드 `x`가 자식 노드를 두 개 가지고 있다면, 일반적으로 `x`의 오른쪽 서브트리에서 `findMin(x->right)`를 호출하여 Inorder Successor를 찾습니다.

- **구체적 예시**:
슬라이드에 제시된 BST에서 `Delete(12, D)` 연산을 수행하려고 합니다.
1.  **삭제할 노드 식별**: `12`는 루트 노드이며, 왼쪽 자식 `5`와 오른쪽 자식 `17`을 모두 가지고 있습니다.
2.  **Degree 2 Node 확인**: `12`는 두 개의 자식을 가지고 있으므로 "Deg. 2 Node"에 해당합니다.
3.  **대체 노드 탐색 준비**: Deg. 2 노드를 삭제할 때는 직접 노드를 제거하는 대신, 해당 노드의 자리에 Inorder Successor 또는 Inorder Predecessor의 값을 복사하고, 그 Successor/Predecessor 노드를 삭제하는 전략을 사용합니다. 이 슬라이드는 이 단계까지를 보여주며, 아직 실제 대체 및 삭제가 이루어지기 전의 상황입니다. 만약 Inorder Successor를 사용한다면, `12`의 오른쪽 서브트리(`17`이 루트)에서 가장 작은 값인 `13`을 찾게 될 것입니다.

- **시험 포인트**:
*   ⭐ **BST 노드 삭제의 세 가지 경우(자식 0개, 1개, 2개)를 정확히 구분하고 각 경우의 처리 방법을 설명할 수 있어야 합니다.**
*   ⭐ **자식 노드가 두 개인 경우, Inorder Successor(오른쪽 서브트리에서 가장 작은 값) 또는 Inorder Predecessor(왼쪽 서브트리에서 가장 큰 값)를 사용하여 삭제하는 과정을 상세히 설명하고 그림으로 표현할 수 있어야 합니다.**
*   ⭐ **Inorder Successor 또는 Predecessor를 찾는 알고리즘을 이해하고 구현할 수 있어야 합니다.** (특히, Inorder Successor는 '오른쪽 자식으로 이동 후, 가장 왼쪽 노드로 계속 이동'하여 찾습니다.)

*   ⭐ "Deg. 2 Node" 삭제 시 Inorder Successor 또는 Predecessor를 선택하는 이유와 그 과정이 왜 BST 속성을 유지하는지 설명할 수 있어야 합니다.

**핵심 개념**:

*   **이진 탐색 트리 (BST) 삭제 - 자식이 두 개인 노드 (Deg. 2 Node)**: 이 슬라이드는 이진 탐색 트리에서 삭제할 노드($D$)가 두 개의 자식(왼쪽 자식과 오른쪽 자식)을 모두 가지고 있을 때, BST의 핵심 속성(왼쪽 서브트리 노드는 부모보다 작고, 오른쪽 서브트리 노드는 부모보다 크다)을 유지하면서 노드를 삭제하는 방법을 설명합니다.
*   **대체 노드 선택 원칙**: 자식이 두 개인 노드를 삭제할 때는 해당 노드의 자리에 **오른쪽 서브트리에서 가장 작은 값 (in-order successor)**을 가진 노드나 **왼쪽 서브트리에서 가장 큰 값 (in-order predecessor)**을 가진 노드를 찾아 대체합니다. 슬라이드는 주로 오른쪽 서브트리에서 가장 작은 값을 사용하는 방법을 예시로 듭니다. 대체 노드를 선택한 후, 삭제할 노드의 값을 대체 노드의 값으로 교체하고, 대체 노드를 원래 위치에서 제거합니다.

**코드/수식 해설**:

주어진 슬라이드는 직접적인 코드를 포함하고 있지는 않지만, 자식이 두 개인 노드(`node_to_delete`)를 삭제하는 `Delete(root, key)` 함수의 논리적 흐름을 다음과 같이 설명할 수 있습니다.

1.  **노드 탐색**: 트리에서 `key` 값을 가진 `node_to_delete`를 찾습니다.
2.  **대체 노드 탐색 (in-order successor)**: `node_to_delete`의 오른쪽 자식(`node_to_delete->right`)부터 시작하여, 가능한 한 왼쪽으로만 내려가면서 가장 작은 값을 가진 노드 `successor_node`를 찾습니다.
    *   이 `successor_node`는 `node_to_delete`의 오른쪽 서브트리 내에서 `node_to_delete` 다음으로 큰 값을 가지며, BST의 in-order 순회 시 `node_to_delete` 바로 다음에 나타나는 노드입니다.
    *   `successor_node`는 항상 자식이 없거나(리프 노드) 오른쪽 자식만 가지는 특징이 있습니다.
3.  **값 교체**: `node_to_delete`의 키 값을 `successor_node`의 키 값으로 대체합니다.
    ```cpp
    node_to_delete->key = successor_node->key;
    ```
4.  **대체 노드 제거**: 이제 `successor_node`를 `node_to_delete`의 오른쪽 서브트리에서 삭제합니다. `successor_node`는 자식이 없거나 오른쪽 자식만 있으므로, 일반적인 BST 삭제 알고리즘의 1개 자식 또는 0개 자식 케이스로 간단히 처리할 수 있습니다.

**구체적 예시**:

슬라이드에 제시된 `Delete(12, D)` 연산을 예시로 들어봅시다.

원본 BST:
```
      12 (삭제 대상)
     /  \
    5    17
   / \   / \
  3   8 13  19
           /
          18
```

1.  **삭제할 노드 식별**: 키 `12`를 가진 노드를 찾습니다. 이 노드는 왼쪽 자식 `5`와 오른쪽 자식 `17`을 모두 가지고 있으므로, 자식이 두 개인 노드입니다.
2.  **대체 노드 찾기**: 노드 `12`의 오른쪽 서브트리(루트 `17`)에서 가장 작은 값을 찾습니다. `17`에서 왼쪽으로 이동하면 `13`을 찾을 수 있습니다. `13`은 더 이상 왼쪽 자식이 없으므로, `13`이 `12`의 in-order successor입니다. (슬라이드에서 녹색으로 표시된 노드)
3.  **값 교체**: 삭제할 노드 `12`의 키 값을 대체 노드 `13`의 키 값으로 바꿉니다. 이제 트리의 루트는 값 `13`을 가집니다.
    ```
          13 (12의 자리에 대체)
         /  \
        5    17
       / \   / \
      3   8 13  19  (이전 13은 이제 원래 위치에서 삭제될 대상)
               /
              18
    ```
4.  **대체 노드 위치에서 제거**: 이제 원래 `13`이 있던 위치(즉, `17`의 왼쪽 자식)에서 노드 `13`을 삭제합니다. `13`은 자식이 없는 리프 노드이므로, 간단히 `17`의 왼쪽 자식 포인터를 `NULL`로 만들고 `13` 노드를 메모리에서 해제합니다.
    ```
          13
         /  \
        5    17
       / \     \
      3   8    19
               /
              18
    ```
    결과적으로 키 `12`를 가진 노드가 성공적으로 삭제되었고, BST의 속성은 유지됩니다.

**시험 포인트**:

*   ⭐ **자식이 두 개인 노드 삭제 방법**: 삭제할 노드의 오른쪽 서브트리에서 가장 작은 값(in-order successor)을 찾거나, 왼쪽 서브트리에서 가장 큰 값(in-order predecessor)을 찾아 값으로 교체한 후, 대체 노드를 원래 위치에서 제거하는 절차를 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **in-order successor/predecessor의 특징**: 이들을 대체 노드로 사용하는 이유 (BST 속성 유지 및 삭제 용이성)를 설명할 수 있어야 합니다. 특히 in-order successor는 왼쪽 자식을 가질 수 없다는 점을 기억하세요.
*   ⭐ **BST 삭제 연산의 시간 복잡도**: 노드 탐색 및 대체 노드 탐색 모두 트리의 높이($h$)에 비례하므로, 최악의 경우 $O(h)$의 시간 복잡도를 가집니다. 균형 잡힌 트리의 경우 $O(\log N)$입니다.

**핵심 개념**:
이 슬라이드는 이진 탐색 트리(Binary Search Tree, BST)에서 두 개의 자식을 가진 노드(Degree 2 Node)를 삭제하는 과정을 설명합니다. 이러한 노드를 삭제할 때는 BST의 속성을 유지하기 위해 특별한 전략이 필요합니다. 핵심은 삭제할 노드의 값을 그 노드의 **오른쪽 서브트리에서 가장 작은 값(Inorder Successor)**으로 대체한 후, 해당 Inorder Successor 노드를 그 원래 위치에서 제거하는 것입니다. (또는 왼쪽 서브트리에서 가장 큰 값, 즉 Inorder Predecessor로 대체할 수도 있습니다.)

**코드/수식 해설**:
BST에서 두 자식을 가진 노드 `N` (삭제 대상)을 삭제하는 일반적인 알고리즘은 다음과 같습니다:

1.  **Inorder Successor 찾기**: `N`의 오른쪽 자식으로 이동한 후, 해당 노드부터 왼쪽 자식이 없을 때까지 계속 왼쪽으로 이동합니다. 이렇게 찾은 노드 `S`가 `N`의 Inorder Successor입니다. `S`는 `N`의 오른쪽 서브트리에서 가장 작은 키를 가집니다.
2.  **값 대체**: `N`의 키를 `S`의 키로 변경합니다. 이로써 `N`의 위치에는 `S`의 값이 들어가게 됩니다.
3.  **Inorder Successor 노드 삭제**: 이제 원래 `S`가 위치했던 곳에서 `S` 노드를 삭제합니다. `S`는 Inorder Successor의 정의상 항상 왼쪽 자식이 없는 노드이므로, 자식이 없거나(Leaf Node) 오른쪽 자식만 하나 있는 노드(Degree 1 Node) 중 하나입니다. 따라서 `S`의 삭제는 Case 0(자식 없음) 또는 Case 1(한 개의 자식)에 해당하여 상대적으로 간단하게 처리됩니다.

다음은 위 과정을 C++ 스타일의 유사 코드(Pseudo-code)로 나타낸 것입니다. 이 코드는 `deleteNode` 함수 내에서 `key == root->key`일 때 두 자식을 가진 노드 삭제 로직을 처리합니다.

```cpp
// BSTNode 구조체 정의 (예시)
// struct BSTNode {
//     int key;
//     BSTNode *left, *right;
// };

BSTNode* deleteNode(BSTNode* root, int key) {
    if (root == nullptr) return root; // 기저 조건: 트리가 비었거나 노드를 찾지 못함

    // 삭제할 키를 탐색
    if (key < root->key) {
        root->left = deleteNode(root->left, key);
    } else if (key > root->key) {
        root->right = deleteNode(root->right, key);
    } else { // key == root->key: 삭제할 노드를 찾음
        // Case 0 또는 Case 1: 자식이 없거나 하나만 있는 경우
        if (root->left == nullptr) {
            BSTNode* temp = root->right;
            delete root; // 현재 노드(root) 삭제
            return temp; // 오른쪽 자식을 부모에게 연결 (temp가 nullptr일 수도 있음)
        } else if (root->right == nullptr) {
            BSTNode* temp = root->left;
            delete root; // 현재 노드(root) 삭제
            return temp; // 왼쪽 자식을 부모에게 연결
        }

        // Case 2: 두 자식을 모두 가진 노드
        // 1. Inorder Successor (오른쪽 서브트리에서 가장 작은 값) 찾기
        BSTNode* successor = root->right;
        while (successor->left != nullptr) {
            successor = successor->left;
        }

        // 2. Inorder Successor의 키를 현재 노드(삭제 대상)로 복사
        root->key = successor->key;

        // 3. Inorder Successor 노드를 그 원래 위치에서 삭제
        //    이제 오른쪽 서브트리에서 successor->key를 가진 노드를 삭제하는 문제가 됨
        root->right = deleteNode(root->right, successor->key);
    }
    return root; // 변경된 서브트리의 루트 반환
}
```

**구체적 예시**:
슬라이드의 예시는 다음과 같습니다.
*   **초기 BST**: 루트 노드 13을 중심으로 왼쪽 서브트리 {3, 5, 8}, 오른쪽 서브트리 {17, 18, 19}가 구성되어 있습니다.
*   **삭제 대상**: 노드 13 (`Delete(13, D)`). (슬라이드 하단의 `Delete(12, D)`는 13의 오타로 보입니다.) 노드 13은 왼쪽 자식 5와 오른쪽 자식 17을 가지고 있으므로, 두 자식을 가진 노드입니다.
*   **Inorder Successor 찾기**:
    1.  13의 오른쪽 자식인 17로 이동합니다.
    2.  17은 왼쪽 자식이 없습니다. 따라서 17이 13의 Inorder Successor입니다. (슬라이드 그림의 파란색 화살표와 점선 원이 이를 가리킵니다.)
*   **값 대체**: 노드 13의 키를 17로 변경합니다. (즉, 원래 13이었던 노드는 이제 값 17을 가지게 됩니다.)
*   **Inorder Successor 노드 삭제**: 원래 17이 있던 위치에서 노드 17을 삭제합니다.
    *   원래 노드 17은 오른쪽 자식 19를 가지고 있습니다 (즉, Degree 1 노드입니다).
    *   따라서 17의 자리에 19가 올라와 17의 부모(원래는 13이었지만, 이제 그 자리는 17로 대체된 노드가 되므로 19는 17로 대체된 노드의 오른쪽 자식이 됩니다)와 연결됩니다.

**슬라이드 하단의 'Note' 설명**:
"The smallest key in the right subtree must be in a `leaf` (like in this example) or `degree 1` node (like in the next example)"
*   이는 Inorder Successor(`S`)의 중요한 특징을 설명합니다. `S`는 BST의 정의상 항상 왼쪽 자식이 없어야 합니다 (만약 왼쪽 자식이 있다면 그 자식이 `S`보다 작아 `S`가 가장 작은 값이 아니게 되기 때문). 따라서 `S`는 반드시 잎 노드(자식이 없는 노드)이거나 오른쪽 자식만 있는 노드(하나의 자식을 가진 노드) 중 하나가 됩니다.
*   슬라이드 그림에서 Inorder Successor인 17은 오른쪽 자식 19를 가지고 있으므로, Degree 1 노드입니다. 따라서 "like in this example"은 `degree 1 node`에 해당하며, 슬라이드 내의 말풍선 "the smallest key in a leaf node"는 이 예시에서는 정확하지 않을 수 있는 일반적인 설명으로 해석해야 합니다.

**시험 포인트**:
*   ⭐ **BST에서 두 자식을 가진 노드를 삭제하는 방법**은 가장 복잡한 경우이며, BST 관련 시험에서 핵심적인 출제 포인트입니다. Inorder Successor (또는 Inorder Predecessor)를 활용한 교체 및 삭제 과정을 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **Inorder Successor의 특징**: Inorder Successor는 항상 왼쪽 자식이 없으므로, 삭제 시에는 자식이 없거나(Leaf Node) 오른쪽 자식만 있는(Degree 1 Node) 경우로 처리된다는 점을 기억하세요. 이는 Inorder Successor의 삭제가 항상 간단하다는 것을 의미하며, 전체 삭제 과정의 효율성을 높입니다.
*   주어진 BST에서 특정 노드를 삭제하는 과정을 그림으로 그리거나, 각 단계별 트리의 변화를 설명하는 문제가 나올 수 있습니다. 반드시 직접 손으로 그려보며 과정을 익혀야 합니다.
*   Inorder Successor를 사용하는 방법 외에 Inorder Predecessor(왼쪽 서브트리에서 가장 큰 값)를 사용하는 경우도 함께 알아두면 좋습니다.

## BST: Deg. 2 노드 삭제 (Delete(x, D) – From a Deg. 2 Node)

### 핵심 개념

이 슬라이드는 이진 탐색 트리(Binary Search Tree, BST)에서 두 개의 자식을 가진 노드(Degree 2 Node)를 삭제하는 방법을 설명합니다. BST에서 노드를 삭제하는 경우, 삭제될 노드의 자식 수에 따라 처리 방식이 달라집니다. 특히 두 개의 자식을 가진 노드를 삭제할 때는 BST의 속성(왼쪽 자식은 부모보다 작고, 오른쪽 자식은 부모보다 크다)을 유지하면서 트리를 재구성해야 합니다.

두 개의 자식을 가진 노드 `N`을 삭제하려면 다음 절차 중 하나를 따릅니다:
1.  **후행자(In-order Successor) 활용**: 노드 `N`의 오른쪽 서브트리에서 가장 작은 값을 찾아 `N`의 값으로 교체하고, 해당 후행자 노드를 원래 위치에서 삭제합니다. 후행자 노드는 항상 0개 또는 1개의 자식만을 가지므로 삭제가 더 용이합니다.
2.  **선행자(In-order Predecessor) 활용**: 노드 `N`의 왼쪽 서브트리에서 가장 큰 값을 찾아 `N`의 값으로 교체하고, 해당 선행자 노드를 원래 위치에서 삭제합니다. 선행자 노드도 항상 0개 또는 1개의 자식만을 가지므로 삭제가 더 용이합니다.

### 코드/수식 해설

BST 노드 삭제 함수는 재귀적으로 구현되는 경우가 많습니다. 다음은 후행자(in-order successor)를 사용하여 Degree 2 노드를 삭제하는 개념적인 단계입니다. 실제 코드 구현 시 `findMin` 또는 `findMax` 함수를 활용하여 후행자/선행자를 찾고, 해당 노드를 삭제하는 함수를 재귀적으로 호출합니다.

```c
// BST 노드 구조체 (개념적)
typedef struct Node {
    int key;
    struct Node *left;
    struct Node *right;
} Node;

// 오른쪽 서브트리에서 가장 작은 값(후행자)을 찾는 함수
Node* findMinNode(Node* node) {
    Node* current = node;
    while (current && current->left != NULL) {
        current = current->left;
    }
    return current;
}

// BST에서 특정 키를 가진 노드를 삭제하는 함수
Node* deleteNode(Node* root, int key) {
    // 1. 노드를 찾는다 (탐색 과정)
    if (root == NULL) {
        return root;
    }

    if (key < root->key) {
        root->left = deleteNode(root->left, key);
    } else if (key > root->key) {
        root->right = deleteNode(root->right, key);
    } else { // key == root->key, 이 노드를 삭제해야 한다.
        // Case 1: 자식이 없거나 하나만 있는 경우
        if (root->left == NULL) { // 오른쪽 자식만 있거나 자식이 없는 경우
            Node* temp = root->right;
            free(root);
            return temp;
        } else if (root->right == NULL) { // 왼쪽 자식만 있는 경우
            Node* temp = root->left;
            free(root);
            return temp;
        }

        // Case 2: 두 개의 자식을 가진 경우 (Degree 2 Node)
        // 오른쪽 서브트리에서 후행자(가장 작은 노드)를 찾는다.
        Node* temp = findMinNode(root->right);

        // 후행자의 값을 현재 노드에 복사한다.
        root->key = temp->key;

        // 후행자 노드를 원래 위치에서 삭제한다.
        root->right = deleteNode(root->right, temp->key);
    }
    return root;
}
```

### 구체적 예시

제시된 슬라이드의 트리와 연산을 따라가 봅시다.

**초기 트리 (Insert(14, D) & Insert(15, D) 연산 후):**

       /  \
      5    17
     / \   / \
    3   8 14  19
           \  /
            15 18

**삭제 과정 (후행자 활용):**

1.  **후행자 찾기**: 노드 `13`의 오른쪽 서브트리(루트 `17`)에서 가장 작은 값을 찾습니다.
    *   `17`에서 왼쪽으로 이동: `14`
    *   `14`에서 왼쪽으로 더 이상 이동할 수 없습니다. (`14`의 오른쪽 자식은 `15`입니다.)
    *   따라서 노드 `13`의 후행자는 `14`입니다.

2.  **값 교체**: 노드 `13`의 값을 후행자 `14`의 값으로 교체합니다.
    *   이제 루트 노드는 `14`가 됩니다.
    ```
            14  (원래 13 자리)
           /  \
          5    17
         / \   / \
        3   8 14  19  (원래 14 노드는 아직 존재)
               \  /
                15 18
    ```

3.  **후행자 노드 삭제**: 원래 위치에 있던 후행자 노드 `14`를 삭제합니다.
    *   원래 노드 `14`는 노드 `17`의 왼쪽 자식입니다.
    *   노드 `14`는 하나의 자식(`15`, 오른쪽 자식)을 가지고 있습니다.
    *   따라서 노드 `17`은 자신의 왼쪽 자식을 `14` 대신 `15`로 연결합니다. (즉, `17`의 왼쪽 자식 = `14`의 오른쪽 자식)

**삭제 후 최종 트리:**

```
        14  (원래 13이 14로 대체됨)
       /  \
      5    17
     / \   / \
    3   8 15  19
               /
              18
```

### 시험 포인트

*   ⭐ **BST 노드 삭제의 세 가지 경우**:
    1.  **리프 노드(Leaf Node)**: 자식이 없는 노드는 단순히 제거합니다.
    2.  **하나의 자식을 가진 노드**: 해당 노드를 제거하고, 그 자식을 부모 노드에 직접 연결합니다.
    3.  **두 개의 자식을 가진 노드(Degree 2 Node)**: 이 슬라이드의 핵심 내용으로, `후행자(in-order successor)` 또는 `선행자(in-order predecessor)`를 찾아 값을 교체한 후, 원래 위치의 후행자/선행자 노드를 삭제합니다.

*   ⭐ **Degree 2 노드 삭제 시 후행자/선행자 선택**: 둘 중 어느 것을 사용해도 BST 속성을 유지할 수 있으며, 일반적으로 후행자를 많이 사용합니다.
    *   **후행자**: 삭제할 노드의 오른쪽 서브트리에서 가장 작은 노드
    *   **선행자**: 삭제할 노드의 왼쪽 서브트리에서 가장 큰 노드

*   **시간 복잡도**: BST에서의 `Delete` 연산은 노드를 탐색하고 (필요한 경우) 후행자/선행자를 찾는 과정이 포함되므로, 트리의 높이($h$)에 비례합니다. 최악의 경우 $O(h)$, 평균적으로 $O(\log N)$ (균형 잡힌 트리의 경우).

*   **BST 속성 유지**: 삭제 연산 후에도 모든 노드가 이진 탐색 트리의 규칙(왼쪽 자식 < 부모 < 오른쪽 자식)을 만족하는지 확인해야 합니다. 후행자/선행자를 이용한 방법은 이 속성을 자동으로 유지합니다.

### **핵심 개념**

이 슬라이드는 이진 탐색 트리(Binary Search Tree, BST)에서 **두 개의 자식을 가진 노드(Degree 2 Node)를 삭제하는 과정**을 설명합니다. BST의 삭제 연산은 크게 세 가지 경우로 나뉩니다:
1.  **자식이 없는 리프 노드 삭제**: 해당 노드를 단순히 제거합니다.
2.  **하나의 자식을 가진 노드 삭제**: 해당 노드를 제거하고, 그 자식 노드를 삭제된 노드의 부모와 직접 연결합니다.
3.  **두 개의 자식을 가진 노드 삭제 (Degree 2 Node)**: 이 경우가 가장 복잡하며, 슬라이드에서 다루는 내용입니다. 이 경우, 삭제할 노드의 값을 **"in-order successor(중위 순회 계승자)"**의 값으로 대체한 후, in-order successor를 원래 위치에서 삭제합니다.
    *   **In-order Successor**: 삭제할 노드의 오른쪽 서브트리에서 가장 작은 값을 가진 노드를 의미합니다. 이 in-order successor는 항상 자식이 없거나(리프 노드) 오른쪽 자식 하나만 가지는(Degree 1 노드) 특성을 가집니다. 따라서, in-order successor를 삭제하는 과정은 위 1번 또는 2번의 더 간단한 경우로 귀결됩니다.

### **코드/수식 해설**

BST에서 두 자식을 가진 노드 `N`을 삭제하는 `Delete(N)` 함수의 핵심 로직은 다음과 같습니다.

```cpp
// BST 노드 구조체 예시
struct Node {
    int key;
    Node* left;
    Node* right;
};

// 오른쪽 서브트리에서 가장 작은 키(in-order successor)를 찾는 함수
Node* findMin(Node* node) {
    Node* current = node;
    while (current && current->left != nullptr) {
        current = current->left;
    }
    return current;
}

// BST에서 특정 키를 가진 노드를 삭제하는 함수 (재귀 버전)
Node* deleteNode(Node* root, int key) {
    if (root == nullptr) {
        return root;
    }

    // 1. 삭제할 노드(key)를 찾는다
    if (key < root->key) {
        root->left = deleteNode(root->left, key);
    } else if (key > root->key) {
        root->right = deleteNode(root->right, key);
    } else { // key == root->key, 이 노드를 삭제해야 함
        // 2. 자식이 없거나 하나인 경우
        if (root->left == nullptr) {
            Node* temp = root->right;
            delete root;
            return temp;
        } else if (root->right == nullptr) {
            Node* temp = root->left;
            delete root;
            return temp;
        }

        // 3. 두 개의 자식을 가진 경우 (Degree 2 Node)
        // 오른쪽 서브트리에서 가장 작은 노드 (in-order successor)를 찾는다
        Node* temp = findMin(root->right);

        // in-order successor의 키를 현재 노드로 복사한다
        root->key = temp->key;

        // in-order successor를 원래 위치에서 삭제한다
        root->right = deleteNode(root->right, temp->key);
    }
    return root;
}
```

### **구체적 예시**

슬라이드의 예시를 따라 노드 13을 삭제하는 과정을 설명합니다.

**초기 BST 상태:**
```
       13 (삭제 대상)
      /  \
     5    17
    / \   / \
   3   8 14  19
         /   /
        15  18
```

1.  **삭제할 노드 식별**: 노드 `13`을 삭제 대상으로 찾습니다.
2.  **노드 유형 확인**: 노드 `13`은 왼쪽 자식(`5`)과 오른쪽 자식(`17`)을 모두 가지고 있으므로, 두 자식을 가진 노드(Degree 2 Node)입니다.
3.  **In-order Successor 찾기**: 노드 `13`의 오른쪽 서브트리(`17`을 루트로 하는 서브트리)에서 가장 작은 값을 가진 노드를 찾습니다.
    *   `17`에서 왼쪽으로 내려가면 `14`가 있습니다.
    *   `14`에서 더 이상 왼쪽 자식이 없으므로, `14`가 in-order successor입니다.
    *   슬라이드에서 `14`가 초록색으로 표시되어 있으며 "the smallest key in degree 1 node"로 설명됩니다. 이는 `14`가 오른쪽 서브트리에서 가장 작은 키이면서, 동시에 하나의 자식(`15`)만을 가진 노드임을 의미합니다.
4.  **값 대체**: 삭제할 노드 `13`의 값을 in-order successor인 `14`의 값으로 대체합니다.
    ```
           14 (새로운 값)
          /  \
         5    17
        / \   / \
       3   8 [14] 19  <- 원래 14의 위치
             /   /
            15  18
    ```
5.  **In-order Successor 삭제**: 원래 `14`가 있던 위치에서 `14` 노드를 삭제합니다.
    *   `14` 노드는 하나의 자식(`15`)만 가지고 있습니다.
    *   `14`를 삭제하고, `14`의 오른쪽 자식인 `15`를 `14`의 부모(`17`)와 직접 연결합니다.
    *   (만약 `14`가 자식이 없었다면 단순히 `14`를 제거합니다.)

**최종 BST 상태:**
```
       14
      /  \
     5    17
    / \   / \
   3   8 15  19
             /
            18
```

### **시험 포인트**

*   ⭐ **BST 삭제의 세 가지 케이스를 정확히 이해하고 설명할 수 있어야 합니다.** 특히 두 자식을 가진 노드 삭제 시의 전략이 중요합니다.
*   ⭐ **In-order successor의 개념과 이를 찾는 방법(오른쪽 서브트리에서 가장 작은 값)을 알아야 합니다.** 또한, in-order successor가 항상 자식이 없거나 하나의 자식만 가지는 노드임을 이해해야 합니다.
*   ⭐ **주어진 BST에 대해 특정 노드를 삭제하는 과정을 단계별로 시뮬레이션할 수 있어야 합니다.** 특히, Degree 2 노드 삭제 시 in-order successor를 찾아 대체하고 삭제하는 과정을 그림으로 그리거나 설명할 수 있어야 합니다.
*   삭제된 노드의 값을 대체할 때 in-order successor 대신 **in-order predecessor (중위 순회 선행자)**를 사용할 수도 있습니다 (왼쪽 서브트리에서 가장 큰 값). 이 두 가지 방법 모두 올바른 BST를 유지합니다. 어떤 것을 선택하든 일관성이 중요합니다. (이 슬라이드는 in-order successor를 사용합니다.)
*   ⭐ **각 삭제 연산의 시간 복잡도는 트리의 높이에 비례합니다.** 최악의 경우 $O(h)$ (h는 트리의 높이), 균형 잡힌 트리의 경우 $O(\log n)$ (n은 노드의 개수)입니다.

## Slide 28

### 핵심 개념

이 슬라이드는 **이진 탐색 트리(Binary Search Tree, BST)** 에서 **두 개의 자식을 가진 노드(Degree 2 Node)** 를 삭제하는 과정을 보여줍니다. 핵심은 삭제할 노드의 키를 **오른쪽 서브트리에서 가장 작은 키(Inorder Successor)** 로 교체한 후, Inorder Successor를 그 원래 위치에서 삭제하는 것입니다. Inorder Successor는 항상 0개 또는 1개의 자식만을 가지므로, 이 경우에 맞는 삭제 절차를 따르게 됩니다.

### 코드/수식 해설

BST에서 키 $x$를 가진 노드 $N$을 삭제하는 일반적인 알고리즘은 다음과 같습니다.

1.  **노드 $N$ 찾기**: 트리를 순회하여 키 $x$를 가진 노드 $N$을 찾습니다.
2.  **삭제할 노드 $N$의 자식 수에 따른 처리**:
    *   **자식이 없는 경우 (Degree 0)**: 단순히 $N$을 삭제하고 부모 노드의 해당 자식 포인터를 `NULL`로 설정합니다.
    *   **자식이 하나인 경우 (Degree 1)**: $N$을 삭제하고 $N$의 유일한 자식을 $N$의 부모에 연결합니다.
    *   **자식이 두 개인 경우 (Degree 2)**:
        a.  $N$의 **Inorder Successor** $S$를 찾습니다. $S$는 $N$의 오른쪽 서브트리에서 가장 작은 키를 가진 노드입니다.
        b.  $N$의 키를 $S$의 키로 **교체**합니다.
        c.  이제 $S$를 그 원래 위치($N$의 오른쪽 서브트리 내)에서 **삭제**합니다. 이때 $S$는 Inorder Successor의 정의에 따라 항상 왼쪽 자식이 없으므로, 0개 또는 1개의 자식을 가진 노드 삭제 규칙을 적용할 수 있습니다. 즉, $S$의 오른쪽 자식을 $S$의 부모에 연결하고 $S$를 제거합니다.

**시간 복잡도**:
BST에서 노드를 검색하고, Inorder Successor를 찾아 이동하며, 최종적으로 노드를 삭제하는 모든 과정은 트리의 높이($h$)에 비례합니다.
따라서, 이 과정의 시간 복잡도는 다음과 같습니다.
$$O(h)$$
균형 잡힌 BST(예: AVL 트리, Red-Black 트리)의 경우 $h = \log N$ (여기서 $N$은 트리의 노드 수)이므로 $O(\log N)$이 됩니다. 그러나 최악의 경우(사향 트리) $h=N$이 될 수 있어 $O(N)$이 될 수도 있습니다.

### 구체적 예시

슬라이드에 제시된 트리에서 노드 `14` (루트 노드)를 삭제하는 과정을 통해 Inorder Successor를 이용한 삭제를 설명합니다. (슬라이드 하단의 "After Delete(13, D)"는 `14`를 삭제하는 시각화와 일치하지 않으므로, 시각화를 기준으로 `Delete(14, D)`로 간주합니다.)

1.  **삭제할 노드 식별**: 현재 루트 노드인 `14`를 삭제하려 합니다. `14`는 왼쪽 자식 `5`와 오른쪽 자식 `17`을 가지므로, 두 개의 자식을 가진 노드입니다.
2.  **Inorder Successor 찾기**: `14`의 오른쪽 서브트리(루트 `17`)에서 가장 작은 키를 찾습니다. `17`의 왼쪽 자식을 따라 내려가면 `15`를 찾을 수 있습니다. 따라서 `15`가 `14`의 Inorder Successor입니다.
3.  **키 교체**: `14`의 키를 `15`로 교체합니다. 이제 루트 노드의 키는 `15`가 됩니다.
4.  **Inorder Successor 원래 위치에서 삭제**: 이제 원래 `15`가 있던 위치, 즉 `17`의 왼쪽 자식인 `15`를 삭제해야 합니다. `15`는 자식이 없는 노드(Degree 0)입니다. 따라서 `17`의 왼쪽 자식 포인터를 `NULL`로 설정하여 `15`를 트리에서 제거합니다.

이 과정을 거치면, 노드 `14`는 `15`로 대체되고, 원래 `15`의 자리는 비워지며 트리의 BST 속성은 유지됩니다.

### 시험 포인트

*   ⭐ **두 개의 자식을 가진 노드 삭제 전략**: BST에서 두 자식을 가진 노드를 삭제할 때, 해당 노드를 Inorder Successor(오른쪽 서브트리의 최소값) 또는 Inorder Predecessor(왼쪽 서브트리의 최대값)로 교체하고, 교체된 노드를 원래 위치에서 삭제하는 과정을 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **Inorder Successor의 특징**: Inorder Successor는 항상 왼쪽 자식을 가지지 않는다는 점을 기억해야 합니다. 이 특징 덕분에 Inorder Successor를 그 원래 위치에서 삭제할 때는 자식이 0개 또는 1개인 노드를 삭제하는 간단한 규칙이 적용됩니다.
*   ⭐ **시간 복잡도**: BST의 삽입, 삭제, 검색 연산의 시간 복잡도가 트리의 높이($h$)에 비례하여 $O(h)$임을 숙지해야 합니다. 균형 잡힌 BST와 사향 트리의 경우를 구분하여 설명할 수 있어야 합니다.
*   ⭐ 슬라이드의 "After Delete(13, D)" 문구는 시각화와 모순되는 명백한 **오타**입니다. 실제 시각화는 `Delete(14, D)` 과정을 보여주고 있습니다. 시험에서 이와 유사한 불일치가 발생할 경우, 맥락을 파악하여 정확한 과정을 유추하거나 명확한 질문을 할 수 있는 능력이 중요합니다.

### **핵심 개념**

*   **이진 탐색 트리 (Binary Search Tree, BST) 속성**:
    *   모든 노드 `N`에 대해, `N`의 왼쪽 서브트리에 있는 모든 키는 `N`의 키보다 작고, `N`의 오른쪽 서브트리에 있는 모든 키는 `N`의 키보다 큽니다.
    *   이 속성은 삽입, 삭제 등의 연산 후에도 반드시 유지되어야 합니다.
*   **BST 노드 삭제 (Deletion in BST)**:
    *   **자식이 없는 노드**: 해당 노드를 단순히 제거합니다.
    *   **자식이 하나인 노드**: 해당 노드를 제거하고, 그 자식을 노드의 부모에 직접 연결합니다.
    *   **자식이 둘인 노드**: 이 경우가 가장 복잡합니다. 일반적으로 두 가지 방법이 있습니다.
        1.  **Inorder Successor (오른쪽 서브트리에서 가장 작은 키)**로 교체: 삭제할 노드의 오른쪽 서브트리에서 가장 작은 키를 찾아(이는 inorder successor입니다) 삭제할 노드의 키와 교체한 후, 원래 inorder successor가 있던 위치에서 해당 노드를 삭제합니다.
        2.  **Inorder Predecessor (왼쪽 서브트리에서 가장 큰 키)**로 교체: 삭제할 노드의 왼쪽 서브트리에서 가장 큰 키를 찾아(이는 inorder predecessor입니다) 삭제할 노드의 키와 교체한 후, 원래 inorder predecessor가 있던 위치에서 해당 노드를 삭제합니다.
*   **Inorder Successor의 특성**: 어떤 노드의 inorder successor는 그 노드의 오른쪽 서브트리에서 가장 왼쪽에 있는 노드입니다. 이 노드는 왼쪽 자식을 가질 수 없습니다 (만약 왼쪽 자식이 있다면 그 자식이 더 작은 키를 가지므로 inorder successor가 아닐 것). 따라서 inorder successor는 0개 또는 1개의 (오른쪽) 자식만을 가집니다.

---

### **코드/수식 해설**

이 슬라이드에는 직접적인 코드나 수식은 없지만, BST 삭제 로직의 핵심은 다음과 같이 개념적으로 표현할 수 있습니다.

**두 자식을 가진 노드 `N`을 Inorder Successor `S`로 교체하여 삭제하는 과정:**

1.  `N`의 오른쪽 서브트리에서 가장 작은 키를 가진 노드 `S`를 찾습니다. (`S`는 `N`의 inorder successor입니다.)
2.  `N`의 키를 `S`의 키로 변경합니다. (이제 `N`은 원래 `S`가 가지고 있던 키를 가집니다.)
3.  원래 `S`가 있던 위치에서 `S` 노드를 삭제합니다. `S`는 최대 하나의 자식(오른쪽 자식)만 가질 수 있으므로, 이 삭제는 자식이 0개 또는 1개인 노드를 삭제하는 간단한 경우에 해당합니다.

### **구체적 예시**

다음과 같은 BST가 있다고 가정합시다. `50` 노드를 삭제하려고 합니다.

```
       50
      /  \
     30   70
    / \   / \
   20 40 60  80
           \
            65
```

1.  **삭제할 노드**: `50` (두 자식을 가짐: `30`, `70`).
2.  **오른쪽 서브트리**: `70`을 루트로 하는 서브트리.
3.  **Inorder Successor `S` 찾기**: `70` 서브트리에서 가장 작은 키는 `60`입니다. (`S = 60`).
4.  **키 교체**: `50`의 키를 `60`으로 바꿉니다.
    ```
           60 (<- 원래 50)
          /  \
         30   70
        / \   / \
       20 40 60  80 (<- 원래 60)
              \
               65
    ```
5.  **`S` (원래 60) 삭제**: 원래 `60` 노드는 오른쪽 자식 `65`를 가집니다. `60`을 삭제하고 `60`의 부모 (`70`)가 `65`를 가리키도록 연결합니다.
    ```
           60
          /  \
         30   70
        / \   / \
       20 40 65  80 (<- 60이 65로 대체됨)
    ```
최종적으로 트리는 여전히 BST 속성을 유지합니다.

### **시험 포인트**

*   ⭐ **BST 삭제 시 두 자식을 가진 노드를 처리하는 방법을 정확히 이해해야 합니다.** 특히 inorder successor (또는 predecessor)를 활용하는 과정과 그 이유를 설명할 수 있어야 합니다.
*   ⭐ **삭제 후 BST 속성 유지에 대한 이해**:
    *   질문: "삭제 후 BST 속성을 복원하기 위해 다시 최소값을 찾아야 할 필요가 있습니까? 왜 그렇습니까/아닙니까?"
    *   **정답**: **아닙니다 (No).**
    *   **이유**:
        1.  **Inorder Successor `S`의 BST 속성 유지**: `S`는 삭제될 노드 `N`의 inorder successor이므로, `N`의 왼쪽 서브트리에 있는 모든 키보다 크고, `N`의 오른쪽 서브트리 중 `S` 자신보다 작은 키는 없습니다. 따라서 `N`의 위치로 `S`가 옮겨가면, `S`를 기준으로 왼쪽 서브트리와 오른쪽 서브트리의 BST 속성이 그대로 유지됩니다.
        2.  **`S` 삭제의 단순성**: `S`는 항상 0개 또는 1개의 (오른쪽) 자식만 가집니다. 이러한 노드를 삭제하는 것은 BST 속성을 깨뜨리지 않고 간단하게 처리됩니다. `S`의 부모는 `S`의 유일한 자식(있다면)을 가리키도록 업데이트되므로, 이 과정에서 트리의 다른 부분에 대한 추가적인 최소값 검색이나 복구 작업은 필요 없습니다.

*   ⭐ 이 문제는 BST의 기본 정의와 핵심 연산 중 하나인 삭제 연산의 상세 절차 및 그 과정에서 트리의 불변성(invariant)이 어떻게 유지되는지에 대한 깊이 있는 이해를 요구합니다. Inorder successor의 특성과 BST 속성을 연결하여 설명할 수 있는 능력이 중요합니다.

## 데이터구조 (CSED233) 슬라이드 노트 - 이진 탐색 트리 (BST) 노드 삭제

이번 슬라이드의 퀴즈는 이진 탐색 트리(BST)에서 두 개의 자식을 가진 노드를 삭제하는 과정에 대한 핵심 질문입니다. 특히, 삭제될 노드를 대체하는 '오른쪽 서브트리에서 가장 작은 키를 가진 노드'의 특성에 대한 이해를 묻고 있습니다.

---

**핵심 개념**:

이 질문은 이진 탐색 트리(BST)에서 노드를 삭제하는 표준 알고리즘 중 하나인 **"in-order successor(중위 순회 계승자)"**를 이용한 삭제 방법에 대한 이해를 요구합니다.

1.  **두 개의 자식을 가진 노드 삭제**: BST에서 삭제할 노드($N$)가 두 개의 자식(왼쪽 자식과 오른쪽 자식)을 가지고 있을 때, 단순히 노드를 제거하면 트리의 구조가 깨지게 됩니다. 이 경우, 트리의 BST 속성을 유지하면서 $N$을 대체할 적절한 노드를 찾아야 합니다.
2.  **대체 노드 선택**: $N$을 대체할 노드로는 다음 두 가지 중 하나를 선택할 수 있습니다.
    *   오른쪽 서브트리에서 가장 작은 키를 가진 노드 (In-order successor)
    *   왼쪽 서브트리에서 가장 큰 키를 가진 노드 (In-order predecessor)
    이 퀴즈는 첫 번째 경우, 즉 오른쪽 서브트리에서 가장 작은 키를 가진 노드를 선택했을 때의 특성을 묻습니다.
3.  **오른쪽 서브트리에서 가장 작은 키를 가진 노드의 특성**:
    *   오른쪽 서브트리에서 가장 작은 키를 가진 노드($S$)는 **삭제될 노드($N$)의 오른쪽 자식 노드부터 시작하여 왼쪽 자식 노드를 따라 계속 내려가서 찾은 가장 마지막 노드**입니다.
    *   이 노드($S$)는 BST의 속성(왼쪽 자식은 부모보다 작고, 오른쪽 자식은 부모보다 크다)에 따라 **왼쪽 자식을 가질 수 없습니다.** 만약 $S$가 왼쪽 자식을 가졌다면, 그 왼쪽 자식은 $S$보다 작은 값을 가지므로 $S$가 '오른쪽 서브트리에서 가장 작은 노드'라는 정의에 모순되기 때문입니다.
    *   그러나 $S$는 **오른쪽 자식을 가질 수 있습니다.** $S$의 오른쪽 자식은 $S$보다 큰 값을 가지며, $S$가 대체 노드로 사용된 후에도 $S$의 오른쪽 서브트리로서 유지될 수 있습니다.
    *   따라서, 이 대체 노드($S$)는 **최대 한 개의 자식(오직 오른쪽 자식만)**을 가질 수 있습니다.

---

**코드/수식 해설**:

오른쪽 서브트리에서 가장 작은 키를 가진 노드(In-order successor)를 찾는 과정은 다음과 같은 재귀 함수나 반복문으로 구현할 수 있습니다.

```cpp
// BST 노드 구조체 (간략화)
struct Node {
    int key;
    Node* left;
    Node* right;
};

// 특정 노드(root)의 오른쪽 서브트리에서 가장 작은 노드를 찾는 함수
Node* findMin(Node* root) {
    if (root == nullptr) {
        return nullptr;
    }
    Node* current = root;
    // 오른쪽 서브트리에서 가장 작은 값은 왼쪽 자식 노드를 따라 끝까지 내려간 노드
    while (current->left != nullptr) {
        current = current->left;
    }
    return current;
}

// 두 자식을 가진 노드 'target'을 삭제하는 로직의 일부
// 'target' 노드의 오른쪽 자식(target->right)에서 findMin을 호출하여 successor를 찾음.
// Node* successor = findMin(target->right);
```

이 `findMin` 함수에 의해 반환된 `successor` 노드가 바로 퀴즈에서 말하는 'replacement node'입니다. 이 `successor`는 `while (current->left != nullptr)` 루프가 끝난 지점에 도달했으므로, 더 이상 왼쪽 자식을 가질 수 없습니다. 즉, `successor->left`는 반드시 `nullptr`입니다.

---

**구체적 예시**:

다음과 같은 BST가 있다고 가정해 봅시다. 여기서 20 노드를 삭제하고 싶습니다.

```
      30
     /  \
    10   20  <- 삭제할 노드
     \     \
      15    25
           /  \
          22   28
```

1.  **삭제할 노드**: 20 (두 개의 자식: 15, 25)
2.  **오른쪽 서브트리**: 20의 오른쪽 자식인 25를 루트로 하는 서브트리입니다.
    ```
       25
      /  \
     22   28
    ```
3.  **오른쪽 서브트리에서 가장 작은 키**: 이 서브트리에서 가장 작은 키를 찾으려면, 25부터 시작하여 왼쪽으로 내려갑니다. 그러면 22를 찾게 됩니다.
4.  **대체 노드**: 22.
    *   22는 왼쪽 자식이 있나요? 아니요, 없습니다. (`22->left == nullptr`)
    *   22는 오른쪽 자식이 있나요? 아니요, 없습니다. (`22->right == nullptr`) (이 예시에서는 없지만, 예를 들어 22 아래에 23이 있다면 22의 오른쪽 자식이 23이 될 수 있습니다.)

따라서 22 노드는 자식이 0개 (최대 한 개)인 노드입니다. 20을 22로 대체한 후, 22는 그 원래 위치에서 제거되고 20의 자리에 들어갑니다. 22의 원래 부모(이 경우 25)는 22의 오른쪽 자식(만약 있다면)을 자신의 새로운 왼쪽 자식으로 연결합니다. 이 예시에서는 22의 자식이 없으므로, 25는 단순히 왼쪽 자식으로 `nullptr`을 가지게 됩니다.

---

**시험 포인트**:

*   ⭐ **BST 노드 삭제 알고리즘**: 특히 두 자식을 가진 노드 삭제 시 대체 노드를 찾는 방법(in-order successor 또는 predecessor)과 그 대체 노드가 가지는 특성을 정확히 이해해야 합니다.
*   ⭐ **In-order successor의 특성**: `findMin` 함수를 통해 오른쪽 서브트리에서 가장 작은 키를 찾으면, 그 노드는 **반드시 왼쪽 자식이 없다는 것**을 기억하세요. 이는 해당 노드를 원래 위치에서 제거할 때 추가적인 복잡성 없이 최대 한 개의 자식(오른쪽 자식)만 처리하면 되기 때문에 삭제 과정을 간소화합니다.
*   삭제된 노드의 자리에는 대체 노드가 들어가고, 대체 노드가 원래 있던 자리에서는 그 노드를 제거해야 합니다. 이때 대체 노드의 오른쪽 자식이 있다면, 대체 노드의 부모가 그 오른쪽 자식을 연결하도록 처리해야 합니다.
*   이 질문은 BST의 기본 정의와 속성(특히 순서 유지)을 깊이 이해하고 있는지를 확인하는 중요한 문제입니다.

## Slide 31

이 슬라이드는 기존 이진 트리(Binary Tree)의 구조를 유지하면서 값을 재배치하여 이진 탐색 트리(Binary Search Tree, BST)로 변환하는 방법을 설명합니다.

---

### **핵심 개념**

기존 이진 트리를 이진 탐색 트리로 변환하는 목표는 **트리의 물리적 구조(노드 간의 연결)는 그대로 유지하되, 각 노드의 값만 변경하여 이진 탐색 트리의 속성(좌측 서브트리의 모든 값은 노드 값보다 작고, 우측 서브트리의 모든 값은 노드 값보다 크다)을 만족**하도록 하는 것입니다.

이를 위한 일반적인 알고리즘은 다음과 같습니다:
1.  **값 추출**: 주어진 이진 트리를 중위 순회(in-order traversal)하여 모든 노드의 값을 순서대로 리스트에 저장합니다.
2.  **값 정렬**: 추출된 값들을 오름차순으로 정렬합니다.
3.  **값 재배치**: 다시 이진 트리를 중위 순회하면서, 정렬된 리스트의 값들을 순서대로 각 노드에 할당합니다. 이진 탐색 트리를 중위 순회하면 값이 오름차순으로 나오기 때문에, 정렬된 값들을 중위 순회 순서로 할당하면 자연스럽게 이진 탐색 트리가 됩니다.

### **코드/수식 해설**

**1. 값 추출 (중위 순회)**

```c++
void inorderTraversal(TreeNode* node, std::vector<int>& values) {
    if (node == nullptr) {
        return;
    }
    inorderTraversal(node->left, values);
    values.push_back(node->val);
    inorderTraversal(node->right, values);
}
```

**2. 값 재배치 (정렬된 값으로 중위 순회)**

```c++
void convertToBST(TreeNode* node, const std::vector<int>& sorted_values, int& index) {
    if (node == nullptr) {
        return;
    }
    convertToBST(node->left, sorted_values, index);
    node->val = sorted_values[index]; // 정렬된 값을 현재 노드에 할당
    index++;
    convertToBST(node->right, sorted_values, index);
}
```

**전체 과정의 복잡도 분석:**
*   **시간 복잡도**:
    *   값 추출: $O(N)$ (N은 노드의 개수)
    *   값 정렬: $O(N \log N)$ (퀵 정렬, 병합 정렬 등)
    *   값 재배치: $O(N)$
    *   총 시간 복잡도: ⭐$O(N \log N)$ (정렬 단계가 지배적)
*   **공간 복잡도**:
    *   값을 저장할 리스트: $O(N)$
    *   재귀 호출 스택: 최악의 경우 $O(N)$ (skewed tree)
    *   총 공간 복잡도: ⭐$O(N)$

### **구체적 예시**

슬라이드에 제시된 트리를 예시로 들어봅시다.

**원본 트리 (왼쪽):**
```
      15
     /  \
    5    7
   / \
  8   6
```
(노드 15의 왼쪽 자식은 5, 오른쪽 자식은 7. 노드 5의 왼쪽 자식은 8, 오른쪽 자식은 6.)

1.  **값 추출 (중위 순회):**
    *   `inorder(15)` 호출
        *   `inorder(5)` 호출
            *   `inorder(8)` 호출
                *   `inorder(null)` 반환
                *   값 `8`을 리스트에 추가 (리스트: `[8]`)
                *   `inorder(null)` 반환
            *   값 `5`를 리스트에 추가 (리스트: `[8, 5]`)
            *   `inorder(6)` 호출
                *   `inorder(null)` 반환
                *   값 `6`을 리스트에 추가 (리스트: `[8, 5, 6]`)
                *   `inorder(null)` 반환
        *   값 `15`를 리스트에 추가 (리스트: `[8, 5, 6, 15]`)
        *   `inorder(7)` 호출
            *   `inorder(null)` 반환
            *   값 `7`을 리스트에 추가 (리스트: `[8, 5, 6, 15, 7]`)
            *   `inorder(null)` 반환
    *   최종 추출된 값 리스트: `[8, 5, 6, 15, 7]`

2.  **값 정렬:**
    *   `[8, 5, 6, 15, 7]`을 정렬하면 `[5, 6, 7, 8, 15]`가 됩니다.

3.  **값 재배치 (중위 순회):**
    정렬된 리스트 `[5, 6, 7, 8, 15]`를 사용하여 다시 중위 순회하며 값을 할당합니다.
    *   `convertToBST(15)` (루트)
        *   `convertToBST(5)` (15의 왼쪽 자식)
            *   `convertToBST(8)` (5의 왼쪽 자식)
                *   `convertToBST(null)` 반환
                *   노드 `8`에 `sorted_values[0]`인 `5` 할당. (이제 `8` 노드의 값은 `5`.) `index`는 `1`.
                *   `convertToBST(null)` 반환
            *   노드 `5`에 `sorted_values[1]`인 `6` 할당. (이제 `5` 노드의 값은 `6`.) `index`는 `2`.
            *   `convertToBST(6)` (5의 오른쪽 자식)
                *   `convertToBST(null)` 반환
                *   노드 `6`에 `sorted_values[2]`인 `7` 할당. (이제 `6` 노드의 값은 `7`.) `index`는 `3`.
                *   `convertToBST(null)` 반환
        *   노드 `15`에 `sorted_values[3]`인 `8` 할당. (이제 `15` 노드의 값은 `8`.) `index`는 `4`.
        *   `convertToBST(7)` (15의 오른쪽 자식)
            *   `convertToBST(null)` 반환
            *   노드 `7`에 `sorted_values[4]`인 `15` 할당. (이제 `7` 노드의 값은 `15`.) `index`는 `5`.
            *   `convertToBST(null)` 반환

**변환된 이진 탐색 트리:**
```
       8
      /  \
     6   15
    / \
   5   7
```

### **시험 포인트**

*   ⭐ **문제의 핵심 이해**: "Keep the structure same, but change the values only" 라는 제약 조건이 주어졌을 때, 이진 탐색 트리의 속성을 만족시키기 위한 논리적 접근 방식을 이해해야 합니다. 구조 변경 없이 값을 재배치하는 것이 중요합니다.
*   ⭐ **알고리즘 절차**:
    1.  원본 트리의 모든 값을 중위 순회로 수집.
    2.  수집된 값을 정렬.
    3.  정렬된 값을 다시 중위 순회로 트리에 할당.
    이 세 단계의 과정을 정확히 기억해야 합니다.
*   ⭐ **시간 및 공간 복잡도**: 변환 과정의 시간 복잡도가 왜 $O(N \log N)$ 이고 공간 복잡도가 $O(N)$ 인지 설명할 수 있어야 합니다. 특히 정렬이 전체 시간 복잡도를 지배하는 이유를 알아야 합니다.
*   **중위 순회의 중요성**: 이진 탐색 트리에서 중위 순회가 정렬된 순서로 노드를 방문한다는 속성을 활용하는 것이 이 문제 풀이의 핵심입니다.
*   주어진 예시와 같이 특정 트리에 대해 이 과정을 단계별로 적용하여 올바른 결과를 도출할 수 있어야 합니다.

## 데이터구조 (CSED233) 슬라이드 노트

### 핵심 개념

이 슬라이드는 **임의의 이진 트리(Binary Tree)를 이진 탐색 트리(Binary Search Tree, BST)로 변환하는 방법**을 다룹니다. 이때 중요한 제약 조건은 **트리의 구조(노드 간의 부모-자식 관계 및 형태)는 유지하되, 각 노드의 값만 변경**하여 BST의 속성을 만족시키는 것입니다.

BST의 핵심 속성은 다음과 같습니다:
1.  모든 노드의 왼쪽 서브트리에 있는 값은 해당 노드의 값보다 작습니다.
2.  모든 노드의 오른쪽 서브트리에 있는 값은 해당 노드의 값보다 큽니다.
3.  좌우 서브트리도 이진 탐색 트리입니다.

이 변환 과정은 다음 세 단계를 통해 이루어집니다.

1.  **중위 순회(Inorder Traversal)**: 주어진 이진 트리를 중위 순회하여 모든 노드의 값을 순서대로 리스트에 저장합니다.
2.  **리스트 정렬(Sort the Traversal List)**: 1단계에서 얻은 값들의 리스트를 오름차순으로 정렬합니다.
3.  **정렬된 값 복사(Copy Sorted List)**: 원본 트리의 구조에 대해 다시 중위 순회를 수행하면서, 2단계에서 정렬된 리스트의 값을 순서대로 각 노드에 할당(덮어쓰기)합니다.

이렇게 하면, 중위 순회는 BST에서 오름차순으로 정렬된 값을 얻는다는 속성과 결합되어, 기존 구조를 유지하면서 모든 노드에 BST 조건을 만족하는 값을 할당할 수 있게 됩니다.

### 코드/수식 해설

**1. 중위 순회 (Inorder Traversal) 및 값 저장**

```c
// 예시: 트리의 노드 구조
struct TreeNode {
    int val;
    struct TreeNode *left;
    struct TreeNode *right;
};

// 중위 순회 함수 (값을 리스트에 저장)
void inorderTraversal(struct TreeNode *root, int *values, int *index) {
    if (root == NULL) {
        return;
    }
    inorderTraversal(root->left, values, index); // 왼쪽 서브트리 방문
    values[(*index)++] = root->val;             // 현재 노드의 값 저장
    inorderTraversal(root->right, values, index); // 오른쪽 서브트리 방문
}
```

**2. 리스트 정렬**

일반적인 정렬 알고리즘을 사용합니다. 예를 들어, 퀵 정렬이나 병합 정렬을 사용할 수 있습니다.
$O(N \log N)$의 시간 복잡도를 가집니다. ($N$은 트리의 노드 개수)

```c
// C 표준 라이브러리의 qsort 함수 사용 예시
// int *values; // 1단계에서 얻은 값들의 배열
// int num_nodes; // 노드 개수
// qsort(values, num_nodes, sizeof(int), compareIntegers);
```

**3. 정렬된 값 복사 (두 번째 중위 순회)**

```c
// 두 번째 중위 순회 함수 (정렬된 값으로 노드 업데이트)
void copySortedValues(struct TreeNode *root, const int *sorted_values, int *index) {
    if (root == NULL) {
        return;
    }
    copySortedValues(root->left, sorted_values, index); // 왼쪽 서브트리 방문
    root->val = sorted_values[(*index)++];             // 현재 노드의 값 업데이트
    copySortedValues(root->right, sorted_values, index); // 오른쪽 서브트리 방문
}
```

### 구체적 예시

슬라이드에 제시된 예시를 통해 과정을 살펴봅시다.

**원본 이진 트리:**
```
     15
    /  \
   5    7
  / \
 8   6
```

1.  **중위 순회**: `inorderTraversal` 함수를 사용하여 노드의 값을 순서대로 리스트에 저장합니다.
    *   (8 방문) -> 리스트: `[8]`
    *   (5 방문) -> 리스트: `[8, 5]`
    *   (6 방문) -> 리스트: `[8, 5, 6]`
    *   (15 방문) -> 리스트: `[8, 5, 6, 15]`
    *   (7 방문) -> 리스트: `[8, 5, 6, 15, 7]`
    최종 리스트: `[8, 5, 6, 15, 7]`

2.  **리스트 정렬**: 위 리스트를 오름차순으로 정렬합니다.
    *   정렬된 리스트: `[5, 6, 7, 8, 15]`

3.  **정렬된 값 복사**: 원본 트리의 구조에 대해 다시 중위 순회를 수행하면서, 정렬된 리스트의 값을 순서대로 할당합니다.
    *   첫 번째로 방문되는 노드 (구조상 8이 있던 자리)에 정렬된 리스트의 첫 번째 값 `5`를 할당.
    *   두 번째로 방문되는 노드 (구조상 5가 있던 자리)에 정렬된 리스트의 두 번째 값 `6`을 할당.
    *   세 번째로 방문되는 노드 (구조상 6이 있던 자리)에 정렬된 리스트의 세 번째 값 `7`을 할당.
    *   네 번째로 방문되는 노드 (구조상 15가 있던 자리)에 정렬된 리스트의 네 번째 값 `8`을 할당.
    *   다섯 번째로 방문되는 노드 (구조상 7이 있던 자리)에 정렬된 리스트의 다섯 번째 값 `15`를 할당.

**변환된 이진 탐색 트리:**
```
     8
    /  \
   6    15
  / \
 5   7
```
이 트리는 이진 탐색 트리의 모든 속성을 만족합니다. (예: 8의 왼쪽 자식은 6 (8보다 작음), 8의 오른쪽 자식은 15 (8보다 큼), 6의 왼쪽 자식은 5, 오른쪽 자식은 7 등)

### 시험 포인트

*   ⭐ **이진 트리를 BST로 변환하는 핵심 아이디어**: 트리의 구조를 유지하면서 값만 변경해야 할 때, 중위 순회가 BST에서 오름차순으로 정렬된 값을 제공한다는 성질을 이용하는 것입니다.
*   ⭐ **변환 과정의 세 단계**: 중위 순회 -> 리스트 정렬 -> 두 번째 중위 순회 및 값 할당. 각 단계를 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **각 단계의 시간 복잡도**:
    *   첫 번째 중위 순회: $O(N)$ (N은 노드의 개수)
    *   리스트 정렬: $O(N \log N)$ (효율적인 정렬 알고리즘 사용 시)
    *   두 번째 중위 순회: $O(N)$
    *   따라서 전체 시간 복잡도는 $O(N \log N)$에 의해 지배됩니다.
*   ⭐ **공간 복잡도**: 노드의 값을 저장하기 위한 리스트에 $O(N)$의 공간이 필요합니다.
*   ⭐ **왜 중위 순회를 사용하는가?**: 중위 순회는 BST의 노드를 오름차순으로 방문합니다. 따라서 이 순서에 맞춰 정렬된 값을 할당하면 자연스럽게 BST 속성을 만족하게 됩니다. 다른 순회 방식(전위, 후위)으로는 이 속성을 보장하기 어렵습니다.

## 데이터구조 (CSED233) 슬라이드 노트: 이진 트리를 이진 탐색 트리로 변환

### 핵심 개념

이 슬라이드는 **기존 이진 트리의 구조는 그대로 유지한 채, 노드의 값만 변경하여 이진 탐색 트리(Binary Search Tree, BST)의 속성을 만족하도록 변환하는 방법**을 설명합니다. 이는 특정 구조를 가진 트리를 BST로 최적화해야 할 때 사용될 수 있습니다.

**이진 탐색 트리(BST)**의 핵심 속성은 다음과 같습니다:
1.  모든 노드의 왼쪽 서브트리에 있는 모든 노드의 값은 현재 노드의 값보다 작거나 같습니다.
2.  모든 노드의 오른쪽 서브트리에 있는 모든 노드의 값은 현재 노드의 값보다 크거나 같습니다.

이 변환 과정은 다음 세 가지 주요 단계로 구성됩니다:
1.  **중위 순회(Inorder Traversal)**: 트리를 중위 순회하여 모든 노드의 값을 추출하여 리스트에 저장합니다.
2.  **리스트 정렬(Sort List)**: 추출된 값들이 담긴 리스트를 오름차순으로 정렬합니다.
3.  **정렬된 값 복사(Copy Sorted Values)**: 다시 트리를 중위 순회하면서, 정렬된 리스트의 값을 순서대로 각 노드에 할당합니다.

### 코드/수식 해설

트리의 노드 개수를 $N$이라고 가정합니다.

1.  **Inorder Traversal (값 추출)**
    *   트리를 재귀적으로 중위 순회하며 각 노드의 값을 `std::vector`와 같은 리스트에 저장합니다.
    *   **시간 복잡도**: 모든 노드를 한 번씩 방문하므로 $O(N)$입니다.

    ```cpp
    void inorder_collect_values(TreeNode* node, std::vector<int>& values) {
        if (node == nullptr) {
            return;
        }
        inorder_collect_values(node->left, values);
        values.push_back(node->val); // 현재 노드의 값 저장
        inorder_collect_values(node->right, values);
    }
    ```

2.  **Sort the traversal list (리스트 정렬)**
    *   단계 1에서 수집된 값을 담은 리스트를 정렬합니다. `std::sort`와 같은 효율적인 정렬 알고리즘을 사용합니다.
    *   **시간 복잡도**: $N$개의 요소를 정렬하는 데는 일반적으로 $O(N \log N)$이 소요됩니다 (예: Merge Sort, Heap Sort, Quick Sort의 평균 경우).

    ```cpp
    // values 벡터는 위 함수를 통해 모든 노드 값으로 채워짐
    std::sort(values.begin(), values.end());
    ```

3.  **Copy the sorted list during another traversal (정렬된 값 재할당)**
    *   다시 트리를 중위 순회하면서, 정렬된 리스트의 값을 순서대로 노드에 할당합니다. 중위 순회의 특성상, 이렇게 값을 할당하면 이진 탐색 트리의 속성이 자동으로 만족됩니다.
    *   **시간 복잡도**: 모든 노드를 다시 한 번씩 방문하므로 $O(N)$입니다.

    ```cpp
    void inorder_assign_values(TreeNode* node, const std::vector<int>& sorted_values, int& index) {
        if (node == nullptr) {
            return;
        }
        inorder_assign_values(node->left, sorted_values, index);
        node->val = sorted_values[index++]; // 정렬된 값을 순서대로 할당
        inorder_assign_values(node->right, sorted_values, index);
    }
    ```
    *   최종 변환 함수:
    ```cpp
    void convert_to_bst(TreeNode* root) {
        std::vector<int> values;
        inorder_collect_values(root, values); // 1. 값 수집
        std::sort(values.begin(), values.end()); // 2. 값 정렬

        int index = 0;
        inorder_assign_values(root, values, index); // 3. 정렬된 값 할당
    }
    ```

*   **전체 시간 복잡도**: 각 단계의 시간 복잡도를 합하면 $O(N) + O(N \log N) + O(N) = O(N \log N)$ 입니다. 정렬 단계가 전체 복잡도를 지배합니다.
*   **공간 복잡도**: 모든 노드의 값을 저장하기 위한 `values` 벡터에 $O(N)$ 공간이 필요하고, 재귀 호출 스택의 깊이에 따라 최대 $O(N)$ (사향 트리) 또는 $O(\log N)$ (균형 트리)의 공간이 추가로 필요합니다. 따라서 총 공간 복잡도는 $O(N)$입니다.

### 구체적 예시

**초기 이진 트리**:
```

     /  \
    5    7
   / \
  8   6
```

1.  **Inorder Traversal (값 추출)**:
    *   왼쪽 서브트리 (8-5-6), 루트 (15), 오른쪽 서브트리 (7) 순으로 방문합니다.
    *   추출된 값 리스트 `values = [8, 5, 6, 15, 7]`

2.  **Sort the traversal list (리스트 정렬)**:
    *   `values` 리스트를 오름차순으로 정렬합니다.
    *   `sorted_values = [5, 6, 7, 8, 15]`

3.  **Copy the sorted list during another traversal (정렬된 값 재할당)**:
    *   다시 트리를 중위 순회하면서 `sorted_values`의 값을 순서대로 노드에 할당합니다.
        *   원래 8이 있던 노드에 5 할당.
        *   원래 5가 있던 노드에 6 할당.
        *   원래 6이 있던 노드에 7 할당.
        *   원래 15가 있던 노드에 8 할당.
        *   원래 7이 있던 노드에 15 할당.

**변환된 이진 탐색 트리**:
```

     /  \
    6    15
   / \

```
*   결과 트리는 초기 트리와 구조는 같지만, 모든 노드의 값이 BST 속성을 만족합니다.
    *   루트 `8`의 왼쪽 자식 `6` (8보다 작음), 오른쪽 자식 `15` (8보다 큼).
    *   노드 `6`의 왼쪽 자식 `5` (6보다 작음), 오른쪽 자식 `7` (6보다 큼).

### 시험 포인트

*   ⭐ **이진 트리를 이진 탐색 트리로 변환하는 세 가지 핵심 단계(Inorder Traversal -> Sort -> Inorder Traversal with sorted values)를 순서대로 나열하고 각각의 역할을 설명할 수 있어야 합니다.**
*   ⭐ **각 단계의 시간 복잡도를 분석하고, 전체 변환 과정의 최종 시간 복잡도가 $O(N \log N)$ (N은 노드의 개수)임을 도출할 수 있어야 합니다.** 특히, 정렬 단계가 지배적인 요인임을 강조하세요.
*   ⭐ **이 변환 방법의 공간 복잡도가 $O(N)$ (정렬된 값을 저장할 리스트와 재귀 스택)임을 이해하고 설명할 수 있어야 합니다.**
*   이 변환은 트리의 **구조는 유지**하고 **값만 변경**한다는 점을 명확히 이해하고 있어야 합니다. (노드의 추가/삭제, 부모-자식 관계 변경은 없음)
*   BST의 중위 순회 결과가 항상 정렬된 순서라는 특성을 역이용하여 문제를 해결하는 방식임을 이해하는 것이 중요합니다.

이 슬라이드는 이진 탐색 트리(BST)에서 최소값을 찾는 `SearchMinimum` 연산을 다룹니다.

---

### **핵심 개념**

*   **이진 탐색 트리의 최소값 특성**: 이진 탐색 트리는 각 노드의 왼쪽 서브트리에 있는 모든 노드의 값이 현재 노드의 값보다 작고, 오른쪽 서브트리에 있는 모든 노드의 값이 현재 노드의 값보다 크다는 핵심 속성을 가집니다. 이 속성 덕분에 트리의 최소값은 항상 **가장 왼쪽 아래에 있는 노드**에 위치하게 됩니다.
*   **SearchMinimum 연산**: 주어진 BST에서 가장 작은 값을 가진 노드를 찾기 위한 연산입니다. 루트 노드부터 시작하여 왼쪽 자식 노드가 존재하는 한 계속해서 왼쪽으로 이동하는 과정을 반복합니다. 더 이상 왼쪽 자식이 없는 노드가 발견되면, 그 노드가 트리의 최소값을 가지는 노드입니다.

### **코드/수식 해설**

BST에서 최소값을 찾는 `SearchMinimum` 함수의 의사 코드입니다.

```cpp
Node* SearchMinimum(Node* root) {
    // 트리가 비어있는 경우
    if (root == NULL) {
        return NULL;
    }

    Node* current = root;
    // 왼쪽 자식이 없을 때까지 계속 왼쪽으로 이동
    while (current->left != NULL) {
        current = current->left;
    }
    return current; // 가장 왼쪽 노드가 최소값을 가진 노드
}
```

*   **시간 복잡도**:
    최소값 탐색은 트리의 루트에서 시작하여 왼쪽으로만 이동하는 경로를 따라갑니다. 따라서 탐색에 걸리는 시간은 트리의 높이($h$)에 비례합니다.
    *   최악의 경우 (Skewed Tree, 편향 트리): 트리가 한쪽으로 치우쳐 링크드 리스트와 유사한 형태일 때, 높이 $h = n$이 되어 $O(n)$의 시간이 소요됩니다.
    *   최적/평균의 경우 (Balanced Tree, 균형 트리): 트리가 균형을 이룰 때, 높이 $h = \log n$이 되어 $O(\log n)$의 시간이 소요됩니다.
    즉, 시간 복잡도는 $O(h)$입니다.

### **구체적 예시**

첨부된 슬라이드의 BST를 사용하여 `SearchMinimum` 연산 과정을 살펴보겠습니다.

1.  **시작**: 루트 노드 `12`에서 탐색을 시작합니다. (`current = 12`)
2.  **왼쪽 자식 확인**: `12`의 왼쪽 자식은 `5`입니다. 왼쪽 자식이 존재하므로 `current`를 `5`로 이동합니다. (`current = 5`)
3.  **왼쪽 자식 확인**: `5`의 왼쪽 자식은 `3`입니다. 왼쪽 자식이 존재하므로 `current`를 `3`으로 이동합니다. (`current = 3`)
4.  **왼쪽 자식 확인**: `3`의 왼쪽 자식은 `NULL`입니다 (존재하지 않습니다).
5.  **종료**: 더 이상 왼쪽 자식이 없으므로 현재 노드인 `3`이 트리의 최소값을 가지는 노드입니다.

따라서 이 BST의 최소값은 **3**입니다.

### **시험 포인트**

*   ⭐ **BST 최소값의 위치**: BST에서 최소값을 갖는 노드는 항상 **가장 왼쪽(leaf) 노드**에 위치합니다. 반대로 최대값은 항상 가장 오른쪽(leaf) 노드에 위치합니다.
*   ⭐ **SearchMinimum 알고리즘**: 루트에서 시작하여 왼쪽 자식이 없을 때까지 현재 노드를 왼쪽 자식으로 계속 업데이트하는 방식으로 구현됩니다.
*   ⭐ **시간 복잡도**: `SearchMinimum` 연산의 시간 복잡도는 트리의 높이($h$)에 비례하며, $O(h)$입니다. 이는 최악의 경우 $O(n)$, 균형 트리의 경우 $O(\log n)$이 됩니다. 이 시간 복잡도 분석은 다른 BST 연산(삽입, 삭제, 탐색)의 복잡도와 연관되어 중요합니다.

이 슬라이드는 이진 탐색 트리(BST)에서 최소값을 찾는 `SearchMinimum(D)` 연산에 대해 설명합니다.

**핵심 개념**
이진 탐색 트리(Binary Search Tree, BST)에서 가장 작은 값을 가진 노드를 찾는 연산입니다. BST의 핵심 속성 덕분에 최소값은 항상 트리의 가장 왼쪽 아래에 위치한 노드에 저장됩니다. 이는 모든 노드에서 왼쪽 자식 노드의 값이 현재 노드의 값보다 작다는 BST의 정의를 따르기 때문입니다. 따라서, 루트 노드부터 시작하여 왼쪽 자식 노드가 존재하는 한 계속해서 왼쪽으로 이동하면 최종적으로 최소값을 찾을 수 있습니다.

**코드/수식 해설**

다음은 C++로 구현된 `SearchMinimum` 함수의 예시입니다.

```cpp
struct Node {
    int data;
    Node* left;
    Node* right;
};

Node* SearchMinimum(Node* root) {
    if (root == nullptr) {
        return nullptr; // 트리가 비어있는 경우
    }
    
    Node* current = root;
    while (current->left != nullptr) {
        current = current->left; // 왼쪽 자식이 있는 한 계속 왼쪽으로 이동
    }
    return current; // 가장 왼쪽 노드가 최소값을 가진 노드
}

// 재귀적 구현 (Recursive implementation)
Node* SearchMinimumRecursive(Node* node) {
    if (node == nullptr || node->left == nullptr) {
        return node;
    }
    return SearchMinimumRecursive(node->left);
}
```

*   **시간 복잡도**: 최소값을 찾기 위해 트리의 루트부터 가장 깊은 왼쪽 노드까지 탐색합니다. 이 경로는 트리의 높이(`h`)에 비례합니다. 따라서 시간 복잡도는 $O(h)$ 입니다.
    *   최악의 경우(skewed tree): 트리가 한쪽으로만 기울어져 연결 리스트 형태일 때 $O(N)$ (N은 노드의 개수)
    *   최상의 경우(balanced tree): 트리가 균형 잡혀 있을 때 $O(\log N)$

**구체적 예시**
슬라이드의 트리를 예시로 들어 `SearchMinimum` 과정을 설명합니다.

1.  **시작**: 루트 노드인 **12**에서 시작합니다.
2.  **왼쪽 이동**: 12의 왼쪽 자식 노드 **5**가 존재하므로 5로 이동합니다.
3.  **왼쪽 이동**: 5의 왼쪽 자식 노드 **3**이 존재하므로 3으로 이동합니다.
4.  **탐색 종료**: 3의 왼쪽 자식 노드는 `NULL`입니다. 따라서 **3**이 이진 탐색 트리에서 가장 작은 값을 가진 노드입니다.

**시험 포인트**
*   ⭐ **BST에서 최소값을 찾는 방법**: 항상 루트에서 시작하여 왼쪽 자식 노드가 `NULL`이 될 때까지 계속해서 왼쪽으로만 탐색합니다.
*   ⭐ **시간 복잡도**: `SearchMinimum` 연산의 시간 복잡도는 트리의 높이 $h$에 비례하는 $O(h)$입니다. 균형 트리의 경우 $O(\log N)$, 최악의 경우(선형 트리) $O(N)$임을 이해해야 합니다.
*   ⭐ **BST의 핵심 속성 이해**: 왜 왼쪽으로만 가면 최소값을 찾을 수 있는지, BST의 기본 정의(왼쪽 < 부모 < 오른쪽)를 바탕으로 설명할 수 있어야 합니다.

## Slide 36

**핵심 개념**:
*   **이진 탐색 트리 (BST, Binary Search Tree)**: 모든 노드에 대해 왼쪽 서브트리의 모든 값은 현재 노드의 값보다 작고, 오른쪽 서브트리의 모든 값은 현재 노드의 값보다 크다는 속성을 만족하는 트리입니다. 이 속성 때문에 BST는 데이터를 효율적으로 탐색, 삽입, 삭제할 수 있습니다.
*   **`SearchMinimum(D)` 연산**: 자료구조 `D`에서 가장 작은 값을 찾는 연산입니다. BST에서는 이 연산이 트리의 가장 왼쪽(leftmost) 노드를 찾는 것과 동일합니다.
*   **O(1) 시간 복잡도**: 연산에 걸리는 시간이 입력 데이터의 크기(`N`, 트리의 노드 수)에 관계없이 항상 일정하다는 것을 의미합니다. 즉, 상수 시간입니다.
*   **트리 구조 변경 없이**: 이 조건은 기존 BST 노드에 추가적인 필드를 도입하거나, 트리의 연결 구조를 바꾸는 등의 행위를 금지합니다.

**코드/수식 해설**:
BST에서 `SearchMinimum` 연산을 수행하는 표준 방법은 트리의 루트 노드부터 시작하여 왼쪽 자식 노드가 없을 때까지 계속해서 왼쪽 자식 노드를 따라 내려가는 것입니다.

```cpp
struct Node {
    int key;
    Node* left;
    Node* right;
    // ... 기타 필드
};

Node* SearchMinimum(Node* root) {
    if (root == nullptr) {
        return nullptr; // 트리가 비어있는 경우
    }
    Node* current = root;
    while (current->left != nullptr) {
        current = current->left;
    }
    return current; // 가장 왼쪽 노드 (최소값을 포함) 반환
}
```
이 알고리즘의 시간 복잡도는 트리의 높이($h$)에 비례합니다. 즉, $O(h)$입니다.
*   최악의 경우 (트리가 한쪽으로 치우친 경우, 예: 모든 노드가 오른쪽 자식만 가지는 경우), 트리의 높이는 노드 수 $N$과 같아져 $h = O(N)$이 됩니다.
*   최적/평균의 경우 (트리가 균형 잡힌 경우), 트리의 높이는 $\log_2 N$에 비례하여 $h = O(\log N)$이 됩니다.

따라서 일반적인 `SearchMinimum` 연산은 $N$에 의존적인 시간 복잡도를 가지므로, `O(1)`이 아닙니다.

**구체적 예시**:
다음과 같은 이진 탐색 트리를 예로 들어 보겠습니다.
```
      10
     /  \
    5    15
   / \   / \
  2   7 12  18
```
`SearchMinimum` 함수를 호출하여 최소값을 찾는 과정은 다음과 같습니다:
1.  `root` 노드 `10`에서 시작합니다. `current`는 `10`입니다.
2.  `current->left` (`5`)가 `nullptr`이 아니므로, `current`를 `5`로 업데이트합니다.
3.  `current->left` (`2`)가 `nullptr`이 아니므로, `current`를 `2`로 업데이트합니다.
4.  `current->left` (`nullptr`)가 `nullptr`이므로, `while` 루프가 종료됩니다.
5.  함수는 `current` 노드인 `2`를 반환합니다.

이 과정에서 3번의 노드 방문 (10, 5, 2)이 이루어졌습니다. 이는 트리의 높이에 비례하는 동작입니다.

**시험 포인트**:
*   ⭐ **결론**: 슬라이드의 질문에 대한 답은 **아니오 (No)** 입니다. 기존 BST 구조를 변경하지 않고는 `SearchMinimum(D)` 연산을 `O(1)` 시간에 지원할 수 없습니다.
*   ⭐ **이유**: `SearchMinimum` 연산은 BST의 정의상 항상 트리의 가장 왼쪽 경로를 따라 내려가야 합니다. 이 경로는 트리의 높이에 따라 길이가 달라지며, 이는 트리의 노드 수 $N$에 따라 달라질 수 있습니다 (최소 $O(\log N)$, 최악 $O(N)$). `O(1)` 시간 복잡도는 입력 크기에 독립적인 상수 시간이므로, $N$에 의존하는 탐색 과정은 `O(1)`이 될 수 없습니다.
*   ⭐ **`O(1)`을 만들기 위한 방법 (단, 문제의 조건 위배)**: 만약 트리의 구조를 변경하는 것이 허용된다면, 트리를 생성하거나 삽입/삭제 연산 시 루트 노드(또는 별도의 트리를 관리하는 객체)에 항상 최소값을 가리키는 포인터(`min_node_ptr`)를 유지하는 방법을 사용할 수 있습니다. 이 경우 `min_node_ptr`을 바로 반환하면 `O(1)`이 가능하지만, 이는 "without modifying the tree structure"라는 조건에 위배됩니다.

## 데이터구조 (CSED233) 튜터 노트: 자료구조 선택 퀴즈

### 핵심 개념
이 슬라이드는 특정 시스템의 요구사항을 만족하는 자료구조를 선택하는 퀴즈를 제시합니다. 시스템은 정수 키(integer keys)의 동적 집합(dynamic set)을 관리하며, 다음 네 가지 연산을 지원해야 합니다.
1.  **키 삽입 (Insert keys)**: 임의의 시간에 새로운 키를 추가.
2.  **키 탐색 (Search for a given key)**: 특정 키의 존재 여부를 확인하거나 위치를 찾음.
3.  **키 삭제 (Remove an arbitrary key)**: 임의의 키를 집합에서 제거.
4.  **정렬된 키 출력 (Output all keys in ascending order)**: 현재 집합에 있는 모든 키를 오름차순으로 출력.

주요 비교 대상은 **이진 탐색 트리 (Binary Search Tree, BST)**와 **우선순위 큐 (Priority Queue), 특히 힙 (heap)**입니다.

*   **이진 탐색 트리 (BST)**: 모든 노드에 대해, 왼쪽 서브트리의 키는 노드 키보다 작고, 오른쪽 서브트리의 키는 노드 키보다 큰 속성을 유지하는 트리 구조. 삽입, 탐색, 삭제 연산의 평균 시간 복잡도는 $O(\log n)$입니다 (트리가 균형 잡혀 있을 때). 모든 키를 정렬된 순서로 출력하는 것은 $O(n)$ 시간에 중위 순회(in-order traversal)를 통해 가능합니다.
*   **우선순위 큐 (Heap)**: 항상 최소값(min-heap) 또는 최대값(max-heap)을 빠르게 찾을 수 있도록 설계된 트리 기반의 자료구조 (주로 배열로 구현). 삽입 및 최소/최대값 삭제 연산은 $O(\log n)$ 시간이 걸립니다. 그러나 임의의 키를 탐색하거나 삭제하는 데는 $O(n)$ 시간이 소요될 수 있습니다.

### 코드/수식 해설
각 연산에 대한 BST와 Heap의 시간 복잡도는 다음과 같습니다 (여기서 $n$은 자료구조에 저장된 키의 개수입니다):

*   **키 삽입 (Insert keys)**
    *   BST (평균/균형 잡힌): $O(\log n)$
    *   Heap: $O(\log n)$
*   **키 탐색 (Search for a given key)**
    *   BST (평균/균형 잡힌): $O(\log n)$
    *   Heap: $O(n)$ (힙은 임의의 키 탐색을 위한 구조가 아님)
*   **키 삭제 (Remove an arbitrary key)**
    *   BST (평균/균형 잡힌): $O(\log n)$
    *   Heap: $O(n)$ (임의의 키를 찾는데 $O(n)$, 삭제 후 재구성하는데 $O(\log n)$)
    (참고: `deleteMin` 또는 `deleteMax`는 $O(\log n)$)
*   **정렬된 키 출력 (Output all keys in ascending order)**
    *   BST (중위 순회): $O(n)$
    *   Heap (반복적인 `deleteMin`): $O(n \log n)$ (각 `deleteMin`이 $O(\log n)$)

### 구체적 예시
학생들의 성적 관리 시스템을 예로 들어 봅시다.
1.  **새로운 학생의 성적 추가**: `Insert` 연산
2.  **특정 학생의 성적 조회**: `Search` 연산
3.  **전학 간 학생의 성적 삭제**: `Remove` 연산
4.  **전체 학생 성적을 석차 순으로 (오름차순) 출력**: `Output all keys in ascending order` 연산

*   **BST를 사용하는 경우**: 모든 연산이 효율적으로 처리됩니다. 새로운 학생이 들어오거나(삽입), 특정 학생 성적을 찾거나(탐색), 전학 간 학생을 지우거나(삭제) 모두 평균적으로 $O(\log n)$ 시간에 가능합니다. 전체 학생의 성적을 오름차순으로 출력할 때도 $O(n)$ 시간에 한 번의 순회로 완료할 수 있습니다. 예를 들어, 성적표를 출력할 때 BST의 중위 순회를 사용하면 학생 이름을 가나다순으로 출력하듯, 키를 정렬된 순서로 출력할 수 있습니다.

*   **Priority Queue (heap)을 사용하는 경우**:
    *   새로운 학생의 성적 추가(`Insert`)는 효율적입니다 ($O(\log n)$).
    *   그러나 특정 학생의 성적을 조회(`Search`)하거나 전학 간 학생의 성적을 삭제(`Remove`)하는 것은 매우 비효율적입니다. 힙은 "최고/최저 성적 학생"을 찾는 데는 $O(1)$ 또는 $O(\log n)$ (삭제 후 재구성) 빠르지만, "철수"의 성적을 찾는 데는 모든 학생을 뒤져봐야 할 수 있습니다.
    *   전체 학생 성적을 석차 순으로 출력(`Output all keys in ascending order`)하려면, 힙에서 최소값을 $n$번 추출해야 하므로 $O(n \log n)$ 시간이 소요되어 BST보다 느립니다.

이러한 비교를 통해, 주어진 시스템의 요구사항을 모두 만족시키기 위해서는 BST가 Heap보다 훨씬 적합함을 알 수 있습니다. 특히, 임의의 키에 대한 탐색 및 삭제, 그리고 효율적인 전체 정렬 출력 요구사항이 중요합니다.

### 시험 포인트
*   ⭐ **각 자료구조의 핵심 연산별 시간 복잡도를 정확히 이해하고 비교할 수 있어야 합니다.** 특히 BST의 균형 상태에 따른 시간 복잡도 변화 (평균 vs. 최악)와 Priority Queue (Heap)의 `search` 및 `remove arbitrary key` 연산의 비효율성을 명확히 알아야 합니다.
*   ⭐ **주어진 요구사항에 가장 적합한 자료구조를 선택하고 그 이유를 설명할 수 있어야 합니다.** 어떤 연산이 주요 요구사항인지 파악하는 것이 중요합니다.
*   ⭐ **BST가 해당 시스템에 적합한 이유:** 모든 연산 (삽입, 탐색, 삭제, 정렬 출력)을 평균적으로 효율적으로 지원하기 때문입니다. 특히, 임의 키에 대한 탐색 및 삭제와 $O(n)$의 정렬된 출력 기능은 BST의 강점입니다.
*   ⭐ **Priority Queue (heap)가 해당 시스템에 부적합한 이유:** 임의의 키 탐색 및 삭제가 $O(n)$으로 비효율적이며, 모든 키를 정렬된 순서로 출력하는 데 $O(n \log n)$이 소요되어 BST의 $O(n)$보다 비효율적이기 때문입니다. Heap은 주로 `min/max` 요소의 빠른 접근 및 삭제에 특화되어 있습니다.

## 데이터구조 (CSED233) 슬라이드 노트: BST와 중복 키

### 핵심 개념

이 슬라이드는 표준 이진 탐색 트리(BST)가 모든 키를 고유(distinct)하다고 가정하는 기본적인 특성을 전제로, 만약 BST에서 중복된 키(duplicate keys)를 허용할 경우 발생할 수 있는 설계 및 구현상의 문제점과 모호성(ambiguities)에 대해 다루고 있습니다. 중복 키를 처리하는 방법에 따라 BST의 탐색, 삽입, 삭제 동작의 정의와 효율성이 크게 달라질 수 있습니다.

### 코드/수식 해설

표준 BST의 속성은 다음과 같습니다:
임의의 노드 $N$에 대해,
*   $N$의 왼쪽 서브트리에 있는 모든 키 $K_L$은 $N$의 키 $K_N$보다 작다 ($K_L < K_N$).
*   $N$의 오른쪽 서브트리에 있는 모든 키 $K_R$은 $N$의 키 $K_N$보다 크다 ($K_R > K_N$).

중복 키를 허용할 경우, 이 속성에 대한 해석이 모호해집니다.
예를 들어, 키 `X`와 동일한 값을 가진 새 키 `X'`를 삽입할 때:

**1. 삽입(Insertion) 시의 모호성**

*   **선택 1: 항상 왼쪽 서브트리에 삽입**
    ```
    insert(node, key):
        if key < node.key:
            node.left = insert(node.left, key)
        else if key >= node.key: // key가 같거나 크면 오른쪽으로
            node.right = insert(node.right, key)
        return node
    ```
    이 경우, 중복 키는 기존 키보다 크거나 같다고 간주하여 오른쪽으로 내려갑니다.
    BST 속성은 $K_L < K_N \le K_R$ 형태로 변경됩니다.

*   **선택 2: 항상 오른쪽 서브트리에 삽입**
    ```
    insert(node, key):
        if key <= node.key: // key가 같거나 작으면 왼쪽으로
            node.left = insert(node.left, key)
        else if key > node.key:
            node.right = insert(node.right, key)
        return node
    ```
    이 경우, 중복 키는 기존 키보다 작거나 같다고 간주하여 왼쪽으로 내려갑니다.
    BST 속성은 $K_L \le K_N < K_R$ 형태로 변경됩니다.

*   **선택 3: 노드 내에 개수(count) 저장**
    ```
    struct Node {
        KeyType key;
        ValueType value;
        int count; // 이 키가 몇 번 등장하는지
        Node *left, *right;
    };

    insert(node, key):
        if key == node.key:
            node.count++
            return node
        else if key < node.key:
            node.left = insert(node.left, key)
        else: // key > node.key
            node.right = insert(node.right, key)
        return node
    ```

*   **선택 4: 노드 내에 리스트(list) 저장**
    각 노드는 해당 키와 함께, 해당 키에 연결된 모든 값(value)들의 리스트를 가집니다.

**2. 탐색(Search) 시의 모호성**

키 $X$를 탐색할 때:
*   하나의 $X$만 찾아서 반환할 것인가?
*   모든 $X$를 찾아서 반환할 것인가?

어떤 $X$를 반환할지는 삽입 시 중복 키를 어느 서브트리에 넣기로 결정했는지에 따라 달라질 수 있습니다. 예를 들어, 항상 오른쪽 서브트리에 중복을 넣었다면, 가장 왼쪽의 중복을 찾기 위해서는 추가적인 탐색 로직이 필요합니다.

**3. 삭제(Deletion) 시의 모호성**

키 $X$를 삭제할 때:
*   중복된 $X$들 중 어떤 것을 삭제할 것인가 (예: 처음 찾은 것)?
*   모든 $X$를 삭제할 것인가?
*   노드 내에 개수를 저장하는 방식이라면, `count`를 1 감소시킬 것인가, 아니면 `count`가 1일 때만 실제 노드를 삭제할 것인가?

### 구체적 예시

**시나리오**: `5, 3, 7, 5, 2, 8, 5` 순서로 키를 삽입하는 경우

**1. 중복 키를 항상 오른쪽 서브트리에 삽입하는 정책 ($K_L < K_N \le K_R$)**

*   `5` 삽입: 루트 `5`
*   `3` 삽입: `5`의 왼쪽 `3`
*   `7` 삽입: `5`의 오른쪽 `7`
*   `5` 삽입: `5`와 같으므로 오른쪽으로 가서 `7`의 왼쪽 `5`
*   `2` 삽입: `3`의 왼쪽 `2`
*   `8` 삽입: `7`의 오른쪽 `8`
*   `5` 삽입: 루트 `5`와 같으므로 오른쪽으로 가서 `7`과 비교, `7`보다 작으므로 왼쪽으로 가서 기존 `5`와 비교, 같으므로 기존 `5`의 오른쪽 `5`

결과 트리 구조 (개략적):
```
      5 (root)
     / \
    3   7
   /   / \
  2   5   8
       \
        5
```
**문제점**: `search(5)`를 하면 루트 `5`를 찾게 되며, 나머지 `5`들을 찾기 위해서는 추가 로직이 필요합니다. 또한, 오른쪽으로만 중복이 치우치면 트리가 한쪽으로 기울어질 위험이 있습니다.

**2. 노드 내에 개수(count)를 저장하는 정책**

*   `5` 삽입: 루트 `(5, count=1)`
*   `3` 삽입: `(5, count=1)`의 왼쪽 `(3, count=1)`
*   `7` 삽입: `(5, count=1)`의 오른쪽 `(7, count=1)`
*   `5` 삽입: 루트 `(5, count=1)`와 같으므로 `(5, count=2)`로 업데이트
*   `2` 삽입: `(3, count=1)`의 왼쪽 `(2, count=1)`
*   `8` 삽입: `(7, count=1)`의 오른쪽 `(8, count=1)`
*   `5` 삽입: 루트 `(5, count=2)`와 같으므로 `(5, count=3)`으로 업데이트

결과 트리 구조 (개략적):
```
      (5, count=3)
     /         \
    (3, count=1)  (7, count=1)
   /               \
  (2, count=1)      (8, count=1)
```
**장점**: 트리의 높이가 낮게 유지되고, `search(5)`는 루트 노드만 반환해도 모든 `5`의 개수를 알 수 있습니다.
**삭제 시 문제점**: `delete(5)`를 호출하면 `count`를 2로 줄일 것인가, 아니면 `count`가 1이 되면 노드를 완전히 삭제할 것인가?

### 시험 포인트

*   ⭐ **표준 BST의 정의 (모든 키는 고유)**를 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **중복 키 허용 시 발생할 수 있는 주요 문제점 (삽입, 탐색, 삭제의 모호성)**을 구체적으로 제시할 수 있어야 합니다.
*   ⭐ **중복 키를 처리하는 세 가지 주요 전략** (특정 서브트리 규칙, 노드 내 카운트, 노드 내 리스트) 각각의 장단점과 구현 시 고려사항을 설명할 수 있어야 합니다.
    *   특히, 특정 서브트리 규칙(예: `key <= node.key`는 왼쪽, `key > node.key`는 오른쪽)을 사용할 경우 **BST 속성이 어떻게 변형되는지 ($K_L \le K_N < K_R$ 또는 $K_L < K_N \le K_R$)**를 LaTeX 수식으로 표현하거나 설명할 수 있어야 합니다.
*   ⭐ 중복 키 처리 방식이 BST의 **균형(balance)** 및 **시간 복잡도(time complexity)**에 미치는 영향을 분석할 수 있어야 합니다. (예: 한쪽으로 치우쳐진 중복 키 삽입은 트리를 불균형하게 만들 수 있음)

## 데이터구조 (CSED233) 슬라이드 요약 노트: Binary Search Tree (BST)

---

### **핵심 개념**

*   **Binary Search Tree (BST)**: 이진 탐색 트리는 이진 트리의 일종으로, 각 노드가 키(key) 값을 가지고 있으며 다음과 같은 핵심 속성을 만족합니다.
*   **BST Property**:
    *   어떤 노드의 왼쪽 서브트리에 있는 모든 노드의 키는 해당 노드의 키보다 작습니다.
    *   어떤 노드의 오른쪽 서브트리에 있는 모든 노드의 키는 해당 노드의 키보다 큽니다.
    *   (중복 키의 처리는 구현에 따라 달라질 수 있으나, 일반적으로 허용하지 않거나 특정 서브트리에만 삽입하는 규칙을 따릅니다.)
*   **Inorder Traversal**: 이진 트리를 탐색하는 방법 중 하나로, `왼쪽 서브트리 -> 현재 노드 -> 오른쪽 서브트리` 순서로 방문합니다. BST에서는 Inorder Traversal을 수행하면 트리에 저장된 모든 요소들이 **오름차순으로 정렬된 순서**로 나옵니다.
*   **Search / Insert / Delete**: BST는 이 속성 덕분에 효율적인 탐색, 삽입, 삭제 연산을 제공합니다. 모든 연산은 트리의 루트에서 시작하여 BST Property에 따라 왼쪽 또는 오른쪽 자식으로 이동하며 특정 키를 찾아갑니다.
*   **Tree Height**: 트리의 높이는 루트 노드부터 가장 깊은 리프 노드까지의 가장 긴 경로에 포함된 간선(edge)의 개수를 의미합니다. (루트만 있는 트리의 높이는 0).
*   **Balanced vs Skewed**:
    *   **Balanced Tree**: 트리의 높이가 최소화된 형태로, 모든 리프 노드들이 루트로부터 거의 비슷한 거리에 있습니다. 균형 잡힌 트리는 모든 연산에서 효율적인 성능을 보장합니다.
    *   **Skewed Tree (Degenerate Tree)**: 트리가 한쪽 방향으로만 노드들이 연결되어 마치 연결 리스트와 같은 형태를 띱니다. 이러한 트리는 BST의 장점을 상실하고 비효율적인 성능을 보입니다.

---

### **코드/수식 해설**

**1. Search (탐색) 연산**
현재 노드의 키와 찾으려는 키를 비교하여, 작으면 왼쪽으로, 크면 오른쪽으로 재귀적으로 또는 반복적으로 이동합니다.

```cpp
// C++ Pseudocode for Search
Node* search(Node* root, int key) {
    // 트리가 비었거나, 키를 찾았을 경우
    if (root == nullptr || root->data == key) {
        return root;
    }
    // 찾으려는 키가 현재 노드보다 작으면 왼쪽 서브트리 탐색
    if (key < root->data) {
        return search(root->left, key);
    }
    // 찾으려는 키가 현재 노드보다 크면 오른쪽 서브트리 탐색
    else {
        return search(root->right, key);
    }
}
```

**2. Insert (삽입) 연산**
탐색 연산과 유사하게 적절한 위치(리프 노드의 아래)를 찾아서 새 노드를 삽입합니다.

```cpp
// C++ Pseudocode for Insert
Node* insert(Node* root, int key) {
    // 트리가 비었을 경우 (삽입할 위치를 찾았을 경우)
    if (root == nullptr) {
        return new Node(key); // 새 노드 생성 및 반환
    }
    // 삽입하려는 키가 현재 노드보다 작으면 왼쪽으로 이동
    if (key < root->data) {
        root->left = insert(root->left, key);
    }
    // 삽입하려는 키가 현재 노드보다 크면 오른쪽으로 이동
    else if (key > root->data) { // 중복 키는 일반적으로 처리하지 않거나 특정 규칙을 따름
        root->right = insert(root->right, key);
    }
    return root; // 변경된 서브트리의 루트 반환
}
```

**3. Delete (삭제) 연산**
삭제 연산은 세 가지 주요 경우로 나뉘며, 가장 복잡한 연산입니다.

*   **Case 1: 삭제할 노드가 리프 노드인 경우 (자식이 없는 경우)**
    *   단순히 노드를 삭제하고, 부모 노드와의 연결을 끊습니다.
*   **Case 2: 삭제할 노드가 자식이 하나인 경우**
    *   삭제할 노드를 삭제하고, 그 자식 노드를 부모 노드에 직접 연결합니다.
*   **Case 3: 삭제할 노드가 자식이 둘인 경우**
    *   삭제할 노드의 **오른쪽 서브트리에서 가장 작은 값 (Inorder Successor)**을 찾거나, **왼쪽 서브트리에서 가장 큰 값 (Inorder Predecessor)**을 찾습니다.
    *   찾은 successor/predecessor의 값을 삭제할 노드에 복사합니다.
    *   그리고 successor/predecessor 노드를 원래 위치에서 삭제합니다. (이때 successor/predecessor는 항상 자식이 없거나 하나만 있는 노드이므로, Case 1 또는 Case 2로 처리됩니다.)

```cpp
// C++ Pseudocode for Delete
// 오른쪽 서브트리에서 가장 작은 노드(Inorder Successor)를 찾는 헬퍼 함수
Node* findMin(Node* node) {
    Node* current = node;
    while (current && current->left != nullptr) {
        current = current->left;
    }
    return current;
}

Node* deleteNode(Node* root, int key) {
    if (root == nullptr) return root;

    // 삭제할 키를 찾기 위해 트리 탐색
    if (key < root->data) {
        root->left = deleteNode(root->left, key);
    } else if (key > root->data) {
        root->right = deleteNode(root->right, key);
    } else { // 삭제할 노드를 찾음 (key == root->data)
        // Case 1 & 2: 자식이 없거나 하나만 있는 경우
        if (root->left == nullptr) {
            Node* temp = root->right; // 오른쪽 자식이 있으면 temp에 저장 (없으면 temp는 nullptr)
            delete root;
            return temp; // 자식(또는 nullptr)을 반환하여 부모와 연결
        } else if (root->right == nullptr) {
            Node* temp = root->left; // 왼쪽 자식만 있는 경우
            delete root;
            return temp; // 자식(또는 nullptr)을 반환하여 부모와 연결
        }

        // Case 3: 자식이 둘인 경우
        // 오른쪽 서브트리에서 가장 작은 노드(inorder successor)를 찾음
        Node* temp = findMin(root->right);
        // successor의 데이터를 현재 노드로 복사
        root->data = temp->data;
        // 이제 원래 successor가 있던 위치에서 해당 노드를 삭제
        root->right = deleteNode(root->right, temp->data);
    }
    return root; // 변경된 서브트리의 루트 반환
}
```

**4. Tree Height (트리 높이) 및 Time Complexity (시간 복잡도)**

*   $N$개의 노드를 가진 BST의 최소 높이 ($h_{min}$)는 트리가 완전 이진 트리에 가까울 때 발생하며, 다음과 같습니다.
    $$ h_{min} = \lfloor \log_2 N \rfloor $$
*   $N$개의 노드를 가진 BST의 최대 높이 ($h_{max}$)는 트리가 Skewed 되었을 때 발생하며, 다음과 같습니다.
    $$ h_{max} = N-1 $$
    (노드 1개일 때 높이 0, 노드 2개일 때 높이 1 등)

*   **Time Complexity**: 탐색, 삽입, 삭제 연산의 시간 복잡도는 트리의 높이에 비례합니다.
    *   **최선/평균 경우 (Balanced BST)**: 트리의 높이가 $O(\log N)$에 가까울 때
        *   $O(\log N)$
    *   **최악 경우 (Skewed BST)**: 트리의 높이가 $O(N)$일 때
        *   $O(N)$

### **구체적 예시**

**예시 1: BST 구축 및 Inorder Traversal**

다음 키들을 순서대로 BST에 삽입해 봅시다: `50, 30, 70, 20, 40, 60, 80`

1.  `50` 삽입: `50`이 루트가 됩니다.
2.  `30` 삽입: `30 < 50`이므로 `50`의 왼쪽 자식이 됩니다.
3.  `70` 삽입: `70 > 50`이므로 `50`의 오른쪽 자식이 됩니다.
4.  `20` 삽입: `20 < 50`, `20 < 30`이므로 `30`의 왼쪽 자식이 됩니다.
5.  `40` 삽입: `40 < 50`, `40 > 30`이므로 `30`의 오른쪽 자식이 됩니다.
6.  `60` 삽입: `60 > 50`, `60 < 70`이므로 `70`의 왼쪽 자식이 됩니다.
7.  `80` 삽입: `80 > 50`, `80 > 70`이므로 `70`의 오른쪽 자식이 됩니다.

완성된 트리의 Inorder Traversal 결과:
`20 -> 30 -> 40 -> 50 -> 60 -> 70 -> 80` (정렬된 순서)

**예시 2: Deletion Cases (삭제 예시)**

위 BST에서 다음 키들을 삭제해 봅시다.

1.  **`20` 삭제 (Case 1: 리프 노드)**:
    *   `20` 노드를 찾습니다. `20`은 자식이 없으므로 단순히 삭제하고 `30`의 왼쪽 자식 연결을 `nullptr`로 설정합니다.

2.  **`70` 삭제 (Case 3: 자식이 둘인 노드)**:
    *   `70` 노드를 찾습니다. `70`은 `60` (왼쪽 자식)과 `80` (오른쪽 자식) 두 개의 자식을 가집니다.
    *   `70`의 오른쪽 서브트리 (`80`만 있음)에서 가장 작은 값인 `80`을 Inorder Successor로 선택합니다.
    *   `70`의 값을 `80`으로 변경합니다.
    *   원래 `80`이 있던 위치(현재 `80`이 된 `70` 노드의 오른쪽 자식)에서 `80`을 삭제합니다. 이 `80`은 리프 노드이므로 Case 1로 처리됩니다. 최종적으로 `70` 노드는 `80`으로 바뀌고 오른쪽 자식이 `nullptr`이 됩니다.

**실생활 비유: 도서관의 책 정리 (Balanced vs Skewed)**

*   **Balanced Tree (균형 잡힌 트리)**: 도서관에서 책들이 주제별, 저자별, 제목별로 잘 분류되어 각 책을 찾기 위해 탐색해야 하는 '분류 단계'가 어떤 책이든 거의 비슷하게 적은 경우입니다. 예를 들어, 컴퓨터 과학 서적 섹션에서 "알고리즘" 책을 찾을 때, 여러 분류를 빠르게 훑어가며 목표에 도달합니다. (탐색 $O(\log N)$)
*   **Skewed Tree (한쪽으로 치우친 트리)**: 만약 도서관의 모든 책이 특정 규칙(예: 최근 입고된 순서)에 따라 단순히 한 줄로만 쭉 나열되어 있고, 특정 책을 찾으려면 처음부터 순서대로 모두 훑어봐야 하는 경우입니다. 예를 들어, "데이터 구조" 책이 '가장 오래된 책'에 해당하는 마지막에 있다면, 모든 책을 하나씩 검토해야 합니다. (탐색 $O(N)$)

### **시험 포인트** ⭐

*   ⭐ **BST Property**를 정확하게 이해하고 설명할 수 있어야 합니다. 이 속성이 BST의 모든 연산에 어떻게 적용되는지 알아야 합니다.
*   ⭐ **Inorder Traversal**이 BST에서 어떤 의미(정렬된 순서)를 가지는지 알고, 재귀적으로 Inorder Traversal을 구현할 수 있어야 합니다. Preorder, Postorder와의 차이점도 중요합니다.
*   ⭐ **탐색, 삽입, 삭제 연산의 상세 동작 방식**을 완벽하게 이해하고 설명할 수 있어야 합니다. 특히 **삭제 연산의 세 가지 경우(자식이 0개, 1개, 2개)**를 구분하고, 각 경우에 대한 처리 로직(특히 Inorder Successor/Predecessor 사용)을 명확히 설명할 수 있어야 합니다.
*   ⭐ **트리 높이**의 정의를 알고, **Balanced Tree와 Skewed Tree의 개념과 차이점**을 정확히 설명할 수 있어야 합니다. 이 두 유형의 트리가 BST 연산의 **시간 복잡도에 미치는 영향**을 비교 설명하는 것이 중요합니다.
*   ⭐ 모든 BST 연산(탐색, 삽입, 삭제)에 대한 **시간 복잡도 (Big-O 표기법)**를 **최선, 평균, 최악의 경우**로 나누어 정확히 제시하고, 그 이유를 트리의 높이와 연관 지어 설명할 수 있어야 합니다. (특히 최악의 경우가 언제 발생하는지 아는 것이 중요합니다.)

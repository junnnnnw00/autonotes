## Slide 1

**핵심 개념**:
본 슬라이드는 CSED233 데이터구조 강의의 첫 페이지로, 이번 섹션에서 다룰 주요 추상 자료형(ADT, Abstract Data Type)인 **List(리스트)**, **Stack(스택)**, **Queue(큐)**를 소개합니다. 이 세 가지 자료구조는 데이터를 저장하고 접근하는 방식을 정의하는 기본적인 논리적 구조입니다.

*   **List**: 순서를 가지는 데이터 항목들의 모음으로, 특정 위치에 항목을 삽입, 삭제하거나 접근할 수 있습니다.
*   **Stack**: LIFO(Last-In, First-Out) 원칙을 따르는 자료구조입니다. 가장 나중에 추가된 항목이 가장 먼저 제거됩니다. 주요 연산은 `push`(추가)와 `pop`(제거)입니다.
*   **Queue**: FIFO(First-In, First-Out) 원칙을 따르는 자료구조입니다. 가장 먼저 추가된 항목이 가장 먼저 제거됩니다. 주요 연산은 `enqueue`(추가)와 `dequeue`(제거)입니다.

**코드/수식 해설**:
본 슬라이드는 소개 페이지이므로, 특정 코드나 수식은 포함되어 있지 않습니다.

**구체적 예시**:
*   **Queue**: 슬라이드 하단의 이미지에서 볼 수 있듯이, 사람들이 줄을 서서 기다리는 모습은 큐의 가장 대표적인 실생활 예시입니다. 먼저 온 사람이 먼저 서비스를 받습니다(FIFO).
*   **Stack**: 식당에서 접시를 쌓아 놓은 모습이나 웹 브라우저의 '뒤로 가기' 기능이 스택의 좋은 예시입니다. 가장 마지막에 방문한 페이지가 '뒤로 가기' 버튼을 눌렀을 때 가장 먼저 나타납니다(LIFO).
*   **List**: 장바구니 목록이나 재생 목록처럼 순서대로 여러 항목을 관리하는 경우가 리스트의 예시입니다. 항목을 중간에 추가하거나 제거할 수 있습니다.

**시험 포인트**:
*   ⭐ List, Stack, Queue 각각의 **정의**, **주요 연산** (예: Stack의 push/pop, Queue의 enqueue/dequeue), 그리고 각 ADT가 따르는 **원칙** (LIFO, FIFO)을 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ 각 ADT를 **배열 기반** 또는 **연결 리스트 기반**으로 구현했을 때의 장단점 및 각 연산의 **시간 복잡도($O(1)$, $O(N)$ 등)**를 분석하는 것이 중요합니다.
*   ⭐ 실제 문제 해결 시 어떤 상황에서 어떤 자료구조를 사용하는 것이 가장 효율적인지 판단할 수 있어야 합니다.

## Slide 2

**핵심 개념**

*   **배열 (Array)**: 동일한 타입의 데이터를 메모리에 연속적으로 저장하는 자료구조입니다. C++에서 배열의 이름은 해당 배열의 첫 번째 요소의 주소를 가리키는 상수 포인터처럼 동작합니다.
*   **포인터 (Pointer)**: 메모리 주소를 저장하는 변수입니다. 특정 타입의 데이터를 가리키며, 포인터 연산을 통해 메모리 상의 다른 위치에 접근할 수 있습니다.
*   **배열-포인터 변환 (Array-to-Pointer Decay)**: 배열의 이름이 포인터가 필요한 컨텍스트에서 사용될 때, 자동으로 배열의 첫 번째 요소에 대한 포인터로 변환되는 C++의 특징입니다.
*   **포인터와 배열 인덱싱**: 포인터 변수도 배열처럼 `pointer_var[index]` 형태로 인덱싱하여 접근할 수 있습니다. 이는 `*(pointer_var + index)`와 동일하게 작동합니다.

**코드/수식 해설**

```cpp
char c[] = {'c', 'a', 't'};
char* p = c;
char* q = &c[0];
cout << c[2] << p[2] << q[2];
```

1.  `char c[] = {'c', 'a', 't'};`
    *   `char` 타입의 배열 `c`를 선언하고, `'c'`, `'a'`, `'t'`로 초기화합니다. 배열의 크기는 초기화 리스트에 따라 3으로 자동 결정됩니다.
    *   메모리에는 `c[0]`에 `'c'`, `c[1]`에 `'a'`, `c[2]`에 `'t'`가 연속적으로 저장됩니다.

2.  `char* p = c;`
    *   `char` 타입의 포인터 `p`를 선언합니다.
    *   배열 이름 `c`가 배열-포인터 변환에 의해 배열의 첫 번째 요소(`c[0]`)의 주소로 해석되어 `p`에 할당됩니다. 즉, `p`는 `c[0]`를 가리키게 됩니다.

3.  `char* q = &c[0];`
    *   `char` 타입의 포인터 `q`를 선언합니다.
    *   배열의 첫 번째 요소 `c[0]`의 주소를 명시적으로 `&` 연산자를 사용하여 구한 후 `q`에 할당합니다.
    *   `q` 또한 `c[0]`를 가리키게 됩니다. (결과적으로 `p`와 `q`는 동일한 메모리 주소를 가리킵니다.)

4.  `cout << c[2] << p[2] << q[2];`
    *   `c[2]`: 배열 `c`의 인덱스 2에 해당하는 요소에 접근합니다. 값은 `'t'`입니다.
    *   `p[2]`: 포인터 `p`를 사용하여 배열 인덱싱 방식으로 요소에 접근합니다. 이는 `*(p + 2)`와 동등하며, `p`가 `c[0]`를 가리키므로 `p + 2`는 `c[2]`의 주소를 가리키게 됩니다. 따라서 값은 `'t'`입니다.
    *   `q[2]`: 포인터 `q`를 사용하여 배열 인덱싱 방식으로 요소에 접근합니다. `p[2]`와 동일한 원리로, 값은 `'t'`입니다.
    *   따라서 최종 출력은 `"ttt"`가 됩니다.

**구체적 예시**

메모리 상태 (예시 주소):

| 주소      | 값    | 변수/요소  |
| :-------- | :---- | :--------- |
| `0x1000`  | `'c'` | `c[0]`     |
| `0x1001`  | `'a'` | `c[1]`     |
| `0x1002`  | `'t'` | `c[2]`     |

변수 `p`의 값은 `0x1000` (c[0]의 주소)
변수 `q`의 값은 `0x1000` (c[0]의 주소)

`cout << c[2] << p[2] << q[2];` 실행 시:
*   `c[2]`는 `0x1002` 주소의 값인 `'t'`를 가져옵니다.
*   `p[2]`는 `*(p + 2)`와 같으므로 `*(0x1000 + 2)` 즉, `*(0x1002)`의 값인 `'t'`를 가져옵니다.
*   `q[2]`는 `*(q + 2)`와 같으므로 `*(0x1000 + 2)` 즉, `*(0x1002)`의 값인 `'t'`를 가져옵니다.

결과적으로 화면에 "ttt"가 출력됩니다.

**시험 포인트**

*   ⭐ **배열 이름의 의미**: `char c[]`에서 `c`가 배열의 첫 번째 요소 주소를 나타내는 상수 포인터처럼 동작한다는 것을 이해해야 합니다.
*   ⭐ **배열-포인터 변환**: `char* p = c;`와 같이 배열 이름이 포인터에 대입될 때, 배열 이름이 첫 번째 요소의 주소로 자동 변환(decay)된다는 개념을 정확히 알아야 합니다.
*   ⭐ **포인터와 배열 인덱싱의 관계**: `p[2]`와 `*(p + 2)`가 동일하며, 배열 인덱싱이 내부적으로 포인터 연산을 통해 이루어진다는 것을 이해하는 것이 중요합니다.
*   ⭐ **`&` 연산자의 사용**: `&c[0]`처럼 특정 요소의 주소를 명시적으로 얻는 방법을 알아야 합니다.

## Slide 3

## 데이터구조 (CSED233) 강의 노트

### 핵심 개념

이 슬라이드는 알고리즘의 효율성을 나타내는 **Big O (빅 오) 표기법**에 대한 이해도를 점검합니다. Big O 표기법은 입력 크기 $n$이 무한대로 커질 때, 알고리즘의 시간 복잡도나 공간 복잡도가 얼마나 빠르게 증가하는지 상한(upper bound)을 나타냅니다. 즉, 알고리즘의 **성능 확장성(scalability)**을 비교하는 데 사용됩니다.

질문은 두 가지 시간 복잡도 $O(2^n)$와 $O(n^2)$ 중 어떤 알고리즘이 더 효율적인지를 묻고 있습니다. 더 "좋은" 알고리즘은 일반적으로 Big O 표기법에서 더 낮은 복잡도를 가지는 알고리즘을 의미하며, 이는 입력 크기가 커질수록 자원(시간 또는 메모리) 사용량이 더 느리게 증가한다는 것을 뜻합니다.

### 코드/수식 해설

슬라이드에서 제시된 두 복잡도 함수는 다음과 같습니다.

1.  **$O(2^n)$**: **지수 시간 복잡도(Exponential Time Complexity)**
    *   입력 크기 $n$이 1 증가할 때마다 연산 횟수가 대략 2배로 증가합니다.
    *   이는 매우 빠르게 증가하는 함수로, $n$이 조금만 커져도 상상할 수 없을 정도로 많은 연산을 필요로 하게 됩니다.
    *   일반적으로 실용적이지 않은 매우 비효율적인 알고리즘을 나타냅니다.

2.  **$O(n^2)$**: **2차(Quadratic) 시간 복잡도**
    *   입력 크기 $n$에 따라 연산 횟수가 $n$의 제곱에 비례하여 증가합니다.
    *   선형 복잡도 $O(n)$보다는 느리지만, 지수 복잡도 $O(2^n)$보다는 훨씬 효율적입니다.
    *   중첩된 루프(nested loops)를 사용하는 많은 알고리즘(예: 버블 정렬, 선택 정렬)에서 나타나는 일반적인 복잡도입니다.

두 함수의 성장률을 비교하면, $n$이 충분히 커질 때 $2^n$은 $n^2$보다 훨씬 빠르게 증가합니다.
이를 수식으로 표현하면 다음과 같습니다.
$$ \lim_{n \to \infty} \frac{n^2}{2^n} = 0 $$
이것은 $2^n$이 $n^2$보다 압도적으로 빠르게 증가한다는 것을 의미합니다.

### 구체적 예시

$n$이 1부터 10까지 증가할 때 두 복잡도의 대략적인 연산 횟수를 비교해봅시다.

| n    | $n^2$       | $2^n$              |
| :--- | :---------- | :----------------- |
| 1    | 1           | 2                  |
| 2    | 4           | 4                  |
| 3    | 9           | 8                  |
| 5    | 25          | 32                 |
| 10   | 100         | 1,024              |
| 20   | 400         | 1,048,576          |
| 30   | 900         | 1,073,741,824 (약 10억) |

표에서 볼 수 있듯이, $n=20$만 되어도 $O(n^2)$는 400회의 연산인 반면, $O(2^n)$는 100만 회 이상의 연산을 필요로 합니다. $n=30$이 되면 $O(n^2)$는 여전히 작은 수이지만, $O(2^n)$는 이미 10억 회를 넘어 실생활에서 처리하기 어려운 수준에 도달합니다.

따라서 **$O(n^2)$ 복잡도를 가진 알고리즘이 $O(2^n)$ 복잡도를 가진 알고리즘보다 훨씬 더 좋습니다.**

### 시험 포인트

*   ⭐ **빅 오 표기법의 정의와 의미**: 알고리즘의 상한 시간/공간 복잡도를 나타내며, 입력 크기 $n$이 커질 때의 성능 변화 추이를 예측하는 데 사용된다는 것을 이해해야 합니다.
*   ⭐ **다양한 시간 복잡도 클래스의 성장률 비교**: $O(\log n) < O(n) < O(n \log n) < O(n^2) < O(n^3) < O(2^n) < O(n!)$ 순서로 효율성이 감소한다는 것을 정확히 알고 있어야 합니다.
*   ⭐ **지수 시간 복잡도($O(2^n)$)의 비효율성**: 지수 시간 복잡도는 입력 크기가 조금만 커져도 비실용적이 된다는 점을 강조합니다. 이는 다항 시간 복잡도($O(n^k)$)와의 주요 차이점입니다.
*   ⭐ **"더 좋은 알고리즘"의 의미**: Big O 복잡도 측면에서 "더 좋다"는 것은 성장률이 더 낮아서 입력 크기가 클 때 더 적은 자원을 사용한다는 의미임을 명확히 인지해야 합니다.

## Slide 4

**핵심 개념**

*   **Big O Notation (빅 오 표기법)**: 알고리즘의 효율성을 나타내는 수학적 표기법입니다. 입력 크기 $N$이 무한대로 커질 때, 알고리즘의 실행 시간이나 공간 사용량이 어떻게 증가하는지 점근적인 상한(upper bound)을 나타냅니다. 주로 최악의 경우(worst-case) 성능을 평가할 때 사용됩니다.
*   **시간 복잡도 (Time Complexity)**: 알고리즘이 주어진 문제를 해결하는 데 필요한 연산의 총 횟수를 입력 크기 $N$의 함수로 나타낸 것입니다. $N$이 커질수록 알고리즘의 성능이 어떻게 변화하는지를 보여줍니다.
*   **팩토리얼 시간 복잡도 ($O(N!)$)**: 입력 크기 $N$에 대해 알고리즘의 실행 시간이 $N$의 계승(factorial)에 비례하여 증가하는 경우를 의미합니다. 이는 매우 빠르게 증가하는 복잡도로, $N$이 조금만 커져도 알고리즘의 실행 시간이 현실적으로 감당하기 어려운 수준이 됩니다. 이는 상수, 로그, 선형, 다항, 지수 시간 복잡도보다 훨씬 더 비효율적입니다.

**코드/수식 해설**

*   **Big O 표기법의 수학적 정의**:
    어떤 함수 $T(N)$이 $O(f(N))$이라는 것은, 양의 상수 $c$와 $N_0$가 존재하여 모든 $N \ge N_0$에 대해 다음 부등식이 성립한다는 의미입니다:
    $$T(N) \le c \cdot f(N)$$
*   **팩토리얼 함수 정의**:
    $N! = N \times (N-1) \times (N-2) \times \dots \times 2 \times 1$
*   **팩토리얼 시간 복잡도를 갖는 알고리즘 (순열 생성 예시)**:
    $N$개의 서로 다른 원소를 가지고 만들 수 있는 모든 순열(permutations)을 생성하는 재귀 함수는 대표적으로 $O(N!)$의 시간 복잡도를 가집니다.
    ```c
    #include <stdio.h>
    #include <string.h>

    // 두 문자를 교환하는 헬퍼 함수
    void swap(char* x, char* y) {
        char temp;
        temp = *x;
        *x = *y;
        *y = temp;
    }

    // 문자열의 모든 순열을 생성하는 함수
    void generatePermutations(char* arr, int start, int end) {
        // base case: start와 end가 같아지면 하나의 순열이 완성된 것
        if (start == end) {
            printf("%s\n", arr);
        } else {
            // start 위치부터 end 위치까지 반복
            for (int i = start; i <= end; i++) {
                // 현재 문자를 start 위치로 가져옴
                swap((arr + start), (arr + i));

                // 나머지 부분에 대해 재귀적으로 순열 생성
                generatePermutations(arr, start + 1, end);

                // 백트래킹: 다음 반복을 위해 원래 상태로 되돌림 (교환 복구)
                swap((arr + start), (arr + i));
            }
        }
    }

    /*
    int main() {
        char str[] = "ABC";
        int n = strlen(str);
        generatePermutations(str, 0, n - 1); // "ABC"의 모든 순열 출력
        return 0;
    }
    */
    ```
    이 `generatePermutations` 함수는 첫 번째 위치에 올 수 있는 문자를 $N$가지 선택하고, 그 각각에 대해 두 번째 위치에 올 수 있는 문자를 $N-1$가지 선택하는 식으로 $N$개의 위치를 채웁니다. 이 과정은 최종적으로 $N \times (N-1) \times \dots \times 1 = N!$ 번의 고유한 순열 생성을 시도하게 됩니다.

**구체적 예시**

*   **순열 생성**: 4개의 숫자 (1, 2, 3, 4)로 만들 수 있는 모든 순서를 나열하는 경우를 생각해봅시다.
    *   첫 번째 위치: 4가지 선택 (1, 2, 3, 4)
    *   두 번째 위치: 남은 3개 중 1가지 선택
    *   세 번째 위치: 남은 2개 중 1가지 선택
    *   네 번째 위치: 남은 1개 중 1가지 선택
    총 $4! = 4 \times 3 \times 2 \times 1 = 24$가지의 순열이 있습니다. 만약 숫자가 15개라면 $15! \approx 1.3 \times 10^{12}$가지가 되어 매우 큰 숫자가 됩니다. 일반적인 컴퓨터로도 20!을 계산하는 것은 거의 불가능합니다.
*   **외판원 문제 (Traveling Salesperson Problem, TSP) - 무차별 대입**: $N$개의 도시를 방문해야 하는 외판원 문제에서, 모든 가능한 경로를 일일이 탐색하여 최단 경로를 찾는 무차별 대입(brute-force) 방식은 $(N-1)!$에 비례하는 시간 복잡도를 가집니다 (시작 도시를 고정했다고 가정). 예를 들어, 15개의 도시만 되어도 $14! \approx 8.7 \times 10^{10}$으로 계산량이 엄청나게 많아집니다.

**시험 포인트**

*   ⭐ **다양한 시간 복잡도 클래스 비교**: $O(1) < O(\log N) < O(N) < O(N \log N) < O(N^2) < O(N^3) < O(2^N) < O(N!)$ 순서로 효율성이 감소한다는 것을 이해하고, 각 클래스의 특징과 대표적인 알고리즘 예시를 설명할 수 있어야 합니다.
*   ⭐ **팩토리얼 복잡도의 의미 및 한계**: 팩토리얼 복잡도를 가진 알고리즘은 입력 크기 $N$이 아주 조금만 증가해도 연산량이 기하급수적으로 늘어나, 실제 적용에 심각한 제약이 있음을 이해해야 합니다. (예: $N=20$일 때 이미 실행 불가능 수준)
*   ⭐ **팩토리얼 복잡도를 가지는 알고리즘 예시**: 순열 생성, 외판원 문제의 무차별 대입, 부분집합 합 문제(Subset Sum Problem)의 naive한 탐색 등 대표적인 문제를 기억하고 설명할 수 있어야 합니다.
*   ⭐ **팩토리얼 복잡도를 피하기 위한 접근법**: 실제 문제 해결 시 $O(N!)$ 복잡도를 가지는 알고리즘은 거의 사용되지 않으며, 대신 동적 계획법(Dynamic Programming), 백트래킹(Backtracking)과 가지치기(Pruning), 휴리스틱(Heuristics) 등 복잡도를 낮추는 기법들이 활용된다는 점을 인지해야 합니다.

## List (리스트)

### 핵심 개념

*   **리스트의 정의**: 리스트 $L$은 $n$개의 원소 $a_1, a_2, ..., a_n$으로 구성된 **유한(finite)하며 순서가 있는(ordered) 원소들의 집합**입니다. 각 원소는 특정 위치(position)를 가집니다.
    *   **길이 (Size)**: 리스트 내 원소의 개수를 $n$으로 나타냅니다.
    *   **빈 리스트 (Empty List)**: 원소가 하나도 없는 리스트를 $< >$로 표기하며, 이때 $n=0$입니다.
*   **원소의 위치**: 리스트에서 $a_i$는 $i$번째 위치에 있는 원소를 의미합니다.
    *   **Head (Front)**: 리스트의 맨 앞 부분을 의미합니다.
    *   **Tail (Rear)**: 리스트의 맨 뒷 부분을 의미합니다.
    *   **Current Position**: 리스트 내에서 현재 작업을 수행할 위치를 가리키는 개념입니다. 이 슬라이드에서는 'fence'라는 가상의 구분자를 사용하여 현재 위치를 표현합니다. `fence`는 두 원소 사이, 또는 리스트의 시작/끝에 위치할 수 있습니다.

### 코드/수식 해설

*   **리스트의 일반적인 표기**:
    리스트 $L$은 다음과 같이 표기합니다.
    $$L = < a_1, a_2, ..., a_n >$$
    여기서 $a_1$은 첫 번째 원소, $a_n$은 마지막 원소이며, $n$은 리스트의 길이를 나타냅니다.

*   **빈 리스트**:
    빈 리스트는 다음과 같이 표기하며, 이때 길이는 0입니다.
    $$< > \quad \text{where } n = 0$$

### 구체적 예시

*   **Current Position (Fence) 예시**:
    리스트 `< 20, 23, 12, 15 >`가 있을 때, 특정 위치에 `current` 포인터(혹은 `fence`)를 둘 수 있습니다. 예를 들어, `23`과 `12` 사이에 `current` 포인터가 있다면 다음과 같이 표현할 수 있습니다.
    $$< 20, 23, | \text{ } 12, 15 >$$
    여기서 수직 막대 `|`가 `fence`를 나타내며, 이것이 `current` position입니다.

*   **Current Position에서의 원소 삽입 예시**:
    위 예시에서 `current` position(즉, `fence` 위치)에 원소 `10`을 삽입하면, `10`은 `23` 뒤에 위치하고 `12`와 `15`는 오른쪽으로 한 칸씩 밀려나게 됩니다.
    $$< 20, 23, | \text{ } 12, 15 > \xrightarrow{\text{insert } 10 \text{ at current}} < 20, 23, \text{ } \boxed{10} \text{ } 12, 15 >$$
    삽입 후 `current` position은 일반적으로 삽입된 원소의 바로 뒤에 위치하거나 (다음 원소를 가리키거나), 삽입된 원소 자체를 가리키도록 구현할 수 있습니다. 슬라이드 예시에서는 삽입된 원소 `10`의 바로 뒤에 가상의 `fence`가 있다고 볼 수 있습니다.

### 시험 포인트

*   ⭐ **리스트의 핵심 특성**: 리스트는 **유한(finite)하며 순서가 있는(ordered) 원소들의 컬렉션**이라는 점을 정확히 이해하고 설명할 수 있어야 합니다. 이는 집합(set)과 같은 다른 자료구조와의 중요한 차이점입니다.
*   ⭐ **`Current Position` 및 `Fence` 개념**: `current` position이 무엇을 의미하며, 리스트 연산(특히 삽입, 삭제)에서 어떤 역할을 하는지 이해하는 것이 중요합니다. `fence`는 `current` position을 시각적으로 표현하는 한 방법입니다.
*   ⭐ **삽입/삭제 연산의 동작 방식**: `current` position에서 원소를 삽입했을 때 리스트의 원소들이 어떻게 이동하는지 (`shift`되는지)를 예시를 통해 설명할 수 있어야 합니다. 이는 리스트의 구현 방식(배열 기반 vs. 연결 리스트 기반)에 따라 시간 복잡도가 달라지는 중요한 이유가 됩니다.

## 데이터구조 (CSED233) 슬라이드 노트 - Quiz: 정렬된 리스트 확인

---

### 핵심 개념

주어진 슬라이드는 '정렬된 리스트(Sorted List)'의 개념을 묻는 퀴즈입니다.
*   **정렬된 리스트**: 리스트 내의 요소들이 특정 순서(오름차순 또는 내림차순)로 배열된 상태를 의미합니다. 일반적으로 '정렬되었다'고 하면 특별한 언급이 없는 한 '오름차순(non-decreasing order)'을 의미하며, 각 요소가 바로 다음 요소보다 작거나 같다는 조건을 만족합니다. 즉, $a_0 \le a_1 \le a_2 \le \dots \le a_{n-1}$ 가 성립해야 합니다.

### 코드/수식 해설

리스트가 오름차순으로 정렬되어 있는지 확인하는 간단한 C 코드입니다.

```c
#include <stdbool.h> // for bool type

bool is_sorted_ascending(int arr[], int n) {
    // 빈 리스트 또는 단일 요소 리스트는 항상 정렬된 것으로 간주합니다.
    if (n <= 1) {
        return true;
    }
    // 리스트의 첫 번째 요소부터 마지막에서 두 번째 요소까지 순회하며 인접한 두 요소를 비교합니다.
    for (int i = 0; i < n - 1; i++) {
        // 현재 요소가 다음 요소보다 크다면, 오름차순 정렬이 깨진 것이므로 false를 반환합니다.
        if (arr[i] > arr[i+1]) {
            return false;
        }
    }
    // 모든 요소를 비교했을 때 위 조건을 만족하지 않았다면, 리스트는 정렬된 것입니다.
    return true;
}
```

오름차순 정렬의 수학적 조건은 다음과 같습니다.
리스트 $A = <a_0, a_1, \dots, a_{n-1}>$ 가 오름차순으로 정렬되어 있다는 것은 모든 $i$ ($0 \le i < n-1$)에 대해 다음 조건을 만족한다는 의미입니다.
$$ a_i \le a_{i+1} $$

### 구체적 예시

슬라이드에 제시된 리스트는 `< 20, 23, 12, 15 >` 입니다.
이 리스트가 오름차순으로 정렬되어 있는지 확인해 봅시다.

1.  첫 번째 요소와 두 번째 요소 비교: $20 \le 23$ (참)
2.  두 번째 요소와 세 번째 요소 비교: $23 \le 12$ (거짓, 왜냐하면 $23 > 12$ 이기 때문)

두 번째 비교에서 정렬 조건이 깨졌으므로, 이 리스트는 정렬되지 않았습니다.

### 시험 포인트

*   ⭐ **정렬된 리스트의 정의**: 오름차순 또는 내림차순 정렬의 정확한 의미와 조건을 이해하고 설명할 수 있어야 합니다. (특히 "non-decreasing" vs "strictly increasing"의 차이)
*   ⭐ **정렬 여부 판별**: 주어진 리스트의 정렬 여부를 빠르게 판단할 수 있어야 합니다. 정렬 여부를 판단하기 위한 반복문 로직을 이해하는 것이 중요합니다.
*   ⭐ **최악의 경우 (Worst-Case)**: 정렬 여부를 확인하는 알고리즘의 시간 복잡도는 O(n)이며, 최악의 경우 (예: 마지막 요소에서 정렬이 깨지는 경우 또는 완벽히 정렬된 경우) 리스트의 모든 요소를 검사해야 합니다.

## 데이터구조 (CSED233) - Quiz: Ordered List

이 슬라이드는 **순서가 있는 리스트(Ordered List)** 개념을 묻는 퀴즈를 제시하고 있습니다. 주어진 리스트 `< 20, 23, 12, 15 >`가 정렬되어 있는지 판별하는 문제입니다.

---

### 핵심 개념

순서가 있는 리스트(Ordered List)는 리스트 내의 모든 요소들이 특정 기준(예: 크기)에 따라 **오름차순(ascending order)** 또는 **내림차순(descending order)**으로 정렬되어 있는 자료구조를 의미합니다. 리스트의 인접한 요소들 간에 일관된 관계(예: 항상 다음 요소가 크거나 같음)가 유지되어야 합니다.

### 코드/수식 해설

리스트가 오름차순으로 정렬되어 있는지 확인하는 의사 코드는 다음과 같습니다. 내림차순의 경우 부등호 방향만 반대로 변경하면 됩니다.

```c
// 주어진 리스트가 오름차순으로 정렬되어 있는지 확인하는 함수
bool is_ordered_ascending(int list[], int n) {
    // 리스트의 각 요소와 다음 요소를 비교합니다.
    // 마지막 요소는 비교할 다음 요소가 없으므로 n-1까지만 반복합니다.
    for (int i = 0; i < n - 1; i++) {
        // 현재 요소가 다음 요소보다 크다면, 오름차순이 깨진 것입니다.
        if (list[i] > list[i+1]) {
            return false; // 정렬되어 있지 않음
        }
    }
    return true; // 모든 요소가 오름차순으로 정렬되어 있음
}
```

순서가 있는 리스트의 수학적 정의 (오름차순 기준):
리스트 $L = \langle L_0, L_1, \ldots, L_{n-1} \rangle$가 오름차순으로 정렬되어 있다는 것은 모든 $i$ ($0 \le i < n-1$)에 대해 다음 조건을 만족한다는 의미입니다.
$$ L_i \le L_{i+1} $$

### 구체적 예시

주어진 리스트 `< 20, 23, 12, 15 >`를 분석하여 순서가 있는지 확인해 봅시다.

1.  **20과 23 비교**: $20 \le 23$ (오름차순 조건을 만족합니다.)
2.  **23과 12 비교**: $23 > 12$ (오름차순 조건을 만족하지 않습니다. 즉, 순서가 깨졌습니다.)

따라서, 이 리스트는 오름차순으로 정렬되어 있지 않습니다. 내림차순으로도 20 < 23이므로 첫 비교에서부터 조건을 만족하지 않습니다. 결론적으로, 이 리스트는 **정렬되어 있지 않습니다 (Not ordered)**.

**실생활 비유**: 서점에 진열된 책들이 ISBN 번호 순서대로 꽂혀있는지 확인하는 것과 같습니다. 만약 번호 100번 책 다음에 200번 책이 오고, 그 다음에 150번 책이 온다면, 그 책꽂이의 책들은 순서대로 정렬되어 있지 않은 것입니다.

### 시험 포인트

*   ⭐ **순서가 있는 리스트의 정의**: 오름차순, 내림차순 정렬의 정확한 의미와 조건을 이해하고 있어야 합니다.
*   ⭐ **정렬 여부 판별**: 주어진 리스트의 정렬 상태(오름차순, 내림차순, 비정렬)를 신속하게 판별할 수 있어야 합니다. 특히, 정렬 조건을 만족하지 않는 첫 번째 지점을 찾는 것이 중요합니다.
*   ⭐ **정렬된 리스트의 활용**: 정렬된 리스트는 이진 탐색(Binary Search)과 같은 효율적인 탐색 알고리즘이나 병합 정렬(Merge Sort)의 병합 단계 등 다양한 알고리즘에서 핵심적으로 활용됩니다. 왜 정렬이 필요한지, 정렬된 상태가 어떤 이점을 주는지 이해하는 것이 중요합니다.

### Sorted vs. Ordered

**핵심 개념**:
`Sorted`와 `Ordered`는 자료구조에서 종종 혼동될 수 있지만, 서로 다른 의미를 가집니다.

*   **Sorted (정렬된)**: 리스트 내의 요소들이 특정 기준(예: 숫자 값의 오름차순/내림차순, 알파벳 순서 등)에 따라 논리적으로 배열된 상태를 의미합니다. 요소들 간의 **값의 관계**에 의해 순서가 결정됩니다. `Sorted` 리스트는 항상 `Ordered` 리스트이기도 합니다.
*   **Ordered (순서가 있는)**: 리스트 내의 요소들이 특정 시퀀스나 위치를 가지고 있는 상태를 의미합니다. 요소들이 어떤 순서로 존재한다는 사실 자체를 강조하며, 이 순서가 반드시 값에 의한 정렬 기준을 따르지는 않습니다. 즉, 요소가 삽입된 순서, 접근된 순서 등 임의의 기준에 따라 순서가 있을 수 있습니다.

**코드/수식 해설**:
이 슬라이드에는 직접적인 코드나 수식은 포함되어 있지 않습니다. 개념적인 차이를 예시를 통해 설명하고 있습니다.

**구체적 예시**:

*   **Sorted list**: `< 12, 15, 20, 23 >`
    *   이 리스트의 요소들은 숫자 값의 오름차순으로 정렬되어 있습니다. 각 요소는 이전 요소보다 크거나 같습니다.
    *   이 리스트는 '정렬된' 동시에 '순서가 있는' 리스트입니다.
*   **Ordered list**: `< 20, 23, 12, 15 >`
    *   이 리스트의 요소들은 특정 순서를 가지고 있지만, 값에 따라 정렬되어 있지는 않습니다. 예를 들어, 이 값들이 삽입된 순서대로 저장되어 있을 수 있습니다.
    *   이 리스트는 '순서가 있는' 리스트이지만, '정렬된' 리스트는 아닙니다.
*   **Ordered list (또 다른 예시)**: `< 12, 15, 20, 23 >`
    *   이 예시는 위 'Sorted list'와 동일합니다. 이는 `Sorted` 리스트가 곧 `Ordered` 리스트의 한 종류임을 보여줍니다.

**실생활 비유**:
*   **Sorted list**: 백화점 진열대의 **사이즈별로 정렬된 옷**들. S, M, L, XL 순서로 배열되어 있어 특정 기준(사이즈)에 따라 순서가 정해져 있습니다. (Ordered & Sorted)
*   **Ordered list**: 내 방 책상 위에 **아무렇게나 놓여 있는 책**들. 책들이 놓여 있는 순서 자체는 있지만 (맨 왼쪽부터, 맨 오른쪽부터 등), 그 순서가 책 제목 가나다순이나 저자 순 같은 특정 기준을 따르지는 않습니다. 오늘 내가 읽은 순서대로 쌓았을 수도 있습니다. (Ordered, but not necessarily Sorted)

**시험 포인트**:
*   ⭐ `Sorted`와 `Ordered`의 개념적 차이를 명확히 이해하고 각각의 특성을 설명할 수 있어야 합니다.
*   ⭐ **`Sorted list`는 항상 `Ordered list`이지만, `Ordered list`는 항상 `Sorted list`인 것은 아님**을 실제 예시를 들어 설명하거나 구분하는 문제가 출제될 수 있습니다. 이 핵심적인 관계를 반드시 기억하세요.

## Slide 9

**핵심 개념**:
*   **리스트 (List)**: 순서가 있는 유한한 원소들의 컬렉션을 추상적으로 정의합니다. 각 원소는 고유한 위치(position)를 가지며, 이 슬라이드에서는 리스트 $L$을 순서대로 정렬된 원소들의 나열 $L = <a_1, a_2, ..., a_n>$ 형태로 표현합니다.
*   **리스트 연산 (List Operations)**: 리스트 자료구조가 제공해야 할 기본적인 기능들을 명세합니다. 이 연산들은 특정 구현 방식(예: 배열, 연결 리스트)에 얽매이지 않고, 리스트라는 추상 데이터 타입(ADT)의 핵심 동작을 정의합니다.

**코드/수식 해설**:

슬라이드에 제시된 리스트 $L$의 기본 형태는 다음과 같습니다:
$$ L = <a_1, a_2, ..., a_{p-1}, \mathbf{a_p}, a_{p+1}, ..., a_n> $$
여기서 $a_i$는 리스트의 $i$번째 원소이며, $p$는 특정 위치를 나타냅니다.

주요 연산들의 결과로 인한 리스트의 변화는 다음과 같습니다:

*   **`Insert(x, p, L)`**: 리스트 $L$의 위치 $p$에 새로운 원소 $x$를 삽입합니다.
    *   결과: $L' = <a_1, a_2, ..., a_{p-1}, \mathbf{x}, a_p, a_{p+1}, ..., a_n>$
    *   기존 위치 $p$부터의 모든 원소들이 한 칸씩 뒤로 밀려납니다.
*   **`Delete(p, L)`**: 리스트 $L$의 위치 $p$에 있는 원소를 삭제합니다.
    *   결과: $L' = <a_1, a_2, ..., a_{p-1}, a_{p+1}, ..., a_n>$
    *   기존 위치 $p+1$부터의 모든 원소들이 한 칸씩 앞으로 당겨집니다.

다른 연산들은 주로 리스트의 정보를 조회하거나 상태를 변경합니다:

*   **`Next(p, L)`**: 리스트 $L$에서 위치 $p$의 바로 다음 위치 또는 포인터를 반환합니다.
*   **`Previous(p, L)`**: 리스트 $L$에서 위치 $p$의 바로 이전 위치 또는 포인터를 반환합니다.
*   **`Locate(x, L)`**: 리스트 $L$에서 원소 $x$가 처음 나타나는 위치 또는 포인터를 반환합니다. (찾지 못할 경우 특정 값을 반환할 수 있음)
*   **`Retrieve(p, L)`**: 리스트 $L$에서 위치 $p$에 있는 원소 자체를 반환합니다.
*   **`MakeNull(L)`**: 리스트 $L$을 비어있는 상태로 만듭니다. 빈 리스트의 끝을 나타내는 `END(L)` 위치를 반환할 수 있습니다.
*   **`First(L)`**: 리스트 $L$의 첫 번째 위치 또는 포인터를 반환합니다.

**구체적 예시**:

초기 리스트 $L = <10, 20, 30, 40>$이 있다고 가정합시다 (여기서 위치 $p$는 1부터 시작하는 인덱스입니다).

1.  **`Insert(25, 3, L)`**: 원소 25를 3번째 위치에 삽입합니다.
    *   결과: $L = <10, 20, \mathbf{25}, 30, 40>$
2.  **`Delete(4, L)`**: 4번째 위치의 원소를 삭제합니다 (현재 리스트에서 30이 삭제됨).
    *   결과: $L = <10, 20, 25, 40>$
3.  **`Retrieve(2, L)`**: 2번째 위치의 원소를 조회합니다.
    *   결과: $20$
4.  **`Locate(40, L)`**: 원소 40의 위치를 찾습니다.
    *   결과: $4$ (4번째 위치)
5.  **`Next(3, L)`**: 3번째 위치의 다음 위치를 찾습니다.
    *   결과: $4$
6.  **`Previous(3, L)`**: 3번째 위치의 이전 위치를 찾습니다.
    *   결과: $2$
7.  **`First(L)`**: 리스트의 첫 번째 위치를 찾습니다.
    *   결과: $1$
8.  **`MakeNull(L)`**: 리스트 $L$을 비웁니다.
    *   결과: $L = < >$ (빈 리스트)

**시험 포인트**:

*   ⭐ **각 리스트 연산의 기능과 목적을 정확히 이해**: 특히 `Locate(x, L)`는 원소 `x`의 *위치*를 반환하고, `Retrieve(p, L)`는 위치 `p`의 *원소*를 반환한다는 점을 명확히 구분해야 합니다.
*   ⭐ **`Insert`와 `Delete` 연산 시 리스트 원소들의 변화 방식**: 삽입 시 뒤로 밀림, 삭제 시 앞으로 당겨짐을 이해해야 합니다. 이는 구현 방식(예: 배열 vs. 연결 리스트)에 따라 성능(시간 복잡도)에 큰 영향을 미칩니다.
*   ⭐ **추상 데이터 타입(ADT)으로서의 리스트 연산**: 슬라이드에 명시된 연산들은 리스트가 제공해야 하는 *기능*에 대한 정의이며, 특정 프로그래밍 언어나 자료구조 구현에 종속되지 않는다는 점을 인지해야 합니다. 실제 구현 시에는 이들의 효율성(시간/공간 복잡도)이 중요한 평가 요소가 됩니다.
*   ⭐ **위치(position)의 개념**: 리스트에서 "위치"는 단순히 정수 인덱스일 수도 있고, 연결 리스트의 경우 포인터가 될 수도 있습니다. 이는 구현에 따라 달라질 수 있음을 이해해야 합니다.

## Slide 10

**핵심 개념**:
*   **배열 기반 리스트 (Array-Based List)**: 리스트 추상 자료형(ADT)을 구현하는 가장 기본적인 방법 중 하나입니다. 고정된 크기의 메모리 블록(배열)을 사용하여 리스트의 원소들을 메모리에 연속적으로 저장합니다.
*   **논리적 길이와 물리적 길이**:
    *   `max`: 배열이 할당받은 실제 물리적 최대 크기입니다. 즉, 이 리스트가 수용할 수 있는 최대 원소의 개수를 나타냅니다.
    *   `size` 또는 `last`: 현재 리스트에 실제로 저장되어 있는 유효한 원소의 개수(`size`) 또는 마지막 유효 원소의 인덱스(`last`)를 나타냅니다. 슬라이드의 예시(`L = <a, b, c, d>`)에서는 `size = 4` 또는 `last = 4`로 표시되어, 현재 리스트에 4개의 원소가 있음을 나타냅니다. 1-based 인덱싱을 가정할 경우 `size`와 `last`는 같은 값을 가집니다.
*   **1-based 인덱싱**: 슬라이드의 그림에서 배열의 인덱스를 `1, 2, 3, 4, ...`로 표기하고 있습니다. 이는 실제 대부분의 프로그래밍 언어에서 사용하는 0-based 인덱싱과 다를 수 있지만, 리스트의 '위치(position)' 개념을 직관적으로 표현하기 위해 사용되는 경우가 많습니다.

**코드/수식 해설**:
*   **배열 선언 및 상태 변수 (C/C++ 예시)**:
    ```c
    #define MAX_LIST_SIZE 100 // 배열의 물리적 최대 크기 (max)
    char L[MAX_LIST_SIZE];    // char 타입의 배열 L 선언
    int size = 0;             // 현재 리스트에 저장된 요소의 개수
    // 또는
    // int last = 0;           // 마지막 요소의 1-based 인덱스
    ```
    슬라이드에서는 `char L[max]`로 선언되어 있으며, `max`는 `MAX_LIST_SIZE`와 같은 역할을 합니다.
*   **요소 삽입 (`Insert(x, p, L)`)의 일반적인 동작**:
    슬라이드에서 질문으로 제시된 `Insert(x, 2, L)`는 값 `x`를 리스트 `L`의 2번째 위치(position)에 삽입하라는 의미입니다. 이 연산을 수행하기 위한 일반적인 단계는 다음과 같습니다:
    1.  리스트가 가득 찼는지 확인합니다 (현재 `size`가 `max`와 같은지). 가득 찼다면 삽입할 수 없습니다.
    2.  삽입할 위치(`p`)부터 현재 리스트의 마지막 요소(`last` 또는 `size`)까지의 모든 요소들을 한 칸씩 뒤로 이동시킵니다. (예: `L[last+1] = L[last]`, `L[last] = L[last-1]`, ..., `L[p+1] = L[p]`).
    3.  비워진 `L[p]` 위치에 새로운 원소 `x`를 저장합니다.
    4.  리스트의 `size` (또는 `last`) 값을 1 증가시킵니다.
*   **삽입 연산의 시간 복잡도**:
    배열의 중간에 요소를 삽입할 경우, 삽입 위치 뒤의 모든 요소들을 이동시켜야 합니다. 최악의 경우 (리스트의 맨 앞에 삽입할 때) $N$개의 모든 요소를 이동시켜야 합니다.
    $$T_{insert}(N) = O(N)$$
    여기서 $N$은 현재 리스트의 요소 개수($size$)입니다.

**구체적 예시**:
현재 리스트 `L = < a, b, c, d >`는 배열에 다음과 같이 1-based 인덱싱으로 저장되어 있다고 가정합니다:
`L[1]='a'`, `L[2]='b'`, `L[3]='c'`, `L[4]='d'`. 현재 `size = 4` (또는 `last = 4`).

`Insert(x, 2, L)` (값 `x`를 2번째 위치에 삽입) 연산을 수행하는 과정:
1.  **공간 확인**: `size = 4`는 `max`보다 작다고 가정하므로 삽입 가능합니다.
2.  **요소 이동**: 2번째 위치(`L[2]`)에 `x`를 넣기 위해, `L[2]`부터 `L[4]`까지의 요소들을 한 칸씩 뒤로 밀어냅니다.
    *   `L[5] = L[4]` (즉, `d`를 `L[5]`로 이동)
    *   `L[4] = L[3]` (즉, `c`를 `L[4]`로 이동)
    *   `L[3] = L[2]` (즉, `b`를 `L[3]`로 이동)
    이 과정 후 배열은 논리적으로 `[a, (empty), b, c, d, ...]` 상태가 됩니다. (여기서 `(empty)`는 원래 `b`가 있던 자리)
3.  **새 값 삽입**: 비워진 2번째 위치(`L[2]`)에 `x`를 저장합니다.
    *   `L[2] = x`
4.  **크기 갱신**: 리스트에 새로운 요소가 추가되었으므로 `size`를 1 증가시킵니다.
    *   `size`는 `4`에서 `5`로 변경됩니다 (또는 `last`가 `4`에서 `5`로 변경).

최종 리스트 상태: `L = < a, x, b, c, d >`
배열 저장 상태 (1-based): `L[1]='a'`, `L[2]='x'`, `L[3]='b'`, `L[4]='c'`, `L[5]='d'`.

**시험 포인트**:
*   ⭐ **배열 기반 리스트의 장점**:
    *   **임의 접근(Random Access)**: 특정 인덱스(위치)의 원소에 $O(1)$ 시간에 즉시 접근할 수 있습니다. 이는 배열이 메모리에 연속적으로 할당되기 때문에 가능합니다.
    *   **메모리 효율성**: 각 원소 저장에 필요한 오버헤드가 적습니다 (포인터 등을 저장할 필요가 없음).
*   ⭐ **배열 기반 리스트의 단점**:
    *   **고정된 크기**: `max`로 배열의 크기가 한정되므로, 리스트가 `max`를 초과하여 성장할 수 없습니다. (동적 배열은 이 문제를 완화하지만, 크기 재할당 시 여전히 $O(N)$의 비용이 발생합니다.)
    *   **비효율적인 삽입/삭제**: 리스트의 중간에 원소를 삽입하거나 삭제할 때, 뒤따르는 모든 원소들을 이동시켜야 하므로 최악의 경우 $O(N)$의 시간 복잡도를 가집니다.
*   ⭐ `size` (또는 `last`) 변수가 배열 기반 리스트에서 어떤 역할을 하는지 정확히 이해하고 설명할 수 있어야 합니다. 이 변수는 현재 리스트의 논리적 범위를 나타내는 필수적인 요소입니다.
*   ⭐ 0-based 인덱싱과 1-based 인덱싱의 차이를 이해하고, 주어진 조건에 따라 리스트 연산을 어떻게 구현할지 (예: 인덱스 `p`를 0-based 배열에 매핑하는 방법) 설명할 수 있어야 합니다.

## Array-Based List 구현 (2)

### 핵심 개념

이 슬라이드는 배열(Array)을 기반으로 리스트(List)를 구현하는 방법을 설명합니다. 특히 리스트의 현재 상태를 유지하고, 중간에 새로운 요소를 삽입하는 `Insert` 연산의 개념을 보여줍니다.

*   **배열 기반 리스트**: 리스트의 요소들을 연속적인 메모리 공간에 할당된 배열에 저장합니다. `char L[max]`는 최대 `max`개의 문자를 저장할 수 있는 배열 `L`을 나타냅니다.
*   **리스트 크기 관리**: 배열은 고정된 크기를 가지지만, 리스트의 실제 요소 수는 가변적입니다. 이를 위해 `size` 변수 또는 `last` 변수를 사용합니다.
    *   `Integer size = 4`: 현재 리스트에 포함된 요소의 총 개수를 나타냅니다.
    *   `Integer last = the position of the last element`: 마지막 요소의 **위치**를 나타냅니다. 슬라이드에서 1-based 인덱싱을 사용하고 있으므로, `size`와 `last`는 동일한 값을 가집니다. 즉, 리스트에 4개의 요소가 있다면 `size=4`이고 마지막 요소의 위치는 4입니다.
*   **삽입(Insert) 연산**: `Insert(x, 2, L)`는 리스트 `L`의 2번 위치에 요소 `x`를 삽입하라는 의미입니다. 배열 기반 리스트에서 중간에 요소를 삽입할 경우, 삽입 위치 이후의 모든 요소들을 한 칸씩 뒤로 밀어내야 빈 공간을 확보할 수 있습니다.

### 코드/수식 해설

슬라이드에는 직접적인 코드가 제시되어 있지 않지만, `Insert` 연산에 대한 가상의 알고리즘 흐름을 유추할 수 있습니다.

**`Insert(element x, position p, List L)` 연산의 일반적인 의사 코드:**

```cpp
// L은 배열, current_size는 현재 리스트의 요소 개수
// max_size는 배열의 최대 크기 (L[max_size])
// (여기서는 1-based 인덱싱을 가정)

Function Insert(x, p, L, current_size, max_size):
    // 1. 리스트가 가득 찼는지 확인
    If current_size == max_size:
        Print "Error: List is full"
        Return
    
    // 2. 삽입 위치가 유효한지 확인 (1 <= p <= current_size + 1)
    If p < 1 or p > current_size + 1:
        Print "Error: Invalid position"
        Return

    // 3. 삽입 위치 p부터 마지막 요소까지 한 칸씩 뒤로 밀기
    //    마지막 요소 (current_size)부터 p까지 역순으로 이동
    For i from current_size down to p:
        L[i + 1] = L[i]
    
    // 4. 새 요소 x를 위치 p에 삽입
    L[p] = x
    
    // 5. 리스트 크기 증가
    current_size = current_size + 1
    
    Return
```

위의 의사 코드에서 **핵심은 `for` 루프를 통해 요소를 쉬프트(shift)하는 부분**입니다. 이 작업은 삽입 위치 뒤에 있는 요소의 개수에 비례합니다.

### 구체적 예시

슬라이드에 제시된 예시를 따라가 봅시다.

**초기 상태:**
*   리스트 `L = < a, b, c, d >`
*   `char L[max]` 배열에 저장된 상태:
    *   `L[1] = 'a'`
    *   `L[2] = 'b'`
    *   `L[3] = 'c'`
    *   `L[4] = 'd'`
*   `size = 4` (또는 `last = 4`)

**연산:** `Insert (x, 2, L)` - 요소 `x`를 2번 위치에 삽입

**동작 과정:**

1.  **공간 확인**: 리스트가 가득 차 있지 않은지 확인합니다. (여기서는 `max`가 충분히 크다고 가정)
2.  **유효성 확인**: 삽입 위치 2는 `1 <= 2 <= 4+1` 이므로 유효합니다.
3.  **요소 쉬프트**: 2번 위치 이후의 요소들을 한 칸씩 뒤로 밀어냅니다.
    *   `L[4]` (`'d'`)를 `L[5]`로 이동: `L[5] = L[4]`
    *   `L[3]` (`'c'`)를 `L[4]`로 이동: `L[4] = L[3]`
    *   `L[2]` (`'b'`)를 `L[3]`로 이동: `L[3] = L[2]`
    *   (이제 `L[2]`는 비어 있는 상태가 됩니다.)
4.  **요소 삽입**: `L[2]`에 `x`를 삽입합니다: `L[2] = 'x'`
5.  **크기 업데이트**: `size`를 1 증가시킵니다. `size = 5`

**최종 상태:**
*   리스트 `L = < a, x, b, c, d >`
*   배열에 저장된 상태:
    *   `L[1] = 'a'`
    *   `L[2] = 'x'`
    *   `L[3] = 'b'`
    *   `L[4] = 'c'`
    *   `L[5] = 'd'`
*   `size = 5` (또는 `last = 5`)

### 시험 포인트

*   ⭐ **배열 기반 리스트의 장단점**:
    *   **장점**: 임의 접근(random access)이 $O(1)$로 빠릅니다. (예: `L[k]` 접근)
    *   **단점**: 삽입(Insert) 또는 삭제(Delete) 연산 시 많은 요소들을 쉬프트해야 하므로, 평균적으로 $O(N)$의 시간 복잡도를 가집니다. 배열이 가득 찼을 때 확장하려면 새 배열을 할당하고 모든 요소를 복사해야 합니다.
*   ⭐ **삽입/삭제 연산의 시간 복잡도**: 배열 중간에 요소를 삽입/삭제하는 경우, 최악의 시나리오(예: 첫 번째 위치에 삽입/삭제)에서는 $N$개의 요소를 쉬프트해야 하므로 시간 복잡도는 $O(N)$입니다. 맨 끝에 삽입/삭제하는 경우는 $O(1)$입니다.
*   ⭐ **`size`와 `last` 변수의 역할**: 리스트의 현재 유효한 데이터 범위를 추적하는 데 사용됩니다. 1-based 인덱싱에서 `size`와 `last`는 같은 의미로 사용될 수 있습니다. 0-based 인덱싱에서는 `size = last + 1`이 됩니다. (문제에서 어떤 인덱싱을 사용하는지 주의!)
*   ⭐ **Full/Empty 조건**: 배열 기반 리스트에서 `is_full()` 또는 `is_empty()`와 같은 연산 구현 방법을 이해해야 합니다. (`is_full()`은 `size == max` 일 때, `is_empty()`는 `size == 0` 일 때)

## Slide 12

**핵심 개념**:
*   **배열 기반 리스트의 삽입(Insert) 연산**: 배열은 물리적으로 연속된 메모리 공간에 데이터를 저장합니다. 특정 위치(`i`)에 새로운 요소(`x`)를 삽입하려면, 해당 위치(`i`)부터 리스트의 끝까지의 모든 요소들을 한 칸씩 뒤로(오른쪽으로) 이동시켜야 합니다. 이 과정을 **쉬프트(shift) 연산**이라고 합니다. 쉬프트 후 빈 공간에 새로운 요소를 삽입합니다.
*   **배열 기반 리스트의 삭제(Delete) 연산**: 특정 위치(`p`)의 요소를 삭제할 경우, 해당 요소를 제거한 후 발생하는 빈 공간을 채우기 위해 삭제된 위치(`p`) 이후의 모든 요소들을 한 칸씩 앞으로(왼쪽으로) 이동시켜야 합니다. 이 역시 쉬프트 연산의 일종입니다.
*   **배열 기반 리스트의 탐색(Locate) 연산**: 특정 값(`x`)을 가진 요소가 리스트 내 어느 위치에 있는지 찾으려면, 배열의 첫 번째 요소부터 순차적으로 탐색하여 해당 값을 찾을 때까지 비교해야 합니다.
*   이러한 연산들은 리스트의 `size` (현재 요소의 개수)에 따라 수행 시간이 달라지며, 이는 주로 시간 복잡도로 표현됩니다.

**코드/수식 해설**:
*   **Insert 함수 시그니처**:
    ```c
    void insert(char x, int i, char *L)
    ```
    `x`는 삽입할 데이터, `i`는 삽입할 인덱스(위치), `L`은 리스트를 나타내는 배열 포인터입니다. 배열 기반 리스트는 보통 1-based index (1부터 시작)를 사용하거나 0-based index (0부터 시작)를 사용하지만, 슬라이드 예시는 1부터 시작하는 것으로 보입니다.

*   **시간 복잡도**:
    배열 기반 리스트에서 `Insert`, `Delete`, `Locate` 연산의 시간 복잡도는 모두 리스트의 길이에 비례합니다.
    *   삽입 (`Insert`): $O(n)$
    *   삭제 (`Delete`): $O(n)$
    *   탐색 (`Locate`): $O(n)$
    여기서 $n$은 리스트에 현재 저장된 요소의 개수를 의미합니다. 쉬프트 연산이나 순차 탐색이 최악의 경우 $n$개의 요소를 다루어야 하기 때문입니다.

**구체적 예시**:
1.  **Insert 예시 (`insert('x', 2, L)`)**:
    *   초기 리스트 상태 (size=4): `[a, b, c, d, _, _, ..., max]` (각각 1, 2, 3, 4번 인덱스)
    *   `'x'`를 2번 인덱스에 삽입하고자 합니다.
    *   `d` (4번 인덱스)를 5번 인덱스로 이동: `[a, b, c, _, d, _, ..., max]`
    *   `c` (3번 인덱스)를 4번 인덱스로 이동: `[a, b, _, c, d, _, ..., max]`
    *   `b` (2번 인덱스)를 3번 인덱스로 이동: `[a, _, b, c, d, _, ..., max]`
    *   2번 인덱스에 `'x'` 삽입: `[a, x, b, c, d, _, ..., max]`
    *   리스트의 `size`는 4에서 5로 증가합니다.
    *   이 과정에서 최악의 경우 (1번 인덱스에 삽입 시) `n`개의 요소가 이동하므로, 선형 시간 복잡도 $O(n)$을 가집니다.

2.  **Delete 예시 (슬라이드에 명시되진 않았지만 유추 가능)**:
    *   리스트 상태 (size=5): `[a, x, b, c, d, _, ..., max]`
    *   `delete(2, L)` (2번 인덱스 요소 `x` 삭제)를 호출했다고 가정합니다.
    *   `b` (3번 인덱스)를 2번 인덱스로 이동: `[a, b, _, c, d, _, ..., max]`
    *   `c` (4번 인덱스)를 3번 인덱스로 이동: `[a, b, c, _, d, _, ..., max]`
    *   `d` (5번 인덱스)를 4번 인덱스로 이동: `[a, b, c, d, _, _, ..., max]`
    *   리스트의 `size`는 5에서 4로 감소합니다.
    *   이 과정에서도 최악의 경우 (1번 인덱스 요소 삭제 시) `n`개의 요소가 이동하므로, 선형 시간 복잡도 $O(n)$을 가집니다.

3.  **Locate 예시 (슬라이드에 명시되진 않았지만 유추 가능)**:
    *   리스트 상태 (size=5): `[a, x, b, c, d, _, ..., max]`
    *   `locate('c', L)`를 호출했다고 가정합니다.
    *   1번 인덱스부터 순차적으로 `a, x, b`와 `'c'`를 비교합니다.
    *   4번 인덱스에서 `'c'`를 발견하고, 해당 인덱스 4를 반환합니다.
    *   최악의 경우 (찾는 요소가 리스트의 마지막에 있거나, 리스트에 없는 경우) `n`개의 요소를 모두 확인해야 하므로, 선형 시간 복잡도 $O(n)$을 가집니다.

**시험 포인트**:
*   ⭐ **배열 기반 리스트의 삽입/삭제 시 성능 병목 지점**: 배열 기반 리스트에서 특정 위치에 요소를 삽입하거나 삭제할 때, 해당 위치 이후의 모든 요소들을 이동시켜야 하는 **쉬프트(shift) 연산**이 발생한다는 점을 이해하는 것이 핵심입니다.
*   ⭐ **시간 복잡도 분석**: `Insert`, `Delete`, `Locate` 연산 모두 최악의 경우 리스트의 길이 $n$에 비례하는 $O(n)$의 시간 복잡도를 갖습니다. 이는 배열의 연속적인 메모리 할당 특성에서 기인하는 주요 단점입니다.
*   ⭐ 배열의 `max` (최대 용량)와 현재 리스트의 `size` (현재 요소 개수)를 구분하고, `Insert` 시 `size` 증가, `Delete` 시 `size` 감소가 필요하다는 점을 명확히 알아야 합니다.

## Slide 13

**핵심 개념**:
*   **단일 연결 리스트 (Singly-linked list / One-way list)**: 데이터를 저장하는 노드들이 체인처럼 연결된 선형 자료구조입니다. 각 노드는 자신의 데이터와 함께 다음 노드의 주소(포인터)를 저장하며, 이 포인터를 통해 다음 노드로 이동할 수 있습니다. "One-way"라는 이름처럼 한 방향으로만 탐색이 가능합니다.
*   **노드 (Node 또는 Cell)**: 연결 리스트의 기본 구성 요소입니다. 슬라이드에서 `Node (or cell) = info + link`로 표현되었듯이, 각 노드는 저장할 **정보(info)**와 다음 노드를 가리키는 **링크(link)** 두 부분으로 구성됩니다.
*   **삽입 연산 (Insert)**: `Insert(x, p, L)`는 리스트 `L`의 특정 위치 `p`에 새로운 요소 `x`를 삽입하는 연산입니다. 슬라이드에서는 `p = current`로, 현재 위치(`b`와 `c` 사이)에 `x`를 삽입하는 과정을 보여줍니다.

**코드/수식 해설**:

*   **노드 구조체 정의 (C/C++ 예시)**:
    ```c
    Struct item {
        char info;       // 노드에 저장될 실제 데이터
        item * link;     // 다음 'item' 노드를 가리키는 포인터
    };
    ```
    이 C/C++ 구조체는 연결 리스트의 각 노드가 어떻게 메모리에 표현되는지를 정의합니다. `info` 필드는 노드가 담고 있는 값을 저장하며, `link` 필드는 자기 자신과 동일한 타입(`item`)의 다른 노드를 가리키는 포인터입니다. 이 `link` 포인터를 통해 노드들이 서로 연결됩니다.

*   **리스트 상태 표현**:
    *   **삽입 전**: $L = <a, b, | c, d>$
        리스트 `L`은 요소 `a`, `b`, `c`, `d`를 순서대로 가지고 있으며, 수직선 `|`은 현재 삽입될 위치를 나타냅니다. 즉, `b` 다음, `c` 이전에 새로운 요소가 삽입될 예정입니다.
    *   **삽입 후**: $L = <a, b, | x, c, d>$
        요소 `x`가 성공적으로 삽입된 후의 리스트 상태를 보여줍니다. 이제 리스트의 순서는 `a`, `b`, `x`, `c`, `d`입니다.

**구체적 예시**:

도서관에서 책을 정리하는 상황을 생각해 봅시다.
1.  **초기 리스트**: 책 `a`, `b`, `c`, `d`가 순서대로 꽂혀 있습니다. (`a` 다음 `b`, `b` 다음 `c`, `c` 다음 `d`)
2.  **노드 구조**: 각 책은 책 자체의 내용(`info`)과 다음 책이 꽂힌 칸의 위치를 가리키는 작은 메모(`link`)를 가지고 있습니다.
3.  **삽입 연산**: 여러분이 새 책 `x`를 구매했습니다. 이 책 `x`를 `b`와 `c` 사이에 꽂고 싶습니다.
    *   먼저, 새 책 `x`를 위한 공간을 만들고, `x`의 메모에 '다음 책은 `c`이다'라고 적습니다. (`x`의 `link`가 `c`를 가리키도록 설정)
    *   다음으로, `b` 책의 메모를 '다음 책은 `c`이다'에서 '다음 책은 `x`이다'로 수정합니다. (`b`의 `link`가 `x`를 가리키도록 변경)
    *   이제 책장은 `a` -> `b` -> `x` -> `c` -> `d` 순서로 연결됩니다.

**시험 포인트**:

*   ⭐ **단일 연결 리스트 노드의 구조와 동작 방식**: `info`와 `link` 필드의 역할, 그리고 `link` 포인터를 통해 다음 노드로 어떻게 이동하는지 정확히 설명할 수 있어야 합니다.
*   ⭐ **삽입 연산의 포인터 조작**: 주어진 위치 `p`에 새로운 노드 `x`를 삽입할 때, 어떤 포인터들을 어떻게 변경해야 하는지 단계를 명확히 설명하고 그림으로 표현할 수 있어야 합니다. 특히, 새로운 노드의 `link`를 먼저 설정한 후, 이전 노드의 `link`를 새로운 노드로 연결하는 순서를 이해하는 것이 중요합니다.
*   ⭐ **시간 복잡도**: 만약 삽입될 위치의 직전 노드(예: `b`)에 대한 포인터(`p`)를 이미 알고 있다면, 삽입 연산 자체는 상수 시간 **$O(1)$**에 수행됩니다. (새로운 노드 생성 및 두 개의 포인터만 변경하면 되기 때문)

### 핵심 개념

단일 연결 리스트(singly-linked list)를 이중 연결 리스트(doubly-linked list)로 변경하는 것은 각 노드에 이전 노드를 가리키는 `prev` 포인터를 추가하고, 이 포인터들을 올바르게 설정하는 과정입니다.

*   **단일 연결 리스트 (Singly-linked list)**: 각 노드는 데이터와 다음 노드를 가리키는 하나의 포인터(`next`)만을 가집니다. 특정 노드에서 다음 노드로의 이동은 쉽지만, 이전 노드로 돌아가려면 리스트의 시작부터 다시 탐색해야 합니다.
    ```c
    struct item {
        char info;
        struct item *link; // Or item *link;
    };
    ```
*   **이중 연결 리스트 (Doubly-linked list)**: 각 노드는 데이터, 다음 노드를 가리키는 포인터(`next`), 그리고 이전 노드를 가리키는 포인터(`prev`)를 가집니다. 이를 통해 양방향 탐색이 가능해집니다.
    ```c
    struct doubly_item {
        char info;
        struct doubly_item *prev;
        struct doubly_item *next;
    };
    ```

### 코드/수식 해설

단일 연결 리스트를 이중 연결 리스트로 변경하는 과정은 크게 두 단계로 나눌 수 있습니다:

1.  **노드 구조 변경**: 기존 `struct item`에 `prev` 포인터를 추가하여 `struct doubly_item`과 같은 형태로 만듭니다. (실제 구현에서는 새로운 `doubly_item` 타입으로 노드를 새로 만들고 데이터를 복사하는 방식이 더 일반적일 수 있으나, 문제의 의도상 기존 노드 자체를 수정하는 것을 가정합니다.)
2.  **`prev` 포인터 설정**: 리스트를 순회하면서 각 노드의 `prev` 포인터를 이전 노드를 가리키도록 설정합니다.

아래는 단일 연결 리스트를 이중 연결 리스트로 변환하는 C 코드 예시입니다. (기존 `item` 구조체에 `prev` 필드를 추가했다고 가정합니다. 실제로는 `struct item`을 `struct doubly_item`으로 명시적으로 변경해야 합니다.)

```c
// 기존 단일 연결 리스트 노드 (prev 필드를 추가할 수 있도록 준비)
typedef struct item {
    char info;
    struct item *prev; // 새로 추가될 필드
    struct item *next; // 기존 link 필드가 next로 이름 변경되었다고 가정
} item;

// 단일 연결 리스트를 이중 연결 리스트로 변환하는 함수
void convertSinglyToDoubly(item *head) {
    if (head == NULL) {
        return; // 빈 리스트는 변환할 것이 없음
    }

    item *current = head;
    item *previous = NULL; // 첫 노드의 prev는 NULL

    while (current != NULL) {
        current->prev = previous; // 현재 노드의 prev 포인터를 이전 노드로 설정
        previous = current;       // 현재 노드를 다음 순회의 이전 노드로 설정
        current = current->next;  // 다음 노드로 이동
    }
}
```

**수식 해설**: 포인터 연산은 대부분 대입(`=`)과 역참조(`->`)로 이루어집니다.
$P_{current} \rightarrow prev = P_{previous}$
$P_{previous} = P_{current}$
$P_{current} = P_{current} \rightarrow next$
위 수식은 `convertSinglyToDoubly` 함수 내부의 핵심 로직을 표현한 것으로, 각 노드의 `prev` 포인터가 올바른 이전 노드를 가리키도록 설정하는 과정을 나타냅니다.

### 구체적 예시

주어진 슬라이드의 단일 연결 리스트 `L -> a -> b -> c -> d`를 예시로 들어보겠습니다.
초기 상태:
- `a`의 `next`는 `b`
- `b`의 `next`는 `c`
- `c`의 `next`는 `d`
- `d`의 `next`는 `NULL`
- 모든 노드의 `prev`는 초기화되지 않았거나 `NULL` (가비지 값)

변환 과정:
1.  `head`는 `a`를 가리킵니다. `current = a`, `previous = NULL`.
2.  **노드 `a`**:
    *   `a->prev = previous` (즉, `a->prev = NULL`)
    *   `previous = a`
    *   `current = a->next` (즉, `current = b`)
3.  **노드 `b`**:
    *   `b->prev = previous` (즉, `b->prev = a`)
    *   `previous = b`
    *   `current = b->next` (즉, `current = c`)
4.  **노드 `c`**:
    *   `c->prev = previous` (즉, `c->prev = b`)
    *   `previous = c`
    *   `current = c->next` (즉, `current = d`)
5.  **노드 `d`**:
    *   `d->prev = previous` (즉, `d->prev = c`)
    *   `previous = d`
    *   `current = d->next` (즉, `current = NULL`)
6.  `current`가 `NULL`이므로 루프 종료.

변환 후 결과:
- `a->prev = NULL`, `a->next = b`
- `b->prev = a`, `b->next = c`
- `c->prev = b`, `c->next = d`
- `d->prev = c`, `d->next = NULL`

이제 리스트는 완벽한 이중 연결 리스트가 됩니다.

### 시험 포인트

*   ⭐ **단일 연결 리스트와 이중 연결 리스트의 구조적 차이**: `prev` 포인터의 유무. (⭐ 정의 및 그림 그리기)
*   ⭐ **변환 알고리즘**:
    *   노드 구조에 `prev` 포인터 필드를 추가해야 함.
    *   리스트를 한 번 순회(traverse)하면서 `prev` 포인터를 설정해야 함.
    *   첫 번째 노드의 `prev`는 `NULL`이라는 점을 명심.
*   ⭐ **시간 복잡도**: 리스트의 각 노드를 한 번씩만 방문하므로 $O(N)$ (N은 리스트의 노드 수)입니다.
*   ⭐ **공간 복잡도**: 기존 노드에 `prev` 포인터 필드를 추가하는 것이라면 추가적인 공간은 $O(1)$ (노드마다 포인터 하나씩 추가). 만약 새로운 노드를 생성하여 복사하는 방식이라면 $O(N)$의 공간이 필요합니다. 문제의 의도는 기존 노드를 `change`하는 것에 가깝습니다.

**핵심 개념**:
*   **포인터 기반 리스트 구현의 논리적 경계**: 연결 리스트에서 삽입 또는 삭제가 일어날 '현재 위치 (current position)'를 어떻게 포인터로 표현할 것인가에 대한 문제입니다. 슬라이드에서는 `b`와 `c` 사이의 빨간 막대가 논리적 경계를 나타내고, `c`가 '현재 위치'로 지정되어 있습니다.
*   **Option-1: `p`가 '현재 요소'를 직접 가리키는 방식**: 포인터 `p`가 삽입/삭제 작업의 대상이 되는 *요소 자체* (`c` 노드)를 직접 가리키는 방법을 제시합니다.
*   **삽입(Insert) 연산의 비효율성**: `p`가 현재 요소를 가리킬 때, `p`가 가리키는 위치 *앞*에 새로운 요소를 삽입하는 `Insert(x, p, L)` 연산에서 발생하는 문제점을 지적합니다.

**코드/수식 해설**:
이 슬라이드에는 직접적인 코드가 없지만, `Insert(x, p, L)` 연산의 개념적인 어려움을 설명할 수 있습니다.
일반적인 단일 연결 리스트에서 노드는 `data`와 다음 노드를 가리키는 `next` 포인터로 구성됩니다:
```c
struct Node {
    DataType data;
    struct Node* next;
};
```
`p`가 현재 노드 `c`를 가리키고 있을 때 (즉, `p`는 `c`의 주소를 가지고 있음), 새 데이터 `x`를 가진 노드 `newNode`를 `b`와 `c` 사이에 삽입하려면 다음 두 단계를 거쳐야 합니다:
1.  `newNode->next = p;` // 새 노드의 다음 포인터가 현재 노드 `c`를 가리키게 합니다.
2.  `b->next = newNode;` // 이전 노드 `b`의 다음 포인터가 새 노드 `newNode`를 가리키게 합니다.

문제는 `p`가 `c`를 가리키는 상황에서는 이전 노드 `b`에 직접 접근할 방법이 없다는 것입니다. 단일 연결 리스트는 역방향 탐색을 지원하지 않으므로, `b`를 찾기 위해서는 리스트의 시작 (`L`)부터 다시 순차적으로 탐색해야 하는 비효율이 발생합니다.

**구체적 예시**:
음악 플레이리스트에서 현재 재생 중인 곡 `C`를 포인터 `p`로 가리키고 있다고 가정해 봅시다. 이제 곡 `C` 바로 앞에 새로운 곡 `X`를 추가하고 싶습니다.
*   `p`는 곡 `C`만 알고 있습니다.
*   곡 `X`를 `C` 앞에 넣으려면, `C` 바로 앞의 곡 `B`의 '다음 곡' 설정을 `X`로 변경해야 합니다. 그리고 `X`의 '다음 곡' 설정을 `C`로 변경해야 합니다.
*   하지만 `p`가 `C`만 가리키고 있으므로, 곡 `B`가 무엇이었는지 `p`만으로는 알 수 없습니다. 플레이리스트의 처음부터 다시 탐색하거나, `p`가 `C`가 아닌 `B`를 가리키도록 설계했어야 이 문제가 해결됩니다.
이처럼 '현재 요소'만을 가리키는 포인터는 '이전 요소'에 대한 접근을 어렵게 만들어, 해당 위치 앞에 삽입하는 연산을 복잡하게 만듭니다.

**시험 포인트**:
*   ⭐ **단일 연결 리스트에서 포인터가 '현재 요소'를 가리킬 때 `Insert` 연산의 문제점을 설명하시오.** (정답: 이전 노드에 접근하기 어렵기 때문에, 이전 노드의 `next` 포인터를 변경하는 것이 비효율적이다. 리스트 시작부터 다시 탐색해야 할 수 있다.)
*   ⭐ **`p`가 `c`를 가리킬 때 `Insert(x, p, L)` 연산 시 링크 변경에 필요한 정보 (이전 노드 `b`의 주소)를 어떻게 얻어야 하는가?** (정답: `L`부터 `p`의 이전 노드까지 탐색하거나, `p`가 현재 노드가 아닌 이전 노드를 가리키도록 구현해야 한다.)
*   ⭐ **논리적 경계를 표현하는 방식이 자료구조 연산의 시간 복잡도에 미치는 영향**을 이해하는 것이 중요합니다. 이 문제를 해결하기 위한 다른 포인터 기반 구현 방식을 이해하는 것이 후속 학습의 핵심이 될 것입니다.

## Slide 16

**핵심 개념**:
이 슬라이드는 연결 리스트(Linked List) 구현에서 '논리적 울타리(logical fence)' 또는 '현재 위치'를 표현하는 한 가지 방법인 **"One-step ahead convention"**을 소개합니다. 이 방식은 포인터 `p`가 실제 '현재' 작업 대상 요소가 아닌, '현재 위치의 바로 이전(previous)' 요소를 가리키도록 정의하는 것입니다. 즉, 포인터 `p`는 삽입/삭제가 일어날 지점의 직전 노드를 참조하여, 다음 노드에 대한 접근을 용이하게 합니다.

**구체적 예시**:
슬라이드의 그림에서 `L`은 리스트의 시작을 나타내고, `a`, `b`, `c`, `d`는 리스트의 요소들입니다.
*   `p`는 요소 `b`를 가리키고 있습니다.
*   "previous position"이라는 빨간 선은 `b`와 `c` 사이에 존재하며, 이 지점이 논리적인 '현재 위치'를 나타냅니다.
*   "One-step ahead convention"에 따르면, `p`가 `b`를 가리키므로 `b`가 '현재 위치'의 이전 요소가 됩니다. 따라서 `p`가 가리키는 것은 요소 `b`이며, '현재 위치'는 `b` *다음*인 `b`와 `c` 사이 지점입니다.

이 컨벤션의 장점은 다음과 같습니다.
*   **삽입**: `p`가 `b`를 가리키고 있을 때, `b`와 `c` 사이에 새로운 요소 `X`를 삽입하려면, `p->next`를 `X`로 변경하고 `X->next`를 `c`로 변경하면 됩니다. `p`만으로 쉽게 접근할 수 있습니다.
    ```c
    Node* newNode = createNode(X);
    newNode->next = p->next; // X의 다음은 p가 원래 가리키던 다음 (c)
    p->next = newNode;      // p의 다음은 X
    ```
*   **삭제**: `p`가 `b`를 가리키고 있을 때, `c`를 삭제하려면, `p->next`를 `c->next`로 변경하여 `c`를 건너뛰면 됩니다.
    ```c
    Node* nodeToDelete = p->next; // 삭제할 노드는 c
    p->next = nodeToDelete->next; // p의 다음을 c의 다음 (d)로 연결
    free(nodeToDelete);           // c 노드 메모리 해제
    ```

**시험 포인트**:
*   ⭐ **"One-step ahead convention"의 정의와 역할**: 포인터가 가리키는 대상이 무엇인지 (현재 위치의 '이전' 요소) 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **이 컨벤션이 리스트 연산 (삽입, 삭제)에 미치는 영향**: 왜 이 방식이 삽입 및 삭제 연산을 단순화하고 효율적으로 만드는지 (예: 별도의 이전 노드 탐색 없이 `p`만으로 작업 가능) 설명할 수 있어야 합니다.
*   ⭐ **그림으로 제시된 상황에서 `p`의 의미**: 그림과 같이 `p`가 특정 노드를 가리킬 때, 이 `p`가 나타내는 '현재 위치'가 어디인지 (예: `b`와 `c` 사이의 논리적 울타리) 정확히 파악할 수 있어야 합니다.

## Pointer-Based List Implementation (4) - 맨 앞 삽입 고려 사항

### 핵심 개념
이 슬라이드는 포인터 기반 연결 리스트에서 요소를 삽입하는 `Insert(x, p, L)` 연산 중, 특히 리스트의 **맨 앞에 새로운 요소를 삽입하는 경우**에 대한 고려 사항을 다룹니다. 여기서 `x`는 삽입할 데이터, `p`는 새로 생성되어 삽입될 노드 (슬라이드 그림에서), `L`은 리스트의 헤드 포인터 또는 삽입 위치를 가리키는 추상적인 포인터입니다. 핵심은 다음과 같습니다.
1.  **빈 리스트에 삽입**: 리스트가 비어있을 때 새로운 요소를 삽입하면, 그 요소가 리스트의 유일한 노드가 되며 헤드 포인터가 이 새 노드를 가리키게 됩니다.
2.  **리스트의 맨 앞에 삽입**: 리스트에 이미 요소들이 있을 때, 첫 번째 요소 바로 앞에 새로운 요소를 삽입하는 경우입니다. 이 경우에도 리스트의 헤드 포인터가 새로운 노드를 가리키도록 업데이트되어야 합니다.
슬라이드의 질문 "When the list is empty or the left partition is empty, what happens?"는 이 두 가지 시나리오를 포괄적으로 묻고 있으며, 그림은 기존 리스트 `a -> b -> c -> d`의 맨 앞에 `p`를 삽입하는 상황을 보여줍니다.

### 코드/수식 해설
연결 리스트에 맨 앞 삽입을 구현하는 표준 C 코드는 다음과 같습니다. `headRef`는 리스트의 헤드 포인터(`struct Node*`)를 변경하기 위해 해당 포인터의 주소(`struct Node**`)를 전달받습니다.

```c
struct Node {
    int data;
    struct Node* next;
};

// InsertAtHead 함수는 연결 리스트의 맨 앞에 새로운 노드를 삽입합니다.
// headRef: 리스트 헤드 포인터의 주소 (포인터 변경을 위해)
// newData: 삽입할 데이터
void InsertAtHead(struct Node** headRef, int newData) {
    // 1. 새 노드를 생성하고 데이터 할당
    struct Node* newNode = (struct Node*)malloc(sizeof(struct Node));
    if (newNode == NULL) {
        // 메모리 할당 실패 시 처리
        fprintf(stderr, "Memory allocation failed!\n");
        return;
    }
    newNode->data = newData;

    // 2. 새 노드의 next 포인터가 현재 리스트의 헤드를 가리키도록 설정
    //    만약 리스트가 비어있다면 (*headRef가 NULL), newNode->next도 NULL이 됩니다.
    newNode->next = *headRef;

    // 3. 리스트의 헤드 포인터가 새 노드를 가리키도록 업데이트
    //    이제 newNode가 새로운 리스트의 시작점이 됩니다.
    *headRef = newNode;
}
```

이 연산의 주요 단계는 포인터 재연결입니다.
-   새 노드 `N` 생성 및 데이터 `x` 할당: `N.data = x`
-   새 노드의 `next` 포인터를 기존 헤드 `H`가 가리키던 곳으로 설정: `N.next = H`
-   리스트의 헤드 포인터를 새 노드 `N`으로 업데이트: `H = N`

### 구체적 예시
도서관에서 책을 정렬하여 꽂아둔 서가에 비유해 봅시다.
1.  **빈 서가에 첫 책 꽂기 (빈 리스트)**: 서가에 아무 책도 없습니다 (`L`이 `NULL`). "데이터구조" 책(`x`)이 도착해서 꽂아야 합니다. 새 칸(`p`)에 "데이터구조" 책을 넣고, 이 칸이 서가의 첫 칸(`L`)이 됩니다. 이 칸 뒤에는 더 이상 책이 없으므로 `p`의 다음은 `NULL`입니다.
2.  **가장 앞에 새 책 꽂기 (맨 앞 삽입)**: 서가에는 이미 "알고리즘" (`a`), "운영체제" (`b`), "네트워크" (`c`) 책이 순서대로 꽂혀 있습니다. 그런데 "데이터구조" 책(`x`)이 도착해서 가장 앞에 꽂아야 합니다. 새 칸(`p`)에 "데이터구조" 책을 넣고, `p`의 다음 칸은 기존의 첫 번째 책인 "알고리즘" 칸을 가리키게 합니다. 그리고 이제 서가의 시작점(`L`)은 `p`를 가리키게 됩니다. 결과적으로 "데이터구조" -> "알고리즘" -> "운영체제" -> "네트워크" 순서가 됩니다.

슬라이드의 그림은 `L`이 현재 `a` 노드를 가리키는 위치(또는 그 앞)에 있고, `p`라는 새 노드를 `a` 앞에 삽입하는 두 번째 시나리오를 시각적으로 보여줍니다.

### 시험 포인트
-   ⭐ **맨 앞 삽입 로직 이해 및 구현**: 연결 리스트의 헤드에 요소를 삽입할 때, `newNode->next`를 현재 헤드로 설정하고, 헤드 포인터 자체를 `newNode`로 업데이트하는 2단계 과정을 정확히 이해하고 코드로 작성할 수 있어야 합니다.
-   ⭐ **포인터-투-포인터 (`struct Node**`) 사용의 중요성**: `InsertAtHead` 함수처럼 리스트의 헤드 포인터를 변경해야 할 때, 함수의 인자로 `struct Node* head` 대신 `struct Node** headRef`를 사용하는 이유와 그 동작 방식을 정확히 설명할 수 있어야 합니다. `struct Node* head`를 사용하면 함수 내에서만 포인터가 변경되고, 호출자의 실제 헤드 포인터는 변경되지 않습니다.
-   ⭐ **엣지 케이스 처리**: 빈 리스트에 삽입하는 경우 (`*headRef`가 `NULL`일 때)와 비어있지 않은 리스트에 삽입하는 경우의 코드가 동일하게 작동하는 원리를 파악해야 합니다. 위에 제시된 코드는 이 두 경우를 모두 자연스럽게 처리합니다.
-   **시간 복잡도**: 연결 리스트의 맨 앞 삽입 연산은 새로운 노드를 생성하고 몇 개의 포인터만 변경하면 되므로, 항상 상수 시간인 ⭐ **O(1)**의 시간 복잡도를 가집니다.

## Slide 18

**핵심 개념**:
*   포인터 기반 연결 리스트에서 `Insert(x, p, L)` 연산은 특정 위치 `p` (대개 새 노드 `x`가 삽입될 위치의 이전 노드) 뒤에 새 노드를 삽입하는 방식으로 구현됩니다.
*   이 슬라이드는 `Insert` 연산을 구현할 때 발생하는 **두 가지 특수 케이스**에 초점을 맞춥니다:
    1.  **리스트가 비어있는 경우**: 리스트에 아무런 노드도 없을 때 첫 번째 노드를 삽입하는 경우입니다. 이 경우 `p`가 가리킬 이전 노드가 없으며, 리스트의 `head` 포인터 자체가 새 노드를 가리키도록 변경되어야 합니다.
    2.  **리스트의 가장 앞에 삽입하는 경우**: 리스트에 노드들이 존재하지만, 새 노드를 기존의 첫 번째 노드보다 앞에 삽입해야 하는 경우입니다. 이 역시 `p`가 가리킬 이전 노드가 없으며, 리스트의 `head` 포인터가 새 노드를 가리키도록 변경되어야 합니다.
*   이러한 특수 케이스들은 일반적인 삽입 로직(예: `p->next = newNode; newNode->next = temp;` 등)만으로는 처리할 수 없으며, `if` 문과 같은 조건문을 사용하여 명시적으로 `head` 포인터를 직접 업데이트하는 특별한 로직을 적용해야 합니다.

**코드/수식 해설**:
슬라이드에는 직접적인 코드나 수식은 없지만, 특수 케이스 처리를 포함한 단일 연결 리스트의 `Insert` 함수 (C언어 기준)는 다음과 같이 작성될 수 있습니다. 여기서 `p`는 `x`를 삽입할 위치의 "이전" 노드를 가리킨다고 가정합니다.

```c
#include <stdlib.h> // for malloc

// 노드 구조체 정의
typedef struct Node {
    int data;
    struct Node* next;
} Node;

/**
 * @brief 연결 리스트에 새로운 노드를 삽입합니다.
 * @param head 리스트의 헤드 포인터에 대한 포인터 (이중 포인터)
 * @param x 삽입할 데이터
 * @param p 새 노드가 삽입될 위치의 이전 노드. NULL이면 리스트의 맨 앞에 삽입.
 */
void insert(Node** head, int x, Node* p) {
    Node* newNode = (Node*)malloc(sizeof(Node));
    if (newNode == NULL) {
        // 메모리 할당 실패 처리
        return;
    }
    newNode->data = x;
    newNode->next = NULL;

    // ⭐ 특수 케이스 1 & 2: 리스트가 비어있거나, p가 NULL인 경우 (맨 앞에 삽입)
    // 이 경우, head 포인터 자체가 변경되어야 합니다.
    if (*head == NULL || p == NULL) {
        newNode->next = *head; // 새 노드의 다음은 현재 head가 가리키던 노드 (NULL이거나 첫 노드)
        *head = newNode;       // head 포인터를 새 노드로 업데이트
    } else {
        // 일반 케이스: p 노드 다음에 삽입
        // newNode를 p와 p->next 사이에 삽입합니다.
        newNode->next = p->next;
        p->next = newNode;
    }
}
```
여기서 `Node** head` (헤드 포인터에 대한 이중 포인터)를 사용하는 것은, `insert` 함수 내에서 리스트의 `head` 포인터 자체의 값을 변경해야 할 때 (예: 리스트의 첫 노드가 바뀌는 경우) 그 변경사항이 호출자에게 반영되도록 하기 위함입니다. 만약 `Node* head`로 전달하면, 함수 내부에서 `head = newNode;`로 변경해도 원본 `head` 포인터는 바뀌지 않습니다.

**구체적 예시**:

어떤 물건들을 순서대로 진열하는 선반을 예로 들어봅시다.

1.  **일반적인 삽입**: "사과"와 "바나나" 사이에 "오렌지"를 진열하고 싶습니다. 이때 "사과" (노드 `p`) 뒤에 "오렌지"를 놓고, "오렌지" 뒤에 "바나나"를 놓으면 됩니다.
2.  **리스트가 비어있는 경우**: 선반이 완전히 비어있습니다. 첫 물건 "책"을 진열합니다. 이때 "책"은 선반의 시작점(head)이자 유일한 물건이 됩니다. 이 상황에서는 `p`가 존재하지 않으므로 `head`를 "책"으로 설정합니다.
3.  **가장 앞에 삽입**: 선반에 이미 "책", "연필", "지우개"가 진열되어 있습니다. "가방"을 "책"보다 앞에 진열하고 싶습니다. 이 경우, "가방"이 선반의 새로운 시작점(head)이 되고, 기존의 "책"은 "가방"의 다음 물건이 됩니다. 여기서도 `p`가 존재하지 않으므로, `head`를 "가방"으로 변경하는 특별한 처리가 필요합니다.

슬라이드 그림은 세 번째 예시(가장 앞에 삽입)를 보여주고 있습니다. `L`은 리스트의 시작을 가리키는데, `a` 앞에 빨간 막대(`x`를 삽입할 위치)가 있습니다. 이 경우 삽입될 노드의 이전 노드 `p`는 사실상 `L` (리스트의 헤드) 자체를 의미하게 되어, `head` 포인터의 변경이 필요합니다.

**시험 포인트**:

*   ⭐ **연결 리스트 삽입 연산의 엣지 케이스**: 빈 리스트에 첫 요소를 삽입하는 경우와 리스트의 가장 앞에 요소를 삽입하는 경우를 명확히 이해하고, 이들이 일반적인 중간 삽입 로직과 다른 이유를 설명할 수 있어야 합니다. (⭐**핵심 중의 핵심!**)
*   ⭐ **`if` 문을 사용한 특수 케이스 처리의 필요성**: 왜 `if` 문을 사용하여 `head` 포인터 변경과 같은 특별한 로직을 적용해야 하는지 논리적으로 설명할 수 있어야 합니다. 특히 `head` 포인터가 변경될 때 `Node** head` (이중 포인터)를 사용하는 이유를 숙지해야 합니다.
*   `Insert` 함수의 C/C++ 코드 (또는 유사한 의사 코드)를 작성하거나, 주어진 코드에서 특수 케이스 처리 로직의 오류를 찾아내고 수정하는 문제가 출제될 수 있습니다.
*   이러한 특수 케이스 처리를 간소화하기 위한 다른 자료구조 디자인 (예: 더미 헤드 노드 사용)과 그 장단점에 대해서도 질문할 수 있습니다. (이 슬라이드에서는 직접 다루지 않았지만, 관련 개념으로 중요합니다.)

### Pointer-Based List Implementation (4) - 연결 리스트 삽입 연산의 엣지 케이스 처리

---

*   **핵심 개념**
    *   **연결 리스트 `Insert(x, p, L)` 연산의 엣지 케이스**: 연결 리스트에서 특정 위치 `p`에 값 `x`를 삽입하는 `Insert` 연산 시, 리스트의 맨 앞(head)에 삽입하거나 리스트가 비어있을 때 발생하는 특수한 상황(edge case)을 다룹니다. 특히, `p`가 삽입할 위치의 직전 노드를 가리키는 포인터라고 가정할 때, `p`가 `NULL`이거나 리스트의 헤드 노드 이전에 삽입해야 하는 경우 기존 `L` (리스트의 헤드 포인터) 자체가 변경되어야 하므로 일반적인 삽입 로직과 다른 처리가 필요합니다.
    *   **문제점**: 일반적인 `insert_after(p)`와 같은 함수는 `p`가 유효한 노드를 가리키고 있을 때만 잘 작동합니다. `p`가 `NULL`이거나 리스트의 첫 번째 요소 앞에 삽입하는 경우, 리스트의 헤드를 가리키는 포인터 `L`을 직접 수정해야 하므로 코드의 복잡성이 증가합니다.
    *   **해결책 1: 조건문(`if` statement)을 사용한 특수 케이스 처리**: `p`가 특정 조건(예: `NULL` 또는 헤드 삽입을 의미하는 특별한 값)을 만족하는지 확인하고, 그에 따라 `L`을 직접 업데이트하는 코드를 작성합니다.
    *   **해결책 2: 더미 헤더 노드(dummy header node) 사용**: 실제 데이터가 없는 더미 노드를 리스트의 맨 앞에 고정적으로 두어, `L`이 항상 이 더미 노드를 가리키도록 합니다. 이렇게 하면 리스트의 '맨 앞' 삽입도 더미 노드 *뒤*에 삽입하는 일반적인 경우로 취급할 수 있어 삽입 로직을 통일시킬 수 있습니다.

*   **코드/수식 해설**

    **1. 조건문 (`if`)을 통한 특수 케이스 처리**

    이 방식은 `p`가 `NULL`일 때(리스트의 맨 앞에 삽입) `L` 포인터를 직접 업데이트하고, 그렇지 않은 경우 `p` 뒤에 삽입하는 일반적인 로직을 따릅니다.

    ```c
    // L은 리스트의 헤드 포인터이며, Insert 함수에서 변경될 수 있으므로 Node** L로 받습니다.
    // p는 삽입할 위치의 직전 노드 포인터입니다. p가 NULL이면 리스트의 맨 앞에 삽입합니다.
    Node* insert(int x, Node* p, Node** L_ptr) {
        Node* new_node = (Node*)malloc(sizeof(Node));
        new_node->data = x;

        if (p == NULL) { // 리스트의 맨 앞에 삽입하는 경우 (새로운 헤드가 됨)
            new_node->next = *L_ptr; // 새 노드의 next는 현재 헤드를 가리키고
            *L_ptr = new_node;      // L_ptr이 새 노드를 가리키도록 업데이트
        } else { // p 노드 뒤에 삽입하는 일반적인 경우
            new_node->next = p->next; // 새 노드의 next는 p의 다음 노드를 가리키고
            p->next = new_node;      // p의 next는 새 노드를 가리키도록 업데이트
        }
        return new_node;
    }
    ```

    **2. 더미 헤더 노드 (`dummy header node`) 사용**

    리스트가 항상 비어있지 않은 것처럼 보이게 하기 위해, 실제 데이터는 포함하지 않는 가상의 노드를 리스트의 맨 앞에 둡니다. `L`은 항상 이 더미 노드를 가리킵니다.

    ```c
    // L은 더미 헤더 노드를 가리키는 포인터입니다.
    // p는 삽입할 위치의 직전 노드 포인터입니다. 더미 노드를 사용하면 p가 NULL일 일이 없습니다.
    Node* insert_with_dummy(int x, Node* p) {
        Node* new_node = (Node*)malloc(sizeof(Node));
        new_node->data = x;

        // 더미 노드를 사용하면 맨 앞 삽입도 동일한 로직으로 처리됩니다.
        // p가 더미 노드일 경우, 더미 노드 뒤에 새 노드를 삽입합니다.
        new_node->next = p->next; // 새 노드의 next는 p의 다음 노드를 가리키고
        p->next = new_node;      // p의 next는 새 노드를 가리키도록 업데이트
        return new_node;
    }
    ```

*   **구체적 예시**

    **예시 1: 조건문 처리 방식**
    *   초기 리스트: `L -> a -> b -> c -> NULL`
    *   **Case 1: `d`를 `a` 앞에 삽입**: `insert(d, NULL, &L)` 호출. `p`가 `NULL`이므로, `d->next = L` (`d->next = a`), `L = d`가 되어 `L -> d -> a -> b -> c -> NULL`이 됩니다.
    *   **Case 2: `e`를 `b` 뒤에 삽입**: `insert(e, b, &L)` 호출. `p`가 `b`이므로, `e->next = b->next` (`e->next = c`), `b->next = e`가 되어 `L -> a -> b -> e -> c -> NULL`이 됩니다.
    *   보시다시피, 헤드 삽입 시에는 `L`을 직접 변경하는 특수 로직이 필요합니다.

    **예시 2: 더미 헤더 노드 방식**
    *   초기 리스트: `L -> (dummy) -> a -> b -> c -> NULL` (여기서 `L`은 더미 노드를 가리킵니다)
    *   **Case 1: `d`를 `a` 앞에 삽입**: `p`를 더미 노드로 설정하여 `insert_with_dummy(d, dummy_node)` 호출. `d->next = dummy_node->next` (`d->next = a`), `dummy_node->next = d`가 되어 `L -> (dummy) -> d -> a -> b -> c -> NULL`이 됩니다.
    *   **Case 2: `e`를 `b` 뒤에 삽입**: `p`를 `b` 노드로 설정하여 `insert_with_dummy(e, b)` 호출. `e->next = b->next` (`e->next = c`), `b->next = e`가 되어 `L -> (dummy) -> a -> b -> e -> c -> NULL`이 됩니다.
    *   이 방식은 모든 삽입(맨 앞, 중간, 맨 뒤)에 대해 동일한 `new_node->next = p->next; p->next = new_node;` 로직을 적용할 수 있어 코드의 일관성과 간결성을 높입니다.

*   **시험 포인트**
    *   ⭐연결 리스트 `Insert` 연산 시 리스트의 맨 앞이나 빈 리스트에 삽입하는 경우에 발생하는 엣지 케이스를 이해하고 설명할 수 있어야 합니다.
    *   ⭐이러한 엣지 케이스를 처리하는 두 가지 주요 방법(조건문 사용, 더미 헤더 노드 사용)의 개념과 구현 아이디어를 비교하여 설명할 수 있어야 합니다.
    *   ⭐특히 더미 헤더 노드의 장점(코드의 간결성, 일반화된 삽입 로직 제공)을 정확히 인지하고 있어야 합니다. 이는 자료구조 구현 시 발생할 수 있는 코너 케이스를 우아하게 처리하는 중요한 기법입니다.

## Slide 20

### 핵심 개념
이 슬라이드는 단일 연결 리스트(Singly Linked List)에서 특정 위치에 노드를 삽입하는 연산의 시간 복잡도를 묻는 퀴즈입니다. 여기서 '특정 위치'는 삽입될 새 노드 바로 앞 노드를 가리키는 포인터 `p`로 주어집니다.

### 코드/수식 해설
`Insert(x, p, L)` 연산은 주어진 데이터 `x`를 새로운 노드로 만들고, 이 새 노드를 포인터 `p`가 가리키는 노드 *뒤*에 삽입하는 것을 의미합니다. 슬라이드의 그림에서 `p`는 `b` 노드를 가리키고 있으며, 삽입은 `b`와 `c` 노드 사이에 이루어집니다.

이 연산을 수행하기 위한 주요 단계는 다음과 같습니다:

1.  데이터 `x`를 저장할 새 노드(`newNode`)를 생성합니다. 이 과정은 메모리 할당을 포함하며 상수 시간($O(1)$)이 소요됩니다.
2.  `newNode`의 다음 포인터(`newNode->next`)를 현재 `p`가 가리키는 노드의 다음 포인터(`p->next`)가 가리키는 노드(즉, 그림의 `c` 노드)로 연결합니다.
3.  `p`가 가리키는 노드의 다음 포인터(`p->next`)를 `newNode`로 변경합니다.

이 과정을 C/C++ pseudo-code로 표현하면 다음과 같습니다:

```c
// 연결 리스트 노드 구조체 (예시)
struct Node {
    int data;         // 노드의 데이터
    struct Node* next; // 다음 노드를 가리키는 포인터
};

// Insert 함수: p가 가리키는 노드 뒤에 x를 삽입
// p: 삽입될 노드의 이전 노드를 가리키는 포인터
// x: 삽입할 데이터
void Insert(int x, struct Node* p) {
    // 1. 새로운 노드 생성 및 데이터 초기화
    struct Node* newNode = (struct Node*)malloc(sizeof(struct Node));
    newNode->data = x;

    // 2. 새 노드의 next를 p->next가 가리키는 곳으로 설정
    //    (기존에 p->next가 가리키던 노드를 새 노드 뒤에 연결)
    newNode->next = p->next;

    // 3. p->next를 새 노드로 변경
    //    (p가 가리키던 노드와 새 노드를 연결)
    p->next = newNode;
}
```
이러한 연산은 메모리 할당과 포인터 조정이라는 고정된 수의 단계로 구성됩니다. 따라서 리스트의 전체 크기 `N`에 관계없이 항상 일정한 시간이 소요됩니다.

따라서 `Insert(x, p, L)` 연산의 시간 복잡도는 $O(1)$입니다.

### 구체적 예시
슬라이드의 그림을 따라가며 삽입 과정을 살펴봅시다.

**초기 연결 리스트 상태:**
`L -> [a] -> [b] -> [c] -> [d] -> NULL`
이때 `p` 포인터는 `b` 노드를 가리키고 있습니다. 삽입할 데이터는 `x`입니다.

**삽입 과정:**

1.  **새 노드 생성**: `[x]` 데이터를 가진 `newNode`를 생성합니다.
2.  **`newNode`의 다음 연결**: `newNode->next`를 `p->next`가 현재 가리키는 노드, 즉 `c` 노드로 설정합니다.
    *   상태: `L -> [a] -> [b] -> [c] -> [d] -> NULL`
    *   `newNode`: `[x] -> [c]`
3.  **`p`의 다음 연결 변경**: `p->next`를 `newNode`로 변경합니다.
    *   `p`는 `b` 노드를 가리키므로, `b` 노드의 `next` 포인터가 이제 `newNode`를 가리키게 됩니다.

**최종 연결 리스트 상태:**
`L -> [a] -> [b] -> [x] -> [c] -> [d] -> NULL`

이 예시에서 볼 수 있듯이, 삽입 연산을 위해 리스트를 처음부터 끝까지 순회할 필요 없이, `p`가 가리키는 노드와 `newNode` 간의 포인터만 재설정하면 됩니다.

### 시험 포인트
*   ⭐ **단일 연결 리스트에서 특정 노드(`p`) *다음에* 새 노드를 삽입하는 연산의 시간 복잡도는 $O(1)$입니다.** 이는 삽입할 위치의 이전 노드에 대한 포인터(`p`)가 이미 주어졌을 때만 해당됩니다.
*   만약 삽입할 위치를 찾기 위해 리스트를 순회해야 한다면 (예: `k`번째 위치에 삽입하거나 특정 값을 가진 노드 뒤에 삽입), 이 탐색 시간이 전체 복잡도에 포함되어 최악의 경우 $O(N)$이 될 수 있습니다.
*   배열과 연결 리스트의 시간 복잡도 차이를 비교하여 이해하는 것이 중요합니다. 배열에서는 중간 삽입 시 뒤의 모든 요소를 이동시켜야 하므로 $O(N)$의 시간이 소요되지만, 연결 리스트는 포인터 조작만으로 $O(1)$에 가능합니다 (단, 삽입 지점을 알고 있는 경우).

## Slide 21

**핵심 개념**

*   **커서(Cursor) 기반 리스트**: 일반적인 연결 리스트가 메모리 주소를 직접 가리키는 포인터를 사용하는 것과 달리, 커서 기반 리스트는 **배열의 정수 인덱스**를 사용하여 다음 노드의 위치를 나타내는 방식으로 포인터 기능을 시뮬레이션합니다. 이는 포인터 개념이 없거나 사용이 제한된 언어 환경에서 연결 리스트를 구현할 때 유용합니다.
*   **리스트 헤드 관리**: 각 리스트의 시작 노드(첫 번째 요소)를 가리키는 배열 인덱스를 별도로 저장하여 리스트에 접근합니다 (예: `Head of L1`, `Head of L2`).
*   **링크(Link) 필드**: 각 노드는 데이터(`info`)와 다음 노드의 배열 인덱스를 저장하는 `link` 필드를 가집니다.
*   **리스트의 끝 표시**: `link` 필드 값이 0(또는 미리 정의된 특정 상수)이면 해당 노드가 리스트의 마지막 노드, 즉 '꼬리(tail)'임을 의미합니다.
*   **시간 복잡도**:
    *   **삽입(Insert) 및 삭제(Delete)**: ⭐ 일반적으로 **$O(1)$** 입니다. 이는 삽입/삭제할 위치를 이미 알고 있거나, 리스트의 맨 앞(head)에 노드를 추가/제거할 때, 단순히 `link` 필드 몇 개만 변경하면 되므로 상수 시간이 소요됩니다. 데이터 이동이 필요 없습니다. (특정 값을 찾아 삽입/삭제하는 경우는 탐색 시간 포함 $O(n)$).
    *   **검색(Retrieve)**: ⭐ **$O(n)$** 입니다. 원하는 데이터를 찾기 위해서는 `head`부터 `link` 필드를 따라 순차적으로 탐색해야 하므로, 리스트 길이에 비례하는 시간이 소요됩니다.

**코드/수식 해설**

커서 기반 리스트는 일반적으로 구조체의 배열로 구현됩니다. 슬라이드의 `info`와 `link` 컬럼을 가진 배열은 C언어에서 다음과 같은 형태로 표현될 수 있습니다.

```c
#define MAX_NODES 100 // 최대 노드 수
#define NIL 0         // 리스트의 끝을 나타내는 값 (여기서는 0)

// 리스트의 각 노드를 표현하는 구조체
typedef struct {
    char info;   // 데이터를 저장하는 필드 (예: 'a', 'b', 'c')
    int link;    // 다음 노드의 배열 인덱스를 저장하는 필드
} Node;

// 모든 노드를 저장할 배열
Node list_array[MAX_NODES];

// 각 리스트의 시작 노드 인덱스
int head_L1; // L1의 헤드 인덱스
int head_L2; // L2의 헤드 인덱스

// 추가적으로, 사용 가능한 빈 노드를 관리하는 free list의 헤드 인덱스도 필요할 수 있습니다.
// int available;
```
위 `list_array` 배열의 각 원소는 `info`와 `link`를 가지며, 이들이 논리적으로 연결되어 리스트를 형성합니다.

**구체적 예시**

슬라이드의 예시 `L1 = <a, b, c>`와 `L2 = <x, y>`가 하나의 배열 `list_array` 내에서 어떻게 구현되는지 살펴보겠습니다. 배열의 인덱스는 1부터 시작한다고 가정합니다.

현재 배열 상태:

| Index | info | link |
| :---- | :--- | :--- |
| 1     | c    | 0    |
| 2     | a    | 4    |
| 3     | y    | 0    |
| 4     | b    | 1    |
| 5     |      |      |
| 6     | x    | 3    |
| 7     | ...  | ...  |

그리고 리스트의 헤드는 다음과 같습니다:
`Head of L1 = 2`
`Head of L2 = 6`

**1. 리스트 L1 (`<a, b, c>`) 순회 과정:**
1.  `Head of L1`이 2이므로, `list_array[2]`에서 시작합니다.
    *   `list_array[2].info`는 `'a'`, `list_array[2].link`는 `4`입니다.
2.  `link`가 4이므로, `list_array[4]`로 이동합니다.
    *   `list_array[4].info`는 `'b'`, `list_array[4].link`는 `1`입니다.
3.  `link`가 1이므로, `list_array[1]`로 이동합니다.
    *   `list_array[1].info`는 `'c'`, `list_array[1].link`는 `0`입니다.
4.  `link`가 `0`이므로, 리스트의 끝에 도달했습니다.
결과적으로 `L1`은 `a -> b -> c` 순서로 구성된 것을 알 수 있습니다.

**2. 리스트 L2 (`<x, y>`) 순회 과정:**
1.  `Head of L2`가 6이므로, `list_array[6]`에서 시작합니다.
    *   `list_array[6].info`는 `'x'`, `list_array[6].link`는 `3`입니다.
2.  `link`가 3이므로, `list_array[3]`으로 이동합니다.
    *   `list_array[3].info`는 `'y'`, `list_array[3].link`는 `0`입니다.
3.  `link`가 `0`이므로, 리스트의 끝에 도달했습니다.
결과적으로 `L2`는 `x -> y` 순서로 구성된 것을 알 수 있습니다.

이 예시를 통해, 노드들이 배열 내에 물리적으로 연속적이지 않아도 `link` 필드를 통해 논리적으로 연결될 수 있으며, 하나의 배열로 여러 리스트를 동시에 관리할 수 있음을 이해할 수 있습니다.

**시험 포인트**

*   ⭐ **커서 기반 리스트의 필요성과 장점**: 포인터가 없는 프로그래밍 환경(예: 일부 옛날 BASIC, FORTRAN 등)에서 연결 리스트를 구현하거나, 메모리 할당/해제 오버헤드를 줄이고 캐시 지역성(cache locality)을 개선하여 성능을 높일 때 사용됩니다. 일반적인 연결 리스트와 비교했을 때 메모리 단편화 문제에 더 강할 수 있습니다.
*   ⭐ **`link` 필드의 `0` (또는 `NIL`) 값의 의미**: 리스트의 끝을 명확히 표시하는 마커 역할을 하며, 리스트 순회 시 종료 조건으로 사용됩니다.
*   ⭐ **$O(1)$ 삽입/삭제의 조건**: 노드 자체를 연결하거나 끊는 작업은 상수 시간이지만, 특정 요소를 찾아 그 앞에 삽입/삭제하는 경우에는 해당 요소를 탐색하는 시간($O(n)$)이 추가됨을 인지해야 합니다.
*   ⭐ **일반 연결 리스트와의 비교**:
    *   **공통점**: 논리적으로 순차적인 구조이며, 노드들이 메모리에 연속적으로 배치될 필요가 없습니다.
    *   **차이점**: 커서 기반은 배열 인덱스를, 일반 연결 리스트는 메모리 주소를 직접 사용합니다. 커서 기반은 고정된 크기의 배열을 사용하고 빈 공간 관리가 필요할 수 있습니다 (Free list 등).
*   ⭐ **구체적인 리스트 구조를 주고 특정 원소 탐색 또는 삽입/삭제 과정을 설명하는 문제**: 슬라이드의 예시처럼 주어진 배열과 헤드 인덱스를 바탕으로 특정 리스트의 원소를 순서대로 나열하거나, 특정 인덱스에 노드를 삽입/삭제했을 때 배열의 `info`와 `link` 필드가 어떻게 변하는지 설명할 수 있어야 합니다.

### Stacks

**핵심 개념**:
스택(Stack)은 삽입과 삭제가 한쪽 끝(⭐**Top**⭐)에서만 일어나는 특수한 형태의 리스트입니다. 이러한 특성 때문에 스택은 ⭐**LIFO (Last-In, First-Out)**⭐, 즉 "가장 나중에 들어온 것이 가장 먼저 나간다"는 원리로 동작합니다. 스택은 포인터, 배열, 커서 등 다양한 리스트 구현 방식을 사용하여 구현할 수 있습니다.

**코드/수식 해설**:
스택 $S$는 요소들의 순서 있는 컬렉션으로, 보통 $S = <a_1, a_2, ..., a_n>$ 과 같이 표현됩니다. 여기서 $a_1$은 스택의 가장 위에 있는 요소 (Top)를 나타냅니다.

주요 스택 연산은 다음과 같습니다:

1.  **Push($x, S$)**:
    *   설명: 스택 $S$의 맨 위에 요소 $x$를 추가합니다. $x$가 새로운 Top이 됩니다.
    *   수식: 기존 스택이 $S = <a_1, a_2, ..., a_n>$ 이라면, `Push(x, S)` 후 스택은 $S = <x, a_1, a_2, ..., a_n>$ 가 됩니다.

2.  **Pop($S$)**:
    *   설명: 스택 $S$의 맨 위에 있는 요소 (Top)를 제거하고 반환합니다.
    *   수식: 기존 스택이 $S = <a_1, a_2, ..., a_n>$ 이라면, `Pop(S)` 후 스택은 $S = <a_2, ..., a_n>$ 가 됩니다. 반환되는 값은 $a_1$입니다.

3.  **Top($S$)**:
    *   설명: 스택 $S$의 맨 위에 있는 요소 (Top)를 반환합니다. 이 연산은 스택에서 요소를 제거하지 않습니다.
    *   수식: 기존 스택이 $S = <a_1, a_2, ..., a_n>$ 이라면, `Top(S)`는 $a_1$을 반환합니다.

4.  **MakeNull($S$)**:
    *   설명: 스택 $S$를 비워 빈 스택으로 만듭니다.
    *   수식: `MakeNull(S)` 후 스택은 $S = < >$ 가 됩니다.

5.  **IsEmpty($S$)**:
    *   설명: 스택 $S$가 비어 있는지 확인합니다.
    *   수식: $S = < >$ 이면 `true`를 반환하고, 그렇지 않으면 `false`를 반환합니다.

**구체적 예시**:
식당에서 사용하는 접시 스택을 생각해보세요.
*   **Push**: 깨끗한 접시를 스택 맨 위에 올려놓는 행위입니다. 가장 최근에 올려놓은 접시가 가장 위에 있습니다.
*   **Pop**: 음식을 담기 위해 가장 위에 있는 접시를 가져가는 행위입니다. 가장 최근에 올려놓은 접시가 가장 먼저 사용됩니다.
*   **Top**: 음식을 담기 전에 가장 위에 있는 접시가 어떤 종류인지 확인하는 행위입니다.
*   **LIFO 원리**: 가장 마지막에 쌓은 접시가 가장 먼저 사용됩니다. (Last-In, First-Out)

**시험 포인트**:
*   ⭐**스택의 정의 (삽입/삭제가 한쪽 끝(Top)에서만 발생)**⭐를 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐**LIFO (Last-In, First-Out)**⭐ 원리를 완벽하게 숙지하고 있어야 합니다.
*   `Push`, `Pop`, `Top` 등 주요 스택 연산들의 동작 방식과 스택 상태 변화를 설명할 수 있어야 합니다. (특히, Pop과 Top의 차이점)
*   `IsEmpty` 및 `MakeNull`의 역할도 중요합니다.
*   다양한 자료구조 (배열, 연결 리스트)를 이용한 스택 구현의 가능성을 인지하고 있어야 합니다. (구체적인 구현 코드보다는 개념적인 이해)

### Stack Implementations

**핵심 개념**:
스택(Stack)은 Last-In, First-Out (LIFO) 원칙을 따르는 추상 자료형으로, 주로 `push` (요소 추가), `pop` (요소 제거), `top`/`peek` (최상단 요소 확인) 연산을 지원합니다. 이 슬라이드에서는 스택을 구현하는 두 가지 주요 방법에 대해 다룹니다: 배열 기반 스택과 연결 리스트 기반 스택.

1.  **배열 기반 스택 (Array-based stacks)**:
    *   고정된 크기의 배열을 사용하여 스택을 구현합니다.
    *   스택의 최상단을 가리키는 `TOP` 포인터(또는 인덱스)를 어떻게 관리하느냐에 따라 `push`/`pop` 연산의 효율성이 달라집니다.
    *   **TOP 포인터 이동 방식**:
        *   일반적으로 `TOP` 포인터는 스택에 $k$개의 요소가 있을 때 $k$번째(즉, 다음 빈 공간 또는 최상단 요소)를 가리키며, `push` 시 증가하고 `pop` 시 감소합니다. 이 경우 `push`, `pop`, `top` 연산은 모두 $O(1)$의 시간 복잡도를 가집니다.
        *   만약 `TOP` 포인터가 배열의 고정된 위치(예: `Position 1`)를 가리키고 새로운 요소를 삽입하거나 제거할 때마다 모든 기존 요소를 이동시켜야 한다면, `push`/`pop` 연산은 $O(n)$의 시간 복잡도를 가집니다. 이는 매우 비효율적입니다.
    *   **단일 배열 내 다중 스택 (Multiple stacks in a single array)**: 하나의 배열에 여러 개의 스택을 저장할 수 있습니다. 예를 들어, 배열의 양 끝에서 서로를 향해 스택을 성장시켜 공간을 효율적으로 활용할 수 있습니다.

2.  **연결 리스트 기반 스택 (Linked stacks)**:
    *   노드(데이터와 다음 노드를 가리키는 포인터)로 구성된 연결 리스트를 사용하여 스택을 구현합니다.
    *   `TOP`은 연결 리스트의 헤드(head) 포인터에 해당하며, 항상 최상단 요소를 가리킵니다. `push`는 새로운 노드를 헤드에 추가하고, `pop`은 헤드 노드를 제거하는 방식입니다.
    *   연결 리스트 구현은 동적인 크기 조절이 가능하며, 배열 기반과 마찬가지로 `push`, `pop`, `top` 연산 모두 $O(1)$의 시간 복잡도를 가집니다.

**코드/수식 해설**:
*   **TOP 포인터가 동적으로 이동하는 배열 기반 스택**:
    *   스택에 $k$개의 요소가 있을 때, `TOP` 인덱스는 다음 요소를 삽입할 위치($k$)를 가리키거나, 최상단 요소가 있는 위치($k-1$)를 가리킬 수 있습니다.
    *   `push` 연산:
        ```c
        void push(Stack* s, int item) {
            if (s->top >= MAX_SIZE - 1) { // 스택 가득 참
                // 에러 처리
                return;
            }
            s->array[++s->top] = item;
        }
        ```
    *   `pop` 연산:
        ```c
        int pop(Stack* s) {
            if (s->top == -1) { // 스택 비어있음
                // 에러 처리
                return -1;
            }
            return s->array[s->top--];
        }
        ```
    *   이 경우, `push`와 `pop` 연산은 인덱스 증감 및 값 할당/반환만 수행하므로 시간 복잡도는 $O(1)$입니다.
    *   수식으로 표기하면, 스택에 $k$개의 요소가 있을 때 `push`/`pop` 비용은 다음과 같습니다.
        $$ Cost(push) = O(1) $$
        $$ Cost(pop) = O(1) $$

*   **TOP 포인터가 고정된 (Fixed) Position 1 방식의 배열 기반 스택**:
    *   이 방식은 스택의 `TOP`이 항상 배열의 첫 번째 인덱스(예: `array[0]`)를 가리킨다고 가정합니다.
    *   `push` 연산: 새로운 요소를 `array[0]`에 삽입하고 기존의 모든 요소를 한 칸씩 뒤로 밀어야 합니다.
    *   `pop` 연산: `array[0]`의 요소를 제거하고 기존의 모든 요소를 한 칸씩 앞으로 당겨야 합니다.
    *   이 경우, $n$개의 요소를 가진 스택에서 `push`와 `pop` 연산은 $n$개의 요소 이동을 수반하므로 시간 복잡도는 $O(n)$입니다.
        $$ Cost(push) = O(n) $$
        $$ Cost(pop) = O(n) $$

**구체적 예시**:

1.  **배열 기반 스택 (TOP이 이동)**:
    *   스택: `[ | | ]` (초기, `top = -1`)
    *   `push(A)`: `[A| | ]` (`top = 0`)
    *   `push(B)`: `[A|B| ]` (`top = 1`)
    *   `pop()`: `B` 반환, 스택 `[A| | ]` (`top = 0`)
    *   `push(C)`: `[A|C| ]` (`top = 1`)
    *   여기서 `TOP`은 항상 최상단 요소를 가리키거나 다음 빈 공간을 가리키는 인덱스 변수로, 그 값만 변경하면 되므로 매우 효율적입니다.

2.  **배열 기반 스택 (TOP이 고정, Position 1)**:
    *   스택: `[ | | ]`
    *   `push(A)`: `[A| | ]` (TOP은 항상 0)
    *   `push(B)`: `[B|A| ]` (기존 `A`가 밀려남)
    *   `push(C)`: `[C|B|A]` (기존 `B, A`가 밀려남)
    *   `pop()`: `C` 반환, 스택 `[B|A| ]` (`B, A`가 앞으로 당겨짐)
    *   이 방식은 그림에서 `(Fixed) Position 1: O(n)`이라고 설명된 상황에 해당하며, 실제 스택 구현에서는 거의 사용되지 않습니다.

3.  **단일 배열 내 다중 스택**:
    *   하나의 큰 배열 `[ | | | | | | | | ]`
    *   `Stack1`은 왼쪽 끝에서 시작하여 오른쪽으로 성장 (`Top1`이 오른쪽으로 이동).
    *   `Stack2`는 오른쪽 끝에서 시작하여 왼쪽으로 성장 (`Top2`가 왼쪽으로 이동).
    *   `Top1`과 `Top2`가 충돌하기 전까지 두 스택이 유연하게 배열 공간을 공유할 수 있습니다. 이는 메모리 단편화를 줄이고 공간 활용을 최적화하는 방법 중 하나입니다.

4.  **예시: 콜 스택 (Call Stack)**:
    *   함수 호출 시 프로그램 내에서 사용되는 스택의 대표적인 예시입니다.
    *   `main` 함수가 `funcA`를 호출하고, `funcA`가 `funcB`를 호출하면, 콜 스택에는 `main` -> `funcA` -> `funcB` 순서로 함수 호출 정보(반환 주소, 지역 변수 등)가 쌓입니다.
    *   `funcB`가 종료되면 스택에서 `funcB` 정보가 `pop`되고, `funcA`로 제어가 돌아옵니다. `funcA`가 종료되면 `funcA` 정보가 `pop`되고, `main`으로 제어가 돌아옵니다. 이는 LIFO 원칙을 따릅니다.

**시험 포인트**:
*   ⭐**배열 기반 스택과 연결 리스트 기반 스택의 시간/공간 복잡도 비교**: `push`, `pop`, `top` 연산의 평균 및 최악 시간 복잡도. (둘 다 일반적으로 $O(1)$)
*   ⭐**배열 기반 스택에서 `TOP` 포인터 구현 방식에 따른 효율성 차이**: `TOP`이 동적으로 이동하는 경우($O(1)$)와 고정된 경우($O(n)$)의 시간 복잡도와 그 이유를 설명할 수 있어야 합니다.
*   ⭐**단일 배열 내 다중 스택 구현 아이디어**: 어떻게 공간을 효율적으로 활용하는지 설명할 수 있어야 합니다. (예: 양 끝에서 성장하는 방식)
*   ⭐**스택의 실제 응용**: 콜 스택(Call Stack)이 무엇이며, 어떻게 스택 자료구조의 LIFO 특성을 활용하는지 설명할 수 있어야 합니다. (재귀 호출, 함수 호출 관리 등)

## Queue

### 핵심 개념
큐(Queue)는 **FIFO (First-In First-Out)** 원칙을 따르는 선형 자료구조입니다. 이는 가장 먼저 삽입된 요소가 가장 먼저 제거된다는 의미입니다. 스택(Stack)이 `top`이라는 한쪽 끝에서만 삽입과 삭제가 일어나는 것과 달리, 큐는 두 개의 포인터를 사용합니다:
*   **Front**: 큐의 가장 앞쪽 요소, 즉 가장 먼저 삽입되어 다음으로 제거될 요소를 가리킵니다.
*   **Rear**: 큐의 가장 뒤쪽 요소, 즉 가장 최근에 삽입된 요소를 가리킵니다. 새로운 요소는 Rear를 통해 추가됩니다.

큐는 다음과 같은 주요 연산을 제공합니다:
*   **Enqueue(x, Q)**: 큐의 Rear에 요소 `x`를 추가합니다.
*   **Dequeue(Q)**: 큐의 Front에 있는 요소를 제거하고 반환합니다.
*   **Front(Q)**: 큐의 Front에 있는 요소를 반환하지만, 제거하지는 않습니다.
*   **MakeNull(Q)**: 큐를 비어있는 상태로 만듭니다.
*   **IsEmpty(Q)**: 큐가 비어있는지 여부를 확인하여 `true` 또는 `false`를 반환합니다.

### 코드/수식 해설
큐 $Q$의 현재 상태를 $<a_1, a_2, ..., a_n>$으로 표현할 때, 여기서 $a_1$은 Front, $a_n$은 Rear에 해당합니다.

*   **`Enqueue(x, Q)`**: 큐 $Q$에 요소 $x$를 추가합니다.
    $$Q = <a_1, a_2, ..., a_n> \implies Q' = <a_1, a_2, ..., a_n, x>$$
*   **`Dequeue(Q)`**: 큐 $Q$의 Front에서 요소를 제거하고 반환합니다. 이 연산은 $a_1$을 반환합니다.
    $$Q = <a_1, a_2, ..., a_n> \implies Q' = <a_2, a_3, ..., a_n>$$
*   **`Front(Q)`**: 큐 $Q$의 Front 요소를 반환합니다.
    $$Q = <a_1, a_2, ..., a_n> \implies a_1$$
*   **`MakeNull(Q)`**: 큐 $Q$를 비어있는 상태로 만듭니다.
    $$Q = <a_1, a_2, ..., a_n> \implies Q' = < >$$
*   **`IsEmpty(Q)`**: 큐 $Q$가 비어있는지 확인합니다.
    $$Q = < > \implies \text{true}$$

### 구체적 예시
가장 대표적인 큐의 실생활 예시는 **은행 창구의 대기 줄**입니다.
1.  **Enqueue**: 새로운 손님(요소)이 은행에 도착하면, 줄의 가장 뒤(Rear)에 가서 섭니다. (예: `Enqueue(NewCustomer, BankQueue)`)
2.  **Dequeue**: 은행 직원이 가장 앞에 서 있는 손님(Front 요소)부터 업무를 처리하고 줄에서 내보냅니다. (예: `Dequeue(BankQueue)`)
3.  **Front**: 직원이 다음으로 처리할 손님이 누구인지 확인하지만, 아직 업무를 시작하지는 않습니다. (예: `Front(BankQueue)`)
4.  **MakeNull**: 은행이 문을 닫고 모든 손님이 떠나면 줄이 비어있는 상태가 됩니다.
5.  **IsEmpty**: 은행 직원이 줄이 비었는지 확인하여 더 이상 처리할 손님이 없는지 봅니다.

**단계별 연산 예시**:
초기 빈 큐: $Q = < >$
1.  `Enqueue(A, Q)`: $Q = <A>$ (A는 Front이자 Rear)
2.  `Enqueue(B, Q)`: $Q = <A, B>$ (A는 Front, B는 Rear)
3.  `Enqueue(C, Q)`: $Q = <A, B, C>$ (A는 Front, C는 Rear)
4.  `Front(Q)`: $A$ 반환
5.  `Dequeue(Q)`: $A$ 반환, $Q = <B, C>$ (B는 Front, C는 Rear)
6.  `Enqueue(D, Q)`: $Q = <B, C, D>$ (B는 Front, D는 Rear)
7.  `Dequeue(Q)`: $B$ 반환, $Q = <C, D>$ (C는 Front, D는 Rear)
8.  `IsEmpty(Q)`: false 반환
9.  `MakeNull(Q)`: $Q = < >$
10. `IsEmpty(Q)`: true 반환

### 시험 포인트
*   ⭐ **FIFO (First-In First-Out) 원칙**을 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ 스택(Stack)과의 차이점, 특히 **Front와 Rear의 개념**을 명확히 구분해야 합니다. 스택은 LIFO, 큐는 FIFO입니다.
*   ⭐ 각 큐 연산(`Enqueue`, `Dequeue`, `Front`, `MakeNull`, `IsEmpty`)의 **기능, 동작 방식, 큐 상태 변화, 반환 값**을 정확히 알아야 합니다.
*   ⭐ **시간 복잡도**: 일반적인 큐 구현(배열 또는 연결 리스트)에서 `Enqueue`, `Dequeue`, `Front`, `IsEmpty` 연산은 이상적으로 ⭐**O(1)**의 시간 복잡도를 가집니다. (`MakeNull`은 구현에 따라 달라질 수 있음)
*   ⭐ 큐의 **다양한 응용 분야** (운영체제의 스케줄링, 프린터 대기열, 네트워크 패킷 처리 등)를 이해하는 것이 좋습니다.

### Array-Based Queues (1)

이 슬라이드는 배열 기반 큐의 기본적인 동작과 함께 발생할 수 있는 주요 문제점을 설명합니다.

-   **핵심 개념**:
    *   **배열 기반 큐 (Array-Based Queue)**: 고정된 크기의 배열을 사용하여 큐를 구현하는 방식입니다. First-In, First-Out (FIFO) 원칙에 따라 데이터가 처리됩니다.
    *   **Front (F) 및 Rear (R) 포인터**:
        *   `F` (Front): 큐에서 가장 먼저 나갈 요소(즉, 다음 `dequeue`될 요소)를 가리키는 인덱스입니다.
        *   `R` (Rear): 큐에 가장 나중에 들어온 요소(즉, 다음 `enqueue`될 요소가 저장될 위치의 바로 이전 인덱스)를 가리키는 인덱스입니다.
    *   **선형 큐의 한계 (Linear Queue Limitation)**: 슬라이드에서 제시된 예시와 같이, `F` 포인터가 계속 앞으로 이동(요소 삭제)함에도 불구하고, `R` 포인터가 배열의 끝에 도달하면 큐가 "가득 찼다"고 판단하여 더 이상 요소를 추가할 수 없는 문제가 발생합니다. 이는 배열 앞부분에 빈 공간이 있음에도 불구하고 발생하는 비효율성입니다. 이를 **"가상 만원 (false full)"** 상태라고 합니다.

-   **코드/수식 해설**:
    슬라이드에는 명시적인 코드는 없지만, 큐의 기본 연산과 인덱스 이동을 설명할 수 있습니다.
    *   큐의 초기 상태 (empty): `F = -1`, `R = -1` (또는 `F = 0`, `R = -1` 등으로 구현 방식에 따라 다름)
    *   **`enqueue(data)`**:
        1.  큐가 비어있다면 (`F == -1`), `F = 0`으로 설정합니다.
        2.  `R`을 1 증가시킵니다: $R = R + 1$
        3.  `array[R]`에 `data`를 저장합니다: `array[R] = data`
    *   **`dequeue()`**:
        1.  `array[F]`에 저장된 데이터를 반환합니다.
        2.  `F`를 1 증가시킵니다: $F = F + 1$
        3.  만약 `dequeue` 후 $F > R$이 되면 큐가 비어있는 상태입니다. 이때 `F`와 `R`을 다시 초기 상태(`-1`, `-1`)로 리셋할 수 있습니다.
    *   **큐가 가득 찼는지 확인 (선형 큐)**: $R == capacity - 1$ 일 경우 가득 찼다고 판단합니다.

    슬라이드의 예시를 따라가 보면:
    *   배열 크기: 6 (인덱스 0부터 5)
    *   `enqueue("a", "b", "c")`:
        `F=0`, `R=2`
        배열: `[a, b, c, _, _, _]`
    *   `dequeue()` 두 번:
        `F`는 0에서 1로, 다시 1에서 2로 이동합니다. (논리적으로 `a`, `b` 제거)
        `F=2`, `R=2`
        배열 (논리적): `[_, _, c, _, _, _]`
    *   `enqueue("d", "e", "f")`:
        `R`은 2에서 3, 4, 5로 이동합니다.
        `F=2`, `R=5`
        배열: `[_, _, c, d, e, f]`
    *   현재 상태에서 $R = 5$ 이며, 이는 $capacity - 1$과 같습니다. 따라서 일반적인 선형 배열 기반 큐는 이 상태를 '가득 찼다'고 인식하여 더 이상 데이터를 `enqueue`할 수 없습니다. (`Can we enqueue more data?` 에 대한 답은 `No`)

-   **구체적 예시**:
    은행에서 번호표를 뽑고 대기하는 줄을 생각해봅시다.
    1.  **배열 크기 6**: 은행 창구 앞에 6명이 설 수 있는 대기 공간이 있습니다.
    2.  **`enqueue("a", "b", "c")`**: 손님 'a', 'b', 'c'가 차례로 줄을 섭니다. (현재 줄: `[a, b, c, _, _, _]`, `F`는 'a', `R`은 'c'를 가리킴)
    3.  **`dequeue()` 두 번**: 손님 'a'와 'b'가 창구로 가서 업무를 봅니다. (현재 줄: `[_, _, c, _, _, _]`, `F`는 'c', `R`은 'c'를 가리킴. 'a'와 'b'가 있던 자리는 비어있지만, 줄 전체가 앞으로 당겨지지는 않습니다.)
    4.  **`enqueue("d", "e", "f")`**: 손님 'd', 'e', 'f'가 추가로 줄을 섭니다. (현재 줄: `[_, _, c, d, e, f]`, `F`는 'c', `R`은 'f'를 가리킴)
    5.  **`Can we enqueue more data?`**: 이제 새로운 손님이 오면 줄의 마지막 공간(`R`이 가리키는 `f` 다음)이 없으므로, 비록 'a'와 'b'가 있던 맨 앞 두 자리는 비어있지만, 더 이상 새로운 손님을 받을 수 없게 됩니다. 이것이 선형 배열 큐의 '가상 만원' 상태입니다.

-   **시험 포인트**:
    *   ⭐ 배열 기반 큐의 `enqueue`, `dequeue` 동작 시 `F`와 `R` 포인터의 변화를 정확히 이해하고 설명할 수 있어야 합니다.
    *   ⭐ **선형 배열 기반 큐의 주요 한계점**인 **"가상 만원 (false full)"** 문제를 명확히 설명할 수 있어야 합니다. 즉, 배열의 앞부분에 빈 공간이 있음에도 불구하고 `R`이 배열의 끝에 도달하여 더 이상 요소를 추가할 수 없는 상황을 인지해야 합니다.
    *   ⭐ 이 문제를 해결하기 위한 다음 단계 자료구조(일반적으로 원형 큐)의 필요성을 이해하는 것이 중요합니다.

**핵심 개념**
*   **배열 기반 큐 (Array-Based Queue)**: 큐는 FIFO(First-In, First-Out) 원칙에 따라 데이터를 관리하는 자료구조이며, 배열을 사용하여 구현할 수 있습니다. 배열 기반 큐는 고정된 크기의 메모리 공간을 미리 할당받아 사용합니다.
*   **Front (F) 및 Rear (R) 포인터**: 큐의 동작을 관리하기 위해 두 개의 인덱스(포인터)가 사용됩니다.
    *   `F` (Front): 큐의 가장 앞에 있는 요소, 즉 가장 먼저 나갈 요소의 인덱스를 가리킵니다.
    *   `R` (Rear): 큐의 가장 뒤에 있는 요소, 즉 가장 마지막에 들어온 요소의 인덱스를 가리킵니다.
*   **Drifting Queue 문제**: 배열 기반 큐에서 `dequeue` 연산이 반복되어 큐의 앞부분에 빈 공간이 생기더라도, `enqueue` 연산으로 `R` 포인터가 배열의 끝에 도달하면 더 이상 요소를 추가할 수 없는 문제가 발생합니다. 이는 배열의 물리적인 끝이 큐의 논리적인 끝을 제한하여, 실제로는 빈 공간이 있음에도 불구하고 큐가 "가득 찼다"고 판단하게 만듭니다.

**코드/수식 해설**
여기서는 `F`는 큐의 첫 번째 원소의 인덱스를, `R`은 큐의 마지막 원소의 인덱스를 가리킨다고 가정합니다. 큐가 비어있는 상태에서는 `F > R`입니다 (예: `F=0, R=-1`). 배열의 크기를 `N`이라고 가정합니다.

*   **`enqueue(data)` 연산**:
    ```
    if (R == N - 1) { // 배열의 끝에 도달했는지 확인
        // Queue is full (drifting problem)
        return false; // 삽입 실패
    }
    if (F == -1) { // 큐가 비어있던 경우
        F = 0;
    }
    R++;
    array[R] = data;
    return true; // 삽입 성공
    ```
*   **`dequeue()` 연산**:
    ```
    if (F > R) { // 큐가 비어있는지 확인
        // Queue is empty
        return null; // 삭제할 요소 없음
    }
    data = array[F];
    F++;
    if (F > R) { // 마지막 요소를 dequeue한 경우 큐를 비어있는 상태로 재설정
        F = -1;
        R = -1;
    }
    return data; // 삭제된 요소 반환
    ```
*   **큐가 가득 찼는지 확인**:
    현재 예시의 `drifting queue` 상황에서는 큐가 가득 찼다고 판단하는 조건은 다음과 같습니다.
    $$ R = N - 1 $$
    여기서 `N`은 배열의 총 크기입니다. 이 조건이 참이면, 큐의 논리적인 앞부분에 빈 공간이 있더라도 `R` 포인터가 배열의 물리적인 끝에 도달했으므로 더 이상 `enqueue`할 수 없게 됩니다.

**구체적 예시**
배열의 크기가 6인 큐를 가정합니다 (`N=6`). 초기 상태: `F=-1, R=-1` (비어있음).

1.  **"a", "b", "c"를 Enqueue**:
    *   `enqueue(a)`: `F=0, R=0`, `array = [a, _, _, _, _, _]`
    *   `enqueue(b)`: `F=0, R=1`, `array = [a, b, _, _, _, _]`
    *   `enqueue(c)`: `F=0, R=2`, `array = [a, b, c, _, _, _]`
    (슬라이드 첫 번째 그림과 동일)

2.  **두 번 Dequeue**:
    *   `dequeue()`: "a" 반환. `F=1, R=2`, `array = [_, b, c, _, _, _]`
    *   `dequeue()`: "b" 반환. `F=2, R=2`, `array = [_, _, c, _, _, _]`

3.  **"d", "e", "f"를 Enqueue**:
    *   `enqueue(d)`: `F=2, R=3`, `array = [_, _, c, d, _, _]`
    *   `enqueue(e)`: `F=2, R=4`, `array = [_, _, c, d, e, _]`
    *   `enqueue(f)`: `F=2, R=5`, `array = [_, _, c, d, e, f]`
    (슬라이드 두 번째 그림과 동일)

4.  **더 많은 데이터를 Enqueue할 수 있는가?**:
    현재 `R`은 5입니다. 배열의 크기가 6이므로, 배열의 최대 인덱스는 `N-1 = 5`입니다. `R == N-1` 조건이 만족되므로, 더 이상 `enqueue`할 수 없습니다. 하지만 배열의 인덱스 0과 1은 비어있는 상태입니다 (`_`). 이처럼 큐의 앞부분에 빈 공간이 있음에도 불구하고 배열의 끝에 도달하여 더 이상 요소를 추가할 수 없는 문제가 바로 **drifting queue 문제**입니다.

**시험 포인트**
*   ⭐ **Drifting Queue 문제의 정의 및 발생 원인**을 정확히 설명할 수 있어야 합니다. (배열의 물리적 한계와 큐의 논리적 상태 불일치)
*   배열 기반 큐에서 `F`와 `R` 포인터가 어떻게 움직이는지 `enqueue`와 `dequeue` 연산을 통해 설명하고 그림으로 나타낼 수 있어야 합니다.
*   주어진 배열 크기와 일련의 `enqueue`/`dequeue` 연산 후 큐의 상태(`F`, `R` 값 및 배열 내용)를 예측할 수 있어야 합니다.
*   ⭐ `drifting queue` 문제의 해결책 (예: 원형 큐, 연결 리스트 기반 큐)에 대한 필요성을 이 문제점을 통해 이해하는 것이 중요합니다.

### 핵심 개념
이 슬라이드는 **배열 기반 선형 큐 (Array-based Linear Queue)**의 `enqueue` 및 `dequeue` 연산에서 발생하는 효율성 문제와 그 해결책, 그리고 관련 시간 복잡도를 다룹니다. 특히, 배열의 물리적 끝에 도달하여 논리적으로는 "가득 찬" 상태가 되었으나, 실제로는 배열의 앞부분에 `dequeue` 연산으로 인해 빈 공간이 남아 있는 상황에 초점을 맞춥니다.

### 코드/수식 해설

#### 1. 데이터 구조 변경 없이 새로운 데이터를 Enqueue하는 방법
제시된 슬라이드의 그림은 배열 기반 큐에서 `a`와 `b`가 `dequeue`되고 `c, d, e, f`가 `enqueue`된 상황을 보여줍니다.
*   초기: `F=0, R=2`, 큐 내용 `[a, b, c]`
*   변환 후: `F=2, R=5`, 큐 내용 `[c, d, e, f]` (배열은 `[_, _, c, d, e, f]`)

이 변환 후 상태에서 `rear` 포인터(`R`)는 배열의 물리적 끝(인덱스 5)에 도달해 있습니다. 만약 이 상태에서 새로운 데이터(예: `g`)를 `enqueue`하려고 하면, `rear`를 더 이상 증가시킬 수 없습니다. 하지만 `front` 포인터(`F`)는 인덱스 2에 있으므로, 배열의 인덱스 0과 1에는 빈 공간이 남아있습니다.

이 문제를 **데이터 구조를 변경하지 않고** 해결하는 방법은 **현재 큐에 저장된 모든 유효한 데이터를 배열의 시작 지점(인덱스 0)으로 이동(shift)시키는 것**입니다. 이 과정을 **압축(compaction)**이라고도 부릅니다.

의사 코드로 표현하면 다음과 같습니다:
```cpp
// 큐가 배열의 끝에 도달했으나, 앞부분에 빈 공간이 있는 경우
if (rear == capacity - 1 && front > 0) {
    int current_elements = rear - front + 1; // 현재 큐에 있는 요소의 개수
    for (int i = 0; i < current_elements; ++i) {
        array[i] = array[front + i]; // 데이터를 배열의 시작으로 이동
    }
    front = 0; // front 포인터 재설정
    rear = current_elements - 1; // rear 포인터 재설정
}

// 이후 일반적인 enqueue (새로운 데이터를 rear 위치에 삽입)
// (앞서 shifting이 발생했거나, 애초에 공간이 충분했던 경우)
rear++;
array[rear] = newItem;
// 만약 큐가 비어있었다면 front도 초기화하는 로직 필요
if (front == -1 && current_elements == 0) front = 0;
```

#### 2. Enqueue 및 Dequeue 연산의 시간 복잡도
위와 같이 데이터를 이동(shifting)하는 방식을 사용하는 배열 기반 선형 큐의 경우 시간 복잡도는 다음과 같습니다.

*   **Dequeue**:
    `front` 포인터를 단순히 증가시키면 되므로, 상수 시간 $O(1)$이 소요됩니다.
    $$ \text{Dequeue Time Complexity} = O(1) $$

*   **Enqueue**:
    새로운 데이터를 삽입하는 일반적인 경우에는 $O(1)$이 소요됩니다. 그러나 `rear`가 배열의 끝에 도달하고 `front`가 0이 아닌 경우, **데이터 이동(shifting)**이 발생합니다. 이 이동 작업은 현재 큐에 있는 $N$개의 모든 요소를 복사해야 하므로 $O(N)$의 시간이 소요됩니다. 따라서 `enqueue` 연산의 최악의 경우 시간 복잡도는 $O(N)$입니다.
    $$ \text{Enqueue Time Complexity (Worst Case)} = O(N) $$
    여기서 $N$은 큐에 현재 저장된 요소의 개수입니다.

### 구체적 예시
버스가 정류장에 정차하여 승객을 태우고 내리는 상황으로 비유할 수 있습니다.
1.  **초기 상태**: 버스 좌석 `[A, B, C, (빈칸), (빈칸), (빈칸)]` (총 6자리)
    *   `front` (첫 승객) = A, `rear` (마지막 승객) = C
2.  **Dequeue (승객 내림)**: A와 B가 버스에서 내립니다.
    *   이제 C가 첫 번째 승객이 됩니다.
    *   버스 좌석 `[(빈칸), (빈칸), C, (빈칸), (빈칸), (빈칸)]`
    *   `front` = C, `rear` = C
3.  **Enqueue (승객 탑승)**: D, E, F가 버스에 탑승합니다.
    *   버스 좌석 `[(빈칸), (빈칸), C, D, E, F]`
    *   `front` = C, `rear` = F
    *   버스는 좌석은 아직 2칸 비어있지만, `rear`가 맨 끝 좌석이므로 더 이상 새로운 승객을 태울 수 없는 것처럼 보입니다.
4.  **Enqueue 문제 및 해결**: 새로운 승객 G가 버스에 타려고 합니다. `rear`는 끝에 도달했지만, `front` 앞에는 빈 좌석이 있습니다.
    *   이때 버스 기사는 **모든 승객(C, D, E, F)에게 앞으로 두 칸씩 이동해달라고 요청합니다.**
    *   이동 후: `[C, D, E, F, (빈칸), (빈칸)]`
    *   `front` = C, `rear` = F
    *   이제 G는 새로 생긴 빈칸에 탑승할 수 있습니다: `[C, D, E, F, G, (빈칸)]`
    *   승객 전체를 이동시키는 이 과정이 바로 $O(N)$ 시간이 소요되는 `enqueue` 연산의 최악의 경우입니다.

### 시험 포인트
*   ⭐ **배열 기반 선형 큐의 한계**: `rear`가 배열의 물리적 끝에 도달했으나 `front`가 0이 아니어서 배열의 앞부분에 빈 공간이 남아있는 상태를 정확히 이해하고 설명할 수 있어야 합니다. (이 현상을 "가비지 공간" 또는 "낭비된 공간"이라고도 합니다.)
*   ⭐ **데이터 이동(Shifting) 전략**: 이 문제를 해결하기 위해 **큐에 있는 모든 유효 데이터를 배열의 시작 지점으로 이동(shift)하는 방법**을 반드시 기억해야 합니다.
*   ⭐ **시간 복잡도 분석**:
    *   `Dequeue` 연산은 항상 $O(1)$입니다.
    *   `Enqueue` 연산은 일반적인 경우 $O(1)$이지만, **데이터 이동(shifting)**이 필요한 최악의 경우(가비지 공간 발생 시)에는 $O(N)$의 시간 복잡도를 가집니다.
    *   $N$은 큐에 현재 저장된 요소의 개수입니다.
*   이러한 선형 배열 큐의 비효율성(특히 `enqueue`의 $O(N)$)을 개선하기 위한 대안으로 **원형 큐(Circular Queue)**가 있다는 점도 함께 알아두면 좋습니다. 원형 큐에서는 `enqueue`와 `dequeue` 모두 $O(1)$을 보장합니다.

## Slide 28

**핵심 개념**

*   **순환 배열 (Circular Array) 기반 큐**: 배열을 사용하여 큐를 구현할 때 발생하는 비효율성(예: `dequeue` 시 모든 요소를 이동시켜야 하는 문제)을 해결하기 위해 고안된 자료구조입니다. 배열의 끝이 논리적으로 배열의 시작과 연결되어 마치 원형처럼 데이터를 저장하고 접근합니다.
*   **모듈로 연산자 (Modulo Operator) 활용**: 순환 배열의 핵심은 모듈로 연산자를 사용하여 배열의 인덱스를 관리하는 것입니다. 이를 통해 배열의 마지막 인덱스 다음에 첫 번째 인덱스가 오도록 수학적으로 연결합니다.

**코드/수식 해설**

*   **모듈로 연산**: 슬라이드에서 제시된 $n + 1 \equiv 1 \pmod n$은 모듈로 연산의 일반적인 합동식 예시입니다. 배열 기반 큐에서 다음 인덱스를 계산할 때 실제로 사용되는 형태는 다음과 같습니다.
    새로운 인덱스 $I'$는 현재 인덱스 $I$와 배열의 총 용량 $C$에 대해 다음처럼 계산됩니다:
    $$ I' = (I + 1) \pmod C $$
    이 수식은 `Rear` 또는 `Front` 포인터를 다음 위치로 이동시킬 때 배열의 끝에 도달하면 자동으로 `0`번 인덱스로 돌아오게 합니다.

*   **Enqueue 연산**: 큐에 새 요소 `x`를 추가(`Enqueue`)하는 경우, `Rear` 포인터를 이동시키고 해당 위치에 `x`를 저장합니다.
    ```cpp
    void Enqueue(Queue& Q, Element x) {
        if (isFull(Q)) { // 큐가 가득 찼는지 확인
            // 오류 처리 또는 크기 확장
            return;
        }
        Q.rear = (Q.rear + 1) % Q.capacity; // rear 포인터 이동
        Q.data[Q.rear] = x; // 새 요소 저장
        Q.size++; // 큐의 크기 증가
    }
    ```
    (참고: `isFull(Q)` 및 `Q.size`와 같은 상태 관리는 순환 배열 큐 구현의 필수 요소입니다.)

**구체적 예시**

슬라이드의 그림은 순환 배열 큐에서 `Enqueue(x, Q)`가 어떻게 동작하는지 보여줍니다.

1.  **초기 상태**:
    *   `F` (Front) 포인터는 'a'가 있는 위치를 가리킵니다.
    *   `R` (Rear) 포인터는 'c'가 있는 위치를 가리킵니다.
    *   배열의 논리적 순서: a, b, c (F에서 R까지)
    *   `R`은 현재 2번 인덱스라고 가정하고, 배열의 총 용량은 10입니다.

2.  **Enqueue(x, Q) 동작**:
    *   새로운 요소를 삽입하기 위해 먼저 `R` 포인터를 다음 위치로 이동시켜야 합니다.
    *   현재 `R`이 가리키는 인덱스가 2이므로, 새로운 `R`의 인덱스는 $(2 + 1) \pmod{10} = 3$이 됩니다.
    *   그림에서는 'c' 다음 위치에 'x'가 삽입되고, `R` 포인터가 'x'를 가리키게 됩니다.

3.  **순환 동작 예시**: 만약 `R` 포인터가 배열의 마지막 인덱스인 9를 가리키고 있었다면, 다음 `Enqueue` 연산 시 `R`의 새 인덱스는 $(9 + 1) \pmod{10} = 0$이 되어 배열의 가장 처음 인덱스(0번)로 돌아와 요소를 삽입하게 됩니다.

**시험 포인트**

*   ⭐ **순환 배열을 사용하는 이유**: 일반 배열 큐에서 `dequeue` 시 발생하는 요소들의 이동(쉬프팅)으로 인한 $O(N)$의 시간 복잡도 문제를 해결하고, `enqueue`와 `dequeue` 모두 $O(1)$의 시간 복잡도로 수행하기 위함입니다.
*   ⭐ **모듈로 연산자의 역할**: 순환 배열에서 `Front`와 `Rear` 포인터를 다음 위치로 이동시킬 때, 배열의 물리적인 끝을 논리적인 시작과 연결하여 원형 구조를 구현하는 데 핵심적인 역할을 합니다.
*   ⭐ **Enqueue/Dequeue 시 포인터 업데이트 로직**: 특히 `(index + 1) % capacity`와 같은 모듈로 연산을 사용하여 포인터를 올바르게 이동시키는 방법을 이해하고 설명할 수 있어야 합니다.
*   ⭐ **빈 큐와 가득 찬 큐의 구분**: (슬라이드에 직접 명시되진 않았지만, 순환 배열 큐의 중요한 개념입니다.) `Front`와 `Rear` 포인터만으로는 빈 큐와 가득 찬 큐를 구분하기 어렵습니다. 이를 해결하기 위해 보통 배열의 한 칸을 비워두거나 (`(R + 1) % C == F`일 때 가득 참), 큐의 현재 요소 개수를 저장하는 `size` 변수를 사용합니다.

## Slide 29

**핵심 개념**: 슬라이드는 배열(array)을 기반으로 구현된 **원형 큐(Circular Queue)**의 `enqueue` (데이터 삽입) 연산에 대한 질문과 그 과정을 시각적으로 보여줍니다. 원형 큐는 일반적인 배열 기반 큐에서 `dequeue` 연산 후 발생하는 앞부분의 메모리 공간 낭비 문제를 해결하기 위해, 배열의 끝과 시작을 논리적으로 연결하여 공간을 효율적으로 재활용하는 자료구조입니다. `front` (F) 포인터는 큐의 가장 앞 원소를, `rear` (R) 포인터는 큐의 가장 뒤 원소를 가리킵니다.

**코드/수식 해설**:
원형 큐에서 `enqueue(item)` 연산은 새로운 데이터를 큐의 `rear`에 추가하는 과정이며, `dequeue()` 연산은 큐의 `front`에서 데이터를 제거하는 과정입니다. 두 연산 모두 원형 배열의 특성을 반영하기 위해 모듈로(modulo) 연산을 사용합니다.

`enqueue(item)` 연산의 로직:
1.  **큐가 가득 찼는지 확인**: 만약 큐가 가득 찼다면 더 이상 데이터를 삽입할 수 없습니다.
    *   일반적인 구현에서는 `rear` 포인터가 가리키는 다음 위치가 `front` 포인터와 같아지면 큐가 가득 찬 것으로 판단합니다. 이 경우 한 칸을 항상 비워두어 `front == rear`일 때 '큐가 비어있음'과 '큐가 가득 참'을 구분합니다.
    $$ \text{isFull}(Q) \quad \text{if } (Q.\text{rear} + 1) \pmod{\text{MAX\_SIZE}} == Q.\text{front} $$
    (`MAX_SIZE`는 큐 배열의 전체 크기)
2.  **`rear` 포인터 업데이트**: 큐가 가득 차지 않았다면, `rear` 포인터를 다음 위치로 이동시킵니다. 원형 배열이므로 `MAX_SIZE`를 나눈 나머지 연산을 사용합니다.
    $$ Q.\text{rear} = (Q.\text{rear} + 1) \pmod{\text{MAX\_SIZE}} $$
3.  **데이터 삽입**: 업데이트된 `rear` 위치에 새로운 데이터를 저장합니다.
    $$ Q.\text{data}[Q.\text{rear}] = \text{item} $$
4.  **초기 상태 처리**: 만약 큐가 비어있던 상태에서 첫 번째 원소를 삽입하는 경우, `front` 포인터도 적절히 초기화해야 합니다 (예: `front = 0`).

`dequeue()` 연산의 로직 (참고):
1.  **큐가 비어있는지 확인**: 만약 큐가 비어있다면 제거할 데이터가 없습니다.
    $$ \text{isEmpty}(Q) \quad \text{if } Q.\text{front} == Q.\text{rear} $$
2.  **`front` 포인터 업데이트**: 큐가 비어있지 않다면, `front` 포인터를 다음 위치로 이동시킵니다.
    $$ Q.\text{front} = (Q.\text{front} + 1) \pmod{\text{MAX\_SIZE}} $$
3.  **데이터 반환**: 업데이트된 `front` 위치의 데이터를 반환합니다.
    $$ \text{return } Q.\text{data}[Q.\text{front}] $$

```c
// 예시: C언어 기반의 원형 큐 구현 (MAX_SIZE = 8, F=0, R=0으로 초기화, F는 첫 번째 원소 직전, R은 마지막 원소를 가리킴)
// 초기: front = 0, rear = 0 (비어있음)
// 큐에 꽉 찼을 때 (rear + 1) % MAX_SIZE == front
#define MAX_SIZE 8 // 배열의 전체 크기

typedef struct {
    int data[MAX_SIZE];
    int front; // 큐의 첫 번째 원소의 '직전' 인덱스
    int rear;  // 큐의 마지막 원소의 인덱스
} CircularQueue;

// 큐 초기화
void initQueue(CircularQueue *q) {
    q->front = 0;
    q->rear = 0;
}

// 큐가 비어있는지 확인
bool isEmpty(CircularQueue *q) {
    return q->front == q->rear;
}

// 큐가 가득 찼는지 확인
bool isFull(CircularQueue *q) {
    return (q->rear + 1) % MAX_SIZE == q->front;
}

// enqueue 연산
void enqueue(CircularQueue *q, int item) {
    if (isFull(q)) {
        printf("Queue is full. Cannot enqueue.\n");
        return;
    }
    q->rear = (q->rear + 1) % MAX_SIZE; // rear를 다음 위치로 이동
    q->data[q->rear] = item;             // 데이터 삽입
    printf("Enqueued: %d. F: %d, R: %d\n", item, q->front, q->rear);
}

// dequeue 연산
int dequeue(CircularQueue *q) {
    if (isEmpty(q)) {
        printf("Queue is empty. Cannot dequeue.\n");
        return -1; // 에러 값 반환
    }
    q->front = (q->front + 1) % MAX_SIZE; // front를 다음 위치로 이동
    int item = q->data[q->front];          // 데이터 추출
    printf("Dequeued: %d. F: %d, R: %d\n", item, q->front, q->rear);
    return item;
}
```

**구체적 예시**:
슬라이드의 그림은 전체 8칸의 원형 큐에서 `Enqueue(x, Q)` 연산을 수행하는 과정을 보여줍니다.
1.  **초기 상태**:
    *   큐에는 'a', 'b', 'c'가 순서대로 저장되어 있습니다.
    *   `F` 포인터는 'a'의 직전 칸을, `R` 포인터는 'c'를 가리킵니다. (예를 들어, `F=7`, `R=2` 또는 `F=0`, `R=2`와 같이 구현에 따라 달라질 수 있습니다. 슬라이드 그림은 F가 a를, R이 c를 가리키는 것으로 보입니다. 위 코드 예시와 일관성을 위해 F는 첫 원소 직전, R은 마지막 원소로 가정하고 설명합니다.)
    *   여기서 `F`가 0, `a`가 1, `b`가 2, `c`가 3번 인덱스에 있다면, `F=0, R=3` 상태입니다.
2.  **`Enqueue(x, Q)` 연산**:
    *   현재 `(R + 1) % 8 = (3 + 1) % 8 = 4` 입니다. 이는 `F`와 다르므로 큐는 가득 차지 않았습니다.
    *   `R` 포인터를 다음 위치로 이동시킵니다: `R = (3 + 1) % 8 = 4`.
    *   `Q.data[4]`에 새로운 데이터 'x'를 저장합니다.
3.  **결과 상태**:
    *   큐에는 'a', 'b', 'c', 'x'가 순서대로 저장됩니다.
    *   `F` 포인터는 여전히 'a'의 직전 칸을 가리키고, `R` 포인터는 새로 삽입된 'x'를 가리킵니다. (예를 들어, `F=0`, `R=4`)

**시험 포인트**:
*   ⭐ **원형 큐의 `enqueue` 및 `dequeue` 연산 구현 로직**: `front`와 `rear` 포인터를 업데이트할 때 **모듈로 연산(`% MAX_SIZE`)**을 사용하는 이유와 방법을 정확히 이해하고 설명할 수 있어야 합니다.
*   ⭐ **원형 큐의 Full (가득 참) 조건과 Empty (비어있음) 조건**: 이 두 가지 상태를 구분하는 방법 (예: 한 칸 비워두기, `count` 변수 사용 등)과 해당 조건식(`(rear + 1) % MAX_SIZE == front`, `front == rear`)을 정확히 암기하고 설명할 수 있어야 합니다.
*   ⭐ **일반 큐와 원형 큐의 장단점 비교**: 원형 큐가 일반 배열 기반 큐의 어떤 문제점(공간 낭비)을 해결하는지 설명할 수 있어야 합니다.
*   ⭐ **시간 복잡도**: 원형 큐의 `enqueue`와 `dequeue` 연산은 항상 포인터 이동과 데이터 삽입/삭제가 상수 시간에 이루어지므로, 시간 복잡도는 $O(1)$ 입니다.

## 배열 기반 원형 큐의 공백/포화 상태 구별 문제

### 핵심 개념
이 슬라이드는 배열 기반의 원형 큐(Circular Queue)를 구현할 때 발생하는 고질적인 문제, 즉 **큐가 비어있는 상태(empty)와 큐가 가득 찬 상태(full)를 두 개의 포인터(Front `F`와 Rear `R`)만으로 구별하기 어려운 경우**를 설명합니다. 원형 큐는 고정된 크기의 배열을 사용하며, `enqueue` 및 `dequeue` 작업 시 포인터들이 배열의 양 끝에 도달하면 다시 처음으로 돌아가는(모듈러 연산 활용) 방식으로 구현됩니다.

### 코드/수식 해설

슬라이드에 제시된 상황과 수식은 다음과 같습니다:

1.  **초기 상태 (`F = R`)**:
    그림 왼쪽에서 `F`와 `R`이 모두 `a`라는 한 요소(element)를 가리키고 있습니다. 이는 큐에 **하나의 원소만 존재**하는 상태를 나타냅니다 (일반적으로 `F`는 첫 번째 원소를, `R`은 마지막 원소를 가리키는 포인터 정의를 가정).

2.  **`Dequeue(Q)` 연산**:
    큐에서 `a`를 `dequeue`하는 연산이 수행됩니다. 이로 인해 큐는 비어있는 상태가 됩니다.

3.  **결과 상태 (`R + 1 = F (mod size)`)**:
    `a`가 제거된 후, 그림 오른쪽을 보면 `R` 포인터는 `a`가 있던 위치를 가리키고, `F` 포인터는 그 다음 위치를 가리키고 있습니다 (원형으로 순환한 상태). 이 때의 포인터 관계는 `$R + 1 \equiv F \pmod{size}$` 로 표현됩니다. 여기서 `size`는 큐 배열의 크기입니다.

문제는 `$R + 1 \equiv F \pmod{size}$` 라는 조건이 **큐가 가득 찬(full) 상태**를 나타내는 데도 사용될 수 있다는 점입니다. 예를 들어, `F`가 첫 번째 원소를 가리키고 `R`이 다음 원소가 들어갈 빈 공간을 가리키는 구현 방식에서, 큐가 가득 차면 `R`은 `F` 바로 앞에 오게 됩니다. 이 경우에도 `$R + 1 \equiv F \pmod{size}$` 조건이 만족될 수 있습니다.

### 구체적 예시

*   **가정**: 크기가 8인 원형 큐(`size = 8`). 인덱스는 0부터 7까지.
*   **포인터 정의**:
    *   `F`: 큐의 첫 번째 원소를 가리킴.
    *   `R`: 큐의 마지막 원소를 가리킴.
*   **시나리오**:
    1.  `Enqueue('a')`만 수행되어 큐에 `'a'`만 들어있는 상태. `F = 0`, `R = 0` (슬라이드의 왼쪽 그림 상태).
    2.  `Dequeue()`를 수행하여 `'a'`를 제거. 큐는 비어있는 상태가 됩니다.
    3.  이 때 `F`와 `R`의 위치는 어떻게 될까요? 슬라이드 그림처럼 `R`이 `0`을 가리키고 `F`가 `1`을 가리키도록 갱신될 수 있습니다 (`F = (R+1)%size`).
    4.  즉, **큐가 비어있는 상태**인데도 `$R + 1 \equiv F \pmod{size}$` 조건이 성립합니다.

*   **대조군 (큐가 가득 찬 상태)**:
    1.  큐가 완전히 가득 찬 상태를 상상해봅시다. 예를 들어, `F = 0`이고 `R = 7`이라면, 8개의 모든 칸에 원소가 들어있을 수 있습니다.
    2.  만약 `F`가 첫 번째 원소의 인덱스, `R`이 마지막 원소의 인덱스라면, 큐가 가득 찬 상태에서 `R`이 `F` 바로 앞에 위치할 수 있습니다. 예를 들어, `F=1, R=0`인 상태는 `$R + 1 \equiv F \pmod{size}$` ($0+1 \equiv 1 \pmod 8$)를 만족하며 큐가 가득 차 있다고 볼 수 있습니다.

이처럼 **두 포인터만으로는 큐의 공백 상태와 포화 상태를 명확히 구별하기 어려운 문제**가 발생합니다. 슬라이드의 스틱맨이 "Empty? or Full?"이라며 혼란스러워하는 이유입니다.

### 시험 포인트
*   ⭐ **원형 배열 기반 큐의 공백/포화 상태 구별 문제의 본질**: `F`와 `R` 두 포인터만 사용하여 `N`개의 모든 슬롯을 활용하는 원형 큐를 구현할 때, 특정 조건(예: `$R + 1 \equiv F \pmod{size}$`)이 공백 상태와 포화 상태를 동시에 나타낼 수 있어 모호함이 발생한다는 점을 이해하고 설명할 수 있어야 합니다.
*   이 문제의 해결책으로는 다음 방법들이 주로 사용됩니다 (이 슬라이드에 직접 제시되지는 않지만, 중요한 관련 개념이므로 함께 알아두세요):
    *   **카운트 변수(`count`) 사용**: 큐에 들어있는 원소의 개수를 별도로 관리하여 `count == 0`이면 공백, `count == size`이면 포화로 판단.
    *   **한 칸 비워두기(One Empty Slot)**: 배열 크기를 `size`로 선언하고 최대 `size - 1`개의 원소만 저장. 이 경우 `$F = R$`이면 공백, `$(R + 1) \pmod{size} == F$`이면 포화로 명확히 구별 가능.
    *   **플래그 변수(`flag`) 사용**: `is_empty` 또는 `is_full`과 같은 불리언 변수를 두어 상태를 명시적으로 저장.

## Slide 31

이 슬라이드는 원형 큐(circular queue) 구현 시 큐가 비어있는 상태(`empty`)와 가득 찬 상태(`full`)를 구분하는 방법에 대한 해결책들을 제시합니다. `front`와 `rear` 포인터만 사용할 경우 발생할 수 있는 모호성을 해소하는 것이 주된 내용입니다.

---

### **핵심 개념**

원형 배열을 이용한 큐 구현에서 `front` (가장 오래된 요소를 가리킴)와 `rear` (가장 최근에 추가된 요소를 가리킴) 포인터만으로는 큐가 비어있는 상태와 가득 찬 상태를 명확히 구분하기 어려운 문제가 발생합니다. 예를 들어, `front`와 `rear`가 같은 위치를 가리킬 때, 큐가 초기 상태로 비어있을 수도 있고, 모든 칸이 채워진 후 `rear`가 `front` 위치로 돌아와서 가득 찬 상태일 수도 있습니다. 이 슬라이드는 이러한 모호성을 해결하기 위한 두 가지 주요 방법을 소개합니다.

### **코드/수식 해설**

1.  **명시적인 요소 개수 변수 (`Explicit count variable`) 사용**:
    큐에 현재 몇 개의 요소가 들어있는지를 추적하는 `cnt` 변수를 사용합니다.
    *   **enqueue 시**: 요소를 큐에 추가할 때마다 `cnt`를 1 증가시킵니다.
        ```cpp
        cnt++;
        ```
    *   **dequeue 시**: 요소를 큐에서 제거할 때마다 `cnt`를 1 감소시킵니다.
        ```cpp
        cnt--;
        ```
    이 방법을 사용하면 큐가 비어있는지 (`cnt == 0`), 가득 찼는지 (`cnt == queue_capacity`)를 명확하게 알 수 있습니다.

2.  **부울 변수 (`Boolean variable`) 사용**:
    큐의 특정 상태를 나타내는 `bool` 변수를 사용할 수 있습니다. 슬라이드에서는 `bool isempty;`를 제시합니다.
    *   `bool isempty;`: 큐가 비어있는지 여부를 나타내는 변수입니다. 이 변수는 주로 큐의 초기 상태나 모든 요소가 제거되었을 때 `true`로 설정되고, 요소가 추가될 때 `false`로 설정됩니다. 하지만 `isempty` 단독으로는 큐가 가득 찬 상태를 구분하는 데는 직접적인 도움을 주지 못하며, 보통 `front`와 `rear` 포인터의 관계 또는 `cnt` 변수와 함께 사용됩니다.

3.  **원형 큐의 가득 찬 상태 수식**:
    $$R + 1 \equiv F \pmod{size}$$
    이 수식은 `rear` 포인터의 다음 위치가 `front` 포인터와 같을 때를 나타냅니다. 원형 큐 구현에서 이 조건은 일반적으로 큐가 *가득 찬* 상태임을 의미하며, 이 경우 배열의 한 칸을 의도적으로 비워두는 관행(sentinel approach)을 통해 `front == rear`가 비어있는 상태로만 사용되도록 합니다. 즉, `(rear + 1) % size == front` 이면 큐가 가득 찬 것으로 판단합니다. 슬라이드의 그림에서 이 수식이 "Empty? or Full?"이라는 질문과 함께 제시된 것은, `front`와 `rear` 포인터만으로는 비어있는 상태(`F == R` 초기 또는 마지막 제거 후)와 가득 찬 상태(`F == R` 또는 `(R+1)%size == F` 같은 특정 조건)를 구분하기 어려운 문제점을 강조하는 것입니다.

### **구체적 예시**

5칸짜리 원형 큐 (인덱스 0~4)를 생각해보겠습니다.
*   **`front`와 `rear`만 사용하는 경우의 모호성**:
    *   **초기 상태**: `front = 0`, `rear = 0`. 큐는 비어있습니다.
    *   **모든 칸을 채운 후 (예: 5개 요소 enqueue)**: `front = 0`, `rear = 0`이 될 수 있습니다 (구현 방식에 따라). 이 경우, `front == rear`는 비어있는 상태를 나타낼 수도, 가득 찬 상태를 나타낼 수도 있어 혼란스럽습니다.
    *   **해결책 1: `cnt` 변수 사용**:
        `cnt` 변수를 추가하여 큐에 있는 요소의 개수를 셉니다.
        *   `cnt = 0`: 큐가 비어있음.
        *   `cnt = 5`: 큐가 가득 참.
        이 방법은 가장 직관적이고 모호성이 없습니다.
    *   **해결책 2: 한 칸 비우기 (implicit sentinel)**:
        배열의 한 칸을 항상 비워두어 `front`와 `rear`가 같으면 비어있고, `(rear + 1) % size == front` 이면 가득 찬 것으로 간주합니다. 이 경우 5칸 배열에는 최대 4개의 요소만 저장할 수 있습니다. 예를 들어, `front = 0`, `rear = 4`일 때 큐는 4개의 요소를 가지고 있으며, `(4 + 1) % 5 == 0`, 즉 `rear` 다음이 `front`와 같으므로 큐는 가득 찬 것으로 판단합니다.

### **시험 포인트**

*   ⭐ **원형 큐에서 `front`와 `rear` 포인터만으로 큐의 `empty` 상태와 `full` 상태를 구분하기 어려운 이유와 그 모호성**: 특히 `F == R` 또는 `(R + 1) % size == F`와 같은 조건이 동시에 두 가지 다른 상태를 나타낼 수 있는 상황을 이해하는 것이 중요합니다.
*   ⭐ **모호성을 해결하는 두 가지 주요 방법**:
    1.  **명시적인 요소 개수 (`count`) 변수를 사용하는 방법**과 각 연산(`enqueue`, `dequeue`) 시 이 변수가 어떻게 갱신되는지.
    2.  **배열의 한 칸을 의도적으로 비워두는 방법**(`(R + 1) % size == F`일 때 `full`로 간주하고, `F == R`일 때 `empty`로 간주)의 원리와 `size` 대비 저장 가능한 실제 요소의 개수.
*   `bool isempty;`와 같은 상태 플래그가 어떻게 사용될 수 있는지 이해하는 것도 중요합니다.

## Slide 32

### **핵심 개념**
*   **연결 리스트 기반 큐 (Linked Queues)**: 데이터를 저장하는 노드들이 포인터로 연결되어 큐를 형성하는 자료구조입니다. 배열 기반 큐와 달리 크기 제한이 없으며, 동적으로 메모리를 할당하고 해제하여 유연하게 동작합니다.
*   **두 개의 포인터 (F, R) 사용**:
    *   `F` (Front): 큐의 가장 앞(first)에 있는 요소를 가리키는 포인터입니다. `Dequeue` 연산 시 제거될 요소를 지칭합니다.
    *   `R` (Rear): 큐의 가장 뒤(last)에 있는 요소를 가리키는 포인터입니다. `Enqueue` 연산 시 새로운 요소가 추가될 위치를 지칭합니다.
*   **더미 헤더 없음 (without a dummy header)**: 이 구현 방식에서는 큐의 시작을 나타내는 특별한 "더미 헤더" 노드를 사용하지 않습니다. 이는 큐가 비어있을 때와 비어있지 않을 때의 `Enqueue`/`Dequeue` 로직에 차이를 둘 필요가 있음을 의미합니다.

### **코드/수식 해설**

**1. 큐의 상태 표현**:
*   큐가 비어 있지 않을 때: `F`는 첫 번째 노드를 가리키고, `R`은 마지막 노드를 가리킵니다. 마지막 노드의 `next` 포인터는 `NULL`입니다.
    *   _예시 다이어그램 1_: `F`는 'a'를, `R`은 'c'를 가리키며, 'c'의 `next` 포인터는 `NULL`입니다.

**2. `Empty(Q)` 연산**:
*   큐가 비어있는지 확인하는 함수입니다.
*   슬라이드에 제시된 조건: `true if (F = R -> null)`
    *   이 표기는 `F`와 `R` 두 포인터가 모두 `NULL`인 경우를 의미합니다. 즉, 큐가 아무런 노드도 가지고 있지 않을 때를 말합니다.
    *   실제 코드에서는 다음과 같이 표현됩니다.
    ```c
    // 큐 노드 구조체 (예시)
    typedef struct Node {
        int data;
        struct Node *next;
    } Node;

    // 큐 구조체 (예시)
    typedef struct {
        Node *front; // F 포인터
        Node *rear;  // R 포인터
    } LinkedQueue;

    // Empty(Q) 함수
    int isEmpty(LinkedQueue *q) {
        // front가 NULL이면 큐는 비어있는 상태이다.
        // 이 경우 rear도 반드시 NULL이어야 한다.
        return (q->front == NULL);
    }
    ```

**3. `Enqueue(x, Q)` 연산**:
*   `x`라는 데이터를 큐의 뒤쪽(`rear`)에 삽입하는 연산입니다.
*   _슬라이드 예시 다이어그램 2_: 기존 큐(a, b, c)에 'x'를 추가합니다.
    *   새 노드 `x`를 생성하고, 기존 `R`이 가리키던 노드('c')의 `next` 포인터가 `x`를 가리키도록 변경합니다.
    *   이후 `R` 포인터를 새로 삽입된 노드(`x`)로 업데이트합니다.
    *   **중요**: 큐가 비어있을 때와 비어있지 않을 때의 로직이 다릅니다.

    ```c
    // Enqueue(x, Q) 함수
    void enqueue(LinkedQueue *q, int item) {
        Node *newNode = (Node *)malloc(sizeof(Node));
        if (newNode == NULL) {
            // 메모리 할당 실패 처리 (예: 오류 메시지 출력 후 종료)
            fprintf(stderr, "Memory allocation failed for new node.\n");
            return;
        }
        newNode->data = item;
        newNode->next = NULL; // 새로 추가되는 노드는 항상 큐의 마지막이므로 next는 NULL

        // 💡 큐가 비어있는 경우 (첫 번째 노드 삽입)
        if (isEmpty(q)) {
            q->front = newNode; // front와 rear 모두 새 노드를 가리킨다.
            q->rear = newNode;
        }
        // 큐가 비어있지 않은 경우
        else {
            q->rear->next = newNode; // 기존 마지막 노드의 next를 새 노드로 연결
            q->rear = newNode;       // rear 포인터를 새 노드로 업데이트
        }
    }
    ```

**4. 슬라이드의 질문**: "Can we use the same code when inserting into an empty queue?"
*   **답변**: **아니요, 일반적으로 동일한 코드를 사용할 수 없습니다.**
*   **이유**:
    *   큐가 비어있지 않을 때의 `Enqueue` 로직 (`q->rear->next = newNode;`)은 `q->rear`가 유효한 노드를 가리키고 있음을 전제로 합니다.
    *   그러나 큐가 비어있는 상태에서는 `q->rear`가 `NULL`입니다. `NULL` 포인터에 대해 `->next`와 같은 멤버 접근을 시도하면 **런타임 오류 (예: Segmentation Fault)**가 발생합니다.
    *   또한, 큐가 비어있을 때 첫 노드를 삽입하면 `q->front` 포인터도 이 새 노드를 가리키도록 초기화해야 합니다. (이후 노드 삽입 시 `q->front`는 변하지 않습니다.) 이러한 특수한 초기화 과정이 필요하므로 `if (isEmpty(q))`와 같은 조건문을 통해 분기 처리가 이루어져야 합니다.

### **구체적 예시**

**예시 1: 큐 초기 상태**
*   `LinkedQueue Q;`
*   `Q.front = NULL; Q.rear = NULL;` (큐는 비어있음)

**예시 2: `enqueue(&Q, 10)` (비어있는 큐에 삽입)**
1.  새 노드(`newNode`) 생성, `data = 10`, `next = NULL`.
2.  `isEmpty(&Q)`는 `true`이므로, `Q.front = newNode`, `Q.rear = newNode`.
    *   결과: `F -> [10 | NULL] <- R`

**예시 3: `enqueue(&Q, 20)` (비어있지 않은 큐에 삽입)**
1.  새 노드(`newNode`) 생성, `data = 20`, `next = NULL`.
2.  `isEmpty(&Q)`는 `false`이므로, `Q.rear->next = newNode` (즉, `[10 | NULL]` 노드의 `next`가 새 노드를 가리키게 됨).
3.  `Q.rear = newNode` (R이 `[20 | NULL]`을 가리키게 됨).
    *   결과: `F -> [10 | ->] -> [20 | NULL] <- R`

**예시 4: 슬라이드 다이어그램에 `enqueue(&Q, 'x')` (F->a->b->c<-R 상태에서 'x' 삽입)**
1.  새 노드(`newNode`) 생성, `data = 'x'`, `next = NULL`.
2.  `isEmpty(&Q)`는 `false`이므로, `Q.rear`가 가리키는 'c' 노드의 `next`를 `newNode`로 설정 (즉, 'c'가 'x'를 가리키게 됨).
3.  `Q.rear = newNode` (R이 'x'를 가리키게 됨).
    *   결과: `F -> [a|->] -> [b|->] -> [c|->] -> [x|NULL] <- R`

### **시험 포인트**
*   ⭐ **연결 리스트 큐에서 `F`와 `R` 포인터의 정확한 역할과 의미**: 각각 큐의 앞과 뒤를 담당하며, `Enqueue`와 `Dequeue` 시 어떻게 업데이트되는지 이해해야 합니다.
*   ⭐ **"더미 헤더 없는" 구현의 특징과 그로 인한 예외 처리의 필요성**: 큐가 비어있을 때 (특히 첫 요소를 삽입할 때) `F`와 `R` 포인터를 모두 초기화/업데이트해야 하는 특수한 경우를 인지하고 있어야 합니다.
*   ⭐ **빈 큐(`Empty(Q)`) 상태의 판단 기준**: `F`와 `R`이 모두 `NULL`인 상태가 빈 큐임을 정확히 알고 있어야 합니다.
*   ⭐ **`Enqueue` 연산 시 `F`와 `R`의 변화 로직**:
    *   **비어있지 않은 큐에 삽입**: `F`는 그대로 유지되고, `R`만 새로 삽입된 노드를 가리키도록 업데이트됩니다.
    *   **비어있는 큐에 삽입**: `F`와 `R` 둘 다 새로 삽입된 노드를 가리키도록 업데이트됩니다. 이 두 경우를 처리하기 위한 조건문(`if (isEmpty(q))`)이 필수적이며, 이를 구현하는 코드를 작성하거나 분석하는 문제가 출제될 수 있습니다.

### Linked Queues (2) - With a Dummy Header

**핵심 개념**:
*   **더미 헤더를 이용한 연결 큐 (Linked Queues with a Dummy Header)**: 연결 리스트를 사용하여 큐를 구현할 때, `Enqueue` (삽입) 및 `Dequeue` (삭제) 연산의 구현을 단순화하기 위해 도입하는 기법입니다. 특히, 큐가 비어있거나 단일 원소를 가질 때 발생하는 특별한 경계 조건(edge cases) 처리를 일반화하여 코드 복잡성을 줄입니다.
*   **더미 헤더 노드 (Dummy Header Node)**: 실제 데이터를 저장하지 않고 큐의 시작점을 나타내는 특별한 노드입니다. 이 노드는 큐가 비어있을 때도 항상 존재합니다.
    *   `F` (Front) 포인터는 항상 이 더미 헤더 노드를 가리킵니다.
    *   `R` (Rear) 포인터는 큐의 마지막 실제 데이터 노드를 가리킵니다.
    *   큐가 비어있을 때는 `R` 포인터도 `F`와 동일하게 더미 헤더 노드를 가리킵니다.
*   **큐의 비어있음 확인 (`Empty(Q)`)**: 큐가 비어있는지 여부는 `F`와 `R` 포인터가 동일한 더미 헤더 노드를 가리키는지로 판단합니다.

**코드/수식 해설**:

먼저, 연결 리스트의 노드 구조와 큐의 구조를 정의합니다.

```c
typedef struct Node {
    int data;          // 노드에 저장될 데이터
    struct Node* next; // 다음 노드를 가리키는 포인터
} Node;

typedef struct {
    Node* F; // Front 포인터: 항상 더미 헤더 노드를 가리킴
    Node* R; // Rear 포인터: 마지막 실제 데이터 노드를 가리킴
} LinkedQueue;
```

**`CreateQueue()` (큐 초기화)**:
큐를 처음 생성할 때, 더미 헤더 노드를 할당하고 `F`와 `R`이 모두 이 노드를 가리키도록 설정합니다.

```c
LinkedQueue* CreateQueue() {
    LinkedQueue* Q = (LinkedQueue*)malloc(sizeof(LinkedQueue));
    if (!Q) { /* 에러 처리 */ }
    Q->F = (Node*)malloc(sizeof(Node)); // 더미 헤더 노드 생성
    if (!Q->F) { /* 에러 처리 */ }
    Q->F->next = NULL; // 더미 헤더의 다음은 처음엔 NULL
    Q->R = Q->F;       // R도 더미 헤더를 가리킴 (큐가 비어있음)
    return Q;
}
```

**`Empty(Q)` (큐가 비었는지 확인)**:
`F`와 `R` 포인터가 동일한 노드(더미 헤더)를 가리키면 큐는 비어있는 상태입니다.

```c
bool Empty(LinkedQueue* Q) {
    return (Q->F == Q->R); // F와 R이 동일한 더미 헤더를 가리키면 비어있음
    // 또는 Q->F->next == NULL; (더미 헤더 다음 노드가 없으면 비어있음)
}
```
슬라이드에 제시된 `true if (F = R -> header)`는 일반적인 표기법이 아니며 혼란을 줄 수 있습니다. 위 코드는 그림 예시를 기반으로 `F`와 `R`이 동일한 더미 헤더 노드를 가리킬 때 큐가 비었다고 판단하는 것을 반영합니다.

**`Enqueue(x, Q)` (원소 삽입)**:
새로운 원소 `x`를 큐의 `R` 포인터가 가리키는 노드(현재 마지막 노드)의 뒤에 삽입하고, `R` 포인터를 새로 삽입된 노드로 업데이트합니다. 더미 헤더 덕분에 큐가 비어있을 때와 아닐 때의 로직이 동일합니다.

```c
void Enqueue(int x, LinkedQueue* Q) {
    Node* newNode = (Node*)malloc(sizeof(Node));
    if (!newNode) { /* 에러 처리 */ }
    newNode->data = x;
    newNode->next = NULL;

    Q->R->next = newNode; // 현재 R이 가리키는 노드의 다음으로 새 노드 연결
    Q->R = newNode;       // R 포인터를 새로 추가된 노드로 업데이트
}
```

**`Dequeue(Q)` (원소 삭제 - 슬라이드에는 명시되지 않았으나 핵심 연산)**:
큐의 `F` 포인터 다음 노드(즉, 첫 번째 실제 데이터 노드)의 원소를 삭제합니다. 더미 헤더 노드 자체는 삭제하지 않습니다. 큐에서 마지막 원소를 삭제하는 경우 `R` 포인터도 업데이트해야 합니다.

```c
int Dequeue(LinkedQueue* Q) {
    if (Empty(Q)) {
        // 큐가 비어있을 때 삭제 요청 시 예외 처리
        fprintf(stderr, "Error: Queue is empty.\n");
        exit(EXIT_FAILURE);
    }

    Node* dequeuedNode = Q->F->next; // 삭제할 노드 (첫 번째 실제 원소)
    int data = dequeuedNode->data;   // 삭제할 노드의 데이터

    Q->F->next = dequeuedNode->next; // 더미 헤더의 다음을 그 다음 노드로 연결

    // 만약 큐에서 마지막 원소를 제거했다면, R 포인터도 더미 헤더를 가리키도록 업데이트
    if (Q->F->next == NULL) {
        Q->R = Q->F;
    }

    free(dequeuedNode); // 노드 메모리 해제
    return data;
}
```

**구체적 예시**:

1.  **초기 상태 (빈 큐)**:
    *   `CreateQueue()` 호출 후 `F`와 `R` 모두 더미 헤더 노드를 가리키며, 더미 헤더의 `next`는 `NULL`입니다.
    ```
    Q->F ----> [ Dummy Header | NULL ] <---- Q->R
    ```
    `Empty(Q)`는 `true`를 반환합니다.

2.  **`Enqueue('a', Q)`**:
    *   새로운 노드 `[a | NULL]`를 생성합니다.
    *   `Q->R->next` (현재 더미 헤더의 `next`)가 `[a | NULL]`를 가리키게 합니다.
    *   `Q->R`이 `[a | NULL]`를 가리키도록 업데이트합니다.
    ```
    Q->F ----> [ Dummy Header | ] ---> [ a | NULL ] <---- Q->R
    ```

3.  **`Enqueue('b', Q)`**:
    *   새로운 노드 `[b | NULL]`를 생성합니다.
    *   `Q->R->next` (현재 `[a | NULL]`의 `next`)가 `[b | NULL]`를 가리키게 합니다.
    *   `Q->R`이 `[b | NULL]`를 가리키도록 업데이트합니다.
    ```
    Q->F ----> [ Dummy Header | ] ---> [ a | ] ---> [ b | NULL ] <---- Q->R
    ```

4.  **`Enqueue('c', Q)` (슬라이드의 그림과 동일한 상태)**:
    *   새로운 노드 `[c | NULL]`를 생성합니다.
    *   `Q->R->next` (현재 `[b | NULL]`의 `next`)가 `[c | NULL]`를 가리키게 합니다.
    *   `Q->R`이 `[c | NULL]`를 가리키도록 업데이트합니다.
    ```
    Q->F ----> [ Dummy Header | ] ---> [ a | ] ---> [ b | ] ---> [ c | NULL ] <---- Q->R
    ```
    이 상태에서 `Empty(Q)`는 `false`를 반환합니다 ($Q->F \neq Q->R$).

5.  **`Dequeue(Q)` (예시: 'a' 제거)**:
    *   `dequeuedNode`는 `Q->F->next` (`[a | ]`)를 가리킵니다.
    *   `data`에 'a'를 저장합니다.
    *   `Q->F->next` (더미 헤더의 `next`)를 `dequeuedNode->next` (`[b | ]`)로 변경합니다.
    *   `dequeuedNode`가 가리키던 노드를 메모리에서 해제합니다.
    ```
    Q->F ----> [ Dummy Header | ] ---> [ b | ] ---> [ c | NULL ] <---- Q->R
    ```

**시험 포인트**:

*   ⭐ **더미 헤더를 사용하는 주된 이유**: 큐 연산(특히 `Enqueue`, `Dequeue`) 구현 시 **경계 조건(empty queue, single-element queue)** 처리를 위한 특별한 코드를 없애고, 일관된 로직으로 코드를 단순화하여 구현 오류 가능성을 줄이기 위함입니다.
*   ⭐ **`F`와 `R` 포인터의 역할 및 상태 변화**:
    *   `F`: 항상 더미 헤더 노드를 가리킵니다.
    *   `R`: 큐의 마지막 실제 데이터 노드를 가리킵니다. 큐가 비어있는 경우 `F`와 동일하게 더미 헤더를 가리킵니다.
*   ⭐ **`Empty(Q)`의 정확한 조건**: `Q->F == Q->R` 또는 `Q->F->next == NULL`. 이 두 조건은 더미 헤더를 사용하는 연결 큐에서 큐가 비어있는지를 정확하게 나타냅니다.
*   ⭐ **`Enqueue` 및 `Dequeue` 연산의 시간 복잡도**: 더미 헤더를 사용하더라도 연결 리스트 기반이므로, 삽입 및 삭제 연산은 항상 $O(1)$의 시간 복잡도를 가집니다.
    *   `Enqueue`: $O(1)$
    *   `Dequeue`: $O(1)$

## 데이터구조 (CSED233) 슬라이드 노트 - Quiz: with vs. without dummy header

이 슬라이드는 연결 리스트(Linked List)에서 "더미 헤더(dummy header)" 노드의 사용 여부에 따른 장단점을 묻는 퀴즈 형식입니다. 주요 평가 기준은 속도, 공간 활용도, 코드 간결성입니다.

---

### **핵심 개념**

*   **더미 헤더(Dummy Header) 노드**: 연결 리스트의 맨 앞에 위치하며 실제 데이터를 저장하지 않는 특별한 노드입니다. 이 노드의 `next` 포인터는 리스트의 첫 번째 실제 데이터 노드를 가리키거나, 리스트가 비어있을 경우 `NULL`을 가리킵니다.
*   **더미 헤더 없음**: 리스트의 `head` 포인터가 직접 첫 번째 실제 데이터 노드를 가리킵니다. 리스트가 비어있을 경우 `head`는 `NULL`입니다.

더미 헤더의 주된 목적은 연결 리스트의 삽입 및 삭제 연산 시 발생할 수 있는 여러 가지 **예외 상황(edge cases)**, 특히 리스트의 맨 앞(head)에서의 연산이나 빈 리스트에 대한 연산을 단순화하여 코드의 복잡성을 줄이는 것입니다.

### **코드/수식 해설**

**1. `Node` 구조체 정의 (공통)**

```c
typedef struct Node {
    int data;
    struct Node* next;
} Node;
```

**2. 삽입 연산 예시 (맨 앞에 삽입)**

*   **더미 헤더 없음 (without dummy header)**
    리스트의 `head` 포인터가 변경될 수 있으므로, `head`가 `NULL`인지 여부에 대한 특별한 처리가 필요합니다. `head`를 인자로 받을 때 이중 포인터(`Node**`)를 사용하는 것이 일반적입니다.

    ```c
    void insert_front_without_dummy(Node** head_ref, int value) {
        Node* new_node = (Node*)malloc(sizeof(Node));
        if (new_node == NULL) { /* handle error */ return; }
        new_node->data = value;

        new_node->next = *head_ref; // 새 노드의 다음은 현재 head가 가리키던 노드
        *head_ref = new_node;       // head를 새 노드로 업데이트
    }
    ```
    이 경우, 빈 리스트에 삽입하든 비어있지 않은 리스트에 삽입하든 `*head_ref`가 업데이트되는 방식은 동일합니다. 하지만 다른 연산 (예: `delete_front`나 `insert_after` 같은 곳)에서 `NULL` 체크나 `head` 변경 여부에 대한 조건문이 더 복잡해질 수 있습니다.

*   **더미 헤더 사용 (with dummy header)**
    `dummy_head` 노드는 항상 존재하므로, `dummy_head->next`를 리스트의 실제 첫 노드처럼 다룰 수 있습니다. `head`가 변경될 걱정 없이 항상 `dummy_head->next`를 조작하면 됩니다. `dummy_head`는 단일 포인터로 전달 가능합니다.

    ```c
    // 리스트는 dummy_head를 가리키는 포인터로 표현되며, dummy_head는 항상 존재
    // ex: Node* list_head = create_dummy_node();
    void insert_front_with_dummy(Node* dummy_head, int value) {
        Node* new_node = (Node*)malloc(sizeof(Node));
        if (new_node == NULL) { /* handle error */ return; }
        new_node->data = value;

        new_node->next = dummy_head->next; // 새 노드의 다음은 dummy_head가 가리키던 실제 첫 노드
        dummy_head->next = new_node;       // dummy_head의 다음을 새 노드로 업데이트
    }
    ```
    여기서 `dummy_head`는 절대 `NULL`이 되지 않기 때문에, `dummy_head->next`에 대한 조작은 빈 리스트든 아니든 항상 동일한 로직으로 처리됩니다.

### **구체적 예시**

**예시: 빈 리스트에 10을 삽입하고, 이어서 20을 맨 앞에 삽입**

**1. 더미 헤더 없음:**
*   **초기 상태**: `head = NULL`
*   **10 삽입**:
    *   `new_node` 생성 (data=10, next=NULL)
    *   `new_node->next = head` (즉, `new_node->next = NULL`)
    *   `head = new_node` (즉, `head`는 10을 가리킴)
    *   결과: `head -> [10|NULL]`
*   **20 삽입**:
    *   `new_node` 생성 (data=20, next=NULL)
    *   `new_node->next = head` (즉, `new_node->next`는 10을 가리킴)
    *   `head = new_node` (즉, `head`는 20을 가리킴)
    *   결과: `head -> [20|.] -> [10|NULL]`

**2. 더미 헤더 사용:**
*   **초기 상태**: `dummy_head -> NULL` (항상 `dummy_head` 노드는 존재)
*   **10 삽입**:
    *   `new_node` 생성 (data=10, next=NULL)
    *   `new_node->next = dummy_head->next` (즉, `new_node->next = NULL`)
    *   `dummy_head->next = new_node` (즉, `dummy_head->next`는 10을 가리킴)
    *   결과: `dummy_head -> [10|NULL]`
*   **20 삽입**:
    *   `new_node` 생성 (data=20, next=NULL)
    *   `new_node->next = dummy_head->next` (즉, `new_node->next`는 10을 가리킴)
    *   `dummy_head->next = new_node` (즉, `dummy_head->next`는 20을 가리킴)
    *   결과: `dummy_head -> [20|.] -> [10|NULL]`

두 경우 모두 최종 결과는 동일하지만, 더미 헤더를 사용한 경우 `head` 포인터 자체가 변경되는 경우를 신경 쓸 필요가 없어 코드 작성이 더 직관적입니다.

### **시험 포인트**

⭐ **더미 헤더 사용 여부에 따른 장단점 비교**는 연결 리스트의 이해도를 측정하는 중요한 문제입니다. 다음 세 가지 기준에 대한 명확한 이해와 설명이 필요합니다.

1.  **Speed (속도)**:
    *   **더미 헤더 없음**: 빈 리스트, 또는 리스트의 시작 부분에서 삽입/삭제 시 `head` 포인터 자체를 변경해야 하거나 `NULL` 체크 같은 조건문이 추가될 수 있습니다. 이는 아주 미미하지만 잠재적인 오버헤드를 발생시킬 수 있습니다.
    *   **더미 헤더 사용**: 모든 삽입/삭제 연산이 `dummy_head->next` 또는 다른 노드의 `next` 포인터를 변경하는 일관된 로직으로 처리될 수 있습니다. `NULL` 체크나 `head` 포인터 변경에 대한 특별한 조건문이 줄어들어, 이론적으로는 약간 더 빠르거나 최소한 예측 가능한 실행 경로를 가질 수 있습니다.
    *   **결론**: 일반적으로 **점근적 시간 복잡도는 동일(O(1) 또는 O(N))**합니다. 하지만 **상수 시간 요소에서 더미 헤더 사용 시 미세한 이점**을 가질 수 있습니다 (조건 분기 감소).

2.  **Space utilization (공간 활용도)**:
    *   **더미 헤더 없음**: 실제 데이터 노드만 저장하므로, 리스트가 비어있을 때는 공간을 거의 차지하지 않습니다 (포인터 하나).
    *   **더미 헤더 사용**: 리스트가 비어있더라도 **하나의 추가 노드(더미 헤더)**를 항상 유지해야 합니다.
    *   **결론**: **더미 헤더 없음이 공간 효율성 측면에서 미세하게 우위**를 가집니다 (하나의 노드만큼). 그러나 전체 리스트 크기에 비하면 상수 오버헤드이므로 큰 차이는 아닙니다.

3.  **Code conciseness (코드 간결성)**:
    *   **더미 헤더 없음**: 빈 리스트, 리스트의 맨 앞에서의 삽입/삭제 등 **다양한 예외 상황(edge cases)을 처리하기 위한 추가적인 `if` 문이나 특별한 로직**이 필요합니다. 이는 코드를 더 길고 복잡하게 만들 수 있으며, 버그 발생 가능성을 높일 수 있습니다.
    *   **더미 헤더 사용**: `dummy_head`가 항상 존재하므로, 리스트가 비어있는 경우에도 `dummy_head->next`를 조작하는 방식으로 **모든 연산에 대해 일관된 코드 로직**을 적용할 수 있습니다. 이는 코드의 길이를 줄이고, 가독성을 높이며, 유지보수를 쉽게 합니다.
    *   **결론**: **더미 헤더 사용이 코드 간결성 및 유지보수성 측면에서 명확히 우위**를 가집니다.

**정리:**

더미 헤더는 공간을 약간 더 차지하지만 (상수 오버헤드), 코드의 간결성과 안정성을 크게 향상시켜 개발 및 디버깅 비용을 줄이는 데 기여합니다. 속도 측면에서는 대부분의 경우 큰 차이가 없거나 미세한 이점이 있을 수 있습니다. 따라서 실용적인 측면에서는 더미 헤더를 사용하는 것이 권장되는 경우가 많습니다.

**핵심 개념**

이 슬라이드는 큐(Queue)의 다양한 변형들을 소개합니다. 기본 큐는 FIFO (First-In, First-Out) 원칙을 따르지만, 특정 요구사항에 맞춰 기능이 확장된 큐들이 존재합니다.

1.  **일반 큐 (Queue)**:
    *   가장 기본적인 큐로, 먼저 들어온 요소가 먼저 나가는 (FIFO) 원칙을 따릅니다.
    *   새로운 요소는 'rear'(후단)에서 삽입되고, 요소는 'front'(전단)에서 삭제됩니다.
    *   슬라이드 상단의 첫 번째 그림이 이를 보여줍니다.

2.  **원형 큐 (Circular Queue)**:
    *   일반 큐의 문제점 중 하나인 '낭비되는 공간' 문제를 해결하기 위해 고안되었습니다.
    *   배열 기반 큐에서 요소 삭제 시 'front'가 이동하면 앞쪽 공간이 비게 되고, 'rear'가 배열의 끝에 도달하면 더 이상 삽입할 수 없게 되는 현상 (full 상태)이 발생합니다. 원형 큐는 배열의 끝과 시작을 연결하여 이 공간을 재활용합니다.
    *   슬라이드 하단 왼쪽 그림에서 마지막 요소가 다시 첫 번째 요소와 연결되는 고리 모양으로 표현됩니다.

3.  **우선순위 큐 (Priority Queue)**:
    *   요소가 큐에 삽입될 때 각 요소에 부여된 '우선순위'에 따라 처리 순서가 결정되는 큐입니다.
    *   가장 높은 우선순위를 가진 요소가 먼저 삭제(처리)됩니다. 우선순위가 같은 요소들 사이에서는 일반 큐의 FIFO 원칙을 따를 수 있습니다.
    *   슬라이드 오른쪽 상단 그림은 우선순위 1, 2, 3에 따라 별도의 큐가 관리되는 것을 시각적으로 보여줍니다 (실제 구현은 단일 구조 내에서 우선순위를 관리하는 경우가 많음).

4.  **덱 (Double-Ended Queue, Deque)**:
    *   양쪽 끝에서 삽입(insertion)과 삭제(deletion)가 모두 가능한 큐입니다.
    *   '덱'이라는 이름은 "Double-Ended Queue"의 줄임말이며, "Deck"으로 발음되기도 합니다.
    *   슬라이드 오른쪽 하단 그림은 큐의 양쪽 끝에서 삽입 및 삭제가 자유롭게 이루어질 수 있음을 보여줍니다.

**구체적 예시**

*   **일반 큐**: 은행 창구의 대기열, 프린터 출력 대기열. 먼저 온 사람이 먼저 서비스를 받거나 먼저 인쇄됩니다.
*   **원형 큐**: 운영체제에서 CPU 스케줄링 시 프로세스들이 번갈아 가며 CPU를 사용하는 라운드 로빈 스케줄링. 제한된 시간 내에 한 바퀴를 돌면서 모든 프로세스에게 공평하게 기회를 줍니다.
*   **우선순위 큐**: 병원의 응급실 환자 대기열. 먼저 온 환자보다는 위급한 환자가 먼저 치료를 받습니다.
*   **덱**: 웹 브라우저의 '뒤로 가기'/'앞으로 가기' 기능 (최근 방문 페이지를 양쪽에서 추가/제거). 또는 작업 스케줄링에서 작업 목록을 관리할 때, 새로운 작업을 앞이나 뒤에 추가하거나, 중요한 작업을 먼저 처리하기 위해 앞쪽에서 빼내는 경우.

**시험 포인트**

*   ⭐ **각 큐의 특징 및 동작 방식**: 일반 큐, 원형 큐, 우선순위 큐, 덱의 정의와 각각의 `enqueue` (삽입), `dequeue` (삭제) 연산이 어떻게 이루어지는지 정확히 이해하고 설명할 수 있어야 합니다. 특히, 원형 큐의 `front`와 `rear` 포인터 관리 로직, 우선순위 큐의 우선순위 결정 기준, 덱의 양방향 연산 가능성은 중요합니다.
*   ⭐ **각 큐가 해결하는 문제 또는 활용 분야**: 각 큐 타입이 어떤 특정 문제점을 해결하기 위해 고안되었는지 (예: 원형 큐는 배열 큐의 공간 낭비 문제) 또는 어떤 상황에 유용하게 사용되는지 (예: 우선순위 큐는 OS 스케줄링) 알아두세요.
*   **구현 시 고려사항**:
    *   일반 큐와 원형 큐는 배열 또는 연결 리스트로 구현할 수 있습니다. 배열 기반 원형 큐의 `front`, `rear` 인덱스 계산 ($$(rear + 1) \% capacity$$)은 중요합니다.
    *   우선순위 큐는 일반적으로 힙(Heap)으로 구현되어 효율적인 삽입/삭제($$O(\log N)$$)가 가능합니다. 연결 리스트로 구현 시 삽입 또는 삭제 중 하나가 $$O(N)$$이 될 수 있습니다.
    *   덱은 양방향 연결 리스트로 구현하는 것이 가장 유연합니다. 배열로 구현할 경우 크기 문제나 오버헤드가 발생할 수 있습니다.

**핵심 개념**:
이 슬라이드는 **코드 테스트(Code Testing)**, 특히 **테스트 케이스 작성**이 자료구조를 깊이 이해하는 데 가장 좋은 방법 중 하나임을 강조합니다. 테스트 케이스를 설계하는 과정에서 자료구조의 동작 방식, 성능 특성, 그리고 잠재적인 문제점을 파악할 수 있습니다.

테스트 케이스를 만들 때 고려해야 할 질문들은 다음과 같습니다:
*   특정 구현이 어떤 경우에 느려질 수 있는가? (시간 복잡도 분석과 연결)
*   특정 연산이 대략 어느 정도의 시간이 걸릴 것으로 예상하는가? (성능 예측 및 Big-O 분석)
*   자료구조의 정의에서 발생할 수 있는 **예외(edge cases)**는 무엇인가? (견고성 및 정확성 확인)
*   어떤 상황에서 버그를 발견할 수 있을까? (일반적인 디버깅 및 안정성 확보)

**코드/수식 해설**:
본 슬라이드에는 직접적인 코드나 수식은 제시되어 있지 않습니다. 다만, "How long do I expect certain operations to take?"와 같은 질문은 암묵적으로 시간 복잡도($O(N)$, $O(\log N)$, $O(1)$ 등)와 같은 수리적 분석의 중요성을 내포하고 있습니다.

**구체적 예시**:
*   **LinkedList의 `remove(index)` 연산**: 특정 인덱스의 요소를 제거하는 함수를 테스트할 때,
    *   **느려지는 경우**: 리스트의 마지막 요소를 제거하는 경우 (처음부터 끝까지 순회해야 할 수 있음).
    *   **예상 시간**: `N`개의 요소가 있다면 평균적으로 $O(N)$ 시간이 걸릴 것으로 예상. (만약 이중 연결 리스트라면 앞/뒤 모두 고려 가능)
    *   **예외(edge cases)**:
        *   빈 리스트에서 제거 시도 (`index`가 유효하지 않음).
        *   리스트의 첫 번째 요소 제거.
        *   리스트의 마지막 요소 제거.
        *   리스트 크기보다 큰 `index`로 제거 시도.
    *   **버그 발견**: `N`개의 요소가 있을 때 `index N-1`까지는 유효하지만 `index N`은 유효하지 않음을 명확히 해야 함. `NULL` 포인터 역참조 또는 잘못된 링크 연결 등의 버그가 발생할 수 있습니다.
*   **HashTable의 충돌(collision) 처리**: 해시 테이블 구현 시, 많은 키들이 동일한 해시 값으로 매핑되는 시나리오를 테스트하여 충돌 해결 전략(예: 체이닝, 선형 탐사)의 효율성을 확인합니다.

**시험 포인트**:
*   ⭐ **테스트 케이스 작성의 중요성**: 단순히 버그를 찾는 것을 넘어, 자료구조의 동작 원리, 성능 특성(시간/공간 복잡도), 그리고 예외 상황에 대한 이해를 높이는 학습 도구로서의 가치를 이해해야 합니다.
*   ⭐ **시간 복잡도와 연계된 테스트**: 특정 연산의 최악의 경우(worst-case) 시나리오를 유도하는 테스트 케이스를 설계하는 능력이 중요합니다. 예를 들어, 특정 자료구조가 언제 $O(N^2)$로 느려지는지, 언제 $O(1)$로 빠르게 동작하는지 등을 테스트를 통해 검증해야 합니다.
*   ⭐ **예외(Edge Cases) 처리**: 빈 자료구조, 가득 찬 자료구조, 단일 요소 자료구조, 경계 값 입력 등 일반적이지 않은 입력에 대한 자료구조의 동작 방식을 정확히 알고 있어야 하며, 이를 테스트 케이스로 구현할 수 있어야 합니다. 이는 자료구조의 견고성(robustness)과 직결됩니다.

## Slide 37

**핵심 개념**
이 슬라이드는 소프트웨어 테스트의 기본 정의와 효과적인 테스트를 위한 핵심 원칙들을 설명합니다.
-   **소프트웨어 테스트(Software Test)**: 평가하고자 하는 코드에 입력을 제공하고, 그 결과가 무엇이어야 하는지에 대한 '단언(assertion)'으로 마무리되는 별도의 코드 조각입니다. 이는 코드의 정확성, 견고성, 예상대로의 동작 여부를 검증하는 데 필수적인 과정입니다.
-   **효과적인 테스트를 위한 원칙**:
    1.  **분리(Isolate)**: 코드를 작고 독립적인 모듈로 나누어 각 모듈을 개별적으로 테스트합니다. 이는 문제 발생 시 원인 파악을 용이하게 합니다.
    2.  **점진적 구축(Build in increments)**: 가장 간단한 경우부터 시작하여 점차 복잡한 경우로 테스트 케이스를 확장해 나가는 계획을 세웁니다.
    3.  **지속적인 테스트(Test as you go)**: 코드가 성장하고 변경됨에 따라 테스트 코드도 함께 성장하고 지속적으로 실행되어야 합니다. 이는 회귀(regression) 오류를 방지하는 데 중요합니다.

**코드/수식 해설**
슬라이드 자체에 코드나 수식은 없지만, 소프트웨어 테스트는 아래와 같은 형태로 코드를 통해 구현됩니다. 예를 들어, 간단한 `add` 함수를 테스트하는 경우를 생각해봅시다.

```c++
// 테스트 대상 코드 (Production Code)
int add(int a, int b) {
    return a + b;
}

// 테스트 코드 (Test Code)
#include <cassert> // C++의 단언(assertion) 라이브러리

void test_add_positive_numbers() {
    int result = add(2, 3);
    assert(result == 5); // 결과가 5여야 한다고 단언
}

void test_add_negative_numbers() {
    int result = add(-2, -3);
    assert(result == -5); // 결과가 -5여야 한다고 단언
}

void test_add_zero() {
    int result = add(5, 0);
    assert(result == 5); // 결과가 5여야 한다고 단언
}

int main() {
    test_add_positive_numbers();
    test_add_negative_numbers();
    test_add_zero();
    // 모든 테스트 통과 시 메시지 출력
    // assert 실패 시 프로그램 종료
    return 0;
}
```
여기서 `assert(result == expected_value);` 부분이 슬라이드에서 언급된 "assertion of what the result should be"에 해당합니다.

**구체적 예시**
여러분들이 `LinkedList` 자료구조를 구현한다고 가정해 봅시다.
1.  **Isolate**: `add_front()`, `add_back()`, `remove_front()`, `remove_back()`, `find()`, `get_size()` 등 각 기능을 독립적인 함수(모듈)로 구현하고, 각 함수에 대한 테스트를 개별적으로 작성합니다. `add_front()`를 테스트할 때 `remove_front()`의 버그가 영향을 미치지 않도록 합니다.
2.  **Build in increments**:
    *   **간단한 경우**: 비어있는 리스트에 요소를 추가하는 경우, 하나의 요소가 있는 리스트에서 요소를 삭제하는 경우.
    *   **점진적 복잡성**: 여러 개의 요소가 있는 리스트의 중간에 추가/삭제, 리스트의 처음/끝에 추가/삭제, 존재하지 않는 요소를 찾는 경우, 리스트가 가득 찼거나 비어있을 때의 엣지 케이스 등.
3.  **Test as you go**: `add_front()` 함수를 구현하고 테스트 통과를 확인한 후에 `add_back()`을 구현합니다. `add_back()` 구현 후 테스트할 때, 이전에 통과했던 `add_front()` 테스트도 다시 실행하여 `add_back()`의 변경이 `add_front()`의 기존 기능에 영향을 주지 않았는지 확인합니다. (이를 회귀 테스트라고 합니다.)

**시험 포인트**
-   ⭐ **소프트웨어 테스트의 정의**를 정확히 이해하고 설명할 수 있어야 합니다. 특히 'input' 제공과 'assertion'을 통해 결과를 확인한다는 점이 중요합니다.
-   ⭐ **효과적인 테스트를 위한 3가지 원칙(Isolate, Build in increments, Test as you go)** 각각의 의미와 왜 중요한지 설명하고, 실제 코드 개발 과정에서 어떻게 적용될 수 있는지 구체적인 예시를 들어 설명할 수 있어야 합니다.
-   자료구조 과목에서는 단순히 자료구조를 구현하는 것뿐만 아니라, **구현한 자료구조가 올바르게 동작하는지 검증하는 테스트 코드의 중요성**을 이해하는 것이 중요합니다.

**데이터구조 (CSED233) 강의 노트**

### Types of Tests (테스트 유형)

**핵심 개념**
이 슬라이드는 소프트웨어 테스트의 두 가지 주요 유형인 블랙박스 테스트와 화이트박스 테스트를 설명합니다. 이 두 유형은 테스트를 수행하는 관점과 대상에 따라 구분됩니다.

*   **블랙박스 테스트 (Black Box Test)**:
    *   시스템의 내부 구현이나 구조를 알지 못한 채, 오직 외부에서 보이는 기능(동작)과 요구사항만을 기반으로 테스트하는 방식입니다.
    *   사용자의 관점에서 코드가 명세된 계약(요구사항)을 잘 준수하는지, 그리고 성능과 효율성은 어떤지를 중점적으로 확인합니다.

*   **화이트박스 테스트 (White Box Test)**:
    *   시스템의 내부 구현, 구조, 코드 흐름 등을 모두 이해한 상태에서 테스트하는 방식입니다.
    *   주로 개발자(작성자)가 코드를 개발하면서 직접 작성하며, 전체 요구사항을 더 작은 단계로 나누어 테스트합니다.
    *   "단위 테스트(unit tests)"가 대표적인 화이트박스 테스트로, 구현의 작은 부분(함수, 메서드)이 단일 어설션(assertion)에 따라 올바르게 동작하는지 검증합니다.

**코드/수식 해설**
해당 슬라이드에는 직접적인 코드나 수식은 포함되어 있지 않습니다.

**구체적 예시**

*   **블랙박스 테스트 예시**:
    *   **자동 판매기 비유**: 여러분이 음료수를 뽑기 위해 자동 판매기 앞에 섰다고 가정해 봅시다. 여러분은 판매기 내부의 회로, 배선, 동작 방식 등을 전혀 알 필요가 없습니다. 그저 돈을 넣고 버튼을 눌렀을 때, 원하는 음료수가 올바르게 나오고 거스름돈이 정확히 나오는지(기능), 그리고 음료수가 너무 오래 걸리지 않는지(성능)만 확인하면 됩니다. 이것이 블랙박스 테스트와 유사합니다.
    *   **데이터구조 적용**: `ArrayList`의 `add(element)` 메서드를 테스트할 때, 내부적으로 배열의 크기를 어떻게 늘리는지, 메모리 재할당이 어떻게 일어나는지 전혀 고려하지 않고, 단지 `add`를 호출했을 때 요소가 성공적으로 추가되고 `size`가 증가하는지, 그리고 많은 요소를 추가했을 때 성능이 저하되지 않는지만 확인하는 것이 블랙박스 테스트입니다.

*   **화이트박스 테스트 예시**:
    *   **자동차 정비공 비유**: 자동차 정비공이 차의 문제를 진단할 때, 단순히 시동이 걸리지 않는다고만 말하지 않습니다. 엔진룸을 열고, 배터리, 점화 플러그, 연료 펌프 등 내부 부품과 연결 상태를 직접 확인하고, 각 부품이 올바르게 작동하는지 세부적으로 점검합니다. 이것이 화이트박스 테스트와 유사합니다.
    *   **데이터구조 적용**: `LinkedList`의 `remove(index)` 메서드를 테스트할 때, 주어진 인덱스에 해당하는 노드를 찾아 연결을 끊고, 이전 노드와 다음 노드를 올바르게 연결하는 로직이 정확한지, 엣지 케이스(리스트의 첫 노드, 마지막 노드를 삭제하는 경우)에서도 포인터가 올바르게 업데이트되는지를 코드의 내부 구현을 직접 보면서 확인하는 것이 화이트박스 테스트입니다. 특히 각 노드를 찾거나 연결을 변경하는 과정의 `NPE` (Null Pointer Exception) 발생 가능성 등 내부적인 오류를 찾습니다.

**시험 포인트**

*   ⭐ **블랙박스 테스트와 화이트박스 테스트의 정의와 핵심 차이점을 정확히 설명할 수 있어야 합니다.** (내부 구현 지식의 유무)
*   ⭐ 각 테스트 유형의 목적과 주안점을 이해하는 것이 중요합니다.
    *   블랙박스: ⭐ **요구사항 준수, 사용자 관점, 성능/효율성**.
    *   화이트박스: ⭐ **구현 상세 분석, 개발자 관점, 단위 테스트, 내부 로직 검증**.
*   ⭐ "단위 테스트(unit tests)"가 화이트박스 테스트의 대표적인 형태로 주로 개발자에 의해 작성된다는 점을 기억해야 합니다.
*   주어진 시나리오나 코드 스니펫에 대해 어떤 테스트 유형이 적합한지 판단할 수 있어야 합니다.

## Slide 39

### 데이터구조 테스트 전략

**핵심 개념**:
데이터 구조의 구현이 올바르게 동작하는지 확인하기 위해서는 다양한 시나리오와 조건에서 철저한 테스트가 필수적입니다. 이 슬라이드는 자료구조를 테스트할 때 고려해야 할 다섯 가지 핵심 범주를 제시합니다. 이는 단순히 기능적 올바름뿐만 아니라, 예상치 못한 입력, 경계 조건, 그리고 성능 측면까지 포괄하여 견고하고 효율적인 자료구조를 개발하는 데 중요합니다.

1.  **예상 동작 (Expected behavior)**: 코드가 일반적이고 '친화적인' 조건에서 의도한 대로 동작하는지 확인합니다.
2.  **금지된 입력 (Forbidden Input)**: 사용자가 잘못된 입력을 제공했을 때 코드가 어떻게 반응하는지, 오류를 적절히 처리하는지 테스트합니다.
3.  **비어있음/널 (Empty/Null)**: 자료구조가 비어있는 초기 상태, 널(null) 값 입력, 또는 0, -1과 같은 특수 값에 대해 안전하게 동작하는지 확인합니다.
4.  **경계/특수 사례 (Boundary/Edge Cases)**: 자료구조의 '끝'에 해당하는 조건들 (첫 번째 항목, 마지막 항목, 가득 찬 컬렉션 등)에서 올바르게 동작하는지 테스트합니다.
5.  **규모 (Scale)**: 자료의 개수가 적을 때와 많을 때(예: 10개 vs 10,000개) 성능이나 동작 방식에 차이가 없는지, 특히 시간/공간 복잡도 측면에서 예상한 대로 동작하는지 확인합니다.

**구체적 예시**:

*   **예상 동작**:
    *   `LinkedList`에 `add("A")`, `add("B")` 후 `remove("A")`를 호출했을 때, `LinkedList`가 `["B"]`가 되고 `size()`가 1을 반환하는지 테스트합니다.
    *   `HashTable`에 키-값 쌍을 여러 개 추가하고, 기존 키로 `get()`을 호출했을 때 정확한 값을 반환하는지 테스트합니다.

*   **금지된 입력**:
    *   `ArrayList`에서 `get(-1)`이나 `get(size())`와 같이 유효하지 않은 인덱스로 접근을 시도했을 때, `IndexOutOfBoundsException`과 같은 적절한 예외를 발생시키는지 테스트합니다.
    *   `Stack` 구현에서 `push(null)`을 허용할지, 아니면 예외를 던질지 결정하고 테스트합니다.

*   **비어있음/널**:
    *   비어있는 `Stack`에서 `pop()`이나 `peek()`을 호출했을 때 `EmptyStackException` 등 적절한 예외를 발생시키는지 테스트합니다.
    *   비어있는 `Queue`에서 `dequeue()`를 호출했을 때 동작을 확인합니다.
    *   `LinkedList` 생성 시 초기 사이즈나 인자를 0 또는 `null`로 주었을 때 정상적으로 초기화되는지 확인합니다.

*   **경계/특수 사례**:
    *   `ArrayList`가 가득 찬 상태에서 `add()`를 호출했을 때, `resize()` 로직이 올바르게 동작하여 새로운 요소를 추가하고 기존 요소들을 유지하는지 테스트합니다.
    *   `LinkedList`에서 첫 번째 노드를 삭제(`removeFirst()`)하거나 마지막 노드를 삭제(`removeLast()`)했을 때, `head` 또는 `tail` 포인터가 올바르게 업데이트되는지 테스트합니다.
    *   `Binary Search Tree`에서 루트 노드만 있는 경우, 또는 리프 노드를 삭제하는 경우를 테스트합니다.

*   **규모**:
    *   `N`개의 요소를 `Heap`에 추가할 때, $N=10, N=1000, N=100000$ 등 다양한 `N` 값에 대해 `add()` 연산의 총 시간 복잡도가 $O(N \log N)$에 가까운지 측정하여 효율성을 검증합니다.
    *   `HashTable`에서 충돌(collision)이 많아지는 특정 `N` 값에 대해 `search()` 연산의 성능이 급격히 저하되지는 않는지 테스트하여 `load factor`나 `rehash` 전략의 유효성을 평가합니다.

**시험 포인트**:

*   ⭐ **자료구조 구현 시 발생할 수 있는 주요 예외 상황(Empty, Null, Index Out of Bounds 등)을 예측하고 적절히 처리하는 방법**을 이해하고 있어야 합니다.
*   ⭐ **경계 조건(Boundary Cases) 테스트의 중요성**: 특히 배열 기반 자료구조의 크기 조절(resizing), 연결 리스트의 첫/마지막 노드 처리, 트리에서의 리프 노드/루트 노드 처리 등이 중요하게 다뤄질 수 있습니다.
*   ⭐ **시간/공간 복잡도 분석의 실제 적용**: 다양한 규모(Scale)에서의 테스트는 이론적인 시간/공간 복잡도 분석(Big-O Notation)과 실제 성능 간의 일치 여부를 확인하는 중요한 과정입니다. 특정 연산에 대한 최악의 경우(Worst-case) 시나리오를 고려하여 테스트 계획을 세울 수 있어야 합니다.
*   ⭐ `Assert` 구문 등을 활용한 테스트 코드 작성 능력 또한 중요합니다. (예: `assert list.size() == 1;`)

### **List (리스트)**

*   **핵심 개념**:
    List는 데이터를 순서대로 나열하여 저장하는 추상 자료형(ADT)입니다. 각 데이터는 인덱스를 통해 위치가 결정되며, 삽입, 삭제, 검색, 접근 등의 연산이 가능합니다. 배열과 달리 크기가 동적으로 변할 수 있는 특징을 가집니다.
*   **코드/수식 해설**:
    List ADT는 일반적으로 다음과 같은 연산들을 포함합니다.
    ```java
    interface List<E> {
        void add(E element); // 리스트의 끝에 원소 추가
        void add(int index, E element); // 특정 인덱스에 원소 삽입
        E get(int index); // 특정 인덱스의 원소 반환
        E remove(int index); // 특정 인덱스의 원소 삭제 및 반환
        int size(); // 리스트의 크기 반환
        boolean isEmpty(); // 리스트가 비어있는지 확인
    }
    ```
*   **구체적 예시**:
    우리가 사용하는 웹 브라우저의 즐겨찾기 목록이나 음악 플레이어의 재생 목록이 List의 좋은 예시입니다. 항목들이 순서대로 나열되어 있고, 특정 위치에 새 항목을 추가하거나 기존 항목을 삭제할 수 있습니다.
*   **시험 포인트**:
    ⭐ List가 배열과 어떤 점에서 다른지 (동적 크기), List ADT가 제공하는 주요 연산들을 이해하는 것이 중요합니다.

### **List implementations (리스트 구현)**

*   **핵심 개념**:
    List는 크게 두 가지 방식으로 구현될 수 있습니다.
    1.  **배열 기반 리스트 (Array-based List / Dynamic Array)**: 내부적으로 배열을 사용하여 데이터를 저장합니다. 필요에 따라 배열의 크기를 동적으로 확장합니다. (예: Java의 `ArrayList`, C++의 `std::vector`)
    2.  **연결 리스트 (Linked List)**: 각 데이터(노드)가 다음/이전 노드의 주소(포인터)를 가지고 연결되어 있는 구조입니다. (예: Java의 `LinkedList`, C++의 `std::list`)
*   **코드/수식 해설**:
    *   **배열 기반 리스트**:
        *   임의 접근 (Random Access): 인덱스를 통한 접근은 O(1)입니다.
        *   중간 삽입/삭제: 원소를 옮겨야 하므로 평균적으로 O(N)입니다.
        *   마지막 삽입/삭제: O(1)입니다.
        *   크기 확장 (resize): 내부 배열이 가득 찼을 때 더 큰 배열을 새로 할당하고 기존 원소를 복사해야 하므로 O(N)입니다.
            $$T_{resize} = O(N)$$
    *   **연결 리스트**:
        *   탐색 (Search): 특정 원소를 찾기 위해 처음부터 순회해야 하므로 O(N)입니다.
        *   중간 삽입/삭제: 삽입/삭제할 위치의 노드를 찾았다면 (포인터 조작만 필요) O(1)입니다. 그러나 해당 노드를 찾는 과정은 O(N)입니다.
        *   머리/꼬리 삽입/삭제: O(1)입니다 (단일 연결 리스트에서는 꼬리 삽입 시 꼬리 노드까지 탐색해야 할 수 있음).
        *   노드 구조 (단일 연결 리스트):
            ```java
            class Node<E> {
                E data;
                Node<E> next;
                Node(E data) {
                    this.data = data;
                    this.next = null;
                }
            }
            ```
*   **구체적 예시**:
    *   **배열 기반 리스트**: 사진 앨범 앱에서 사진들을 썸네일로 볼 때, 특정 인덱스의 사진을 바로 보고 싶을 때 유용합니다.
    *   **연결 리스트**: 어떤 책의 목차에서 특정 장(chapter)을 삽입하거나 삭제할 때, 해당 장의 앞뒤 페이지만 변경하면 되므로 전체 구조를 크게 바꾸지 않아도 됩니다.
*   **시험 포인트**:
    ⭐ 배열 기반 리스트와 연결 리스트의 연산별 시간 복잡도를 정확히 비교하고 설명할 수 있어야 합니다. 특히 **임의 접근**, **중간 삽입/삭제**, **동적 배열의 크기 확장** 시의 시간 복잡도 차이에 주목하세요.

### **Stack (스택)**

*   **핵심 개념**:
    Stack은 LIFO(Last-In, First-Out) 원칙을 따르는 선형 자료구조입니다. 가장 마지막에 삽입된 원소가 가장 먼저 삭제됩니다. `push` (삽입), `pop` (삭제), `peek` (최상단 원소 확인), `isEmpty` (스택이 비어있는지 확인) 연산을 제공합니다.
*   **코드/수식 해설**:
    스택의 핵심 연산은 다음과 같습니다.
    *   `push(element)`: 스택의 최상단에 `element`를 추가합니다.
    *   `pop()`: 스택의 최상단 원소를 제거하고 반환합니다. 스택이 비어있으면 오류를 발생시킵니다.
    *   `peek()`: 스택의 최상단 원소를 반환하지만 제거하지는 않습니다.
    ```java
    interface Stack<E> {
        void push(E element);
        E pop();
        E peek();
        boolean isEmpty();
        int size();
    }
    ```
*   **구체적 예시**:
    식당에서 쌓여있는 접시 더미를 생각해보세요. 가장 위에 있는 접시를 먼저 가져가고(pop), 새 접시는 항상 맨 위에 쌓입니다(push). 웹 브라우저의 '뒤로 가기' 기능도 스택을 사용하여 방문 기록을 관리하는 대표적인 예입니다.
*   **시험 포인트**:
    ⭐ Stack의 LIFO 원칙을 정확히 이해하고, `push`, `pop` 연산이 $O(1)$의 시간 복잡도를 갖는 이유를 설명할 수 있어야 합니다.

### **Stack implementations (스택 구현)**

*   **핵심 개념**:
    Stack은 배열 기반 또는 연결 리스트 기반으로 구현될 수 있습니다. 두 구현 방식 모두 스택의 핵심 연산인 `push`, `pop`, `peek`를 $O(1)$의 시간 복잡도로 제공할 수 있습니다.
    1.  **배열 기반 스택**: 고정된 크기의 배열을 사용하여 구현하거나, 동적 배열(dynamic array)을 사용하여 필요에 따라 크기를 확장할 수 있습니다. `top` 인덱스를 사용하여 최상단 원소를 가리킵니다.
    2.  **연결 리스트 기반 스택**: 연결 리스트의 `head` 노드를 스택의 `top`으로 사용하여 구현합니다. `push`는 `head`에 새 노드를 추가하고, `pop`은 `head` 노드를 제거합니다.
*   **코드/수식 해설**:
    *   **배열 기반 스택**: `top` 인덱스를 사용하여 배열의 마지막 원소를 추적합니다. `push` 시 `top++` 후 삽입, `pop` 시 반환 후 `top--`.
        ```java
        // 배열 기반 스택의 push (배열이 가득 차지 않았다고 가정)
        public void push(E element) {
            array[++top] = element;
        }
        // 배열 기반 스택의 pop (스택이 비어있지 않다고 가정)
        public E pop() {
            return array[top--];
        }
        ```
    *   **연결 리스트 기반 스택**: `head` (또는 `top`) 포인터만 관리하면 됩니다.
        ```java
        // 연결 리스트 기반 스택의 push
        public void push(E element) {
            Node<E> newNode = new Node<>(element);
            newNode.next = head;
            head = newNode;
        }
        // 연결 리스트 기반 스택의 pop
        public E pop() {
            if (isEmpty()) throw new EmptyStackException();
            E data = head.data;
            head = head.next;
            return data;
        }
        ```
    두 구현 방식 모두 `push`, `pop`, `peek`는 $$O(1)$$의 시간 복잡도를 가집니다.
*   **구체적 예시**:
    (스택 ADT의 예시와 동일하게 적용 가능합니다.)
*   **시험 포인트**:
    ⭐ 배열 기반과 연결 리스트 기반 스택 구현 모두 `push`와 `pop` 연산이 상수 시간 $O(1)$에 수행되는 이유를 설명할 수 있어야 합니다. 배열 기반의 경우 `Overflow`, `Underflow` 처리 방안도 중요합니다.

### **Queue (큐)**

*   **핵심 개념**:
    Queue는 FIFO(First-In, First-Out) 원칙을 따르는 선형 자료구조입니다. 가장 먼저 삽입된 원소가 가장 먼저 삭제됩니다. `enqueue` (삽입), `dequeue` (삭제), `peek` (전단 원소 확인), `isEmpty` (큐가 비어있는지 확인) 연산을 제공합니다.
*   **코드/수식 해설**:
    큐의 핵심 연산은 다음과 같습니다.
    *   `enqueue(element)`: 큐의 후단(rear)에 `element`를 추가합니다.
    *   `dequeue()`: 큐의 전단(front) 원소를 제거하고 반환합니다. 큐가 비어있으면 오류를 발생시킵니다.
    *   `peek()`: 큐의 전단 원소를 반환하지만 제거하지는 않습니다.
    ```java
    interface Queue<E> {
        void enqueue(E element);
        E dequeue();
        E peek();
        boolean isEmpty();
        int size();
    }
    ```
*   **구체적 예시**:
    은행 창구의 대기줄을 생각해보세요. 먼저 온 손님이 먼저 서비스를 받고(dequeue), 새로운 손님은 항상 줄의 맨 뒤에 섭니다(enqueue). 프린터 인쇄 대기열이나 운영체제의 작업 스케줄링에도 큐가 사용됩니다.
*   **시험 포인트**:
    ⭐ Queue의 FIFO 원칙을 정확히 이해하고, `enqueue`, `dequeue` 연산이 $O(1)$의 시간 복잡도를 갖는 이유를 설명할 수 있어야 합니다.

### **Queue implementations (큐 구현)**

*   **핵심 개념**:
    Queue도 배열 기반 또는 연결 리스트 기반으로 구현될 수 있습니다. 두 구현 방식 모두 큐의 핵심 연산인 `enqueue`, `dequeue`, `peek`를 $O(1)$의 시간 복잡도로 제공할 수 있습니다.
    1.  **배열 기반 큐 (원형 큐, Circular Queue)**: 선형 배열로 구현할 경우 `dequeue` 시 앞의 원소들을 당겨야 하므로 비효율적입니다. 이를 해결하기 위해 원형 배열(circular array)을 사용하여 `front`와 `rear` 포인터를 원형으로 이동시킵니다.
    2.  **연결 리스트 기반 큐**: 연결 리스트의 `head` 노드를 `front`로, `tail` 노드를 `rear`로 사용하여 구현합니다. `enqueue`는 `tail`에 새 노드를 추가하고, `dequeue`는 `head` 노드를 제거합니다.
*   **코드/수식 해설**:
    *   **배열 기반 (원형 큐)**: `front`와 `rear` 인덱스를 사용하여 큐의 앞과 뒤를 추적합니다. 모듈로 연산을 사용하여 배열의 끝에 도달하면 다시 처음으로 돌아오게 합니다.
        *   `enqueue` 시: `rear = (rear + 1) % capacity`
        *   `dequeue` 시: `front = (front + 1) % capacity`
        $$next\_index = (current\_index + 1) \pmod{capacity}$$
    *   **연결 리스트 기반 큐**:
        ```java
        // 연결 리스트 기반 큐의 enqueue
        public void enqueue(E element) {
            Node<E> newNode = new Node<>(element);
            if (isEmpty()) {
                front = newNode;
            } else {
                rear.next = newNode;
            }
            rear = newNode;
        }
        // 연결 리스트 기반 큐의 dequeue
        public E dequeue() {
            if (isEmpty()) throw new EmptyQueueException();
            E data = front.data;
            front = front.next;
            if (front == null) { // 큐가 비게 되면 rear도 null로
                rear = null;
            }
            return data;
        }
        ```
    두 구현 방식 모두 `enqueue`, `dequeue`, `peek`는 $$O(1)$$의 시간 복잡도를 가집니다.
*   **구체적 예시**:
    (큐 ADT의 예시와 동일하게 적용 가능합니다.)
*   **시험 포인트**:
    ⭐ **원형 큐의 필요성**과 동작 원리를 정확히 이해해야 합니다. `front`, `rear` 인덱스 관리 및 모듈로 연산의 의미를 설명할 수 있어야 합니다. 배열 기반과 연결 리스트 기반 큐 구현 모두 `enqueue`와 `dequeue` 연산이 상수 시간 $O(1)$에 수행되는 이유를 숙지하세요.

### **Code testing (코드 테스팅)**

*   **핵심 개념**:
    구현한 자료구조가 올바르게 작동하는지, 예상치 못한 상황에서도 견고하게 동작하는지, 그리고 성능 요구사항을 충족하는지 검증하는 과정입니다. 이는 소프트웨어 개발에서 버그를 조기에 발견하고 코드의 신뢰성을 높이는 데 필수적입니다.
*   **코드/수식 해설**:
    단위 테스트(Unit Testing)는 특정 함수나 모듈이 제대로 동작하는지 개별적으로 검증하는 가장 기본적인 테스트 방법입니다. 다양한 테스트 케이스를 설계하여 다음을 확인해야 합니다.
    *   **정상 동작 (Happy Path)**: 일반적인 사용 시나리오에서 올바르게 동작하는가?
    *   **엣지 케이스 (Edge Cases)**: 자료구조가 비어있을 때, 가득 찼을 때, 한 개의 원소만 있을 때 등 경계 조건에서 올바르게 동작하는가?
    *   **예외 처리 (Exception Handling)**: 잘못된 입력이나 불가능한 연산 시 (예: 빈 스택에서 `pop`) 적절하게 예외를 처리하는가?
    *   **성능 테스트 (Performance Testing)**: 특히 시간 복잡도가 중요한 자료구조에서 대량의 데이터를 처리할 때 이론적 복잡도에 맞게 동작하는가? (예: N이 충분히 클 때 O(N) vs O(N^2)의 차이 확인)
*   **구체적 예시**:
    *   **스택 테스트**:
        *   빈 스택에서 `pop()`을 호출했을 때 `EmptyStackException`이 발생하는지 확인.
        *   `push(1)`, `push(2)` 후 `pop()`하면 2가 반환되는지, 그 다음 `pop()`하면 1이 반환되는지 확인.
        *   `push`와 `pop`을 스택이 가득 찰 때까지 반복했을 때 오버플로우/언더플로우 처리가 잘 되는지 확인.
*   **시험 포인트**:
    ⭐ 자료구조 구현 시 테스트의 중요성과 그 목적을 설명할 수 있어야 합니다. 특히 **정상 동작** 외에 **엣지 케이스**와 **예외 처리**에 대한 테스트가 왜 중요한지 강조할 수 있어야 합니다.
